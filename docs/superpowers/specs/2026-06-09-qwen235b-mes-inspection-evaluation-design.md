# Qwen235B MES巡检推理能力评估设计

## 概述

本设计旨在评估Qwen235B模型在MES系统巡检场景下的多步推理能力。通过模拟一个真实的JDBC Connection reset事故场景，测试模型是否能够正确分析巡检数据、识别根本原因并提出合理的解决方案。

## 目标

1. **模拟事故场景**：基于真实的JDBC Connection reset事故报告，生成模拟的MES巡检脚本输出数据。
2. **测试推理能力**：调用Qwen235B API，要求模型对模拟数据进行多步推理分析。
3. **评估模型表现**：从诊断准确性、推理过程质量和解决方案建议三个维度对模型响应进行评分。
4. **生成评估报告**：输出详细的评估报告，为决策提供依据。

## 范围

- **仅针对提供的JDBC Connection reset事故报告**
- **不实际调用MES巡检脚本**，而是模拟其JSON输出
- **评估Qwen3-235B-A22B-w8a8模型**（通过提供的API端点）

## 组件设计

### 1. 事故场景模拟器

**职责**：基于事故报告生成模拟的巡检JSON数据。

**输入**：事故报告文本
**输出**：模拟的巡检JSON数据（模拟`oracle_check.py`、`jvm_check.py`等脚本的输出格式）

**模拟数据示例**：
```json
{
  "service": "oracle",
  "timestamp": "2026-05-28T11:00:00Z",
  "status": "critical",
  "database": "MESDB",
  "checks": {
    "sessions": {
      "status": "critical",
      "active": 1400,
      "inactive": 50,
      "total": 1450,
      "max_sessions": 500,
      "usage_percent": 290.0
    },
    "connection_errors": {
      "status": "critical",
      "error_type": "JDBC Connection reset",
      "count": 1450,
      "affected_nodes": "cluster-wide"
    }
  },
  "exit_code": 2
}
```

### 2. API调用器

**配置**：

- API密钥：`sk-dooFBpzVWgrvf32YLPFfq5r63dEYHELlUjMT84KrEH5wG0zN`
- 模型：`Qwen3-235B-A22B-w8a8`
- 最大token数：2000（足够详细的推理）

**提示词设计**：
```
你是一位MES系统专家，擅长分析系统巡检数据并诊断问题。请分析以下巡检数据，并按照以下步骤进行推理：

1. **识别异常指标**：从数据中找出异常的指标。
2. **分析可能的原因**：基于异常指标，分析可能导致问题的原因。
3. **确定根本原因**：从可能的原因中，确定最可能的根本原因。
4. **提出解决方案**：针对根本原因，提出具体的解决方案。

请确保你的推理过程清晰、逻辑严密，并提供详细的解释。

**巡检数据**：
{模拟的JSON数据}

**事故背景**：
{事故报告摘要}
```

### 3. 推理评估器

**职责**：分析模型响应，根据预设标准进行评分。

**评估维度**：
1. **诊断准确性（40%）**：模型是否正确识别根本原因（DNS问题）。
   - 5分：明确识别DNS解析问题为根本原因
   - 4分：识别出DNS相关问题，但不够明确
   - 3分：识别出网络或连接问题，但未具体到DNS
   - 2分：识别出一些相关问题，但偏离根本原因
   - 1分：未能识别出相关问题

2. **推理过程质量（30%）**：推理步骤是否清晰、逻辑是否严密。
   - 5分：推理步骤完整、逻辑清晰、解释详细
   - 4分：推理步骤基本完整，逻辑较清晰
   - 3分：推理步骤有缺失，但整体逻辑尚可
   - 2分：推理步骤混乱，逻辑不清晰
   - 1分：几乎没有推理过程

3. **解决方案建议（30%）**：提出的解决方案是否合理、可行。
   - 5分：解决方案具体、可行、针对性强
   - 4分：解决方案合理，但可行性或针对性稍弱
   - 3分：解决方案有一定道理，但不够具体
   - 2分：解决方案偏离问题或不可行
   - 1分：没有提出解决方案

**评分方法**：使用关键词匹配和语义分析相结合的方式。对于诊断准确性，检查响应中是否包含“DNS”、“解析”、“InetAddress.getLocalHost()”等关键词。对于推理过程质量和解决方案建议，使用简单的启发式规则进行评估。

### 4. 报告生成器

**职责**：将评估结果生成Markdown报告。

**报告结构**：
1. **执行摘要**：总体评分和结论。
2. **模拟数据**：生成的巡检JSON数据。
3. **模型响应**：模型的完整推理响应。
4. **详细评估**：每个维度的评分和分析。
5. **结论和建议**：对Qwen235B模型是否适合MES巡检推理的最终评估。

## 数据流

1. **事故场景模拟器**生成模拟的巡检JSON数据。
2. 将数据与提示词一起发送给**API调用器**。
3. **API调用器**调用Qwen235B API，接收模型的推理响应。
4. **推理评估器**根据预设标准对响应进行评分。
5. **报告生成器**将评分和详细分析输出为Markdown报告。

## 错误处理

- **API调用失败**：记录错误并重试（最多3次）。
- **模型响应格式错误**：尝试解析，如果失败则记录错误并继续评估。
- **评估过程异常**：记录异常并继续生成报告。

## 验证标准

1. **脚本可运行**：Python脚本能够成功执行，无语法错误。
2. **API调用成功**：能够成功调用Qwen235B API并获取响应。
3. **评估逻辑正确**：评估器能够根据模型响应生成合理的评分。
4. **报告生成完整**：报告包含所有必要部分，格式正确。

## 风险与缓解

1. **API端点不可用**：缓解措施：脚本应包含错误处理和重试逻辑。
2. **模型响应质量低**：缓解措施：评估器应能处理低质量响应，并给出相应评分。
3. **评估标准主观性**：缓解措施：使用明确的评分标准和关键词匹配，减少主观性。
