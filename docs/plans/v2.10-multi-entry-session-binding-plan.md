## Phase 3: EntryAdapter Interface

### Files to create

| File | Action |
|------|--------|
| `agent/managed_agents/entry_adapter.py` | **New** — Abstract base + registry |

### Interface design

```python
class EntryAdapter(Protocol):
    entrypoint: str  # "feishu", "discord", "web", "cli"

    def normalize_event(raw: dict) -> EntryEvent:
        """Convert adapter-specific inbound to EntryEvent."""

    def resolve_workspace(event: EntryEvent) -> Workspace | None:
        """Map external source to Workspace."""

    def resolve_session(event: EntryEvent, workspace: Workspace) -> Session | None:
        """Map external channel/thread to Session."""

    def health() -> dict:
        """Adapter health status."""

class EntryAdapterRegistry:
    adapters: dict[str, EntryAdapter]
    
    def register(adapter: EntryAdapter): ...
    def get(entrypoint: str) -> EntryAdapter | None: ...
    def ingest(raw: dict, entrypoint: str) -> EntryEvent: ...
```

### Compatibility strategy

- Adapters are registered at startup
- Missing adapter: event ingested as `entrypoint="unknown"`
- All adapters produce the same EntryEvent shape
- No adapter changes core execution behavior

### Tests required

- registry registration and lookup
- adapter health reporting
- fallback when adapter not found

### Risks

- **Low**: Protocol-based interface is non-invasive
- **Mitigation**: adapters are optional; core works without any adapters registered

### Non-goals

- No DiscordAdapter implementation
- No FeishuAdapter refactor
- No health dashboard UI

## Phase 4: Discord Adapter MVP

### Files to create

| File | Action |
|------|--------|
| `agent/managed_agents/discord_adapter.py` | **New** — DiscordAdapter, bot process, message handler |
| `configs/managed_agents/discord.yaml` | **New** — bot token, server ID, category/channel mappings |

### Implementation

```
DiscordAdapter (implements EntryAdapter):
  → On message_create:
      1. Resolve guild → deployment (always "local")
      2. Resolve category → workspace (create if not found)
      3. Resolve channel → session (create if not found)
      4. Thread → sub-session (for threaded discussions)
      5. Create EntryEvent
      6. Call EntryAdapterRegistry.ingest()
      7. Reply with resolved workspace/session name
  → On health check:
      Report bot process status, connection state
```

### Session binding mapping

```
Discord guild/server → Hermes deployment (always local)
Discord category → workspace
Discord channel → session
Discord thread → sub-session
Message "@Hermes" → EntryEvent with intent="mention"
```

### Tests required

- DiscordAdapter session resolution
- message normalization
- health endpoint
- category→workspace binding persistence

### Risks

- **Medium**: Local bot process lifecycle management
- **Mitigation**: run as subprocess managed by Hermes scheduler

### Non-goals

- No OAuth2 / webhook integration (Phase 4 uses bot token only)
- No global Discord fallback
- No cross-server support
- No voice channel support
- No slash commands

### Implementation

```
FeishuAdapter (implements EntryAdapter):
  → On card button or message:
      1. Extract thread_id → session
      2. Create EntryEvent
      3. Route to EntryAdapterRegistry.ingest()
```
