## Executive verdict

Do not ship this as production-ready Office automation yet.

The work is directionally good: it is Python-first, keeps Telegram sending dry-run, preserves no-npm constraints, has useful docs, and has a real test suite around redaction, scorecard validation, watchdog findings, and outbox idempotency.

But it does not clear a FAANG-style launch gate because the control plane still has reliability and truthfulness gaps:

1. The deployed cron evidence does not currently reproduce from this reviewer profile. `python3 scripts/office_watchdog_cron_install.py status` reports `runner_exists=false` and `jobs=[]`, while the parent devops handoff claimed a local-delivery cron job was installed. That may be profile-scoping, but the release claim is not globally verifiable as written.
2. Notification delivery is intentionally dry-run only. That is an acceptable MVP limitation, but it means any doc or handoff implying live Telegram state-change delivery is not launch-ready.
3. The watchdog is diagnostic-only. It detects stale claims, nonspawnable assignees, blocked-protocol issues, missing scorecards, and outbox backlog, but it does not implement the safe repair workflows described by the product direction.
4. Evidence gates validate artifact presence and schema, not artifact truth. The validator can reject empty heavy claims, but it cannot prove that a benchmark/deploy/GPU/release artifact is semantically valid.
5. Security posture is documented well, but broad default-profile access still relies on policy compliance and regex redaction. There is no enforceable permission boundary around protected artifacts, browser state, or secrets.

This should proceed as an internal alpha reliability kit with blockers below. It should not be represented as production-ready, fully deployed, or live-notifying.

## Present-day industry reality check

Sources checked during review:

- SLSA v1.1 specification: SLSA is an industry consensus spec for incrementally improving software supply-chain security using levels and provenance/attestation formats.
- CISA SBOM page: SBOM remains a key building block for software security and supply-chain risk management, with 2025 minimum-elements guidance highlighted by CISA.
- OpenTelemetry docs: current observability baseline is traces, metrics, and logs generated by instrumented systems, with backends intentionally separate.

Impact on verdict:

- A no-npm claim is not enough for supply-chain readiness. A production gate should also include dependency provenance or SBOM/lockfile review for the Python side and copied scripts.
- A Doctor/Watchdog CLI is not enough for operability. A production gate needs durable metrics, alert grouping/silencing, runbook-tested rollback, and state-machine invariants.
- Evidence scorecards are good, but modern release gates expect machine-verifiable provenance and artifact semantics, not just file existence.

## Blockers before signoff

### B1. Deployment evidence is not reproducible across profiles

Files/commands:

- `scripts/office_watchdog_cron_install.py:72-75`
- `docs/office-superpowers/DEPLOYMENT.md:32-42`
- Review command: `python3 scripts/office_watchdog_cron_install.py status`

Finding:

The devops handoff says a reversible local-delivery no-agent cron job is installed every 30m. From the reviewer profile, the status command reports no copied runner and no matching jobs. The likely root cause is profile-specific `HERMES_HOME`, but the docs present the installed state as if it is the local Office state, not a profile-scoped state.

Failure mode:

Akhil reads the runbook, assumes the watchdog is actually protecting the Office, and walks away. In reality, the scheduler for the active/default profile may not be running the job. That is a classic launch-review fail: the safety system is documented, but not actually attached to the production control plane being protected.

Required fix:

Add a deployment verification section that states which profile owns the cron job, which `HERMES_HOME` contains the copied runner, and which scheduler process will tick it. The install/status script should print the resolved `HERMES_HOME`, profile, job IDs, delivery target, next run, and scheduler/gateway status. The gate should fail unless those match the intended production Office profile.

### B2. Telegram delivery is not implemented, only queued/dry-run

Files:

- `hermes_cli/office_superpowers.py:605-609`
- `docs/office-superpowers/DEPLOYMENT.md:13,124-170`
- `docs/office-superpowers/TEMPLATE_LIBRARY.md:228`

Finding:

`send-due` and `retry-failed` return a dry-run-only message. This is honest in the implementation and several docs, but the overall product requirement included low-noise Telegram status updates. That requirement is not satisfied yet.

Failure mode:

State changes can complete without the operator ever receiving remote notification. Worse, if docs are skimmed, a downstream worker can treat queued intent as delivered message.

Required fix:

Keep the MVP label until a reviewed gateway sender is implemented. Add outbox record state transitions for `pending`, `sent`, `failed`, `last_error`, `attempts`, `next_attempt_at`, and `sent_at`. Require a live smoke artifact before any signoff says Telegram delivery works.

### B3. Watchdog safe repair language is ahead of implementation

Files:

- `hermes_cli/office_superpowers.py:412-497`
- `hermes_cli/office_superpowers.py:540-554`
- `docs/office-superpowers/SECURITY_MODEL.md:47-50`

Finding:

The watchdog emits `safe_auto_repair` hints but never mutates board state. `--repair-routine` only gates argument acceptance; it does not perform repairs. The security model says the default-profile operator may reclaim stale routine work, unblock protocol-only stale blocks, and reassign nonspawnable tasks under conditions. Those capabilities are not implemented in this code path.

Failure mode:

A stuck Office stays stuck. The watchdog pages, but the human still has to interpret and fix it. That is useful, but not the promised hands-free recovery.

Required fix:

Either downgrade docs to diagnostic-only for this release or implement a reviewed repair engine with idempotency, audit comments, PID/heartbeat checks, dry-run diffs, and tests for stale claim races.

### B4. Evidence gate can be gamed by artifact presence

Files:

- `hermes_cli/office_superpowers.py:142-189`
- `tests/hermes_cli/test_office_superpowers.py:50-74`

Finding:

The validator checks required fields and that heavy-claim gates have at least one existing artifact path. It does not validate artifact type, contents, timestamp, command provenance, exit code consistency, or whether the artifact actually supports the claim.

Failure mode:

A worker can attach a stale JSON file or irrelevant log and pass schema validation for a performance/deploy/GPU claim. That is evidence laundering.

Required fix:

Add artifact-class validators for benchmark, deploy/release, GPU/Colab, live-server, and security-scan claims. At minimum: require artifact type, generating command, generated_at, workspace-relative path, checksum, and claim-specific fields. Reviewer/QA should fail gates whose artifacts are semantically unrelated.

### B5. Security boundary is policy-only, not enforcement

Files/docs:

- `docs/office-superpowers/SECURITY_MODEL.md:37-68`
- `docs/office-superpowers/SECURITY_MODEL.md:87-205`
- `hermes_cli/office_superpowers.py:40-47`

Finding:

The docs correctly define protected assets and denied actions. The code only provides regex redaction and diagnostics. It does not enforce protected artifact writes, broad default-profile access, browser cookie access, memory writes, or approval policy changes.

Failure mode:

A future worker with broad profile access can still modify protected files or leak private browser/log content if it ignores policy. Regex redaction will miss some secret formats and cannot judge PII in screenshots or page text.

Required fix:

Before production signoff, add enforcement at the tool/dispatcher boundary or permission-policy layer for protected paths and secret-bearing artifacts. Treat the current docs as a threat model, not a control.

## Required fixes before launch

1. Make deployment status profile-aware and fail closed when the active Office profile has no installed/ticking watchdog job.
2. Implement or explicitly defer live Telegram sender semantics. If deferred, rename the feature to report-outbox dry-run and remove any launch language implying delivery.
3. Tighten the evidence validator from schema/artifact-exists to claim-specific artifact validation.
4. Add watchdog state-machine tests for stale-running claim race windows, blocked-task taxonomy, nonspawnable reassignment, and missing-scorecard follow-up routing.
5. Add security enforcement for protected artifacts or mark all broad-access security controls as policy-only residual risk.
6. Add operational metrics or durable telemetry for watchdog runs, outbox backlog, send failures, stale claims, and repair decisions. Current JSON CLI output is not enough for 3am operations.
7. Make browser/Chrome access claims testable. Docs should distinguish remote browser tool access, local dashboard HTTP access, and logged-in Chrome profile access. The last one remains a human-login/privacy blocker.
8. Add rollback verification that proves the cron job was paused/removed from the same profile/scheduler where it was installed.

## Panel review

### Priya Venkataraman, Principal Engineer, Google

Verdict: The abstraction is promising, but the control-plane invariant is not explicit enough to launch.

- `build_watchdog_report` reports findings, but the invariant is unclear: is the system supposed to diagnose, notify, repair, or all three? The docs span all three, while the implementation mostly diagnoses. Split the interface into `detect`, `plan_repair`, `apply_repair`, and `notify`, with explicit state transitions.
- The outbox idempotency key includes payload hash, task, run, board, and report type. Good. But there is no sender state machine. A durable outbox without a durable sender is a queue-shaped TODO.
- `validate_scorecard` conflates evidence presence with evidence validity. That is not a load-bearing abstraction for trust. The domain needs typed evidence contracts.
- Profile-scoped deployment is the kind of boundary that becomes a production outage. The docs must name the profile and scheduler being protected.

### Marcus Okafor, Engineering Director, Meta

Verdict: This is alpha-grade ops tooling, not launch-ready automation.

- The best thing here is that the watchdog returns nonzero with actionable findings and stays dry-run. That is safer than pretending to heal the board.
- The worst thing is the deployment claim. If the status command says no job, your runbook cannot say the job is installed. At Meta, that fails launch review because the mitigation itself is not deployed.
- There are no SLOs, no dashboard, no alert grouping/silencing beyond truncation, and no runbook-tested incident path. Current output is enough for a developer at a terminal, not for an operator away from the machine.
- Current board state has 21 watchdog findings. That proves the detector works, but it also means the Office is already noisy. Do not route that to Telegram until grouping and suppression are real.

### Ken Tanaka, Principal Security Engineer, AWS

Verdict: The threat model is useful, but the controls are mostly advisory.

- Regex redaction is a guardrail, not a security boundary. It will miss provider-specific tokens, PII in natural language, screenshots, and browser-visible private data.
- The no-npm constraint is preserved for this change, but supply-chain posture in 2026 needs provenance. SLSA and SBOM expectations mean you should record dependencies, script origins, and artifact checksums if this becomes an operator control plane.
- Broad default-profile access is dangerous unless enforced at the tool boundary. A policy doc saying not to edit protected artifacts does not stop a compromised or confused worker.
- The browser profile boundary is correctly documented. Keep it strict. Do not let future docs imply unrestricted logged-in Chrome control.

### Sam Delacroix, Staff Engineer, ex-Netflix / ex-Stripe

Verdict: The CLI ergonomics are close, but the operator story is not yet coherent.

- `office_doctor.py`, `office_watchdog.py`, and `office_report_outbox.py status` are the right primitive shape. Small commands, JSON output, no npm, easy to script.
- `office_watchdog_cron_install.py status` needs to print the resolved profile/home and explain why zero jobs is bad or expected. Right now a user has to know Hermes profile scoping to interpret the output.
- The docs are comprehensive, but they are too generous with future-tense capability. Put a current capability matrix at the top: implemented, dry-run only, documented only, out of scope.
- Error messages should distinguish expected findings from deployment failures. A watchdog exit 1 due to real findings is okay; a missing installed job is not.

## Top 3 fixes before anyone else sees this

1. Rewrite the top-level capability statement to avoid overclaiming: diagnostic/dry-run now, live Telegram and safe repair later.
2. Fix deployment evidence so `status` proves the active profile has the job or explicitly says no job is installed.
3. Add a short architecture section defining the core invariant: durable Kanban state is source of truth; reports are derived; delivery/repair must never claim success before committed evidence.

## Top 3 fixes before production

1. Implement profile-aware cron deployment verification and rollback verification.
2. Implement live outbox sender with durable send states, retries, idempotency, and smoke-test artifacts, or remove Telegram delivery from production scope.
3. Add enforcement, not just docs, around protected artifacts, secrets, browser profile boundaries, and approval policy changes.

## The thing this project is avoiding

It is avoiding the hard state-machine work. Detecting bad states is much easier than safely transitioning them. The product promise is "hands-free Office recovery," but the implementation is currently "tell me what is broken." That gap is manageable if named honestly. It is dangerous if hidden behind good docs.
