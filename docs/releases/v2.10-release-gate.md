## 3. Safety Checks

### 3.1 Feishu transport modification is minimal and guarded

The `feishu.py` change is exactly **9 lines** inserted at line 2564:

- Checks if incoming card action has `action == "select_session"`
- If so, imports and calls `handle_select_session_card_action()` from the bridge
- Returns immediately to avoid double-dispatch
- If the check fails or the bridge raises, falls through to existing behavior
- This is a **sidecar pattern**: the import is lazy, the function is wrapped in try/except in the bridge, and failures cannot block the main Feishu message flow

**Assessment: SAFE**

### 3.2 Ambiguity card behavior is opt-in and disabled by default

`FEISHU_AMBIGUITY_CARD_ENABLED` defaults to `"false"`:

```python
FEISHU_AMBIGUITY_CARD_ENABLED = os.getenv(
    "FEISHU_AMBIGUITY_CARD_ENABLED", "false"
).lower() in ("true", "1", "yes")
```

When disabled (current default):
- `feishu_session_resolver.py:159` skips ambiguity detection entirely
- `feishu_session_resolver.py:192` `check_ambiguity()` returns `None` immediately
- Resolution falls through to step 7: derive default group session
- **No interactive card is ever sent** in the default configuration

**Assessment: SAFE — opt-in only**

### 3.3 No silent global fallback for ambiguous Feishu messages

The resolution chain (`feishu_session_resolver.py`) follows this priority:

1. Explicit card session_id
2. Explicit alias (reserved)
3. P2P current-session binding
4. Thread/root-message binding
5. Group workspace binding
6. **Ambiguity card (behind feature flag)**
7. Default group session — **only after all above fail**

Key point: The ambiguity card (step 6) is only reached when the feature flag is enabled AND multiple active sessions are detected. Without the flag, the resolver proceeds to step 7. This does NOT produce an "I don't know which session" error — it produces a deterministic default group session. This is the designed behavior for Phase 1 compatibility.

**Assessment: SAFE — deterministic, no silent fallback to wrong session**

### 3.4 Permission gate deny behavior is safe

`feishu_permission_gate.py` produces structured `PermissionCheckResult` objects with `allowed=False` and a clear `reason` string. It does NOT:

- Raise exceptions that could crash the transport
- Auto-block agents
- Auto-demote agents
- Modify routing behavior
- Inject side effects

It is explicitly documented as "advisory input only" for `_check_sender_allowed()`:

```
This module is advisory input only. It does not modify routing or policy
execution. It provides check() results that the Feishu transport can use
to accept or reject messages.
```

**Assessment: SAFE — advisory-only, no side effects**

### 3.5 Web Console unknown/unavailable/disconnected states

`AdapterHealthCard.tsx`:
- Handles `disconnected` → destructive badge
- Handles `unknown` → secondary badge
- Handles `unregistered` → outline badge
- Handles `error` → destructive badge + error message
- Default fallback for any unrecognized status → secondary badge with raw status text
- When `mode == "cli_legacy"`, displays explicit message: "No external adapters registered. System is in CLI-only legacy mode."

`V210WorkspaceContext.tsx`:
- When no data loaded (empty workspaces, empty sessions, no health), returns null (component does not render)
- When `mode == "cli_legacy"`, displays "CLI Legacy" badge
- Never invents adapter health data

**Assessment: SAFE — explicit, no fabricated certainty**

- Feishu interactive card auto-enablement (behind feature flag, disabled by default)
- Streaming cards, approval buttons, notification cards
- Mac App shell (design only)
- LarkCliToolRuntime / LarkSkillLoader (not scoped, no code)
- Automatic demotion or agent switching (preserved invariant)
- Automatic routing weight mutation (preserved invariant)

- Cross-posting between Feishu and Discord
- Public exposure of local Hermes API
- Global fallback session for ambiguous messages
