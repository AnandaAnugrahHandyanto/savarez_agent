# QMT 盘中流判定说明

> 结论先行：当前 QMT 盘中流使用**候选池快照序列**做事件判定，已统一服务于 `qmt_intraday_refresh.py` 与 `qmt_intraday_state_matrix.py`。它不是 tick 级逐笔成交订阅，而是**快照驱动的离散事件层**。

## 1. Source of truth

代码以以下实现为准：

- `qmt_intraday_state_matrix.py`
  - `build_transition_window(...)`
  - `build_flow_summary(...)`
  - `is_sealed_limit(...)`
  - `is_blowup(...)`
  - `support_strengthened(...)`
  - `support_weakened(...)`
- `qmt_intraday_refresh.py`
  - 通过共享 `build_transition_window(...)` 输出 `## 零、最新窗口变化`

## 2. 当前覆盖的盘中流

当前统一判定 5 类事件：

1. 封单
2. 回封
3. 炸板
4. 承接转强
5. 承接转弱

## 3. 判定边界

### 3.1 封单

**定义**：前一快照未封死，当前快照首次进入封死涨停。

当前判定条件：

- 当前快照满足：
  - `pct >= 9.8`
  - `ask1_vol == 0`
- 且前一快照不满足上面条件
- 且历史上该票尚未被记为“已封过”

**解释**：
- 这是“首次封住”的语义
- 不是所有涨幅接近涨停都算封单，必须满足封死条件

### 3.2 回封

**定义**：此前已经封过，或前一窗口已经显式处于炸板/开板状态，当前再次封死涨停。

当前判定条件：

- 当前快照满足封死涨停：
  - `pct >= 9.8`
  - `ask1_vol == 0`
- 且前一快照未封死
- 且满足以下任一：
  - 该票历史上已经封过一次
  - 前一快照已被识别为炸板态

**解释**：
- 回封强调“重新封回去”
- 因此依赖历史 `seen_sealed` 状态，而不只是前后两帧裸比较

### 3.3 炸板

**定义**：已封住的涨停被打开，或者当前快照显式标注为炸板。

当前判定条件：

- `limit_up_type == "炸板"`
- 或前一快照封死、当前快照不再封死且 `pct < 9.8`

**解释**：
- 当前实现不把所有“高位未封死”都视作炸板
- 只有显式炸板标记，或“封死 → 打开”的转移，才算炸板

### 3.4 承接转强

**定义**：当前窗口盘口承接明显增强，但没有进入封单/炸板事件。

当前判定条件：

- 当前窗口不是封单/回封/炸板
- `bid_ask_ratio >= 1.8`
- 且满足以下任一：
  - 前一窗口 `bid_ask_ratio < 1.1`
  - 当前窗口相对前一窗口提升 `>= 1.0`

**解释**：
- 这是“盘口承接改善”的事件，不代表一定要买
- 若同窗口已经发生封单/炸板，则优先记更强的结构事件，不再重复记承接变化

### 3.5 承接转弱

**定义**：当前窗口盘口承接明显走弱，但没有进入封单/炸板事件。

当前判定条件：

- 当前窗口不是封单/回封/炸板
- 前一窗口 `bid_ask_ratio >= 1.8`
- 且满足以下任一：
  - 当前窗口 `bid_ask_ratio < 1.0`
  - 前后下降幅度 `>= 1.0`

**解释**：
- 这是“承接恶化”的事件
- 若该窗口已经被识别为炸板，则优先记炸板，不重复记承接转弱

## 4. 单窗口事件 vs 全日累计语义

### 4.1 单窗口事件

单窗口事件指的是：

- 只比较**上一快照 → 当前快照**
- 只反映“最新一跳”发生了什么
- 主要给 IM 首屏 / 刷新报告顶部使用

当前使用位置：

- `qmt_intraday_refresh.py`
  - `## 零、最新窗口变化`
- `qmt_intraday_state_matrix.py`
  - `## IM 极简摘要`
中的 `最新盘中流` / `最新焦点`

当前 `## IM 极简摘要` 首屏还应同时携带：
- `题材判定：...`
- `动作模板：<action_label>（标签=<固定原因标签>；原因=...；计划=...）`

这样 push 侧无需重写正文，也能从首屏直接提取 `theme_line / template_line / reason_tags`。

### 4.2 全日累计语义

全日累计语义指的是：

- 把同日所有快照窗口串起来
- 统计当天到目前为止累计发生过多少次：
  - 封单
  - 回封
  - 炸板
  - 承接转强
  - 承接转弱

当前用途：

- 主要用于 `build_flow_summary(...)` 的内部汇总
- 适合做盘后复盘或长报告统计
- **当前极简摘要默认不展示全日累计**，避免首屏过长

## 5. 当前实现边界

1. **不是 tick 级成交事件流**
   - 当前仍基于离散快照
   - 因此事件只在“相邻快照比较”上成立

2. **不是实时订阅器**
   - 依赖外部调度产生快照
   - 快照越密，事件识别越接近真实盘中节奏

3. **盘口承接使用 `bid_ask_ratio` 代理**
   - 这是当前最稳定、最便宜的 live 字段
   - 还没有接更高频的逐笔/分时成交结构

4. **封单/炸板优先级高于承接变化**
   - 同一个窗口内，如果已经判定为封单/回封/炸板
   - 就不再重复记承接转强/转弱

## 6. 对外阅读建议

- 看 **刷新报告顶部 `## 零、最新窗口变化`**：适合看“这一跳发生了什么”
- 看 **推送顶部 `## IM 极简摘要`**：适合 Feishu 首屏
- 看 **完整状态迁移报告**：适合盘中复盘与盘后归因

## 7. 结论

当前这套盘中流已经满足：

- 首屏短摘要可读
- refresh / push 双栈统一
- 事件边界一致
- 单窗口与全日累计语义分离

但它仍然是**快照驱动事件层**，不是逐笔行情层。后续若接更高频数据，应在不破坏当前摘要协议的前提下扩展判定器。
