## 1. The single most important paragraph in the harness

From `CLAUDE.md` lines 16–25:

> tanner-stack is a drop-in starter kit for AI-assisted software engineering. It is not an application. It ships four things and nothing else:
>
> | Asset | Where | What it is |
> |---|---|---|
> | **Sub-agents** | `.claude/agents/` | Named sub-agents (Architect, Chen, Code Reviewer, Grep Verifier) with auto-delegation via `description`-field triggers, declared modes, and staged approval gates. Reference-doc copies at `personas/`. |

The harness is **markdown plus YAML frontmatter plus a small `settings.json`**. There is no application code, no runtime, no server, no scheduler, no daemon. The "engine" is Claude Code's native delegation reading `description` fields.

## 2. Sub-agents — the only persistent operating identities

Sub-agents live at `.claude/agents/*.md`. There are **four** and only four. Each is a markdown body with YAML frontmatter that Claude Code reads at session start to decide auto-delegation. Reference-doc mirrors at `personas/<name>.md` exist for historical continuity; the sub-agent files at `.claude/agents/` are authoritative (`CLAUDE.md` line 66).

A sub-agent prompt should specify the natural-language signal that triggers the right sub-agent. Auto-dispatch is description-driven: each sub-agent's `description` field opens with `MUST BE USED` and lists `Use PROACTIVELY when user says ...` trigger phrases plus explicit `DO NOT USE for X — that is the Y sub-agent` carve-outs. Match your prompt's wording to those triggers and the right sub-agent fires without needing any explicit invocation syntax. (Quoted from `README.md` lines 18–20: "You speak to Claude Code naturally. ... Claude Code reads each sub-agent's and each skill's `description` frontmatter field at session start and routes your signal to the best match.")

### 2.1 architect (`.claude/agents/architect.md`)

| Field | Value |
|---|---|
| `name` | architect |
| `model` | opus |
| `tools` | Read, Write, Edit, Glob, Grep, Bash |
| `color` | blue |
| `memory` | project |
| `skills` (preloaded) | swarm |
| Expected posture | "high effort, long runs expected (design for up to ~60 turns). Staged work with operator gates between modes." (line 24) |

**Auto-dispatch trigger** (line 10, verbatim):

> MUST BE USED for system-level design, repo scaffolding, methodology extraction, staged audit/extract/verify workflows, AND all swarm orchestration decisions. Use PROACTIVELY when user says "architect", "scaffold", "extract", "audit a system", "parallel", "swarm", "multi-file refactor", or needs decisions about decomposing work across multiple independent execution units.

**Modes.** Multi-mode with halt-and-await-approval gates: DISCOVER → EXTRACT → VERIFY. Architect declares `[MODE: DISCOVER]` / `[MODE: EXTRACT]` / `[MODE: VERIFY]` at the start of every response and never mixes modes (`.claude/agents/architect.md` lines 27–35).

**Authority unique to architect.** Architect is the **sole sub-agent with swarm-invocation authority**. Other sub-agents request a swarm by routing through architect; they do not invoke `swarm` directly. The `swarm` skill is preloaded only in architect's frontmatter (line 16, plus the `Swarm Orchestration` section, lines 52–69).

**Halt rules** (lines 92–98): operator-decision flag on any genericity classification ambiguity, path-sanity halt on missing source paths, provenance discipline on every extracted file, no silent decisions on lost behavior, swarm-veto authority over sequential-deps-disguised-as-swarm requests.

**When to route to architect:** distill methodology from a codebase, scaffold a new repo from one, decompose multi-file/multi-subsystem work for parallel execution. Example trigger phrases verbatim (lines 84–90): "Extract the audit/fix/build methodology from `<path>` into a generic template." / "Pull Claude Code configs, skills, and personas from `<path>` and genericize them." / "I need to refactor these 40 files consistently — should I swarm this?" / "Run a parallel audit across these 12 subsystems."

**When NOT to route to architect:** post-diff PR review (route to code-reviewer), latent-defect audit on existing code (route to chen), one-shot textual claim verification (route to grep-verifier).

### 2.2 chen (`.claude/agents/chen.md`)

| Field | Value |
|---|---|
| `name` | chen |
| `model` | opus |
| `tools` | Read, Grep, Glob, Bash (read-only — no Write, no Edit) |
| `color` | orange |
| `memory` | project |
| `skills` (preloaded) | chen, audit-deep-subsystem, audit-finding-expansion, audit-spec-to-code-delta, audit-pre-launch-failure |
| Expected posture | "max effort, long runs (design for up to ~80 turns). Audits are thorough by design — do not shortcut to hit a turn budget." (line 30) |

**Auto-dispatch trigger** (line 10, verbatim):

> MUST BE USED for audits of existing code in the repo. Use PROACTIVELY when user says "audit", "is this safe to ship", "what could break", "pre-launch check", "spec-to-code delta", or wants latent-issue discovery in a subsystem, module, or file that already exists (no pending diff or PR). Chen operates in four focused audit modes — deep subsystem, finding expansion, spec-to-code delta, pre-launch failure. Audit-only; does not fix, does not invent findings, requires grep evidence for every claim. DO NOT USE for reviewing proposed changes, diffs, or PRs — that is the code-reviewer sub-agent. DO NOT USE for first-pass textual verification of a claim — that is the grep-verifier sub-agent; chen's finding-expansion (MODE 2) is for drilling into an existing chen finding, not one-shot pattern checks.

**Focus modes** (lines 32–51). Chen runs in exactly one of four modes per invocation. Mode is declared in the opening line of every response. Selection table verbatim from `chen.md` lines 44–49:

| User signal | Mode |
|---|---|
| "audit this subsystem", "full audit of X", "what's wrong with X" | DEEP SUBSYSTEM |
| "dig into this bug", "how far does this defect spread", "trace this finding" | FINDING EXPANSION |
| "does the code match the spec", "verify against architecture", "has it drifted" | SPEC-TO-CODE DELTA |
| "launch-ready?", "what would break in production", "pre-demo sanity" | PRE-LAUNCH FAILURE |

If the signal is ambiguous between two modes, Chen states both candidates and picks the tighter one with justification; if irreducibly ambiguous, halts and asks (line 51).

**Output contract** (lines 78–81). Chen produces a 12-section structured report (defined in the chen-audit superprompt — see §6 of this manual). Every finding labeled with severity (P0–P3), certainty (CONFIRMED / HYPOTHESIS / UNVERIFIED / DISPROVEN), blast radius, upstream cause, downstream consequence, recommended fix, live-verification bridge. Final verdict is one of: CLEAN, PROVISIONALLY CLEAN PENDING LIVE VERIFICATION, NO MATERIAL DEFECT FOUND IN REPO-ONLY REVIEW, NOT CLEAN, HIGH RISK / DO NOT SHIP.

**What chen will not do** (lines 67–76): soften findings, narratively merge contradictions, mark CLEAN unless every false-clean-prevention criterion is met, carry a prior Claude answer forward as proof, declare a pattern-level finding closed from a single instance, Write or Edit files, mix modes within a single audit run.

### 2.5 The personas/ directory — distinction

Files at `personas/<name>.md` exist for the same four sub-agents (`architect`, `chen`, `code-reviewer`, `grep-verifier`). Each carries an explicit reference-doc disclaimer. From `personas/chen.md` line 17 (verbatim):

> **REFERENCE DOC.** Operational form at [.claude/agents/chen.md](../.claude/agents/chen.md). Keep in sync on major edits; the sub-agent file is authoritative for Claude Code delegation.

A prompt should never cite `personas/` for behavior. Cite `.claude/agents/` for delegation behavior; cite `personas/` only when discussing extension/historical-continuity context per `docs/extending.md` lines 9–15.

There is **no separate "persona" concept beyond the sub-agents themselves** in this harness. The terminology "persona" appears in `docs/methodology.md` lines 50–62 as a general framing for "operating identities," but operationally a persona = a sub-agent.

## 3. Skills — the rigor and capability layer

Skills live at `skills/<name>/SKILL.md`. There are 28 directories total: 8 sub-agent-preloaded, 19 main-session auto-invoke, 1 scaffold (`_template`). Per `docs/build-notes/skill-audit-2026-04-21.md` lines 144–157.

Skills auto-invoke based on `description` matching, the same way sub-agents do. The harness's discipline — quoted from `docs/build-notes/skill-audit-2026-04-21.md` line 12 — is:

> every skill must be either (a) preloaded in a sub-agent's `skills:` frontmatter OR (b) have auto-invocation language (`Use PROACTIVELY` / `MUST BE USED` / `Use when ...`) in its description — no orphans.

### 3.1 Skill frontmatter vocabulary

Recurring frontmatter fields (observed across all `SKILL.md` files):

| Field | Meaning observed in the files |
|---|---|
| `name` | Lowercase hyphenated skill name, must match directory. |
| `description` | Auto-invocation trigger language. |
| `disable-model-invocation: true` | Model cannot auto-fire this skill from arbitrary prompts; operator (or sub-agent preload) must explicitly invoke. Present on most rigor and audit skills. |
| `user-invocable: true` | Operator can directly invoke. Observed on `100`, `pr`, `soft`, `zero`. |
| `context: fork` | Skill runs in a forked context window, isolating from main thread. Observed on most fork-style skills (`adverse`, `code`, `lossy`, `diagnose`, `chen`, `decide`, `grep-verify`, `audit-*`, `_template`, `st`). |
| `effort: low \| medium \| high \| max` | Effort hint. `max` is observed on `adverse`, `code`, `lossy`, `st`, `chen`, `swarm`, `audit-*`. |
| `argument-hint: "[...]"` | Hint string for the operator-supplied `$ARGUMENTS`. |
| `allowed-tools: ...` | Tool allowlist for the skill body. |

### 3.2 Rigor modes — the operator's primary discipline lever

Four rigor modes ship. Each stays active for the rest of the session once activated.

| Skill | Trigger phrases (from `description`) | Standard imposed |
|---|---|---|
| **`100`** (`skills/100/SKILL.md`) | "I need this right", "no margin for error", "prove it" | Strictest. file:line proof on every finding, proven root cause on every fix, full-file pattern census, deep wire trace, explicit CONFIRMED/HYPOTHESIS/UNVERIFIED labels. (Lines 24–59.) |
| **`zero`** (`skills/zero/SKILL.md`) | "no false positives", "don't miss any", "find all of them" | Zero False Positive AND Zero False Negative. Pattern census, no clean verdict without exhaustive search. Lighter than `/100` (no deep-wire-trace requirement), stricter than `/soft`. (Lines 19–63.) |
| **`soft`** (`skills/soft/SKILL.md`) | "no guessing", "hard evidence only", "prove it with the source" | Bans soft conclusions — no "likely", "probably", "should work", "might be" without exact file/line/runtime value/fix. (Lines 19–48.) |
| **`shutup`** (`skills/shutup/SKILL.md`) | "stop talking", "shut up", "quiet" | Silent mode. Claude stops generating tokens until explicitly told to resume. Context-preservation tool. (Lines 14–22.) |

The strictness ordering is documented in each skill's `description` field cross-references. From `skills/100/SKILL.md` line 10: "Strictest rigor tier — tighter than /zero (proof but not full census) and /soft (only bans soft conclusions)." From `skills/zero/SKILL.md` line 10: "Lighter than /100 (which also requires deep wire trace to execution proof) but stricter than /soft." From `skills/soft/SKILL.md` line 10: "Less strict than /100 (which also requires full-file pattern census and deep wire trace)."

A `/max` command at `.claude/commands/max.md` activates `/100 + /zero + /soft` simultaneously as a one-shot wrapper (lines 1–32 of `max.md`).

A prompt-writer declares rigor by either (a) using the natural-language trigger phrases above (auto-fire), (b) typing the slash form `/100`, `/zero`, `/soft`, `/shutup`, `/max` (operator override per `disable-model-invocation: true` semantics), or (c) writing the trigger phrases in the prompt body so the model self-elects (when `disable-model-invocation: true`, this requires the operator to explicitly invoke; the `100`/`soft`/`zero` skills carry both `user-invocable: true` and `disable-model-invocation: true`).

## 5. Workflows — multi-step processes

Workflows live at `workflows/*.md`. Four workflow files ship.

### 5.1 `workflows/audit-fix-build.md` — the default loop

The harness's default for any non-trivial change. Three phases with halt gates.

**Phase 1 — AUDIT** (find truth, propose nothing). Invocation: chen sub-agent (auto-delegates from "audit this subsystem" / "is this safe to ship") OR `diagnose` skill (auto-invokes from error / "why is this failing") OR code-reviewer sub-agent (auto-delegates from "review this diff"). Deliverable: findings report with severity + certainty + file:line evidence + pattern-vs-instance + blast radius + upstream/downstream + fix sketch (no implementation) + missing-evidence list. **Halt gate: operator reviews the report and approves specific findings for Phase 2. Never self-promote from AUDIT to FIX.** (Lines 22–48.)

**Phase 2 — FIX** (one finding at a time). One finding per commit unless a grouped change set. Invoke grep-verifier on each claim. Run typecheck after every edit (PostToolUse hook). For pattern-level findings, run full-file or full-codebase census before claiming complete. Discipline rules quoted (lines 62–68): surgical changes (Karpathy 8.3); no speculative fixes; restrictive fallbacks (LEARNINGS §5); live path only; write-side before read-side (LEARNINGS §4). (Lines 50–73.)

**Phase 3 — BUILD** (integrate, verify, ship). Invocation: `/verify <scope>` (3-leg) → `adverse` skill (auto-fires from "what am I missing") → optionally `st` skill → `pr` skill (preloaded in code-reviewer). Rules quoted (lines 91–97): execution proof not wiring proof; no "probably fixed"; session-end discipline. **Halt gate: operator owns PR merge. Never merge to main/master/LIVE/Production from a Claude Code session.** (Lines 75–98.)

**Loop semantics** (lines 102–107): out-of-scope discoveries → BACKLOG.md, never inline. Bug that invalidates a Phase 1 finding → return to Phase 1. Multi-file refactors sharing a contract → grouped change set → one Phase 2 commit, not N. If batching is happening, STOP.

**Anti-patterns named verbatim** (lines 111–117): "I already know the bug" (skip-Phase-1), pattern-level close-from-one-hit, writing code before success criteria, dead-code edits, "should work" / "looks correct" language.

### 5.2 `workflows/pr-review.md` — three-layer PR discipline

Three layers, none optional (lines 14–62). Layer 1: Author preflight via `pr` skill (branch safety / code-reviewer delegation / typecheck / test suite / explicit file staging / push + `gh pr create`). Layer 2: code-reviewer sub-agent structural sweep. Verdict CLEAN / WARNINGS / BLOCKED. Layer 3: human merge decision (operator — never the Claude Code session — makes the merge call).

Binding rules quoted (lines 91–97): no force-push; no direct commits to protected branches; explicit file staging (`git add .` banned); commit messages follow convention or PR is blocked; code-reviewer runs on every PR; WARNINGS do not silently become CLEAN.

### 5.3 `workflows/commit-conventions.md` — commit format

Conventional Commits–style: `<type>(<scope>): <short description>` with body and footer. Subject ≤ 72 chars. Types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `style`. Body required for every `fix` commit (lines 76–88). Decision-code reference convention. `BREAKING CHANGE:` footer for public-contract changes. Footer: `Co-Authored-By: Claude <noreply@anthropic.com>` per project attribution.

Binding rules (lines 172–180): every fix has a body; imperative mood; no WIP commits on protected branches; one logical change per commit; reference decision codes; never `git add .`; never amend a pushed commit.

### 5.4 `workflows/audit-report-template.md` — audit deliverable shape

Structural template for any formal audit. Sections: Executive Summary → Scope → Findings (Critical/High/Medium/Low) → False Positives Investigated → Coverage Summary (Security / Business Logic / Integration with %) → Open Questions → Recommended Actions → Audit Checklist. (Lines 1–123.)

## 9. Auto-dispatch — how routing actually works

Quoted from `CLAUDE.md` §3.5 line 74:

> Routing sub-agents and skills from natural-language task signals is handled primarily by Claude Code's native delegation: each sub-agent's `description` frontmatter field (in `.claude/agents/*.md`) and each main-session skill's `description` field (in `skills/*/SKILL.md`) describe when they fire, and Claude Code picks the best match. **Structured slash-command workflows in `.claude/commands/` always override auto-dispatch** when you want direct control.

The §3.5.1 routing table (`CLAUDE.md` lines 80–89) maps user signals to default dispatch. Quoted relevant rows:

| User signal | Default dispatch |
|---|---|
| "audit" / "review this" / "is this correct" / "find issues" | chen sub-agent. Pick audit submode. |
| "fix X" / "resolve Y" / "clean up Z" | `workflows/audit-fix-build.md` Phase 2 (FIX). If a prior audit report exists, resume at Phase 2; otherwise dispatch to chen first, then Phase 2. |
| "build X" / "add X" / "create a feature" | Phase 3 (BUILD) via main session; conclude by auto-invoking the `pr` skill. |
| "refactor" / "restructure" / "reorganize" | Full audit-fix-build loop starting with chen MODE 1 or MODE 2. Use `/blast` on all modified symbols. |
| "PR review" / "review this pull request" / "review this diff" | code-reviewer sub-agent + `workflows/pr-review.md`. |
| "verify X" / "confirm Y" / "check that Z" | grep-verifier (textual) or chen MODE 2 (semantic). |
| "explain this codebase" / "onboard me" / "how does X work" | architect DISCOVER mode. GitNexus skill auto-invokes if available. |
| "bulk task" / "do X across all files" / "parallel" / "swarm" | architect — architect decides swarm vs serial and invokes preloaded `swarm` skill when warranted. |

**Clarification gate** (CLAUDE.md §3.5.5, lines 136–145, paraphrase): the harness binds itself to halt and ask exactly one clarifying question when (a) the signal does not clearly match any row in §3.5.1, (b) two conflicting signals are present, (c) target scope is undefined, or (d) the desired output shape is ambiguous (findings report vs implemented fix vs both). ### 10.1 Direct evidence on parallel execution

**Architect owns swarm.** From `CLAUDE.md` lines 58–60 (verbatim):

**Triggers and prohibitions** verbatim from `CLAUDE.md` §3.5.4 lines 116–132:

> - No cross-file dependencies between the subtasks.
> - Audit work where findings are independent per subsystem.

> - Single-file work.

> 3. **Max 5 subagents per swarm** — context budget constraint.
> 4. **Each agent must complete independently** — no agent waits on another's output.

### 10.4 Verdict for an external prompt-writer

**Question: Can this harness safely run multiple agents in parallel overnight?**

**Answer: Partial.**

- **Single-session intra-process parallelism: Yes.** The architect sub-agent can decompose a parallelizable task and invoke the `swarm` skill, which spawns up to 5 subagents in their own context windows on non-overlapping files. The synthesis step is mandatory and runs in the main thread. Read-only subagents are the rule; the main thread does the editing. This is a documented and supported primitive.
- **Multi-process or cross-session parallelism: No.** From `docs/architecture.md` line 46 verbatim: "Full multi-process swarm coordination is not implemented." There is no documented primitive for spawning concurrent independent Claude Code sessions or for a coordinator daemon that runs across sessions.
- **Unattended overnight execution: Not documented as supported.** Every halt gate in every workflow waits for operator approval. The standing prohibition `CLAUDE.md` line 274 forbids inventing the missing pieces. Every fix workflow ends with "operator reviews," every PR workflow ends with "operator merges," every push ends with "operator initiates." The harness's single-most-cited binding rule (CLAUDE.md line 145) is `clarification cost < misroute cost` — implying preference for halting until operator answers, not for guessing through to completion.
- **Pattern the harness does document for "long-running unattended work": session-handoff via `session-end` skill** (`skills/session-end/SKILL.md`). The skill produces a structured handoff document that is "self-contained — a reader with no session context should understand what happened and what to do next" (line 113). The next session starts via `session-start` skill which re-establishes branch / commits / dirty files / risks. The cross-session continuity primitive is **the handoff document, not a daemon**.

A prompt-writer who wants overnight unattended work should either (a) confine it to a single architect-led swarm of ≤5 read-only investigators with a synthesizer waiting to greet the operator at AM-time, or (b) treat the work as a sequence of discrete sessions each ending with a `session-end` handoff and starting from a `session-start` briefing, with operator-in-the-loop between sessions. The harness as configured does not authorize agents to push to production, merge PRs, or self-promote across approval gates without the operator.

**Most important citation supporting this answer:** `CLAUDE.md` line 274:

> Do not invent architecture for Hermes / Mythos / Arsenal. If a task requires them, halt and ask the operator.

Combined with `docs/architecture.md` line 46:

> Full multi-process swarm coordination is not implemented.

And the consistent operator-gate language across every workflow.

## 12. LEARNINGS.md — anti-patterns, verbatim

`LEARNINGS.md` documents seven hard-won failure modes plus a meta-lesson. Every prompt-writer should respect every one of these. Quoted in full per the prompt's instructions:

### 1. The Dead Code Trap (lines 13–25)

> **What happened:** Multiple fix sessions applied correct changes to a dead file. Every fix was technically correct. Every fix had zero production impact because the file was no longer on the live execution path.
>
> **The lesson:** ALWAYS grep-verify the live execution path before editing any file. Trace the call chain forward from the known entry point — don't assume a file is active because it exists and compiles.
>
> **Prevention:** Before editing any file, run: `grep -r "import.*from.*{filename}" {source_dir}/` to confirm something actually imports it. If zero results and it's not a known entry point, it's dead.

### 2. False Positive Audit Findings (38–54% Rate) (lines 28–41)

> **What happened:** AI-generated audit reports consistently flagged bugs that didn't exist in the actual code. Across multiple audit sessions, 38–54% of reported findings were false positives — phantom bugs created by the auditor misreading code, missing context, or hallucinating function behavior.
>
> **The lesson:** Never accept an audit finding at face value. Every finding must be grep-verified against the actual source before it's acted on. The forcing function: "Are you 100% certain? Show me the grep evidence."
>
> **Common false positive patterns:**
> - "Missing null check" — value is guaranteed by a prior step or database constraint
> - "Unused variable" — used via destructuring, spread, or dynamic access
> - "Hardcoded value" — actually imported from config; the import just isn't on the same screen
> - "Function never called" — called via event routing or dynamic dispatch
> - "Dead code path" — actually the active path (the reverse of Lesson #1)

### 3. CRLF Line Terminators — Silent Edit Failures (lines 44–50)

> **What happened:** Claude Code's Edit tool silently failed when files had CRLF line terminators. The edit appeared to succeed but the file was unchanged. No error message. No warning. Nothing happened.
>
> **The fix:** `git config --global core.autocrlf input`. This is permanent. Never revert to `true`.
>
> **Detection:** If a Claude Code edit seems to have no effect, check line endings: `file <filename>` or `cat -A <filename> | head -5` (look for `^M` at line ends). Fix with `dos2unix <filename>`.

### 4. Write-Side vs Read-Side Fix Ordering (lines 53–61)

> **What happened:** Read-side fixes were technically correct but complete no-ops because the field being read was never written upstream. The read-side consumers would correctly use the field — but only if the write-side actually populated it first.
>
> **The fix:** Find and fix all write sites first. Verify the data is written. Then verify the read side uses it correctly.
>
> **The lesson:** ALWAYS trace BOTH directions before declaring a fix complete. Ask: "Is this value actually being written before it's read?" A read-side fix only works if the write-side populates the data. Trace the full pipeline: origin → write → read → consumer.

### 5. Undefined Fallbacks Must Be Restrictive (lines 64–71)

> **What happened:** A fallback value for an undefined state was set to the permissive option. TypeScript guaranteed the value was always present at runtime, but the fallback existed as a safety net. When the safety net triggered unexpectedly, it let bad data through instead of blocking it.
>
> **The fix:** Every fallback and default in a data pipeline must be the most restrictive option. Unknown or missing states should produce the safest outcome, not the happiest path.
>
> **The lesson:** Every fallback/default should answer: "If this value is unexpectedly missing, what's the safest thing to do?" The answer is always the restrictive option.

### 6. Pattern Census Before Declaring Complete (lines 74–81)

> **What happened:** One instance of a pattern was found and fixed. The fix was correct. But 10+ sibling instances of the same pattern existed elsewhere in the same file and weren't checked. The bug continued via the unfixed instances.
>
> **The fix:** Run a full-file (or full-codebase) pattern census before declaring any pattern-level bug fixed. One correct instance does NOT mean all sibling instances are correct.
>
> **The lesson:** For any fix that addresses a pattern (not just a single specific line), run: `grep -n '{pattern}' {file}` to find every instance. Fix all of them. Report the total count.

### 7. Silent Data Truncation at Stage Boundaries (lines 84–98)

> **What happened:** A JSON repair / parser library silently truncated oversized output from an upstream stage before the downstream stage consumed it. No error was thrown. No warning logged. The downstream stage received partial data, accepted it as valid, and proceeded to produce incomplete results. The truncation was only discovered weeks later when output quality metrics diverged from expectations.
>
> **The fix:** Every cross-stage handoff that uses serialization, repair, or transformation must (a) log when a size limit is reached, (b) fail loud rather than pass partial data downstream, and (c) include a length/completeness check on the receiving side that rejects obviously-truncated payloads.
>
> **The lesson:** Silent data loss is the worst kind of bug — it produces plausible-looking output that is quietly wrong. Any step that can truncate, repair, coerce, or re-serialize data across a boundary is a ticking time bomb unless you explicitly verify completeness. Invoke the `lossy` skill before trusting any cross-boundary transformation.

### Meta-Lesson: Schema Before Code (lines 102–105)

> Database migrations must deploy before code changes. The deployment sequence is non-negotiable: **schema → code → verify**. If your code change depends on a new column, table, or constraint, the migration must land first. The operator owns migration deployment.

## 13. Karpathy's four LLM-coding pitfalls — the prompt-writer's mental checklist

`CLAUDE.md` §8 lines 280–304 names the four LLM-coding failure modes the harness's discipline layer was built to counter (synthesized from forrestchang/andrej-karpathy-skills, credit to Karpathy and Forrest Chang per line 302).

1. **Think Before Coding** — state assumptions explicitly; if uncertain, ask. Enforced by `/100`, `/soft`, `/zero`, every persona's mode-declaration rule.
2. **Simplicity First** — minimal code; no speculative features. Enforced by chen's "no rewrites when precise fixes are available" directive and code-reviewer's scope-drift check.
3. **Surgical Changes** — touch only what you must; match surrounding style; do not "improve" unrelated areas; do not refactor as a side effect. Enforced by `/adverse` integration-wiring check and code-reviewer's blast-radius analysis.
4. **Goal-Driven Execution** — transform request into verifiable success criteria before implementing; pair each plan step with a concrete check. Enforced by `/diagnose` Phase 3, `/verify` three-leg loop, chen's EXECUTION PROOF RULE.

`CLAUDE.md` line 304: "Karpathy's principles are the *why*; §3.5 is the *how*."

## 14. The well-formed prompt — anatomy

A prompt that exercises this harness at full capacity follows a recognizable shape. Most of the structure is implicit when natural-language signals match the auto-dispatch table (§9), but explicit prompts for high-stakes work should declare the following:

1. **Mission statement.** One paragraph. What is the work, why does it matter, what shape of output do you want.
2. **Sub-agent routing cue.** Match wording to the `description` triggers (§2). E.g., to invoke chen MODE 1: "Audit `<subsystem>` thoroughly — I want the whole failure surface mapped." To invoke architect for parallel work: "I need to refactor these 40 files consistently — should I swarm this?"
3. **Rigor mode declaration.** One of `/100` (strictest), `/zero` (FP+FN), `/soft` (no hand-waving), `/max` (combined), or simply use the trigger phrases ("I need this right", "no margin for error", "no false positives", "no guessing", "hard evidence only").
4. **Scope boundaries.** Explicit. What files? What subsystem? What's in scope, what's out. The harness will halt and ask if undefined (§3.5.5 clarification gate).
5. **Halt gates / preflight conditions.** State the gates explicitly. Examples: "Halt after Phase 1 audit — do not implement until I approve specific findings." "Halt before push — let me review the commit." "Stop after writing the failing test."
6. **Deliverable path and format.** Where should output land? In a report file? In `BACKLOG.md`? As inline code? What format — markdown headers, structured table, prose summary? The harness has multiple structured-output skills (`adverse`, `diagnose`, `chen`, `verify`, etc.) — pick or reference the format you want.
7. **Out-of-scope and NOT-to-do list.** Explicit. What the agent must not touch (e.g., dead-code paths, production branches, untested fields). The PreToolUse hook in `settings.json` blocks edits to paths matching `DEAD_FILE_PATTERN_*` placeholders, but project-specific dead paths must be enumerated.
8. **References to LEARNINGS.md anti-patterns.** If the work is in territory the lessons cover (dead code, write-side ordering, restrictive fallbacks, pattern census, silent data truncation), name the lesson explicitly so the agent attends to it.
9. **GitNexus availability declaration** if relevant. The four `gitnexus-*` skills are gated on availability (`docs/build-notes/adversarial-cleanup-2026-04-22.md` lines 49–50). State whether GitNexus is indexed; if it isn't, the harness falls back to grep.
10. **Session-handoff plan** for long work. State that the work should `/end` (`session-end` skill) at a clean stopping point with a structured handoff doc, so the next session can `/start` (`session-start` skill) and resume.

## 15. Cross-session handoff — the only documented "long work" primitive

The session-end / session-start pair is the harness's documented mechanism for cross-session continuity.

**`session-end` skill** (`skills/session-end/SKILL.md`) generates a structured handoff document. Sections (lines 75–106): SESSION HANDOFF header (date / branch / commit count), Changes Made (per-file: path / summary / decision ID / risk LOW/MEDIUM/HIGH), Decisions / Work Items Touched (table), Remaining Work (P0 merge blockers / P1 should-fix / P2 deferrable), Blockers, Next Step (single concrete action), Session Stats. Hard rule (line 113): "The handoff must be self-contained — a reader with no session context should understand what happened and what to do next."

**`session-start` skill** (`skills/session-start/SKILL.md`) loads pre-state (branch / dev-server status / recent commits / working tree / changed files) and presents a briefing. Hard rule on production-branch detection (lines 45–49): "If the current branch is `main`, `master`, or contains `LIVE` or `Production` → report **🚨 DANGER: You are on a protected branch. Do NOT edit files. Switch to a feature branch immediately.**"

A long unattended sequence is, per harness convention, structured as: session-start briefing → bounded work → session-end handoff → operator review → next session-start. Not as: continuous unattended overnight execution.

## 16. The three session-level modes — `AGENTS.md`

Per `AGENTS.md` lines 9–22 verbatim:

> ## Mode Selection
>
> Sessions operate in one of three modes. Declare your mode at the start of a session.
>
> | Mode | Trigger | Behavior |
> |------|---------|----------|
> | **IMPLEMENT** | Default — active coding task | High-effort editing, verification after every change, GitNexus impact check before every symbol edit |
> | **AUDIT** | auto-delegation to chen sub-agent, `adverse` skill auto-invocation, or explicit operator request | Read-only investigation. No edits. Produce findings with evidence labels. Await operator approval before implementing. |
> | **DISCOVERY** | New codebase, onboarding | Inventory mode. Read, map, document. No edits. Produce a structured report. |
>
> **Mode-switching rules:**
> - Switching from AUDIT → IMPLEMENT requires explicit operator approval ("proceed", "implement", "go ahead").
> - DISCOVERY produces a written report before any mode switch.
> - Never self-promote to IMPLEMENT from AUDIT without approval.

**Inviolable session rules** verbatim from `AGENTS.md` lines 96–103:

> 1. **Branch check first** — `git branch --show-current` before any edit.
> 2. **Grep before trust** — never accept an audit finding without grep evidence.
> 3. **Dead code check** — verify the live path before editing any file.
> 4. **Impact before edit** — GitNexus impact analysis before modifying any symbol.
> 5. **Operator owns production** — never push to production branches without explicit operator instruction.
> 6. **Scope discipline** — out-of-scope findings go to BACKLOG.md, not inline fixes.

A prompt-writer should declare the session mode either explicitly ("Operating in AUDIT mode...") or by the choice of routing signal (e.g., "audit" routes to chen → AUDIT; "implement X" stays in IMPLEMENT; "explain this codebase" routes to architect DISCOVER → DISCOVERY).

## 17. The GitNexus inviolable mandate — cited in the project template, not the harness itself

The harness's project template (`docs/project-claude-md-template.md` lines 109–188) ships a GitNexus integration block intended to be copied into adopting-project CLAUDE.md files. The block enumerates "Always Do" / "When Debugging" / "When Refactoring" / "Never Do" rules. Quoted "Never Do" rules (lines 140–144 of the template):

> - NEVER edit a function, class, or method without first running `gitnexus_impact` on it.
> - NEVER ignore HIGH or CRITICAL risk warnings from impact analysis.
> - NEVER rename symbols with find-and-replace — use `gitnexus_rename` which understands the call graph.
> - NEVER commit changes without running `gitnexus_detect_changes()` to check affected scope.

These rules are inviolable **for projects that adopt the template**. The harness itself does not enforce them — they apply to downstream projects that vendor tanner-stack and configure GitNexus. The `code-reviewer` sub-agent's `GitNexus availability contract` (line 65 of `code-reviewer.md`) explicitly handles the case where GitNexus is unavailable: continue with grep-only checks and note in the verdict.

## 18. Finish-line checklist — what every well-formed tanner-stack prompt must hit

Use this checklist to verify a prompt before sending it to the harness. Each item ties back to a specific harness primitive cited in this manual.

- [ ] **Sub-agent routing matches `description` triggers.** (`§2`) Use the trigger phrases verbatim from the relevant `.claude/agents/<name>.md` file. Auto-dispatch is description-driven.
- [ ] **Rigor mode declared.** (`§3.2`) Either invoke a slash form (`/100`, `/zero`, `/soft`, `/max`, `/shutup`) or use the natural-language trigger phrases ("I need this right" / "no false positives" / "no guessing" / "stop talking").
- [ ] **Scope boundary explicit.** (`§9 / §3.5.5 clarification gate`) Name the file, subsystem, or PR. The harness halts and asks if undefined.
- [ ] **Audit mode named when invoking chen.** (`§2.2`) Pick from DEEP SUBSYSTEM / FINDING EXPANSION / SPEC-TO-CODE DELTA / PRE-LAUNCH FAILURE. Match wording to the chen.md mode-selection table.
- [ ] **Halt gates and preflight conditions specified.** (`§5.1 / §16`) State where the agent should stop and wait for operator approval. Audit-fix-build's halt gates between Phases 1 / 2 / 3 are the canonical pattern.
- [ ] **Deliverable path and format specified.** (`§14 item 6`) Report file path, BACKLOG entry, structured table, etc. Reference `audit-report-template.md` if the deliverable is an audit.
- [ ] **NOT-to-do list and out-of-scope items enumerated.** (`§14 item 7 / §16 rule 6`) Out-of-scope findings → BACKLOG.md, never inline.
- [ ] **LEARNINGS.md anti-patterns respected.** (`§12`) Name the lessons that apply (dead code, write-side ordering, restrictive fallbacks, pattern census, silent truncation).
- [ ] **No invocation of unimplemented primitives.** (`§10.2 / §11`) Mythos / Hermes / Arsenal are documented as future hooks. Do not assume their existence. CLAUDE.md line 274: "If a task requires them, halt and ask the operator."
- [ ] **Parallel work routed through architect, with the swarm rules respected.** (`§10.1`) ≤5 read-only subagents, no two agents on the same file, mandatory main-thread synthesis. Do not ask sub-agents other than architect to spawn parallel work.
- [ ] **Branch safety acknowledged.** (`§16 rule 1 / settings.json deny list`) Declare that destructive actions on `main` / `master` / `LIVE` / `Production` branches are operator-only.
- [ ] **Session-handoff plan if work spans multiple sessions.** (`§15`) End with `/end` (`session-end` skill); next session begins with `/start` (`session-start` skill). Do not rely on cross-session memory persistence — it is not implemented.
- [ ] **`gh` / `git push` is operator-initiated, not agent-initiated** — unless explicitly authorized for the specific task. (`workflows/pr-review.md` lines 91–97; `skills/pr/SKILL.md` line 78; `skills/finish/SKILL.md` line 102.)
- [ ] **GitNexus availability declared** if any structural-query work is requested. (`§3.8 / §17`) Otherwise the agent will run in degraded grep-only mode and the prompt-writer should know that.

## 19. Closing notes on what the manual is and is not

This manual documents the harness as it ships at v0.3.1. It does not document Claude Code's underlying delegation engine (which is upstream of tanner-stack), it does not document how Claude Code matches paths to rule files (which is also upstream), and it does not document any Anthropic-internal capabilities that may exist outside of what tanner-stack configures.

If a future prompt-writer wants a primitive this manual does not list, the answer is **not to invent it**. Per the harness's own standing prohibition (CLAUDE.md line 274): halt and ask the operator. Per its own clarification-cost discipline (CLAUDE.md line 145): clarification cost < misroute cost.

Two observations marked explicitly as **inferences** rather than as harness facts:

- **Inference:** The harness's documented dispatch model is one-level (main → sub-agent), with one specific fan-out exception (architect → swarm subagents up to 5). The harness does not document deeper nesting; the most defensible reading is that deeper nesting is not authorized.
- **Inference:** "Long-running unattended overnight work" as a single continuous session is not a configuration the harness was built for. The session-end / session-start pair is the documented continuity primitive. A prompt-writer who interprets this otherwise should expect halt-and-ask behavior at every gate.

Beyond these two inferences, every claim in this manual ties to a specific file and (where useful) a specific line in the harness. Absences — "Boris Cherny", "anthropic cheat code", "focus feature", any cron / scheduler primitive — are stated plainly.
