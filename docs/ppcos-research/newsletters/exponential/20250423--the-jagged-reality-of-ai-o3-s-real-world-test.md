o3 is somewhat of an enigma among the best AI models. When it launched last week, it caused a stir because using it not only delivered better output but it felt different. Economist Tyler Cowen  called o3 “AGI [ https://substack.com/redirect/443e1f0b-0b11-4182-a29d-000880e1d2b5?j=eyJ1IjoicXd0ciJ9.0tEpqnOrcfuKYRYnEOurmlOZ_pVIUnS06konjQA-bCs ].” Ethan Mollick  wrote that the model resembles a “jagged AGI” [ https://substack.com/redirect/13bd2061-9983-4c10-b6bd-99cde33bd9bc?j=eyJ1IjoicXd0ciJ9.0tEpqnOrcfuKYRYnEOurmlOZ_pVIUnS06konjQA-bCs ] – super‑human in some areas, surprisingly brittle in others. The pushback arrived just as fast: Bojan Tunguz ’s satirical strawberry tweet  [ https://substack.com/redirect/b7f6b502-fd2c-40b8-b6e7-4d69e4b9503d?j=eyJ1IjoicXd0ciJ9.0tEpqnOrcfuKYRYnEOurmlOZ_pVIUnS06konjQA-bCs ]and Colin Fraser’s algebra critique [ https://substack.com/redirect/71b374ce-97f4-4981-8018-8e2449f646a8?j=eyJ1IjoicXd0ciJ9.0tEpqnOrcfuKYRYnEOurmlOZ_pVIUnS06konjQA-bCs ] capture the gap between benchmarks and basic slips.
Benchmarks poured in. A VCT benchmark that measures the capability to troubleshoot complex virology laboratory protocols [ https://substack.com/redirect/cdf10101-a661-4ccc-880e-0f804a8025f9?j=eyJ1IjoicXd0ciJ9.0tEpqnOrcfuKYRYnEOurmlOZ_pVIUnS06konjQA-bCs ] now has o3 in dominance with 43.8% accuracy vs. 22.1% that human PhD-level virologists scored. On ARC‑AGI v1 [ https://substack.com/redirect/c2f11c8c-81b2-4cb4-b3ca-db325e9ace62?j=eyJ1IjoicXd0ciJ9.0tEpqnOrcfuKYRYnEOurmlOZ_pVIUnS06konjQA-bCs ], it solves more than half the secret set at a twentieth of the cost of the next best chain‑of‑thought model. Yet on the tougher ARC‑AGI v2 [ https://substack.com/redirect/c2f11c8c-81b2-4cb4-b3ca-db325e9ace62?j=eyJ1IjoicXd0ciJ9.0tEpqnOrcfuKYRYnEOurmlOZ_pVIUnS06konjQA-bCs ] it barely clears 3%. The signal is getting harder to hear.
The more we measure AI, the less we seem to know. To cut through the noise, we stepped away from dashboards and put o3 – alongside Gemini 2.5 Pro, Claude 3.7 Sonnet and GPT-4o – through eight messy, real-world tests that reflect some of our day-to-day tasks:
Auditing codebases and building new features
Choosing the perfect laptop case
Crafting tricky international travel plans
Summarising dense essays
Reading scientific papers accurately
Executing research workflows
Translating chaotic brainstorming sessions into clear next steps
Ruthless fact-checking
Here’s what we learned from using the leading models in parallel.
Choosing the perfect laptop case
Nathan Warren  began this test with a straightforward, consumer-facing prompt:
I need a laptop case for a MacBook Pro 17". Please find me the best product that fits my characteristics.
This vague prompt was his way of testing how well each model could scope a messy real-world query, ask the right clarifying questions and return a concise, high-confidence recommendation.
He scored each mode on three Cs: clarity, curation and confidence – awarding up to two points a piece. o3 began by firing five rapid‑fire questions about style, protection level, preferred materials, bonus features, budget and then delivered a shortlist of just two SKUs that met every constraint. The answer arrived in under three‑and‑a‑half minutes and included a neat comparison grid plus a one‑sentence “if/then” recommendation.
Gemini took the opposite approach. It was quicker, but asked no clarifiers, it just provided a selection of ten bags to choose from. Gemini felt like a search engine, o3 more of a personal concierge.
Auditing codebases and building new features
For the codebase audit test, Nathan chose to examine a large RAG-style project we use to create a searchable archive of our newsletter. The repository holds more than 2400 files and the only practical way to parse it is using OpenAI’s Codex or Anthropic’s Claude Code which lets the AI walk the file system, run shell commands, and read source in bulk. But because we don’t have API access to o3, we had to settle for o4-mini. Practically, this is what most people would use for coding anyway – it is much cheaper while being only slightly behind o3 on software engineering benchmarks.
Nathan graded performance on diagnosis accuracy, implementation advice, and whether the proposed patch ran on the first try – two points each, for a maximum of six. o4-mini received maximum marks. It loaded the full directory tree, crawled 2 400 files easily and behaved like a seasoned mechanic. It listed modules, in‑lined code and flagged exact inconsistencies. On the other hand, Claude Sonnet 3.7 offered a fine but higher‑level review, more driving‑impressions than under‑the‑hood specifics. Nathan gave it a 4/6.
The second part of Nathan’s coding test was to get the models to build a new feature. He handed both contenders the same brief: “After retrieval, run an LLM ‘aggregate & summarise’ step so you return a concise answer rather than raw chunks.” Claude 3.7 Sonnet treated it like a pull‑request ticket. In a single burst it created a tiny helper that grabs the top search chunks, feeds them to an OpenAI call and prints the resulting blurb. The patch slotted in cleanly: no hard‑coding of keys, clear logging and a new --raw flag for anyone who still wants the old behaviour. Nathan hit enter, the code compiled and the CLI spat out a neat three‑sentence answer – job done in under four minutes, API cost ¢68.
o4‑mini followed the same brief but took a slightly bumpier road. It altered fewer lines, ran faster and even wrote an “agent‑readable” guide inside the comments, yet it called an outdated OpenAI function. Fixing that import took thirty seconds, after which the feature worked exactly as advertised. Total bill: $1.
Both models could wire up a real feature in a single conversation, but Claude was slightly more robust. o4‑mini remains the fastest diagnostic mechanic, while Claude is still the safer pair of hands when the code needs to ship today.
Plan tricky international travel
The British Airways website is one of my most frequently visited sites. My trips often involve tight turnarounds and a dash to another commitment. So far, the large language models have been utterly useless. They haven’t browsed the web; they can’t plan through the combinatorial complexity of trip routings and they lack the nuance and understanding I expect.
One example, I generally prefer the earlier flight to the later flight, except when coming back from San Francisco as the early BA flight is a janky A380 but the later one gets a more comfortable 777.
My current travel test to the LLMs is to respond accurately to this made-up challenge:
I am in Tianjin on 26 June finishing a meeting by 11.45am. I must be in Watford, UK for 11am on Friday 27 June. What travel options do I have? Business class preferred.
This isn’t straightforward. It is a multimodal query, involving combinations of trains, taxis, planes and stop-overs in potentially many countries. It’s reasonably easy for me to solve: leave Beijing on a flight that gets into London by 9pm UK time; or take an overnight routing via one of the likely candidate cities (Helsinki, Istanbul, Dubai, Doha, Abu Dhabi, Paris or Frankfurt) and make sure you can get a flight first thing in the morning to London.
Most AI systems fail even if they try hard. Gemini 2.5 with Deep Research was like a breathless intern. It went off and boiled the ocean, eagerly offering me a 5500-word report which was completely wrong. So, I wanted to know, how does o3 compare?
It was remarkable, as you’ll see further below.
o3 understood the possibilities and responded in a structured format: to arrive in London in the evening of the 26th and stay overnight. Or take an overnight flight (via Hong Kong or Istanbul) to arrive in London in the morning, with time to travel to Watford, a town outside London. It even considered flying private.
o3 didn’t do this perfectly.
Its preferred routing via Helsinki would get me to Heathrow at 9am the following day, too tight given the vagaries of the airport and traffic. It missed two other things. One was a routing via Paris which would get me into London by 7.30am, plenty of time to get to Watford. But more critically, it missed an Air China flight departing Beijing at 16.15 to London. This flight lands in London in the late evening, giving time for a night's rest and a relaxed journey the following morning.
When I asked why it had missed that particular routing, it came up with a plausible cockamamie excuse: that convenient flight was often over capacity and, in any case, OAG had filled the schedule late. Neither of those were true. It also claimed the aircraft was an A350-900 (which it isn’t).
So apart from the lying, o3 is quite impressive.
I also tried this search in Perplexity. It doesn’t do a bad job of outlining the overall trip, but it got excited about a non-existent BA flight. Like o3 it also missed the Air China 1615 flight. But far worse than o3, it completely ignored the need to arrive at the airport before the flight leaves!
Planning and executing a research plan
Chantal Smith  evaluated how three leading AI models – Claude 3.7 Sonnet, Gemini 2.5 and ChatGPT o3 – approach constructing and implementing research plans. Each AI was given identical instructions: create a detailed research plan to prepare for a live recording with economist Tyler Cowen and then execute that plan. Throughout the process, she minimized intervention, simply instructing AIs to “keep going” when they paused for direction. When completed, Chantal asked each AI to produce a final research document synthesizing their work. This experiment wasn’t focused on testing their deep research capabilities, but rather on comparing their approaches to planning, implementation and final deliverable creation.
ChatGPT o3 demonstrated superior methodology rigor with highly specific planning. It created a precise timeline and included specific automation instructions like “Schedule daily feed digests (05:30 UTC)” and “python → batch‑download URLs, extract full text, pipe through nltk summariser.” This level of precision and tool specification was unmatched by the other models.
Claude’s approach was more exploratory, with significantly less specific methodology. It used phrases like “I’ll begin executing the research plan” without detailing exact processes or tools. While the final research was excellent, its methodological approach lacked transparency.
Gemini took a middle approach with a clear 5-phase plan but no technical transparency. It mixed speed and substance; it was quick to emphasise an important clash: Tyler’s focus on bottlenecks versus Exponential View’s faith in exponential tech – “How does Tyler Cowen square the two?”. It rounded things out with  decent audience hooks.
Claude excelled at connecting complex ideas and showed a deeper understanding of Tyler’s intellectual evolution. It zeroed in on interview craft. Sections like “Creating Memorable Moments” and “Follow-up Techniques” showed some savvy about keeping the dialogue sharp and memorable.
When o3 regrouped, its final packet looked broadcast-ready: a minute-by-minute “conversation spine,” promo copy for every platform, a run-of-show checklist and spotless tables citing Cowen’s work with correct dates.
Overall, we’re moving forward with o3 and Gemini 2.5 Pro for this kind of work.
Summarising dense essays
For article summarisation, I have my own panel test which I’ve used to compare recent models. In this case, I take a couple of long-ish articles from two Substackers Jon Russell  and Michael Spencer , both quite different in tone and test the LLMs with my refined prompt.
The summariser is designed to give me a clean executive summary, identify the key arguments, spot standout data points, quotes and give me a long-ish summary. I’ll typically use it for longer pieces of work to help me better understand what I’m reading.
I grade each LLM on four criteria:
Comprehensiveness & salience: Captures the most critical points and main thesis.
Clarity & conciseness: Provides an easy-to-understand, well-structured and suitably brief summary.
Executive summary: How good was the executive summary and capturing the argument.
Insight & synthesis: Demonstrates deeper understanding by synthesizing information, capturing nuance and highlighting implications.
I barely now check for accuracy and fidelity as recent models rarely fail on this particular task in my experience.
The stand-out winner was Google’s Gemini 2.5 Pro. But the results are not as obvious as that, OpenAI’s o1 and o1 Pro produce by far the best executive summaries, really distilling the underlying arguments into a short statement. By contrast, o3’s executive summary felt a bit high school.
Reading scientific papers and graphs accurately
To test the latest multimodal models, Hannah Petrovic  asked OpenAI o3, Google’s Gemini 2.5 Pro and Anthropic’s Claude 3.7 Sonnet Ext Thinking to analyse her own paper published last year [ https://substack.com/redirect/06111687-9325-4e77-a493-f71de0d97485?j=eyJ1IjoicXd0ciJ9.0tEpqnOrcfuKYRYnEOurmlOZ_pVIUnS06konjQA-bCs ]. The prompt included the name of the paper and the first author. She tasked each model to (1) locate the paper, (2) read the text and figures and (3) summarise the main findings and make a note of the key point in each figure.
While Gemini and o3 found the paper online without help, Claude needed the PDF uploaded manually. Otherwise, the task was identical for all three systems.
o3 struggled most with figure interpretation despite leading the MMMU benchmark for visual reasoning. It produced an accurate commentary on the first figure, but from the third plot onwards its descriptions began to drift: it inferred trends that were mentioned in captions yet never appeared in the images themselves. In other words, the textual discussion anchored its answers more than the visuals did.
Outside the figure-by-figure breakdown, o3’s narrative summary of the paper was solid, but as a “graph reader” it underperformed.
Gemini delivered the strongest overall result. Every figure – including those tucked away in the appendix – was recognised and paired with a concise, correct takeaway. The model even went a step further, constructing a table that mapped the results space to predicted outcomes where they weren’t explicitly mentioned in the text, a move Hannah found to be both creative and genuinely useful. The only drawback was verbosity: the report was long enough to need trimming for most practical uses.
Claude landed midway between the two. It identified all non-appendix figures and linked each cluster of plots to the overarching claims they supported. What it did not do was spell out the unique insight of every single figure individually; instead, it grouped related charts under a shared takeaway. The result was more accurate than o3’s but slightly less granular than Gemini’s.

Translating chaotic brainstorming sessions into clear next steps
In any fast‑moving team, sticky notes are great for sparking ideas but they’re terrible for recall a week later. You may snap a picture of the board or take sticky notes home with you with the plan to turn them into notes, but then something else comes in the way and you postpone the task. In the past, we tried to use Claude and earlier OpenAI models to turn a photo of sticky notes into a write-up that the team can access quickly after the meeting to keep the momentum going. But it was never reliable enough to turn it into a practice. How about o3?
We uploaded three hard-to-read images of a messy board and post-its with the following prompt:
You are an AI assistant analyzing a photo of a messy whiteboard from a "Start / Stop / Continue" team retrospective. Extract all legible text from sticky notes in the image, regardless of color or position. Organize the extracted content under three clear headings: **Start**, **Stop**, and **Continue**. Then: 1. Summarize the key themes or recurring ideas in each category. 2. Propose action-oriented next steps based on these themes, phrased as specific, achievable tasks. The goal is to convert an unstructured brainstorming session into a clear summary and a set of team-aligned recommendations. Tone: concise, practical, and focused on moving things forward. Prioritize ideas over wording. Input: image of the whiteboard with sticky notes Output: summary + next steps
It took 4 minutes 30 seconds to make sense of the board and organize ideas.
The whiteboard was messy even for a human to interpret – handwriting of five different people, arrows everywhere, half the text lost in the glare. o3 began by stripping away everything that was not a sticky note. It masked the neon pink and yellow rectangles, ignored doodles and arrows and treated the whiteboard background as irrelevant noise. That pre-processing step gave the optical-character-recognition engine a clean target: handwriting only.
The output was the most accurate and reliable summary we’ve ever seen from an AI on this task – it could read the hand-writing, correctly classify notes into the right category (even where humans may find it tricky to distinguish the content).
After translating the notes into text, o3 went on to organise the ideas into common themes – something a skilled moderator would do extremely well during a brainstorming session; then it outlined the next steps. Next time, for an even more powerful post-brainstorm review, we’ll give it more context and possibly the transcript of the meeting (if all participants agree to it).
We compared o3 to our other go-to models, Claude Sonnet 3.7, 4o and Gemini 2.5 Pro. Claude stumbled the most. It runs a generic vision pass without pre‑processing, so glare and low‑ink strokes send swathes of text to the void. Because it doesn’t filter doodles, arrows and marker borders slip into the transcript, which then arrives as one long, unstructured paragraph; it made mistakes – didn’t categorise the right tasks in the right place nor transcribe them correctly. 4o made fewer mistakes and did a decent job categorizing tasks but it was less accurate than 3o – it misinterpreted some notes and thus categorize reliably. And Gemini 2.5 Pro got it wrong. It misinterpreted the writing so the verbatim list was false. It assembled similar words together trying to use this grouping to make sense of the content but this only led to inaccurate interpretation of the text. Gemini 2.5 Pro is a no-go – at least with a low-quality photo and no additional context.
Fact-checking
For the fact-checking task, we ran the most recent Sunday newsletter through three AI models with the following prompt:
You are a meticulous fact-checking assistant. Review the following curated newsletter for factual accuracy. Your job is to identify any inaccurate or unsupported claims, statistics, names, dates, and figures. For each issue you flag, provide a corrected version **with a citation from a reputable source** (e.g., peer-reviewed journals, mainstream news outlets, government or institutional data). If a claim cannot be verified or is misleading, explain why and suggest rewording or removal. Newsletter content: [PASTE]
o3 gave the newsletter a forensic once‑over, scanning every sentence for claims that could be anchored to hard evidence – revenues, yields, chip specs, inequality ratios, you name it – then dived into primary data and mainstream reportage to see which figures still stood up in April 2025. Where the numbers lined up, it left them; where they wobbled, it traced the latest filings, academic papers or Bloomberg terminals to steady them. Anything o3 couldn’t source at all went onto an “unverifiable” list for the chop or a fresh citation.
GPT-4o cited older studies and occasionally slipped between US and global baselines, surfacing only seven problems. Claude, enhanced with Perplexity MCP, performed better, but still missed several subtleties and flagged nine items. Gemini’s review wandered off-brief, offering edits where none were needed and overlooking genuine errors.
By comparison, o3 enforced a consistent standard: current data, precise scope and transparent sources. Its “character” also felt different – it doesn’t crumble under the user’s feedback, it stands its ground and explains why it did what it did. And it apologises much less than some other very apologetic AIs.
The scores are in
LLMs indeed are jagged. o3 is impressive in careful planning, rigorous fact-checking and real-world adaptability, yet oddly fragile with some visual nuances and logistical finesse. Benchmarks won’t fully capture this tension; daily human interactions will. Ironically, o3’s imperfections might be exactly what we need – a constant reminder that, as we integrate AI deeper into our workflows, embracing its flaws could be the key to genuinely productive collaboration. Perhaps the true breakthrough of advanced AI won’t be flawless performance, but rather learning how to skillfully manage imperfection.
