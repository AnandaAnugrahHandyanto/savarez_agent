# M5 Automated Judgment Criteria / M5 自动化裁决判定标准

> 定位：这是最小版 M5 Review Orchestrator 的客观判据。
> 它只负责输出裁决草案，不移除人工审批。

## 1. 固定复审顺序
M5 必须固定按以下顺序执行：
1. M3 Diff 审计
2. 调用链差异提取
3. pytest 执行与 exit code 采集
4. 实现报告读取

**任何实现若重排这四步，判定为不合格。**

---

## 2. 裁决类型

### 2.1 真赢候选（`APPROVE_CANDIDATE`）
必须同时满足：
- M3 无 `REJECT_HARD`
- 调用链差异证明目标统一或目标修复成立
- pytest exit code = 0
- 报告未声明越界，且与前 3 步证据一致

### 2.2 假赢（`FAKE_WIN`）
出现以下任一：
- pytest 通过，但调用链仍保留双实现核心链
- pytest 通过，但报告与 diff 证据不一致
- help / 文案看似对齐，但控制流未真实变化

### 2.3 越界（`OUT_OF_SCOPE`）
出现以下任一：
- 改了合同范围外文件
- 触碰 shell / PATH / wrapper 等非目标高危链路
- 顺手实现非本模块功能
- 业务代码被修改，但当前任务明令禁止

### 2.4 硬拒绝（`REJECT_HARD`）
出现以下任一：
- M3 命中 TTY 降级硬规则
- M3 命中新增高危删除/写入路径且与目标无关
- M3 命中 shell / PATH 触碰
- 代码结构表现出明显契约级越界

---

## 3. 优先级规则
- `REJECT_HARD` 最高优先级，不能被后续步骤覆盖
- `OUT_OF_SCOPE` 高于测试结果
- 测试通过不能消除越界
- 报告解释不能覆盖前 3 步机器证据

---

## 4. 调用链判定最低要求
M5 至少要能回答：
- 目标路径是否复用同一 helper
- 是否仍保留双实现核心链
- 是否只是“名字统一”，实际上逻辑仍分叉

如果回答不了以上问题，不得输出 `APPROVE_CANDIDATE`。

---

## 5. pytest 判定最低要求
- 必须真实执行指定测试命令
- 必须采集 exit code
- 必须把失败测试与裁决关联
- 不能只打印 stdout 后继续判绿

---

## 6. 报告读取原则
报告只能在前 3 步完成后读取。
报告的作用：
- 补充上下文
- 确认故意未处理的技术债
- 解释最小必要联动修正

报告不能：
- 推翻 `REJECT_HARD`
- 推翻 `OUT_OF_SCOPE`
- 作为唯一放行依据

---

## 7. 当前主控标准
M5 要想被视为可用，至少必须做到：
- 自动调用 M3
- 自动执行 pytest
- 自动提取最小调用链差异
- 自动输出结构化裁决草案
- 保留人工审批保险丝
