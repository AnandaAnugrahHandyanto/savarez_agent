## Task Contract（任务合同）

**Objective (目标)**  
实现最小版 `M5 Review Orchestrator`，把当前人工主控的首轮复审顺序程序化：自动调用 M3 审计、自动执行 pytest、自动提取调用链差异，并汇总为“真赢 / 假赢 / 越界”的客观裁决草案。

**Scope / Watchouts (范围 / 警戒线)**  
IN: 只实现最小版 M5 复审编排能力；固定复审顺序为“Diff 审计 -> 调用链差异 -> 测试结果 -> 报告”；允许调用现有 M3；允许运行指定 pytest 子集；允许做最小调用链差异提取；允许输出结构化裁决结果。  
OUT: 不顺手接管 Task Contract 生成；不实现完整调度器；不实现 Hook；不移除人工审批点；不直接修改 TDB-3 业务代码；不改文档体系。  
WATCHOUTS: 最大风险是把 M5 做成“自动放行器”，而不是“自动复审器”；第二风险是错误地让测试结果覆盖 `REJECT_HARD`；第三风险是调用链差异提取流于文本比对，无法识别是否仍保留双实现核心链；第四风险是把报告读得过早，让实现体的话术带节奏。

**Inputs (输入)**  
当前已存在的 `hermes_cli/safe_refactor_audit.py`；`docs/exec-plans/in-progress/m3-audit-rules-checklist.md`；AR-1 战时文档；当前成功的人类复审顺序：`Diff -> 调用链 -> 测试 -> 报告`；目标试点任务：`TDB-3`。

**Deliverables / Evidence (交付物 / 证据)**  
交付物：M5 最小可运行代码；最小测试；结构化裁决输出；简短报告。  
证据：M5 能自动调用 M3；能读取 pytest exit code；能提取最小调用链差异；能输出 `APPROVE_CANDIDATE / FAKE_WIN / OUT_OF_SCOPE / REJECT_HARD` 之类裁决草案；`REJECT_HARD` 不会被后续步骤覆盖；人工审批点仍保留。

**Done (完成标准)**  
以下条件必须同时成立：

1. **顺序被写死**  
   - 复审顺序固定为：M3 -> 调用链 -> 测试 -> 报告  
   - 不允许跳步或重排

2. **M3 结果真正进入裁决**  
   - `REJECT_HARD` 会直接阻断后续自动放行  
   - 测试通过不能覆盖 `REJECT_HARD`

3. **pytest exit code 真正进入裁决**  
   - 不是只打印结果，而是能改变裁决状态

4. **调用链差异可用**  
   - 至少能判断是否复用同一 helper  
   - 至少能判断是否仍保留双实现核心链

5. **未越界**  
   - 不实现完整调度器  
   - 不移除人工审批点  
   - 不修改业务代码  
   - 不改文档文件

6. **可审计**  
   - 报告能说明：调用了哪些审计、跑了哪些测试、调用链比较得出什么结论、最终裁决为何成立
