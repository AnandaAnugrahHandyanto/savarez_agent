# DevOps portfolio CV positioning pattern

Use when creating a CV/resume from a user's live DevOps portfolio projects.

## Evidence-first source material
Pull claims from verified artifacts where possible:
- Live URLs and health endpoints
- GitHub repos and CI runs
- Portfolio project cards
- README/runbook architecture
- AWS/DNS/runtime facts already verified in the session

Do not invent certifications, employment titles, production scale, or Kubernetes/EKS experience.

## Strong junior DevOps positioning
Good target title examples:
- Junior Cloud / DevOps Engineer
- Freelance IT & Web Operator
- Junior Linux / Cloud Support Engineer

Strong project bullet shape:
- **Action:** Deployed / built / maintained / automated / troubleshot
- **Stack:** AWS EC2, Docker Compose, Caddy HTTPS, PostgreSQL, GitHub Actions, DNS
- **Operational proof:** health checks, backup/restore scripts, runbook, CI smoke checks, security group cleanup
- **Evidence:** live site + GitHub repo

Example bullet:
> Deployed a Node.js/PostgreSQL online shop on AWS EC2 using Docker Compose, Caddy automatic HTTPS, Hostinger DNS, Elastic IP, health endpoints, production dependency audit, backup/restore scripts, and GitHub Actions CI.

## Outputs
For this user/task class, create:
- `*.pdf` — readable human-facing CV
- `*.png` — shareable visual preview
- `*.docx` — simple ATS/editable version
- `*.txt` — ATS keyword/plain-text version

## Visual layout notes
- One-page is fine even with some whitespace at bottom; avoid cramming.
- Header + clear section dividers + compact bullets worked well.
- Verify with vision for clipping/overlap/readability.

## Pitfalls
- Do not call ongoing studies a completed degree/certification.
- Do not let a pretty visual CV be the only output; ATS TXT/DOCX matters for applications.
- Do not include every debugging detail; CVs need final operational impact and tools.
