# Внутрішня робота циклу агента

Основний движок оркестрації – це клас `AIAgent` у `run_agent.py` — великий файл (~4 400 рядків), який обробляє все: від складання підказки до диспетчеризації інструментів та відмовостійкості провайдера.

## Основні обов’язки

`AIAgent` відповідає за:

- Складання ефективної системної підказки та схем інструментів за допомогою `prompt_builder.py`
- Вибір правильного провайдера/режиму API (`chat_completions`, `codex_responses`, `anthropic_messages`)
- Виконання переривних викликів моделі з підтримкою скасування
- Виконання викликів інструментів (послідовно або паралельно через пул потоків)
- Підтримку історії розмов у форматі повідомлень OpenAI
- Обробку стиснення, повторних спроб і переключення на запасний (фолбек) режим
- Відстеження бюджету ітерацій між батьківськими та дочірніми агентами
- Очищення постійної пам’яті перед втратою контексту

## Два точки входу

```python
# Simple interface — returns final response string
response = agent.chat("Fix the bug in main.py")

# Full interface — returns dict with messages, metadata, usage stats
result = agent.run_conversation(
    user_message="Fix the bug in main.py",
    system_message=None,           # auto-built if omitted
    conversation_history=None,      # auto-loaded from session if omitted
    task_id="task_abc123"
)
```

`chat()` — тонка обгортка навколо `run_conversation()`, яка витягує поле `final_response` із словника результату.

## Режими API

Hermes підтримує три режими виконання API, які визначаються вибором провайдера, явними аргументами та евристикою базового URL:

| API mode | Використовується для | Тип клієнта |
|----------|----------------------|-------------|
| `chat_completions` | Сумісні з OpenAI кінцеві точки (OpenRouter, кастомні, більшість провайдерів) | `openai.OpenAI` |
| `codex_responses` | OpenAI Codex / Responses API | `openai.OpenAI` з форматом Responses |
| `anthropic_messages` | Рідний Anthropic Messages API | `anthropic.Anthropic` через адаптер |

Режим визначає, як формуються повідомлення, як структуруються виклики інструментів, як парсяться відповіді та як працює кешування/стрімінг. Усі три збігаються в один внутрішній формат повідомлень (словники OpenAI‑стилю `role`/`content`/`tool_calls`) до і після викликів API.

**Порядок визначення режиму:**
1. Явний аргумент конструктора `api_mode` (найвища пріоритетність)
2. Специфічне визначення провайдера (наприклад, провайдер `anthropic` → `anthropic_messages`)
3. Евристика базового URL (наприклад, `api.anthropic.com` → `anthropic_messages`)
4. За замовчуванням: `chat_completions`

## Життєвий цикл туру

Кожна ітерація циклу агента виконується за такою послідовністю:

```text
run_conversation()
  1. Generate task_id if not provided
  2. Append user message to conversation history
  3. Build or reuse cached system prompt (prompt_builder.py)
  4. Check if preflight compression is needed (>50% context)
  5. Build API messages from conversation history
     - chat_completions: OpenAI format as-is
     - codex_responses: convert to Responses API input items
     - anthropic_messages: convert via anthropic_adapter.py
  6. Inject ephemeral prompt layers (budget warnings, context pressure)
  7. Apply prompt caching markers if on Anthropic
  8. Make interruptible API call (_interruptible_api_call)
  9. Parse response:
     - If tool_calls: execute them, append results, loop back to step 5
     - If text response: persist session, flush memory if needed, return
```

### Формат повідомлення

Усі повідомлення використовують сумісний з OpenAI формат внутрішньо:

```python
{"role": "system", "content": "..."}
{"role": "user", "content": "..."}
{"role": "assistant", "content": "...", "tool_calls": [...]}
{"role": "tool", "tool_call_id": "...", "content": "..."}
```

Контент роздумів (від моделей, що підтримують розширене мислення) зберігається у `assistant_msg["reasoning"]` і за потреби виводиться через `reasoning_callback`.

### Правила чергування повідомлень

Цикл агента суворо дотримується чергування ролей повідомлень:

- Після системного повідомлення: `User → Assistant → User → Assistant → …`
- Під час виклику інструменту: `Assistant (з tool_calls) → Tool → Tool → … → Assistant`
- **Ніколи** не два повідомлення Assistant підряд
- **Ніколи** не два повідомлення User підряд
- **Лише** роль `tool` може мати послідовні записи (паралельні результати інструментів)

Провайдери валідовують ці послідовності і відхиляють некоректну історію.

## Переривні виклики API

Запити API обгорнуті у `_interruptible_api_call()`, який виконує реальний HTTP‑запит у фоновому потоці, одночасно стежачи за подією переривання:

```text
┌────────────────────────────────────────────────────┐
│  Main thread                  API thread           │
│                                                    │
│   wait on:                     HTTP POST           │
│    - response ready     ───▶   to provider         │
│    - interrupt event                               │
│    - timeout                                       │
└────────────────────────────────────────────────────┘
```

При перериванні (користувач надсилає нове повідомлення, команда `/stop` або сигнал):
- Потік API відкидається (відповідь відкидається)
- Агент може обробити новий ввід або чисто завершитися
- Жодна часткова відповідь не вставляється в історію розмови

## Виконання інструментів

### Послідовно vs Паралельно

Коли модель повертає виклики інструментів:

- **Один виклик** → виконується безпосередньо у головному потоці
- **Кілька викликів** → виконуються паралельно через `ThreadPoolExecutor`
  - Виняток: інструменти, позначені як інтерактивні (наприклад, `clarify`), примушують послідовне виконання
  - Результати вставляються у вихідний порядок викликів, незалежно від порядку завершення

### Потік виконання

```text
for each tool_call in response.tool_calls:
    1. Resolve handler from tools/registry.py
    2. Fire pre_tool_call plugin hook
    3. Check if dangerous command (tools/approval.py)
       - If dangerous: invoke approval_callback, wait for user
    4. Execute handler with args + task_id
    5. Fire post_tool_call plugin hook
    6. Append {"role": "tool", "content": result} to history
```

### Інструменти рівня агента

Деякі інструменти перехоплюються `run_agent.py` *до* досягнення `handle_function_call()`:

| Інструмент | Чому перехоплюється |
|------------|----------------------|
| `todo` | Читає/записує локальний стан завдань агента |
| `memory` | Записує у файли постійної пам’яті з обмеженням кількості символів |
| `session_search` | Запитує історію сесії через базу даних сесій агента |
| `delegate_task` | Створює підагент(и) з ізольованим контекстом |

Ці інструменти змінюють стан агента безпосередньо і повертають синтетичні результати без проходження через реєстр.

## Точки входу для колбеків

`AIAgent` підтримує платформо‑специфічні колбеки, які забезпечують прогрес у реальному часі в CLI, шлюзі та інтеграціях ACP:

| Колбек | Коли викликається | Використовують |
|--------|-------------------|-----------------|
| `tool_progress_callback` | До/після кожного виконання інструменту | CLI‑спіннер, повідомлення про прогрес шлюзу |
| `thinking_callback` | Коли модель починає/закінчує мислити | CLI‑індикатор “thinking…” |
| `reasoning_callback` | Коли модель повертає контент роздумів | Відображення роздумів у CLI, блоки роздумів шлюзу |
| `clarify_callback` | Коли викликається інструмент `clarify` | Запит вводу у CLI, інтерактивне повідомлення шлюзу |
| `step_callback` | Після кожного завершеного туру агента | Відстеження кроків шлюзу, прогрес ACP |
| `stream_delta_callback` | Кожен токен стріму (за увімкнення) | Відображення стріму у CLI |
| `tool_gen_callback` | Коли виклик інструменту парситься зі стріму | Попередній перегляд інструменту у спіннері CLI |
| `status_callback` | Зміни стану (thinking, executing тощо) | Оновлення статусу в ACP |

## Бюджет та поведінка запасного (fallback) режиму

### Бюджет ітерацій

Агент відстежує ітерації через `IterationBudget`:

- За замовчуванням: 90 ітерацій (можна налаштувати через `agent.max_turns`)
- Кожен агент має власний бюджет. Підагенти отримують незалежні бюджети, обмежені `delegation.max_iterations` (за замовчуванням — 50) — загальна кількість ітерацій батька + підагентів може перевищувати ліміт батька
- При досягненні 100 % агент зупиняється і повертає підсумок виконаної роботи

### Запасна модель

Коли основна модель падає (429 — обмеження швидкості, 5xx — серверна помилка, 401/403 — помилка автентифікації):

1. Перевіряється список `fallback_providers` у конфігурації
2. Кожен запасний провайдер пробується по порядку
3. При успіху розмова продовжується новим провайдером
4. При 401/403 спроба оновити облікові дані перед переключенням

Система запасних провайдерів також охоплює допоміжні завдання окремо — vision, стиснення та веб‑видобуток мають власні ланцюжки fallback, які налаштовуються у розділі `auxiliary.*` конфігурації.

## Стиснення та постійність

### Коли спрацьовує стиснення

- **Попередньо** (до виклику API): якщо розмова перевищує 50 % вікна контексту моделі
- **Автоматичне стиснення шлюзу**: якщо розмова перевищує 85 % (агресивніше, виконується між турами)

### Що відбувається під час стиснення

1. Пам’ять спочатку вивантажується на диск (запобігає втраті даних)
2. Середні тури розмови підсумовуються у компактний резюме
3. Останні N повідомлень залишаються незмінними (`compression.protect_last_n`, за замовчуванням — 20)
4. Пари повідомлень інструмент‑результат зберігаються разом (не розділяються)
5. Генерується новий ідентифікатор лінійки сесії (стиснення створює “дочірню” сесію)

### Постійність сесії

Після кожного туру:
- Повідомлення зберігаються у сховищі сесій (SQLite через `hermes_state.py`)
- Зміни пам’яті вивантажуються у `MEMORY.md` / `USER.md`
- Сесію можна відновити пізніше за допомогою `/resume` або `hermes chat --resume`

## Ключові файли джерела

| Файл | Призначення |
|------|-------------|
| `run_agent.py` | Клас AIAgent — повний цикл агента |
| `agent/prompt_builder.py` | Складання системної підказки з пам’яті, навичок, контекстних файлів, персональності |
| `agent/context_engine.py` | Абстрактний ContextEngine — підключуване управління контекстом |
| `agent/context_compressor.py` | Типовий движок — алгоритм втратоїсного підсумовування |
| `agent/prompt_caching.py` | Маркери кешування підказок Anthropic та метрики кешу |
| `agent/auxiliary_client.py` | Додатковий LLM‑клієнт для побічних завдань (vision, підсумовування) |
| `model_tools.py` | Колекція схем інструментів, диспетчер `handle_function_call()` |
