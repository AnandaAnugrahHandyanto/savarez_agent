# Внутрішня структура шлюзу

Шлюз обміну повідомленнями — це тривалий процес, який з’єднує Hermes з понад 20 зовнішніми платформами обміну повідомленнями через уніфіковану архітектуру.
## Ключові файли

| Файл | Призначення |
|------|-------------|
| `gateway/run.py` | `GatewayRunner` — головний цикл, slash‑команди, розподіл повідомлень (великий файл; перевір git для актуальної кількості рядків) |
| `gateway/session.py` | `SessionStore` — збереження розмов і формування ключа сесії |
| `gateway/delivery.py` | Доставка вихідних повідомлень до цільових платформ/каналів |
| `gateway/pairing.py` | Процес парування DM для авторизації користувача |
| `gateway/channel_directory.py` | Відображення ідентифікаторів чатів у зрозумілі назви для cron‑доставки |
| `gateway/hooks.py` | Виявлення хуків, їх завантаження та розсилання подій життєвого циклу |
| `gateway/mirror.py` | Дзеркальне копіювання повідомлень між сесіями для `send_message` |
| `gateway/status.py` | Керування блокуванням токенів для екземплярів шлюзу, прив’язаних до профілю |
| `gateway/builtin_hooks/` | Точка розширення для завжди‑зареєстрованих хуків (не постачається) |
| `gateway/platforms/` | Адаптери платформ (по одному на кожну платформу обміну повідомленнями) |

## Потік повідомлень

Коли повідомлення надходить з будь‑якої платформи:

1. **Platform adapter** отримує сиру подію, нормалізує її у `MessageEvent`
2. **Base adapter** перевіряє активний захист сесії:
   - Якщо агент запущений для цієї сесії → ставить повідомлення в чергу, встановлює подію переривання
   - Якщо `/approve`, `/deny`, `/stop` → обходить захист (виконується inline)
3. **GatewayRunner._handle_message()** отримує подію:
   - Визначає ключ сесії за допомогою `_session_key_for_source()` (формат: `agent:main:{platform}:{chat_type}:{chat_id}`)
   - Перевіряє авторизацію (див. розділ Authorization нижче)
   - Перевіряє, чи це slash‑команда → передає до обробника команд
   - Перевіряє, чи агент вже запущений → перехоплює команди типу `/stop`, `/status`
   - Інакше → створює екземпляр `AIAgent` і запускає розмову
4. **Відповідь** надсилається назад через **platform adapter**

### Формат ключа сесії

Ключі сесії кодують повний контекст маршрутизації:

```
agent:main:{platform}:{chat_type}:{chat_id}
```

Наприклад: `agent:main:telegram:private:123456789`

Платформи, що підтримують теми (Telegram forum topics, Discord threads, Slack threads), можуть включати ідентифікатори тем у частину `chat_id`. **Ніколи не створюй ключі сесії вручну** — завжди використовуйте `build_session_key()` з `gateway/session.py`.

### Двохрівневий захист повідомлень

Коли агент активно працює, вхідні повідомлення проходять через два послідовних захисти:

1. **Рівень 1 — Base adapter** (`gateway/platforms/base.py`): Перевіряє `_active_sessions`. Якщо сесія активна, ставить повідомлення в `_pending_messages` і встановлює подію переривання. Це ловить повідомлення *до* того, як вони потраплять до **gateway runner**.

2. **Рівень 2 — Gateway runner** (`gateway/run.py`): Перевіряє `_running_agents`. Перехоплює специфічні команди (`/stop`, `/new`, `/queue`, `/status`, `/approve`, `/deny`) і маршрутизує їх відповідно. Все інше викликає `running_agent.interrupt()`.

Команди, які мають дістатися **runner’а**, коли агент заблокований (наприклад `/approve`), відправляються **inline** через `await self._message_handler(event)` — вони обходять систему фонових завдань, щоб уникнути гонок.
## Авторизація

Шлюз використовує багаторівневу перевірку авторизації, що виконується у зазначеному порядку:

1. **Прапорець allow‑all для платформи** (наприклад, `TELEGRAM_ALLOW_ALL_USERS`) — якщо встановлено, усі користувачі на цій платформі авторизовані
2. **Білий список платформи** (наприклад, `TELEGRAM_ALLOWED_USERS`) — ідентифікатори користувачів, розділені комами
3. **DM‑парування** — автентифіковані користувачі можуть парувати нових користувачів за допомогою коду парування
4. **Глобальний прапорець allow‑all** (`GATEWAY_ALLOW_ALL_USERS`) — якщо встановлено, усі користувачі на всіх платформах авторизовані
5. **За замовчуванням: відхилення** — неавторизовані користувачі відхиляються

### Потік DM‑парування

```text
Admin: /pair
Gateway: "Pairing code: ABC123. Share with the user."
New user: ABC123
Gateway: "Paired! You're now authorized."
```

Стан парування зберігається у `gateway/pairing.py` і зберігається між перезапусками.
## Slash Command Dispatch

Усі slash‑команди в шлюзі проходять через один і той самий конвеєр розв’язання:

1. `resolve_command()` з `hermes_cli/commands.py` зіставляє вхід з канонічною назвою (обробляє псевдоніми, пошук за префіксом)
2. Канонічна назва перевіряється у `GATEWAY_KNOWN_COMMANDS`
3. Обробник у `_handle_message()` розподіляє виконання за канонічною назвою
4. Деякі команди обмежені конфігурацією (`gateway_config_gate` у `CommandDef`)

### Running-Agent Guard

Команди, які НЕ ПОВИННІ виконуватись, поки агент обробляє запит, відхиляються одразу:

```python
if _quick_key in self._running_agents:
    if canonical == "model":
        return "⏳ Agent is running — wait for it to finish or /stop first."
```

Команди‑обхід (`/stop`, `/new`, `/approve`, `/deny`, `/queue`, `/status`) мають спеціальну обробку.
## Джерела конфігурації

Шлюз читає конфігурацію з кількох джерел:

| Джерело | Що воно надає |
|--------|-----------------|
| `~/.hermes/.env` | API‑ключі, токени ботів, облікові дані платформ |
| `~/.hermes/config.yaml` | Налаштування моделі, конфігурація інструментів, параметри відображення |
| Environment variables | Перезаписують будь‑яке з вищезазначених |

На відміну від CLI (яке використовує `load_cli_config()` з жорстко закодованими значеннями за замовчуванням), шлюз читає `config.yaml` безпосередньо через YAML‑завантажувач. Це означає, що ключі конфігурації, які існують у словнику за замовчуванням CLI, але відсутні у файлі конфігурації користувача, можуть поводитися по‑різному між CLI та шлюзом.
## Адаптери платформ

Кожна платформа обміну повідомленнями має адаптер у `gateway/platforms/`:

```text
gateway/platforms/
├── base.py              # BaseAdapter — shared logic for all platforms
├── telegram.py          # Telegram Bot API (long polling or webhook)
├── discord.py           # Discord bot via discord.py
├── slack.py             # Slack Socket Mode
├── whatsapp.py          # WhatsApp Business Cloud API
├── signal.py            # Signal via signal-cli REST API
├── matrix.py            # Matrix via mautrix (optional E2EE)
├── mattermost.py        # Mattermost WebSocket API
├── email.py             # Email via IMAP/SMTP
├── sms.py               # SMS via Twilio
├── dingtalk.py          # DingTalk WebSocket
├── feishu.py            # Feishu/Lark WebSocket or webhook
├── wecom.py             # WeCom (WeChat Work) callback
├── weixin.py            # Weixin (personal WeChat) via iLink Bot API
├── bluebubbles.py       # Apple iMessage via BlueBubbles macOS server
├── qqbot/               # QQ Bot (Tencent QQ) via Official API v2 (sub-package: adapter.py, crypto.py, keyboards.py, …)
├── yuanbao.py           # Yuanbao (Tencent) DM/group adapter
├── feishu_comment.py    # Feishu document/drive comment-reply handler
├── msgraph_webhook.py   # Microsoft Graph change-notification webhook (Teams, Outlook, etc.)
├── webhook.py           # Inbound/outbound webhook adapter
├── api_server.py        # REST API server adapter
└── homeassistant.py     # Home Assistant conversation integration
```

Адаптери реалізують спільний інтерфейс:
- `connect()` / `disconnect()` — керування життєвим циклом
- `send_message()` — надсилання вихідних повідомлень
- `on_message()` — нормалізація вхідних повідомлень → `MessageEvent`

### Блокування токенів

Адаптери, які підключаються за допомогою унікальних облікових даних, викликають `acquire_scoped_lock()` у `connect()` та `release_scoped_lock()` у `disconnect()`. Це запобігає одночасному використанню одного й того ж токену бота двома профілями.
## Шлях доставки

Вихідні доставки (`gateway/delivery.py`) обробляють:

- **Пряма відповідь** — надсилати відповідь назад у чат, звідки вона походить
- **Доставка у домашній канал** — маршрутизувати вивід cron‑завдань та результати у фоні у налаштований домашній канал
- **Явна доставка до цілі** — інструмент `send_message`, що вказує `telegram:-1001234567890`, або [`hermes send` CLI](/guides/pipe-script-output), який обгортає той самий інструмент для скриптів оболонки
- **Крос‑платформна доставка** — доставляти на іншу платформу, ніж та, з якої надійшло повідомлення

Доставки cron‑завдань НЕ віддзеркалюються у історії сесії шлюзу — вони існують лише у власній cron‑сесії. Це навмисний дизайн, щоб уникнути порушень чергування повідомлень.
## Hooks

Gateway hooks — це Python‑модулі, які реагують на події життєвого циклу:

### Події шлюзу (Gateway Hook Events)

| Подія | Коли спрацьовує |
|-------|-----------------|
| `gateway:startup` | Запуск процесу шлюзу |
| `session:start` | Початок нової сесії розмови |
| `session:end` | Сесія завершується або закінчується за тайм‑аутом |
| `session:reset` | Користувач скидає сесію командою `/new` |
| `agent:start` | Агент починає обробляти повідомлення |
| `agent:step` | Агент завершує одну ітерацію виклику інструменту |
| `agent:end` | Агент завершує роботу і повертає відповідь |
| `command:*` | Виконується будь‑яка слеш‑команда |

Hook‑и виявляються у `gateway/builtin_hooks/` (точка розширення — наразі порожня у постаченій збірці; `_register_builtin_hooks()` — пустий заглушковий метод) та `~/.hermes/hooks/` (встановлені користувачем). Кожен hook — це каталог з маніфестом `HOOK.yaml` та файлом `handler.py`.
## Інтеграція провайдера пам'яті

Коли плагін провайдера пам'яті (наприклад, Honcho) увімкнено:

1. Gateway створює `AIAgent` для кожного повідомлення з ідентифікатором сесії
2. `MemoryManager` ініціалізує провайдера з контекстом сесії
3. Інструменти провайдера (наприклад, `honcho_profile`, `viking_search`) маршрутизуються через:

```text
AIAgent._invoke_tool()
  → self._memory_manager.handle_tool_call(name, args)
    → provider.handle_tool_call(name, args)
```

4. При завершенні/скиданні сесії викликається `on_session_end()` для очищення та фінального запису даних

### Життєвий цикл скидання пам'яті

Коли сесія скидається, відновлюється або закінчується:
1. Вбудовані пам'яті записуються на диск
2. Хук `on_session_end()` провайдера пам'яті викликається
3. Тимчасовий `AIAgent` виконує розмовний раунд лише з пам'яттю
4. Контекст потім відкидається або архівується
## Фонове обслуговування

Шлюз виконує періодичне обслуговування разом із обробкою повідомлень:

- **Cron ticking** — перевіряє розклади завдань і запускає заплановані завдання
- **Session expiry** — видаляє залишені сесії після закінчення їх терміну дії
- **Memory flush** — проактивно очищає пам'ять перед закінченням сесії
- **Cache refresh** — оновлює списки моделей та статус провайдерів
## Управління процесами

Шлюз працює як довгоживучий процес, яким можна керувати за допомогою:

- `hermes gateway start` / `hermes gateway stop` — ручне керування
- `systemctl` (Linux) або `launchctl` (macOS) — управління сервісом
- PID‑файл у `~/.hermes/gateway.pid` — відстеження процесу в межах профілю

**Профіль‑специфічний vs глобальний**: `start_gateway()` використовує PID‑файли, прив’язані до профілю. `hermes gateway stop` зупиняє лише шлюз поточного профілю. `hermes gateway stop --all` виконує глобальне сканування `ps aux` і завершує всі процеси шлюзу (використовується під час оновлень).
