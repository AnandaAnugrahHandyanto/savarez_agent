# Git Worktrees

Hermes Agent часто використовується у великих, довгоживучих репозиторіях. Коли потрібно:

- Запускати **кілька агентів паралельно** в одному проєкті, або
- Тримати експериментальні рефакторинги ізольованими від основної гілки,

Git **worktrees** — найнадійніший спосіб надати кожному агенту власний **checkout** без дублювання всього репозиторію.

Ця сторінка показує, як поєднати worktrees з Hermes, щоб кожна сесія мала чистий, ізольований робочий каталог.

## Чому варто використовувати worktrees з Hermes?

Hermes розглядає **поточний робочий каталог** як корінь проєкту:

- CLI: каталог, у якому ти запускаєш `hermes` або `hermes chat`
- Шлюзи обміну повідомленнями: каталог, встановлений змінною `MESSAGING_CWD`

Якщо запустити кілька агентів у **одному checkout**, їхні зміни можуть взаємодіяти:

- Один агент може видалити або перезаписати файли, які використовує інший.
- Складніше зрозуміти, які зміни належать яким експериментам.

За допомогою worktrees кожен агент отримує:

- **Власну гілку та робочий каталог**
- **Власну історію Checkpoint Manager** для `/rollback`

Дивись також: [Checkpoints and /rollback](./checkpoints-and-rollback.md).

## Швидкий старт: створення worktree

З головного репозиторію (що містить `.git/`) створіть новий worktree для гілки‑фічі:

```bash
# From the main repo root
cd /path/to/your/repo

# Create a new branch and worktree in ../repo-feature
git worktree add ../repo-feature feature/hermes-experiment
```

Це створює:

- Новий каталог: `../repo-feature`
- Нову гілку: `feature/hermes-experiment`, що checkout‑иться у цьому каталозі

Тепер можна `cd` у новий worktree і запустити Hermes там:

```bash
cd ../repo-feature

# Start Hermes in the worktree
hermes
```

Hermes:

- Бачить `../repo-feature` як корінь проєкту.
- Використовує цей каталог для контекстних файлів, правок коду та інструментів.
- Має **окрему історію контрольних точок** для `/rollback`, прив’язану до цього worktree.

## Запуск кількох агентів паралельно

Можна створити кілька worktrees, кожен зі своєю гілкою:

```bash
cd /path/to/your/repo

git worktree add ../repo-experiment-a feature/hermes-a
git worktree add ../repo-experiment-b feature/hermes-b
```

У різних терміналах:

```bash
# Terminal 1
cd ../repo-experiment-a
hermes

# Terminal 2
cd ../repo-experiment-b
hermes
```

Кожен процес Hermes:

- Працює у своїй гілці (`feature/hermes-a` vs `feature/hermes-b`).
- Записує контрольні точки у різний shadow‑repo hash (виходить з шляху worktree).
- Може використовувати `/rollback` незалежно, не впливаючи на інший.

Це особливо корисно, коли:

- Виконуєш пакетні рефакторинги.
- Пробуєш різні підходи до однієї задачі.
- Поєднуєш CLI‑ та шлюз‑сесії проти одного upstream‑репо.

## Безпечне видалення worktrees

Коли експеримент завершено:

1. Визнач, чи зберігати роботу, чи відкинути її.
2. Якщо треба зберегти:
   - Змерджи гілку у свою основну гілку, як зазвичай.
3. Видали worktree:

```bash
cd /path/to/your/repo

# Remove the worktree directory and its reference
git worktree remove ../repo-feature
```

Примітки:

- `git worktree remove` відмовиться видаляти worktree з незакоміченими змінами, якщо не вказати примусове видалення.
- Видалення worktree **не** видаляє гілку автоматично; гілку можна видалити або залишити за допомогою звичних команд `git branch`.
- Дані контрольних точок Hermes у `~/.hermes/checkpoints/` не очищуються автоматично при видаленні worktree, проте зазвичай займають дуже мало місця.

## Кращі практики

- **Один worktree на один експеримент Hermes**
  - Створюй окрему гілку/worktree для кожної суттєвої зміни.
  - Це робить diff‑и сфокусованими, а PR‑и маленькими та зручними для рев’ю.
- **Назви гілки за експериментом**
  - напр. `feature/hermes-checkpoints-docs`, `feature/hermes-refactor-tests`.
- **Комітити часто**
  - Використовуй git‑коміти для високорівневих етапів.
  - Використовуй [checkpoints and /rollback](./checkpoints-and-rollback.md) як страховку для інструментальних правок між ними.
- **Уникай запуску Hermes з кореня «голого» репо, коли працюєш з worktrees**
  - Користуйся каталогами worktree, щоб у кожного агента був чітко визначений контекст.

## Використання `hermes -w` (автоматичний режим worktree)

Hermes має вбудований прапорець `-w`, який **автоматично створює тимчасовий git worktree** зі своєю гілкою. Не потрібно налаштовувати worktrees вручну — просто `cd` у свій репо і запусти:

```bash
cd /path/to/your/repo
hermes -w
```

Hermes:

- Створить тимчасовий worktree у `.worktrees/` всередині репо.
- Checkout‑не ізольовану гілку (наприклад `hermes/hermes-<hash>`).
- Запустить повну CLI‑сесію всередині цього worktree.

Це найпростіший спосіб отримати ізоляцію worktree. Можна також поєднати це з однією запит‑командою:

```bash
hermes -w -q "Fix issue #123"
```

Для паралельних агентів відкрий кілька терміналів і запусти `hermes -w` у кожному — кожен виклик отримає свій власний worktree і гілку автоматично.

## Підсумок

- Використовуй **git worktrees**, щоб кожна сесія Hermes мала чистий **checkout**.
- Використовуй **гілки**, щоб фіксувати високорівневу історію експериментів.
- Використовуй **контрольні точки + `/rollback`**, щоб відновлюватись після помилок у кожному worktree.

Таке поєднання дає:

- Надійні гарантії, що різні агенти та експерименти не заважатимуть один одному.
- Швидкі цикли ітерацій з легким відновленням після невдалих правок.
- Чисті, зручні для рев’ю pull‑request’и.
