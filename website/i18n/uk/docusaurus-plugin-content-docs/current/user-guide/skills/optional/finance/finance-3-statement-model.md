# 3-Statement Financial Model Template Completion

Заповни інтегровані шаблони фінансових моделей із правильними зв’язками між Звітом про прибутки, Балансовим звітом та Звітом про грошові потоки.
## ⚠️ КРИТИЧНІ ПРИНЦИПИ — Прочитай перед заповненням будь‑якого шаблону

**Формули замість жорстких значень (непідлягає обговоренню):**
- Кожна клітинка проекції, roll‑forward, зв’язок і subtotal МАЄ бути формулою Excel — ніколи не попередньо обчисленим значенням
- При використанні Python/openpyxl: записуй рядки формул (`ws["D15"] = "=D14*(1+Assumptions!$B$5)"`), А НЕ обчислені результати (`ws["D15"] = 12500`)
- ЄДИНІ клітинки, які можуть містити жорстко закодовані числа, це: (1) історичні фактичні дані, (2) драйвери припущень у вкладці **Assumptions**
- Якщо ти знаходиш себе, обчислюючи значення в Python і записуючи результат у клітинку — ЗУПИНИ. Запиши формулу замість цього.
- Чому: модель повинна гнучко реагувати на перемикання сценаріїв або зміну припущень. Жорсткі значення порушують усі подальші перевірки цілісності без повідомлення.

**Перевіряй крок за кроком разом з користувачем:**
1. **Після картографування шаблону** → покажи користувачеві, які вкладки/розділи ти визначив, і підтверди перед тим, як змінювати будь‑які клітинки
2. **Після заповнення історичних даних** → покажи користувачеві блок історичних даних і підтверди, що значення/періоди відповідають вихідним даним
3. **Після побудови проекцій IS** → запусти перевірки subtotal, покажи користувачеві проєктований IS, підтверди перед переходом до BS
4. **Після побудови BS** → покажи користувачеві баланс (Assets = L+E) для кожного періоду, підтверди перед переходом до CF
5. **Після побудови CF** → покажи користувачеві звірку готівки (CF ending cash = BS cash), підтверди перед фіналізацією
6. **НЕ заповнюй всю модель від початку до кінця і не представляй її готовою** — розбивай на кожен крок, показуй роботу, виявляй помилки рано.
## Форматування — Професійна синя/сіра палітра (за замовчуванням, якщо шаблон/користувач не вказали інше)

**Тримай кольори мінімальними.** Використовуй лише сині та сірі для заповнення клітинок. Не вводь зелений, жовтий, оранжевий або кілька акцентних кольорів — чиста модель потребує стриманості.

| Елемент | Заповнення | Шрифт |
|---|---|---|
| Заголовки розділів (IS / BS / CF titles) | Темно‑синій `#1F4E79` | Білий **жирний** |
| Заголовки колонок (FY2024A, FY2025E тощо) | Світло‑синій `#D9E1F2` | Чорний **жирний** |
| Клітинки вводу (історичні дані, драйвери припущень) | Світло‑сірий `#F2F2F2` або білий | Синій `#0000FF` |
| Клітинки формул | Білий | Чорний |

**Це 3 сині + 1 сірий + білий.** Якщо шаблон має власну колірну схему, дотримуйся шаблону.

Колір шрифту сигналізує *що* є клітинка (ввід/формула/посилання). Колір заповнення сигналізує *де* ти знаходишся (заголовок/дані/підсумок).
## Структура моделі

### Організація вкладок шаблону

Шаблони різняться назвою вкладок та їхньою організацією. Перш ніж заповнювати, переглянь усі вкладки, щоб зрозуміти структуру шаблону. Нижче наведено типові назви вкладок та їхній вміст:

| Типові назви вкладок | Що шукати у вмісті |
|---------------------|--------------------|
| IS, P&L, Income Statement | Звіт про прибутки та збитки |
| BS, Balance Sheet | Балансовий звіт |
| CF, CFS, Cash Flow | Звіт про грошові потоки |
| WC, Working Capital | Розклад робочого капіталу |
| DA, D&A, Depreciation, PP&E | Розклад амортизації та зносу |
| Debt, Debt Schedule | Розклад боргу |
| NOL, Tax, DTA | Розклад чистих операційних збитків |
| Assumptions, Inputs, Drivers | Припущення та вхідні дані |
| Checks, Audit, Validation | Панель перевірки помилок |

**Контрольний список перегляду шаблону**
- Визначити, які вкладки присутні у шаблоні (не всі шаблони містять кожен розклад)
- Зафіксувати будь‑які специфічні для шаблону вкладки, яких немає у списку вище
- Розуміти залежності між вкладками (наприклад, які розклади живлять основні звіти)
- Знайти клітини вводу та клітини формул на кожній вкладці

### Розуміння структури шаблону

Перш ніж заповнювати шаблон, ознайомся з його існуючим макетом, щоб дані вводились у правильних місцях і формули залишались незмінними.

**Визначення структури рядків**
- Знайди назву моделі у верхній частині кожної вкладки
- Визначи заголовки розділів та їх візуальне розділення
- Знайди рядок одиниць, що вказує $ мільйони, %, x тощо
- Зафіксуй заголовки колонок, що розрізняють фактичні дані та оцінки
- Перевір позначення періодів (наприклад, FY2024A, FY2025E)
- Визначи клітини вводу та клітини формул (зазвичай розрізняються кольором шрифту)

**Визначення структури колонок**
- Перевір назви рядків у найлівішій колонці
- Переконайся, що історичні роки йдуть перед проекційними
- Зафіксуй візуальну межу, що розділяє історичні та проекційні періоди
- Перевір послідовність колонок у всіх вкладках

**Робота з іменованими діапазонами**
Шаблони часто використовують іменовані діапазони для ключових ввідних та вихідних даних. Перед введенням даних:
- Переглянь існуючі іменовані діапазони у шаблоні (`Formulas → Name Manager` в Excel)
- Типові іменовані діапазони включають: темпи зростання доходу, відсотки витрат, ключові вихідні (Net Income, EBITDA, Total Debt, Cash), клітина‑вибір сценарію
- Переконайся, що ввідні дані вводяться у клітини, що живлять ці іменовані діапазони

### Період проекції
- Шаблони зазвичай проєктують 5 років вперед від останнього історичного року
- Перевір, щоб колонки історичних (A) та проекційних (E) даних були чітко розділені
- Переконайся, що колонки використовують позначення фінансового року (наприклад, FY2024A, FY2025E)
## Аналіз маржі

**Примітка: Наведений аналіз маржі слід виконувати лише за запитом користувача або якщо шаблон явно вимагає це. Якщо запиту немає, пропусти цей розділ.**

Розрахуй і відобрази маржі прибутковості на вкладці **Income Statement (IS)**, щоб відстежувати операційну ефективність і забезпечити порівняння з колегами.

### Основні маржі, які слід включити

| Маржа | Формула | Що вимірює |
|--------|---------|------------------|
| Gross Margin | Gross Profit / Revenue | Цінова потужність, ефективність виробництва |
| EBITDA Margin | EBITDA / Revenue | Основна операційна прибутковість |
| EBIT Margin | EBIT / Revenue | Операційна прибутковість після амортизації та зносу |
| Net Income Margin | Net Income / Revenue | Прибутковість чистого результату |

### Макет звіту про прибутки та збитки з маржами

Відображай відсотки маржі безпосередньо під кожним рядком прибутку:
- Gross Margin % під Gross Profit
- EBIT Margin % під EBIT
- EBITDA Margin % під EBITDA
- Net Income Margin % під Net Income
## Кредитні метрики

**Примітка: Аналіз кредиту слід виконувати лише за запитом користувача або якщо шаблон явно вимагає це. Якщо запит не надано, пропусти цей розділ.**

Розрахуй і відобрази метрики кредиту/важеля на вкладці **Balance Sheet (BS)**, щоб оцінити фінансове здоров’я, боргову спроможність і дотримання ковенантів.

### Основні кредитні метрики, які слід включити

| Метрика | Формула | Що вимірює |
|--------|---------|------------|
| Total Debt / EBITDA | Total Debt / LTM EBITDA | Мультиплікатор важеля |
| Net Debt / EBITDA | (Total Debt - Cash) / LTM EBITDA | Важель за вирахуванням готівки |
| Interest Coverage | EBITDA / Interest Expense | Спосіб обслуговування боргу |
| Debt / Total Cap | Total Debt / (Total Debt + Equity) | Капітальна структура |
| Debt / Equity | Total Debt / Total Equity | Фінансовий важель |
| Current Ratio | Current Assets / Current Liabilities | Короткострокова ліквідність |
| Quick Ratio | (Current Assets - Inventory) / Current Liabilities | Негайна ліквідність |

### Перевірки ієрархії кредитних метрик

Переконайся, що **Upside** демонструє найсильніший кредитний профіль:
- **Leverage**: Upside < Base < Downside (нижче — краще)
- **Coverage**: Upside > Base > Downside (вище — краще)
- **Liquidity**: Upside > Base > Downside (вище — краще)

### Відстеження дотримання ковенантів

Якщо відомі боргові ковенанти, додай явні перевірки дотримання, порівнюючи фактичні метрики з пороговими значеннями ковенантів.
## Аналіз сценаріїв (Base / Upside / Downside)

Використай перемикач сценаріїв (випадаючий список) у вкладці **Assumptions** з формулами `CHOOSE` або `INDEX/MATCH`.

| Сценарій | Опис |
|----------|------|
| Base Case | Керівництво компанії або консенсус‑оцінки |
| Upside Case | Зростання вище рекомендацій, розширення маржі |
| Downside Case | Зростання нижче тренду, стискання маржі |

**Ключові драйвери для чутливості**: зростання доходу, валова маржа, % SG&A, DSO/DIO/DPO, % CapEx, процентна ставка, податкова ставка.

**Перевірки аудиту сценаріїв**: перемикачі охоплюють усі заяви, баланси (BS) у всіх сценаріях, готівка збігається, ієрархія дотримується (Upside > Base > Downside для NI, EBITDA, FCF, марж).

## Заповнення шаблонів моделей

У цьому розділі наведено загальні рекомендації щодо заповнення будь‑якого шаблону фінансової моделі з трьома звітами, зберігаючи існуючі формули та забезпечуючи цілісність даних.

### Крок 1: Аналіз структури шаблону

Перш ніж вводити дані, ретельно переглянь шаблон, щоб зрозуміти його архітектуру:

**Визначення клітинок вводу та формул**
- Шукай візуальні підказки (колір шрифту, заливка клітинок), які розрізняють клітинки вводу від клітинок формул.
- Типові конвенції: синій шрифт = вхідні дані, чорний шрифт = формули, зелений шрифт = посилання на інші листи.
- Використовуй **Trace Precedents/Dependents** в Excel (Formulas → Trace Precedents), щоб зрозуміти взаємозв’язки клітинок.
- Перевіряй названі діапазони, які можуть керувати ключовими вводами (Formulas → Name Manager).

**Картування потоку шаблону**
- Визнач, які вкладки постачають дані іншим (наприклад, Assumptions → IS → BS → CF).
- Зафіксуй будь‑які допоміжні розклади та їх зв’язки з основними звітами.
- Документуй конкретні рядки та структуру шаблону перед заповненням.

### Крок 2: Заповнення даних без порушення формул

**Золоті правила вводу даних**

| Правило | Опис |
|------|------|
| Редагувати лише клітинки вводу | Ніколи не перезаписуй клітинки, що містять формули, якщо не замінюєш формулу навмисно |
| Зберігати посилання на клітинки | При копіюванні даних використовуйте **Paste Values** (Ctrl+Shift+V), щоб не перезаписати формули форматуванням джерела |
| Відповідність одиниць шаблону | Перевір, чи шаблон працює в тисячах, мільйонах чи в реальних значеннях, перед вводом даних |
| Дотримання конвенцій знаків | Дотримуйся існуючої конвенції знаків у шаблоні (наприклад, витрати як позитивні або негативні) |
| Перевірка на циклічні посилання | Якщо шаблон використовує ітеративні обчислення, переконайся, що **Enable Iterative Calculation** увімкнено |

**Безпечний процес вводу даних**
1. Визнач точні клітинки, призначені для вводу (зазвичай підсвічені або позначені).
2. Спочатку введи історичні дані, потім перевір, чи формули правильно розраховують ці періоди.
3. Введи драйвери‑припущення, які живлять прогностичні розрахунки.
4. Переглянь розраховані результати, щоб підтвердити коректність формул.
5. Якщо потрібно змінити клітинку з формулою, задокументуй оригінальну формулу перед зміною.

**Обробка вбудованих формул**
- Якщо формули посилаються на клітинки, які ще не заповнені, очікуй тимчасові помилки (#REF!, #DIV/0!) до завершення вводу.
- Коли формули дають неочікувані результати, використай **Trace Precedents**, щоб знайти відсутні або неправильні вводи.
- Ніколи не видаляй рядки/стовпці без перевірки залежностей формул у всіх вкладках.

### Крок 3: Перевірка формул

**Перевірки цілісності формул**

Перед тим як покладатися на результати шаблону, переконайся, що формули працюють правильно:

| Тип перевірки | Метод |
|--------------|-------|
| Trace precedents | Вибери клітинку з формулою → Formulas → Trace Precedents, щоб перевірити, чи посилається вона на правильні вводи |
| Trace dependents | Переконайся, що ключові вводи потрапляють у очікувані клітинки результату |
| Evaluate formula | Використай Formulas → Evaluate Formula, щоб покроково пройти складні обчислення |
| Перевірка на жорсткі значення | Прогностичні формули мають посилатися на припущення, а не містити жорстко закодовані значення |
| Тест з відомими значеннями | Введи прості тестові значення, щоб підтвердити очікувані результати |
| Узгодженість між листами | Переконайся, що однакова логіка формул застосовується у всіх періодах прогнозу |

**Типові проблеми формул**
- Змішані абсолютні/відносні посилання, що дають неправильні результати при копіюванні між періодами.
- Розірвані посилання на зовнішні файли або видалені діапазони (помилки #REF!).
- Ділення на нуль у ранніх періодах перед зростанням доходу (помилки #DIV/0!).
- Попередження про циклічні посилання (можуть бути навмисними для розрахунків відсотків).
- Несумісність формул між колонками прогнозу (використай Ctrl+\, щоб знайти відмінності).

**Перевірка зв’язків між листами**
- Підтверди, що значення, які з’являються на кількох листах, пов’язані (не дублюються).
- Переконайся, що підсумки розкладів відповідають відповідним рядкам у головних звітах.
- Перевір, чи мітки періодів збігаються на всіх листах.

### Крок 4: Перевірки якості за листами

Після заповнення шаблону виконай ці перевірки на кожному листі:

**Income Statement (IS) – Перевірки якості**
- Фігури доходу відповідають вихідним даним за історичні періоди.
- Усі статті витрат сумуються до зазначених підсумків.
- Проміжні підсумки (Gross Profit, EBIT, EBT, Net Income) розраховуються правильно.
- Логіка розрахунку податків адекватна (правильно обробляє збитки).
- Драйвери прогнозу посилаються на вкладку Assumptions (без жорстких значень).
- Зміни період‑за‑періодом є розумними.

**Balance Sheet (BS) – Перевірки якості**
- Assets = Liabilities + Equity для кожного періоду (основна перевірка).
- Баланс готівки відповідає кінцевій готівці у Cash Flow Statement.
- Робочий капітал збігається з підтримуючими розкладами (за потреби).
- Retained Earnings переноситься правильно: Prior RE + Net Income - Dividends +/- Adjustments = Ending RE.
- Баланси боргу відповідають розкладу боргу (за потреби).
- Усі статті балансу мають правильні знаки (активи позитивні, більшість зобов’язань позитивні).

**Cash Flow Statement (CF) – Перевірки якості**
- Net Income у розділі CFO збігається з Net Income у Income Statement.
- Негрошові надбавки (D&A, SBC тощо) відповідають їх джерельним розкладам/звітам.
- Зміни робочого капіталу мають правильні знаки (збільшення активу = використання готівки = негатив).
- CapEx відповідає розкладу PP&E або рол‑форварду основних засобів.
- Фінансові операції відповідають змінам у боргових та капітальних рахунках у BS.
- Кінцева готівка збігається з Cash у Balance Sheet.
- Початкова готівка дорівнює кінцевій готівці попереднього періоду.

**Перевірки підтримуючих розкладів**
- Початкові залишки дорівнюють закритим залишкам попереднього періоду.
- Логіка рол‑форварду завершена (Beginning + Additions - Deductions = Ending).
- Підсумки розкладів відповідають рядкам у головному звіті.
- Припущення, використані у розрахунках, збігаються з вкладкою Assumptions.

### Крок 5: Перевірки цілісності між звітами

Після перевірки окремих листів підтвердь правильну інтеграцію трьох звітів:

| Перевірка | Формула | Очікуваний результат |
|----------|--------|-----------------------|
| Баланс балансу | Assets - Liabilities - Equity | = 0 |
| Зв’язок готівки | CF Ending Cash - BS Cash | = 0 |
| Зв’язок чистого прибутку | IS Net Income - CF Starting Net Income | = 0 |
| Нерозподілений прибуток | Prior RE + NI - Dividends - BS Ending RE | = 0 (коригувати для SBC/інших пунктів за потреби) |

### Крок 6: Останній огляд

Перш ніж вважати модель завершеною:
- Переглянь усі сценарії (за потреби), щоб переконатися, що перевірки проходять у кожному випадку.
- Переглянь усі помилки #REF!, #DIV/0!, #VALUE! та #NAME? і виправ їх або задокументуй.
- Переконайся, що всі клітинки вводу заповнені (пошукай заповнювачі).
- Перевір, чи одиниці вимірювання узгоджені у всіх листах.
- Збережи чисту версію перед будь‑якими додатковими змінами.

### Circular Reference Handling

Interest expense creates circularity: Interest → Net Income → Cash → Debt Balance → Interest

Enable iterative calculation in Excel: **File → Options → Formulas → Enable iterative calculation**. Set maximum iterations to **100**, maximum change to **0.001**. Add a circuit‑breaker toggle in the **Assumptions** tab.

### Check Categories

**Section 1: Currency Consistency**
- Currency identified and documented in **Assumptions**
- All tabs use the same currency symbol and scale
- Units row matches model currency

**Section 2: Balance Sheet Integrity**
- Assets = Liabilities + Equity (for each period)
- Formula: Assets - Liabilities - Equity (must = 0)

**Section 3: Cash Flow Integrity**
- Cash ties to BS (CF Ending Cash = BS Cash)
- Cash Monthly vs Annual: Closing Cash (Monthly) = Closing Cash (Annual)
- NI ties to IS (CF Net Income = IS Net Income)
- D&A ties to schedule
- SBC ties to IS
- ΔAR, ΔInventory, ΔAP tie to WC schedule
- CapEx ties to DA schedule

**Section 4: Retained Earnings**
- RE roll‑forward check: Prior RE + NI + SBC - Dividends = Ending RE
- Show component breakdown for debugging

**Section 5: Working Capital**
- AR, Inventory, AP tie to BS
- DSO, DIO, DPO reasonability checks (flag if outside normal ranges)

**Section 6: Debt Schedule**
- Total Debt ties to BS (Current + LT Debt)
- Interest calculation ties to IS

**Section 6b: Equity Financing**
- Equity issuance proceeds tie to BS Common Stock/APIC increase
- Cash increase from equity = Equity account increase (must balance)
- Equity Raise Tie‑Out: ΔCommon Stock/APIC (BS) = Equity Issuance (CFF) (must = 0)
- Year 0 Equity Tie‑Out: Equity Raised (Year 0) = Beginning Equity Capital (Year 1)

**Section 6c: NOL Schedule**
- Beginning NOL (Year 1 / Formation) = 0 (new business starts with zero NOL)
- NOL increases only when EBT < 0 (losses must be realized to generate NOL)
- DTA ties to BS (NOL Schedule DTA = BS Deferred Tax Asset)
- NOL utilization ≤ 80 % of EBT (post‑2017 federal limitation)
- NOL balance is non‑negative (cannot utilize more than available)
- NOL generated only when EBT < 0
- Tax expense = 0 when taxable income ≤ 0

**Section 7: Scenario Hierarchy**
- Absolute metrics: Upside > Base > Downside (NI, EBITDA, FCF)
- Margins: Upside > Base > Downside (GM %, EBITDA %, NI %)
- Credit metrics: Upside < Base < Downside for leverage (inverted)

**Section 8: Formula Integrity**
- COGS, S&M, G&A, R&D, SBC driven by % of Revenue (no hard‑codes)
- Consistent formulas across projection years
- No #REF!, #DIV/0!, #VALUE! errors

**Section 9: Credit Metric Thresholds**
- Flag metrics as Green/Yellow/Red based on covenant thresholds
- Summary of any red flags

### Master Check Formula

Aggregate all section statuses into a single master check:
- If all sections pass → **"✓ ALL CHECKS PASS"**
- If any section fails → **"✗ ERRORS DETECTED - REVIEW BELOW"**

### Quick Debug Workflow

When Master Status shows errors:
1. Scroll to find red‑highlighted sections
2. Identify which check category has failures
3. Navigate to the source tab to investigate
4. Fix the underlying issue
5. Return to the **Checks** tab to verify resolution
