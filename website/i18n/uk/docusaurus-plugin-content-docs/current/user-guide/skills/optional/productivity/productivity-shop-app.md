# Shop.app — Персональний помічник покупок

Використовуй цю навичку, коли користувач хоче **шукати товари в різних магазинах, порівнювати ціни, знаходити схожі товари, відстежувати замовлення, керувати поверненням або повторно замовити попередню покупку** через API агента Shop.app.

Для пошуку товарів автентифікація не потрібна. Автентифікація (device-authorization flow) потрібна для будь‑якої операції, що стосується користувача: замовлення, відстеження, повернення, повторне замовлення. Зберігай токени **лише у своїй робочій пам’яті для поточної сесії** — ніколи не записуй їх на диск, ніколи не проси користувача вставляти їх.

Всі кінцеві точки повертають **plain‑text markdown** (включаючи помилки, які виглядають як `# Error\n\n{message} ({status})`). Використовуй `curl` через інструмент `terminal`; для функції примірки використай інструмент `image_generate`.

## Пошук продукту (без автентифікації)

**Endpoint:** `GET https://shop.app/agents/search`

| Parameter | Type | Required | Default | Description |
|---|---|---|---|---|
| `query` | string | yes | — | Ключові слова пошуку |
| `limit` | int | no | 10 | Результати 1–10 |
| `ships_to` | string | no | `US` | ISO‑3166 код країни (контролює валюту + доступність) |
| `ships_from` | string | no | — | ISO‑3166 код країни походження продукту |
| `min_price` | decimal | no | — | Мінімальна ціна |
| `max_price` | decimal | no | — | Максимальна ціна |
| `available_for_sale` | int | no | 1 | `1` = лише в наявності |
| `include_secondhand` | int | no | 1 | `0` = лише нові |
| `categories` | string | no | — | Ідентифікатори таксономії Shopify, розділені комами |
| `shop_ids` | string | no | — | Фільтр за конкретними магазинами |
| `products_limit` | int | no | 10 | Кількість варіантів на продукт, 1–10 |

```
curl -s 'https://shop.app/agents/search?query=wireless+earbuds&limit=10&ships_to=US'
```

**Формат відповіді:** простий текст. Продукти розділяються рядком `\n\n---\n\n`.

**Поля, які потрібно витягнути для кожного продукту:**
- **Title** — перший рядок
- **Price + Brand + Rating** — другий рядок (`$PRICE at BRAND — RATING`)
- **Product URL** — рядок, що починається з `https://`
- **Image URL** — рядок, що починається з `Img: `
- **Product ID** — рядок, що починається з `id: `
- **Variant IDs** — у розділі Variants або з параметра `variant=` у URL продукту
- **Checkout URL** — рядок, що починається з `Checkout: ` (містить плейсхолдер `{id}`; заміни його реальним ідентифікатором варіанту)

**Пагінація:** відсутня. Щоб отримати інші результати, **змінюй запит** (різні ключові слова, синоніми, більш вузькі чи широкі терміни). До ≈ 3 пошукових раундів.

**Помилки:** відсутній/порожній `query` повертає `# Error\n\nquery is missing (400)`.

## Знайти схожі продукти

Той самий формат відповіді, що й у пошуку продуктів.

**За ідентифікатором варіанту (GET):**

```
curl -s 'https://shop.app/agents/search?variant_id=33169831854160&limit=10&ships_to=US'
```

`variant_id` має надходити з параметра запиту `variant=` у URL продукту — поле `id:` з результатів пошуку **не** приймається.

**За зображенням (POST):**

```
curl -s -X POST https://shop.app/agents/search \
  -H 'Content-Type: application/json' \
  -d '{"similarTo":{"media":{"contentType":"image/jpeg","base64":"<BASE64>"}},"limit":10}'
```

Потрібні байти зображення, закодовані у base64. URL‑адреси **не** приймаються — спочатку завантажте зображення (`curl -o`), потім `base64 -w0 file.jpg` для вбудовування.
## Аутентифікація — Потік авторизації пристрою (RFC 8628)

Обов’язково для замовлень, відстеження, повернень, повторного замовлення. Не потрібно для пошуку товарів.

**Стан сесії (утримуй у контексті розмови лише для цього діалогу):**

| Ключ | Тривалість | Опис |
|---|---|---|
| `access_token` | доки не закінчиться / 401 | Bearer‑токен для автентифікованих кінцевих точок |
| `refresh_token` | доки оновлення не зазнає невдачі | Оновлює `access_token` без повторної автентифікації |
| `device_id` | вся сесія | `shop-skill--<uuid>` — генерується один раз, використовується для кожного запиту |
| `country` | вся сесія | ISO‑код країни (`US`, `CA`, `GB`, …) — запитуй або виводь |

**Правила:**
- `user_code` завжди 8 символів A‑Z, у форматі `XXXXXXXX`.
- Не потрібні `client_id`, `client_secret` або callback — проксі обробляє це.
- **Ніколи не проси користувача вставляти токени в чат.**
- Токени живуть лише протягом цієї розмови. Не записуй їх у `.env` чи будь‑який файл.

### Потік

**1. Запитати код пристрою:**
```
curl -s -X POST https://shop.app/agents/auth/device-code
```
Відповідь містить `device_code`, `user_code`, `sign_in_url`, `interval`, `expires_in`. Показати користувачеві `sign_in_url` (та `user_code`).

**2. Опитувати для отримання токену** кожні `interval` секунд:
```
curl -s -X POST https://shop.app/agents/auth/token \
  --data-urlencode 'grant_type=urn:ietf:params:oauth:grant-type:device_code' \
  --data-urlencode "device_code=$DEVICE_CODE"
```
Обробляти помилки: `authorization_pending` (продовжувати опитування), `slow_down` (додати 5 с до інтервалу), `expired_token` / `access_denied` (перезапустити потік). При успіху повертаються `access_token` + `refresh_token`.

**3. Перевірка:**
```
curl -s https://shop.app/agents/auth/userinfo \
  -H "Authorization: Bearer $ACCESS_TOKEN"
```

**4. Оновлення при 401:**
```
curl -s -X POST https://shop.app/agents/auth/token \
  --data-urlencode 'grant_type=refresh_token' \
  --data-urlencode "refresh_token=$REFRESH_TOKEN"
```
Якщо оновлення не вдається, перезапусти потік авторизації пристрою.
## Замовлення

> **Обсяг:** Shop.app агрегує замовлення з **усіх магазинів** (не лише Shopify), використовуючи електронні квитанції, які користувач підключив у додатку Shop. Цей інструмент ніколи не працює безпосередньо з електронною поштою користувача.

**Прогрес статусу:** `paid → fulfilled → in_transit → out_for_delivery → delivered`
**Інші:** `attempted_delivery`, `refunded`, `cancelled`, `buyer_action_required`

### Шаблон отримання

```
curl -s 'https://shop.app/agents/orders?limit=50' \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "x-device-id: $DEVICE_ID"
```

Параметри: `limit` (1–50, за замовчуванням 20), `cursor` (з попередньої відповіді).

**Ключові поля для вилучення:**
- **UUID замовлення** — `uuid: …`
- **Магазин** — `at …`, `Store domain: …`, `Store URL: …`
- **Ціна** — рядок після `Store URL`
- **Дата** — `Ordered: …`
- **Статус / Доставка** — `Status: …`, `Delivery: …`
- **Можливість повторного замовлення** — `Can reorder: yes`
- **Товари** — під розділом `— Items —`, кожен з необов’язковими `[product:ID]` `[variant:ID]` та `Img:`
- **Відстеження** — під розділом `— Tracking —` (перевізник, код, URL відстеження, ETA)
- **ID трекера** — `tracker_id: …`
- **URL повернення** — `Return URL: …` (тільки якщо можливо)

**Пагінація:** якщо перший рядок — `cursor: <value>`, передай його назад як `?cursor=<value>` для наступної сторінки. Продовжуй, доки не з’явиться рядок без `cursor:`.

**Фільтрація:** застосовуй на боці клієнта після отримання (за датою `Ordered:`, статусом `Delivery:` тощо).

**Помилки:** при 401 оновити токен і повторити запит. При 429 чекати 10 сек і повторити.

### Деталі відстеження

Відстеження розташоване в розділі `— Tracking —` кожного замовлення:
```
delivered via UPS — 1Z999AA10123456784
Tracking URL: https://ups.com/track?num=…
ETA: Arrives Tuesday
```

**Попередження про застаріле відстеження:** якщо `Ordered:` має кілька місяців, а доставка все ще `in_transit`, повідом користувача, що відстеження може бути застарілим.
## Повернення

Два джерела:

**1. URL повернення на рівні замовлення** — шукай `Return URL: …` у даних замовлення.

**2. Політика повернення на рівні продукту:**
```
curl -s 'https://shop.app/agents/returns?product_id=29923377167' \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "x-device-id: $DEVICE_ID"
```

Поля: `Returnable` (`yes` / `no` / `unknown`), `Return window` (days), `Return policy URL`, `Shipping policy URL`.

Для отримання повного тексту політики, отримай URL політики повернення за допомогою `web_extract` (або `curl` + видалення тегів) — це HTML.
## Повторне замовлення

1. Отримай замовлення з `limit=50`, знайди ціль за `uuid:` або збігом магазину/товару.
2. Перевір `Can reorder: yes` — якщо параметр відсутній, повторне замовлення може не працювати.
3. Витягни `[variant:ID]` та назву товару з `— Items —`, а домен магазину — з `Store domain:` або `Store URL:`.
4. Сформуй URL оформлення: `https://{domain}/cart/{variantId}:{quantity}`.

**Приклад:** `at Allbirds` + `Store domain: allbirds.myshopify.com` + `[variant:789012]` → `https://allbirds.myshopify.com/cart/789012:1`

**Відсутній варіант (наприклад, замовлення Amazon, без `[variant:ID]`):** використай запасний (фолбек) посилання на пошук у магазині: `https://{domain}/search?q={title}`.
## Build a Checkout URL

| Parameter | Description |
|---|---|
| `items` | Масив об’єктів `{ variant_id, quantity}` |
| `store_url` | URL магазину (наприклад `https://allbirds.ca`) |
| `email` | Попередньо заповнити email — лише з інформації, яку вже маєш |
| `city` | Попередньо заповнити місто |
| `country` | Попередньо заповнити код країни |

**Pattern:** `https://{store}/cart/{variant_id}:{qty},{variant_id}:{qty}?checkout[email]=…`

URL `Checkout:` у результатах пошуку містить `{id}` як заповнювач — заміни його реальним `variant_id`.

- **Default:** посилання на сторінку продукту, щоб користувач міг переглянути.
- **"Buy now":** використай checkout‑URL з конкретним варіантом.
- **Multi-item, same store:** один комбінований URL.
- **Multi-store:** окремі checkout‑URL для кожного магазину — повідом користувача.
- **Never claim the purchase is complete.** Користувач платить на сайті магазину.
## Віртуальна примірка та візуалізація

Коли доступна функція `image_generate`, пропонуй візуалізувати товари для користувача:
- Одяг / взуття / аксесуари → віртуальна примірка за допомогою фото користувача
- Меблі / декор → розміщення у фото кімнати користувача
- Мистецтво / принти → попередній перегляд на стіні користувача

Під час першого пошуку одягу, аксесуарів, меблів, декору або мистецтва, згадай це **один раз**: *«Хочеш побачити, як це виглядатиме на тобі? Надішли мені фото, і я створю візуалізацію.»*

Результати є приблизними (кольори, пропорції, посадка) — для натхнення, а не точного відтворення.
## Політики магазину

Отримуй безпосередньо з домену магазину:
```
https://{shop_domain}/policies/shipping-policy
https://{shop_domain}/policies/refund-policy
```

Вони повертають HTML — використай `web_extract` (or `curl` + strip tags) перед показом.

Коли маєш `product_id` з позицій замовлення, надавай перевагу `GET /agents/returns?product_id=…` для перевірки права на повернення + посилань на політики.

## Бути A+ шопінг‑асистентом

Вести розмову з **продуктами**, а не з розповіддю.

**Стратегія пошуку:**
1. **Спочатку шукай широко** — варіюй терміни, змішуй синоніми + категорію + бренд. Використовуй фільтри (`min_price`, `max_price`, `ships_to`), коли це доречно.
2. **Оцінюй** — прагни 8–10 результатів за ціною / брендом / стилем. До 3‑х раундів повторного пошуку з різними запитами. Не «сторінка 2» — змінюй запит.
3. **Організуй** — згрупуй у 2–4 теми (використання, ціновий діапазон, стиль).
4. **Представляй** — 3–6 продуктів у кожній групі з зображенням, назвою + брендом, ціною (локальна валюта, коли можливо, діапазон, якщо min ≠ max), рейтинг + кількість відгуків, однорядковий опис з реальних даних продукту, підсумок варіантів («6 кольорів, розміри S‑XXL»), посилання на сторінку продукту та посилання **Buy Now** для оформлення покупки.
5. **Рекомендуй** — виділи 1–2 найкращих з конкретною причиною («4.8 / 5 за 2 000+ відгуків»).
6. **Задай одне цілеспрямоване уточнення**, яке наблизить до рішення.

**Відкриття** (загальний запит): шукай одразу, не задавай зайвих уточнюючих питань.
**Уточнення** («до $50», «в синьому»): коротко підтверди, покажи підходящі варіанти, при потребі виконай повторний пошук.
**Порівняння**: починай з ключової компромісної характеристики, порівнюй специфікації бок‑о‑бок, давай рекомендації під конкретну ситуацію.

**Слабкі результати?** Не здавайся після одного запиту. Спробуй ширші терміни, прибери прикметники, запити лише за категорією, назви брендів або розділи складні запити. Приклад: `dimmable vintage bulbs e27` → `vintage edison bulbs` → `e27 dimmable bulbs` → `filament bulbs`.

**Стратегія пошуку замовлень:**
1. Отримай 50 замовлень (`limit=50`) — використовуй великий ліміт для пошуку.
2. Переглянь на відповідність за магазином (`at <store>`) або назвою товару у `— Items —`. Підбирай приблизно — «Yoto» підходить до «Yoto Ltd».
3. Дій за знайденим: відстеження, повернення або повторне замовлення.
4. Якщо нічого не знайдено? Пагінуй за допомогою `cursor` або попроси більше деталей.

| Користувач каже | Стратегія |
|---|---|
| "Where's my Yoto order?" | Отримати 50 → знайти `at Yoto` → показати відстеження |
| "Show me recent orders" | Отримати 20 (за замовчуванням) |
| "Return the shoes from January?" | Отримати 50 → відфільтрувати за `Ordered:` у січні → перевірити повернення |
| "Reorder the coffee" | Отримати 50 → знайти позицію кави → сформувати URL оформлення |
| "Did I order one of these before?" | Отримати 50 → зіставити з поточними результатами пошуку → показати збіги |
## Форматування

**Кожен товар:**
- Зображення
- Назва + бренд
- Ціна (місцева валюта; показувати діапазони, коли мінімум ≠ максимум)
- Оцінка + кількість відгуків
- Одне речення‑диференціатор на основі реальних даних про товар
- Короткий опис доступних варіантів
- Посилання на сторінку товару
- Посилання на оформлення покупки «Buy Now» (будується з ідентифікатора варіанту за шаблоном оформлення)

**Замовлення:**
- Підсумовуй природно — не вставляй сирі поля.
- Виділяй ETA для товарів у дорозі; дати для доставлених.
- Пропонуй подальші дії: «Хочеш деталі відстеження?», «Хочеш повторно замовити?»
- Пам’ятай: охоплення — це всі магазини, підключені до Shop, а не лише Shopify.

Адаптери шлюзу Hermes (Telegram, Discord, Slack, iMessage, …) автоматично рендерять markdown та URL‑зображень. Пиши звичайний markdown, розміщуючи URL‑зображень на окремому рядку — адаптер сам подбає про специфічне для платформи форматування. Не вигадуй виклик інструменту `message()` (це належить власному середовищу виконання Shop.app, а не Hermes).
## Правила

- Використовуй те, що вже знаєш про користувача (країна, розмір, вподобання) — не запитуй повторно.
- Ніколи не вигадуй URL‑адреси та не створюй специфікації.
- Ніколи не розповідай користувачеві про використання інструментів, внутрішні ідентифікатори чи параметри API.
