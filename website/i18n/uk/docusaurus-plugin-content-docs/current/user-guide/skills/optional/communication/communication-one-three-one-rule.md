# Правило 1‑3‑1

Структурована рамка прийняття рішень для технічних пропозицій та аналізу компромісів. Коли користувач стикається з вибором між кількома підходами (архітектурні рішення, вибір інструментів, стратегії рефакторингу, шляхи міграції), цей скілл генерує формат 1‑3‑1: одне чітке формулювання проблеми, три різні варіанти з перевагами/недоліками і одну конкретну рекомендацію з визначенням готовності та планом впровадження. # 1‑3‑1 Правило Комунікації

Структурований формат прийняття рішень, коли завдання має кілька життєздатних підходів і користувач потребує чіткої рекомендації. Генерує стислий опис проблеми, три варіанти з компромісами та дієвий план для рекомендованого шляху.

## Коли використовувати

- Користувач явно просить про відповідь у форматі «1‑3‑1».
- Користувач каже «дай мені варіанти» або «які у мене вибори» щодо технічного рішення.
- Завдання має кілька життєздатних підходів із значущими компромісами (архітектура, інструменти, стратегія міграції).
- Користувач потребує пропозиції, яку можна передати команді або зацікавленій стороні.

Не використовуйте для простих питань з очевидною відповіддю, сеансів налагодження або завдань, де користувач вже визначився з підходом.

## Процедура

1. **Problem** (one sentence)
   - Сформулюй основне рішення або бажаний результат в одному стислому реченні.
   - Зосередься на *чому*, а не на *як* — без деталей реалізації, назв інструментів чи конкретних технологій.
   - Тримай його коротким. Якщо потрібне «і», ти описуєш вже два проблеми.

2. **Options** (exactly three)
   - Представ три різні, життєздатні підходи, позначені A, B, C.
   - Кожен варіант містить короткий опис, переваги та недоліки.
   - Варіанти мають бути справді різними стратегіями, а не незначними варіаціями одного підходу.

3. **Recommendation** (one option)
   - Вкажи, який варіант ти рекомендуєш і чому, орієнтуючись на контекст і пріоритети користувача.
   - Будь прямим — це твоє професійне судження, а не ухиляння.

4. **Definition of Done**
   - Перерахуйте конкретні критерії успішності для рекомендованого варіанту.
   - Це мають бути вимірювані, перевіряються результати — а не розпливчасті прагнення.
   - Якщо користувач обирає інший варіант, онови цей розділ відповідно.

5. **Implementation Plan**
   - Конкретні кроки для реалізації рекомендованого варіанту.
   - Додай конкретні команди, інструменти або дії, якщо це доречно.
   - Якщо користувач обирає інший варіант, онови цей розділ відповідно.

## Перевірка

- Відповідь містить рівно одне речення Проблеми.
- Відповідь містить рівно три Варіанти (A, B, C) з перевагами та недоліками для кожного.
- Відповідь містить одну Рекомендацію, яка обирає один варіант з обґрунтуванням.
- Definition of Done та Implementation Plan відповідають обраному варіанту.
- Якщо користувач вибирає інший варіант, Рекомендація, DoD і План впровадження оновлюються.

## Приклад

**User:** “Give me a 1-3-1 for adding retry logic to our API client.”

**Agent:**

**Problem:** API‑виклики до зовнішніх сервісів іноді падають під навантаженням і потребують плавного повтору.

**Options:**

- **Option A: Exponential backoff with jitter in a shared utility.**
  **Pros:** Простий, добре відомий шаблон; одна реалізація, спільна для всіх викликів.
  **Cons:** Немає circuit‑breaking; виклики блокуються під час повторів.

- **Option B: Retry library (tenacity) as a decorator on each endpoint call.**
  **Pros:** Налаштовується per‑endpoint; вбудована підтримка circuit‑breaker‑ів і rate‑limit‑ів.
  **Cons:** Додає залежність; декоратор може ускладнити контроль потоку.

- **Option C: Queue‑based async retry with dead letter handling.**
  **Pros:** Не блокує; невдалі виклики відстежуються і можуть бути повторені; добре масштабується.
  **Cons:** Значне збільшення складності; потрібен бекенд черги; надмірно для сервісів з низьким обсягом.

**Recommendation:** Option A. Обсяг викликів не виправдовує інфраструктуру черги, а спільна утиліта з експоненціальним бекофом + jitter покриває випадки помилок без нових залежностей.

**Definition of Done:**
- Утиліта `api_retry()` обробляє транзиторні HTTP‑помилки (429, 502, 503, 504).
- Повтори до 3 разів з експоненціальним бекофом і jitter.
- Усі зовнішні API‑виклики проходять через утиліту повтору.
- Логується кожна спроба повтору з endpoint та номером спроби.
- Тести охоплюють вичерпання повторів, миттєвий успіх та сценарій “транзит‑потім‑успіх”.

**Implementation Plan:**
1. Створити `utils/api_retry.py` з налаштовуваними max‑retries, base‑delay та списком статус‑кодів, які підлягають повтору.
2. Додати jitter через `random.uniform(0, base_delay)` для запобігання “thundering herd”.
3. Обгорнути існуючі виклики API у `api_client.py` за допомогою утиліти повтору.
4. Додати юніт‑тести, що мокають HTTP‑відповіді для кожного сценарію повтору.
5. Перевірити під навантаженням за допомогою простого стрес‑тесту проти мок‑ендпоінту з випадковими помилками.
