# Honcho Memory for Hermes

Honcho забезпечує AI‑нативне крос‑сесійне моделювання користувача. Він навчається розпізнавати, хто користувач, у різних розмовах, і надає кожному профілю Hermes власну ідентичність співвиконавця, одночасно забезпечуючи єдине уявлення про користувача.
## Коли використовувати

- Налаштування Honcho (хмара або самостійний хостинг)
- Усунення проблем, коли пам'ять не працює / однорангові вузли не синхронізуються
- Створення багатопрофільних налаштувань, коли кожен агент має свій власний Honcho‑вузол
- Налаштування параметрів спостереження, відновлення, діалектичної глибини або частоти запису
- Розуміння того, що роблять 5 інструментів Honcho і коли їх слід застосовувати
- Конфігурація бюджетів контексту та ін’єкція підсумків сесії

## Архітектура

### Вставка базового контексту

Коли Honcho вставляє контекст у системний підказник (у режимах `hybrid` або `context` recall), він формує блок базового контексту у такому порядку:

1. **Резюме сесії** — короткий підсумок поточної сесії (розташовується першим, щоб модель одразу мала безперервність розмови)
2. **Представлення користувача** — накопичена Honcho модель користувача (уподобання, факти, патерни)
3. **Картка AI‑партнера** — ідентифікаційна картка AI‑партнера цього профілю Hermes

Резюме сесії генерується автоматично Honcho на початку кожного кроку (коли існує попередня сесія). Воно дає моделі «теплий старт», не відтворюючи всю історію.

### Вибір холодного / теплого підказника

Honcho автоматично обирає одну з двох стратегій підказки:

| Умова | Стратегія | Що відбувається |
|-----------|----------|--------------|
| Немає попередньої сесії або порожнє представлення | **Cold start** | Легкий вступний підказник; пропускає вставку резюме; спонукає модель дізнатися про користувача |
| Існуюче представлення та/або історія сесії | **Warm start** | Повна вставка базового контексту (резюме → представлення → картка); багатший системний підказник |

Налаштовувати це не потрібно — це автоматично залежить від стану сесії.

### Партнери

Honcho моделює розмови як взаємодії між **партнерами**. Hermes створює двох партнерів на сесію:

- **Партнер‑користувач** (`peerName`) — представляє людину. Honcho будує представлення користувача на основі спостережуваних повідомлень.
- **AI‑партнер** (`aiPeer`) — представляє цей екземпляр Hermes. Кожен профіль отримує свого AI‑партнера, щоб агенти мали незалежні уявлення.

### Спостереження

Кожен партнер має два перемикачі спостереження, які контролюють, що Honcho вивчає:

| Перемикач | Що він робить |
|-----------|---------------|
| `observeMe` | Спостерігаються власні повідомлення партнера (будується самопредставлення) |
| `observeOthers` | Спостерігаються повідомлення інших партнерів (будується крос‑партнерське розуміння) |

За замовчуванням: усі чотири перемикачі **ввімкнені** (повне двостороннє спостереження).

Налаштуй per‑partner у `honcho.json`:

```json
{
  "observation": {
    "user": { "observeMe": true, "observeOthers": true },
    "ai":   { "observeMe": true, "observeOthers": true }
  }
}
```

Або використай скорочені пресети:

| Пресет | Користувач | AI | Випадок використання |
|--------|------------|----|----------------------|
| `"directional"` (за замовчуванням) | me:on, others:on | me:on, others:on | Багатоагентна система, повна пам’ять |
| `"unified"` | me:on, others:off | me:off, others:on | Одиничний агент, моделювання лише користувача |

Налаштування, змінені у [панелі Honcho](https://app.honcho.dev), синхронізуються під час ініціалізації сесії — серверна конфігурація має пріоритет над локальними значеннями за замовчуванням.

### Сесії

Сесії Honcho визначають, куди потрапляють повідомлення та спостереження. Варіанти стратегії:

| Стратегія | Поведінка |
|-----------|----------|
| `per-directory` (за замовчуванням) | Одна сесія на робочий каталог |
| `per-repo` | Одна сесія на корінь git‑репозиторію |
| `per-session` | Нова сесія Honcho при кожному запуску Hermes |
| `global` | Одна сесія для всіх каталогів |

Ручне перевизначення: `hermes honcho map my-project-name`

### Режими recall

Як агент отримує доступ до пам’яті Honcho:

| Режим | Автовставка контексту? | Доступні інструменти? | Випадок використання |
|-------|------------------------|-----------------------|----------------------|
| `hybrid` (за замовчуванням) | Так | Так | Агент вирішує, коли використовувати інструменти, а коли — автоконтекст |
| `context` | Так | Ні (приховано) | Мінімальні токени, без викликів інструментів |
| `tools` | Ні | Так | Агент явно керує всім доступом до пам’яті |
## Три ортогональні регулятори

Діалектична поведінка Honcho контролюється трьома незалежними вимірами. Кожен можна налаштувати, не впливаючи на інші:

### Cadence (when)

Керує **частотою** викликів діалектики та контексту.

| Key | Default | Description |
|-----|---------|-------------|
| `contextCadence` | `1` | Мінімальна кількість ходів між викликами API контексту |
| `dialecticCadence` | `2` | Мінімальна кількість ходів між викликами API діалектики. Рекомендовано 1–5 |
| `injectionFrequency` | `every-turn` | `every-turn` або `first-turn` для базової інжекції контексту |

Вищі значення cadence зменшують частоту запуску діалектичного LLM. `dialecticCadence: 2` означає, що движок працює кожен другий хід. Встановлення `1` запускає його кожен хід.

### Depth (how many)

Керує **кількістю** раундів діалектичного міркування, які Honcho виконує на запит.

| Key | Default | Range | Description |
|-----|---------|-------|-------------|
| `dialecticDepth` | `1` | 1-3 | Кількість раундів діалектичного міркування на запит |
| `dialecticDepthLevels` | -- | array | Додаткові перевизначення рівня для кожного раунду (див. нижче) |

`dialecticDepth: 2` означає, що Honcho виконує два раунди діалектичного синтезу. Перший раунд створює початкову відповідь, другий — уточнює її.

`dialecticDepthLevels` дозволяє задати рівень міркування для кожного раунду окремо:

```json
{
  "dialecticDepth": 3,
  "dialecticDepthLevels": ["low", "medium", "high"]
}
```

Якщо `dialecticDepthLevels` не вказано, раунди використовують **пропорційні рівні**, отримані з `dialecticReasoningLevel` (базового):

| Depth | Pass levels |
|-------|-------------|
| 1 | [base] |
| 2 | [minimal, base] |
| 3 | [minimal, base, low] |

Так раніше проходи залишаються недорогими, а повна глибина використовується лише на фінальному синтезі.

**Depth при старті сесії.** Попереднє прогрівання при старті сесії виконує повністю налаштований `dialecticDepth` у фоні перед ходом 1. Однопрохідне прогрівання на холодному пірі часто повертає обмежений результат — багатопрохідна глибина запускає цикл аудиту/узгодження ще до того, як користувач щось скаже. Хід 1 безпосередньо споживає результат прогрівання; якщо прогрівання не встигло завершитися, хід 1 переходить до синхронного виклику з обмеженим тайм‑аутом.

### Level (how hard)

Керує **інтенсивністю** кожного раунду діалектичного міркування.

| Key | Default | Description |
|-----|---------|-------------|
| `dialecticReasoningLevel` | `low` | `minimal`, `low`, `medium`, `high`, `max` |
| `dialecticDynamic` | `true` | Коли `true`, модель може передати `reasoning_level` у `honcho_reasoning`, щоб перевизначити значення за замовчуванням для окремого виклику. `false` = завжди використовується `dialecticReasoningLevel`, перевизначення моделі ігноруються |

Вищі рівні забезпечують більш багатий синтез, але споживають більше токенів на бекенді Honcho.
## Налаштування кількох профілів

Кожен профіль Hermes отримує свого власного Honcho AI peer, при цьому використовуючи один і той самий робочий простір (контекст користувача). Це означає:

- Усі профілі бачать одне й те саме представлення користувача
- Кожен профіль створює свою власну AI‑особистість та спостереження
- Висновки, написані одним профілем, видимі іншим через спільний робочий простір

### Створити профіль із Honcho peer

```bash
hermes profile create coder --clone
# creates host block hermes.coder, AI peer "coder", inherits config from default
```

Що робить `--clone` для Honcho:
1. Створює блок хоста `hermes.coder` у `honcho.json`
2. Встановлює `aiPeer: "coder"` (назва профілю)
3. Наслідує `workspace`, `peerName`, `writeFrequency`, `recallMode` тощо з типових налаштувань
4. Активно створює peer у Honcho, щоб він існував до першого повідомлення

### Заповнити існуючі профілі

```bash
hermes honcho sync    # creates host blocks for all profiles that don't have one yet
```

### Конфігурація для кожного профілю

Перезаписати будь‑яке налаштування у блоці хоста:

```json
{
  "hosts": {
    "hermes.coder": {
      "aiPeer": "coder",
      "recallMode": "tools",
      "dialecticDepth": 2,
      "observation": {
        "user": { "observeMe": true, "observeOthers": false },
        "ai": { "observeMe": true, "observeOthers": true }
      }
    }
  }
}
```
## Інструменти

Агент має 5 двосторонніх Honcho‑інструментів (прихованих у режимі `context` recall):

| Інструмент | Виклик LLM? | Вартість | Коли використовувати |
|------|-----------|------|----------|
| `honcho_profile` | No | minimal | Швидкий фактичний знімок на початку розмови або для швидкого пошуку імені/ролі/уподобань |
| `honcho_search` | No | low | Отримати конкретні минулі факти для власного роздуму — сирі уривки, без синтезу |
| `honcho_context` | No | low | Повний знімок контексту сесії: резюме, представлення, картка, останні повідомлення |
| `honcho_reasoning` | Yes | medium–high | Питання природньою мовою, синтезоване діалектичним двигуном Honcho |
| `honcho_conclude` | No | minimal | Записати або видалити постійний висновок; передай `peer: "ai"` для самопізнання ШІ |

### `honcho_profile`
Читання або оновлення картки співрозмовника — відібрані ключові факти (ім’я, роль, уподобання, стиль спілкування). Передай `card: [...]` для оновлення; опусти параметр — для читання. Без виклику LLM.

### `honcho_search`
Семантичний пошук по збереженому контексту для конкретного співрозмовника. Повертає сирі уривки, впорядковані за релевантністю, без синтезу. За замовчуванням 800 токенів, максимум 2000. Корисно, коли потрібні конкретні минулі факти для власного роздуму, а не синтезована відповідь.

### `honcho_context`
Повний знімок контексту сесії від Honcho — резюме сесії, представлення співрозмовника, картка співрозмовника та останні повідомлення. Без виклику LLM. Використовуй, коли хочеш бачити все, що Honcho знає про поточну сесію та співрозмовника, в один клік.

### `honcho_reasoning`
Питання природньою мовою, на яке відповідає діалектичний двигун роздумів Honcho (виклик LLM у бекенді Honcho). Вища вартість, вища якість. Передай `reasoning_level` для контролю глибини: `minimal` (швидко/дешево) → `low` → `medium` → `high` → `max` (ретельно). Якщо параметр не вказано, використовується налаштоване за замовчуванням (`low`). Використовуй для синтезованого розуміння патернів користувача, цілей або поточного стану.

### `honcho_conclude`
Запис або видалення постійного висновку про співрозмовника. Передай `conclusion: "..."` для створення. Передай `delete_id: "..."` для видалення (для видалення ПІІ — Honcho самостійно виправляє неправильні висновки з часом, тому видалення потрібне лише для ПІІ). Ти маєш передати рівно один із цих параметрів.

### Двостороннє таргетування співрозмовників

Усі 5 інструменти приймають необов’язковий параметр `peer`:
- `peer: "user"` (за замовчуванням) — працює з користувачем
- `peer: "ai"` — працює з ШІ‑профілем цього профілю
- `peer: "<explicit-id>"` — будь‑який ID співрозмовника у робочому просторі

Приклади:
```
honcho_profile                        # read user's card
honcho_profile peer="ai"              # read AI peer's card
honcho_reasoning query="What does this user care about most?"
honcho_reasoning query="What are my interaction patterns?" peer="ai" reasoning_level="medium"
honcho_conclude conclusion="Prefers terse answers"
honcho_conclude conclusion="I tend to over-explain code" peer="ai"
honcho_conclude delete_id="abc123"    # PII removal
```
## Шаблони використання агента

Рекомендації для Hermes, коли пам’ять Honcho активна.

### При старті розмови

```
1. honcho_profile                  → fast warmup, no LLM cost
2. If context looks thin → honcho_context  (full snapshot, still no LLM)
3. If deep synthesis needed → honcho_reasoning  (LLM call, use sparingly)
```

Не викликай `honcho_reasoning` на кожному кроці. Авто‑вставка вже обробляє оновлення поточного контексту. Використовуй інструмент міркування лише тоді, коли дійсно потрібен синтезований інсайт, якого базовий контекст не дає.

### Коли користувач ділиться чимось, що треба запам’ятати

```
honcho_conclude conclusion="<specific, actionable fact>"
```

Добрі висновки: «Віддає перевагу прикладам коду над прозовими поясненнями», «Працює над проєктом Rust async до квітня 2026 року»
Погані висновки: «Користувач сказав щось про Rust» (занадто загально), «Користувач здається технічним» (вже в репрезентації)

### Коли користувач запитує про минулий контекст / потрібно згадати деталі

```
honcho_search query="<topic>"       → fast, no LLM, good for specific facts
honcho_context                       → full snapshot with summary + messages
honcho_reasoning query="<question>"  → synthesized answer, use when search isn't enough
```

### Коли використовувати `peer: "ai"`

Використовуй AI‑пір для побудови та запиту власних знань агента:
- `honcho_conclude conclusion="I tend to be verbose when explaining architecture" peer="ai"` — самокорекція
- `honcho_reasoning query="How do I typically handle ambiguous requests?" peer="ai"` — самодіагностика
- `honcho_profile peer="ai"` — перегляд власної ідентифікаційної картки

### Коли НЕ викликати інструменти

У режимах `hybrid` та `context` базовий контекст (репрезентація користувача + картка + резюме сесії) авто‑вставляється перед кожним кроком. Не повторюй запит того, що вже було вставлено. Викликай інструменти лише коли:
- Потрібно щось, чого немає в авто‑вставленому контексті
- Користувач явно просить тебе згадати або перевірити пам’ять
- Ти формуєш висновок про щось нове

### Усвідомлення каденції

`honcho_reasoning` на стороні інструменту має ту ж вартість, що і діалектика авто‑вставки. Після явного виклику інструменту каденція авто‑вставки скидається — уникаючи подвійного стягнення за один і той же крок.
## Посилання на конфігурацію

Файл конфігурації: `$HERMES_HOME/honcho.json` (профіль‑локальний) або `~/.honcho/config.json` (глобальний).

### Ключові налаштування

| Ключ | За замовчуванням | Опис |
|-----|------------------|------|
| `apiKey` | -- | API‑ключ ([отримати](https://app.honcho.dev)) |
| `baseUrl` | -- | Базовий URL для самохостингу Honcho |
| `peerName` | -- | Ідентифікатор користувача‑піра |
| `aiPeer` | host key | Ідентифікатор AI‑піра |
| `workspace` | host key | Ідентифікатор спільного робочого простору |
| `recallMode` | `hybrid` | `hybrid`, `context` або `tools` |
| `observation` | all on | Булеві значення `observeMe`/`observeOthers` для кожного піра |
| `writeFrequency` | `async` | `async`, `turn`, `session` або ціле число N |
| `sessionStrategy` | `per-directory` | `per-directory`, `per-repo`, `per-session`, `global` |
| `messageMaxChars` | `25000` | Максимальна кількість символів у повідомленні (розбивається на частини, якщо перевищено) |

### Налаштування діалектики

| Ключ | За замовчуванням | Опис |
|-----|------------------|------|
| `dialecticReasoningLevel` | `low` | `minimal`, `low`, `medium`, `high`, `max` |
| `dialecticDynamic` | `true` | Автоматичне підвищення рівня міркувань залежно від складності запиту. `false` = фіксований рівень |
| `dialecticDepth` | `1` | Кількість діалектичних раундів на запит (1‑3) |
| `dialecticDepthLevels` | -- | Необов’язковий масив рівнів для кожного раунду, напр. `["low", "high"]` |
| `dialecticMaxInputChars` | `10000` | Максимальна кількість символів у вхідних даних діалектичного запиту |

### Бюджет контексту та ін’єкція

| Ключ | За замовчуванням | Опис |
|-----|------------------|------|
| `contextTokens` | uncapped | Максимальна кількість токенів для комбінованої ін’єкції базового контексту (резюме + представлення + картка). Опціональне обмеження — залишити без обмеження або задати ціле число для обмеження розміру ін’єкції. |
| `injectionFrequency` | `every-turn` | `every-turn` або `first-turn` |
| `contextCadence` | `1` | Мінімальна кількість ходів між викликами API контексту |
| `dialecticCadence` | `2` | Мінімальна кількість ходів між діалектичними викликами LLM (рекомендовано 1‑5) |

Бюджет `contextTokens` застосовується під час ін’єкції. Якщо резюме сесії + представлення + картка перевищують бюджет, Honcho спочатку обрізає резюме, потім представлення, зберігаючи картку. Це запобігає вибуху контексту у довгих сесіях.

### Санітизація `memory-context`

Honcho санітує блок `memory-context` перед ін’єкцією, щоб запобігти ін’єкції підказок та некоректному вмісту:

- Видаляє XML/HTML‑теги з користувацьких висновків
- Нормалізує пробіли та керуючі символи
- Обрізає окремі висновки, що перевищують `messageMaxChars`
- Екранує послідовності розділювачів, які можуть порушити структуру системного підказки

Це виправлення вирішує крайові випадки, коли необроблені користувацькі висновки з розміткою або спеціальними символами можуть пошкодити ін’єкційний блок контексту.
## Усунення проблем

### «Honcli не налаштовано»
Запусти `hermes honcho setup`. Переконайся, що `memory.provider: honcho` присутній у `~/.hermes/config.yaml`.

### Пам'ять не зберігається між сесіями
Перевір `hermes honcho status` — переконайся, що `saveMessages: true` і `writeFrequency` не встановлено в `session` (це записує лише при виході).

### Профіль не отримує свій власний peer
Використовуй `--clone` під час створення: `hermes profile create <name> --clone`. Для існуючих профілів: `hermes honcho sync`.

### Зміни спостережень у панелі не відображаються
Конфігурація спостережень синхронізується з сервером під час ініціалізації кожної сесії. Запусти нову сесію після зміни налаштувань у інтерфейсі Honcho.

### Повідомлення обрізані
Повідомлення, що перевищують `messageMaxChars` (за замовчуванням 25 k), автоматично розбиваються маркерами `[continued]`. Якщо це трапляється часто, перевір, чи результати інструментів або вміст навичок не збільшують розмір повідомлення.

### Введення контексту занадто велике
Якщо бачиш попередження про перевищення бюджету контексту, зменши `contextTokens` або знизь `dialecticDepth`. Спочатку скорочується підсумок сесії, коли бюджет обмежений.

### Відсутній підсумок сесії
Підсумок сесії вимагає принаймні одного попереднього ходу в поточній сесії Honcho. При холодному запуску (нова сесія, без історії) підсумок пропускається, і Honcho використовує стратегію підказки холодного старту.

| Command | Description |
|---------|-------------|
| `hermes honcho setup` | Інтерактивний майстер налаштування (cloud/local, identity, observation, recall, sessions) |

| `hermes honcho enable` | Увімкнути Honcho для активного профілю (створює блок хоста, якщо потрібно) |

| `hermes honcho peers` | Показати ідентичності peer у всіх профілях |
| `hermes honcho mode` | Показати або встановити режим recall (`hybrid`, `context`, `tools`) |
| `hermes honcho tokens` | Показати або встановити бюджети токенів (`--context <N>`, `--dialectic <N>`) |
| `hermes honcho sessions` | Перелік відомих відповідностей директорія‑до‑імені сесії |
| `hermes honcho map <name>` | Прив’язати поточну робочу директорію до імені сесії Honcho |
| `hermes honcho identity` | Задати ідентичність AI‑peer або показати обидва представлення peer |
| `hermes honcho sync` | Створити блоки хостів для всіх профілів Hermes, які ще їх не мають |
