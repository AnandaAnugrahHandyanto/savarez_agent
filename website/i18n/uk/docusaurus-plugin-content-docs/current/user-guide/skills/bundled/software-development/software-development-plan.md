# Режим Плану

Використовуй цю навичку, коли користувач хоче отримати план замість виконання.

## Основна поведінка

На цьому кроці ти лише плануєш.

- Не реалізовуй код.
- Не редагуй файли проєкту, окрім markdown‑файлу плану.
- Не виконуй мутуючі команди терміналу, commit, push або зовнішні дії.
- За потреби можеш переглядати репозиторій або інший контекст лише за допомогою команд/інструментів з правом лише читання.
- Твій результат — markdown‑план, збережений у активному робочому просторі під `.hermes/plans/`.

## Вимоги до виведення

Напиши markdown‑план, який є конкретним і дієвим.

Включай, коли це доречно:
- Мета
- Поточний контекст / припущення
- Запропонований підхід
- Покроковий план
- Файли, які, ймовірно, зміняться
- Тести / валідація
- Ризики, компроміси та відкриті питання

Якщо завдання пов’язане з кодом, вкажи точні шляхи файлів, ймовірні цілі тестів та кроки перевірки.

## Місце збереження

Збережи план за допомогою `write_file` у:
- `.hermes/plans/YYYY-MM-DD_HHMMSS-<slug>.md`

Розглядай це як шлях відносно активної робочої директорії / бекенд‑простору. Інструменти Hermes орієнтовані на бекенд, тому використання відносного шляху зберігає план у робочому просторі локально, у Docker, SSH, Modal та Daytona бекендах.

Якщо середовище виконання надає конкретний цільовий шлях, використай саме його.
Якщо ні — створи розумне ім’я файлу з тайм‑стампом у `.hermes/plans/`.

## Стиль взаємодії

- Якщо запит достатньо чіткий, одразу напиши план.
- Якщо до `/plan` не додається явна інструкція, визнач задачу з поточного контексту розмови.
- Якщо запит справді недостатньо уточнений, постав коротке уточнююче питання замість вгадування.
- Після збереження плану коротко відповідай, що саме ти запланував, і вкажи шлях до збереженого файлу.
