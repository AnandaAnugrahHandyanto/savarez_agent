## Архітектура конвеєра

Кожен режим слідує одному й тому ж 6‑етапному конвеєру:

```
INPUT → ANALYZE → SCENE_FN → TONEMAP → SHADE → ENCODE
```

1. **INPUT** — Завантаження/декодування вихідного матеріалу (відеокадри, аудіо‑зразки, зображення або нічого)
2. **ANALYZE** — Витяг ознак з кожного кадру (аудіо‑полоски, яскравість/краї відео, вектори руху)
3. **SCENE_FN** — Функція сцени рендерить у піксельне полотно (`uint8 H,W,3`). Компонуватиме кілька сіток персонажів за допомогою `_render_vf()` + режимів змішування пікселів. Див. `references/composition.md`
4. **TONEMAP** — Нормалізація яскравості на основі процентилів. Див. `references/composition.md` § Adaptive Tonemap
5. **SHADE** — Пост‑обробка за допомогою `ShaderChain` + `FeedbackBuffer`. Див. `references/shaders.md`
6. **ENCODE** — Передає необроблені RGB‑кадри у ffmpeg для кодування H.264/GIF

## Критичні нотатки щодо реалізації

### Яскравість — використовуйте `tonemap()`, а не лінійні множники

Це головна візуальна проблема. ASCII на чорному фоні природно темний. **Ніколи не використовуйте множники `canvas * N`** — вони обрізають яскраві ділянки. Використовуйте адаптивний `tonemap()`:

```python
def tonemap(canvas, gamma=0.75):
    f = canvas.astype(np.float32)
    lo, hi = np.percentile(f[::4, ::4], [1, 99.5])
    if hi - lo < 10: hi = lo + 10
    f = np.clip((f - lo) / (hi - lo), 0, 1) ** gamma
    return (f * 255).astype(np.uint8)
```

Конвеєр: `scene_fn() → tonemap() → FeedbackBuffer → ShaderChain → ffmpeg`

Гамма для кожної сцени: за замовчуванням 0.75, solarize 0.55, posterize 0.50, яскраві сцени 0.85. Використовуйте змішування `screen` (не `overlay`) для темних шарів.

### Висота клітини шрифту

macOS Pillow: `textbbox()` повертає неправильну висоту. Використовуйте `font.getmetrics()`: `cell_height = ascent + descent`. Див. `references/troubleshooting.md`.

### Застрявання pipe ffmpeg

Ніколи не використовуйте `stderr=subprocess.PIPE` з довготривалим ffmpeg — буфер заповнюється на 64 KB і виникає зависання. Перенаправляйте в файл. Див. `references/troubleshooting.md`.

### Сумісність шрифтів

Не всі Unicode‑символи відображаються у всіх шрифтах. Перевіряйте палітри під час ініціалізації — рендерьте кожен символ, перевіряйте, чи не отримано порожній результат. Див. `references/troubleshooting.md`.

### Архітектура per‑clip

Для сегментованих відео (цитати, сцени, глави) рендерьте кожен як окремий файл кліпу для паралельного рендерингу та вибіркового повторного рендерингу. Див. `references/scenes.md`.

- **Forced Connections** — коли користувач хоче натхнення з іншої галузі («зроби це органічним», «промисловий естетика»)
- **Conceptual Blending** — коли користувач називає два об’єкти для поєднання («океан зустрічає музику», «космос + каліграфія»)
- **Oblique Strategies** — коли користувач максимально відкритий («здивуй мене», «щось, чого я ще не бачив»)

### Forced Connections
1. Вибери домен, не пов’язаний з візуальною метою (метеорологічні системи, мікробіологія, архітектура, гідродинаміка, текстильне плетіння)
2. Перелічити його ключові візуальні/структурні елементи (ерозія → поступове розкриття; мітоз → поділ‑дуплікація; плетіння → взаємозв’язані візерунки)
3. Відобрази ці елементи за допомогою ASCII‑символів та анімаційних патернів
4. Синтезуй — як виглядає «ерозія» або «кристалізація» у сітці символів?

### Conceptual Blending
1. Назви два різні візуальні/концептуальні простори (наприклад, хвилі океану + нотний лист)
2. Визнач відповідності (вершини = високі ноти, впадини = паузи, пінка = стаккато)
3. Поєднуй вибірково — залишай найцікавіші відповідності, відкидай нав’язані
4. Розвивай нові властивості, які існують лише у цьому поєднанні

### Oblique Strategies
1. Випади одну: «Шануй свою помилку як приховану інтенцію» / «Використай стару ідею» / «Що б зробив твій найкращий друг?» / «Підкресли недоліки» / «Переверни все догори дном» / «Тільки частину, а не ціле» / «Зворотний хід»
2. Інтерпретуй директиву щодо поточної задачі ASCII‑анімації
3. Застосуй латеральний інсайт до візуального дизайну перед написанням коду
