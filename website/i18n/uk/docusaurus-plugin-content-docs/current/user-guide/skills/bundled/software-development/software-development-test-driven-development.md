# Розробка, орієнтована на тестування (TDD)
## Огляд

Пиши тест спочатку. Спостерігай, як він падає. Пиши мінімальний код, щоб пройти тест.

**Основний принцип:** Якщо ти не бачив, як тест падає, ти не знаєш, чи він тестує правильне.

**Порушення букви правил — це порушення духу правил.**
## Коли використовувати

**Завжди:**
- Нові функції
- Виправлення помилок
- Рефакторинг
- Зміни поведінки

**Винятки (спершу запитай у користувача):**
- Прототипи, які викидаються
- Згенерований код
- Файли конфігурації

Думаєш «пропустити TDD лише цього разу»? Стоп. Це раціоналізація.
## Залізний закон

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

Писати код перед тестом? Видалити його. Почати спочатку.

**Без виключень:**
- Не залишай його як «reference»
- Не «адаптуй» його під час написання тестів
- Не дивись на нього
- Видаляти — це видаляти

Реалізовуй чисто з тестів. Точка.
## Цикл Red‑Green‑Refactor

### RED — Напиши тест, який падає

Напиши один мінімальний тест, що показує, що має статися.

**Добрий тест:**
```python
def test_retries_failed_operations_3_times():
    attempts = 0
    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise Exception('fail')
        return 'success'

    result = retry_operation(operation)

    assert result == 'success'
    assert attempts == 3
```
Чітка назва, тестує реальну поведінку, один випадок.

**Поганий тест:**
```python
def test_retry_works():
    mock = MagicMock()
    mock.side_effect = [Exception(), Exception(), 'success']
    result = retry_operation(mock)
    assert result == 'success'  # What about retry count? Timing?
```
Неоднозначна назва, тестує мок, а не реальний код.

**Вимоги:**
- Одна поведінка на тест
- Чітка описова назва («and» у назві? Розділи її)
- Реальний код, а не моки (хіба що неминуче)
- Назва описує поведінку, а не реалізацію

### Verify RED — Перевір, що тест падає

**ОБОВʼЯЗКОВО. Ніколи не пропускай.**

```bash
# Use terminal tool to run the specific test
pytest tests/test_feature.py::test_specific_behavior -v
```

Перевір:
- Тест падає (не помилки через типографічні помилки)
- Повідомлення про падіння відповідає очікуванням
- Падає тому, що функціональність відсутня

**Тест проходить одразу?** Ти тестуєш вже існуючу поведінку. Виправ тест.

**Тест викидає помилки?** Виправ помилку, запусти знову, доки тест не падатиме правильно.

### GREEN — Мінімальний код

Напиши найпростіший код, щоб тест пройшов. Ніщо більше.

**Добре:**
```python
def add(a, b):
    return a + b  # Nothing extra
```

**Погано:**
```python
def add(a, b):
    result = a + b
    logging.info(f"Adding {a} + {b} = {result}")  # Extra!
    return result
```

Не додавай функціональність, не рефактори інший код і не «покращуй» поза межами тесту.

**Читання дозволено у GREEN:**
- Жорстко закодовані значення повернення
- Копіювання‑вставка
- Дублювання коду
- Пропуск крайових випадків

Виправимо це у REFACTOR.

### Verify GREEN — Перевір, що тест проходить

**ОБОВʼЯЗКОВО.**

```bash
# Run the specific test
pytest tests/test_feature.py::test_specific_behavior -v

# Then run ALL tests to check for regressions
pytest tests/ -q
```

Перевір:
- Тест проходить
- Інші тести все ще проходять
- Вивід чистий (без помилок, попереджень)

**Тест падає?** Виправ код, а не тест.

**Інші тести падають?** Виправ регресії негайно.

### REFACTOR — Очисти код

Після успішного GREEN:
- Видали дублювання
- Покращи назви
- Витягни допоміжні функції
- Спрощуй вирази

Тримай тести GREEN протягом усього процесу. Не додавай нову поведінку.

**Якщо тести падають під час рефакторингу:** Відміни зміни одразу. Роби менші кроки.

### Повтори

Наступний тест, який падає, для наступної поведінки. По одному циклу за раз.
## Чому порядок має значення

**«Я напишу тести після, щоб перевірити, що все працює»**

Тести, написані після коду, проходять одразу. Одразу пройти — це нічого не доводить:
- Можна тестувати не те, що треба
- Можна тестувати реалізацію, а не поведінку
- Можна пропустити крайові випадки, які ти забув
- Ти ніколи не бачив, як тест ловить помилку

Тестування спочатку змушує тебе побачити, як тест падає, доводячи, що він дійсно щось тестує.

**«Я вже вручну протестував усі крайові випадки»**

Ручне тестування — це випадкове. Ти думаєш, що протестував усе, але:
- Немає запису про те, що саме ти тестував
- Не можна повторно запустити, коли код змінюється
- Легко забути випадки під тиском
- «Працювало, коли я спробував» ≠ всеохоплююче

Автоматизовані тести систематичні. Вони виконуються однаково щоразу.

**«Видаляти X годин роботи — марнотратство»**

Помилка «заплатити за втрачене». Час уже витрачено. Тепер твій вибір:
- Видалити і переписати за допомогою TDD (висока впевненість)
- Залишити і додати тести після (низька впевненість, ймовірні баги)

«Витрата» — це збереження коду, якому ти не можеш довіряти.

**«TDD догматичний, бути практичним означає адаптуватися»**

TDD — практичний:
- Знаходить баги до коміту (швидше, ніж налагодження після)
- Запобігає регресіям (тести одразу ловлять поломки)
- Документує поведінку (тести показують, як користуватись кодом)
- Дозволяє рефакторинг (змінюй вільно, тести ловлять поломки)

«Практичні» скорочення = налагодження в продакшені = повільніше.

**«Тести після досягають тих самих цілей — це дух, а не ритуал»**

Ні. Тести після відповідають на питання «Що це робить?». Тести спочатку відповідають на питання «Що це має робити?».

Тести після упереджені твоєю реалізацією. Ти тестуєш те, що створив, а не те, що потрібно. Тести спочатку змушують виявляти крайові випадки ще до реалізації.
## Common Rationalizations

| Excuse | Reality |
|--------|---------|
| «Занадто простий, щоб тестувати» | Навіть простий код може зламатися. Тест займає 30 секунд. |
| «Потім протестую» | Проходження тестів одразу нічого не доводить. |
| «Тести після досягають тієї ж мети» | Тести‑після = «Що це робить?». Тести‑спочатку = «Що це має робити?». |
| «Вже протестовано вручну» | Ад‑хок ≠ систематичний. Немає запису, не можна повторити. |
| «Видаляти X годин – марно» | Помилка «заплативши за глибину». Збереження неперевіреного коду – технічний борг. |
| «Залишити як довідку, написати тести потім» | Ти все одно адаптуєш його. Це тестування після. Видалити – означає видалити. |
| «Спочатку треба дослідити» | Добре. Викинь дослідження і починай з TDD. |
| «Тест важкий = дизайн неясний» | Слухай тест. Важко протестувати – важко використовувати. |
| «TDD сповільнить мене» | TDD швидше, ніж налагодження. Прагматичний підхід = тестування спочатку. |
| «Ручне тестування швидше» | Ручне тестування не доводить крайові випадки. Ти будеш повторно тестувати кожну зміну. |
| «У існуючому коді немає тестів» | Ти його покращуєш. Додай тести для коду, який змінюєш. |
## Червоні прапорці — ЗУПИНИ та починай спочатку

Якщо ти помітиш, що робиш будь‑яке з наведеного, видали код і починай заново за допомогою TDD:

- Код до тесту
- Тест після реалізації
- Тест проходить одразу з першого запуску
- Не можеш пояснити, чому тест провалився
- Тести додані «пізніше»
- Виправдання «лише цього разу»
- «Я вже вручну протестував це»
- «Тести після досягають тієї ж мети»
- «Залишити як довідку» або «адаптувати існуючий код»
- «Вже витрачено X годин, видаляти — марнотратство»
- «TDD — догматизм, я прагну практичності»
- «Це інше, бо…»

**Все це означає: видали код. Починай заново за допомогою TDD.**
## Контрольний список перевірки

Перед позначенням роботи як завершеної:

- [ ] Кожна нова функція/метод має тест
- [ ] Спостерігав, як кожен тест не проходив перед реалізацією
- [ ] Кожен тест не проходив з очікуваної причини (відсутня функція, а не помилка друку)
- [ ] Написав мінімальний код, щоб пройти кожен тест
- [ ] Усі тести проходять
- [ ] Вивід бездоганний (без помилок і попереджень)
- [ ] Тести використовують реальний код (моки лише у випадку крайньої необхідності)
- [ ] Покрито граничні випадки та помилки

Не можеш відмітити всі галочки? Ти пропустив TDD. Почни спочатку.
## Коли застряг

| Проблема | Рішення |
|---------|----------|
| Не знаєш, як тестувати | Напиши потрібний API. Спочатку напиши асерцію. Запитай користувача. |
| Тест занадто складний | Дизайн занадто складний. Спрощуй інтерфейс. |
| Потрібно мокати все | Код надто зв’язаний. Використовуй ін’єкцію залежностей. |
| Налаштування тесту величезне | Витягни допоміжні функції. Все ще складно? Спрощуй дизайн. |
## Інтеграція Hermes Agent

### Запуск тестів

Використовуй інструмент `terminal` для запуску тестів на кожному кроці:

```python
# RED — verify failure
terminal("pytest tests/test_feature.py::test_name -v")

# GREEN — verify pass
terminal("pytest tests/test_feature.py::test_name -v")

# Full suite — verify no regressions
terminal("pytest tests/ -q")
```

### З delegate_task

При розсиланні підагентів для реалізації забезпеч TDD у цілі:

```python
delegate_task(
    goal="Implement [feature] using strict TDD",
    context="""
    Follow test-driven-development skill:
    1. Write failing test FIRST
    2. Run test to verify it fails
    3. Write minimal code to pass
    4. Run test to verify it passes
    5. Refactor if needed
    6. Commit

    Project test command: pytest tests/ -q
    Project structure: [describe relevant files]
    """,
    toolsets=['terminal', 'file']
)
```

### З systematic-debugging

Знайдено баг? Напиши тест, який його відтворює. Дотримуйся циклу TDD. Тест доводить виправлення і запобігає регресії.

Ніколи не виправляй баги без тесту.
## Тестування антипатернів

- **Тестування поведінки моків замість реальної поведінки** — моки мають перевіряти взаємодії, а не замінювати систему, що тестується
- **Тестування деталей реалізації** — слід перевіряти поведінку/результати, а не внутрішні виклики методів
- **Тільки «щасливий шлях»** — обов’язково тестуй граничні випадки, помилки та межі
- **Крихкі тести** — тести мають перевіряти поведінку, а не структуру; рефакторинг не повинен їх ламати
## Фінальне правило

```
Production code → test exists and failed first
Otherwise → not TDD
```
