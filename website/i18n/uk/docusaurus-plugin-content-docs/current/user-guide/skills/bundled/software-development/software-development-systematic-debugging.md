# Систематичне налагодження
## Огляд

Випадкові виправлення марнують час і створюють нові помилки. Швидкі патчі приховують глибинні проблеми.

**Основний принцип:** **ЗАВЖДИ** знаходити кореневу причину перед тим, як намагатися виправляти. Виправлення симптомів — це провал.

**Порушення букви цього процесу — порушення духу налагодження.**
## Залізний закон

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

Якщо ти не завершив фазу 1, ти не можеш пропонувати виправлення.
## Коли використовувати

Використовуй для БУДЬ‑якої технічної проблеми:
- Помилки тестів
- Баги в продакшені
- Неочікувана поведінка
- Проблеми продуктивності
- Помилки збірки
- Проблеми інтеграції

**Використовуй це ОСОБЛИВО, коли:**
- Під тиском часу (надзвичайні ситуації змушують гадати)
- «Ще одне швидке виправлення» здається очевидним
- Ти вже спробував кілька виправлень
- Попереднє виправлення не спрацювало
- Ти не повністю розумієш проблему

**Не пропускай, якщо:**
- Проблема здається простою (прості баги теж мають глибинні причини)
- Ти поспішаєш (поспішність гарантує переделку)
- Хтось вимагає виправити ЗАРАЗ (системний підхід швидший за хаотичний)

## Phase 1: Розслідування кореневої причини

**BEFORE attempting ANY fix:**

### 1. Читай повідомлення про помилки уважно

- Не пропускай помилки чи попередження
- Вони часто містять точне рішення
- Читай стек‑трейси повністю
- Зафіксуй номери рядків, шляхи до файлів, коди помилок

**Action:** Використай `read_file` на відповідних вихідних файлах. Використай `search_files`, щоб знайти рядок помилки в кодовій базі.

### 2. Відтворюй стабільно

- Чи можеш ти викликати її надійно?
- Які точні кроки?
- Чи відбувається це щоразу?
- Якщо не відтворюється → збирай більше даних, не вгадуй

**Action:** Використай інструмент `terminal`, щоб запустити провальний тест або викликати баг:

```bash
# Run specific failing test
pytest tests/test_module.py::test_name -v

# Run with verbose output
pytest tests/test_module.py -v --tb=long
```

### 3. Перевір останні зміни

- Що змінилося і могло спричинити це?
- Git diff, останні коміти
- Нові залежності, зміни конфігурації

**Action:**

```bash
# Recent commits
git log --oneline -10

# Uncommitted changes
git diff

# Changes in specific file
git log -p --follow src/problematic_file.py | head -100
```

### 4. Збирай докази в системах з кількома компонентами

**WHEN system has multiple components (API → service → database, CI → build → deploy):**

**BEFORE proposing fixes, add diagnostic instrumentation:**

Для КОЖНОЇ межі компоненту:
- Логуй, які дані входять у компонент
- Логуй, які дані виходять з компоненту
- Перевір propagation середовища/конфігурації
- Перевір стан на кожному шарі

Запусти один раз, щоб зібрати докази, які показують, ДЕ відбувається збій.
THEN аналізуй докази, щоб визначити, який компонент не працює.
THEN розслідуй саме цей компонент.

### 5. Трасуй потік даних

**WHEN error is deep in the call stack:**

- Звідки походить неправильне значення?
- Яка функція викликала цю функцію з неправильним значенням?
- Продовжуй трасувати вгору, доки не знайдеш джерело
- Виправляй у джерелі, а не на симптомі

**Action:** Використай `search_files`, щоб трасувати посилання:

```python
# Find where the function is called
search_files("function_name(", path="src/", file_glob="*.py")

# Find where the variable is set
search_files("variable_name\\s*=", path="src/", file_glob="*.py")
```

### Phase 1 Completion Checklist

- [ ] Повідомлення про помилки повністю прочитані та зрозумілі
- [ ] Проблему відтворено стабільно
- [ ] Останні зміни ідентифіковано та переглянуто
- [ ] Зібрано докази (логи, стан, потік даних)
- [ ] Проблему ізольовано до конкретного компоненту/коду
- [ ] Сформовано гіпотезу про кореневу причину

**STOP:** Не переходь до Phase 2, доки не зрозумієш, ЧОМУ це відбувається.
## Phase 2: Аналіз шаблону

**Знайди шаблон перед виправленням:**

### 1. Знайди працюючі приклади

- Знайди схожий працюючий код у тому ж кодовому базі
- Що працює і схоже на те, що зламано?

**Дія:** Використай `search_files`, щоб знайти подібні шаблони:

```python
search_files("similar_pattern", path="src/", file_glob="*.py")
```

### 2. Порівняй з референсами

- Якщо впроваджуєш шаблон, прочитай референсну реалізацію **повністю**
- Не переглядай поверхнево — читай кожен рядок
- Повністю зрозумій шаблон перед застосуванням

### 3. Визнач різниці

- Що відрізняється між працюючим і зламаним кодом?
- Переліч усі відмінності, навіть найменші
- Не припускай, що «це не має значення»

### 4. Розбери залежності

- Які інші компоненти потрібні?
- Які налаштування, конфігурації, середовище?
- Які припущення робить код?
## Phase 3: Гіпотеза та тестування

**Науковий метод:**

### 1. Сформулюй одну гіпотезу

- Чітко сформулюй: «Я вважаю, що X є причиною, бо Y»
- Запиши її
- Будь конкретним, а не розпливчастим

### 2. Тестуй мінімально

- Зроби найменшу можливу зміну для перевірки гіпотези
- Одна змінна за раз
- Не виправляй одразу кілька речей

### 3. Перевір перед продовженням

- Чи спрацювало? → Phase 4
- Не спрацювало? → Сформулюй нову гіпотезу
- НЕ додавай інші виправлення поверх

### 4. Коли не знаєш

- Скажи «I don't understand X»
- Не вдавай, що розумієш
- Попроси користувача про допомогу
- Досліджуй далі

## Фаза 4: Реалізація

**Виправляй кореневу причину, а не симптом:**

### 1. Створи тест, який провалюється

- Найпростіше можливе відтворення
- Автоматизований тест, якщо можливо
- ПОВИНЕН бути до виправлення
- Використовуй навичку `test-driven-development`

### 2. Реалізуй одиничне виправлення

- Виріши виявлену кореневу причину
- Одна зміна за раз
- Без покращень типу «поки я тут»
- Без згрупованого рефакторингу

### 3. Перевір виправлення

```bash
# Run the specific regression test
pytest tests/test_module.py::test_regression -v

# Run full suite — no regressions
pytest tests/ -q
```

### 4. Якщо виправлення не працює — Правило трьох

- **STOP.**
- Підрахуй: скільки виправлень ти спробував?
- Якщо < 3: Повернись до Фази 1, проаналізуй заново з новою інформацією
- **Якщо ≥ 3: STOP і постав під сумнів архітектуру (крок 5 нижче)**
- НЕ намагайся застосувати виправлення #4 без обговорення архітектури

### 5. Якщо 3+ виправлення не вдалося: Постав під сумнів архітектуру

**Патерн, що вказує на проблеми архітектури:**
- Кожне виправлення виявляє новий спільний стан/зв’язок в іншому місці
- Виправлення вимагають «масивного рефакторингу» для реалізації
- Кожне виправлення створює нові симптоми в інших частинах

**STOP і постав під сумнів фундаментальні питання:**
- Чи є цей патерн принципово правильним?
- Чи «тримаємося» ми його лише через інерцію?
- Чи варто рефакторити архітектуру, а не продовжувати виправляти симптоми?

**Обговори це з користувачем перед тим, як робити подальші виправлення.**

Це НЕ провальна гіпотеза — це неправильна архітектура.

## Червоні прапорці — ЗУПИНИСЬ і Дотримуйся процесу

Якщо ти помічаєш, що думаєш:
- «Швидке виправлення на даний момент, розслідуватиму пізніше»
- «Просто спробую змінити X і подивитися, чи працює»
- «Додати кілька змін, запустити тести»
- «Пропустити тест, я перевірю вручну»
- «Ймовірно це X, давай виправимо це»
- «Я не повністю розумію, але це може спрацювати»
- «Шаблон каже X, але я адаптую його інакше»
- «Ось головні проблеми: [перелік виправлень без розслідування]»
- Пропонування рішень до відстеження потоку даних
- **«Ще одна спроба виправлення» (коли вже спробовано 2 + рази)**
- **Кожне виправлення виявляє нову проблему в іншому місці**

**Все це означає: ЗУПИНИСЬ. Повернись до Фази 1.**

**Якщо не вдалося 3 + виправлення:** Став під сумнів архітектуру (крок 5 Фази 4).
## Загальні виправдання

| Виправдання | Реальність |
|--------|---------|
| «Проблема проста, процес не потрібен» | Навіть прості проблеми мають кореневі причини. Процес швидкий навіть для простих багів. |
| «Невідкладно, немає часу на процес» | Систематичне налагодження ШВИДШЕ, ніж вгадування та перевірка. |
| «Спочатку спробуй це, а потім розслідуй» | Перше виправлення задає шаблон. Робі й його правильно з самого початку. |
| «Напишу тест після підтвердження, що виправлення працює» | Неперевірені виправлення не тримаються. Тест спочатку доводить їх працездатність. |
| «Одночасно кілька виправлень економить час» | Не можеш визначити, що саме спрацювало. Це створює нові баги. |
| «Посилання занадто довге, я адаптую шаблон» | Часткове розуміння гарантує баги. Прочитай його повністю. |
| «Бачу проблему, дай мені виправити» | Бачити симптоми ≠ розуміти кореневу причину. |
| «Ще одна спроба виправлення» (після 2+ невдач) | 3+ невдачі = архітектурна проблема. Став під сумнів шаблон, а не виправляй знову. |
## Швидка довідка

| Фаза | Ключові дії | Критерії успіху |
|------|------------|-----------------|
| **1. Root Cause** | Читати помилки, відтворювати їх, перевіряти зміни, збирати докази, трасувати потік даних | Розуміти **ЩО** і **ЧОМУ** |
| **2. Pattern** | Шукати робочі приклади, порівнювати, визначати відмінності | Знати, що відрізняється |
| **3. Hypothesis** | Формувати теорію, тестувати мінімально, змінюючи по одній змінній | Підтверджена або нова гіпотеза |
| **4. Implementation** | Створити регресійний тест, виправити кореневу причину, перевірити | Баг виправлено, всі тести проходять |
## Інтеграція Hermes Agent

### Інструменти розслідування

Використовуй ці інструменти Hermes під час Фази 1:

- **`search_files`** — знаходити рядки помилок, відстежувати виклики функцій, виявляти шаблони
- **`read_file`** — читати вихідний код з номерами рядків для точного аналізу
- **`terminal`** — запускати тести, перевіряти історію git, відтворювати баги
- **`web_search`/`web_extract`** — досліджувати повідомлення про помилки, документацію бібліотек

### З `delegate_task`

Для складного багатокомпонентного налагодження розподіли підагенти розслідування:

```python
delegate_task(
    goal="Investigate why [specific test/behavior] fails",
    context="""
    Follow systematic-debugging skill:
    1. Read the error message carefully
    2. Reproduce the issue
    3. Trace the data flow to find root cause
    4. Report findings — do NOT fix yet

    Error: [paste full error]
    File: [path to failing code]
    Test command: [exact command]
    """,
    toolsets=['terminal', 'file']
)
```

### З `test-driven-development`

При виправленні багів:
1. Напиши тест, який відтворює баг (RED)
2. Систематично налагоджуй, щоб знайти причину
3. Виправ причину (GREEN)
4. Тест доводить виправлення і запобігає регресії
## Реальний вплив у практиці

З сеансів відладки:
- Систематичний підхід: 15‑30 хвилин на виправлення
- Підхід випадкових виправлень: 2‑3 години безцільного кидання
- Рівень успішних виправлень з першої спроби: 95 % проти 40 %
- Нових помилок, що з’являються: майже нульовий проти звичних випадків

**Без скорочень. Без здогадок. Систематичний підхід завжди переможе.**
