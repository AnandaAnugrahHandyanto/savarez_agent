# Конвеєр виробництва p5.js
## Коли використовувати

## Що всередині

Конвеєр виробництва для інтерактивного та генеративного візуального мистецтва з використанням p5.js. Створює браузерні скетчі, генеративне мистецтво, візуалізації даних, інтерактивні досвіди, 3D‑сцени, аудіореактивні візуали та моушн‑графіку — експортуються як HTML, PNG, GIF, MP4 або SVG. Охоплює: 2D/3D рендеринг, шум і системи часток, поля потоків, шейдери (GLSL), маніпуляція пікселями, кінетична типографіка, WebGL‑сцени, аудіо‑аналіз, взаємодія миші/клавіатури та безголовий експорт у високій роздільності.
## Creative Standard

Це візуальне мистецтво, що відображається в браузері. Полотно — середовище; алгоритм — пензель.

**Перш ніж написати жодного рядка коду**, сформулюй креативну концепцію. Що передає цей твір? Що змушує глядача зупинитися і не прокручувати далі? Чим це відрізняється від прикладу‑уроку коду? Запит користувача — точка старту, інтерпретуй його з творчою амбіцією.

**Якість першого рендеру — непідлязна компромісу.** Результат має одразу вразити візуально. Якщо він виглядає як вправа з p5.js, стандартна конфігурація або «AI‑згенерований креативний код», це помилка. Перепрацюй перед випуском.

**Виходь за межі референсного словника.** Шумові функції, системи частинок, колірні палітри та шейдерні ефекти в прикладах — лише стартовий словник. Для кожного проєкту комбінуй, накладай і вигадуй нове. Каталог — це палітра фарб, а ти малюєш картину.

**Будь проактивно креативним.** Якщо користувач просить «систему частинок», надай систему частинок з виникаючою стайковою поведінкою, хвостовими привидами, глибинним туманом зі зміною палітри та фоном шумового поля, що «дихає». Додай хоча б одну візуальну деталь, яку користувач не запитував, але оцінить.

**Щільно, багатошарово, продумано.** Кожен кадр має винагороджувати перегляд. Ніколи безплідні біло‑білі фони. Завжди композиційна ієрархія. Завжди навмисний колір. Завжди мікродеталь, що з’являється лише при уважному розгляді.

**Єдина естетика важливіша за кількість функцій.** Усі елементи повинні слугувати єдиній візуальній мові — спільна колірна температура, послідовний словник товщини штрихів, гармонійна швидкість руху. Ескіз з десятьма несумісними ефектами гірший за той, у якому три ефекти гармонійно поєднані.

### Примітка щодо версії

**p5.js 1.x** (1.11.3) — це типове значення — стабільне, добре задокументоване, найширша сумісність бібліотек. Використовуй його, якщо проєкт не потребує функцій 2.x.

**p5.js 2.x** (2.2+) додає: `async setup()` замість `preload()`, режими кольорів OKLCH/OKLAB, `splineVertex()`, API шейдеру `.modify()`, змінні шрифти, `textToContours()`, події вказівника. Потрібно для p5.brush. Дивись `references/core-api.md` § p5.js 2.0.
## Конвеєр

Кожен проєкт слідує одному й тому ж 6‑етапному шляху:

```
CONCEPT → DESIGN → CODE → PREVIEW → EXPORT → VERIFY
```

1. **CONCEPT** — Сформулюй творче бачення: настрій, колірний світ, словник руху, що робить його унікальним
2. **DESIGN** — Вибери режим, розмір полотна, модель взаємодії, колірну систему, формат експорту. Перетвори концепт у технічні рішення
3. **CODE** — Напиши один HTML‑файл з вбудованим p5.js. Структура: globals → `preload()` → `setup()` → `draw()` → helpers → classes → event handlers
4. **PREVIEW** — Відкрий у браузері, перевір якість зображення. Протести на цільовій роздільності. Перевір продуктивність
5. **EXPORT** — Зафіксуй результат: `saveCanvas()` для PNG, `saveGif()` для GIF, `saveFrames()` + ffmpeg для MP4, Puppeteer для безголового пакетного режиму
6. **VERIFY** — Чи відповідає результат концепту? Чи виглядає він вражаюче у передбачуваному розмірі дисплея? Чи варто його оформити у рамку?

### Правила варіації проєкту

Ніколи не використовуйте типові налаштування. Для кожного проєкту:
- **Кастомна колірна палітра** — не використовуйте «чистий» `fill(255, 0, 0)`. Завжди створюйте палітру з 3‑7 кольорів.
- **Кастомний словник товщини штриха** — тонкі акценти (0.5), середня структура (1‑2), жирний акцент (3‑5).
- **Обробка фону** — не використовуйте простий `background(0)` або `background(255)`. Завжди текстуруйте, використовуйте градієнт або шарування.
- **Різноманітність руху** — різні швидкості для різних елементів. Основний — 1×, другорядний — 0.3×, амбієнт — 0.1×.
- **Принаймні один вигаданий елемент** — кастомна поведінка часток, нова комбінація шуму, унікальна реакція взаємодії.

### Проєктно‑специфічна інновація

Для кожного проєкту придумайте хоча б один із:
- Кастомна колірна палітра, що відповідає настрою (не готовий пресет).
- Новий комбінований шумовий поле (наприклад, крутильний шум + доменне викривлення + зворотний зв’язок).
- Унікальна поведінка часток (кастомні сили, кастомні сліди, кастомне спавнування).
- Механіка взаємодії, яку користувач не просив, але яка підвищує якість роботи.
- Техніка композиції, що створює візуальну ієрархію.

### Філософія дизайну параметрів

Параметри мають виникати з алгоритму, а не з універсального меню. Питай: «Які властивості *цього* системи мають бути налаштовуваними?»

**Хороші параметри** розкривають характер алгоритму:
- **Кількості** — скільки часток, гілок, клітин (регулює густоту).
- **Масштаби** — частота шуму, розмір елементів, інтервали (регулює текстуру).
- **Швидкості** — темп, швидкість росту, згасання (регулює енергію).
- **Пороги** — коли змінюється поведінка? (регулює драматичність).
- **Відношення** — пропорції, баланс між силами (регулює гармонію).

**Погані параметри** — це загальні контролі, не пов’язані з алгоритмом:
- «color1», «color2», «size» — без контексту беззмістовні.
- Перемикачі для несуттєвих ефектів.
- Параметри, що змінюють лише зовнішній вигляд, а не поведінку.

Кожен параметр має змінювати спосіб, яким алгоритм *думатиме*, а не лише те, як він *виглядатиме*. Параметр «турбулентність», що змінює октави шуму, — хороший. Слайдер «розмір частки», що лише змінює радіус `ellipse()`, — поверхневий.
## Робочий процес

### Крок 1: Креативне бачення

Перед будь‑яким кодом сформулюй:

- **Настрій / атмосфера**: Які відчуття має виникнути у глядача? Розмірковуючі? Енергійні? Тривожні? Грайливі?
- **Візуальна історія**: Що відбувається з часом (або під час взаємодії)? Будується? Розпадається? Трансформується? Осцилює?
- **Колірний світ**: Теплий/холодний? Монохромний? Комплементарний? Який домінуючий відтінок? Який акцент?
- **Мова форм**: Органічні криві? Гостра геометрія? Точки? Лінії? Змішані?
- **Словник руху**: Повільний дрейф? Вибуховий сплеск? Дихальний пульс? Механічна точність?
- **Що робить ЦЕ унікальним**: Яка одна риса робить цей ескіз особливим?

Зістав запит користувача з естетичними виборами. «Розслаблююче генеративне тло» вимагає зовсім іншого підходу, ніж «гліч‑візуалізація даних».

### Крок 2: Технічний дизайн

- **Режим** — який із 7 режимів у таблиці вище
- **Розмір полотна** — landscape 1920x1080, portrait 1080x1920, square 1080x1080, або responsive `windowWidth/windowHeight`
- **Рендерер** — `P2D` (за замовчуванням) або `WEBGL` (для 3D, шейдерів, розширених режимів змішування)
- **Частота кадрів** — 60fps (інтерактивно), 30fps (фонова анімація) або `noLoop()` (статичне генеративне)
- **Ціль експорту** — відображення в браузері, PNG‑зображення, GIF‑цикл, MP4‑відео, SVG‑вектор
- **Модель взаємодії** — пасивна (без вводу), керована мишею, клавіатурою, аудіо‑реактивна, керована прокруткою
- **Інтерфейс глядача** — для інтерактивного генеративного мистецтва починай з `templates/viewer.html`, який забезпечує навігацію по seed, слайдери параметрів і завантаження. Для простих ескізів або експорту відео використай чистий HTML

### Крок 3: Код ескізу

Для **інтерактивного генеративного мистецтва** (дослідження seed, налаштування параметрів): починай з `templates/viewer.html`. - **Колірний режим**: Використовуй `colorMode(HSB, 360, 100, 100, 100)` для інтуїтивного керування кольорами
- **Розділення стану**: `CONFIG` для параметрів, `PALETTE` для кольорів, глобальні змінні для змінного стану
- **Класи‑сутності**: Частинки, агенти, форми як класи з методами `update()` + `display()`
- **Бафери поза екраном**: `createGraphics()` для шаруватої композиції, слідів, масок

### Крок 4: Попередній перегляд та ітерація

- Відкрий HTML‑файл безпосередньо в браузері — сервер не потрібен для базових ескізів
- Для `loadImage()`/`loadFont()` з локальних файлів: використай `scripts/serve.sh` або `python3 -m http.server`
- Вкладка **Performance** у Chrome DevTools для перевірки 60fps
- Тестуй у цільовій роздільній здатності, а не лише у вікні
- Корегуй параметри, доки візуал не відповідатиме концепції з Кроку 1

### Крок 6: Перевірка якості

- **Чи відповідає візуал баченню?** Порівняй результат з креативною концепцією. Якщо виглядає загально, повернись до Кроку 1
- **Перевірка роздільної здатності**: Чи чітко виглядає на цільовому розмірі? Чи немає артефактів згладжування?
- **Перевірка продуктивності**: Чи тримає 60fps у браузері? (мінімум 30fps для анімацій)
- **Перевірка кольорів**: Чи гармонують кольори? Протести на світлому та темному моніторах
- **Крайові випадки**: Що відбувається на краях полотна? При зміні розміру? Після 10‑хвилинної роботи?
## Критичні нотатки щодо реалізації

### Продуктивність — Спочатку вимкнути FES

Friendly Error System (FES) додає до 10‑х разів навантаження. Вимкни його в кожному продакшн‑скетчі:

```javascript
p5.disableFriendlyErrors = true;  // BEFORE setup()

function setup() {
  pixelDensity(1);  // prevent 2x-4x overdraw on retina
  createCanvas(1920, 1080);
}
```

У гарячих циклах (частинки, піксельні операції) використовуйте `Math.*` замість обгорток p5 — вимірно швидше:

```javascript
// In draw() or update() hot paths:
let a = Math.sin(t);          // not sin(t)
let r = Math.sqrt(dx*dx+dy*dy); // not dist() — or better: skip sqrt, compare magSq
let v = Math.random();        // not random() — when seed not needed
let m = Math.min(a, b);       // not min(a, b)
```

Ніколи не викликайте `console.log()` всередині `draw()`. Ніколи не маніпулюйте DOM у `draw()`. Див. `references/troubleshooting.md` § Performance.

### Seeded Randomness — Завжди

Кожен генеративний скетч має бути відтворюваним. Одна seed — один результат.

```javascript
function setup() {
  randomSeed(CONFIG.seed);
  noiseSeed(CONFIG.seed);
  // All random() and noise() calls now deterministic
}
```

Не використовуйте `Math.random()` для генеративного контенту — лише для продуктивно‑критичного не‑візуального коду. Для візуальних елементів завжди використовуйте `random()`. Якщо потрібен випадковий seed: `CONFIG.seed = floor(random(99999))`.

### Підтримка платформ генеративного мистецтва (fxhash / Art Blocks)

Для платформ генеративного мистецтва замініть PRNG p5 на детерміновану випадковість платформи:

```javascript
// fxhash convention
const SEED = $fx.hash;              // unique per mint
const rng = $fx.rand;               // deterministic PRNG
$fx.features({ palette: 'warm', complexity: 'high' });

// In setup():
randomSeed(SEED);   // for p5's noise()
noiseSeed(SEED);

// Replace random() with rng() for platform determinism
let x = rng() * width;  // instead of random(width)
```

Див. `references/export-pipeline.md` § Platform Export.

### Колірний режим — Використовуй HSB

HSB (Hue, Saturation, Brightness) значно простіший у роботі, ніж RGB, для генеративного мистецтва:

```javascript
colorMode(HSB, 360, 100, 100, 100);
// Now: fill(hue, sat, bri, alpha)
// Rotate hue: fill((baseHue + offset) % 360, 80, 90)
// Desaturate: fill(hue, sat * 0.3, bri)
// Darken: fill(hue, sat, bri * 0.5)
```

Ніколи не хардкодьте «чисті» RGB‑значення. Визначте об’єкт палітри, отримуйте варіації процедурно. Див. `references/color-systems.md`.

### Шум — Багатооктавний, а не «чистий»

«Чистий» `noise(x, y)` виглядає як гладкі плями. Шаруйте октави для природної текстури:

```javascript
function fbm(x, y, octaves = 4) {
  let val = 0, amp = 1, freq = 1, sum = 0;
  for (let i = 0; i < octaves; i++) {
    val += noise(x * freq, y * freq) * amp;
    sum += amp;
    amp *= 0.5;
    freq *= 2;
  }
  return val / sum;
}
```

Для плавних органічних форм використовуйте **domain warping**: подавайте вихід шуму назад як координати входу шуму. Див. `references/visual-effects.md`.

### `createGraphics()` для шарів — Не опціонально

Однопрохідне рендеринг виглядає плоско. Використовуйте буфери поза екраном для композиції:

```javascript
let bgLayer, fgLayer, trailLayer;
function setup() {
  createCanvas(1920, 1080);
  bgLayer = createGraphics(width, height);
  fgLayer = createGraphics(width, height);
  trailLayer = createGraphics(width, height);
}
function draw() {
  renderBackground(bgLayer);
  renderTrails(trailLayer);   // persistent, fading
  renderForeground(fgLayer);  // cleared each frame
  image(bgLayer, 0, 0);
  image(trailLayer, 0, 0);
  image(fgLayer, 0, 0);
}
```

### Продуктивність — Векторизуй, коли можливо

Виклики `draw` у p5.js дорогі. Для тисяч частинок:

```javascript
// SLOW: individual shapes
for (let p of particles) {
  ellipse(p.x, p.y, p.size);
}

// FAST: single shape with beginShape()
beginShape(POINTS);
for (let p of particles) {
  vertex(p.x, p.y);
}
endShape();

// FASTEST: pixel buffer for massive counts
loadPixels();
for (let p of particles) {
  let idx = 4 * (floor(p.y) * width + floor(p.x));
  pixels[idx] = r; pixels[idx+1] = g; pixels[idx+2] = b; pixels[idx+3] = 255;
}
updatePixels();
```

Див. `references/troubleshooting.md` § Performance.

### Instance Mode для кількох скетчів

Глобальний режим забруднює `window`. Для продакшну використовуйте instance mode:

```javascript
const sketch = (p) => {
  p.setup = function() {
    p.createCanvas(800, 800);
  };
  p.draw = function() {
    p.background(0);
    p.ellipse(p.mouseX, p.mouseY, 50);
  };
};
new p5(sketch, 'canvas-container');
```

Потрібно, коли вбудовуєте кілька скетчів на одну сторінку або інтегруєте їх у фреймворки.

### Підводні камені WebGL‑режиму

- `createCanvas(w, h, WEBGL)` — початок у центрі, а не у верхньому‑лівому куті
- Вісь Y інвертована (позитивна Y йде вгору у WEBGL, вниз у P2D)
- `translate(-width/2, -height/2)` — щоб отримати координати, схожі на P2D
- `push()`/`pop()` навколо кожного трансформу — стек матриць може переповнитися без повідомлення
- `texture()` перед `rect()`/`plane()` — не після
- Кастомні шейдери: `createShader(vert, frag)` — тестуйте у різних браузерах

### Експорт — Конвенція клавіш

Кожен скетч має включати ці прив’язки у `keyPressed()`:

```javascript
function keyPressed() {
  if (key === 's' || key === 'S') saveCanvas('output', 'png');
  if (key === 'g' || key === 'G') saveGif('output', 5);
  if (key === 'r' || key === 'R') { randomSeed(millis()); noiseSeed(millis()); }
  if (key === ' ') CONFIG.paused = !CONFIG.paused;
}
```

### Безголовий відео‑експорт — Використовуй `noLoop()`

Для безголового рендерингу через Puppeteer скетч **повинен** використовувати `noLoop()` у `setup`. Без цього цикл `draw` p5 працює безперервно, а скріншоти робляться повільно — скетч «переганяє» і ви отримуєте пропущені/дубльовані кадри.

```javascript
function setup() {
  createCanvas(1920, 1080);
  pixelDensity(1);
  noLoop();                    // capture script controls frame advance
  window._p5Ready = true;      // signal readiness to capture script
}
```

Вбудований `scripts/export-frames.js` виявляє `_p5Ready` і викликає `redraw()` один раз на захоплення для точної 1:1 відповідності кадрів. Див. `references/export-pipeline.md` § Deterministic Capture.

Для багатосценних відео використовуйте архітектуру per‑clip: один HTML на сцену, рендер окремо, склейте за допомогою `ffmpeg -f concat`. Див. `references/export-pipeline.md` § Per‑Clip Architecture.

### Робочий процес агента

Коли створюєш скетчі p5.js:

1. **Напиши HTML‑файл** — один самодостатній файл, весь код inline
2. **Відкрий у браузері** — `open sketch.html` (macOS) або `xdg-open sketch.html` (Linux)
3. **Локальні ресурси** (шрифти, зображення) потребують сервера: `python3 -m http.server 8080` у каталозі проєкту, потім відкрий `http://localhost:8080/sketch.html`
4. **Експорт PNG/GIF** — додай скорочення у `keyPressed()` як показано вище, повідом користувачу, яку клавішу натиснути
5. **Безголовий експорт** — `node scripts/export-frames.js sketch.html --frames 300` для автоматизованого захоплення кадрів (скетч має використовувати `noLoop()` + `_p5Ready`)
6. **Рендеринг MP4** — `bash scripts/render.sh sketch.html output.mp4 --duration 30`
7. **Ітеративне уточнення** — редагуй HTML‑файл, користувач оновлює сторінку, щоб побачити зміни
8. **Завантаження довідок за потребою** — використай `skill_view(name="p5js", file_path="references/...")` для завантаження конкретних файлів довідки під час реалізації

## Creative Divergence (використовуй лише коли користувач запитує експериментальний/креативний/унікальний результат)

Якщо користувач просить креативний, експериментальний, несподіваний або нестандартний результат, обери стратегію, яка найкраще підходить, і обґрунтуй її кроки **до** генерації коду.

- **Conceptual Blending** — коли користувач називає два об’єкти для поєднання або хоче гібридну естетику
- **SCAMPER** — коли користувач хоче варіацію відомого шаблону генеративного мистецтва
- **Distance Association** — коли користувач дає один концепт і хоче його розгорнути («створи щось про час»)

### Conceptual Blending
1. Назви два різні візуальні системи (наприклад, фізика часток + рукопис)
2. Визнач відповідності (частинки = краплі чорнила, сили = тиск пера, поля = форми літер)
3. Поєднуй вибірково — залишай лише ті відповідності, які створюють цікаві emergent‑візуали
4. Реалізуй поєднання як єдину систему, а не як два окремих підсистеми

### SCAMPER Transformation
Візьми відомий генеративний шаблон (flow field, particle system, L‑system, cellular automata) і систематично трансформуй його:
- **Substitute**: заміни кола текстовими символами, лінії — градієнтами
- **Combine**: об’єднай два шаблони (flow field + voronoi)
- **Adapt**: застосуй 2D‑шаблон до 3D‑проекції
- **Modify**: перебільши масштаб, спотвор координатний простір
- **Purpose**: використай фізичну симуляцію для типографіки, алгоритм сортування — для кольору
- **Eliminate**: прибери сітку, прибери колір, прибери симетрію
- **Reverse**: запусти симуляцію у зворотному напрямку, інвертуй простір параметрів

### Distance Association
1. Зафіксуй концепт користувача (наприклад, «самотність»)
2. Створи асоціації на трьох відстанях:
   - **Близька** (очевидна): порожня кімната, одинична фігура, тиша
   - **Середня** (цікава): одна риба в школі, що плаває у протилежному напрямку, телефон без сповіщень, проміжок між вагонами метро
   - **Далека** (абстрактна): прості числа, асимптотичні криві, колір 3 ранку
3. Розвивай середньовіддалені асоціації — вони достатньо конкретні для візуалізації, але досить несподівані, щоб бути цікавими.
