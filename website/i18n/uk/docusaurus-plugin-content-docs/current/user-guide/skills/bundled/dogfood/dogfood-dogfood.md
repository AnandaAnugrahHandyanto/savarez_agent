# Dogfood: Систематичне тестування веб‑додатків QA

## Огляд

Ця навичка проводить тебе через систематичне дослідницьке QA тестування веб‑додатків за допомогою набору інструментів браузера. Ти будеш навігувати по застосунку, взаємодіяти з елементами, фіксувати докази проблем і створювати структурований звіт про баги.

## Передумови

- Набір інструментів браузера має бути доступний (`browser_navigate`, `browser_snapshot`, `browser_click`, `browser_type`, `browser_vision`, `browser_console`, `browser_scroll`, `browser_back`, `browser_press`)
- Цільовий URL та область тестування, надані користувачем

## Вхідні дані

Користувач надає:
1. **Target URL** — точка входу для тестування
2. **Scope** — які ділянки/фічі тестувати (або «full site» для повного охоплення)
3. **Output directory** (необов’язково) — куди зберігати скріншоти та звіт (за замовчуванням: `./dogfood-output`)

## Робочий процес

Дотримуйся цього 5‑етапного систематичного процесу:

### Етап 1: Планування

1. Створи структуру вихідного каталогу:
<!-- ascii-guard-ignore -->
   ```
   {output_dir}/
   ├── screenshots/       # Evidence screenshots
   └── report.md          # Final report (generated in Phase 5)
   ```
<!-- ascii-guard-ignore-end -->
2. Визнач область тестування згідно вводу користувача.
3. Склади приблизну карту сайту, плануючи, які сторінки та функції тестувати:
   - Головна/домашня сторінка
   - Навігаційні посилання (header, footer, sidebar)
   - Ключові користувацькі потоки (реєстрація, вхід, пошук, оформлення замовлення тощо)
   - Форми та інтерактивні елементи
   - Граничні випадки (порожні стани, сторінки помилок, 404)

### Етап 2: Дослідження

Для кожної сторінки або функції з плану:

1. **Navigate** до сторінки:
   ```
   browser_navigate(url="https://example.com/page")
   ```

2. **Take a snapshot** для розуміння структури DOM:
   ```
   browser_snapshot()
   ```

3. **Check the console** на наявність JavaScript‑помилок:
   ```
   browser_console(clear=true)
   ```
   Роби це після кожної навігації та після кожної значущої взаємодії. Тихі JS‑помилки — цінні знахідки.

4. **Take an annotated screenshot** для візуальної оцінки сторінки та виявлення інтерактивних елементів:
   ```
   browser_vision(question="Describe the page layout, identify any visual issues, broken elements, or accessibility concerns", annotate=true)
   ```
   Параметр `annotate=true` накладає нумеровані мітки `[N]` на інтерактивні елементи. Кожен `[N]` відповідає рефу `@eN` для подальших команд браузера.

5. **Test interactive elements** систематично:
   - Клікати кнопки та посилання: `browser_click(ref="@eN")`
   - Заповнювати форми: `browser_type(ref="@eN", text="test input")`
   - Тестувати навігацію клавіатурою: `browser_press(key="Tab")`, `browser_press(key="Enter")`
   - Прокручувати контент: `browser_scroll(direction="down")`
   - Перевіряти валідацію форм некоректними даними
   - Тестувати порожні відправки

6. **Після кожної взаємодії** перевір:
   - Помилки в консолі: `browser_console()`
   - Візуальні зміни: `browser_vision(question="What changed after the interaction?")`
   - Відповідність очікуваного та фактичного поведінки

### Етап 3: Збір доказів

Для кожної знайденої проблеми:

1. **Take a screenshot** зображення проблеми:
   ```
   browser_vision(question="Capture and describe the issue visible on this page", annotate=false)
   ```
   Збережи `screenshot_path` з відповіді — він буде використаний у звіті.

2. **Record the details**:
   - URL, де виникла проблема
   - Кроки для відтворення
   - Очікувана поведінка
   - Фактична поведінка
   - Помилки консолі (за наявності)
   - Шлях до скріншоту

3. **Classify the issue** згідно таксономії (див. `references/issue-taxonomy.md`):
   - Severity: Critical / High / Medium / Low
   - Category: Functional / Visual / Accessibility / Console / UX / Content

### Етап 4: Класифікація

1. Переглянь усі зібрані проблеми.
2. Де‑дуплікат — об’єднай баги, які однакові, але проявляються в різних місцях.
3. Признач остаточну severity та category кожній проблемі.
4. Сортуй за severity (Critical → High → Medium → Low).
5. Підрахуй кількість проблем за severity та category для executive summary.

### Етап 5: Звіт

Згенеруй фінальний звіт, використовуючи шаблон `templates/dogfood-report-template.md`.

Звіт має містити:
1. **Executive summary** з загальною кількістю проблем, розподілом за severity та областю тестування
2. **Per‑issue sections** з:
   - Номером та назвою проблеми
   - Бейджами severity та category
   - URL, де виявлено
   - Описом проблеми
   - Кроками для відтворення
   - Очікуваною vs фактичною поведінкою
   - Посиланнями на скріншоти (використовуй `MEDIA:<screenshot_path>` для вбудованих зображень)
   - Помилками консолі, якщо релевантно
3. **Summary table** всіх проблем
4. **Testing notes** — що протестовано, що не протестовано, будь‑які блокери

Збережи звіт у `{output_dir}/report.md`.

## Довідка по інструментам

| Інструмент | Призначення |
|------|---------|
| `browser_navigate` | Перейти за URL |
| `browser_snapshot` | Отримати текстовий знімок DOM (дерево доступності) |
| `browser_click` | Клікнути елемент за рефом (`@eN`) або текстом |
| `browser_type` | Ввести текст у поле |
| `browser_scroll` | Прокрутити сторінку вгору/вниз |
| `browser_back` | Повернутись у історії браузера |
| `browser_press` | Натиснути клавішу клавіатури |
| `browser_vision` | Скріншот + AI‑аналіз; використай `annotate=true` для міток елементів |
| `browser_console` | Отримати вивід та помилки JS‑консолі |

## Поради

- **Завжди перевіряй `browser_console()` після навігації та після значущих взаємодій.** Тихі JS‑помилки — одні з найцінніших знахідок.
- **Використовуй `annotate=true` з `browser_vision`**, коли треба зрозуміти позиції інтерактивних елементів або коли рефи зі скріншоту неясні.
- **Тестуй як валідні, так і невалідні дані** — баги валідації форм поширені.
- **Прокручуй довгі сторінки** — контент під “fold” може мати проблеми рендерингу.
- **Тестуй навігаційні потоки** — пройди багатокрокові процеси від початку до кінця.
- **Перевір адаптивність**, фіксуючи будь‑які проблеми розмітки на скріншотах.
- **Не забувай про граничні випадки**: порожні стани, надто довгі тексти, спеціальні символи, швидкі кліки.
- При подачі скріншотів користувачу включай `MEDIA:<screenshot_path>`, щоб вони могли бачити докази вбудовано.
