# Сесії

Hermes Agent автоматично зберігає кожну розмову як сесію. Сесії дозволяють відновлювати розмову, здійснювати пошук між сесіями та повністю керувати історією розмов.
## Як працюють сесії

Кожна розмова — будь‑то з CLI, Telegram, Discord, Slack, WhatsApp, Signal, Matrix, Teams або будь‑якої іншої платформи обміну повідомленнями — зберігається як сесія з повною історією повідомлень. Сесії відстежуються у:

1. **SQLite database** (`~/.hermes/state.db`) — структуровані метадані сесії з FTS5 повнотекстовим пошуком, плюс повна історія повідомлень

База даних SQLite зберігає:
- Session ID, source platform, user ID
- **Session title** (унікальна, зрозуміла людиною назва)
- Model name and configuration
- System prompt snapshot
- Full message history (role, content, tool calls, tool results)
- Token counts (input/output)
- Timestamps (started_at, ended_at)
- Parent session ID (для розділення сесії, викликаного компресією)

### Що враховується в контекст

Hermes зберігає історію сесії, щоб можна було відновлювати розмови, але не
пересилає кожен байт, який коли‑небудь обробляв. На кожному кроці модель бачить
вибраний системний промпт, поточне вікно розмови та будь‑який вміст,
який Hermes явно вставляє для цього кроку.

Медіа‑вкладення обробляються як вхідні дані, прив’язані до кроку:

- Зображення можуть бути прикріплені безпосередньо до наступного виклику моделі, або попередньо проаналізовані в
  текстовий опис, коли активна модель не підтримує нативне бачення.
- Аудіо транскрибується в текст, коли налаштовано speech‑to‑text.
- Текстові документи можуть мати включений витягнутий текст; інші типи документів
  зазвичай представляються збереженим локальним шляхом і короткою нотаткою.
- Шляхи до вкладень та витягнутий/виведений текст можуть з’являтися в транскрипті, але
  сирі байти зображення, аудіо чи бінарного файлу не копіюються повторно у
  майбутні підказки.

Наприклад, якщо користувач надсилає зображення і просить Hermes створити мем,
Hermes може один раз проаналізувати це зображення за допомогою бачення та запустити скрипт обробки зображень. Майбутні кроки автоматично не несуть оригінальний JPEG у контексті.
Вони несуть лише те, що було записано в розмову, наприклад запит користувача,
короткий опис зображення, локальний шлях до кешу або остаточну відповідь асистента.

Найчастіша причина росту контексту — не медіа‑файл, а
об’ємний текст: вставлені транскрипти, повні логи, великі виводи інструментів, довгі дифи,
повторювані звіти про статус і детальні дампи доказів. Віддавай перевагу
резюме, шляхам до файлів, фокусованим уривкам і пошукам за допомогою інструментів,
замість копіювання великих артефактів у чат.

:::tip
Використовуй `/compress`, коли сесія стає довгою, `/new` для нової гілки, і
`hermes sessions prune` лише коли хочеш видалити старі завершені сесії зі сховища. Компресія зменшує активний контекст; це не видалення даних для конфіденційності.
Передай назву до `/new` (наприклад `/new payments-refactor`), щоб задати початкову назву нової сесії — це зручно для пошуку пізніше за допомогою `/resume <name>` або в вибірці `/sessions`.
:::

## CLI Session Resume

Віднови попередні розмови з CLI за допомогою `--continue` або `--resume`:

### Continue Last Session

```bash
# Resume the most recent CLI session
hermes --continue
hermes -c

# Or with the chat subcommand
hermes chat --continue
hermes chat -c
```

Це шукає останню `cli` сесію у базі даних SQLite і завантажує її повну історію розмов.

### Resume by Name

Якщо ти дав сесії назву (див. [Session Naming](#session-naming) нижче), її можна відновити за назвою:

```bash
# Resume a named session
hermes -c "my project"

# If there are lineage variants (my project, my project #2, my project #3),
# this automatically resumes the most recent one
hermes -c "my project"   # → resumes "my project #3"
```

### Resume Specific Session

```bash
# Resume a specific session by ID
hermes --resume 20250305_091523_a1b2c3d4
hermes -r 20250305_091523_a1b2c3d4

# Resume by title
hermes --resume "refactoring auth"

# Or with the chat subcommand
hermes chat --resume 20250305_091523_a1b2c3d4
```

Ідентифікатори сесій показуються при виході з CLI‑сесії і їх можна знайти за допомогою `hermes sessions list`.

### Conversation Recap on Resume

Коли ти відновлюєш сесію, Hermes відображає компактний підсумок попередньої розмови у стилізованій панелі перед запитом вводу:

<img className="docs-terminal-figure" src="/img/docs/session-recap.svg" alt="Stylized preview of the Previous Conversation recap panel shown when resuming a Hermes session." />
<p className="docs-figure-caption">Режим відновлення показує компактну панель підсумку з останніми повідомленнями користувача та асистента перед поверненням до живого запиту.</p>

Підсумок:
- Показує **повідомлення користувача** (золотий `●`) і **відповіді асистента** (зелений `◆`);
- **Обрізає** довгі повідомлення (300 символів для користувача, 200 символів / 3 рядки для асистента);
- **Згортає** виклики інструментів до кількості з назвами інструментів (наприклад, `[3 tool calls: terminal, web_search]`);
- **Приховує** системні повідомлення, результати інструментів та внутрішнє міркування;
- **Обмежує** підсумок останніми 10 обмінами з індикатором «... N earlier messages ...»;
- Використовує **приглушене оформлення**, щоб відрізнити його від активної розмови.

Щоб вимкнути підсумок і залишити мінімальну однорядкову поведінку, встанови у `~/.hermes/config.yaml`:

```yaml
display:
  resume_display: minimal   # default: full
```

:::tip
Ідентифікатори сесій мають формат `YYYYMMDD_HHMMSS_<hex>` — CLI/TUI сесії використовують 6‑символьний hex‑суфікс (наприклад, `20250305_091523_a1b2c3`), сесії шлюзу — 8‑символьний суфікс (наприклад, `20250305_091523_a1b2c3d4`). Ти можеш відновити за ID (повним або унікальним префіксом) або за назвою — обидва варіанти працюють з `-c` і `-r`.
:::
## Крос‑платформений Handoff

Використовуй `/handoff <platform>` у CLI‑сесії, щоб передати живу розмову у домашній канал обраної платформи обміну повідомленнями. Агент підхоплює розмову саме там, де CLI залишив її — той самий ідентифікатор сесії, повний ролевий транскрипт, виклики інструментів та все інше.

```bash
# Inside a CLI session
/handoff telegram
```

Що відбувається:

1. CLI перевіряє, що `<platform>` увімкнено і в ньому встановлено домашній канал (одноразово запусти `/sethome` у чаті‑призначенні, щоб його налаштувати).
2. CLI позначає сесію як очікуючу і **block‑polls шлюзом**. Відмовляє, якщо агент знаходиться посеред ходи — спочатку дочекайся завершення поточної відповіді.
3. Спостерігач шлюзу захоплює handoff і запитує у адаптера призначення нову гілку:
   - **Telegram** — відкриває нову тему форуму (DM‑теми, якщо в Bot API 9.4+ увімкнено режим Topics у чаті, або тему супер‑групи форуму).
   - **Discord** — створює 1440‑хвилинну автоматично‑архівовану гілку під домашнім текстовим каналом.
   - **Slack** — публікує стартове повідомлення і використовує його `ts` як прив’язку до гілки.
   - **WhatsApp / Signal / Matrix / SMS** — немає вбудованих гілок, переходить безпосередньо у домашній канал.
4. Шлюз повторно прив’язує ключ призначення до існуючого ідентифікатора твоєї CLI‑сесії, після чого створює синтетичний хід користувача, просячи агента підтвердити та підсумувати. Відповідь з’являється у новій гілці.
5. Коли шлюз підтверджує успіх, CLI виводить підказку `/resume` і коректно завершується:

   ```
   ↻ Handoff complete. The session is now active on telegram.
     Resume it on this CLI later with: /resume my-session-title
   ```

6. Від цього моменту розмова живе на платформі. Відповідай у новій гілці — будь‑хто, хто має доступ до цього каналу, ділиться тією ж сесією, і будь‑яке подальше реальне повідомлення користувача в гілці автоматично приєднується, бо сесії гілок не потребують `user_id`.

**Повернутись у CLI:** коли захочеш повернутись до десктопа, просто запусти `/resume <title>` (або `hermes -r "<title>"` у терміналі) і підхопи розмову там, де її залишила платформа.

**Режими відмови:**
- Не налаштовано домашній канал → CLI відмовляє з підказкою `/sethome`.
- Платформу не увімкнено / шлюз не запущено → CLI завершує очікування через 60 секунд з чітким повідомленням, а твоя CLI‑сесія залишається недоторканою.
- Не вдалося створити гілку (відсутні права, вимкнено topics‑mode) → переходить безпосередньо у домашній канал і все одно завершує; ізоляція гілки відсутня, але handoff працює.
- `adapter.send` не вдається (обмеження швидкості, тимчасова помилка API) → handoff позначається як невдалий з причиною; рядок очищується, і ти можеш спробувати ще раз.

**Обмеження, яке варто знати:** для платформ без підтримки гілок, які мають багатокористувацькі групові домашні канали, синтетичний хід працює як DM‑стиль сесії. Це підходить для домашніх каналів типу self‑DM (типове налаштування), але не ідеально для справжніх спільних групових чатів. Тема підтримується у Telegram, Discord, Slack — найпоширеніший випадок, тому більшість налаштувань ніколи не стикаються з цим обмеженням.
## Іменування сесій

Давай сесіям зрозумілі для людини назви, щоб їх легко знаходити та відновлювати.

### Автоматично згенеровані назви

Hermes автоматично створює коротку описову назву (3–7 слів) для кожної сесії після першого обміну. Це виконується у фоні за допомогою швидкої допоміжної моделі, тому не додає затримки. Ти побачиш автоматично згенеровані назви під час перегляду сесій за допомогою `hermes sessions list` або `hermes sessions browse`.

Автоматичне називання спрацьовує лише один раз за сесію і пропускається, якщо ти вже вручну встановив назву.

### Встановлення назви вручну

Використай команду `/title` у будь‑якій чат‑сесії (CLI або gateway):

```
/title my research project
```

Назва застосовується одразу. Якщо сесія ще не створена в базі даних (наприклад, ти виконав `/title` до надсилання першого повідомлення), вона ставиться в чергу і застосовується, коли сесія стартує.

Ти також можеш перейменувати існуючі сесії з командного рядка:

```bash
hermes sessions rename 20250305_091523_a1b2c3d4 "refactoring auth module"
```

### Правила назви

- **Унікальна** — дві сесії не можуть мати одну й ту ж назву
- **Максимум 100 символів** — зберігає вивід списку чистим
- **Санітизована** — контрольні символи, нуль‑ширинні символи та RTL‑перевизначення автоматично видаляються
- **Звичайний Unicode підходить** — emoji, CJK, символи з діакритиками працюють без проблем

### Автоматичне успадкування при стисканні

Коли контекст сесії стискається (вручну через `/compress` або автоматично), Hermes створює нову продовжувальну сесію. Якщо у оригіналу була назва, нова сесія автоматично отримує нумеровану назву:

```
"my project" → "my project #2" → "my project #3"
```

Коли ти відновлюєш за назвою (`hermes -c "my project"`), система автоматично вибирає найновішу сесію в лінійці.

### `/title` у платформах шлюзу

Команда `/title` працює у всіх платформах шлюзу (Telegram, Discord, Slack, WhatsApp):

- `/title My Research` — встановити назву сесії
- `/title` — показати поточну назву
## Команди керування сесіями

Hermes надає повний набір команд керування сесіями через `hermes sessions`:

### Список сесій

```bash
# List recent sessions (default: last 20)
hermes sessions list

# Filter by platform
hermes sessions list --source telegram

# Show more sessions
hermes sessions list --limit 50
```

Коли сесії мають назви, у виводі показуються назви, попередні перегляди та відносні часові мітки:

```
Title                  Preview                                  Last Active   ID
────────────────────────────────────────────────────────────────────────────────────────────────
refactoring auth       Help me refactor the auth module please   2h ago        20250305_091523_a
my project #3          Can you check the test failures?          yesterday     20250304_143022_e
—                      What's the weather in Las Vegas?          3d ago        20250303_101500_f
```

Якщо жодна сесія не має назви, використовується спрощений формат:

```
Preview                                            Last Active   Src    ID
──────────────────────────────────────────────────────────────────────────────────────
Help me refactor the auth module please             2h ago        cli    20250305_091523_a
What's the weather in Las Vegas?                    3d ago        tele   20250303_101500_f
```

### Експорт сесій

```bash
# Export all sessions to a JSONL file
hermes sessions export backup.jsonl

# Export sessions from a specific platform
hermes sessions export telegram-history.jsonl --source telegram

# Export a single session
hermes sessions export session.jsonl --session-id 20250305_091523_a1b2c3d4
```

Експортовані файли містять один JSON‑об’єкт на рядок із повними метаданими сесії та всіма повідомленнями.

### Видалити сесію

```bash
# Delete a specific session (with confirmation)
hermes sessions delete 20250305_091523_a1b2c3d4

# Delete without confirmation
hermes sessions delete 20250305_091523_a1b2c3d4 --yes
```

### Перейменувати сесію

```bash
# Set or change a session's title
hermes sessions rename 20250305_091523_a1b2c3d4 "debugging auth flow"

# Multi-word titles don't need quotes in the CLI
hermes sessions rename 20250305_091523_a1b2c3d4 debugging auth flow
```

Якщо назва вже використовується іншою сесією, буде показано помилку.

### Обрізання старих сесій

```bash
# Delete ended sessions older than 90 days (default)
hermes sessions prune

# Custom age threshold
hermes sessions prune --older-than 30

# Only prune sessions from a specific platform
hermes sessions prune --source telegram --older-than 60

# Skip confirmation
hermes sessions prune --older-than 30 --yes
```

:::info
Обрізання видаляє лише **завершені** сесії (сесії, які були явно завершені або автоматично скинуті). Активні сесії ніколи не обрізаються.
:::

### Статистика сесій

```bash
hermes sessions stats
```

Вивід:

```
Total sessions: 142
Total messages: 3847
  cli: 89 sessions
  telegram: 38 sessions
  discord: 15 sessions
Database size: 12.4 MB
```

Для глибшого аналізу — використання токенів, оцінки вартості, розподіл інструментів та патерни активності — скористайся [`hermes insights`](/reference/cli-commands#hermes-insights).
## Інструмент пошуку сесій

Агент має вбудований інструмент `session_search`, який виконує повнотекстовий пошук по всіх минулих розмовах за допомогою движка SQLite FTS5 — і дозволяє агенту прокручувати будь‑яку знайдену сесію. Без викликів LLM, без підсумовування, без обрізання. Кожна форма повертає фактичні повідомлення з БД.

### Три форми виклику

Інструмент визначає, що тобі потрібно, за заданими аргументами. Параметр `mode` відсутній.

**1. Discovery — передай `query`:**

```python
session_search(query="auth refactor", limit=3)
```

Запускає FTS5, видаляє дублікати за лінійністю сесії, повертає топ N сесій. Кожен результат містить:

- `session_id`, `title`, `when`, `source`
- `snippet` — фрагмент збігу, підсвічений FTS5
- `bookend_start` — перші 3 повідомлення користувача + асистента сесії (ціль/старт)
- `messages` — ±5 повідомлень навколо збігу FTS5, з позначеним якорним повідомленням (збіг у контексті)
- `bookend_end` — останні 3 повідомлення користувача + асистента сесії (резолюція/рішення)
- `match_message_id`, `messages_before`, `messages_after`

`bookend` + вікно разом відтворюють ціль → збіг → резолюцію без потреби обробляти весь транскрипт. Типовий час виконання: 15–50 мс на реальній БД сесій.

**2. Scroll — передай `session_id` + `around_message_id`:**

```python
session_search(session_id="20260510_174648_805cc2", around_message_id=590803, window=10)
```

Повертає вікно ±`window` повідомлень, центроване навколо якоря. Без FTS5, без `bookend` — лише зріз. Використовуй після виклику discovery, коли потрібен більший контекст, ніж стандартне вікно ±5.

- Щоб прокручувати **вперед**: передай `messages[-1].id` як `around_message_id`
- Щоб прокручувати **назад**: передай `messages[0].id` як `around_message_id`
- Граничне повідомлення з’являється в обох вікнах як орієнтир
- Коли `messages_before` або `messages_after` менше `window`, ти знаходишся на початку або в кінці сесії

Типовий час виконання: 1–2 мс за виклик прокручування.

**3. Browse — без аргументів:**

```python
session_search()
```

Повертає останні сесії хронологічно (заголовки, прев’ю, часові мітки). Корисно, коли користувач запитує «на чому я працював», не називаючи тему.

### Синтаксис запиту FTS5

Режим ключових слів підтримує стандартний синтаксис запитів FTS5:

- Прості ключові слова: `docker deployment` (FTS5 за замовчуванням виконує AND)
- Фрази: `"exact phrase"`
- Булеві оператори: `docker OR kubernetes`, `python NOT java`
- Префікс: `deploy*`

### Додаткові параметри

- `sort` — `newest` або `oldest`, над ранжуванням FTS5. Якщо не вказано, використовується лише релевантність (за замовчуванням; підходить для дослідницького відновлення). Використовуй `newest` для питань типу «де ми залишили X», `oldest` — для «як починалось X».
- `role_filter` — ролі, розділені комами, які треба включити. За замовчуванням у discovery `user,assistant` (вихід інструмента зазвичай шум). Передай `user,assistant,tool`, щоб включити вихід інструмента (налагодження поведінки), або `tool`, щоб шукати лише вихід інструмента.

### Коли використовується

Агент отримує підказку використовувати пошук сесій автоматично:

> *«Коли користувач посилається на щось із минулої розмови або ти підозрюєш, що існує релевантний попередній контекст, використай session_search, щоб згадати його, перш ніж просити їх повторити.»*

Типові тригери: «ми робили це раніше», «пам’ятаєш, коли», «в останній раз», «як я згадував», або будь‑яке посилання на проєкт/особу/концепт, які не входять у поточне вікно.
## Відстеження сесій за платформою

### Сесії шлюзу

На платформах обміну повідомленнями сесії ідентифікуються детермінованим **ключем сесії**, сформованим з джерела повідомлення:

| Тип чату | Формат ключа за замовчуванням | Поведінка |
|-----------|--------------------|----------|
| Telegram DM | `agent:main:telegram:dm:<chat_id>` | Одна сесія на DM‑чат |
| Discord DM | `agent:main:discord:dm:<chat_id>` | Одна сесія на DM‑чат |
| WhatsApp DM | `agent:main:whatsapp:dm:<canonical_identifier>` | Одна сесія на користувача DM (LID/телефонні псевдоніми зводяться до однієї ідентичності, коли існує відповідність) |
| Груповий чат | `agent:main:<platform>:group:<chat_id>:<user_id>` | По‑користувацьки всередині групи, якщо платформа надає ID користувача |
| Групова гілка/тема | `agent:main:<platform>:group:<chat_id>:<thread_id>` | Спільна сесія для всіх учасників гілки (за замовчуванням). По‑користувацька при `thread_sessions_per_user: true`. |
| Канал | `agent:main:<platform>:channel:<chat_id>:<user_id>` | По‑користувацьки всередині каналу, якщо платформа надає ID користувача |

Коли Hermes не може отримати ідентифікатор учасника для спільного чату, він переходить до **однієї спільної сесії** для цієї кімнати.

### Спільні vs ізольовані групові сесії

За замовчуванням Hermes використовує `group_sessions_per_user: true` у `config.yaml`. Це означає:

- Alice і Bob можуть одночасно спілкуватися з Hermes в одному Discord‑каналі, не ділячись історією транскриптів
- довге завдання одного користувача, насичене інструментами, не забруднює контекстне вікно іншого користувача
- обробка переривань також залишається по‑користувацькій, бо ключ `running-agent` відповідає ізольованому ключу сесії

Якщо потрібен один спільний «мозок кімнати», встанови:

```yaml
group_sessions_per_user: false
```

Це повертає групи/канали до **однієї спільної сесії** на кімнату, зберігаючи спільний контекст розмови, але також спільно використовуючи токени, стан переривань та зростання контексту.

### Політики скидання сесії

Сесії шлюзу автоматично скидаються згідно з налаштованими політиками:

- **idle** — скидання після N хвилин бездіяльності
- **daily** — скидання у визначену годину кожного дня
- **both** — скидання за тим, що станеться раніше (idle або daily)
- **none** — ніколи не скидатись автоматично

Перед автоматичним скиданням сесії агент отримує шанс зберегти важливі **пам’яті** або **навички** з розмови.

Сесії з **активними фоновими процесами** ніколи не скидаються автоматично, незалежно від політики.
## Місця зберігання

| Що | Шлях | Опис |
|------|------|-------------|
| SQLite‑база даних | `~/.hermes/state.db` | Усі метадані сесії + повідомлення з підтримкою FTS5 |
| Повідомлення gateway | `~/.hermes/state.db` | SQLite — канонічне сховище для всіх повідомлень сесії |
| Індекс маршрутизації gateway | `~/.hermes/sessions/sessions.json` | Відображає ключі сесій на активні ідентифікатори сесій (метадані походження, прапори закінчення терміну) |

SQLite‑база даних використовує режим WAL для одночасних читачів і одного записувача, що добре підходить для багатоплатформенної архітектури gateway.

:::note Legacy JSONL transcripts
Сесії, створені до того, як `state.db` став канонічним, можуть мати залишкові
`*.jsonl` файли у `~/.hermes/sessions/`. Вони більше не записуються і
не читаються Hermes. Безпечно видалити після перевірки, що відповідна сесія
існує в `state.db`.
:::

### Схема бази даних

Ключові таблиці у `state.db`:

- **sessions** — метадані сесії (id, source, user_id, model, title, timestamps, token counts). Заголовки мають унікальний індекс (дозволені `NULL`‑заголовки, лише не‑`NULL` мають бути унікальними).
- **messages** — повна історія повідомлень (role, content, tool_calls, tool_name, token_count)
- **messages_fts** — віртуальна таблиця FTS5 для повнотекстового пошуку по вмісту повідомлень
## Термін дії сесії та очистка

### Автоматичне очищення

- Сесії шлюзу автоматично скидаються згідно налаштованої політики скидання
- Перед скиданням агент зберігає пам'ять та навички з сесії, термін якої спливає
- Додаткова автоматична обрізка: коли `sessions.auto_prune` має значення `true`, завершені сесії, старші за `sessions.retention_days` (за замовчуванням 90), видаляються під час запуску CLI/шлюзу
- Після обрізки, яка фактично видалила рядки, `state.db` підлягає `VACUUM`, щоб звільнити місце на диску (SQLite не зменшує файл при простому `DELETE`)
- Обрізка виконується не частіше, ніж один раз за `sessions.min_interval_hours` (за замовчуванням 24); мітка часу останнього запуску зберігається всередині самого `state.db`, тому вона спільна для всіх процесів Hermes у тому ж `HERMES_HOME`

За замовчуванням **вимкнено** — історія сесій цінна для відновлення `session_search`, і її тихе видалення може здивувати користувачів. Увімкни в `~/.hermes/config.yaml`:

```yaml
sessions:
  auto_prune: true          # opt in — default is false
  retention_days: 90        # keep ended sessions this many days
  vacuum_after_prune: true  # reclaim disk space after a pruning sweep
  min_interval_hours: 24    # don't re-run the sweep more often than this
```

Активні сесії ніколи не підлягають автоматичній обрізці, незалежно від їх віку.

### Ручне очищення

```bash
# Prune sessions older than 90 days
hermes sessions prune

# Delete a specific session
hermes sessions delete <session_id>

# Export before pruning (backup)
hermes sessions export backup.jsonl
hermes sessions prune --older-than 30 --yes
```

:::tip
База даних росте повільно (типово: 10‑15 МБ для сотень сесій), а історія сесій живить відновлення `session_search` у минулих розмовах, тому автоматична обрізка за замовчуванням вимкнена. Увімкни її, якщо ти працюєш з важким навантаженням шлюзу/cron, коли `state.db` суттєво впливає на продуктивність (спостережуваний випадок: 384 МБ `state.db` з ~1000 сесій уповільнює вставки FTS5 та список `/resume`). Використовуй `hermes sessions prune` для одноразового очищення без включення автоматичного сканування.
:::
