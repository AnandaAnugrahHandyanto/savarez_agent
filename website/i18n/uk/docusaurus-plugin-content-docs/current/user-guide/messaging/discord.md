# Налаштування Discord

Hermes Agent інтегрується з Discord як бот, дозволяючи тобі спілкуватися зі своїм AI‑асистентом через прямі повідомлення або канали сервера. Бот отримує твої повідомлення, обробляє їх через pipeline Hermes Agent (включаючи використання інструментів, пам'ять і міркування) і відповідає в режимі реального часу. Підтримуються текстові, голосові повідомлення, вкладення файлів та slash‑команди.

Перед налаштуванням ось та частина, яку більшість людей хоче знати: як Hermes поводиться, коли він вже знаходиться на твоєму сервері.
## Як поводиться Hermes

| Context | Behavior |
|---------|----------|
| **DMs** | Hermes відповідає на кожне повідомлення. Не потрібен `@mention`. Кожен DM має свою сесію. |
| **Server channels** | За замовчуванням Hermes відповідає лише коли його `@mention`‑нуть. Якщо ти публікуєш у каналі без згадки, Hermes ігнорує повідомлення. |
| **Free-response channels** | Ти можеш зробити певні канали без згадок за допомогою `DISCORD_FREE_RESPONSE_CHANNELS`, або вимкнути згадки глобально за допомогою `DISCORD_REQUIRE_MENTION=false`. Повідомлення в цих каналах відповідаються **inline** — авто‑трединг пропускається, тож канал залишається легким чатом. |
| **Threads** | Hermes відповідає в тому ж треді. Правила згадок залишаються діючими, якщо тільки цей тред або його батьківський канал не налаштовані як free‑response. Треди залишаються ізольованими від батьківського каналу щодо історії сесії. |
| **Shared channels with multiple users** | За замовчуванням Hermes ізолює історію сесії для кожного користувача в межах каналу задля безпеки та ясності. Двоє людей, що розмовляють в одному каналі, не ділять один транскрипт, якщо ти явно не вимкнув це. |
| **Messages mentioning other users** | Коли `DISCORD_IGNORE_NO_MENTION` встановлено в `true` (за замовчуванням), Hermes мовчить, якщо повідомлення @згадує інших користувачів, але **не** згадує бота. Це запобігає втручанню бота в розмови, спрямовані до інших людей. Встанови `false`, якщо хочеш, щоб бот відповідав на всі повідомлення незалежно від того, кого згадали. Це діє лише в серверних каналах, а не в DMs. |

:::tip
Якщо ти хочеш звичайний канал довідки бота, де люди можуть спілкуватися з Hermes без тегу щоразу, додай цей канал до `DISCORD_FREE_RESPONSE_CHANNELS`.
:::

### Модель шлюзу Discord

Hermes у Discord — це не webhook, який відповідає без стану. Він працює через повний шлюз обміну повідомленнями, що означає, що кожне вхідне повідомлення проходить через:

1. авторизацію (`DISCORD_ALLOWED_USERS`);
2. перевірки згадок / free‑response;
3. пошук сесії;
4. завантаження транскрипту сесії;
5. звичайне виконання агента Hermes, включаючи інструменти, пам'ять та slash‑команди;
6. доставку відповіді назад у Discord.

Це важливо, бо поведінка в зайнятому сервері залежить і від маршрутизації Discord, і від політики сесій Hermes.

### Модель сесії в Discord

За замовчуванням:

- кожен DM має свою сесію;
- кожен серверний тред має власний простір імен сесії;
- кожен користувач у спільному каналі має свою сесію всередині цього каналу.

Тож якщо Аліса і Боб обидва розмовляють з Hermes у `#research`, Hermes розглядає їх як окремі розмови за замовчуванням, навіть якщо вони використовують один і той же видимий канал Discord.

Це контролюється у `config.yaml`:

```yaml
group_sessions_per_user: true
```

Встанови `false` лише якщо ти явно хочеш одну спільну розмову для всього простору:

```yaml
group_sessions_per_user: false
```

Спільні сесії можуть бути корисними для колаборативної кімнати, але вони також означають:

- користувачі ділять зріст контексту та витрати токенів;
- довге завдання з важкими інструментами одного користувача може «заповнити» контекст усіх інших;
- запущений процес одного користувача може перервати подальший запит іншого в тому ж просторі.

### Переривання та конкурентність

Hermes відстежує запущені агенти за ключем сесії.

З параметром за замовчуванням `group_sessions_per_user: true`:

- переривання власного запиту Аліси впливає лише на сесію Аліси в цьому каналі;
- Боб може продовжувати розмову в тому ж каналі, не успадковуючи історію Аліси і не перериваючи її запит.

З `group_sessions_per_user: false`:

- вся кімната ділить один слот запущеного агента для цього каналу/треду;
- подальші повідомлення від різних людей можуть переривати або ставати в чергу одне за одним.

Цей посібник проведе тебе через весь процес налаштування — від створення бота у **Developer Portal** Discord до надсилання першого повідомлення.
## Крок 1: Створи Discord‑застосунок

1. Перейди до [Discord Developer Portal](https://discord.com/developers/applications) і ввійди у свій Discord‑акаунт.
2. Натисни **New Application** у правому верхньому куті.
3. Введи назву для свого застосунку (наприклад, "Hermes Agent") і прийми Developer Terms of Service.
4. Натисни **Create**.

Ти опинишся на сторінці **General Information**. Запиши **Application ID** — він знадобиться пізніше для створення URL‑запрошення.
## Крок 2: Створи бота

1. У лівій бічній панелі натисни **Bot**.
2. Discord автоматично створює користувача‑бота для твого застосунку. Ти побачиш ім’я бота, яке можна змінити.
3. У розділі **Authorization Flow**:
   - Увімкни **Public Bot** → **ON** — це потрібно, щоб скористатися запрошувальним посиланням, наданим Discord (рекомендовано). Тоді вкладка **Installation** згенерує типову URL‑адресу авторизації.
   - Залиши **Require OAuth2 Code Grant** у стані **OFF**.

:::tip
Ти можеш задати власний аватар і банер для свого бота на цій сторінці. Це те, що користувачі бачитимуть у Discord.
:::

:::info[Private Bot Alternative]
Якщо ти хочеш залишити бота приватним (Public Bot = OFF), ти **повинен** використати метод **Manual URL** у Кроці 5 замість вкладки **Installation**. Запрошувальне посилання від Discord працює лише при увімкненому Public Bot.
:::
## Крок 3: Увімкнути привілейовані інтенти шлюзу

Це найважливіший крок у всій **налаштуванні**. Без правильних інтентів ваш бот підключиться до Discord, але **не зможе читати вміст повідомлень**.

На сторінці **Bot** прокрутіть вниз до **Privileged Gateway Intents**. Ви побачите три перемикачі:

| Інтент | Призначення | Потрібно? |
|--------|-------------|-----------|
| **Presence Intent** | Перегляд статусу користувачів (онлайн/офлайн) | Необов’язково |
| **Server Members Intent** | Доступ до списку учасників, розв’язання імен користувачів | **Потрібно** |
| **Message Content Intent** | Читання текстового вмісту повідомлень | **Потрібно** |

**Увімкніть Server Members Intent та Message Content Intent**, перемкнувши їх **ON**.

- Без **Message Content Intent** ваш бот отримує події повідомлень, але текст повідомлення порожній — бот буквально не бачить, що ти написав.
- Без **Server Members Intent** бот не може розпізнавати імена користувачів у списку дозволених і може не визначити, хто надсилає йому повідомлення.

:::warning[Це причина №1, чому Discord‑боти не працюють]
Якщо твій бот онлайн, але ніколи не відповідає на повідомлення, **Message Content Intent** майже напевно вимкнено. Повернись до [Developer Portal](https://discord.com/developers/applications), вибери свою програму → Bot → Privileged Gateway Intents і переконайся, що **Message Content Intent** перемкнено в **ON**. Натисни **Save Changes**.
:::

**Щодо кількості серверів:**
- Якщо твій бот знаходиться **менше ніж на 100 серверах**, ти можеш вільно перемикати інтенти.
- Якщо твій бот знаходиться **на 100 і більше серверах**, Discord вимагає подати заявку на верифікацію для використання привілейованих інтентів. Для особистого використання це не є проблемою.

Натисни **Save Changes** внизу сторінки.
## Крок 4: Отримати токен бота

Токен бота — це облікові дані, які Hermes Agent використовує для входу як твій бот. На сторінці **Bot**:

1. У розділі **Token** натисни **Reset Token**.
2. Якщо у твоєму обліковому записі Discord увімкнено двофакторну автентифікацію, введи код 2FA.
3. Discord покаже новий токен. **Одразу скопіюй його**.

:::warning[Token shown only once]
Токен відображається лише один раз. Якщо ти його втратиш, доведеться скинути його та згенерувати новий. Ніколи не публікуй токен і не додавай його в Git — будь‑хто, хто має цей токен, отримує повний контроль над твоїм ботом.
:::

Збережи токен у безпечному місці (наприклад, у менеджері паролів). Тобі знадобиться він у кроці 8.
## Крок 5: Генерація URL‑запрошення

Тобі потрібен OAuth2‑URL, щоб запросити бота на свій сервер. Є два способи це зробити:

### Варіант A: Використання вкладки **Installation** (рекомендовано)

:::note[Requires Public Bot]
Цей метод вимагає, щоб **Public Bot** був встановлений у **ON** у Кроці 2. Якщо ти вимкнув **Public Bot**, використай метод **Manual URL**, описаний нижче.
:::

1. У лівій бічній панелі натисни **Installation**.
2. У розділі **Installation Contexts** увімкни **Guild Install**.
3. Для **Install Link** вибери **Discord Provided Link**.
4. У розділі **Default Install Settings** для **Guild Install**:
   - **Scopes**: вибери `bot` і `applications.commands`
   - **Permissions**: вибери дозволи, перелічені нижче.

### Варіант B: **Manual URL**

Ти можеш сконструювати URL‑запрошення безпосередньо, використовуючи такий формат:

```
https://discord.com/oauth2/authorize?client_id=YOUR_APP_ID&scope=bot+applications.commands&permissions=274878286912
```

Заміні `YOUR_APP_ID` на ідентифікатор застосунку з Кроку 1.

### Необхідні дозволи

Це мінімальні дозволи, які потрібні твоєму боту:

- **View Channels** — бачити канали, до яких має доступ
- **Send Messages** — відповідати на твої повідомлення
- **Embed Links** — форматувати багаті відповіді
- **Attach Files** — надсилати зображення, аудіо та інші файли
- **Read Message History** — зберігати контекст розмови

### Рекомендовані додаткові дозволи

- **Send Messages in Threads** — відповідати в розмовах у гілках
- **Add Reactions** — реагувати на повідомлення для підтвердження

### Цілі числа дозволів

| Рівень | Ціле число дозволів | Що включено |
|-------|-------------------|-----------------|
| Minimal | `117760` | View Channels, Send Messages, Read Message History, Attach Files |
| Recommended | `274878286912` | All of the above plus Embed Links, Send Messages in Threads, Add Reactions |
## Крок 6: Запросити на свій сервер

1. Відкрий URL‑запрошення у браузері (з вкладки **Installation** або з вручну сконструйованого URL).
2. У випадаючому списку **Add to Server** вибери свій сервер.
3. Натисни **Continue**, потім **Authorize**.
4. За потреби пройди CAPTCHA.

:::info
Тобі потрібен дозвіл **Manage Server** на Discord‑сервері, щоб запросити бота. Якщо ти не бачиш свій сервер у випадаючому списку, попроси адміністратора сервера використати посилання запрошення.
:::

Після авторизації бот з’явиться у списку учасників твого сервера (він буде позначений як офлайн, доки ти не запустиш Hermes gateway).
## Крок 7: Знайди свій Discord User ID

Hermes Agent використовує твій Discord User ID, щоб контролювати, хто може взаємодіяти з ботом. Щоб знайти його:

1. Відкрий Discord (десктопний або веб‑додаток).
2. Перейди до **Settings** → **Advanced** → увімкни **Developer Mode**.
3. Закрий налаштування.
4. Клацни правою кнопкою миші на своєму імені користувача (у повідомленні, у списку учасників або у профілі) → **Copy User ID**.

Твій User ID — це довге число, наприклад `284102345871466496`.

:::tip
Developer Mode також дозволяє копіювати **Channel IDs** та **Server IDs** тим же способом — клацни правою кнопкою миші на назві каналу або сервера і вибери **Copy ID**. Тобі знадобиться Channel ID, якщо хочеш вручну встановити домашній канал.
:::
## Крок 8: Налаштування Hermes Agent

### Варіант A: Інтерактивне налаштування (рекомендовано)

Запусти команду керованого налаштування:

```bash
hermes gateway setup
```

Вибери **Discord**, коли буде запит, потім встав свій токен бота та ID користувача, коли запитають.

### Варіант B: Ручна конфігурація

Додай наступне до файлу `~/.hermes/.env`:

```bash
# Required
DISCORD_BOT_TOKEN=your-bot-token
DISCORD_ALLOWED_USERS=284102345871466496

# Multiple allowed users (comma-separated)
# DISCORD_ALLOWED_USERS=284102345871466496,198765432109876543
```

Потім запусти **gateway**:

```bash
hermes gateway
```

Бот має з’явитися в Discord протягом кількох секунд. Надішли йому повідомлення — будь‑то DM або в канал, який він бачить — щоб перевірити.

:::tip
Ти можеш запустити `hermes gateway` у фоні або як службу systemd для постійної роботи. Дивись документацію з розгортання для деталей.
:::
## Довідник конфігурації

Поведінка Discord контролюється двома файлами: **`~/.hermes/.env`** для облікових даних та перемикачів рівня середовища, і **`~/.hermes/config.yaml`** для структурованих налаштувань. Змінні середовища завжди мають пріоритет над значеннями у `config.yaml`, коли обидва встановлені.
### Змінні середовища (`.env`)

| Змінна | Обов’язково | За замовчуванням | Опис |
|----------|----------|----------------|------|
| `DISCORD_BOT_TOKEN` | **Yes** | — | Токен бота з [Discord Developer Portal](https://discord.com/developers/applications). |
| `DISCORD_ALLOWED_USERS` | **Yes** | — | Кома‑розділені ID користувачів Discord, яким дозволено взаємодіяти з ботом. Без цього **або** `DISCORD_ALLOWED_ROLES` шлюз відхиляє всіх користувачів. |
| `DISCORD_ALLOWED_ROLES` | No | — | Кома‑розділені ID ролей Discord. Будь‑який учасник з однією з цих ролей авторизований — логіка **OR** з `DISCORD_ALLOWED_USERS`. Автоматично вмикає **Server Members Intent** під час підключення. Корисно, коли команди модерації змінюються: нові модератори отримують доступ одразу після надання ролі, без необхідності пушити конфіг. |
| `DISCORD_HOME_CHANNEL` | No | — | ID каналу, куди бот надсилає проактивні повідомлення (вивід cron, нагадування, сповіщення). |
| `DISCORD_HOME_CHANNEL_NAME` | No | `"Home"` | Відображувана назва домашнього каналу в логах та статусному виводі. |
| `DISCORD_COMMAND_SYNC_POLICY` | No | `"safe"` | Керує синхронізацією нативних slash‑команд під час старту. `"safe"` порівнює існуючі глобальні команди і оновлює лише зміни, відтворюючи команди, коли зміни метаданих Discord не можна застосувати патчем. `"bulk"` зберігає стару поведінку `tree.sync()`. `"off"` повністю пропускає синхронізацію під час старту. |
| `DISCORD_REQUIRE_MENTION` | No | `true` | Якщо `true`, бот відповідає у серверних каналах лише коли його `@згадують`. Встанови `false`, щоб відповідати на всі повідомлення в кожному каналі. |
| `DISCORD_THREAD_REQUIRE_MENTION` | No | `false` | Якщо `true`, скорочення згадки в темі вимикається — теми підлягають тому ж правилу, що й канали, вимагаючи `@згадку` навіть після того, як бот вже брав участь. Використовуй, коли кілька ботів ділять одну тему і потрібно, щоб кожен реагував лише на явну `@згадку`. |
| `DISCORD_FREE_RESPONSE_CHANNELS` | No | — | Кома‑розділені ID каналів, у яких бот відповідає без вимоги `@згадки`, навіть коли `DISCORD_REQUIRE_MENTION` встановлено в `true`. |
| `DISCORD_IGNORE_NO_MENTION` | No | `true` | Якщо `true`, бот мовчить, коли повідомлення `@згадує` інших користувачів, але не згадує бота. Запобігає втручанню бота в розмови, адресовані іншим людям. Діє лише у серверних каналах, не в DM. |
| `DISCORD_AUTO_THREAD` | No | `true` | Якщо `true`, автоматично створює нову тему для кожної `@згадки` у текстовому каналі, так що кожна розмова ізольована (подібно до поведінки Slack). Повідомлення, що вже знаходяться в темах або DM, не змінюються. |

### Config File (`config.yaml`)

Розділ `discord` у `~/.hermes/config.yaml` відображає вищезазначені змінні оточення. Параметри у `config.yaml` застосовуються як значення за замовчуванням — якщо відповідна змінна оточення вже встановлена, вона має перевагу.

```yaml
# Discord-specific settings
discord:
  require_mention: true           # Require @mention in server channels
  thread_require_mention: false   # If true, require @mention in threads too (multi-bot threads)
  free_response_channels: ""      # Comma-separated channel IDs (or YAML list)
  auto_thread: true               # Auto-create threads on @mention
  reactions: true                 # Add emoji reactions during processing
  ignored_channels: []            # Channel IDs where bot never responds
  no_thread_channels: []          # Channel IDs where bot responds without threading
  history_backfill: true          # Prepend recent channel scrollback on mention (default: true)
  history_backfill_limit: 50      # Max messages to scan backwards (default: 50)
  channel_prompts: {}             # Per-channel ephemeral system prompts
  allow_mentions:                 # What the bot is allowed to ping (safe defaults)
    everyone: false               # @everyone / @here pings (default: false)
    roles: false                  # @role pings (default: false)
    users: true                   # @user pings (default: true)
    replied_user: true            # reply-reference pings the author (default: true)

# Session isolation (applies to all gateway platforms, not just Discord)
group_sessions_per_user: true     # Isolate sessions per user in shared channels
```

#### `discord.require_mention`

**Type:** boolean — **Default:** `true`

Коли ввімкнено, бот відповідає у каналах сервера лише коли його **@згадують** безпосередньо. У приватних повідомленнях (DM) відповідь надходить незалежно від цього налаштування.

#### `discord.thread_require_mention`

**Type:** boolean — **Default:** `false`

За замовчуванням, після того як бот взяв участь у **thread** (автоматично створеному при `@mention` або після відповіді), він продовжує відповідати на кожне наступне повідомлення в цьому thread без потреби знову його **@згадувати**. Це правильне налаштування для розмов один‑на‑один.

У **multi‑bot threads**, де користувачі звертаються до одного бота за ходом, це налаштування стає підступом — кожен інший бот у thread також реагує на кожне повідомлення, витрачаючи кредити та спамлячи канал. Встанови `thread_require_mention: true`, щоб вимкнути скорочення в thread і обмежити thread так само, як канали. Явні `@mentions` і надалі працюватимуть як раніше.

```yaml
discord:
  require_mention: true
  thread_require_mention: true    # multi-bot setup
```

#### `discord.free_response_channels`

**Type:** string or list — **Default:** `""`

Ідентифікатори каналів, у яких бот відповідає на всі повідомлення без потреби `@mention`. Приймає або рядок, розділений комами, або список у форматі YAML:

```yaml
# String format
discord:
  free_response_channels: "1234567890,9876543210"

# List format
discord:
  free_response_channels:
    - 1234567890
    - 9876543210
```

Якщо батьківський канал thread входить до цього списку, і сам thread також стає вільним від згадок.

Канали вільної відповіді також **пропускають автоматичне створення thread** — бот відповідає вбудовано, а не створює новий thread для кожного повідомлення. Це дозволяє використовувати канал як легку чат‑поверхню. Якщо потрібна поведінка з thread, не додавай канал до `free_response_channels` (використовуй звичайний потік `@mention`).

#### `discord.auto_thread`

**Type:** boolean — **Default:** `true`

Коли ввімкнено, кожен `@mention` у звичайному текстовому каналі автоматично створює новий thread для розмови. Це зберігає головний канал чистим і надає кожній розмові власну ізольовану історію сесії. Після створення thread подальші повідомлення в ньому не потребують `@mention` — бот вже бере участь. Встанови [`thread_require_mention`](#discordthread_require_mention) у `true`, щоб вимкнути це скорочення в thread для налаштувань з кількома ботами.

Повідомлення, надіслані в існуючих thread або DM, не залежать від цього параметра. Канали, зазначені у `discord.free_response_channels` або `discord.no_thread_channels`, також обходять автоматичне створення thread і отримують вбудовані відповіді.

#### `discord.reactions`

**Type:** boolean — **Default:** `true`

Керує тим, чи додає бот емодзі‑реакції до повідомлень як візуальний зворотний зв’язок:
- 👀 додається, коли бот починає обробляти твоє повідомлення
- ✅ додається, коли відповідь успішно доставлена
- ❌ додається, якщо під час обробки сталася помилка

Вимкни, якщо реакції заважають або у бота немає дозволу **Add Reactions**.

#### `discord.ignored_channels`

**Type:** string or list — **Default:** `[]`

Ідентифікатори каналів, у яких бот **ніколи** не відповідає, навіть коли його `@mentioned` безпосередньо. Це має найвищий пріоритет — якщо канал є у цьому списку, бот мовчки ігнорує всі повідомлення, незалежно від `require_mention`, `free_response_channels` чи будь‑якого іншого налаштування.

```yaml
# String format
discord:
  ignored_channels: "1234567890,9876543210"

# List format
discord:
  ignored_channels:
    - 1234567890
    - 9876543210
```

Якщо батьківський канал thread входить до цього списку, повідомлення в цьому thread також ігноруються.

#### `discord.no_thread_channels`

**Type:** string or list — **Default:** `[]`

Ідентифікатори каналів, у яких бот відповідає безпосередньо в каналі, а не створює автоматично thread. Це діє лише коли `auto_thread` встановлено у `true` (за замовчуванням). У цих каналах бот відповідає вбудовано, як звичайне повідомлення, а не створює новий thread.

```yaml
discord:
  no_thread_channels:
    - 1234567890  # Bot responds inline here
```

Корисно для каналів, присвячених взаємодії з ботом, де thread лише створювали б зайвий шум.

#### `discord.channel_prompts`

**Type:** mapping — **Default:** `{}`

Каналь‑специфічні ефемерні системні підказки, які інжектуються на кожен хід у відповідному Discord‑каналі або thread без збереження в історії транскрипту.

```yaml
discord:
  channel_prompts:
    "1234567890": |
      This channel is for research tasks. Prefer deep comparisons,
      citations, and concise synthesis.
    "9876543210": |
      This forum is for therapy-style support. Be warm, grounded,
      and non-judgmental.
```

Поведінка:
- Пріоритет мають точні збіги ID thread/каналу.
- Якщо повідомлення надходить у thread або форум і для цього thread немає явного запису, Hermes використовує ID батьківського каналу/форуму.
- Підказки застосовуються ефемерно під час виконання, тому їх зміна впливає на майбутні ходи одразу, не переписуючи історію попередніх сесій.

#### `discord.history_backfill`

**Type:** boolean — **Default:** `true`

Коли ввімкнено, бот відновлює пропущені повідомлення каналу при кожному `@mention`. При `require_mention: true` бот обробляє лише ті повідомлення, які його явно тегують — все інше в каналі невидиме для історії сесії. Backfill сканує історію каналу назад, збираючи повідомлення між останньою відповіддю бота і поточним згадуванням, і додає їх як контекст.

Поведінка за типом поверхні:

- **Канали сервера** (з `require_mention: true`): backfill сканує канал від останньої відповіді бота. Корисно, коли інші учасники писали, доки бот не був згаданий.
- **Threads**: backfill сканує лише thread — `channel.history()` у thread повертає лише повідомлення цього thread, а не батьківського каналу. Це правильний обсяг, бо thread зазвичай самодостатні.
- **DM**: пропускаються. Кожне DM‑повідомлення активує бота, тому історія сесії вже повна — немає проміжку між згадками.
- **Канали вільної відповіді** та **автоматично створені ботом threads**: пропускаються з тієї ж причини — відсутність згадування означає відсутність проміжку.

Сесії, прив’язані до користувача (`group_sessions_per_user: true`, за замовчуванням), також виграють: у користувача може не бути контексту, створеного іншими учасниками каналу, або його власних повідомлень до того, як він тегнув бота. Backfill заповнює обидва прогалини.

```yaml
discord:
  history_backfill: true   # default
```

Щоб вимкнути:

```yaml
discord:
  history_backfill: false
```

> **Note:** Повідомлення, які надходять *під час* обробки бота (між тригером і його відповіддю), не захоплюються. Це прийняте спрощення — користувач може надіслати їх повторно або знову тегнути.

#### `discord.history_backfill_limit`

**Type:** integer — **Default:** `50`

Максимальна кількість повідомлень, які скануються назад при відновленні контексту каналу. На практиці сканування зазвичай зупиняється значно раніше — на останньому повідомленні бота в каналі, що є природною межою між ходами. Це обмеження служить захистом від холодних запусків і довгих прогалин, коли в недавній історії немає попереднього повідомлення бота.

```yaml
discord:
  history_backfill: true
  history_backfill_limit: 50
```

#### `group_sessions_per_user`

**Type:** boolean — **Default:** `true`

Глобальне налаштування шлюзу (не специфічне для Discord), яке визначає, чи отримують користувачі в одному каналі ізольовані історії сесій.

Коли `true`: Alice і Bob, які спілкуються в `#research`, мають окремі розмови з Hermes. Коли `false`: весь канал ділить одну спільну історію та один слот запущеного агента.

```yaml
group_sessions_per_user: true
```

Дивись розділ [Session Model](#session-model-in-discord) вище для повного опису наслідків кожного режиму.

#### `display.tool_progress`

**Type:** string — **Default:** `"all"` — **Values:** `off`, `new`, `all`, `verbose`

Керує тим, чи надсилає бот повідомлення про прогрес у чаті під час обробки (наприклад, “Reading file…”, “Running terminal command…”). Це глобальне налаштування шлюзу, що застосовується до всіх платформ.

```yaml
display:
  tool_progress: "all"    # off | new | all | verbose
```

- `off` — без повідомлень про прогрес
- `new` — показувати лише перший виклик інструменту за хід
- `all` — показувати всі виклики інструментів (скорочено до 40 символів у повідомленнях шлюзу)
- `verbose` — показувати повні деталі виклику інструменту (може створювати довгі повідомлення)

#### `display.tool_progress_command`

**Type:** boolean — **Default:** `false`

Коли ввімкнено, робить доступною слеш‑команду `/verbose` у шлюзі, дозволяючи перемикатися між режимами прогресу інструменту (`off → new → all → verbose → off`) без редагування `config.yaml`.

```yaml
display:
  tool_progress_command: true
```
## Керування доступом до Slash‑команд

За замовчуванням кожен дозволений користувач може запускати будь‑яку Slash‑команду. Щоб розділити ваш список дозволених на **адміністраторів** (повний доступ до Slash‑команд) і **звичайних користувачів** (тільки команди, які ви явно дозволяєте), додайте `allow_admin_from` та `user_allowed_commands` у блок `extra` платформи Discord:

```yaml
gateway:
  platforms:
    discord:
      extra:
        # Existing user allowlist (unchanged)
        allow_from:
          - "123456789012345678"  # admin user ID
          - "999888777666555444"  # regular user ID

        # NEW — admins get all slash commands (built-in + plugin)
        allow_admin_from:
          - "123456789012345678"

        # NEW — non-admin allowed users can only run these slash commands.
        # /help and /whoami are always allowed so users can see their access.
        user_allowed_commands:
          - status
          - model
          - history

        # Optional: separate admin / command lists for server channels
        group_allow_admin_from:
          - "123456789012345678"
        group_user_allowed_commands:
          - status
```

**Поведінка:**

- Користувач у `allow_admin_from` для області (DM або канал сервера) може запускати **будь‑яку** зареєстровану Slash‑команду — вбудовану ТА зареєстровану плагіном — через живий реєстр команд.
- Користувач, який не входить до `allow_admin_from`, може запускати лише команди, зазначені в `user_allowed_commands`, плюс завжди дозволений базовий набір: `/help` та `/whoami`.
- Звичайний чат (повідомлення без Slash) не змінюється. Користувачі без прав адміністратора можуть й надалі спілкуватися з агентом у звичному режимі; вони просто не можуть ініціювати довільні команди.
- **Зворотна сумісність:** якщо `allow_admin_from` не встановлено для області, фільтрація Slash‑команд вимикається для цієї області. Існуючі інсталяції продовжують працювати без змін.
- Статус адміністратора в DM не означає статус адміністратора в каналі сервера. Кожна область має власний список адміністраторів.

Використай `/whoami`, щоб побачити активну область, твій рівень (admin / user / unrestricted) та які Slash‑команди ти можеш запускати.
## Інтерактивний вибір моделі

Надішли `/model` без аргументів у каналі Discord, щоб відкрити випадаючий вибір моделі:

1. **Вибір провайдера** — випадаючий список (Select), що показує доступних провайдерів (до 25).
2. **Вибір моделі** — другий випадаючий список з моделями для обраного провайдера (до 25).

Вибірник завершує роботу через 120 секунд. Тільки авторизовані користувачі (ті, що в `DISCORD_ALLOWED_USERS`) можуть взаємодіяти з ним. Якщо ти знаєш назву моделі, введи `/model <name>` безпосередньо.
## Native Slash Commands for Skills

Hermes автоматично реєструє встановлені **skills** як native Discord Application Commands. Це означає, що skills з’являються в меню автодоповнення Discord `/` разом із вбудованими командами.

- Кожен skill стає Discord slash‑командою (наприклад, `/code-review`, `/ascii-art`)
- Skills приймають необов’язковий параметр рядка `args`
- Discord має обмеження у 100 application commands на бота — якщо у вас більше skills, ніж доступних слотів, зайві skills пропускаються з попередженням у логах
- Skills реєструються під час запуску бота разом із вбудованими командами, такими як `/model`, `/reset` та `/background`

Ніякої додаткової конфігурації не потрібно — будь‑який skill, встановлений за допомогою `hermes skills install`, автоматично реєструється як Discord slash‑команда під час наступного перезапуску шлюзу.

### Disabling Slash Command Registration

Якщо ти запускаєш кілька шлюзів Hermes проти одного і того ж Discord‑застосунку (наприклад, staging + production), лише один з них повинен володіти глобальною реєстрацією slash‑команд — інакше останній запуск переважає, і реєстрації «перемикаються». Вимкни реєстрацію slash‑команд на «follower» шлюзі:

```yaml
gateway:
  platforms:
    discord:
      extra:
        slash_commands: false   # default: true
```

Залишивши це значення `true` на «primary» шлюзі, ти зберігаєш звичайну поведінку — глобальні `/`‑меню команди для вбудованих skills та встановлених skills.
## Надсилання медіа (`send_message` + `MEDIA:` теги)

Адаптер Discord підтримує рідне завантаження файлів для будь‑якого поширеного типу медіа за допомогою інструмента `send_message` та інлайн‑тегів `MEDIA:/path/to/file`, які генерує агент:

| Тип | Як доставляється |
|---|---|
| Зображення (PNG/JPG/WebP) | Рідне вкладення зображення Discord з інлайн‑переглядом |
| Анімовані GIF | `send_animation` завантажує як `animation.gif`, тож Discord відтворює його інлайн (не як статичну мініатюру) |
| Відео (MP4/MOV) | `send_video` — рідний відеоплеєр |
| Аудіо / Голос | `send_voice` — рідне голосове повідомлення, коли можливо, інакше вкладення файлу |
| Документи (PDF/ZIP/docx/тощо) | `send_document` — рідне вкладення з кнопкою завантаження |

Обмеження розміру завантаження в Discord залежить від рівня бусту сервера (25 МБ безкоштовно, до 500 МБ). Якщо Hermes отримує HTTP 413, адаптер переходить до запасного варіанту — посилання на локальний шлях кешу, а не «тихо» падає.
## Отримання довільних типів файлів

За замовчуванням бот кешує завантаження, які відповідають вбудованому білому списку — зображення, аудіо, відео, PDF, текст/markdown/csv/log, JSON/XML/YAML/TOML, zip, docx/xlsx/pptx. Будь‑що інше (наприклад `.wav`, `.bin`, дамп з власним розширенням) реєструється як `Unsupported document type` і відкидається ще до того, як агент його побачить.

Щоб приймати довільні типи файлів, увімкни `discord.allow_any_attachment`:

```yaml
discord:
  allow_any_attachment: true
  # Optional — raise/disable the per-file size cap. Default is 32 MiB.
  # The whole file is held in memory while being cached, so unlimited
  # uploads carry a real memory cost.
  max_attachment_bytes: 33554432   # bytes; 0 = unlimited
```

Коли прапорець увімкнено, будь‑який завантажений файл завантажується, кешується у `~/.hermes/cache/documents/` і передається агенту як подія повідомлення типу `DOCUMENT` з MIME `application/octet-stream`. Агент отримує нотатку контексту, що вказує на локальний шлях (автоперекладений для Docker/Modal пісочних терміналів через `to_agent_visible_cache_path`) і може дослідити файл за допомогою `terminal` (`ffprobe`, `unzip`, `file`, `strings` тощо) або `read_file`. Тіло файлу **не** вбудовується у підказку — лише шлях — тому бінарні завантаження не збільшують вікно контексту.

Відомі текстові формати, вже включені у білий список (`.txt`, `.md`, `.log`), і надалі автоматично інжектуються до 100 КіБ; ця поведінка не змінюється при увімкненому прапорці.

Еквівалентні змінні середовища: `DISCORD_ALLOW_ANY_ATTACHMENT=true` і `DISCORD_MAX_ATTACHMENT_BYTES=33554432` (або `0` для відсутності обмеження).

:::warning Витрати пам’яті при необмеженій кількості
Вимкнення обмеження розміру (`max_attachment_bytes: 0`) означає, що користувач може кинути багатогігабайтовий файл боту, і шлюз буде буферизувати його в пам’яті під час кешування на диск. Встановлюй це лише в довірених однокористувацьких інсталяціях. Для спільних ботів залишай значення за замовчуванням — 32 MiB, або підвищуй його обережно.
:::
## Interactive Prompts (clarify)

Коли агент викликає інструмент `clarify` — щоб запитати, який підхід ти віддаєш перевагу, отримати зворотний зв’язок після виконання завдання або перевірити рішення перед нетривіальним вибором — Discord відображає питання **по одній кнопці на кожний варіант**:

> Який фреймворк використати для дашборду?
>
> [1. Next.js] [2. Remix] [3. Astro] [Other (type answer)]

Натисни кнопку з номером, щоб відповісти, або натисни **Other**, щоб ввести вільну відповідь (наступне повідомлення, яке ти надішлеш у цьому каналі, стане відповіддю). Відкриті виклики `clarify` (без попередньо заданих варіантів) не показують кнопки і просто захоплюють твоє наступне повідомлення.

Кнопки вимикаються після того, як вибір зроблено, щоб дублюючі кліки не призвели до подвійного розв’язання запиту. Налаштуй тайм‑аут відповіді через `agent.clarify_timeout` у `~/.hermes/config.yaml` (за замовчуванням `600` секунд). Якщо ти не відповіси протягом тайм‑ауту, агент розблокується за допомогою sentinel‑повідомлення і адаптується, а не зависне.
## Канал «домашній»

Ти можеш призначити «домашній канал», куди бот надсилатиме проактивні повідомлення (наприклад, вивід результатів cron‑завдань, нагадування та сповіщення). Є два способи його встановити:

### За допомогою слеш‑команди

Введи `/sethome` у будь‑якому каналі Discord, у якому присутній бот. Цей канал стане домашнім каналом.

### Ручна конфігурація

Додай наступне до свого `~/.hermes/.env`:

```bash
DISCORD_HOME_CHANNEL=123456789012345678
DISCORD_HOME_CHANNEL_NAME="#bot-updates"
```

Замінити ID на фактичний ідентифікатор каналу (правою кнопкою миші → Copy Channel ID при увімкненому режимі розробника).
## Голосові повідомлення

Hermes Agent підтримує голосові повідомлення Discord:

- **Вхідні голосові повідомлення** автоматично транскрибуються за допомогою налаштованого провайдера STT: локальний `faster-whisper` (без ключа), Groq Whisper (`GROQ_API_KEY`) або OpenAI Whisper (`VOICE_TOOLS_OPENAI_KEY`).
- **Текст‑у‑мову**: використай `/voice tts`, щоб бот надсилатиме голосові відповіді разом із текстовими.
- **Голосові канали Discord**: Hermes може приєднуватися до голосового каналу, слухати, як користувачі говорять, і відповідати у цьому каналі.

Для повного налаштування та посібника з експлуатації дивись:
- [Voice Mode](/user-guide/features/voice-mode)
- [Use Voice Mode with Hermes](/guides/use-voice-mode-with-hermes)
## Канали форуму

Discord‑канали форуму (тип 15) не приймають прямі повідомлення — кожен пост у форумі має бути темою. Hermes автоматично виявляє канали форуму і створює нову тему, коли потрібно надіслати повідомлення, тому `send_message`, TTS, зображення, голосові повідомлення та вкладення файлів працюють без спеціальної обробки агентом.

- **Назва теми** формується з першого рядка повідомлення (префікс заголовка markdown видаляється, обмеження — 100 символів). Якщо повідомлення містить лише вкладення, як назва теми використовується ім’я файлу (запасна назва теми).
- **Вкладення** передаються разом із стартовим повідомленням нової теми — без окремого кроку завантаження, без часткових надсилань.
- **Один виклик — одна тема**: кожне надсилання у форум створює нову тему. Подальші надсилання в той самий форум, отже, будуть створювати окремі теми.
- **Виявлення тришарові**: спочатку кеш каталогу каналів, потім локальний кеш процесу, і в останню чергу живий запит `GET /channels/{id}` як останній варіант (результат якого потім мемоізується на весь час роботи процесу).

Оновлення каталогу (`/channels refresh` на платформах, які це підтримують, або перезапуск шлюзу) заповнює кеш усіма каналами форуму, створеними після запуску бота.
## Усунення проблем

### Бот онлайн, але не відповідає на повідомлення

**Причина**: Message Content Intent вимкнено.

**Виправлення**: Перейди до [Developer Portal](https://discord.com/developers/applications) → твій додаток → Bot → Privileged Gateway Intents → увімкни **Message Content Intent** → Save Changes. Перезапусти шлюз.

### Помилка «Disallowed Intents» під час запуску

**Причина**: Твій код запитує intents, які не ввімкнені в Developer Portal.

**Виправлення**: Увімкни всі три Privileged Gateway Intents (Presence, Server Members, Message Content) у налаштуваннях Bot, потім перезапусти.

### Бот не бачить повідомлення в певному каналі

**Причина**: Роль бота не має дозволу переглядати цей канал.

**Виправлення**: У Discord відкрий налаштування каналу → Permissions → додай роль бота з увімкненими **View Channel** та **Read Message History**.

### Помилки 403 Forbidden

**Причина**: У бота відсутні необхідні дозволи.

**Виправлення**: Перезапроси запрошення бота з правильними дозволами, використовуючи URL зі кроку 5, або вручну відкоригуй дозволи ролі бота в Server Settings → Roles.

### Бот офлайн

**Причина**: Шлюз Hermes не працює або токен неправильний.

**Виправлення**: Перевір, що `hermes gateway` запущений. Переконайся, що `DISCORD_BOT_TOKEN` у твоєму файлі `.env` правильний. Якщо ти нещодавно скинув токен, онови його.

### «User not allowed» / Бот ігнорує тебе

**Причина**: Твій User ID відсутній у `DISCORD_ALLOWED_USERS`.

**Виправлення**: Додай свій User ID до `DISCORD_ALLOWED_USERS` у `~/.hermes/.env` і перезапусти шлюз.

### Люди в одному каналі діляться контекстом несподівано

**Причина**: `group_sessions_per_user` вимкнено, або платформа не може надати User ID для повідомлень у цьому контексті.

**Виправлення**: Встанови це у `~/.hermes/config.yaml` і перезапусти шлюз:

```yaml
group_sessions_per_user: true
```

Якщо ти навмисно хочеш спільну розмову в кімнаті, залиш це вимкненим — просто очікуй спільну історію транскрипту та спільну поведінку переривань.
## Security

:::warning
Завжди встановлюй `DISCORD_ALLOWED_USERS` (або `DISCORD_ALLOWED_ROLES`), щоб обмежити, хто може взаємодіяти з ботом. Якщо жодної змінної не вказано, шлюз за замовчуванням відхиляє всіх користувачів як захисний захід. Дозволяй лише людей, яким довіряєш — авторизовані користувачі мають повний доступ до можливостей агента, включно з використанням інструментів та доступом до системи.
:::

### Role‑Based Access Control

Для серверів, де доступ керується ролями, а не окремими списками користувачів (команди модераторів, служба підтримки, внутрішні інструменти), використай `DISCORD_ALLOWED_ROLES` — кома‑розділений список ідентифікаторів ролей. Будь‑який учасник, який має одну з цих ролей, буде авторизований.

```bash
# ~/.hermes/.env — works alongside or instead of DISCORD_ALLOWED_USERS
DISCORD_ALLOWED_ROLES=987654321098765432,876543210987654321
```

Семантика:

- **OR з білим списком користувачів.** Користувач авторизований, якщо його ID є в `DISCORD_ALLOWED_USERS` **або** він має будь‑яку роль із `DISCORD_ALLOWED_ROLES`.
- **Server Members Intent автоматично вмикається.** Коли встановлено `DISCORD_ALLOWED_ROLES`, бот активує інтенцію Members під час підключення — це необхідно, щоб Discord надсилав інформацію про ролі разом із записами учасників.
- **ID ролей, а не назви.** Отримай їх у Discord: **User Settings → Advanced → Developer Mode ON**, потім клацни правою кнопкою миші по ролі → **Copy Role ID**.
- **DM fallback.** У приватних повідомленнях перевірка ролі сканує спільні гільдії; користувач з дозволеною роллю в будь‑якому спільному сервері також авторизований у DM.

Це рекомендований підхід, коли команда модераторів часто змінюється — нові модератори отримують доступ одразу після надання ролі, без редагування `.env` чи перезапуску шлюзу.

### Mention Control

За замовчуванням Hermes блокує бот від згадок `@everyone`, `@here` та ролей, навіть якщо їх містить відповідь бота. Це запобігає випадковому спаму всього серверу через погано сформульований запит або віддзеркалений вміст користувача. Окремі згадки `@user` та згадки у відповідях (маленька «chip» «replying to…») залишаються активними, щоб звичайна розмова працювала.

Ти можеш пом’якшити ці налаштування через змінні середовища або `config.yaml`:

```yaml
# ~/.hermes/config.yaml
discord:
  allow_mentions:
    everyone: false      # allow the bot to ping @everyone / @here
    roles: false         # allow the bot to ping @role mentions
    users: true          # allow the bot to ping individual @users
    replied_user: true   # ping the author when replying to their message
```

```bash
# ~/.hermes/.env — env vars win over config.yaml
DISCORD_ALLOW_MENTION_EVERYONE=false
DISCORD_ALLOW_MENTION_ROLES=false
DISCORD_ALLOW_MENTION_USERS=true
DISCORD_ALLOW_MENTION_REPLIED_USER=true
```

:::tip
Залишай `everyone` і `roles` встановленими в `false`, якщо точно не знаєш, навіщо вони потрібні. LLM дуже легко може згенерувати рядок `@everyone` у звичайній відповіді; без цього захисту це повідомлення сповістить усіх учасників твого серверу.
:::

Для отримання додаткової інформації про захист розгортання Hermes Agent, переглянь [Security Guide](../security.md).
