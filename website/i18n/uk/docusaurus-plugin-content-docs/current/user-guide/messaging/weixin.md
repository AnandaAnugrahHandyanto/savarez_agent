# Weixin (WeChat)

Підключи Hermes до [WeChat](https://weixin.qq.com/) (微信), персональної платформи обміну повідомленнями від Tencent. Адаптер використовує **iLink Bot API** Tencent для персональних облікових записів WeChat — це відрізняється від WeCom (Enterprise WeChat). Повідомлення доставляються через long‑polling, тому публічний кінцевий пункт або webhook не потрібні.

:::info
Цей адаптер призначений для **персональних облікових записів WeChat** (微信). Якщо потрібен корпоративний/підприємницький WeChat, дивись [адаптер WeCom](./wecom.md).
:::

:::warning iLink bot identity — ordinary WeChat groups may not work
QR‑вхід підключає Hermes до **iLink bot identity** (наприклад `a5ace6fd482e@im.bot`), **не** до повністю скриптового звичайного персонального облікового запису WeChat. Наслідки:

- iLink bot identity зазвичай **не може бути запрошений у звичайні групи WeChat** так, як це робить звичайний контакт.
- iLink, як правило, **не передає події звичайних груп WeChat** (включаючи `@`‑згадки персонального облікового запису, використаного для QR‑входу) у шлюз для більшості бот‑типів облікових записів.
- `@`‑згадка персонального облікового запису WeChat, який сканує QR‑код, **не є** тією ж самою, що `@`‑згадка iLink bot — бот має окрему ідентичність.
- Налаштування `WEIXIN_GROUP_POLICY` / `WEIXIN_GROUP_ALLOWED_USERS`, наведені нижче, діють лише тоді, коли iLink фактично повертає події груп для вашого типу облікового запису. Якщо цього не відбувається, групові повідомлення ніколи не досягнуть Hermes, незалежно від політики.

На практиці більшість розгортань отримують лише DMs до iLink bot і працюють надійно. Якщо доставка в групи не працює після налаштування, обмеження знаходиться на боці iLink, а не в Hermes. Шлюз записує `WARNING` під час запуску, коли `WEIXIN_GROUP_POLICY` встановлено не в `disabled`.
:::
## Передумови

- Особистий обліковий запис WeChat
- Пакети Python: `aiohttp` і `cryptography`
- Відображення QR‑коду в терміналі включено, коли Hermes встановлено з додатковим модулем `messaging`

Встанови необхідні залежності:

```bash
pip install aiohttp cryptography
# Optional: for terminal QR code display
pip install hermes-agent[messaging]
```
## Налаштування

### 1. Запусти Майстра Налаштування

Найпростіший спосіб підключити свій обліковий запис WeChat – через інтерактивний майстер налаштування:

```bash
hermes gateway setup
```

Вибери **Weixin**, коли буде запит. Майстер виконає:

1. Запит QR‑коду у iLink Bot API
2. Виведе QR‑код у твоєму терміналі (або надасть URL)
3. Очікуватиме, доки ти відскануєш QR‑код за допомогою мобільного додатку WeChat
4. Запитає підтвердження входу на твоєму телефоні
5. Автоматично збереже облікові дані облікового запису у `~/.hermes/weixin/accounts/`

Після підтвердження ти побачиш повідомлення типу:

```
微信连接成功，account_id=your-account-id
```

Майстер зберігає `account_id`, `token` та `base_url`, тому їх не потрібно налаштовувати вручну.

### 2. Налаштуй змінні середовища

Після початкового входу через QR‑код встанови принаймні ID облікового запису у `~/.hermes/.env`:

```bash
WEIXIN_ACCOUNT_ID=your-account-id

# Optional: override the token (normally auto-saved from QR login)
# WEIXIN_TOKEN=your-bot-token

# Optional: restrict access
WEIXIN_DM_POLICY=open
WEIXIN_ALLOWED_USERS=user_id_1,user_id_2

# Optional: restore legacy multiline splitting behavior
# WEIXIN_SPLIT_MULTILINE_MESSAGES=true

# Optional: home channel for cron/notifications
WEIXIN_HOME_CHANNEL=chat_id
WEIXIN_HOME_CHANNEL_NAME=Home
```

### 3. Запусти gateway

```bash
hermes gateway
```

Адаптер відновить збережені облікові дані, підключиться до iLink API та розпочне довге опитування повідомлень.
## Функції

- **Long-poll transport** — не потрібен публічний кінцевий пункт, webhook або WebSocket
- **QR code login** — налаштування підключення скануванням за допомогою `hermes gateway setup`
- **DM messaging** — налаштовувані політики доступу; груповий обмін повідомленнями залежить від того, чи iLink фактично доставляє групові події для підключеної ідентичності (часто це не так для облікових записів iLink‑бота — дивись попередження вище)
- **Media support** — зображення, відео, файли та голосові повідомлення
- **AES-128-ECB encrypted CDN** — автоматичне шифрування/дешифрування всіх медіапередач
- **Context token persistence** — збереження токену контексту на диску для безперервності відповідей після перезапуску
- **Markdown formatting** — зберігає Markdown, включаючи заголовки, таблиці та блоки коду, тому клієнти WeChat, що підтримують Markdown, можуть відображати його нативно
- **Smart message chunking** — повідомлення залишаються в одній бульбашці, доки не перевищують ліміт; лише надмірно великі навантаження розбиваються на логічні частини
- **Typing indicators** — показує статус «typing…» у клієнті WeChat, доки агент обробляє
- **SSRF protection** — URL‑адреси зовнішніх медіа перевіряються перед завантаженням
- **Message deduplication** — 5‑хвилинне ковзне вікно запобігає подвійній обробці
- **Automatic retry with backoff** — відновлює роботу після тимчасових помилок API
## Параметри конфігурації

Встанови їх у `config.yaml` під `platforms.weixin.extra`:

| Ключ | За замовчуванням | Опис |
|-----|------------------|------|
| `account_id` | — | iLink Bot account ID (обов’язково) |
| `token` | — | iLink Bot token (обов’язково, автоматично зберігається після QR‑входу) |
| `base_url` | `https://ilinkai.weixin.qq.com` | базовий URL iLink API |
| `cdn_base_url` | `https://novac2c.cdn.weixin.qq.com/c2c` | базовий URL CDN для передачі медіа |
| `dm_policy` | `open` | доступ до DM: `open`, `allowlist`, `disabled`, `pairing` |
| `group_policy` | `disabled` | доступ до груп: `open`, `allowlist`, `disabled` |
| `allow_from` | `[]` | ID користувачів, яким дозволено надсилати DM (коли `dm_policy=allowlist`) |
| `group_allow_from` | `[]` | ID груп, яким дозволено (коли `group_policy=allowlist`) |
| `split_multiline_messages` | `false` | Якщо `true`, розбиває багаторядкові відповіді на кілька чат‑повідомлень (застаріла поведінка). Якщо `false`, залишає багаторядкові відповіді в одному повідомленні, якщо тільки вони не перевищують ліміт довжини. |
## Політики доступу

### DM‑політика

Керує тим, хто може надсилати прямі повідомлення (DM) боту:

| Value | Behavior |
|-------|----------|
| `open` | Будь‑хто може надсилати DM боту (за замовчуванням) |
| `allowlist` | Тільки користувачі з ID, зазначеними в `allow_from`, можуть надсилати DM |
| `disabled` | Усі DM ігноруються |
| `pairing` | Режим парування (для початкового налаштування) |

```bash
WEIXIN_DM_POLICY=allowlist
WEIXIN_ALLOWED_USERS=user_id_1,user_id_2
```

`WEIXIN_ALLOWED_USERS` — це **вхідний фільтр**, а не система запрошень. QR‑login підключає одну ідентичність iLink‑бота до Hermes. Інші користувачі не сканують QR‑код Hermes своїми обліковими записами; вони повинні надсилати повідомлення підключеному iLink‑боту/контакту через WeChat, і Hermes обробить DM лише тоді, коли Weixin‑ID відправника присутній у `WEIXIN_ALLOWED_USERS`.

Практичний порядок налаштування:

1. Одноразово спарити Hermes за допомогою `hermes gateway setup` і зафіксувати підключений обліковий запис iLink‑бота.
2. Кожен дозволений користувач надсилає пряме повідомлення цьому боту/контакту.
3. Прочитати ID відправника/користувача з журналів шлюзу або з корисного навантаження вхідної події.
4. Додати ці ID до `WEIXIN_ALLOWED_USERS`, після чого перезапустити шлюз.

Якщо лише обліковий запис, який сканував QR‑код, може спілкуватися з Hermes, переконайся, що інші користувачі надсилають повідомлення саме ідентичності iLink‑бота, а не особистому обліковому запису WeChat, який виконав QR‑login. iLink‑бот — це окрема ідентичність, і звичайна маршрутизація контактів/груп у WeChat може бути обмежена поведінкою iLink від Tencent.

### Group‑політика

Керує тим, у яких групах бот відповідає **коли iLink доставляє події групи для підключеної ідентичності**. Для iLink‑ботів, що входять через QR‑login (наприклад, `...@im.bot`), події груп зазвичай не доставляються, тому ця політика може не мати ефекту — дивись попередження про обмеження iLink‑бота у верхній частині сторінки.

| Value | Behavior |
|-------|----------|
| `open` | Бот відповідає у всіх групах (якщо події доставлені) |
| `allowlist` | Бот відповідає лише у групах з ID, зазначеними в `group_allow_from` (якщо події доставлені) |
| `disabled` | Усі групові повідомлення ігноруються (за замовчуванням) |

```bash
WEIXIN_GROUP_POLICY=allowlist
# NOTE: this is a comma-separated list of group chat IDs, NOT member user IDs,
# despite the variable name containing "USERS". Keep this in mind when configuring.
WEIXIN_GROUP_ALLOWED_USERS=group_id_1,group_id_2
```

:::note
Політика груп за замовчуванням — `disabled` для Weixin (на відміну від WeCom, де за замовчуванням `open`). Це навмисно — особисті облікові записи WeChat можуть бути у багатьох групах, а iLink‑боти, як правило, не отримують звичайні групові повідомлення WeChat. Шлюз виводить `WARNING` під час запуску, якщо встановити `WEIXIN_GROUP_POLICY` на будь‑яке значення, відмінне від `disabled`.
:::
## Підтримка медіа

### Вхідні (отримання)

Адаптер отримує медіа‑вкладення від користувачів, завантажує їх з CDN WeChat, розшифровує та кешує локально для обробки агентом:

| Тип | Як обробляється |
|------|-----------------|
| **Зображення** | Завантажуються, розшифровуються AES‑шифром і кешуються як JPEG. |
| **Відео** | Завантажуються, розшифровуються AES‑шифром і кешуються як MP4. |
| **Файли** | Завантажуються, розшифровуються AES‑шифром і кешуються. Оригінальна назва файлу зберігається. |
| **Голос** | Якщо доступна текстова транскрипція, вона витягується як текст. Інакше аудіо (формат SILK) завантажується і кешується. |

**Цитовані повідомлення:** Медійні дані з цитованих (відповідних) повідомлень також витягуються, щоб агент мав контекст того, на що користувач відповідає.

### AES-128-ECB зашифрований CDN

Медіафайли WeChat передаються через зашифрований CDN. Адаптер обробляє це прозоро:

- **Вхідні:** Зашифровані медіа завантажуються з CDN за допомогою URL‑ів `encrypted_query_param`, потім розшифровуються AES‑128‑ECB за допомогою ключа, що надається в корисному навантаженні повідомлення.
- **Вихідні:** Файли шифруються локально випадковим ключем AES‑128‑ECB, завантажуються до CDN, і зашифроване посилання включається у вихідне повідомлення.
- Ключ AES має 16 байт (128 біт). Ключі можуть надходити у вигляді raw base64 або hex‑кодування — адаптер підтримує обидва формати.
- Потрібен пакет Python `cryptography`.

Налаштування не потрібне — шифрування та розшифрування виконуються автоматично.

### Вихідні (надсилання)

| Метод | Що надсилає |
|--------|--------------|
| `send` | Текстові повідомлення з форматуванням Markdown |
| `send_image` / `send_image_file` | Нативні повідомлення‑зображення (через завантаження в CDN) |
| `send_document` | Вкладення файлів (через завантаження в CDN) |
| `send_video` | Відеоповідомлення (через завантаження в CDN) |

Усі вихідні медіа проходять через процес завантаження в зашифрований CDN:

1. Згенерувати випадковий ключ AES‑128.
2. Зашифрувати файл за допомогою AES‑128‑ECB + паддінг PKCS#7.
3. Запитати URL для завантаження у iLink API (`getuploadurl`).
4. Завантажити зашифрований файл у CDN.
5. Надіслати повідомлення з зашифрованим медіа‑посиланням.
## Персистентність токену контексту

API iLink Bot вимагає, щоб `context_token` повертався у відповіді з кожним вихідним повідомленням для даного співрозмовника. Адаптер підтримує сховище токену контексту на диску:

- Токени зберігаються окремо для кожного `account+peer` у файлі `~/.hermes/weixin/accounts/<account_id>.context-tokens.json`
- При запуску раніше збережені токени відновлюються
- Кожне вхідне повідомлення оновлює збережений токен для відправника
- Вихідні повідомлення автоматично включають останній токен контексту

Це забезпечує безперервність відповідей навіть після перезапуску **gateway**.
## Форматування Markdown

Клієнти WeChat, підключені через iLink Bot API, можуть відображати Markdown безпосередньо, тому адаптер зберігає Markdown замість його переписування:

- **Headers** залишаються як заголовки Markdown (`#`, `##`, …)
- **Tables** залишаються як таблиці Markdown
- **Code fences** залишаються як блоки коду з огороджувачами
- **Excessive blank lines** стискаються до подвійних переносів рядка поза блоками коду
## Розбиття повідомлень

Повідомлення надсилаються як єдине чат‑повідомлення, якщо вони вкладаються в межі платформи. Тільки надмірно великі повідомлення розбиваються для доставки:

- Максимальна довжина повідомлення: **4000 символів**
- Повідомлення, що не перевищують межу, залишаються цілісними, навіть якщо містять кілька абзаців або переноси рядків
- Надмірно великі повідомлення розбиваються на логічних межах (абзаци, порожні рядки, блоки коду)
- Блоки коду зберігаються цілісними, коли це можливо (ніколи не розбиваються всередині блоку, якщо лише сам блок не перевищує межу)
- Надмірно великі окремі блоки повертаються до базової логіки обрізання адаптера
- Затримка **0.3 s** між частинами запобігає відкиданню через обмеження швидкості WeChat, коли надсилаються кілька частин
## Індикатори набору

Адаптер показує статус набору в клієнті WeChat:

1. Коли надходить повідомлення, адаптер отримує `typing_ticket` через API `getconfig`
2. Тікети набору кешуються 10 хвилин для кожного користувача
3. `send_typing` надсилає сигнал початку набору; `stop_typing` надсилає сигнал завершення набору
4. gateway автоматично активує індикатори набору, доки агент обробляє повідомлення
## Long-Poll Connection

Адаптер використовує HTTP long‑polling (не WebSocket) для отримання повідомлень:

### Як це працює

1. **Connect:** Перевіряє облікові дані та запускає цикл опитування
2. **Poll:** Викликає `getupdates` з тайм‑аутом 35 секунд; сервер утримує запит, доки не надійдуть повідомлення або не сплине тайм‑аут
3. **Dispatch:** Вхідні повідомлення розподіляються одночасно за допомогою `asyncio.create_task`
4. **Sync buffer:** Постійний курсор синхронізації (`get_updates_buf`) зберігається на диск, щоб адаптер відновився з правильного місця після перезапуску

### Поведінка при повторних спробах

При помилках API адаптер застосовує просту стратегію повторних спроб:

| Condition | Behavior |
|-----------|----------|
| Transient error (1st–2nd) | Повторити через 2 секунди |
| Repeated errors (3+) | Затримка 30 секунд, потім скинути лічильник |
| Session expired (`errcode=-14`) | Пауза 10 хвилин (може знадобитися повторний вхід) |
| Timeout | Негайно повторити опитування (звичайна поведінка long‑poll) |

### Дедуплікація

Вхідні повідомлення дедуплікуються за ідентифікаторами повідомлень у вікні 5 хвилин. Це запобігає подвійній обробці під час мережевих збоїв або накладання відповідей опитування.

### Блокування токену

Лише один екземпляр шлюзу **Weixin** може використовувати даний токен одночасно. Адаптер отримує scoped‑lock під час запуску та звільняє його під час завершення роботи. Якщо інший шлюз вже використовує той самий токен, запуск завершується помилкою з інформативним повідомленням.

## Устранення проблем

| Проблема | Виправлення |
|---------|------------|
| `Weixin startup failed: aiohttp and cryptography are required` | Встанови обидва: `pip install aiohttp cryptography` |
| `Weixin startup failed: WEIXIN_TOKEN is required` | Запусти `hermes gateway setup` для завершення QR‑логіну або встанови `WEIXIN_TOKEN` вручну |
| `Weixin startup failed: WEIXIN_ACCOUNT_ID is required` | Встанови `WEIXIN_ACCOUNT_ID` у своєму `.env` або запусти `hermes gateway setup` |
| `Another local Hermes gateway is already using this Weixin token` | Спершу зупини інший екземпляр шлюзу — дозволено лише один поллер на токен |
| Session expired (`errcode=-14`) | Твоя сесія входу закінчилася. Запусти `hermes gateway setup` ще раз, щоб відсканувати новий QR‑код |
| QR code expired during setup | QR‑код автоматично оновлюється до 3 разів. Якщо він продовжує закінчуватись, перевір підключення до мережі |
| Bot doesn't respond to DMs | Перевір `WEIXIN_DM_POLICY` — якщо встановлено `allowlist`, відправник має бути в `WEIXIN_ALLOWED_USERS` |
| Bot ignores group messages | Політика групи за замовчуванням `disabled`. Встанови `WEIXIN_GROUP_POLICY=open` або `allowlist` — але зауваж, що iLink‑боти (`...@im.bot`) зазвичай не можуть отримувати звичайні повідомлення груп WeChat. Якщо в логах шлюзу немає raw‑подій вхідних групових повідомлень, обмеження знаходиться на стороні iLink, а не в Hermes. |
| Media download/upload fails | Переконайся, що `cryptography` встановлено. Перевір доступ до `novac2c.cdn.weixin.qq.com` |
| `Blocked unsafe URL (SSRF protection)` | URL вихідних медіа вказує на приватну/внутрішню адресу. Дозволені лише публічні URL |
| Voice messages show as text | Якщо WeChat надає транскрипцію, адаптер використовує текст. Це очікувана поведінка |
| Messages appear duplicated | Адаптер видаляє дублікати за ідентифікатором повідомлення. Якщо бачиш дублікати, перевір, чи не запущено кілька екземплярів шлюзу |
| `iLink POST ... HTTP 4xx/5xx` | Помилка API сервісу iLink. Перевір дійсність токену та підключення до мережі |
| Terminal QR code doesn't render | Перевстанови з додатковим пакетом messaging: `pip install hermes-agent[messaging]`. Або відкрий URL, виведений над QR‑кодом |
