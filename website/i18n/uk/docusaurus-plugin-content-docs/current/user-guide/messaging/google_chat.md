# Налаштування Google Chat

Підключи Hermes Agent до Google Chat як бота. Інтеграція використовує pull‑підписки Cloud Pub/Sub для вхідних подій і Chat REST API для вихідних повідомлень. Ергономічно еквівалентно Slack Socket Mode або Telegram long‑polling: твій Hermes‑процес не потребує публічної URL‑адреси, тунелю чи TLS‑сертифіката. Він підключається, автентифікується та слухає підписку — так само, як бот Telegram слухає токен.

> Запусти `hermes gateway setup` і обери **Google Chat** для покрокового майстра.

:::note Workspace edition
Google Chat є частиною Google Workspace. Ти можеш використовувати цю інтеграцію з
особистим Workspace (`@yourdomain.com`, зареєстрованим через Google) або робочим
Workspace, де маєш права адміністратора для публікації додатку. Облікові записи лише
з Gmail не можуть розміщувати додатки Chat.
:::
## Огляд

| Компонент | Значення |
|-----------|----------|
| **Бібліотеки** | `google-cloud-pubsub`, `google-api-python-client`, `google-auth` |
| **Вхідний транспорт** | Cloud Pub/Sub pull subscription (no public endpoint) |
| **Вихідний транспорт** | Chat REST API (`chat.googleapis.com`) |
| **Аутентифікація** | Service Account JSON with `roles/pubsub.subscriber` on the subscription |
| **Ідентифікація користувача** | Chat resource names (`users/{id}`) + email |

---
## Крок 1: Створити або вибрати проєкт GCP

Тобі потрібен проєкт Google Cloud, щоб розмістити тему Pub/Sub. Якщо у тебе його ще немає, створи його у [console.cloud.google.com](https://console.cloud.google.com) — особисті облікові записи отримують безкоштовний рівень, який легко покриває трафік бота.

Запиши ідентифікатор проєкту (наприклад, `my-chat-bot-123`). Ти використаєш його в кожному наступному кроці.

---
## Крок 2: Увімкнути два API

У консолі перейдіть до **APIs & Services → Library** і увімкніть:

- **Google Chat API**
- **Cloud Pub/Sub API**

Обидва безкоштовні для обсягів, які генерує твій особистий бот.

---
## Крок 3: Створити обліковий запис служби

**IAM & Admin → Service Accounts → Create Service Account.**

- Name: `hermes-chat-bot`
- Пропусти крок “Grant this service account access to project”. IAM на конкретній
  підписці — це все, що тобі потрібно — **НЕ** надавай ролі рівня проєкту для Pub/Sub.

Після створення відкрий SA, перейди до **Keys → Add Key → Create new key → JSON** і
завантаж файл. Збережи його в місці, доступному лише Hermes (наприклад,
`~/.hermes/google-chat-sa.json`, `chmod 600`).

:::caution Немає ролі “Chat Bot Caller”
Поширена помилка — шукати IAM‑роль, специфічну для Chat, і надавати її на рівні
проєкту. Такої ролі не існує. Повноваження чат‑бота отримується завдяки
встановленню в просторі, а не через IAM. Єдине, що потрібно твоєму SA, — це роль
підписника Pub/Sub на підписці, яку ти створиш у наступному кроці.
:::
## Крок 4: Створити тему та підписку Pub/Sub

**Pub/Sub → Topics → Create topic.**

- Topic ID: `hermes-chat-events`
- Залиши всі інші параметри за замовчуванням.

Після створення на сторінці деталізації теми є вкладка **Subscriptions**. Створи підписку:

- Subscription ID: `hermes-chat-events-sub`
- Delivery type: **Pull**
- Message retention: **7 days** (щоб черга залишалася після перезапуску hermes)
- Залиши решту параметрів за замовчуванням.

---
## Крок 5: Прив’язка IAM до теми (критично)

На **тему** (не на підписку) додай IAM‑принципала:

- Принципал: `chat-api-push@system.gserviceaccount.com`
- Роль: `Pub/Sub Publisher`

Без цього Google Chat не зможе публікувати події у твою тему, і твій бот ніколи нічого не отримає.

---
## Крок 6: Прив’язка IAM до підписки

На **підписці** додай свій власний Service Account як principal:

- Principal: `hermes-chat-bot@<your-project>.iam.gserviceaccount.com`
- Role: `Pub/Sub Subscriber`

Також надай роль `Pub/Sub Viewer` на тій самій підписці — Hermes викликає
`subscription.get()` під час запуску як перевірку доступності.
## Крок 7: Налаштуй чат‑додаток

Перейди до **APIs & Services → Google Chat API → Configuration**.

- **App name**: назва, яку ти хочеш, щоб бачили користувачі («Hermes» — підходить).
- **Avatar URL**: будь‑який публічний PNG (Google має кілька типових).
- **Description**: коротке речення, що показується в каталозі додатків.
- **Functionality**: увімкни **Receive 1:1 messages** та **Join spaces and group conversations**.
- **Connection settings**: вибери **Cloud Pub/Sub**, введи назву теми `projects/<your-project>/topics/hermes-chat-events`.
- **Visibility**: обмеж доступ до твого робочого простору (або конкретних користувачів) — не публікуй для всіх, доки тестуєш.

Save.
## Крок 8: Встановити бота у тестовому просторі

Відкрий Google Chat у браузері. Розпочни пряме повідомлення (DM) зі своїм ботом, шукаючи його назву в меню **+ New Chat**. При першому надсиланні повідомлення Google надсилає подію `ADDED_TO_SPACE`, яку Hermes використовує для кешування власного `users/{id}` бота з метою фільтрації власних повідомлень.
## Крок 9: Налаштуй Hermes

Додай розділ Google Chat у `~/.hermes/.env`:

```bash
# Required
GOOGLE_CHAT_PROJECT_ID=my-chat-bot-123
GOOGLE_CHAT_SUBSCRIPTION_NAME=projects/my-chat-bot-123/subscriptions/hermes-chat-events-sub
GOOGLE_CHAT_SERVICE_ACCOUNT_JSON=/home/you/.hermes/google-chat-sa.json

# Authorization — paste the emails of people allowed to talk to the bot
GOOGLE_CHAT_ALLOWED_USERS=you@yourdomain.com,coworker@yourdomain.com

# Optional
GOOGLE_CHAT_HOME_CHANNEL=spaces/AAAA...         # default delivery destination for cron jobs
GOOGLE_CHAT_MAX_MESSAGES=1                      # Pub/Sub FlowControl; 1 serializes commands per session
GOOGLE_CHAT_MAX_BYTES=16777216                  # 16 MiB — cap on in-flight message bytes
```

Ідентифікатор проєкту також запасний варіант — `GOOGLE_CLOUD_PROJECT`, а шлях до SA — `GOOGLE_APPLICATION_CREDENTIALS` — використай будь‑яку з цих конвенцій.

Встанови залежності, потрібні адаптеру Google Chat (наразі жоден Hermes‑extra не опубліковано — встанови їх безпосередньо):

```bash
pip install google-cloud-pubsub google-api-python-client google-auth google-auth-oauthlib
```

Запусти шлюз:

```bash
hermes gateway
```

Ти маєш побачити рядок журналу типу:

```
[GoogleChat] Connected; project=my-chat-bot-123, subscription=<redacted>,
             bot_user_id=users/XXXX, flow_control(msgs=1, bytes=16777216)
```

Надішли «hola» у тестовому DM. Бот опублікує маркер «Hermes is thinking…», а потім відредагує це саме повідомлення, вставивши реальну відповідь — без «message deleted»‑томбстоунів.
## Форматування та можливості

Google Chat підтримує обмежену підмножину markdown:

| Підтримується | Не підтримується |
|-----------|---------------|
| `*bold*`, `_italic_`, `~strike~`, `` `code` `` | Заголовки, списки |
| Вбудовані зображення через URL | Кнопки Interactive Card v2 (v1 цього шлюзу) |
| Вбудовані файлові вкладення (після `/setup-files` — дивись Крок 10) | Вбудовані голосові нотатки / круглі відео‑нотатки |

Системна підказка агента містить специфічну підказку для Google Chat, щоб він знав про ці обмеження і уникав форматування, яке не буде відображено.

Обмеження розміру повідомлення: 4000 символів на повідомлення. Довгі відповіді агента автоматично розбиваються на кілька повідомлень.

Підтримка потоків: коли користувач відповідає у потоці, Hermes виявляє `thread.name` і публікує свою відповідь у тому ж потоці, тому кожен потік отримує окрему сесію Hermes.

## Крок 10: Доставка нативних вкладень (необов’язково)

Виходячи з коробки, бот може публікувати текст, вбудовані зображення за URL та завантажувати картки для аудіо/відео/документів. Щоб доставляти **нативні** вкладення Chat — той самий віджет файлу, який з’являється, коли людина перетягує файл — кожен користувач авторизує бота один раз через OAuth‑потік для окремого користувача.

### Чому окремий потік

Точка входу Google Chat `media.upload` жорстко відхиляє автентифікацію сервісного облікового запису:

> This method doesn't support app authentication with a service account.
> Authenticate with a user account.

Немає ролі IAM чи області, яка це виправляє. Точка входу приймає лише облікові дані користувача. Тому бот має діяти *як користувач* щоразу, коли завантажує файл — саме як користувач, який запросив файл.

### Одноразове налаштування хоста

1. Перейди до **APIs & Services → Credentials** у тому ж проєкті GCP.
2. **Create credentials → OAuth client ID → Desktop app**.
3. Завантаж JSON. Перемісти його на хост, на якому працює Hermes.
4. На хості зареєструй клієнт у Hermes:

```bash
python -m plugins.platforms.google_chat.oauth \
    --client-secret /path/to/client_secret.json
```

Це створює `~/.hermes/google_chat_user_client_secret.json`. Це спільна інфраструктура — вона ідентифікує OAuth *додаток*, а не окремого користувача. Один файл на хост достатній, незалежно від кількості користувачів, які пізніше авторизуються.

### Авторизація користувача (у чаті)

Кожен користувач запускає потік один раз у своєму DM з ботом:

1. Відправляє `/setup-files` боту. Бот відповідає статусом і наступним кроком.
2. Відправляє `/setup-files start`. Бот відповідає OAuth‑URL.
3. Відкриває URL, натискає **Allow** і бачить, як браузер не вдається завантажити `http://localhost:1/?...&code=...`. Ця помилка очікувана — код авторизації знаходиться в адресному рядку.
4. Копіює невдалий URL (або лише значення `code=...`) і вставляє його в чат як `/setup-files <PASTED_URL>`. Бот обмінює його на токен оновлення.

Токен зберігається у `~/.hermes/google_chat_user_tokens/<sanitized_email>.json`. Подальші запити файлів у DM цього користувача використовують *його* токен, тому бот завантажує файли від його імені, а повідомлення потрапляє у його простір.

Щоб відкликати пізніше: `/setup-files revoke` видаляє лише токен цього користувача. Токени інших користувачів залишаються недоторканими.

### Область

Потік запитує лише одну область: `chat.messages.create`. Вона охоплює і `media.upload`, і `messages.create`, які посилаються на завантажений `attachmentDataRef`. Ніякого Drive, жодних ширших областей Chat — це навмисно мінімальний привілей.

### Поведінка в багатокористувацькому режимі

Коли у запитувача ще немає токена користувача, бот переходить до **запасного (варіанту)** однокористувацького токена у `~/.hermes/google_chat_user_token.json` (якщо він залишився після попередньої установки без багатокористувацької підтримки). Якщо жоден токен недоступний, бот надсилає чітке текстове повідомлення з інструкцією запустити `/setup-files`.

Користувач, який відкликає токен, очищає лише свою «слот». 401/403 від одного токена виключає лише кеш цього користувача. Користувачі не заважають один одному.
## Устранення неполадок

**Бот мовчить після надсилання «hola».**

1. Перевірте, чи має підписка Pub/Sub **не доставлені** повідомлення в консолі.
   Якщо так, Hermes не автентифікований — перевірте `GOOGLE_CHAT_SERVICE_ACCOUNT_JSON`
   і чи зазначений SA у ролі `Pub/Sub Subscriber` на підписці.
2. Якщо підписка не містить жодного повідомлення, Google Chat не публікує.
   Подвійно перевірте прив’язку IAM на **топіку**:
   `chat-api-push@system.gserviceaccount.com` має мати `Pub/Sub Publisher`.
3. Перевірте логи `hermes gateway` на наявність `[GoogleChat] Connected`. Якщо бачите
   `[GoogleChat] Config validation failed`, повідомлення про помилку підкаже, яку
   змінну оточення виправити.

**Бот відповідає, але замість відповіді агента з’являється повідомлення про помилку.**

Перевірте логи на `[GoogleChat] Pub/Sub stream died` — якщо це повторюється, ваші
облікові дані SA могли бути змінені або підписка видалена. Після 10 спроб адаптер
позначає себе як фатальний.

**«403 Forbidden» у кожному вихідному повідомленні.**

Бот був видалений зі простору або ви відкликали його в консолі Chat API.
Перевстановіть його у просторі (наступна подія `ADDED_TO_SPACE` автоматично
включить обмін повідомленнями).

**Занадто багато попереджень «Rate limit hit».**

Стандартні квоти Chat API дозволяють 60 повідомлень на простір за хвилину. Якщо ваш
агент генерує довгі потокові відповіді, що перевищують це, адаптер повторює запит
з експоненціальним бекофсом — проте ви все одно побачите затримку, видиму користувачеві.
Розгляньте можливість стислих відповідей або збільшення квоти в консолі GCP.

**Бот продовжує публікувати повідомлення «/setup-files» замість файлів.**

У запитувача немає токену OAuth для користувача, і немає резервного варіанту.
Запустіть `/setup-files` у їхньому DM і виконайте крок 10. Після завершення обміну
наступний запит файлу буде завантажений безперешкодно, без перезапуску шлюзу.

**`/setup-files start` повідомляє «No client credentials stored on the host».**

Одноразове налаштування хоста не було виконано. На терміналі хоста, який запускає
Hermes:

```bash
python -m plugins.platforms.google_chat.oauth \
    --client-secret /path/to/client_secret.json
```

Потім знову надішліть `/setup-files start`.

**`/setup-files <PASTED_URL>` повідомляє «Token exchange failed».**

Код автентифікації одноразовий і має короткий термін дії (зазвичай кілька хвилин).
Надішліть `/setup-files start`, щоб отримати нову URL‑адресу, і спробуйте ще раз.
## Примітки щодо безпеки

- **Область прав Service Account**: адаптер запитує області `chat.bot` і `pubsub`.
  IAM має бути фактичним засобом примусу — надайте вашому SA мінімальні права
  (`roles/pubsub.subscriber` + `roles/pubsub.viewer` на підписці), а не ролі Pub/Sub на рівні проєкту чи організації.
- **Захист завантаження вкладень**: Hermes додає токен SA лише до URL‑адрес, чиї хости входять до короткого білого списку доменів, що належать Google (`googleapis.com`, `drive.google.com`, `lh[3-6].googleusercontent.com` та кілька інших). Будь‑який інший хост відхиляється ще до HTTP‑запиту, щоб захистити від сценаріїв SSRF, коли спеціально сформована подія може перенаправити токен до сервісу метаданих GCE.
- **Редакція**: електронні адреси Service Account, шляхи підписок та тем видаляються з виводу журналу скриптом `agent/redact.py`. Дамп конверту (`GOOGLE_CHAT_DEBUG_RAW=1`) проходить через той самий фільтр редагування і записується на рівні DEBUG.
- **Відповідність**: якщо плануєш підключати цього бота до регульованого робочого простору (будь‑що з політикою резидентності даних або управління ШІ), отримай відповідний дозвіл до першої інсталяції.
- **Область прав User OAuth**: потік вкладень для окремих користувачів запитує *лише* `chat.messages.create` — мінімум, який охоплює `media.upload` плюс наступний `messages.create`. Токени зберігаються у вигляді простого JSON у
  `~/.hermes/google_chat_user_tokens/<sanitized_email>.json` (захист забезпечують дозволи файлової системи — та сама модель, що й для файлу ключа SA). Кожен токен належить лише одному користувачу; відкликання обмежене цим користувачем.
