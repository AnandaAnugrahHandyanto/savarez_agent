# ntfy

[ntfy](https://ntfy.sh/) — це простий HTTP‑базований сервіс публікації‑підписки сповіщень. Працює з безкоштовним публічним сервером `ntfy.sh` або будь‑яким самостійно розгорнутим інстансом і підтримує будь‑якого клієнта, який може робити HTTP‑запити — телефони, браузери, скрипти, годинники.

ntfy — чудовий легковаговий канал push для Hermes: підпишись на тему у [ntfy mobile app](https://ntfy.sh/docs/subscribe/phone/), надсилай повідомлення в тему, щоб спілкуватись з агентом, отримуй відповідь назад на телефон.

> Запусти `hermes gateway setup` і обери **ntfy** для покрокового налаштування.

## Вимоги

- Назва теми (будь‑який унікальний рядок — `hermes-myname-2026` підходить)
- [ntfy mobile app](https://ntfy.sh/docs/subscribe/phone/) встановлений і підписаний на цю тему
- За бажанням: самостійно розгорнутий сервер ntfy або токен облікового запису `ntfy.sh` для приватних/зарезервованих тем

Все. Ніякого SDK, демона, Node.js. Адаптер використовує `httpx`, який вже є залежністю Hermes.

## Налаштування Hermes

### Через майстер налаштувань

```bash
hermes setup gateway
```

Обери **ntfy** і слідуй підказкам.

### Через змінні середовища

Додай це до `~/.hermes/.env`:

```
NTFY_TOPIC=hermes-myname-2026
NTFY_ALLOWED_USERS=hermes-myname-2026
NTFY_HOME_CHANNEL=hermes-myname-2026
```

| Variable | Required | Description |
|---|---|---|
| `NTFY_TOPIC` | Yes | Тема для підписки (вхідні повідомлення) |
| `NTFY_SERVER_URL` | Optional | URL сервера (за замовчуванням: `https://ntfy.sh`) — вкажи самостійно розгорнутий ntfy для приватності |
| `NTFY_TOKEN` | Optional | Bearer‑токен (наприклад `tk_xyz`) або `user:pass` для Basic auth |
| `NTFY_PUBLISH_TOPIC` | Optional | Інша тема для вихідних відповідей (за замовчуванням `NTFY_TOPIC`) |
| `NTFY_MARKDOWN` | Optional | Встанови `true`, щоб надсилати відповіді з заголовком `X-Markdown: true` |
| `NTFY_ALLOWED_USERS` | Recommended | Кома‑розділений список тем, які дозволені (сприймаються як ідентифікатори користувачів; дивись нижче) |
| `NTFY_ALLOW_ALL_USERS` | Optional | Встанови `true`, щоб дозволити всім публікаторам — безпечно лише для приватних тем з токенами читання |
| `NTFY_HOME_CHANNEL` | Optional | Тема за замовчуванням для cron / доставки сповіщень |
| `NTFY_HOME_CHANNEL_NAME` | Optional | Людська назва домашнього каналу |

## Модель ідентичності — прочитай перед розгортанням

ntfy не має вбудованої автентифікованої користувацької ідентичності. Поле `title` у опублікованому повідомленні **контролюється публікатором** і може бути будь‑яким. Адаптер Hermes НЕ використовує `title` для авторизації — інакше будь‑який публікатор, що знає тему, міг би підмінити дозволеного користувача.

Натомість **сама назва теми є ідентичністю**. Кожне повідомлення, опубліковане в тему, розглядається як надіслане від одного логічного користувача (теми). Тому `NTFY_ALLOWED_USERS` зазвичай просто назва теми — одноелементний білий список, який контролює весь канал.

Це означає, що **будь‑хто, хто знає тему, може спілкуватись з агентом**. Щоб зробити це реальним кордоном довіри:

- **Самостійно розгорни ntfy** і закрий тему за допомогою [Access Control](https://docs.ntfy.sh/config/#access-control). Тільки авторизовані клієнти з токеном читання/запису можуть публікувати.
- Або **використай приватну тему на ntfy.sh** ([reserved topics](https://docs.ntfy.sh/publish/#reserved-topics) вимагають облікового запису) і захисти її `NTFY_TOKEN`.
- Або **вибери довгу, важко вгадувану назву теми** (`hermes-7d4f9c8b-2026`) і розглядай її як спільний секрет. Це найпростіше налаштування, проте назва теми може потрапити у логи чи скріншоти.

У всіх випадках не передавай конфіденційні дані через ntfy, якщо тема не захищена контролем доступу.

## Швидкий старт — спілкуйся з агентом зі свого телефону

1. Вибери назву теми: `hermes-myname-2026`
2. На телефоні: встанови [ntfy app](https://ntfy.sh/docs/subscribe/phone/), натисни **+**, введи `hermes-myname-2026`
3. На хості:
   ```bash
   echo 'NTFY_TOPIC=hermes-myname-2026' >> ~/.hermes/.env
   echo 'NTFY_ALLOWED_USERS=hermes-myname-2026' >> ~/.hermes/.env
   hermes gateway restart
   ```
4. У додатку ntfy надішли повідомлення в тему. Відповідь агента прийде як push‑сповіщення.

## Використання ntfy з cron‑задачами

Після встановлення `NTFY_HOME_CHANNEL` cron‑задачі можуть надсилати в ntfy:

```python
cronjob(
    action="create",
    schedule="every 1h",
    deliver="ntfy",          # uses NTFY_HOME_CHANNEL
    prompt="Check for alerts and summarise."
)
```

Або явно вказати конкретну тему:

```python
send_message(target="ntfy:alerts-channel", message="Done!")
```

Працює навіть коли cron запускається поза процесом шлюзу — плагін реєструє `standalone_sender_fn`, який відкриває власне HTTP‑з’єднання.

## Самостійний хостинг ntfy

Якщо потрібен повний контроль:

```bash
# Docker
docker run -p 80:80 -it binwiederhier/ntfy serve

# Native
go install heckel.io/ntfy/v2@latest
ntfy serve
```

Потім вкажи Hermes на цей сервер:

```
NTFY_SERVER_URL=https://ntfy.mydomain.com
NTFY_TOPIC=hermes
NTFY_TOKEN=tk_abc123  # if you've set up access control
```

Самостійний хостинг дає контроль доступу до тем, політики збереження повідомлень, вкладень та емодзі‑тегів. Дивись [ntfy server docs](https://docs.ntfy.sh/install/).

## Форматування Markdown

Клієнти ntfy рендерять markdown, коли публікатор встановлює заголовок `X-Markdown: true`. Щоб увімкнути це для вихідних відповідей Hermes:

```
NTFY_MARKDOWN=true
```

Або в `config.yaml`:

```yaml
platforms:
  ntfy:
    extra:
      markdown: true
```

Мобільний додаток підтримує підмножину CommonMark — жирний, курсив, списки, посилання, блоки коду. Дивись [ntfy's markdown docs](https://docs.ntfy.sh/publish/#markdown-formatting) для повного переліку.

## Налаштування лише для вихідних повідомлень (сповіщення без вхідних)

Якщо потрібно лише, щоб Hermes *надсилав* сповіщення в ntfy (зведення cron, тривоги) і ніколи не приймав вхідні повідомлення, встанови і `NTFY_TOPIC`, і `NTFY_PUBLISH_TOPIC` в одне й те ж значення і пропусти `NTFY_ALLOWED_USERS`. Без білого списку агент не відповідає на вхідні повідомлення — телефон отримує лише push‑повідомлення, а розмова одностороння.

## Обмеження

- **Розмір повідомлення**: ntfy обмежує тіло повідомлення 4096 символами. Hermes обрізає його з попередженням, коли перевищено ліміт.
- **Немає індикаторів набору**: протокол не передає їх; `send_typing` — ніяк не працює.
- **Немає тем чи вкладень**: ntfy — це прості push‑сповіщення. Довгі відповіді залишаються в тілі повідомлення, без розгалуження тем.
- **Немає вбудованої ідентичності користувача**: дивись розділ «Модель ідентичності» вище.

## Устранення проблем

**Auth failure / 401** — `NTFY_TOKEN` неправильний, або токен не має прав публікації/підписки на цю тему. Адаптер зупиняє цикл перепідключення на 401, а статус шлюзу покаже `fatal: ntfy_unauthorized`. Виправ токен і перезапусти шлюз.

**Topic not found / 404** — `NTFY_TOPIC` не існує на вказаному сервері. Для ntfy.sh теми створюються автоматично при першій публікації, тому 404 означає, що ти вказав самостійно розгорнутий сервер без створеної теми. Адаптер зупиняє цикл перепідключення з `fatal: ntfy_topic_not_found`.

**Connected but no messages** — Перевір, що `NTFY_ALLOWED_USERS` містить назву теми. У моделі ідентичності ntfy тема — користувач; порожній білий список відхиляє все.

**Reconnects every 60s** — За замовчуванням keepalive потоку 55 s; у ntfy можуть бути періодичні мережеві проблеми. Адаптер застосовує експоненціальне збільшення інтервалу (2 → 5 → 10 → 30 → 60 s) і скидає його до 0, коли потік живий ≥60 s.
