# Налаштування Signal

Hermes підключається до Signal через демон [signal-cli](https://github.com/AsamK/signal-cli), запущений у режимі HTTP. Адаптер передає повідомлення в режимі реального часу за допомогою SSE (Server‑Sent Events) і надсилає відповіді через JSON‑RPC.

Signal – це найприватніший серед масових мессенджерів: сквозне шифрування за замовчуванням, протокол з відкритим кодом, мінімальний збір метаданих. Це робить його ідеальним для робочих процесів агентів, чутливих до безпеки.

:::info No New Python Dependencies
Адаптер Signal використовує `httpx` (вже є основною залежністю Hermes) для всього зв’язку. Додаткові Python‑пакети не потрібні. Потрібно лише встановити signal-cli зовнішньо.
:::

---
## Передумови

- **signal-cli** — Java‑базований клієнт Signal ([GitHub](https://github.com/AsamK/signal-cli))
- **Java 17+** runtime — потрібна для signal-cli
- **Номер телефону** з встановленим Signal (для підключення як вторинного пристрою)

### Встановлення signal-cli

```bash
# macOS
brew install signal-cli

# Linux (download latest release)
VERSION=$(curl -Ls -o /dev/null -w %{url_effective} \
  https://github.com/AsamK/signal-cli/releases/latest | sed 's/^.*\/v//')
curl -L -O "https://github.com/AsamK/signal-cli/releases/download/v${VERSION}/signal-cli-${VERSION}.tar.gz"
sudo tar xf "signal-cli-${VERSION}.tar.gz" -C /opt
sudo ln -sf "/opt/signal-cli-${VERSION}/bin/signal-cli" /usr/local/bin/
```

:::caution
signal-cli **не** знаходиться в репозиторіях apt або snap. Linux‑встановлення вище завантажує його безпосередньо з [GitHub releases](https://github.com/AsamK/signal-cli/releases).
:::

## Крок 1: Підключи свій обліковий запис Signal

Signal-cli працює як **пов’язаний пристрій** — схоже на WhatsApp Web, але для Signal. Твій телефон залишається основним пристроєм.

```bash
# Generate a linking URI (displays a QR code or link)
signal-cli link -n "HermesAgent"
```

1. Відкрий **Signal** на телефоні
2. Перейди до **Налаштування → Пов’язані пристрої**
3. Натисни **Link New Device**
4. Відскануй QR‑код або введи URI

---
## Крок 2: Запусти демон signal‑cli

```bash
# Replace +1234567890 with your Signal phone number (E.164 format)
signal-cli --account +1234567890 daemon --http 127.0.0.1:8080
```

:::tip
Тримай його працюючим у фоні. Можеш використати `systemd`, `tmux`, `screen` або запустити його як сервіс.
:::

Перевір, що він працює:

```bash
curl http://127.0.0.1:8080/api/v1/check
# Should return: {"versions":{"signal-cli":...}}
```

## Крок 3: Налаштування Hermes

Найпростіший спосіб:

```bash
hermes gateway setup
```

Вибери **Signal** у меню платформи. Майстер:

1. Перевірить, чи встановлено `signal-cli`
2. Запитає HTTP‑URL (за замовчуванням: `http://127.0.0.1:8080`)
3. Перевірить з’єднання з демоном
4. Запитає номер телефону твого облікового запису
5. Налаштує дозволені користувачі та політики доступу

### Ручна конфігурація

Додай у `~/.hermes/.env`:

```bash
# Required
SIGNAL_HTTP_URL=http://127.0.0.1:8080
SIGNAL_ACCOUNT=+1234567890

# Security (recommended)
SIGNAL_ALLOWED_USERS=+1234567890,+0987654321    # Comma-separated E.164 numbers or UUIDs

# Optional
SIGNAL_GROUP_ALLOWED_USERS=groupId1,groupId2     # Enable groups (omit to disable, * for all)
SIGNAL_HOME_CHANNEL=+1234567890                  # Default delivery target for cron jobs
```

Потім запусти шлюз:

```bash
hermes gateway              # Foreground
hermes gateway install      # Install as a user service
sudo hermes gateway install --system   # Linux only: boot-time system service
```

## Контроль доступу

### DM доступ

DM доступ слідує тому ж шаблону, що й інші платформи Hermes:

1. **`SIGNAL_ALLOWED_USERS` встановлено** → лише ці користувачі можуть надсилати повідомлення
2. **Не встановлено allowlist** → невідомі користувачі отримують код парування DM (підтвердження через `hermes pairing approve signal CODE`)
3. **`SIGNAL_ALLOW_ALL_USERS=true`** → будь‑хто може надсилати повідомлення (використовуй з обережністю)

### Груповий доступ

Груповий доступ керується змінною середовища `SIGNAL_GROUP_ALLOWED_USERS`:

| Конфігурація | Поведінка |
|---------------|----------|
| Не встановлено (за замовчуванням) | Всі групові повідомлення ігноруються. Бот відповідає лише на DM. |
| Встановлено зі списком ID груп | Моніторяться лише вказані групи (наприклад, `groupId1,groupId2`). |
| Встановлено `*` | Бот відповідає в будь‑якій групі, учасником якої він є. |

## Функції

### Прикріплення

Адаптер підтримує надсилання та отримання медіа в обох напрямках.

**Вхідні** (користувач → агент):

- **Зображення** — PNG, JPEG, GIF, WebP (автоматично визначається за magic‑bytes)
- **Аудіо** — MP3, OGG, WAV, M4A (голосові повідомлення транскрибуються, якщо налаштовано Whisper)
- **Документи** — PDF, ZIP та інші типи файлів

**Вихідні** (агент → користувач):

Агент може надсилати медіа‑файли за допомогою тегів `MEDIA:` у відповідях. Підтримуються такі методи доставки:

- **Зображення** — `send_multiple_images` і `send_image_file` надсилають PNG, JPEG, GIF, WebP як рідні прикріплення Signal
- **Голос** — `send_voice` надсилає аудіофайли (OGG, MP3, WAV, M4A, AAC) як прикріплення
- **Відео** — `send_video` надсилає MP4‑відеофайли
- **Документи** — `send_document` надсилає будь‑який тип файлу (PDF, ZIP тощо)

Усі вихідні медіа проходять через стандартний API прикріплень Signal. На відміну від деяких платформ, Signal не розрізняє голосові повідомлення та файлові прикріплення на рівні протоколу.

Обмеження розміру прикріплення: **100 МБ** (в обох напрямках).

:::warning
**Сервери Signal обмежують швидкість завантаження прикріплень**, адаптер використовує планувальник для надсилання кількох зображень, який групує їх по 32 і регулює швидкість завантаження відповідно до політики серверів Signal.
:::

### Рідне форматування, цитати у відповідях та реакції

Повідомлення Signal відображаються з **рідним форматуванням** замість буквальних символів markdown. Адаптер перетворює markdown (`**bold**`, `*italic*`, `` `code` ``, `~~strike~~`, `||spoiler||`, заголовки) у `bodyRanges` Signal, тож текст показується зі справжнім стилем у клієнті отримувача, а не як видимі `**` / `` ` `` символи.

**Цитати у відповідях.** Коли Hermes відповідає на конкретне повідомлення, він тепер публікує рідну відповідь, що цитує оригінал — та ж UI‑можливість, яку бачать користувачі Signal, коли використовують «Reply». Це автоматично для відповідей, згенерованих у відповідь на вхідне повідомлення.

**Реакції.** Агент може реагувати на повідомлення через стандартний API реакцій; реакції відображаються у Signal як емодзі‑реакції до відповідного повідомлення, а не як додатковий текст.

Нічого додаткового налаштовувати не потрібно — це включено за замовчуванням у нових збірках signal‑cli. Якщо твоя версія `signal-cli` застаріла, Hermes переходить у режим простого тексту і виводить одноразове попередження.

### Індикатори набору

Бот надсилає індикатори набору під час обробки повідомлень, оновлюючи їх кожні 8 секунд.

### Приховування номерів телефонів

Усі номери телефонів автоматично приховуються в логах:
- `+15551234567` → `+155****4567`
- Це стосується як логів шлюзу Hermes, так і глобальної системи приховування.

### Примітка собі (налаштування з одним номером)

Якщо ти запускаєш `signal-cli` як **пов’язаний вторинний пристрій** на власному номері телефону (а не окремий номер бота), ти можеш взаємодіяти з Hermes через функцію Signal «Note to Self».

Просто надішли повідомлення самому собі зі свого телефону — `signal-cli` його підхопить, і Hermes відповість у тому ж чаті.

**Як це працює:**
- Повідомлення «Note to Self» надходять як конверти `syncMessage.sentMessage`
- Адаптер виявляє, коли вони адресовані власному обліковому запису бота, і обробляє їх як звичайні вхідні повідомлення
- Захист від ехо‑відповідей (відстеження timestamp надсилання) запобігає нескінченним циклам — власні відповіді бота автоматично фільтруються

**Додаткове налаштування не потрібне.** Це працює автоматично, доки `SIGNAL_ACCOUNT` збігається з твоїм номером телефону.

### Моніторинг стану

Адаптер стежить за з’єднанням SSE і автоматично перепід’єднується, якщо:
- З’єднання розривається (з експоненціальним відкатом: 2 s → 60 s)
- Протягом 120 секунд не виявлено активності (пінгує `signal-cli` для перевірки)
## Устранення проблем

| Проблема | Рішення |
|---------|----------|
| **"Cannot reach signal-cli"** під час налаштування | Переконайся, що daemon **signal-cli** запущений: `signal-cli --account +YOUR_NUMBER daemon --http 127.0.0.1:8080` |
| **Повідомлення не надходять** | Перевір, чи `SIGNAL_ALLOWED_USERS` містить номер відправника у форматі E.164 (з префіксом `+`) |
| **"signal-cli not found on PATH"** | Встанови **signal-cli** і переконайся, що він у твоєму `PATH`, або використай Docker |
| **З’єднання постійно розривається** | Переглянь логи **signal-cli** на предмет помилок. Переконайся, що встановлено Java 17+ |
| **Групові повідомлення ігноруються** | Налаштуй `SIGNAL_GROUP_ALLOWED_USERS` зі специфічними ідентифікаторами груп або `*`, щоб дозволити всі групи |
| **Bot не відповідає ніким** | Налаштуй `SIGNAL_ALLOWED_USERS`, використай DM‑парування або явно дозволь всім користувачам через політику шлюзу, якщо потрібен ширший доступ |
| **Дублюються повідомлення** | Переконайся, що лише один екземпляр **signal-cli** слухає твій номер телефону |
## Безпека

:::warning
**Завжди налаштовуй контроль доступу.** Бот має доступ до терміналу за замовчуванням. Без `SIGNAL_ALLOWED_USERS` або DM‑парування шлюз відхиляє всі вхідні повідомлення як запобіжний захід.
:::

- Номери телефонів заміняються у всіх виводах журналу
- Використовуй DM‑парування або явні allowlist‑и для безпечного підключення нових користувачів
- Тримай групи вимкненими, якщо тобі не потрібна підтримка груп, або додавай у allowlist лише ті групи, яким довіряєш
- Сквозне шифрування Signal захищає вміст повідомлень під час передачі
- Дані сесії signal-cli у `~/.local/share/signal-cli/` містять облікові дані — захищай їх, як пароль

## Довідник змінних середовища

| Змінна | Обов’язково | За замовчуванням | Опис |
|----------|----------|----------------|------|
| `SIGNAL_HTTP_URL` | Так | — | HTTP‑endpoint signal-cli |
| `SIGNAL_ACCOUNT` | Так | — | Номер телефону бота (E.164) |
| `SIGNAL_ALLOWED_USERS` | Ні | — | Номери телефонів/UUID, розділені комами |
| `SIGNAL_GROUP_ALLOWED_USERS` | Ні | — | Ідентифікатори груп для моніторингу, або `*` для всіх (пропусти, щоб вимкнути групи) |
| `SIGNAL_ALLOW_ALL_USERS` | Ні | `false` | Дозволити будь‑якому користувачу взаємодіяти (пропустити список дозволених) |
| `SIGNAL_HOME_CHANNEL` | Ні | — | Цільова точка доставки за замовчуванням для cron‑завдань |
