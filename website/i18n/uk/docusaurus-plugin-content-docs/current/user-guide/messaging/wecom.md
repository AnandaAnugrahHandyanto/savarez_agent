# WeCom (Enterprise WeChat)

Підключи Hermes до [WeCom](https://work.weixin.qq.com/) (企业微信), корпоративної платформи обміну повідомленнями від Tencent. Адаптер використовує AI‑бот WebSocket шлюз WeCom для реального часу двостороннього спілкування — без публічної кінцевої точки або webhook.

Дивись також: [WeCom Callback](./wecom-callback.md) для налаштування вхідного webhook.
## Передумови

- Обліковий запис організації WeCom
- AI‑бот, створений у консолі адміністрування WeCom
- Ідентифікатор бота та секрет з сторінки облікових даних бота
- Пакети Python: `aiohttp` і `httpx`
## Налаштування

### Крок 1: Створити AI‑бота

#### Рекомендовано: Scan-to-Create (одна команда)

```bash
hermes gateway setup
```

Вибери **WeCom** і відскануй QR‑код за допомогою мобільного додатку WeCom. Hermes автоматично створить застосунок‑бота з потрібними дозволами та збереже облікові дані.

Майстер налаштування виконає:
1. Показ QR‑коду у твоєму терміналі
2. Чекає, доки ти відскануєш його мобільним додатком WeCom
3. Автоматично отримує Bot ID та Secret
4. Проведе тебе через конфігурацію контролю доступу

#### Альтернатива: Ручне налаштування

Якщо Scan-to-Create недоступний, майстер переходить до ручного вводу:

1. Увійди в [WeCom Admin Console](https://work.weixin.qq.com/wework_admin/frame)
2. Перейди до **Applications** → **Create Application** → **AI Bot**
3. Налаштуй назву та опис бота
4. Скопіюй **Bot ID** і **Secret** зі сторінки облікових даних
5. Запусти `hermes gateway setup`, вибери **WeCom** і введи облікові дані, коли буде запит

:::warning
Тримай Secret бота в таємниці. Будь-хто, хто його отримає, зможе видавати себе за твого бота.
:::

### Крок 2: Налаштувати Hermes

#### Варіант A: Інтерактивне налаштування (рекомендовано)

```bash
hermes gateway setup
```

Вибери **WeCom** і слідуй підказкам. Майстер проведе тебе через:
- Облікові дані бота (через скан QR або ручний ввід)
- Налаштування контролю доступу (білий список, режим парування або відкритий доступ)
- Канал «домашній» для сповіщень

#### Варіант B: Ручна конфігурація

Додай наступне до `~/.hermes/.env`:

```bash
WECOM_BOT_ID=your-bot-id
WECOM_SECRET=your-secret

# Optional: restrict access
WECOM_ALLOWED_USERS=user_id_1,user_id_2

# Optional: home channel for cron/notifications
WECOM_HOME_CHANNEL=chat_id
```

### Крок 3: Запустити шлюз

```bash
hermes gateway
```
## Функції

- **WebSocket transport** — постійне з’єднання, публічний кінцевий пункт не потрібен
- **DM and group messaging** — налаштовувані політики доступу
- **Per-group sender allowlists** — детальний контроль над тим, хто може взаємодіяти в кожній групі (білий список відправників)
- **Media support** — завантаження та скачування зображень, файлів, голосу, відео
- **AES-encrypted media** — автоматичне розшифрування вхідних вкладень
- **Quote context** — зберігає контекст цитати
- **Markdown rendering** — відповіді у форматі багатого тексту
- **Reply-mode streaming** — стримінг у режимі відповіді, корелює відповіді з контекстом вхідних повідомлень
- **Auto-reconnect** — експоненційне збільшення інтервалу при втраті з’єднання
## Параметри конфігурації

Встанови їх у `config.yaml` під `platforms.wecom.extra`:

| Ключ | За замовчуванням | Опис |
|-----|------------------|------|
| `bot_id` | — | WeCom AI Bot ID (обов’язково) |
| `secret` | — | WeCom AI Bot Secret (обов’язково) |
| `websocket_url` | `wss://openws.work.weixin.qq.com` | URL шлюзу WebSocket |
| `dm_policy` | `open` | Доступ до DM: `open`, `allowlist`, `disabled`, `pairing` |
| `group_policy` | `open` | Доступ до груп: `open`, `allowlist`, `disabled` |
| `allow_from` | `[]` | Ідентифікатори користувачів, яким дозволено надсилати DM (коли `dm_policy=allowlist`) |
| `group_allow_from` | `[]` | Ідентифікатори груп, яким дозволено доступ (коли `group_policy=allowlist`) |
| `groups` | `{}` | Конфігурація для окремих груп (див. нижче) |
## Політики доступу

### DM Policy

Керує тим, хто може надсилати прямі повідомлення (DM) боту:

| Value | Behavior |
|-------|----------|
| `open` | Будь‑хто може надсилати DM боту (за замовчуванням) |
| `allowlist` | Тільки користувачі з `allow_from` можуть надсилати DM |
| `disabled` | Всі DM ігноруються |
| `pairing` | Режим парингу (для початкового налаштування) |

```bash
WECOM_DM_POLICY=allowlist
```

### Group Policy

Керує тим, у яких групах бот відповідає:

| Value | Behavior |
|-------|----------|
| `open` | Бот відповідає у всіх групах (за замовчуванням) |
| `allowlist` | Бот відповідає лише у групах, ідентифікатори яких зазначені в `group_allow_from` |
| `disabled` | Всі повідомлення в групах ігноруються |

```bash
WECOM_GROUP_POLICY=allowlist
```

### Per-Group Sender Allowlists

Для детального контролю можна обмежити, які користувачі можуть взаємодіяти з ботом у конкретних групах. Це налаштовується у `config.yaml`:

```yaml
platforms:
  wecom:
    enabled: true
    extra:
      bot_id: "your-bot-id"
      secret: "your-secret"
      group_policy: "allowlist"
      group_allow_from:
        - "group_id_1"
        - "group_id_2"
      groups:
        group_id_1:
          allow_from:
            - "user_alice"
            - "user_bob"
        group_id_2:
          allow_from:
            - "user_charlie"
        "*":
          allow_from:
            - "user_admin"
```

**Як це працює:**

1. Параметри `group_policy` та `group_allow_from` визначають, чи дозволена група взагалі.
2. Якщо група проходить верхній контроль, список `groups.<group_id>.allow_from` (за наявності) додатково обмежує, які відправники в цій групі можуть взаємодіяти з ботом.
3. Запис `"*"` у групі слугує значенням за замовчуванням для груп, які явно не вказані.
4. Записи у allowlist підтримують підстановочний символ `*`, який дозволяє всіх користувачів, і вони не залежать від регістру.
5. Записи можуть опціонально мати префікс `wecom:user:` або `wecom:group:` — префікс автоматично видаляється.

Якщо для групи не налаштовано `allow_from`, усі користувачі в цій групі дозволені (за умови, що сама група проходить верхню перевірку політики).
## Підтримка медіа

### Вхідні (отримання)

Адаптер отримує медіа‑вкладення від користувачів і кешує їх локально для обробки агентом:

| Тип | Як обробляється |
|------|-----------------|
| **Зображення** | Завантажуються та кешуються локально. Підтримуються як URL‑зображення, так і base64‑закодовані зображення. |
| **Файли** | Завантажуються та кешуються. Ім’я файлу зберігається з оригінального повідомлення. |
| **Голос** | Текстова транскрипція голосового повідомлення витягується, якщо доступна. |
| **Змішані повідомлення** | Змішані типи повідомлень WeCom (текст + зображення) розбираються, і всі компоненти витягуються. |

**Цитовані повідомлення:** Медiа з цитованих (відповідних) повідомлень також витягується, щоб агент мав контекст щодо того, на що користувач відповідає.

### Дешифрування AES‑зашифрованих медіа

WeCom шифрує деякі вхідні медіа‑вкладення за допомогою AES‑256‑CBC. Адаптер обробляє це автоматично:

- Коли вхідний медіа‑елемент містить поле `aeskey`, адаптер завантажує зашифровані байти і розшифровує їх за допомогою AES‑256‑CBC з паддінгом PKCS#7.
- Ключ AES — це base64‑декодоване значення поля `aeskey` (повинно бути саме 32 байти).
- IV отримується з перших 16 байт ключа.
- Потрібен пакет Python `cryptography` (`pip install cryptography`).

Ніякої конфігурації не потрібно — розшифрування відбувається прозоро, коли отримано зашифровану медіа.

### Вихідні (надсилання)

| Метод | Що надсилає | Обмеження розміру |
|--------|--------------|-------------------|
| `send` | Текстові повідомлення у форматі Markdown | 4000 символів |
| `send_image` / `send_image_file` | Нативні повідомлення‑зображення | 10 МБ |
| `send_document` | Вкладення‑файли | 20 МБ |
| `send_voice` | Голосові повідомлення (лише формат AMR для нативного голосу) | 2 МБ |
| `send_video` | Відеоповідомлення | 10 МБ |

**Завантаження частинами:** Файли завантажуються частинами по 512 KB за трьохетапним протоколом (init → chunks → finish). Адаптер виконує це автоматично.

**Автоматичне зниження типу:** Коли медіа перевищує ліміт розміру свого нативного типу, але знаходиться під абсолютним лімітом у 20 МБ, воно автоматично надсилається як загальне файлове вкладення:

- Зображення > 10 МБ → надсилаються як файл
- Відео > 10 МБ → надсилаються як файл
- Голос > 2 МБ → надсилаються як файл
- Аудіо не у форматі AMR → надсилаються як файл (WeCom підтримує AMR лише для нативного голосу)

Файли, що перевищують абсолютний ліміт у 20 МБ, відхиляються з інформативним повідомленням, яке надсилається в чат.
## Режим відповіді (Stream Responses)

Коли бот отримує повідомлення через зворотний виклик WeCom, адаптер запам’ятовує ідентифікатор вхідного запиту. Якщо відповідь надсилається, доки контекст запиту ще активний, адаптер використовує режим відповіді WeCom (`aibot_respond_msg`) зі стримінгом, щоб безпосередньо пов’язати відповідь із вхідним повідомленням. Це забезпечує більш природний досвід спілкування у клієнті WeCom.

Якщо контекст вхідного запиту завершився або недоступний, адаптер переходить до проактивного надсилання повідомлень через `aibot_send_msg`.

Режим відповіді також працює з медіа: завантажені медіа‑файли можна надсилати у відповідь на початкове повідомлення.
## Підключення та повторне підключення

Адаптер підтримує постійне WebSocket‑з’єднання з шлюзом WeCom за адресою `wss://openws.work.weixin.qq.com`.

### Життєвий цикл підключення

1. **Connect:** відкриває WebSocket‑з’єднання та надсилає кадр автентифікації `aibot_subscribe` з `bot_id` і `secret`.
2. **Heartbeat:** надсилає ping‑кадри рівня застосунку кожні 30 секунд, щоб підтримувати з’єднання живим.
3. **Listen:** безперервно читає вхідні кадри та викликає колбеки повідомлень.

### Поведінка при повторному підключенні

При втраті з’єднання адаптер використовує експоненціальне збільшення затримки для повторних спроб:

| Спроба | Затримка |
|--------|----------|
| 1‑ша спроба | 2 секунди |
| 2‑га спроба | 5 секунд |
| 3‑я спроба | 10 секунд |
| 4‑та спроба | 30 секунд |
| 5‑та+ спроба | 60 секунд |

Після кожного успішного повторного підключення лічильник затримки скидається до нуля. Усі очікуючі запити `future` помічаються як невдалі при роз’єднанні, щоб виклики не зависали безкінечно.

### Дедуплікація

Вхідні повідомлення дедуплікуються за ідентифікаторами повідомлень протягом 5‑хвилинного вікна та з максимальною кеш‑кількістю 1000 записів. Це запобігає подвійній обробці повідомлень під час повторного підключення або мережевих збоїв.

## Усунення проблем

| Проблема | Виправлення |
|---------|------------|
| `WECOM_BOT_ID and WECOM_SECRET are required` | Встанови обидві змінні середовища або налаштуй їх у майстрі налаштувань |
| `WeCom startup failed: aiohttp not installed` | Встанови aiohttp: `pip install aiohttp` |
| `WeCom startup failed: httpx not installed` | Встанови httpx: `pip install httpx` |
| `invalid secret (errcode=40013)` | Перевір, чи секрет збігається з обліковими даними твого бота |
| `Timed out waiting for subscribe acknowledgement` | Перевір підключення до `openws.work.weixin.qq.com` |
| Bot doesn't respond in groups | Перевір налаштування `group_policy` і впевнись, що ID групи включений у `group_allow_from` |
| Bot ignores certain users in a group | Перевір списки `allow_from` для кожної групи у розділі конфігурації `groups` |
| Media decryption fails | Встанови `cryptography`: `pip install cryptography` |
| `cryptography is required for WeCom media decryption` | Вхідні медіа зашифровані AES. Встанови: `pip install cryptography` |
| Voice messages sent as files | WeCom підтримує лише формат AMR для вбудованих голосових повідомлень. Інші формати автоматично знижуються до файлу. |
| `File too large` error | У WeCom абсолютний ліміт на всі завантаження файлів — 20 МБ. Стисни або розділи файл. |
| Images sent as files | Зображення > 10 МБ перевищують нативний ліміт і автоматично знижуються до вкладень‑файлів. |
| `Timeout sending message to WeCom` | Можливо, WebSocket розірвався. Перевір журнали на предмет повідомлень про повторне підключення. |
| `WeCom websocket closed during authentication` | Проблема з мережею або неправильні облікові дані. Перевір `WECOM_BOT_ID` і `WECOM_SECRET`. |
