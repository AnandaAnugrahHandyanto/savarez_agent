# Налаштування Telegram

Hermes Agent інтегрується з Telegram як повнофункціональний чат‑бот. Після підключення ти можеш спілкуватися зі своїм агентом з будь‑якого пристрою, надсилати голосові повідомлення, які автоматично транскрибуються, отримувати результати запланованих завдань і використовувати агента в групових чатах. Інтеграція побудована на [python-telegram-bot](https://python-telegram-bot.org/) і підтримує текст, голос, зображення та вкладення файлів.
## Крок 1: Створи бота через BotFather

Кожен Telegram‑бот потребує API‑токен, виданий [@BotFather](https://t.me/BotFather), офіційним інструментом керування ботами Telegram.

1. Відкрий Telegram і знайди **@BotFather**, або перейди за посиланням [t.me/BotFather](https://t.me/BotFather)
2. Надішли `/newbot`
3. Вибери **display name** (наприклад, "Hermes Agent") — це може бути будь‑яка назва
4. Вибери **username** — воно має бути унікальним і закінчуватись на `bot` (наприклад, `my_hermes_bot`)
5. BotFather відповість твоїм **API‑токеном**. Він виглядає так:

```
123456789:ABCdefGHIjklMNOpqrSTUvwxYZ
```

:::warning
Тримай токен бота в таємниці. Будь‑хто, хто має цей токен, може керувати твоїм ботом. Якщо він вийде в інтернет, негайно відклич його за допомогою `/revoke` у BotFather.
:::
## Крок 2: Налаштуй свого бота (необов’язково)

Ці команди BotFather покращують досвід користувачів. Напиши @BotFather і використай:

| Команда | Призначення |
|---------|--------------|
| `/setdescription` | Текст «Що може робити цей бот?», який показується перед тим, як користувач розпочне спілкування |
| `/setabouttext` | Короткий текст на сторінці профілю бота |
| `/setuserpic` | Завантажити аватар для твого бота |
| `/setcommands` | Визначити меню команд (кнопка `/` у чаті) |
| `/setprivacy` | Керувати тим, чи бачить бот усі повідомлення в групі (дивись Крок 3) |

:::tip
Для `/setcommands` корисний початковий набір:

```
help - Show help information
new - Start a new conversation
sethome - Set this chat as the home channel
```
:::
## Крок 3: Режим конфіденційності (Критично для груп)

Telegram‑боти мають **режим конфіденційності**, який **увімкнено за замовчуванням**. Це найпоширеніше джерело плутанини під час використання ботів у групах.

**При увімкненому режимі конфіденційності** ваш бот бачить лише:
- Повідомлення, що починаються з команди `/`
- Відповіді безпосередньо на повідомлення бота
- Службові повідомлення (приєднання/вихід учасників, закріплені повідомлення тощо)
- Повідомлення в каналах, де бот є адміністратором

**При вимкненому режимі конфіденційності** бот отримує кожне повідомлення в групі.

### Як вимкнути режим конфіденційності

1. Напиши **@BotFather**
2. Відправ `/mybots`
3. Вибери свого бота
4. Перейди до **Bot Settings → Group Privacy → Turn off**

:::warning
**Ти маєш видалити та знову додати бота до будь‑якої групи** після зміни налаштування конфіденційності. Telegram кешує стан конфіденційності, коли бот приєднується до групи, і він не оновиться, доки бот не буде видалений і знову доданий.
:::

:::tip
Альтернатива вимкненню режиму конфіденційності: підвищити бота до **адміністратора групи**. Адміністратори‑боти завжди отримують усі повідомлення незалежно від налаштування конфіденційності, і це дозволяє уникнути глобального перемикання режиму.
:::

### Спостерігай за розмовами в групі без автоматичних відповідей

Для поведінки групи у стилі OpenClaw/Yuanbao налаштуй Telegram так, щоб бот **бачив** звичайні повідомлення групи, але **відповідав** лише коли його безпосередньо активують:

```yaml
telegram:
  allowed_chats:
    - "-1001234567890"
  group_allowed_chats:
    - "-1001234567890"
  require_mention: true
  observe_unmentioned_group_messages: true
```

При увімкненому цьому режимі неозначені повідомлення групи з явно дозволених чатів/топіків додаються до спільної транскрипції сесії чату/топіка як спостережуваний контекст, проте не запускають агента. `allowed_chats` визначає, де бот відповідає; `group_allowed_chats` авторизує спільну групову сесію, що використовується для спостережуваного контексту, тому використай ті ж ідентифікатори чатів для цього режиму. Пізніше згадка `@botname`, відповідь боту або налаштований шаблон згадки в тому ж дозволеному чаті/топіку можуть використати цей спостережуваний контекст. Запущене повідомлення також позначається `[nickname|user_id]` і отримує безпековий підказник за кожен хід, щоб модель розглядала попередні спостережувані рядки як контекст, а не як інструкції, адресовані боту.

Еквівалентна змінна середовища:

```bash
TELEGRAM_ALLOWED_CHATS=-1001234567890
TELEGRAM_GROUP_ALLOWED_CHATS=-1001234567890
TELEGRAM_OBSERVE_UNMENTIONED_GROUP_MESSAGES=true
```

Для цього Telegram має доставляти звичайні групові повідомлення до шлюзу, тому вимкни режим конфіденційності BotFather або підвищи бота до адміністратора групи, як описано вище.
## Крок 4: Знайди свій User ID

Hermes Agent використовує числові Telegram User ID для контролю доступу. Твій User ID **не** є твоїм ім’ям користувача — це число типу `123456789`.

**Метод 1 (рекомендовано):** Напиши повідомлення [@userinfobot](https://t.me/userinfobot) — він миттєво відповість твоїм User ID.

**Метод 2:** Напиши повідомлення [@get_id_bot](https://t.me/get_id_bot) — ще один надійний варіант.

Збережи це число; воно знадобиться тобі на наступному кроці.
## Крок 5: Налаштування Hermes

### Варіант A: Інтерактивне налаштування (рекомендовано)

```bash
hermes gateway setup
```

Вибери **Telegram**, коли буде запропоновано. Майстер запитає токен твого бота та дозволені ідентифікатори користувачів, потім запише конфігурацію за тебе.

### Варіант B: Ручна конфігурація

Додай наступне до `~/.hermes/.env`:

```bash
TELEGRAM_BOT_TOKEN=123456789:ABCdefGHIjklMNOpqrSTUvwxYZ
TELEGRAM_ALLOWED_USERS=123456789    # Comma-separated for multiple users
```

### Запуск шлюзу

```bash
hermes gateway
```

Бот має з’явитися в мережі протягом кількох секунд. Надішли йому повідомлення в Telegram, щоб перевірити.
## Надсилання згенерованих файлів з Docker‑базованих терміналів

Якщо бекенд твого терміналу — `docker`, май на увазі, що вкладення Telegram надсилає **процес шлюзу**, а не зсередини контейнера. Це означає, що остаточний шлях `MEDIA:/...` має бути доступний для читання на хості, де працює шлюз.

Типова помилка:

- агент записує файл у Docker за шляхом `/workspace/report.txt`
- модель генерує `MEDIA:/workspace/report.txt`
- доставка в Telegram не вдається, бо `/workspace/report.txt` існує лише всередині контейнера, а не на хості

Рекомендований шаблон:

```yaml
terminal:
  backend: docker
  docker_volumes:
    - "/home/user/.hermes/cache/documents:/output"
```

Потім:

- записуй файли у Docker у каталог `/output/...`
- генеруй **видимий хостом** шлях у `MEDIA:`, наприклад:
  `MEDIA:/home/user/.hermes/cache/documents/report.txt`

Якщо у тебе вже є розділ `docker_volumes:`, додай нове монтування до того ж списку. Дублікати ключів у YAML без повідомлення перезаписують попередні значення.

### Підтримувані розширення файлів `MEDIA:`

Шлюз витягує теги `MEDIA:/path/to/file` з відповідей агента і надсилає вказаний файл як нативне вкладення платформи. Підтримувані розширення на всіх платформах шлюзу:

| Категорія | Розширення |
|---|---|
| Зображення | `png`, `jpg`, `jpeg`, `gif`, `webp`, `bmp`, `tiff`, `svg` |
| Аудіо | `mp3`, `wav`, `ogg`, `m4a`, `opus`, `flac`, `aac` |
| Відео | `mp4`, `mov`, `webm`, `mkv`, `avi` |
| **Документи** | `pdf`, `txt`, `md`, `csv`, `json`, `xml`, `html`, `yaml`, `yml`, `log` |
| **Офіс** | `docx`, `xlsx`, `pptx`, `odt`, `ods`, `odp` |
| **Архіви** | `zip`, `rar`, `7z`, `tar`, `gz`, `bz2` |
| **Книги / пакети** | `epub`, `apk`, `ipa` |

Все, що входить до цього списку, доставляється як нативне вкладення на платформах, які його підтримують (Telegram, Discord, Signal, Slack, WhatsApp, Feishu, Matrix тощо); на платформах без нативної підтримки це переходить у посилання або простий текстовий індикатор. **Жирні** категорії були додані в останніх випусках — якщо ти розраховував на те, що модель скаже `here is the file: /path/to/report.docx`, заміни це на `MEDIA:/path/to/report.docx` для нативної доставки.
## Режим webhook

За замовчуванням Hermes підключається до Telegram за допомогою **long polling** — gateway надсилає вихідні запити до серверів Telegram, щоб отримати нові оновлення. Це добре працює для локальних та завжди‑увімкнених розгортань.

Для **хмарних розгортань** (Fly.io, Railway, Render тощо) **режим webhook** є більш економічним. Ці платформи можуть автоматично пробуджувати призупинені машини при вхідному HTTP‑трафіку, але не при вихідних з’єднаннях. Оскільки polling — це вихідний запит, бот у режимі polling ніколи не може спати. Режим webhook змінює напрямок — Telegram надсилає оновлення на HTTPS‑URL твого бота, що дозволяє спати під час бездіяльності.

| | Polling (за замовчуванням) | Webhook |
|---|---|---|
| Напрямок | Gateway → Telegram (вихідний) | Telegram → Gateway (вхідний) |
| Найкраще для | Локальні, завжди‑увімкнені сервери | Хмарні платформи з авто‑пробудженням |
| Налаштування | Ніяких додаткових параметрів | Встановити `TELEGRAM_WEBHOOK_URL` |
| Вартість у стані бездіяльності | Машина повинна залишатися запущеною | Машина може спати між повідомленнями |

### Конфігурація

Додай наступне до `~/.hermes/.env`:

```bash
TELEGRAM_WEBHOOK_URL=https://my-app.fly.dev/telegram
TELEGRAM_WEBHOOK_SECRET="$(openssl rand -hex 32)"  # required
# TELEGRAM_WEBHOOK_PORT=8443        # optional, default 8443
```

| Змінна | Обов’язково | Опис |
|----------|----------|-------------|
| `TELEGRAM_WEBHOOK_URL` | Так | Публічний HTTPS‑URL, куди Telegram надсилатиме оновлення. Шлях URL автоматично витягується (наприклад, `/telegram` у наведеному вище прикладі). |
| `TELEGRAM_WEBHOOK_SECRET` | **Так** (коли встановлено `TELEGRAM_WEBHOOK_URL`) | Секретний токен, який Telegram повертає в кожному запиті webhook для верифікації. Gateway відмовиться запускатися без нього — див. [GHSA-3vpc-7q5r-276h](https://github.com/NousResearch/hermes-agent/security/advisories/GHSA-3vpc-7q5r-276h). Згенеруй за допомогою `openssl rand -hex 32`. |
| `TELEGRAM_WEBHOOK_PORT` | Ні | Локальний порт, на якому сервер webhook слухає (за замовчуванням: `8443`). |

Коли встановлено `TELEGRAM_WEBHOOK_URL`, gateway запускає HTTP‑сервер webhook замість polling. Якщо змінна не задана, використовується режим polling — без змін у поведінці порівняно з попередніми версіями.

### Приклад хмарного розгортання (Fly.io)

1. Додай змінні середовища до секретів твого застосунку Fly.io:

```bash
fly secrets set TELEGRAM_WEBHOOK_URL=https://my-app.fly.dev/telegram
fly secrets set TELEGRAM_WEBHOOK_SECRET=$(openssl rand -hex 32)
```

2. Відкрий порт webhook у файлі `fly.toml`:

```toml
[[services]]
  internal_port = 8443
  protocol = "tcp"

  [[services.ports]]
    handlers = ["tls", "http"]
    port = 443
```

3. Розгорни:

```bash
fly deploy
```

У логах gateway має з’явитися: `[telegram] Connected to Telegram (webhook mode)`.
## Підтримка проксі

Якщо API Telegram заблоковано або потрібно направляти трафік через проксі, встанови специфічний для Telegram URL проксі. Це має пріоритет над загальними змінними середовища `HTTPS_PROXY` / `HTTP_PROXY`.

**Option 1: config.yaml (рекомендовано)**

```yaml
telegram:
  proxy_url: "socks5://127.0.0.1:1080"
```

**Option 2: змінна середовища**

```bash
TELEGRAM_PROXY=socks5://127.0.0.1:1080
```

Підтримувані схеми: `http://`, `https://`, `socks5://`.

Проксі застосовується як до основного з’єднання Telegram, так і до запасного (fallback) IP‑транспорту. Якщо специфічний проксі для Telegram не встановлено, шлюз повертається до `HTTPS_PROXY` / `HTTP_PROXY` / `ALL_PROXY` (або автоматичного визначення системного проксі macOS).
## Канал дому

Використай команду `/sethome` у будь‑якому чаті Telegram (прямому повідомленні або групі), щоб позначити його як **домашній канал**. Заплановані завдання (cron‑jobs) надсилають свої результати в цей канал.

Ти також можеш встановити його вручну у `~/.hermes/.env`:

```bash
TELEGRAM_HOME_CHANNEL=-1001234567890
TELEGRAM_HOME_CHANNEL_NAME="My Notes"
```

:::tip
Ідентифікатори групових чатів – це від’ємні числа (наприклад, `-1001234567890`). Твій особистий DM‑чат має той самий ідентифікатор, що й твій користувач.
:::

### Доставки cron у режимі тем

Якщо у твоєму боті ввімкнено режим тем у DM, повідомлення cron, які надходять у кореневий чат, потрапляють у системну лобі — відповісти там не відкриває сесію, і ти бачиш повідомлення «головний чат зарезервовано для системних команд». Створи окрему тему форуму (наприклад, `Cron`) і встанови:

```bash
TELEGRAM_CRON_THREAD_ID=<topic_thread_id>
```

`TELEGRAM_CRON_THREAD_ID` перевизначає `TELEGRAM_HOME_CHANNEL_THREAD_ID` лише для доставок cron. Відповіді в цій темі продовжують існуючу сесію теми.
## Голосові повідомлення

### Вхідні голосові (Speech-to-Text)

Голосові повідомлення, які ти надсилаєш у Telegram, автоматично транскрибуються налаштованим провайдером STT Hermes і вставляються як текст у розмову.

- `local` використовує `faster-whisper` на машині, де запущений Hermes — API‑ключ не потрібен
- `groq` використовує Groq Whisper і потребує `GROQ_API_KEY`
- `openai` використовує OpenAI Whisper і потребує `VOICE_TOOLS_OPENAI_KEY`

#### Пропуск STT: передай сирий аудіофайл агенту

Якщо ти хочеш, щоб **сам агент** обробляв аудіо — для діаризації, власного інструменту транскрипції або просто архівування запису — встанови `stt.enabled: false` у `~/.hermes/config.yaml`:

```yaml
stt:
  enabled: false
```

При вимкненому STT шлюз все одно завантажує голосове/аудіо‑вкладення у кеш аудіо Hermes, **але не транскрибує його**. Агент отримує повідомлення з маркером типу:

```
[The user sent a voice message: /home/<user>/.hermes/cache/audio/<hash>.ogg]
```

Твої інструменти або навички можуть потім безпосередньо читати цей шлях (наприклад, передати його в локальний конвеєр діаризації, більш потужну модель транскрипції або завантажити у довготривале сховище). Розширення файлу відповідає оригінальному формату, яким Telegram доставив його (`.ogg` для голосових нотаток, `.mp3`/`.m4a`/тощо для аудіо‑вкладень).

Це природньо поєднується з розділом [local Bot API server](#large-files-20mb-via-local-bot-api-server) нижче, який піднімає обмеження Telegram у 20 МБ на `getFile` до 2 ГБ — корисно, коли записи, які треба обробити, довші за кілька хвилин.

### Вихідні голосові (Text-to-Speech)

Коли агент генерує аудіо за допомогою TTS, воно надсилається як рідні **голосові бульбашки** Telegram — круглі, відтворювані вбудовано.

- **OpenAI і ElevenLabs** генерують Opus нативно — додаткових налаштувань не потрібно
- **Edge TTS** (провайдер за замовчуванням, безкоштовний) виводить MP3 і потребує **ffmpeg** для конвертації в Opus:

```bash
# Ubuntu/Debian
sudo apt install ffmpeg

# macOS
brew install ffmpeg
```

Без ffmpeg аудіо Edge TTS надсилається як звичайний аудіофайл (все ще відтворюваний, але використовується прямокутний плеєр замість голосової бульбашки).

Налаштуй провайдера TTS у своєму `config.yaml` під ключем `tts.provider`.
## Великі файли (>20 МБ) через локальний Bot API сервер

Публічний Bot API Telegram обмежує завантаження `getFile` **20 МБ**, тому будь‑яка голосова нотатка, аудіофайл, відео чи документ більшого розміру без повідомлення відхиляються Hermes з відповіддю «занадто великий». Документований спосіб обійти це — запустити **локальний** [telegram-bot-api](https://github.com/tdlib/telegram-bot-api) демон — те саме серверне ПЗ, яке використовує Telegram, лише у твоїй мережі. Локальний сервер підвищує ліміт файлів до **2 ГБ**, а Hermes автоматично знімає власне внутрішнє обмеження, коли бачить налаштований `base_url`.

Це відкриває такі робочі процеси:

- Надсилання довгих голосових нотаток (45‑хвилинні зустрічі, подкасти) боту
- Завантаження великих відео для обробки інструментом vision‑tool
- Архівація сирих аудіо для офлайн‑конвеєрів, наприклад діаризації, вирівнювання чи навчальних даних

### Крок 1: Отримай облікові дані Telegram API

Локальний сервер спілкується безпосередньо з MTProto‑шаром Telegram (а не з публічним Bot API), тому потрібні **MTProto‑облікові дані**:

1. Перейди на [my.telegram.org/apps](https://my.telegram.org/apps) і ввійди у свій обліковий запис Telegram.
2. Створи новий застосунок (будь‑яка назва та короткий опис підходять).
3. Скопіюй `api_id` та `api_hash` — обидва обов’язкові.

### Крок 2: Запусти сервер telegram-bot-api

Підтримуваний спільнотою образ Docker [`aiogram/telegram-bot-api`](https://hub.docker.com/r/aiogram/telegram-bot-api) — найпростіший шлях. Мінімальний `docker‑compose.yaml` (використовуй режим `--local`, щоб увімкнути підвищені ліміти):

```yaml
services:
  tg-bot-api:
    image: aiogram/telegram-bot-api:latest
    container_name: tg-bot-api
    restart: unless-stopped
    ports:
      - "127.0.0.1:8081:8081"   # bind to loopback only; see security note
    environment:
      TELEGRAM_API_ID: "12345"           # your api_id from Step 1
      TELEGRAM_API_HASH: "abcdef..."     # your api_hash from Step 1
      TELEGRAM_LOCAL: "1"                # enable --local mode (raises 20MB → 2GB)
    volumes:
      - ./tg-bot-api-data:/var/lib/telegram-bot-api
```

Запусти його:

```bash
docker compose up -d tg-bot-api
docker logs --tail 20 tg-bot-api
```

:::warning Security
Локальний Bot API сервер отримує токен бота у шляху URL (наприклад `/bot<TOKEN>/getMe`) **без додаткової автентифікації**. Будь‑хто, хто може дістатися до порту, отримує повний контроль над ботом — читає всі повідомлення, надсилає їх від імені бота тощо. Прив’язуй контейнер до `127.0.0.1` і/або розміщуй його за зворотним проксі у приватній мережі. **Ніколи не відкривай порт 8081 у публічний інтернет.**
:::

### Крок 3: Вийди бота з публічного API (одноразово)

Бот може бути активним лише на **одному** Bot API сервері одночасно. Якщо твій бот вже працював проти `api.telegram.org` (а це, швидше за все, так), треба явно вийти звідти, перш ніж локальний сервер його прийме:

```bash
curl "https://api.telegram.org/bot<YOUR_BOT_TOKEN>/logOut"
# expected response: {"ok":true,"result":true}
```

Це одноразовий крок міграції — його не треба повторювати при кожному перезапуску. Telegram доставляє всі повідомлення, отримані після `logOut`, через новий сервер.

Перевір, чи локальний сервер може спілкуватися з Telegram від імені бота:

```bash
curl "http://127.0.0.1:8081/bot<YOUR_BOT_TOKEN>/getMe"
# expected response: {"ok":true,"result":{"id":...,"is_bot":true,...}}
```

### Крок 4: Вкажи Hermes локальний сервер

Додай URL‑и у `platforms.telegram.extra` у `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      base_url: "http://127.0.0.1:8081/bot"
      base_file_url: "http://127.0.0.1:8081/file/bot"
      local_mode: true        # see Step 5 below — only set this if the bot's data
                              # directory is readable by the Hermes process
```

:::caution Use `platforms.telegram.extra`, not `telegram.extra`
Наразі лише форма `platforms.<name>.extra` глибоко зливається у конфігурацію платформи. Ключі, розташовані безпосередньо під верхньорівневим блоком `telegram.extra`, без повідомлення ігноруються.
:::

Коли встановлено `base_url`, Hermes:

- Будує клієнт python‑telegram‑bot проти локального сервера
- Автоматично піднімає внутрішній ліміт розміру документів/аудіо з 20 МБ → 2 ГБ
- Повідомляє активний ліміт у повідомленні про помилку «занадто великий» (`Maximum: 2048 MB.`), щоб було зрозуміло, в якому режимі ти працюєш

Перезапусти шлюз і шукай рядок підтвердження у логах:

```bash
hermes gateway restart
grep -E "Using custom Telegram base_url|Using Telegram local_mode" ~/.hermes/logs/gateway.log | tail
```

### Крок 5: `local_mode` — доступ до файлів на диску

Локальний сервер має **два способи** доставки файлів:

1. **Без `--local`** (за замовчуванням): файли подаються по HTTP за `/file/bot<TOKEN>/<path>`, як у публічному Bot API. Ліміт 20 МБ залишається. Корисно лише як мережеве рішення (наприклад, коли `api.telegram.org` недоступний, а ти можеш самостійно хостити); не підходить для підвищення розміру.
2. **З `--local`** (встановлюється через `TELEGRAM_LOCAL=1` вище): файли записуються у файлову систему сервера, а відповідь `getFile` повертає **абсолютний шлях** замість HTTP‑URL. Ліміт 20 МБ знімається. Hermes тоді читає байти **з диска**, а не по HTTP.

Щоб шлях читання з диска працював, встанови `local_mode: true` у конфігурації вище **і** переконайся, що процес Hermes може читати шлях, який повертає сервер. Два сценарії:

- **Той самий хост** — telegram‑bot‑api і Hermes працюють на одній машині. Прив’яжи том даних до каталогу, який Hermes може читати (наприклад `/var/lib/telegram-bot-api`), і впевнись, що права власності збігаються. Контейнер спускає привілеї до внутрішнього користувача `telegram-bot-api` (uid залежить від образу); найпростіше додати `user: "<UID>:<GID>"` у сервіс compose, щоб файли належали uid, під яким вже працює Hermes.
- **Різні машини** — сервер бота працює на одному хості (NAS, окрема VM), а Hermes — на іншому. Каталог даних сервера має бути спільним з машиною Hermes за **тим самим абсолютним шляхом**, який сервер повідомляє (зазвичай `/var/lib/telegram-bot-api`). NFS добре підходить; CIFS/SMB з параметром `uid=` для переназначення uid зручніший, якщо не хочеш розбиратись з невідповідністю uid на рівні файлової системи.

Якщо `local_mode: true` встановлено, а Hermes не може `stat` повернений шлях (проблеми з правами або неправильне монтування), python‑telegram‑bot тихо повертається до HTTP‑`getFile` до локального сервера — який у режимі `--local` відповідає `404 Not Found`. Симптом проявляється у `gateway.log` так:

```
[Telegram] Failed to cache voice: Not Found
telegram.error.InvalidToken: Not Found
```

Якщо бачиш це, підвищення ліміту працює, а спільний доступ до файлів — ні. Перевір `ls -la /var/lib/telegram-bot-api/<TOKEN>/voice/` з хоста Hermes під користувачем, від якого запускається шлюз, і впевнись, що файл можна `cat`‑нути без помилки прав.

### Крок 6: Перевірка

Надішли боту голосову нотатку або аудіофайл, розмір якого перевищує 20 МБ. Підключи `tail` до логу шлюзу:

```bash
tail -f ~/.hermes/logs/gateway.log | grep -iE "telegram|cache"
```

Ти маєш побачити рядок типу `[Telegram] Cached user voice at /home/<user>/.hermes/cache/audio/...` і **не** отримати відмову «занадто великий». У поєднанні з `stt.enabled: false` (вище) шлях до оригінального аудіофайлу потрапляє у вхідне повідомлення агента для подальшої обробки.
## Використання групових чатів

Hermes Agent працює в групових чатах Telegram з кількома зауваженнями:

- **Privacy mode** визначає, які повідомлення бот може бачити (див. [Step 3](#step-3-privacy-mode-critical-for-groups))
- `TELEGRAM_ALLOWED_USERS` все ще діє — лише уповноважені користувачі можуть ініціювати бота, навіть у групах
- Ти можеш запобігти відповіді бота на звичайну групову розмову за допомогою `telegram.require_mention: true`
- При `telegram.require_mention: true` групові повідомлення приймаються, коли вони:
  - є відповіддю на одне з повідомлень бота
  - містять згадку `@botusername`
  - `/command@botusername` (форма команди меню бота Telegram, що включає ім’я бота)
  - збігаються з одним із налаштованих regex‑слово‑пробудження у `telegram.mention_patterns`
- У групах з кількома Hermes‑ботами `telegram.exclusive_bot_mentions` робить маршрутизацію детермінованою. Коли повідомлення явно згадує один або кілька імен користувачів Telegram‑бота, лише згадані профілі бота його обробляють; інші Hermes‑боти ігнорують його до того, як запрацюють відповіді та запасні (fallback) тригери. Це ввімкнено за замовчуванням.
- Використовуй `telegram.ignored_threads`, щоб Hermes мовчав у конкретних темах форуму Telegram, навіть коли група інакше дозволяє вільні відповіді або відповіді за згадкою.
- Якщо `telegram.require_mention` залишено незаданим або `false`, Hermes зберігає попередню поведінку відкритої групи і відповідає на звичайні групові повідомлення, які він може бачити.

### Кілька Hermes‑ботів в одній групі

Якщо ти запускаєш кілька профілів Hermes в одному Telegram‑групі, створи окремий токен Telegram‑бота для кожного профілю і запусти окремий шлюз для кожного профілю. Не використовуйте один і той самий токен бота в кількох запущених шлюзах; Telegram відхилить одночасне опитування одного токену.

**Рекомендована конфігурація групи:**

```yaml
telegram:
  require_mention: true
  exclusive_bot_mentions: true
  mention_patterns: []
```

З такою налаштуванням групове повідомлення типу `@research_bot @ops_bot summarize this` обробляється лише `research_bot` та `ops_bot`. Інші Hermes‑боти в групі залишаються мовчазними, навіть якщо повідомлення є відповіддю на одне з їхніх попередніх повідомлень або інакше підходить під спільне слово‑пробудження.

Встанови `exclusive_bot_mentions: false` лише для застарілих груп, де явні згадки не повинні переважати над відповідями та тригерами слів‑пробудження.

Щоб працювати з кількома профілями, запускай команду шлюзу один раз для кожного профілю. Наприклад:

```bash
# default profile
hermes gateway start
hermes gateway status
hermes gateway stop

# named profiles
hermes -p research gateway start
hermes -p research gateway status
hermes -p research gateway stop
```

Для невеликого фіксованого флоту використай цикл оболонки або скрипт, який викликає `hermes gateway <action>` для профілю за замовчуванням і `hermes -p <profile> gateway <action>` для кожного іменованого профілю. Це надійніше, ніж припускати, що одна команда на рівні процесу керує кожним іменованим профілем у кожному менеджері сервісів.

### Усунення неполадок: працює в ДМ, а не в групах

Якщо бот відповідає у приватному чаті, але мовчить у групі, перевіряй ці «шлюзи» у порядку:

1. **Доставка Telegram:** вимкни режим приватності BotFather, підвищи бота до адміністратора або згадуй бота безпосередньо. Hermes не може відповісти на групові повідомлення, які Telegram ніколи не доставляє боту.
2. **Повторне приєднання після зміни приватності:** видали бота з групи та додай його знову після зміни налаштувань приватності BotFather. Telegram може зберігати стару поведінку доставки для існуючих членств.
3. **Авторизація Hermes:** переконайся, що відправник зазначений у `TELEGRAM_ALLOWED_USERS` або `TELEGRAM_GROUP_ALLOWED_USERS`, або дозволь груповий чат за допомогою `TELEGRAM_GROUP_ALLOWED_CHATS`.
4. **Фільтри згадок:** якщо встановлено `telegram.require_mention: true`, звичайна групова розмова ігнорується, якщо повідомлення не є slash‑командою, відповіддю боту, згадкою `@botusername` або не відповідає налаштованим `mention_patterns`.
5. **Маршрутизація багатоботних груп:** якщо група містить кілька ботів, переконайся, що кожен профіль Hermes використовує унікальний токен бота і залишай `exclusive_bot_mentions` увімкненим, якщо лише не хочеш навмисно стару спільну поведінку тригерів.

Від’ємні ідентифікатори чатів — це нормально для груп та супергруп Telegram. Якщо ти використовуєш авторизацію за межами чату, розмісти ці ідентифікатори у `TELEGRAM_GROUP_ALLOWED_CHATS`, а не у списку дозволених користувачів‑відправників.

### Приклад конфігурації тригеру групи

Додай це до `~/.hermes/config.yaml`:

```yaml
telegram:
  require_mention: true
  exclusive_bot_mentions: true
  mention_patterns:
    - "^\\s*chompy\\b"
  ignored_threads:
    - 31
    - "42"
```

Цей приклад дозволяє всі звичні прямі тригери плюс повідомлення, що починаються з `chompy`, навіть якщо вони не використовують `@mention`. Повідомлення у темах Telegram `31` та `42` завжди ігноруються до того, як запускаються перевірки згадок та вільних відповідей.

### Примітки щодо `mention_patterns`

- Шаблони використовують регулярні вирази Python
- Пошук нечутливий до регістру
- Шаблони перевіряються як у текстових повідомленнях, так і у підписах медіа
- Неправильні регулярні вирази ігноруються з попередженням у логах шлюзу, а не призводять до падіння бота
- Якщо потрібно, щоб шаблон збігався лише на початку повідомлення, зафіксуй його символом `^`
## Приватні теми чату (Bot API 9.4)

Telegram Bot API 9.4 (лютий 2026) представив **приватні теми чату** — боти можуть створювати теми у стилі форуму безпосередньо в 1‑on‑1 DM‑чатах, без необхідності суперґрупи. Це дозволяє запускати кілька ізольованих робочих просторів у твоєму існуючому DM з Hermes.

### Випадок використання

Якщо ти працюєш над кількома довготривалими проєктами, теми зберігають їхній контекст окремо:

- **Тема "Website"** — робота над твоїм продакшн‑веб‑сервісом
- **Тема "Research"** — огляд літератури та дослідження статей
- **Тема "General"** — різноманітні завдання та швидкі питання

Кожна тема отримує свою сесію розмови, історію та контекст — повністю ізольовані від інших.

### Конфігурація

:::caution Prerequisites
Перш ніж додавати теми до конфігурації, користувач повинен **увімкнути режим Topics** у DM‑чаті з ботом:

1. Відкрий приватний чат з Hermes‑ботом у Telegram
2. Натисни на ім’я бота у верхній частині, щоб відкрити інформацію про чат
3. Увімкни **Topics** (перемикач, що перетворює чат у форум)

Без цього Hermes під час запуску виведе `The chat is not a forum` і пропустить створення теми. Це налаштування клієнта Telegram — бот не може увімкнути його програмно.
:::

Додай теми у `platforms.telegram.extra.dm_topics` у `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      dm_topics:
      - chat_id: 123456789        # Your Telegram user ID
        topics:
        - name: General
          icon_color: 7322096
        - name: Website
          icon_color: 9367192
        - name: Research
          icon_color: 16766590
          skill: arxiv              # Auto-load a skill in this topic
```

**Поля:**

| Поле | Обов’язково | Опис |
|------|--------------|------|
| `name` | Yes | Відображувана назва теми |
| `icon_color` | No | Код кольору іконки Telegram (ціле число) |
| `icon_custom_emoji_id` | No | ID кастомного емодзі для іконки теми |
| `skill` | No | Skill, який автозавантажується у нових сесіях цієї теми |
| `thread_id` | No | Автозаповнюється після створення теми — не встановлюй вручну |

### Як це працює

1. При запуску шлюзу Hermes викликає `createForumTopic` для кожної теми, у якої ще немає `thread_id`
2. `thread_id` автоматично зберігається назад у `config.yaml` — наступні перезапуски пропускають виклик API
3. Кожна тема відповідає ізольованому ключу сесії: `agent:main:telegram:dm:{chat_id}:{thread_id}`
4. Повідомлення в кожній темі мають власну історію розмов, скидання пам’яті та вікно контексту

### Обробка кореневого DM

За замовчуванням повідомлення, надіслані в кореневий DM (поза будь‑якою темою), обробляються звично. Встанови `ignore_root_dm: true`, щоб перетворити кореневий DM у лобі — звичайні повідомлення будуть тихо ігноруватись у користувачів, у яких налаштовані DM‑теми, тоді як системні команди (`/start`, `/help`, `/status` тощо) залишаться працездатними.

```yaml
platforms:
  telegram:
    extra:
      ignore_root_dm: true
      dm_topics:
        - chat_id: 123456789
          topics:
            - name: General
```

Перевірка виконується **для кожного чату**: лише користувачі, у яких хоча б один запис у `dm_topics`, матимуть змінену поведінку кореневого DM. Користувачі без налаштованих тем залишаються без змін.

### Прив’язка skill

Теми з полем `skill` автоматично завантажують вказаний skill, коли в цій темі стартує нова сесія. Це працює точно так само, як введення `/skill-name` на початку розмови — вміст skill ін’єкціюється у перше повідомлення, і подальші повідомлення бачать його в історії розмов.

Наприклад, тема з `skill: arxiv` матиме попередньо завантажений skill `arxiv` щоразу, коли її сесія скидається (через тайм‑аут бездіяльності, щоденний скидання або ручну команду `/reset`).

:::tip
Теми, створені поза конфігурацією (наприклад, вручну через API Telegram), автоматично виявляються, коли надходить сервісне повідомлення `forum_topic_created`. Ти також можеш додати теми до конфігурації під час роботи шлюзу — вони будуть підхоплені під час наступного кеш‑місу.
:::
## Режим багатосесійного DM (`/topic`)

ChatGPT‑подібний багатосесійний DM — один бот, багато паралельних розмов. На відміну від оператором‑curated `extra.dm_topics` вище, цей режим **керується користувачем**: без конфігурації, без попередньо оголошених імен тем. Кінцевий користувач вмикає його за допомогою `/topic`, а потім натискає кнопку Telegram **+**, щоб створити стільки тем, скільки захоче, кожна з яких — повністю незалежна Hermes сесія.
### `/topic` підкоманди

| Форма | Контекст | Ефект |
|------|----------|-------|
| `/topic` | Root DM, ще не ввімкнено | Перевірити можливості BotFather, увімкнути режим мульти‑сесії, створити закріплену системну тему |
| `/topic` | Root DM, вже ввімкнено | Показати статус: доступні незв’язані сесії для відновлення |
| `/topic` | Усередині теми | Показати поточне прив’язування сесії до теми |
| `/topic help` | Будь‑який | Коротка інлайн‑інструкція |
| `/topic off` | Root DM | Вимкнути режим мульти‑сесії та очистити всі прив’язки тем для цього чату |
| `/topic <session-id>` | Усередині теми | Відновити попередню Telegram‑сесію в поточній темі |

Лише авторизовані користувачі (білий список через `TELEGRAM_ALLOWED_USERS` / конфігурація автентифікації платформи) можуть виконувати `/topic`. Неавторизований відправник отримує відмову замість активації.
### DM Topics vs Multi-session DM mode

| | `extra.dm_topics` (config-driven) | `/topic` (user-driven) |
|---|---|---|
| Хто активує | Оператор, у `config.yaml` | Кінцевий користувач, відправляючи `/topic` |
| Список тем | Фіксований набір, оголошений у конфігурації | Користувач вільно створює/видаляє теми |
| Назви тем | Вибирає оператор | Вибирає користувач; автоматично перейменовується під заголовок сесії Hermes |
| Поведінка кореневого DM | Звичайний чат (лобі, якщо `ignore_root_dm: true`) | Перетворюється на системне лобі (не‑командні повідомлення відхиляються) |
| Основний випадок використання | Постійні робочі простори з необов’язковим прив’язуванням навичок | Ад‑hoc паралельні сесії |
| Збереження | `extra.dm_topics` у конфігурації | Таблиці SQLite `telegram_dm_topic_mode` + `telegram_dm_topic_bindings` |

Обидві функції можуть співіснувати в одному боті — ти запускаєш `/topic` у DM користувача, а `extra.dm_topics` продовжує керувати темами, оголошеними оператором, для інших чатів.
### Prerequisites

У **@BotFather** відкрий свого бота → **Bot Settings → Threads Settings**:

1. Увімкни **Threaded Mode** (вмикає `has_topics_enabled`);
2. **Не** вимикай можливість користувачам створювати теми (залишає `allows_users_to_create_topics` увімкненим).

Коли користувач вперше виконує `/topic`, Hermes викликає `getMe` для перевірки обох прапорців. Якщо якийсь з них вимкнено, Hermes надсилає скріншот сторінки налаштувань Threads у BotFather і пояснює, що потрібно перемкнути — активація не відбудеться, доки не будуть виконані всі вимоги.
### Потік активації

З кореневого DM надішли:

```
/topic
```

Hermes виконає:

1. Перевірить `getMe().has_topics_enabled` та `allows_users_to_create_topics`.
2. Якщо обидва значення true, увімкне багатосесійний режим тем для цього DM.
3. Створить і закріпить тему **System** для статусу/команд (за можливості).
4. Відповість списком попередніх незв’язаних Telegram‑сесій, які користувач може відновити.

Після активації **кореневий DM стає лобі**: звичайні запити відхиляються з рекомендацією перейти до **All Messages**. Системні команди (`/status`, `/sessions`, `/usage`, `/help` тощо) продовжують працювати в кореневому DM.
### Створення нової теми (потік користувача)

1. Відкрий DM‑бота в Telegram
2. Натисни **All Messages** у верхній частині інтерфейсу бота, потім надішли будь‑яке повідомлення
3. Telegram створює нову тему для цього повідомлення
4. Hermes відповідає всередині цієї теми — тема тепер є окремою сесією

Кожна тема має власну історію розмови, стан моделі, виконання інструментів та ідентифікатор сесії. Ключ ізоляції — `agent:main:telegram:dm:{chat_id}:{thread_id}` — ідентичний ізоляції тем DM, що задається конфігурацією.
### Автоматично перейменовані теми

Коли Hermes генерує назву сесії для теми (через конвеєр автозаголовку після першого обміну), сама тема в Telegram перейменовується, щоб відповідати — напр., «New Topic» стає «Database migration plan». Перейменування виконується за принципом «best‑effort»: помилки реєструються, але не переривають сесію.

Щоб вимкнути це і залишити вручну обрані назви тем без змін, встанови:

```yaml
gateway:
  platforms:
    telegram:
      extra:
        disable_topic_auto_rename: true
```

Коли цей прапорець увімкнено, Hermes все одно генерує внутрішню назву сесії (використовується `hermes sessions`, TUI тощо), але ніколи не змінює назву теми в Telegram. Це корисно, коли ти організовуєш теми вручну в режимі Threaded Mode BotFather і не хочеш, щоб кожна перша відповідь перезаписувала назву.
### `/new` всередині теми

Скидає сесію поточної теми (новий ідентифікатор сесії, чиста історія), не зачіпаючи інші теми. Hermes відповідає нагадуванням, що для паралельної роботи зазвичай варто створити іншу тему (через **All Messages**).
### Відновлення попередньої сесії

Всередині теми надішли:

```
/topic <session-id>
```

Це прив’язує поточну тему до існуючої сесії Hermes замість створення нової. Корисно для продовження розмови, яка розпочалася до ввімкнення режиму теми. Обмеження:

- Цільова сесія має належати тому самому користувачу Telegram
- Цільова сесія не повинна вже бути прив’язана до іншої теми

Hermes підтверджує назвою сесії та відтворює останнє повідомлення асистента для контексту.

Щоб дізнатися ідентифікатори сесій, надішли `/topic` (без аргументу) у кореневому DM — Hermes виведе список непов’язаних Telegram‑сесій користувача.
### `/topic` inside a topic (no argument)

Показує поточне прив’язування теми: назву сесії, ID сесії та підказки щодо `/new` замість створення іншої теми.
### Під капотом

- Активація зберігається у `telegram_dm_topic_mode(chat_id, user_id, enabled, ...)` у `state.db`.
- Кожне прив’язування теми зберігається у `telegram_dm_topic_bindings(chat_id, thread_id, session_id, ...)` з `ON DELETE CASCADE` на `session_id` — видалення сесії автоматично очищає її прив’язку до теми.
- Міграція режиму тем у SQLite **opt‑in**: вона виконується під час першого виклику `/topic`, ніколи під час запуску шлюзу. Поки користувач не виконає `/topic` у цьому профілі, `state.db` залишається незмінним.
- Кожне вхідне DM‑повідомлення шукає свою прив’язку `(chat_id, thread_id)`. Якщо вона існує, пошук маршрутизує повідомлення до прив’язаної сесії через `SessionStore.switch_session()`, тож відображення ключ‑сесії‑до‑id‑сесії залишається послідовним на диску.
- `/new` всередині теми переписує рядок прив’язки, вказуючи на новий `session_id`, тому наступне повідомлення залишиться у новій сесії.
- Теми, оголошені в `extra.dm_topics`, **ніколи не перейменовуються автоматично** — обране оператором ім’я зберігається, навіть коли ввімкнено режим багатосесійності.
- Встанови `extra.disable_topic_auto_rename: true`, щоб вимкнути автоматичне перейменування **усіх** тем у чаті (включаючи ad‑hoc теми, створені через Threaded Mode).
- Загальна (закріплена вгорі) тема у DM з увімкненим форумом розглядається як коренева лобі, незалежно від того, чи надсилає Telegram її повідомлення з `message_thread_id=1`, чи без `thread_id`.
- Нагадування у кореневій лобі обмежені до одного повідомлення кожні 30 секунд на чат — користувач, який забув, що режим тем увімкнено, і надсилає десять запитів у корені, не отримає десяти відповідей.
- Скриншоти налаштувань BotFather обмежені до однієї відправки кожні 5 хвилин на чат — повторні спроби `/topic`, доки налаштування Threads залишаються вимкненими, не перезавантажать те саме зображення.
- `/background <prompt>`, запущений всередині теми, повертає результат у ту ж тему; фонові сесії не викликають автоматичного перейменування власної теми.
- `/topic` сам по собі підлягає перевірці авторизації користувача бота — неавторизовані DM отримують відмову замість активації.
### Вимкнення режиму багатосесійності

Надішли `/topic off` у кореневому DM. Hermes вимикає рядок, очищає прив’язки чату `(thread_id → session_id)`, і кореневий DM повертається до звичайного чату Hermes. Існуючі теми в Telegram не видаляються — вони просто перестають бути окремими сесіями. Пізніше запусти `/topic` знову, щоб увімкнути їх.

Якщо потрібно очистити вручну (наприклад, масове скидання у багатьох чатах), видали рядки безпосередньо:

```bash
sqlite3 ~/.hermes/state.db \
  "UPDATE telegram_dm_topic_mode SET enabled = 0 WHERE chat_id = '<your_chat_id>'; \
   DELETE FROM telegram_dm_topic_bindings WHERE chat_id = '<your_chat_id>';"
```
### Пониження версії Hermes

Якщо ти понизиш версію Hermes до такої, що передує `/topic`, функція просто перестає працювати — таблиці `telegram_dm_topic_mode` і `telegram_dm_topic_bindings` залишаються в `state.db`, але ігноруються старим кодом. Прямі повідомлення повертаються до рідної ізоляції за потоками (кожен `message_thread_id` і надалі отримує свою **сесію** через `build_session_key`), тому твої існуючі теми Telegram продовжують працювати як паралельні **сесії**. Кореневе пряме повідомлення більше не є лобі — повідомлення там потрапляють до агента, як і раніше. Повторне оновлення знову активує режим багатосесійності саме там, де він був.
## Прив’язка навички до теми групового форуму

Супергрупи з увімкненим **режимом Тем** (також називаються «теми форуму») вже мають ізоляцію сесій за темою — кожен `thread_id` відповідає власній розмові. Але ти можеш захотіти **автозавантажувати навичку**, коли повідомлення надходять у певну тему групи, так само, як працює прив’язка навички до теми в DM.

### Випадок використання

Супергрупа команди з темами форуму для різних робочих потоків:

- Тема **Engineering** → автозавантажує навичку `software-development`
- Тема **Research** → автозавантажує навичку `arxiv`
- Тема **General** → без навички, універсальний помічник

### Конфігурація

Додай прив’язки тем у `platforms.telegram.extra.group_topics` у `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      group_topics:
      - chat_id: -1001234567890       # Supergroup ID
        topics:
        - name: Engineering
          thread_id: 5
          skill: software-development
        - name: Research
          thread_id: 12
          skill: arxiv
        - name: General
          thread_id: 1
          # No skill — general purpose
```

**Поля:**

| Поле | Обов’язково | Опис |
|------|-------------|------|
| `chat_id` | Так | Числовий ID супергрупи (від’ємне число, що починається з `-100`) |
| `name` | Ні | Людсько‑читабельна мітка теми (лише для інформації) |
| `thread_id` | Так | ID теми форуму Telegram — видно у посиланнях `t.me/c/<group_id>/<thread_id>` |
| `skill` | Ні | Навичка, яку автозавантажити у нових сесіях у цій темі |

### Як це працює

1. Коли повідомлення надходить у прив’язану тему групи, Hermes шукає `chat_id` і `thread_id` у конфігурації `group_topics`.
2. Якщо знайдено запис із полем `skill`, ця навичка автозавантажується для сесії — так само, як прив’язка навички до теми в DM.
3. Теми без ключа `skill` отримують лише ізоляцію сесії (існуюча поведінка, без змін).
4. Неприв’язані значення `thread_id` або `chat_id` просто ігноруються — без помилок, без навички.

### Відмінності від DM‑тем

| | DM‑теми | Групові теми |
|---|---|---|
| Ключ конфігурації | `extra.dm_topics` | `extra.group_topics` |
| Створення теми | Hermes створює теми через API, якщо `thread_id` відсутній | Адміністратор створює теми в інтерфейсі Telegram |
| `thread_id` | Автозаповнюється після створення | Потрібно встановити вручну |
| `icon_color` / `icon_custom_emoji_id` | Підтримуються | Не застосовується (зовнішній вигляд контролює адміністратор) |
| Прив’язка навички | ✓ | ✓ |
| Ізоляція сесії | ✓ | ✓ (вже вбудована для тем форуму) |

:::tip
Щоб знайти `thread_id` теми, відкрий тему в Telegram Web або Desktop і подивись на URL: `https://t.me/c/1234567890/5` — останнє число (`5`) є `thread_id`. `chat_id` для супергрупи — це ID групи з префіксом `-100` (наприклад, група `1234567890` стає `-1001234567890`).
:::
## Останні функції Bot API

- **Bot API 9.4 (Feb 2026):** Приватні теми чату — боти можуть створювати теми форуму в 1‑on‑1 DM‑чатах за допомогою `createForumTopic`. Hermes використовує це для двох окремих функцій: оператором‑curated [Приватні теми чату](#private-chat-topics-bot-api-94) (конфігураційно‑керований, фіксований список тем) та користувачем‑driven [Режим багатосесійних DM](#multi-session-dm-mode-topic) (активується командою `/topic`, не обмежена кількість створених користувачем тем).
- **Політика конфіденційності:** Telegram тепер вимагає, щоб у ботів була політика конфіденційності. Встанови її через BotFather за допомогою `/setprivacy_policy`, або Telegram може автоматично згенерувати заповнювач. Це особливо важливо, якщо твій бот публічний.
- **Bot API 9.5 (Mar 2026):** Нативний стримінг через `sendMessageDraft`. Hermes підтримує нативний API стримінгу чернеток Telegram як опціональний транспорт для приватних чатів. За замовчуванням залишається застарілий шлях `editMessageText`, оскільки попередні перегляди чернеток можуть візуально згортатися та перерисовуватись у деяких клієнтах Telegram.

### Транспорт стримінгу (`gateway.streaming.transport`)

Коли стримінг увімкнено (`gateway.streaming.enabled: true`), Hermes обирає один із чотирьох транспортів:

| Значення | Поведінка |
|---|---|
| `auto` | Нативний стримінг чернеток у підтримуваних чатах (наразі Telegram DMs); застарілий шлях на основі `edit` у іншому випадку. При невдачі кадру чернетки плавно переходить назад. |
| `draft` | Примусово використовувати нативні чернетки. Логує пониження та переходить до `edit`, якщо чат не підтримує чернетки (наприклад, групи/теми). |
| `edit` (default) | Застарілий прогресивний поллінг `editMessageText` для будь‑якого типу чату. |
| `off` | Повністю вимкнути стримінг (лише фінальна відповідь, без прогресивних оновлень). |

У `~/.hermes/config.yaml`:

```yaml
gateway:
  streaming:
    enabled: true
    transport: edit    # edit | auto | draft | off
```

**Що ти побачиш у DM‑чатах з `edit` (за замовчуванням)** — шлюз надсилає звичайне повідомлення‑попередній перегляд і поступово оновлює його через `editMessageText`, уникаючи ефекту згортання/відкату чернетки у Telegram.

**Що ти побачиш у DM‑чатах з `auto` або `draft`** — Telegram показує анімований попередній перегляд чернетки, який оновлюється токен‑за‑токеном. Коли відповідь завершиться, вона доставляється як звичайне повідомлення, а попередній перегляд чернетки природно зникає у клієнті. Чернетки не мають ідентифікатора повідомлення, тому фінальна відповідь залишається в історії чату.

**Що щодо груп, супергруп, тем форуму?** Telegram обмежує `sendMessageDraft` лише приватними чатами (DM). Шлюз прозоро переходить до шляху на основі `edit` для всього іншого — той самий UX, що і раніше.

**Що, якщо кадр чернетки не вдається?** Будь‑яка помилка (тимчасова мережна помилка, відхилення сервером, старіша установка python-telegram-bot) переключає відповідь назад на шлях на основі `edit` на решту стріму. Наступна відповідь отримує нову спробу.
## Rendering: Tables and Link Previews

Telegram's MarkdownV2 has no native table syntax — pipe tables render as backslash-escaped noise if passed through raw. Hermes normalizes markdown tables automatically:

- **Small tables** are flattened into **row-group bullets** — each row becomes a readable bulleted list under the column headings. Good for 2–4 columns and short cells.
- **Larger or wider tables** fall back to a **fenced code block** with aligned columns so nothing collapses. A one-line prompt hint is added so the agent knows to prefer prose follow-ups over more tables on Telegram.

There's nothing to configure — the adapter picks the right fallback per message. If you want the legacy "always code-block" behavior, disable table normalization by setting `telegram.pretty_tables: false` in `config.yaml` (default: `true`).

**Link previews.** Telegram auto-generates link previews for URLs in bot messages. If you'd rather suppress those (long `/tools` output, agent reply that mentions ten links, etc.):

```yaml
gateway:
  platforms:
    telegram:
      extra:
        disable_link_previews: true
```

When enabled, Hermes attaches Telegram's `LinkPreviewOptions(is_disabled=True)` to every outgoing message and falls back to the legacy `disable_web_page_preview` parameter on older `python-telegram-bot` versions.
## Group Allowlisting

Telegram‑групи та чати форуму мають два ортогональні шлюзи, які можна налаштувати:

- **Ідентифікатори користувачів‑відправників** (`group_allow_from` / `TELEGRAM_GROUP_ALLOWED_USERS`) — список дозволених у межах відправника, який застосовується лише до повідомлень груп/форуму. Використовуй це, коли потрібно, щоб певні користувачі могли викликати бота в групах, не додаючи їх до `TELEGRAM_ALLOWED_USERS` (що також надає їм доступ у DM).
- **Ідентифікатори чатів** (`group_allowed_chats` / `TELEGRAM_GROUP_ALLOWED_CHATS`) — список дозволених у межах чату. Будь‑який учасник цих груп/форумів може взаємодіяти з ботом. Корисно для ботів підтримки/команди, коли саме членство в групі є сигналом доступу.

```yaml
gateway:
  platforms:
    telegram:
      extra:
        # Global access (DMs + groups). Users here can always invoke the bot.
        allow_from:
          - "123456789"
        # Sender IDs allowed in groups/forums only. Does NOT grant DM access.
        group_allow_from:
          - "987654321"
        # Entire groups/forums — any member is authorized.
        group_allowed_chats:
          - "-1001234567890"
```

Еквівалентні змінні середовища:

```bash
TELEGRAM_ALLOWED_USERS="123456789"
TELEGRAM_GROUP_ALLOWED_USERS="987654321"
TELEGRAM_GROUP_ALLOWED_CHATS="-1001234567890"
```

**Поведінка**

- `TELEGRAM_ALLOWED_USERS` охоплює всі типи чатів (DM, групи, форуми).
- `TELEGRAM_GROUP_ALLOWED_USERS` авторизує лише зазначених відправників у групах/форумах. Вони все ще не можуть писати боту в DM, якщо не вказані в `TELEGRAM_ALLOWED_USERS`.
- Чат у `TELEGRAM_GROUP_ALLOWED_CHATS` авторизує кожного його учасника, незалежно від відправника.
- Використовуй `*` у будь‑якому з цих параметрів, щоб дозволити будь‑якого відправника/чат.
- Це накладається поверх існуючих тригерів згадок/шаблонів та поверх `group_topics` + `ignored_threads`.

### Міграція з версії до PR #17686

До цього розділення `TELEGRAM_GROUP_ALLOWED_USERS` був єдиним параметром, і користувачі ставили **ідентифікатори чатів** у нього. Для зворотної сумісності значення у вигляді ідентифікаторів чатів (що починаються з `-`) у `TELEGRAM_GROUP_ALLOWED_USERS` все ще розглядаються як ідентифікатори чатів, і один раз виводиться попередження про застарілість. Міграція:

```bash
# Old (still works, but deprecated)
TELEGRAM_GROUP_ALLOWED_USERS="-1001234567890"

# New
TELEGRAM_GROUP_ALLOWED_CHATS="-1001234567890"
```

### Обхід @згадки для гостей (`guest_mode`)

У типовій конфігурації `group_allowed_chats` є жорстким шлюзом: повідомлення з груп, які не входять до списку, тихо відкидаються, навіть якщо учасник явно @згадує бота. Це правильне значення за замовчуванням для ботів підтримки/команди.

Для більш неформальних налаштувань — групові чати друзів, де бот **переважно мовчить**, але **іноді доступний за явним пінгом** — увімкни `guest_mode`:

```yaml
gateway:
  platforms:
    telegram:
      extra:
        group_allowed_chats:
          - "-1001234567890"   # your main allowlisted group
        guest_mode: true       # non-allowlisted groups: allow on @mention only
```

Еквівалент у середовищі:

```bash
TELEGRAM_GUEST_MODE=true
```

За замовчуванням: `false`.

Якщо `guest_mode: true`, повідомлення з не дозволеної групи обробляється **лише**, якщо воно явно @згадує бота. Згадка потрібна кожного разу — немає «прилипання» сесії для взаємодій гостей, тому бот ніколи не ініціює діалог у групі друзів, якщо його не згадали.

DM та дозволені групи працюють точно так само, як і раніше.
## Керування доступом до Slash‑команд

За замовчуванням кожен дозволений користувач може виконувати будь‑яку Slash‑команду. Щоб розділити ваш allowlist на **адміністраторів** (повний доступ до Slash‑команд) і **звичайних користувачів** (лише команди, які ви явно дозволили), додайте `allow_admin_from` і `user_allowed_commands` у блок `extra` платформи:

```yaml
gateway:
  platforms:
    telegram:
      extra:
        # Existing allowlists (unchanged)
        allow_from:
          - "123456789"     # admin
          - "555555555"     # regular user
          - "777777777"     # regular user

        # NEW — admins get all slash commands (built-in + plugin)
        allow_admin_from:
          - "123456789"

        # NEW — non-admin allowed users can only run these slash commands.
        # /help and /whoami are always allowed so users can see their access.
        user_allowed_commands:
          - status
          - model
          - history

        # Optional: separate admin/command lists for groups
        group_allow_admin_from:
          - "123456789"
        group_user_allowed_commands:
          - status
```

**Поведінка:**

- Користувач, зазначений у `allow_admin_from` для області (DM або група), може виконувати **будь‑яку** зареєстровану Slash‑команду — вбудовані команди ТА команди, зареєстровані плагінами — через живий реєстр.
- Користувач у `allow_from`, але **не** у `allow_admin_from`, може виконувати лише команди, зазначені в `user_allowed_commands`, плюс завжди дозволені базові: `/help` і `/whoami`.
- Звичайний чат (повідомлення без Slash) не змінюється. Користувачі без прав адміністратора можуть продовжувати спілкуватися з агентом у звичайному режимі, вони лише не можуть запускати довільні команди.
- **Зворотна сумісність:** якщо `allow_admin_from` не встановлено для області, фільтрація Slash‑команд вимикається для цієї області. Існуючі інсталяції продовжують працювати без змін.
- Статус адміністратора в DM не означає статус адміністратора в групі. Кожна область має власний список адміністраторів.
- Якщо встановлено лише `group_allow_admin_from`, область DM залишається в режимі без обмежень (зворотна сумісність).

Використай `/whoami`, щоб побачити активну область, свій рівень (admin / user / unrestricted) та які Slash‑команди ти можеш виконувати.
## Інтерактивний вибір моделі

Коли ти надсилаєш `/model` без аргументів у чаті Telegram, Hermes показує інтерактивну inline‑клавіатуру для перемикання моделей:

1. **Вибір провайдера** — кнопки, що відображають кожного доступного провайдера з кількістю моделей (наприклад, “OpenAI (15)”, “✓ Anthropic (12)” для поточного провайдера).
2. **Вибір моделі** — пагінований список моделей з навігацією **Prev**/**Next**, кнопкою **Back** для повернення до провайдерів та **Cancel**.

Поточна модель і провайдер відображаються вгорі. Уся навігація здійснюється шляхом редагування того самого повідомлення на місці (без зайвого шуму в чаті).

:::tip
Якщо ти знаєш точну назву моделі, введи `/model <name>` безпосередньо, щоб пропустити вибір. Також можна ввести `/model <name> --global`, щоб зберегти зміну між сесіями.
:::
## DNS-over-HTTPS Fallback IPs

У деяких обмежених мережах `api.telegram.org` може резольвитися в IP‑адресу, недоступну для з’єднання. Адаптер Telegram включає механізм **запасний IP**, який прозоро повторює спроби підключення до альтернативних IP‑адрес, зберігаючи правильне TLS‑ім’я хоста та SNI.

### Як це працює

1. Якщо встановлено `TELEGRAM_FALLBACK_IPS`, ці IP‑адреси використовуються безпосередньо.
2. Інакше адаптер автоматично запитує **Google DNS** та **Cloudflare DNS** через DNS-over-HTTPS (DoH), щоб знайти альтернативні IP‑адреси для `api.telegram.org`.
3. IP‑адреси, повернені DoH і відрізняються від результату системного DNS, використовуються як запасні.
4. Якщо DoH також заблоковано, використовується жорстко закодована seed‑IP (`149.154.167.220`) як останній варіант.
5. Після успішного підключення до запасного IP він стає «липким» — подальші запити використовують його без повторної спроби первинного шляху.

### Налаштування

```bash
# Explicit fallback IPs (comma-separated)
TELEGRAM_FALLBACK_IPS=149.154.167.220,149.154.167.221
```

Або у `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      fallback_ips:
        - "149.154.167.220"
```

:::tip
Зазвичай не потрібно налаштовувати це вручну. Автовиявлення через DoH обробляє більшість сценаріїв у обмежених мережах. Змінна середовища `TELEGRAM_FALLBACK_IPS` потрібна лише якщо DoH також заблоковано у твоїй мережі.
:::
## Підтримка проксі

Якщо твоя мережа вимагає HTTP‑проксі для доступу до інтернету (часто у корпоративних середовищах), адаптер Telegram автоматично читає стандартні змінні середовища проксі та маршрутизує всі з’єднання через проксі.

### Підтримувані змінні

Адаптер перевіряє ці змінні середовища у зазначеному порядку, використовуючи першу встановлену:

1. `HTTPS_PROXY`
2. `HTTP_PROXY`
3. `ALL_PROXY`
4. `https_proxy` / `http_proxy` / `all_proxy` (нижні варіанти)

### Конфігурація

Встанови проксі у своєму середовищі перед запуском шлюзу:

```bash
export HTTPS_PROXY=http://proxy.example.com:8080
hermes gateway
```

Або додай його до `~/.hermes/.env`:

```bash
HTTPS_PROXY=http://proxy.example.com:8080
```

Проксі застосовується як до основного транспорту, так і до всіх запасних (варіант) IP‑транспортів. Додаткова конфігурація Hermes не потрібна — якщо змінна середовища встановлена, вона використовується автоматично.

:::note
Це охоплює користувацький запасний (варіант) транспортний шар, який Hermes використовує для з’єднань Telegram. Стандартний клієнт `httpx`, який використовується в інших місцях, вже нативно підтримує змінні проксі.
:::
## Реакції на повідомлення

Бот може додавати emoji‑реакції до повідомлень як візуальний зворотний зв’язок процесу обробки:

- 👀 коли бот починає обробляти твоє повідомлення
- ✅ коли відповідь успішно доставлена
- ❌ якщо під час обробки сталася помилка

Реакції **вимкнено за замовчуванням**. Увімкни їх у `config.yaml`:

```yaml
telegram:
  reactions: true
```

Або через змінну середовища:

```bash
TELEGRAM_REACTIONS=true
```

:::note
На відміну від Discord (де реакції додаються), Telegram Bot API замінює всі реакції бота одним викликом. Перехід від 👀 до ✅/❌ відбувається атомарно — ти не побачиш їх одночасно.
:::

:::tip
Якщо у бота немає дозволу додавати реакції в групі, виклики реакцій завершуються без повідомлення про помилку, і обробка повідомлення продовжується звичайно.
:::
## Підказки для окремих каналів

Призначай тимчасові системні підказки конкретним групам Telegram або темам форуму. Підказка впроваджується під час виконання на кожному кроці — ніколи не зберігається в історії транскрипту — тому зміни набувають чинності одразу.

```yaml
telegram:
  channel_prompts:
    "-1001234567890": |
      You are a research assistant. Focus on academic sources,
      citations, and concise synthesis.
    "42":  |
      This topic is for creative writing feedback. Be warm and
      constructive.
```

Ключами є ідентифікатори чатів (групи/супергрупи) або ідентифікатори тем форуму. Для форумних груп підказки на рівні теми переважають над підказками на рівні групи:

- Повідомлення в темі `42` у групі `-1001234567890` → використовується підказка теми `42`
- Повідомлення в темі `99` (без явного запису) → використовується підказка групи `-1001234567890`
- Повідомлення в групі без запису → підказка каналу не застосовується

Числові ключі YAML автоматично нормалізуються до рядків.
## Устранення проблем

| Проблема | Рішення |
|---------|----------|
| Bot не відповідає взагалі | Перевір, чи `TELEGRAM_BOT_TOKEN` правильний. Переглянь логи `hermes gateway` на предмет помилок. |
| Bot відповідає «unauthorized» | Твій ID користувача відсутній у `TELEGRAM_ALLOWED_USERS`. Перевір ще раз за допомогою @userinfobot. |
| Bot ігнорує повідомлення групи | Ймовірно, ввімкнено режим приватності. Вимкни його (Крок 3) або зроби бота адміністратором групи. **Не забудь видалити та знову додати бота після зміни налаштувань приватності.** |
| Голосові повідомлення не транскрибуються | Переконайся, що STT доступний: встанови `faster-whisper` для локальної транскрипції, або задай `GROQ_API_KEY` / `VOICE_TOOLS_OPENAI_KEY` у `~/.hermes/.env`. |
| Голосові відповіді – це файли, а не «бульбашки» | Встанови `ffmpeg` (потрібен для конвертації Edge TTS Opus). |
| Токен бота відкликано/недійсний | Згенеруй новий токен через `/revoke`, потім `/newbot` або `/token` у BotFather. Онови файл `.env`. |
| Webhook не отримує оновлення | Перевір, чи `TELEGRAM_WEBHOOK_URL` публічно доступний (перевір за допомогою `curl`). Переконайся, що твоя платформа/зворотний проксі маршрутизує вхідний HTTPS‑трафік з порту URL до локального порту, налаштованого в `TELEGRAM_WEBHOOK_PORT` (вони не обов’язково однакові). Переконайся, що SSL/TLS активний — Telegram надсилає лише на HTTPS‑URL. Перевір правила брандмауера. |

## Interactive Prompts (clarify)

Коли агент викликає інструмент `clarify` — щоб запитати, який підхід ти віддаєш перевагу, отримати зворотний зв’язок після виконання завдання або перевірити рішення перед нетривіальним вибором — Telegram відображає питання у вигляді **inline‑кнопок**:

> ❓ Який фреймворк використати для дашборду?
>

Натисни кнопку, щоб відповісти, або **Інше**, щоб ввести вільну відповідь (наступне повідомлення, яке ти надішлеш, стане відповіддю). Для відкритих викликів `clarify` (без попередньо заданих варіантів) кнопки не показуються, і просто фіксується твоє наступне повідомлення.

Налаштуй тайм‑аут відповіді через `agent.clarify_timeout` у `~/.hermes/config.yaml` (за замовчуванням `600` секунд). Якщо ти не відповіси протягом тайм‑ауту, агент розблокується за допомогою sentinel‑повідомлення і адаптується, а не зависне.
## Обсяг push‑повідомлень

Telegram надсилає push‑повідомлення на кожне повідомлення, яке відправляє бот. Для довгих ходів агента, які генерують бульбашки прогресу інструменту, потокові оновлення та зворотні виклики статусу, це швидко стає надмірним. Адаптер Telegram має два режими сповіщень:

| Mode | Behavior |
|------|----------|
| `important` (default) | Сповіщає лише **остаточні відповіді**, **запити на схвалення** та **підтвердження slash‑команд**. Прогрес інструменту, потокові фрагменти та статусні повідомлення надсилаються з `disable_notification=true`. |
| `all` | Кожне вихідне повідомлення викликає push‑повідомлення. Спадковий режим; використай, якщо дійсно хочеш отримувати сповіщення про кожен виклик інструменту. |

Налаштуй у `~/.hermes/config.yaml`:

```yaml
display:
  platforms:
    telegram:
      notifications: important   # or "all"
```

Перевизначення через змінну середовища (зручно для швидкого A/B тестування):

```bash
HERMES_TELEGRAM_NOTIFICATIONS=all
```

Невідомі значення записуються у журнал як попередження і відкат до `important`.
## Повідомлення про стан, що редагуються на місці

Telegram‑адаптер маршрутизує повторювані зворотні виклики стану агента (наприклад, «Compressing context…», «Calling tool…») через `send_or_update_status()`, який зберігає кеш `{(chat_id, status_key) → message_id}` і **редагує існуюче повідомлення** при наступних викликах замість додавання нового кожного разу. Різні значення `status_key` отримують власні повідомлення; різні чати ніколи не конфліктують. Якщо редагування не вдається (наприклад, користувач видалив повідомлення або воно старше, ніж дозволяє Telegram для редагувань), запис у кеші видаляється, і наступний виклик публікує нове повідомлення та знову кешує його ID. Ніяких налаштувань не потрібно — це поведінка за замовчуванням у Telegram. Інші адаптери, які не реалізують `send_or_update_status`, переходять до простого `send()` без змін.
## Закріпити вхідне повідомлення користувача під час ходу агента

Коли користувач надсилає повідомлення, яке ініціює хід агента, Telegram‑адаптер **закріплює** це вхідне повідомлення на час ходу і **знімає закріплення**, коли відповідь завершена — легкий візуальний індикатор того, що бот активно працює над повідомленням, а не ігнорує його. Прикріплення використовує `disable_notification=true`, щоб уникнути зайвих сповіщень. Налаштування не потрібні.
## Безпека

:::warning
Завжди встановлюй `TELEGRAM_ALLOWED_USERS`, щоб обмежити, хто може взаємодіяти з твоїм ботом. Без цього gateway за замовчуванням відхиляє всіх користувачів як запобіжний захід.
:::

Ніколи не публікуй токен свого бота. Якщо він був скомпрометований, негайно відклич його за допомогою команди `/revoke` у BotFather.

Для отримання додаткової інформації переглянь [документацію щодо безпеки](/user-guide/security). Також можна скористатися [DM pairing](/user-guide/messaging#dm-pairing-alternative-to-allowlists) для більш динамічного підходу до авторизації користувачів.
