# Налаштування Feishu / Lark

Hermes Agent інтегрується з Feishu та Lark як повнофункціональний бот. Після підключення ти можеш спілкуватися з агентом у приватних повідомленнях або групових чатах, отримувати результати cron‑завдань у домашньому чаті та надсилати текст, зображення, аудіо і файли‑вкладення через звичайний потік шлюзу.

Інтеграція підтримує обидва режими підключення:

- `websocket` — рекомендовано; Hermes відкриває вихідне з’єднання, і тобі не потрібна публічна кінцева точка webhook
- `webhook` — корисно, коли ти хочеш, щоб Feishu/Lark надсилали події у твій шлюз через HTTP
## Як поводиться Hermes

| Контекст | Поведінка |
|----------|----------|
| Прямі повідомлення | Hermes відповідає на кожне повідомлення. |
| Групові чати | Hermes відповідає лише коли бот згаданий у чаті. |
| Спільні групові чати | За замовчуванням історія сесії ізольована для кожного користувача в спільному чаті. |

Ця поведінка спільного чату контролюється `config.yaml`:

```yaml
group_sessions_per_user: true
```

Встанови `false` лише якщо ти явно хочеш одну спільну розмову в чаті.
## Крок 1: Створити Feishu / Lark додаток

### Рекомендовано: Scan-to-Create (одна команда)

```bash
hermes gateway setup
```

Вибери **Feishu / Lark** і відскануй QR‑код у своєму мобільному додатку Feishu або Lark. Hermes автоматично створить бот‑додаток з потрібними дозволами та збереже облікові дані.

### Альтернатива: Ручне налаштування

Якщо Scan-to-Create недоступний, майстер переходить до ручного вводу:

1. Відкрий консоль розробника Feishu або Lark:
   - Feishu: [https://open.feishu.cn/](https://open.feishu.cn/)
   - Lark: [https://open.larksuite.com/](https://open.larksuite.com/)
2. Створи новий додаток.
3. У **Credentials & Basic Info** скопіюй **App ID** та **App Secret**.
4. Увімкни можливість **Bot** для додатку.
5. Запусти `hermes gateway setup`, вибери **Feishu / Lark** і введи облікові дані, коли буде запропоновано.

:::warning
Тримай App Secret у таємниці. Будь‑хто, хто його має, може видавати себе за твій додаток.
:::

### Налаштування дозволів

У консолі розробника Feishu перейдіть до **Permission Management** і додайте наступні області. Ви можете пакетно імпортувати їх на сторінці дозволів.

**Обов’язкові дозволи:**

| Scope | Призначення |
|-------|-------------|
| `im:message` | Отримання та читання повідомлень |
| `im:message:send_as_bot` | Надсилання повідомлень від імені бота |
| `im:resource` | Доступ до зображень, файлів та аудіо, надісланих користувачами |
| `im:chat` | Доступ до метаданих чатів/груп |
| `im:chat:readonly` | Читання списку чатів та їх учасників |

**Рекомендовані дозволи (для повної функціональності):**

| Scope | Призначення |
|-------|-------------|
| `im:message.reactions:readonly` | Отримання подій реакцій‑емодзі |
| `admin:app.info:readonly` | Автоматичне визначення ідентичності бота для @згадування |
| `contact:user.id:readonly` | Визначення ID користувачів для порівняння зі списком дозволених |

### Налаштування подій

У **Events and Callbacks**:

1. Встанови режим підключення **Long Connection (WebSocket)** (рекомендовано) або налаштуй URL веб‑хука.
2. У розділі **Event Configuration** підпишись на:
   - `im.message.receive_v1` — необхідно для отримання повідомлень

### Публікація додатку

Після налаштування дозволів і подій перейдіть до **Version Management** і опублікуйте нову версію додатку. Дозволи набудуть чинності лише після публікації та затвердження версії (для корпоративних додатків може знадобитися схвалення адміністратора).
## Крок 2: Вибір режиму підключення

### Рекомендовано: режим WebSocket

Використовуй режим WebSocket, коли Hermes працює на твоєму ноутбуці, робочій станції або приватному сервері. Публічний URL не потрібен. Офіційний Lark SDK відкриває та підтримує постійне вихідне WebSocket‑з’єднання з автоматичним перепідключенням.

```bash
FEISHU_CONNECTION_MODE=websocket
```

**Вимоги:** Пакет Python `websockets` має бути встановлений. SDK обробляє життєвий цикл підключення, heartbeat та автоматичне перепідключення внутрішньо.

**Як це працює:** Адаптер запускає WebSocket‑клієнт Lark SDK у фоні в окремому потоці‑виконавці. Вхідні події (повідомлення, реакції, дії карток) розсилаються у головний цикл asyncio. При роз’єднанні SDK спробує автоматично перепідключитися.

### Додатково: режим Webhook

Використовуй режим webhook лише тоді, коли Hermes вже працює за доступною HTTP‑точкою.

```bash
FEISHU_CONNECTION_MODE=webhook
```

У режимі webhook Hermes запускає HTTP‑сервер (через `aiohttp`) і надає endpoint Feishu за адресою:

```text
/feishu/webhook
```

**Вимоги:** Пакет Python `aiohttp` має бути встановлений.

Ти можеш налаштувати адресу прив’язки та шлях сервера webhook:

```bash
FEISHU_WEBHOOK_HOST=127.0.0.1   # default: 127.0.0.1
FEISHU_WEBHOOK_PORT=8765         # default: 8765
FEISHU_WEBHOOK_PATH=/feishu/webhook  # default: /feishu/webhook
```

Коли Feishu надсилає запит на верифікацію URL (`type: url_verification`), webhook відповідає автоматично, щоб ти міг завершити налаштування підписки у консолі розробника Feishu. Відповідь на виклик контролюється змінною `FEISHU_VERIFICATION_TOKEN`, якщо вона встановлена — запити з відсутнім або невідповідним токеном відхиляються, тому неавтентифікований віддалений клієнт не зможе довести контроль над кінцевою точкою, підсилаючи дані, контрольовані атакувальником.
## Крок 3: Налаштування Hermes

### Варіант A: Інтерактивне налаштування

```bash
hermes gateway setup
```

Вибери **Feishu / Lark** і заповни запити.

### Варіант B: Ручна конфігурація

Додай наступне до `~/.hermes/.env`:

```bash
FEISHU_APP_ID=cli_xxx
FEISHU_APP_SECRET=secret_xxx
FEISHU_DOMAIN=feishu
FEISHU_CONNECTION_MODE=websocket

# Optional but strongly recommended
FEISHU_ALLOWED_USERS=ou_xxx,ou_yyy
FEISHU_HOME_CHANNEL=oc_xxx
```

`FEISHU_DOMAIN` приймає:

- `feishu` для Feishu China
- `lark` для Lark (міжнародний)
## Крок 4: Запустити gateway

```bash
hermes gateway
```

Потім надішли боту повідомлення з Feishu/Lark, щоб підтвердити, що з’єднання активне.

Ти також можеш попередньо налаштувати його:

## Security

### User Allowlist

Для продакшн‑використання встанови білий список (`allowlist`) з Feishu Open ID:

```bash
FEISHU_ALLOWED_USERS=ou_xxx,ou_yyy
```

Якщо залишити білий список порожнім, будь‑хто, хто може дістатися до бота, зможе його використовувати. У групових чатах білий список перевіряється проти `open_id` відправника перед обробкою повідомлення.

### Webhook Encryption Key

При запуску в режимі webhook встанови ключ шифрування, щоб увімкнути перевірку підпису вхідних webhook‑payload:

```bash
FEISHU_ENCRYPT_KEY=your-encrypt-key
```

Цей ключ знаходиться у розділі **Event Subscriptions** конфігурації твого Feishu‑додатку. Після встановлення адаптер перевіряє кожен запит webhook, використовуючи алгоритм підпису:

```
SHA256(timestamp + nonce + encrypt_key + body)
```

Обчислений хеш порівнюється зі значенням заголовка `x-lark-signature` за допомогою timing‑safe порівняння. Запити з недійсними або відсутніми підписами відхиляються з HTTP 401.

:::tip
У режимі WebSocket перевірка підпису здійснюється самим SDK, тому `FEISHU_ENCRYPT_KEY` є необов’язковим. У режимі webhook його настійно рекомендують використовувати у продакшн‑середовищі.
:::

### Verification Token

Додатковий рівень автентифікації, який перевіряє поле `token` у webhook‑payload:

```bash
FEISHU_VERIFICATION_TOKEN=your-verification-token
```

Цей токен також знаходиться у розділі **Event Subscriptions** Feishu‑додатку. Після встановлення кожен вхідний webhook‑payload має містити відповідний `token` у своєму об’єкті `header`. Токени, що не збігаються, відхиляються з HTTP 401.

Обидва `FEISHU_ENCRYPT_KEY` і `FEISHU_VERIFICATION_TOKEN` можна використовувати разом для захисту в глибині.
## Політика групових повідомлень

Змінна середовища `FEISHU_GROUP_POLICY` керує тим, чи і як Hermes відповідає у групових чатах:

```bash
FEISHU_GROUP_POLICY=allowlist   # default
```

| Value | Behavior |
|-------|----------|
| `open` | Hermes відповідає на @згадки від будь‑якого користувача в будь‑якій групі. |
| `allowlist` | Hermes відповідає лише на @згадки від користувачів, зазначених у `FEISHU_ALLOWED_USERS`. |
| `disabled` | Hermes ігнорує всі групові повідомлення повністю. |

У всіх режимах бот має бути явно @згаданий (або @all) у групі, перш ніж повідомлення буде оброблено. Прямі повідомлення завжди обходять цей шлюз.

Встанови `FEISHU_REQUIRE_MENTION=false`, щоб дозволити Hermes читати весь груповий трафік без вимоги @згадки:

```bash
FEISHU_REQUIRE_MENTION=false
```

Для керування окремими чатами встанови `require_mention` у записі `group_rules` — дивись [Per-Group Access Control](#per-group-access-control) нижче.

### Ідентифікатор бота

Hermes автоматично визначає `open_id` бота та його відображуване ім’я під час запуску. Ти лише потрібно задати їх вручну, коли автоматичне визначення не може отримати доступ до Feishu API, або коли твій застосунок використовує tenant‑scoped ідентифікатори користувачів:

```bash
FEISHU_BOT_OPEN_ID=ou_xxx     # only when auto-detection fails
FEISHU_BOT_USER_ID=xxx        # required if your app uses sender_id_type=user_id
FEISHU_BOT_NAME=MyBot         # only when auto-detection fails
```
## Повідомлення між ботами

За замовчуванням Hermes ігнорує повідомлення, надіслані іншими ботами. Увімкни повідомлення між ботами, коли потрібно, щоб Hermes брав участь у A2A‑оркестрації або отримував сповіщення від інших ботів у тій самій групі.

```bash
FEISHU_ALLOW_BOTS=mentions   # default: none
```

| Value | Behavior |
|-------|----------|
| `none` | Ігнорувати всі повідомлення від інших ботів (за замовчуванням). |
| `mentions` | Приймати лише коли інший бот @mentions Hermes. |
| `all` | Приймати кожне повідомлення від іншого бота. |

Також можна налаштувати через `feishu.allow_bots` у `config.yaml` (змінна оточення має пріоритет, якщо обидва задані).

Боти‑партнери не потрібно додавати до `FEISHU_ALLOWED_USERS` — цей білий список застосовується лише до людей‑відправників.

Надай область `application:bot.basic_info:read`, щоб відображати імена ботів‑партнерів; без неї боти‑партнери все одно правильно маршрутизуються, але будуть показані їхні `open_id`.
## Інтерактивні дії карток

Коли користувачі натискають кнопки або взаємодіють з інтерактивними картками, надісланими ботом, адаптер маршрутизує їх як синтетичні події команди `/card`:

- Натискання кнопки перетворюється на: `/card button {"key": "value", ...}`
- У навантаженні `value` дії з визначення картки передається JSON.
- Дії карток дедуплікуються у вікні 15 хвилин, щоб запобігти подвійній обробці.

Підказки оновлення, керовані шлюзом, використовують нативну Feishu‑картку `Yes` / `No` замість повернення простих текстових відповідей. Коли `hermes update --gateway` потребує підтвердження, адаптер записує вибрану відповідь у файл `.update_response` Hermes і замінює картку в рядку на розв’язаний стан.

Події дій карток надсилаються з `MessageType.COMMAND`, тому вони проходять через звичайний конвеєр обробки команд.

Саме так працює **затвердження команд** — коли агент потребує виконати небезпечну команду, він надсилає інтерактивну картку з кнопками **Allow Once** / **Session** / **Always** / **Deny**. Користувач натискає кнопку, і зворотний виклик дії картки передає рішення про затвердження агенту.

### Необхідна конфігурація Feishu App

Інтерактивні картки вимагають **трьох** кроків налаштування в Feishu Developer Console. Відсутність будь‑якого з них викликає помилку **200340**, коли користувачі натискають кнопки картки.

1. **Підписка на подію дії картки:**
   У **Event Subscriptions** додайте `card.action.trigger` до підписаних подій.

2. **Увімкнення можливості Interactive Card:**
   У **App Features > Bot** переконайтеся, що перемикач **Interactive Card** увімкнено. Це повідомляє Feishu, що ваш додаток може отримувати зворотні виклики дій картки.

3. **Налаштування Card Request URL (лише режим webhook):**
   У **App Features > Bot > Message Card Request URL** встановіть URL, який збігається з кінцевою точкою вашого webhook‑події (наприклад `https://your-server:8765/feishu/webhook`). У режимі WebSocket це обробляється автоматично SDK.

:::warning
Без усіх трьох кроків Feishu успішно *надсилатиме* інтерактивні картки (надсилання вимагає лише дозволу `im:message:send`), але натискання будь‑якої кнопки поверне помилку 200340. Картка здається працюючою — помилка проявляється лише під час взаємодії користувача з нею.
:::
## Document Comment Intelligent Reply

Окрім чату, адаптер може також відповідати на `@`‑згадки у **Feishu/Lark документах**. Коли користувач коментує документ (вибір локального тексту або коментар до всього документа) і @‑згадує бота, Hermes читає документ разом із навколишньою гілкою коментарів і публікує відповідь LLM безпосередньо в цій гілці.

Працює завдяки події `drive.notice.comment_add_v1`; обробник:

- Паралельно отримує вміст документа та хронологію коментарів (20 повідомлень для гілок всього документа, 12 для гілок локального виділення).
- Запускає агента з наборами інструментів `feishu_doc` + `feishu_drive`, обмежений однією сесією коментаря.
- Розбиває відповіді на частини по 4000 символів і публікує їх як відповіді в гілці.
- Кешує сесії на документ протягом 1 години з обмеженням у 50 повідомлень, щоб подальші коментарі в тому ж документі зберігали контекст.

### 3‑рівневий контроль доступу

Відповіді на коментарі в документі доступні **лише за явним дозволом** — немає режиму неявного дозволу всім. Дозволи визначаються у такому порядку (перший збіг виграє, за полем):

1. **Точний документ** — правило, прив’язане до конкретного токену документа.
2. **Шаблон** — правило, що відповідає шаблону документів.
3. **Верхній рівень** — правило за замовчуванням для робочого простору.

Для кожного правила доступні два типи політик:

- **`allowlist`** — статичний список користувачів/орендарів.
- **`pairing`** — статичний список ∪ сховище, схвалене під час виконання. Корисно для розгортань, коли модератори можуть надавати доступ у реальному часі.

Правила зберігаються у `~/.hermes/feishu_comment_rules.json` (надання прав у `~/.hermes/feishu_comment_pairing.json`) з кешуванням за часом зміни файлу — зміни набувають чинності під час наступної події коментаря без перезапуску шлюзу.

CLI:

```bash
# Inspect current rules and pairing state
python -m gateway.platforms.feishu_comment_rules status

# Simulate an access check for a specific doc + user
python -m gateway.platforms.feishu_comment_rules check <fileType:fileToken> <user_open_id>

# Manage pairing grants at runtime
python -m gateway.platforms.feishu_comment_rules pairing list
python -m gateway.platforms.feishu_comment_rules pairing add <user_open_id>
python -m gateway.platforms.feishu_comment_rules pairing remove <user_open_id>
```

### Необхідна конфігурація Feishu App

Окрім вже наданих дозволів чату/картки, додайте подію коментаря в Drive:

- Підпишіться на `drive.notice.comment_add_v1` у **Event Subscriptions**.
- Наділіть області `docs:doc:readonly` та `drive:drive:readonly`, щоб обробник міг читати вміст документа.
## Підтримка медіа

### Вхідні (отримання)

Адаптер отримує та кешує наступні типи медіа від користувачів:

| Тип | Розширення | Як обробляється |
|------|-----------|-------------------|
| **Зображення** | .jpg, .jpeg, .png, .gif, .webp, .bmp | Завантажуються через Feishu API та кешуються локально |
| **Аудіо** | .ogg, .mp3, .wav, .m4a, .aac, .flac, .opus, .webm | Завантажуються та кешуються; маленькі текстові файли автоматично витягуються |
| **Відео** | .mp4, .mov, .avi, .mkv, .webm, .m4v, .3gp | Завантажуються та кешуються як документи |
| **Файли** | .pdf, .doc, .docx, .xls, .xlsx, .ppt, .pptx та інші | Завантажуються та кешуються як документи |

Медіа з повідомлень у форматі **rich‑text** (post), включаючи вбудовані зображення та вкладені файли, також витягуються та кешуються.

Для невеликих текстових документів (.txt, .md) вміст файлу автоматично вставляється у текст повідомлення, щоб агент міг прочитати його без використання інструментів.

### Вихідні (надсилання)

| Метод | Що надсилає |
|--------|--------------|
| `send` | Текст або повідомлення у форматі **rich post** (автоматично визначається за вмістом markdown) |
| `send_image` / `send_image_file` | Завантажує зображення у Feishu, потім надсилає його як нативний **bubble‑зображення** (з необов’язковою підписом) |
| `send_document` | Завантажує файл у Feishu API, потім надсилає його як вкладення |
| `send_voice` | Завантажує аудіофайл як вкладення файлу у Feishu |
| `send_video` | Завантажує відео та надсилає його як нативне медіаповідомлення |
| `send_animation` | GIF‑файли конвертуються у вкладення (Feishu не підтримує нативний **bubble‑GIF**) |

Маршрутизація завантаження файлів виконується автоматично за розширенням:

- `.ogg`, `.opus` → завантажується як аудіо `opus`
- `.mp4`, `.mov`, `.avi`, `.m4v` → завантажується як медіа `mp4`
- `.pdf`, `.doc(x)`, `.xls(x)`, `.ppt(x)` → завантажується зі своїм типом документа
- Все інше → завантажується як загальний **stream‑файл**
## Відображення Markdown та запасний (фолбек) режим посту

Коли вихідний текст містить форматування markdown (заголовки, **жирний**, списки, блоки коду, посилання тощо), адаптер автоматично надсилає його як **post**‑повідомлення Feishu з вбудованим тегом `md` замість простого тексту. Це забезпечує багатоформатне відображення у клієнті Feishu.

Якщо API Feishu відхиляє payload посту (наприклад, через непідтримувані конструкції markdown), адаптер автоматично переходить у запасний (фолбек) режим і надсилає повідомлення як простий текст без markdown. Такий двоетапний запасний (фолбек) режим гарантує, що повідомлення завжди доставляються.

Повідомлення простим текстом (markdown не виявлено) надсилаються як тип повідомлення `text`.
## Статус реакцій обробки

Поки агент працює, бот показує реакцію `Typing` на твоє повідомлення. Вона очищується, коли надходить відповідь, або замінюється на `CrossMark`, якщо обробка завершилась помилкою.

Встанови `FEISHU_REACTIONS=false`, щоб вимкнути її.
## Захист від сплесків і пакетування

Адаптер включає дебаунсинг для швидких сплесків повідомлень, щоб не перевантажити агента:

### Пакетування тексту

Коли користувач надсилає кілька текстових повідомлень поспіль, вони об’єднуються в одну подію перед відправкою:

| Setting | Env Var | Default |
|---------|---------|---------|
| Період тиші | `HERMES_FEISHU_TEXT_BATCH_DELAY_SECONDS` | 0.6s |
| Максимальна кількість повідомлень у пакеті | `HERMES_FEISHU_TEXT_BATCH_MAX_MESSAGES` | 8 |
| Максимальна кількість символів у пакеті | `HERMES_FEISHU_TEXT_BATCH_MAX_CHARS` | 4000 |

### Пакетування медіа

Кілька медіа‑вкладень, надісланих поспіль (наприклад, перетягування кількох зображень), об’єднуються в одну подію:

| Setting | Env Var | Default |
|---------|---------|---------|
| Період тиші | `HERMES_FEISHU_MEDIA_BATCH_DELAY_SECONDS` | 0.8s |

### Серіалізація в межах чату

Повідомлення в одному чаті обробляються послідовно (по одному), щоб зберегти зв’язність розмови. Кожен чат має власне блокування, тому повідомлення в різних чатах обробляються одночасно.
## Обмеження швидкості (режим Webhook)

У режимі webhook адаптер застосовує обмеження швидкості за IP‑адресою, щоб захистити від зловживань:

- **Вікно:** 60‑секундне ковзне вікно
- **Ліміт:** 120 запитів за вікно для трійки (app_id, path, IP)
- **Максимальна кількість відстежуваних ключів:** до 4096 унікальних ключів (запобігає необмеженому зростанню пам’яті)

Запити, що перевищують ліміт, отримують HTTP 429 (Too Many Requests).

### Відстеження аномалій Webhook

Адаптер відстежує послідовні помилкові відповіді за IP‑адресою. Після 25 послідовних помилок з одного IP протягом 6‑годинного вікна записується попередження. Це допомагає виявляти неправильно налаштованих клієнтів або спроби сканування.

Додаткові захисти webhook:
- **Ліміт розміру тіла:** максимум 1 MB
- **Тайм‑аут читання тіла:** 30 секунд
- **Примусове використання Content‑Type:** приймається лише `application/json`
## Налаштування WebSocket

При використанні режиму `websocket` ти можеш налаштувати поведінку перепідключення та ping‑запитів:

```yaml
platforms:
  feishu:
    extra:
      ws_reconnect_interval: 120   # Seconds between reconnect attempts (default: 120)
      ws_ping_interval: 30         # Seconds between WebSocket pings (optional; SDK default if unset)
```

| Параметр | Ключ конфігурації | За замовчуванням | Опис |
|---------|-------------------|-------------------|------|
| Reconnect interval | `ws_reconnect_interval` | 120s | Як довго чекати між спробами перепідключення |
| Ping interval | `ws_ping_interval` | _(SDK default)_ | Частота keepalive‑ping‑запитів WebSocket |
## Керування доступом за групами

Окрім глобального `FEISHU_GROUP_POLICY`, ти можеш встановлювати деталізовані правила для кожного групового чату за допомогою `group_rules` у `config.yaml`:

```yaml
platforms:
  feishu:
    extra:
      default_group_policy: "open"     # Default for groups not in group_rules
      admins:                          # Users who can manage bot settings
        - "ou_admin_open_id"
      group_rules:
        "oc_group_chat_id_1":
          policy: "allowlist"          # open | allowlist | blacklist | admin_only | disabled
          allowlist:
            - "ou_user_open_id_1"
            - "ou_user_open_id_2"
        "oc_group_chat_id_2":
          policy: "admin_only"
        "oc_group_chat_id_3":
          policy: "blacklist"
          blacklist:
            - "ou_blocked_user"
        "oc_free_chat":
          policy: "open"
          require_mention: false       # overrides FEISHU_REQUIRE_MENTION for this chat
```

| Policy | Description |
|--------|-------------|
| `open` | Будь‑хто в групі може використовувати бота |
| `allowlist` | Тільки користувачі, що входять у `allowlist` групи, можуть використовувати бота |
| `blacklist` | Усі, крім користувачів у `blacklist` групи, можуть використовувати бота |
| `admin_only` | Тільки користувачі зі глобального списку `admins` можуть використовувати бота в цій групі |
| `disabled` | Бот ігнорує всі повідомлення в цій групі |

Встанови `require_mention: false` у записі `group_rules`, щоб вимкнути вимогу @‑згадки для конкретного чату. Якщо його не вказано, чат успадковує глобальне значення `FEISHU_REQUIRE_MENTION`.

Групи, які не вказані в `group_rules`, використовують `default_group_policy` (за замовчуванням — значення `FEISHU_GROUP_POLICY`).

## Устранення проблем

| Проблема | Виправлення |
|----------|------------|
| `lark-oapi not installed` | Встанови SDK: `pip install lark-oapi` |
| `websockets not installed; websocket mode unavailable` | Встанови websockets: `pip install websockets` |
| `aiohttp not installed; webhook mode unavailable` | Встанови aiohttp: `pip install aiohttp` |
| `FEISHU_APP_ID or FEISHU_APP_SECRET not set` | Встанови обидві змінні середовища або налаштуй їх через `hermes gateway setup` |
| `Another local Hermes gateway is already using this Feishu app_id` | Одна інстанція Hermes може використовувати один і той самий app_id одночасно. Спочатку зупини інший шлюз. |
| Bot doesn't respond in groups | Переконайся, що бот згаданий (`@`), перевір `FEISHU_GROUP_POLICY` і впевнись, що відправник входить до `FEISHU_ALLOWED_USERS`, якщо політика `allowlist` |
| `Webhook rejected: invalid verification token` | Переконайся, що `FEISHU_VERIFICATION_TOKEN` збігається з токеном у конфігурації **Event Subscriptions** твого Feishu‑додатку |
| `Webhook rejected: invalid signature` | Переконайся, що `FEISHU_ENCRYPT_KEY` збігається з ключем шифрування у конфігурації Feishu‑додатку |
| Post messages show as plain text | Feishu API відхилив payload повідомлення; це нормальна запасна (fallback) поведінка. Переглянь логи для деталей. |
| Images/files not received by bot | Надай дозволи `im:message` та `im:resource` у Feishu‑додатку |
| Bot identity not auto-detected | Зазвичай це тимчасова проблема мережі при доступі до endpoint‑у інформації про бота Feishu. Встанови `FEISHU_BOT_OPEN_ID` і `FEISHU_BOT_NAME` вручну як обхідний шлях. |
| Peer bot messages still ignored after enabling `FEISHU_ALLOW_BOTS` | Hermes ще не може ідентифікувати себе — встанови `FEISHU_BOT_OPEN_ID` (і `FEISHU_BOT_USER_ID`, якщо твій додаток використовує `sender_id_type=user_id`). |
| Peer bots show as `ou_xxxxxx` instead of by name | Надай дозвіл `application:bot.basic_info:read`. |
| Error 200340 when clicking approval buttons | Увімкни можливість **Interactive Card** і налаштуй **Card Request URL** у Feishu Developer Console. Дивись [Required Feishu App Configuration](#required-feishu-app-configuration) вище. |
| `Webhook rate limit exceeded` | Понад 120 запитів/хвилину з одного IP. Зазвичай це помилкова конфігурація або цикл. |
