# Налаштування Matrix

Hermes Agent інтегрується з Matrix, відкритим федеративним протоколом обміну повідомленнями. Matrix дозволяє запускати власний homeserver або використовувати публічний, наприклад matrix.org — у будь‑якому випадку ти зберігаєш контроль над своїми комунікаціями. Бот підключається через Python SDK `mautrix`, обробляє повідомлення через конвеєр Hermes Agent (включаючи використання інструментів, пам'ять та міркування) і відповідає в режимі реального часу. Підтримуються текст, файлові вкладення, зображення, аудіо, відео та опційне сквозне шифрування (E2EE).

Hermes працює з будь‑яким Matrix homeserver — Synapse, Conduit, Dendrite або matrix.org.

Перед налаштуванням ось частина, яку більшість людей хоче дізнатися: як Hermes поводиться після підключення.
## Як поводиться Hermes

| Context | Behavior |
|---------|----------|
| **DMs** | Hermes відповідає на кожне повідомлення. Не потрібен `@mention`. Кожен DM має свою сесію. Встанови `MATRIX_DM_MENTION_THREADS=true`, щоб створювати тред, коли бот `@mentioned` у DM. |
| **Rooms** | За замовчуванням Hermes вимагає `@mention` для відповіді. Встанови `MATRIX_REQUIRE_MENTION=false` або додай ідентифікатори кімнат до `MATRIX_FREE_RESPONSE_ROOMS` для кімнат із вільною відповіддю. Запрошення в кімнати приймаються автоматично. |
| **Threads** | Hermes підтримує треди Matrix (MSC3440). Якщо ти відповідаєш у треді, Hermes зберігає контекст треду ізольованим від головної часової шкали кімнати. Треди, у яких бот вже брав участь, не потребують згадки. |
| **Auto‑threading** | За замовчуванням Hermes автоматично створює тред для кожного повідомлення, на яке він відповідає в кімнаті. Це ізолює розмови. Встанови `MATRIX_AUTO_THREAD=false`, щоб вимкнути. |
| **Shared rooms with multiple users** | За замовчуванням Hermes ізолює історію сесії для кожного користувача всередині кімнати. Двоє людей, які спілкуються в одній кімнаті, не ділять один транскрипт, якщо ти явно не вимкнеш це. |

:::tip
Бот автоматично приєднується до кімнат, коли його запрошують. Просто запроси користувача Matrix бота в будь‑яку кімнату, і він приєднається та почне відповідати.
:::

### Модель сесії в Matrix

За замовчуванням:

- кожен DM отримує свою сесію
- кожен тред отримує власний простір імен сесії
- кожен користувач у спільній кімнаті отримує свою сесію всередині цієї кімнати

Це контролюється у `config.yaml`:

```yaml
group_sessions_per_user: true
```

Встанови `false` лише якщо ти явно хочеш одну спільну розмову для всієї кімнати:

```yaml
group_sessions_per_user: false
```

Спільні сесії можуть бути корисними для колаборативної кімнати, але вони також означають:

- користувачі ділять зростання контексту та витрати токенів
- довге завдання одного користувача, насичене інструментами, може збільшити контекст усіх інших
- запущений процес одного користувача може перервати продовження іншого користувача в тій самій кімнаті

### Налаштування згадок та тредів

Ти можеш налаштувати поведінку згадок та автоматичного створення тредів через змінні середовища або `config.yaml`:

```yaml
matrix:
  require_mention: true           # Require @mention in rooms (default: true)
  free_response_rooms:            # Rooms exempt from mention requirement
    - "!abc123:matrix.org"
  auto_thread: true               # Auto-create threads for responses (default: true)
  dm_mention_threads: false       # Create thread when @mentioned in DM (default: false)
```

Або через змінні середовища:

```bash
MATRIX_REQUIRE_MENTION=true
MATRIX_FREE_RESPONSE_ROOMS=!abc123:matrix.org,!def456:matrix.org
MATRIX_AUTO_THREAD=true
MATRIX_DM_MENTION_THREADS=false
MATRIX_REACTIONS=true          # default: true — emoji reactions during processing
```

:::tip Вимкнення реакцій
`MATRIX_REACTIONS=false` вимикає емодзі‑реакції (👀/✅/❌), які бот додає до вхідних повідомлень у процесі життєвого циклу. Корисно для кімнат, де реакції шумлять або не підтримуються всіма клієнтами.
:::

:::note
Якщо ти оновлюєшся з версії, в якій не було `MATRIX_REQUIRE_MENTION`, бот раніше відповідав на всі повідомлення в кімнатах. Щоб зберегти цю поведінку, встанови `MATRIX_REQUIRE_MENTION=false`.
:::

Цей посібник проведе тебе через повний процес налаштування — від створення облікового запису бота до надсилання першого повідомлення.
## Крок 1: Створити обліковий запис бота

Тобі потрібен обліковий запис користувача Matrix для бота. Є кілька способів це зробити:

### Варіант A: Зареєструватися на власному **homeserver** (рекомендовано)

Якщо ти запускаєш власний **homeserver** (Synapse, Conduit, Dendrite):

1. Використай **admin API** або інструмент реєстрації, щоб створити нового користувача:

```bash
# Synapse example
register_new_matrix_user -c /etc/synapse/homeserver.yaml http://localhost:8008
```

2. Вибери ім’я користувача, наприклад `hermes` — повний ID користувача буде `@hermes:your-server.org`.

### Варіант B: Використати matrix.org або інший публічний **homeserver**

1. Перейди до [Element Web](https://app.element.io) і створити новий обліковий запис.
2. Вибери ім’я користувача для свого бота (наприклад `hermes-bot`).

### Варіант C: Використати власний обліковий запис

Ти також можеш запускати Hermes під своїм користувачем. Це означає, що бот публікує повідомлення від твого імені — корисно для персональних асистентів.
## Крок 2: Отримати токен доступу

Hermes потрібен токен доступу для автентифікації **на** homeserver. У тебе є два варіанти:

### Варіант A: Токен доступу (рекомендовано)

Найнадійніший спосіб отримати токен:

**Через Element:**
1. Увійди в [Element](https://app.element.io) за допомогою облікового запису бота.
2. Перейди до **Settings** → **Help & About**.
3. Прокрути вниз і розгорни **Advanced** — токен доступу буде показаний там.
4. **Скопіюй його одразу.**

**Через API:**

```bash
curl -X POST https://your-server/_matrix/client/v3/login \
  -H "Content-Type: application/json" \
  -d '{
    "type": "m.login.password",
    "user": "@hermes:your-server.org",
    "password": "your-password"
  }'
```

У відповіді є поле `access_token` — скопіюй його.

:::warning[Зберігай токен доступу у безпеці]
Токен доступу дає повний доступ до облікового запису бота в Matrix. Ніколи не публікуй його і не додавай у репозиторій Git. Якщо він став відомий, відкликати його можна, вийшовши зі всіх сесій цього користувача.
:::

### Варіант B: Вхід за паролем

Замість токену доступу можна надати Hermes ідентифікатор користувача бота та пароль. Hermes автоматично ввійде під час запуску. Це простіше, але пароль зберігається у твоєму файлі `.env`.

```bash
MATRIX_USER_ID=@hermes:your-server.org
MATRIX_PASSWORD=your-password
```
## Крок 3: Знайди свій Matrix User ID

Hermes Agent використовує твій Matrix User ID, щоб контролювати, хто може взаємодіяти з ботом. Matrix User ID мають формат `@username:server`.

Щоб знайти його:

1. Відкрий [Element](https://app.element.io) (або улюблений клієнт Matrix).
2. Натисни на свою аватарку → **Settings**.
3. Твій Matrix User ID відображається у верхній частині профілю (наприклад, `@alice:matrix.org`).

:::tip
Matrix User ID завжди починається з `@` і містить `:` перед назвою сервера. Наприклад: `@alice:matrix.org`, `@bob:your-server.com`.
:::
## Крок 4: Налаштування Hermes Agent

### Варіант A: Інтерактивне налаштування (рекомендовано)

Запусти команду керованого налаштування:

```bash
hermes gateway setup
```

Вибери **Matrix**, коли буде запит, потім вкажи URL твого homeserver, access token (або user ID + password) і дозволені user ID, коли запитають.

### Варіант B: Ручна конфігурація

Додай наступне у файл `~/.hermes/.env`:

**Використовуючи access token:**

```bash
# Required
MATRIX_HOMESERVER=https://matrix.example.org
MATRIX_ACCESS_TOKEN=***

# Optional: user ID (auto-detected from token if omitted)
# MATRIX_USER_ID=@hermes:matrix.example.org

# Security: restrict who can interact with the bot
MATRIX_ALLOWED_USERS=@alice:matrix.example.org

# Multiple allowed users (comma-separated)
# MATRIX_ALLOWED_USERS=@alice:matrix.example.org,@bob:matrix.example.org
```

**Використовуючи вхід за паролем:**

```bash
# Required
MATRIX_HOMESERVER=https://matrix.example.org
MATRIX_USER_ID=@hermes:matrix.example.org
MATRIX_PASSWORD=***

# Security
MATRIX_ALLOWED_USERS=@alice:matrix.example.org
```

Опціональні налаштування поведінки у `~/.hermes/config.yaml`:

```yaml
group_sessions_per_user: true
```

- `group_sessions_per_user: true` зберігає контекст кожного учасника ізольовано в спільних кімнатах

### Запуск шлюзу

Після налаштування запусти шлюз Matrix:

```bash
hermes gateway
```

Бот має підключитися до твого homeserver і почати синхронізацію протягом кількох секунд. Надішли йому повідомлення — або DM, або в кімнаті, до якої він приєднався — для тесту.

:::tip
Ти можеш запустити `hermes gateway` у фоні або як службу systemd для постійної роботи. Дивись документацію з розгортання для деталей.
:::
## End-to-End Encryption (E2EE)

Hermes підтримує сквозне шифрування Matrix, тож ти можеш спілкуватися зі своїм ботом у зашифрованих кімнатах.

### Requirements

E2EE вимагає бібліотеку `mautrix` з extras для шифрування та C‑бібліотеку `libolm`:

```bash
# Install mautrix with E2EE support
pip install 'mautrix[encryption]'

# Or install with hermes extras
pip install 'hermes-agent[matrix]'
```

Тобі також потрібно встановити `libolm` у системі:

```bash
# Debian/Ubuntu
sudo apt install libolm-dev

# macOS
brew install libolm

# Fedora
sudo dnf install libolm-devel
```

### Enable E2EE

Додай у свій `~/.hermes/.env`:

```bash
MATRIX_ENCRYPTION=true
```

Коли E2EE увімкнено, Hermes:

- Зберігає ключі шифрування у `~/.hermes/platforms/matrix/store/` (у старих інсталяціях: `~/.hermes/matrix/store/`)
- Завантажує ключі пристрою під час першого підключення
- Автоматично розшифровує вхідні повідомлення та шифрує вихідні
- Автоматично приєднується до зашифрованих кімнат за запрошенням

### Cross-Signing Verification (Recommended)

Якщо у твоєму обліковому записі Matrix увімкнено крос‑підпис (за замовчуванням у Element), встанови ключ відновлення, щоб бот міг самостійно підписати свій пристрій під час запуску. Без цього інші клієнти Matrix можуть відмовитися ділитися сеансами шифрування з ботом після ротації ключа пристрою.

```bash
MATRIX_RECOVERY_KEY=EsT... your recovery key here
```

**Де його знайти:** У Element відкрий **Settings** → **Security & Privacy** → **Encryption** → твій ключ відновлення (також називається «Security Key»). Це ключ, який ти зберігав під час першого налаштування крос‑підпису.

При кожному запуску, якщо встановлено `MATRIX_RECOVERY_KEY`, Hermes імпортує ключі крос‑підпису з безпечного сховища секретів homeserver‑а та підписує поточний пристрій. Це ідемпотентно і безпечно залишати увімкненим постійно.

:::warning[Deleting the crypto store]
Якщо ти видалиш `~/.hermes/platforms/matrix/store/crypto.db`, бот втратить свою ідентичність шифрування. Простий перезапуск з тим самим device ID **не** відновить повністю — homeserver все ще зберігає одноразові ключі, підписані старим ідентифікаційним ключем, і peers не зможуть встановити нові Olm‑сеанси.

Hermes виявляє цю ситуацію під час запуску і відмовляється увімкнути E2EE, записуючи в лог: `device XXXX has stale one-time keys on the server signed with a previous identity key`.

**Найпростіше відновлення:** згенеруй новий access token (який створить новий device ID без застарілої історії ключів). Дивись розділ «Upgrading from a previous version with E2EE» нижче. Це найнадійніший шлях і не вимагає змін у базі даних homeserver‑а.

**Ручне відновлення** (для досвідчених — зберігає той самий device ID):

1. Зупини Synapse і видали старий пристрій з його бази даних:
   ```bash
   sudo systemctl stop matrix-synapse
   sudo sqlite3 /var/lib/matrix-synapse/homeserver.db "
     DELETE FROM e2e_device_keys_json WHERE device_id = 'DEVICE_ID' AND user_id = '@hermes:your-server';
     DELETE FROM e2e_one_time_keys_json WHERE device_id = 'DEVICE_ID' AND user_id = '@hermes:your-server';
     DELETE FROM e2e_fallback_keys_json WHERE device_id = 'DEVICE_ID' AND user_id = '@hermes:your-server';
     DELETE FROM devices WHERE device_id = 'DEVICE_ID' AND user_id = '@hermes:your-server';
   "
   sudo systemctl start matrix-synapse
   ```
   Або через API адміністрування Synapse (зверни увагу на URL‑закодований user ID):
   ```bash
   curl -X DELETE -H "Authorization: Bearer ADMIN_TOKEN" \
     'https://your-server/_synapse/admin/v2/users/%40hermes%3Ayour-server/devices/DEVICE_ID'
   ```
   Примітка: видалення пристрою через API адміністрування може також анулювати пов’язаний access token. Після цього, можливо, доведеться згенерувати новий токен.

2. Видали локальне сховище криптографії та перезапусти Hermes:
   ```bash
   rm -f ~/.hermes/platforms/matrix/store/crypto.db*
   # restart hermes
   ```

Інші клієнти Matrix (Element, matrix-commander) можуть кешувати старі ключі пристрою. Після відновлення введи `/discardsession` у Element, щоб примусово створити новий сеанс шифрування з ботом.
:::

:::info
Якщо `mautrix[encryption]` не встановлено або `libolm` відсутня, бот автоматично переходить у режим простого (нешифрованого) клієнта. У логах ти побачиш попередження.
:::
## Home Room

Ти можеш призначити «домашню кімнату», куди бот надсилатиме проактивні повідомлення (наприклад, вивід cron‑завдань, нагадування та сповіщення). Є два способи це зробити:

### За допомогою слеш‑команди

Введи `/sethome` у будь‑якій кімнаті Matrix, де присутній бот. Ця кімната стане домашньою.

### Ручна конфігурація

Додай це до твого `~/.hermes/.env`:

```bash
MATRIX_HOME_ROOM=!abc123def456:matrix.example.org
```
## Дозволений список кімнат (`allowed_rooms`)

Обмежити бота фіксованим набором кімнат Matrix. Коли встановлено, бот **лише** відповідає в кімнатах, ідентифікатор яких присутній у списку — повідомлення з будь‑якої іншої кімнати ігноруються без повідомлення, навіть якщо бот згадано.

**Прямі повідомлення (DM) виключені** з цього фільтра, тому уповноважені користувачі завжди можуть спілкуватися з ботом один‑на‑один.

```yaml
matrix:
  allowed_rooms:
    - "!abc123def456:matrix.example.org"
    - "!opsroom789:matrix.example.org"
```

Або через змінну середовища (через кому):

```bash
MATRIX_ALLOWED_ROOMS="!abc123def456:matrix.example.org,!opsroom789:matrix.example.org"
```

**Поведінка**:

- Порожнє / не встановлено → без обмежень (за замовчуванням).
- Непорожнє → ідентифікатор кімнати має бути у списку. Перевірка виконується **до** будь‑якого іншого обмеження (вимога згадки, список дозволених відправників тощо).
- Використовуйте **внутрішній ID** кімнати (`!abc...:server`), а не її псевдонім (`#room:server`). Внутрішній ID кімнати можна знайти в Element у розділі **Кімната → Налаштування → Додатково**.

Дивись також: [admin/user slash command split](../../reference/slash-commands.md#permissions-and-adminuser-split).


:::tip
Щоб знайти ID кімнати: у Element перейдіть у кімнату → **Налаштування** → **Додатково** → **Внутрішній ID кімнати** показано там (починається з `!`).
:::
## Усунення проблем

### Бот не відповідає на повідомлення

**Причина**: Бот не приєднався до кімнати, або `MATRIX_ALLOWED_USERS` не містить вашого User ID.

**Виправлення**: Запроси бота до кімнати — він автоматично приєднується після запрошення. Перевір, чи твій User ID знаходиться в `MATRIX_ALLOWED_USERS` (використовуй повний формат `@user:server`). Перезапусти шлюз.

### Бот приєднується до кімнат, але беззвучно відкидає кожне повідомлення (зсув часу)

**Причина**: Системний годинник хоста встановлений вперед щодо реального часу. Адаптер Matrix застосовує 5‑секундний фільтр «grace» під час запуску (`event_ts < startup_ts - 5`), щоб ігнорувати події, відтворені під час початкової синхронізації. Коли стінковий годинник випереджає, кожна вхідна подія виглядає «старішою за запуск» і відкидається до досягнення обробника повідомлень — бот виглядає підключеним, але ніколи не відповідає. Див. [#12614](https://github.com/NousResearch/hermes-agent/issues/12614).

**Симптом**: У журналі шлюзу показує `Matrix: dropped N live events as 'too old' more than 30s after startup`.

**Виправлення**: Синхронізуй годинник хоста за допомогою NTP і перезапусти бота:

```bash
# Debian/Ubuntu
sudo timedatectl set-ntp true
timedatectl status   # confirm "System clock synchronized: yes"

# macOS
sudo sntp -sS time.apple.com
```

### «Failed to authenticate» / «whoami failed» під час запуску

**Причина**: Токен доступу або URL домашнього сервера неправильні.

**Виправлення**: Переконайся, що `MATRIX_HOMESERVER` вказує на твій домашній сервер (включай `https://`, без кінцевого слешу). Перевір, чи `MATRIX_ACCESS_TOKEN` дійсний — спробуй його за допомогою curl:

```bash
curl -H "Authorization: Bearer YOUR_TOKEN" \
  https://your-server/_matrix/client/v3/account/whoami
```

Якщо команда повертає інформацію про твого користувача, токен дійсний. Якщо повертає помилку, згенеруй новий токен.

### Помилка «mautrix not installed»

**Причина**: Пакет Python `mautrix` не встановлений.

**Виправлення**:
```bash
pip install 'mautrix[encryption]'
```

Або з extras Hermes:

```bash
pip install 'hermes-agent[matrix]'
```

### Помилки шифрування / «could not decrypt event»

**Причина**: Відсутні ключі шифрування, `libolm` не встановлений, або пристрій бота не довірений.

**Виправлення**:
1. Переконайся, що `libolm` встановлений у твоїй системі (див. розділ E2EE вище).
2. Переконайся, що `MATRIX_ENCRYPTION=true` встановлено у твоєму `.env`.
3. У клієнті Matrix (Element) відкрий профіль бота → Sessions → перевір/доверь пристрій бота.
4. Якщо бот лише приєднався до зашифрованої кімнати, він може розшифровувати лише повідомлення, надіслані *після* приєднання. Старі повідомлення недоступні.

### Оновлення з попередньої версії з E2EE

:::tip
Якщо ти також вручну видалив `crypto.db`, дивись попередження «Deleting the crypto store» у розділі E2EE вище — потрібні додаткові кроки для очищення застарілих одноразових ключів з домашнього сервера.
:::

Якщо ти раніше користувався Hermes з `MATRIX_ENCRYPTION=true` і оновлюєшся до
версії, що використовує нове криптосховище на базі SQLite, ідентичність шифрування бота змінилася. Твій клієнт Matrix (Element) може кешувати старі ключі пристрою
і відмовлятися ділитися сесіями шифрування з ботом.

**Симптоми**: Бот підключається і в журналах показує «E2EE enabled», але всі
повідомлення мають «could not decrypt event» і бот не відповідає.

**Що відбувається**: Стан старого шифрування (з попереднього бекенду `matrix-nio` або
серіалізації `mautrix`) несумісний з новим криптосховищем SQLite. Бот створює нову ідентичність шифрування, але твій клієнт Matrix все ще має кешовані старі ключі і не ділиться сесією шифрування з пристроєм, чиї ключі змінилися. Це функція безпеки Matrix — клієнти розглядають зміну ключів пристрою як підозрілу.

**Виправлення** (одноразова міграція):

1. **Згенеруй новий токен доступу** для отримання нового Device ID. Найпростіший спосіб:

      ```bash
   curl -X POST https://your-server/_matrix/client/v3/login \
     -H "Content-Type: application/json" \
     -d '{
       "type": "m.login.password",
       "identifier": {"type": "m.id.user", "user": "@hermes:your-server.org"},
       "password": "***",
       "initial_device_display_name": "Hermes Agent"
     }'
   ```

   Скопіюй новий `access_token` і онови `MATRIX_ACCESS_TOKEN` у `~/.hermes/.env`.

2. **Видали старий стан шифрування**:

      ```bash
   rm -f ~/.hermes/platforms/matrix/store/crypto.db
   rm -f ~/.hermes/platforms/matrix/store/crypto_store.*
   ```

3. **Вкажи свій recovery key** (якщо ти користуєшся cross‑signing — більшість користувачів Element це роблять). Додай до `~/.hermes/.env`:

      ```bash
   MATRIX_RECOVERY_KEY=EsT... your recovery key here
   ```

   Це дозволяє боту самопідписатися за допомогою ключів cross‑signing під час запуску, тож Element довіряє новому пристрою одразу. Без цього Element може бачити новий пристрій як неперевірений і відмовлятися ділитися сесіями шифрування. Знайди свій recovery key у Element у **Settings** → **Security & Privacy** → **Encryption**.

4. **Примусово зміни сесію шифрування у клієнті Matrix**. У Element
   відкрий DM‑кімнату з ботом і введи `/discardsession`. Це змушує Element
   створити нову сесію шифрування і поділитися нею з новим пристроєм бота.

5. **Перезапусти шлюз**:

      ```bash
   hermes gateway run
   ```

   Якщо встановлено `MATRIX_RECOVERY_KEY`, у журналах має з’явитися `Matrix: cross-signing verified via recovery key`.

6. **Надішли нове повідомлення**. Бот має розшифрувати його і відповісти нормально.

:::note
Після міграції повідомлення, надіслані *до* оновлення, розшифрувати неможливо — старі
ключі шифрування втрачено. Це впливає лише на перехід; нові повідомлення працюють
звично.
:::

:::tip
**Нові інсталяції не підлягають цьому.** Міграція потрібна лише, якщо у тебе був
робочий E2EE‑набір з попередньою версією Hermes і ти оновлюєшся.

**Навіщо новий токен доступу?** Кожен токен Matrix прив’язаний до конкретного Device ID. Повторне використання того ж Device ID з новими ключами шифрування змушує інші клієнти Matrix не довіряти пристрою (вони бачать зміну ідентифікаційних ключів як потенційну загрозу). Новий токен створює новий Device ID без історії старих ключів, тому інші клієнти довіряють йому одразу.
:::
## Режим проксі (E2EE на macOS)

Matrix E2EE вимагає `libolm`, який не компілюється на macOS ARM64 (Apple Silicon). Додаток `hermes-agent[matrix]` доступний лише для Linux. Якщо ти користуєшся macOS, режим проксі дозволяє запускати E2EE у Docker‑контейнері на Linux‑VM, тоді як сам агент працює нативно на macOS з повним доступом до твоїх локальних файлів, пам’яті та інструментів.

### Як це працює

```
macOS (Host):
  └─ hermes gateway
       ├─ api_server adapter ← listens on 0.0.0.0:8642
       ├─ AIAgent ← single source of truth
       ├─ Sessions, memory, skills
       └─ Local file access (Obsidian, projects, etc.)

Linux VM (Docker):
  └─ hermes gateway (proxy mode)
       ├─ Matrix adapter ← E2EE decryption/encryption
       └─ HTTP forward → macOS:8642/v1/chat/completions
           (no LLM API keys, no agent, no inference)
```

Docker‑контейнер обробляє лише протокол Matrix + E2EE. Коли надходить повідомлення, він розшифровує його і пересилає текст хосту через стандартний HTTP‑запит. Хост запускає агента, викликає інструменти, генерує відповідь і передає її назад у потоці. Контейнер шифрує і надсилає відповідь у Matrix. Усі сесії об’єднані — CLI, Matrix, Telegram та будь‑яка інша платформа діляться однією пам’яттю та історією розмов.

### Крок 1: Налаштування хоста (macOS)

Увімкни API‑сервер, щоб хост приймав вхідні запити від Docker‑контейнера.

Додай у `~/.hermes/.env`:

```bash
API_SERVER_ENABLED=true
API_SERVER_KEY=your-secret-key-here
API_SERVER_HOST=0.0.0.0
```

- `API_SERVER_HOST=0.0.0.0` прив’язується до всіх інтерфейсів, щоб Docker‑контейнер міг до нього дістатися.
- `API_SERVER_KEY` потрібен для прив’язки, що не є loopback. Вибери надійний випадковий рядок.
- API‑сервер працює на порту 8642 за замовчуванням (можна змінити за допомогою `API_SERVER_PORT`, якщо потрібно).

Запусти шлюз:

```bash
hermes gateway
```

Ти маєш побачити, як API‑сервер стартує разом з будь‑якими іншими налаштованими платформами. Перевір, чи доступний він з VM:

```bash
# From the Linux VM
curl http://<mac-ip>:8642/health
```

### Крок 2: Налаштування Docker‑контейнера (Linux VM)

Контейнеру потрібні облікові дані Matrix та URL проксі. Йому НЕ потрібні ключі API LLM.

**`docker-compose.yml`:**

```yaml
services:
  hermes-matrix:
    build: .
    environment:
      # Matrix credentials
      MATRIX_HOMESERVER: "https://matrix.example.org"
      MATRIX_ACCESS_TOKEN: "syt_..."
      MATRIX_ALLOWED_USERS: "@you:matrix.example.org"
      MATRIX_ENCRYPTION: "true"
      MATRIX_DEVICE_ID: "HERMES_BOT"

      # Proxy mode — forward to host agent
      GATEWAY_PROXY_URL: "http://192.168.1.100:8642"
      GATEWAY_PROXY_KEY: "your-secret-key-here"
    volumes:
      - ./matrix-store:/root/.hermes/platforms/matrix/store
```

**`Dockerfile`:**

```dockerfile
FROM python:3.11-slim

RUN apt-get update && apt-get install -y libolm-dev && rm -rf /var/lib/apt/lists/*
RUN pip install 'hermes-agent[matrix]'

CMD ["hermes", "gateway"]
```

Ось весь вміст контейнера. Жодних API‑ключів для OpenRouter, Anthropic чи будь‑якого іншого провайдера інференсу.

### Крок 3: Запуск обох

1. Спочатку запусти шлюз на хості:
      ```bash
   hermes gateway
   ```

2. Потім запусти Docker‑контейнер:
      ```bash
   docker compose up -d
   ```

3. Надішли повідомлення в зашифрований чат Matrix. Контейнер розшифрує його, передасть хосту, а хост потоком відправить відповідь назад.

### Довідка з налаштувань

Режим проксі налаштовується **на стороні контейнера** (тонкий шлюз):

| Налаштування | Опис |
|--------------|------|
| `GATEWAY_PROXY_URL` | URL віддаленого Hermes API‑сервера (наприклад, `http://192.168.1.100:8642`) |
| `GATEWAY_PROXY_KEY` | Токен Bearer для автентифікації (повинен збігатися з `API_SERVER_KEY` на хості) |
| `gateway.proxy_url` | Те саме, що `GATEWAY_PROXY_URL`, але у `config.yaml` |

На стороні хоста потрібні:

| Налаштування | Опис |
|--------------|------|
| `API_SERVER_ENABLED` | Встановити `true` |
| `API_SERVER_KEY` | Токен Bearer (спільний з контейнером) |
| `API_SERVER_HOST` | Встановити `0.0.0.0` для мережевого доступу |
| `API_SERVER_PORT` | Номер порту (за замовчуванням: `8642`) |

### Працює на будь‑якій платформі

Режим проксі не обмежений лише Matrix. Будь‑який адаптер платформи може його використовувати — встанови `GATEWAY_PROXY_URL` у будь‑якому екземплярі шлюзу, і він буде пересилати запити до віддаленого агента замість запуску локального. Це корисно, коли адаптер платформи має працювати в іншому середовищі, ніж агент (мережна ізоляція, вимоги E2EE, обмеження ресурсів).

:::tip
Безперервність сесії підтримується через заголовок `X-Hermes-Session-Id`. API‑сервер хоста відстежує сесії за цим ID, тому розмови зберігаються між повідомленнями так само, як і при локальному агенті.
:::

:::note
**Обмеження (v1):** Повідомлення про прогрес інструментів від віддаленого агента не передаються назад — користувач бачить лише потокову фінальну відповідь, а не окремі виклики інструментів. Запити на підтвердження небезпечних команд обробляються на стороні хоста і не пересилаються користувачеві Matrix. Це може бути виправлено в майбутніх оновленнях.
:::

### Проблеми синхронізації / бот відстає

**Причина**: Довгі виконання інструментів можуть затримувати цикл синхронізації, або homeserver працює повільно.

**Виправлення**: Цикл синхронізації автоматично повторює спробу кожні 5 секунд у разі помилки. Перевір журнали Hermes на предмет попереджень, пов’язаних із синхронізацією. Якщо бот постійно відстає, переконайся, що у homeserver достатньо ресурсів.

### Бот офлайн

**Причина**: Шлюз Hermes не запущений або не зміг під’єднатися.

**Виправлення**: Переконайся, що запущено `hermes gateway`. Переглянь вивід у терміналі на предмет повідомлень про помилки. Типові проблеми: неправильний URL homeserver, прострочений токен доступу, недоступний homeserver.

### «Користувач не дозволений» / бот ігнорує тебе

**Причина**: Твій User ID відсутній у `MATRIX_ALLOWED_USERS`.

**Виправлення**: Додай свій User ID у `MATRIX_ALLOWED_USERS` у `~/.hermes/.env` і перезапусти шлюз. Використовуй повний формат `@user:server`.
## Безпека

:::warning
Завжди встановлюй `MATRIX_ALLOWED_USERS`, щоб обмежити, хто може взаємодіяти з ботом. Без цього шлюзу за замовчуванням відхиляються всі користувачі як запобіжний захід. Додавай лише ID користувачів людей, яким ти довіряєш — авторизовані користувачі мають повний доступ до можливостей агента, включаючи використання інструментів та доступ до системи.
:::

Для отримання додаткової інформації про захист розгортання Hermes Agent, переглянь [Посібник з безпеки](../security.md).
## Примітки

- **Any homeserver**: Працює з Synapse, Conduit, Dendrite, matrix.org або будь‑яким homeserver, що відповідає специфікації Matrix. Не потрібне жодне конкретне програмне забезпечення homeserver.
- **Federation**: Якщо ти на федеративному homeserver, бот може спілкуватися з користувачами інших серверів — просто додай їхні повні `@user:server` ID у `MATRIX_ALLOWED_USERS`.
- **Auto-join**: Бот автоматично приймає запрошення в кімнати та приєднується. Він починає відповідати одразу після приєднання.
- **Media support**: Hermes може надсилати та отримувати зображення, аудіо, відео та файлові вкладення. Медіа завантажується на твій homeserver за допомогою API репозиторію вмісту Matrix.
- **Native voice messages (MSC3245)**: Адаптер Matrix автоматично позначає вихідні голосові повідомлення прапорцем `org.matrix.msc3245.voice`. Це означає, що відповіді TTS та голосове аудіо відображаються як **нативні голосові бульбашки** в Element та інших клієнтах, які підтримують MSC3245, а не як загальні аудіофайли‑вкладення. Вхідні голосові повідомлення з прапорцем MSC3245 також правильно ідентифікуються та передаються до транскрипції «мова‑в‑текст». Ніяких налаштувань не потрібно — це працює автоматично.
