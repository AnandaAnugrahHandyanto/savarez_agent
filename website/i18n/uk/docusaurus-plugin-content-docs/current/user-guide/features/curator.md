# Curator

Curator — це процес фонового обслуговування **створених агентом навичок**. Він відстежує, як часто кожна навичка переглядається, використовується та оновлюється, переміщує довго не використовувані навички через стани `active → stale → archived` і періодично запускає короткий перегляд допоміжної моделі, який пропонує консолідації або виправлення відхилень.

Він існує, щоб навички, створені через [цикл навчання](/user-guide/features/skills#agent-managed-skills-skill_manage-tool), не накопичувалися назавжди. Кожного разу, коли агент вирішує нову задачу і зберігає навичку, ця навичка потрапляє в `~/.hermes/skills/`. Без обслуговування ти отримуєш десятки вузьких майже‑дублікатів, які забруднюють каталог і витрачають токени.

Curator **ніколи не торкається** вбудованих навичок (поставлених разом з репозиторієм) або навичок, встановлених через hub (з [agentskills.io](https://agentskills.io)). Він переглядає лише навички, створені самим агентом. Також він **ніколи не видаляє автоматично** — найгірший результат — архівація в `~/.hermes/skills/.archive/`, що можна відновити.

Відстежує [issue #7816](https://github.com/NousResearch/hermes-agent/issues/7816).
## Як це працює

Куратор запускається перевіркою неактивності, а не демоном cron. При старті CLI‑сесії та під час повторюваного тіку у потоці **cron‑ticker** шлюзу, Hermes перевіряє, чи:

1. Минуло достатньо часу з останнього запуску куратора (`interval_hours`, за замовчуванням **7 днів**), і
2. Агент був неактивний достатньо довго (`min_idle_hours`, за замовчуванням **2 години**).

Якщо обидві умови виконані, створюється фоновий форк `AIAgent` — той самий шаблон, що використовується підказками **самонавчального агента** пам’яті/навичок. Форк працює у власному кеші підказок і ніколи не торкається активної розмови.

:::info First-run behavior
При новій інсталяції (або під час першого тіку після `hermes update`) куратор **не запускається одразу**. Перше спостереження встановлює `last_run_at` у «зараз» і відкладає перший реальний прохід на повний `interval_hours`. Це дає тобі повний інтервал для перегляду бібліотеки навичок, закріплення важливих елементів або повної відмови, перш ніж куратор щось змінить.

Якщо хочеш побачити, що куратор *зробив би* перед реальним запуском, виконай `hermes curator run --dry-run` — він створює той самий звіт огляду без зміни бібліотеки.
:::

Запуск має два етапи:

1. **Автоматичні переходи** (детерміновані, без LLM). Навички, які не використовувалися `stale_after_days` (30), стають `stale`; навички, які не використовувалися `archive_after_days` (90), переміщуються до `~/.hermes/skills/.archive/`.
2. **Огляд LLM** (один прохід aux‑моделі, `max_iterations=8`). Форкований агент оглядає створені агентом навички, може прочитати будь‑яку з них за допомогою `skill_view` і вирішує для кожної навички, чи залишити, виправити (через `skill_manage`), консолідувати перекриваючі або архівувати за допомогою інструменту терміналу. Консолідація розглядає навичку як повний пакет: якщо навичка має `references/`, `templates/`, `scripts/`, `assets/` або відносні посилання на ці шляхи, куратор повинен або залишити її окремо, перенести потрібні підтримуючі файли та переписати шляхи, або архівувати весь пакет без змін — а не лише `SKILL.md` у файл `references/` іншої навички.

Закріплені навички недоступні ні для автоматичних переходів куратора, ні для інструмента `skill_manage` агента. Дивись [Pinning a skill](#pinning-a-skill) нижче.
## Конфігурація

Усі налаштування зберігаються у `config.yaml` під розділом `curator:` (не в `.env` — це не секрет). За замовчуванням:

⟦HOLD_0⟩

Щоб повністю вимкнути, встанови `curator.enabled: false`.

### Запуск огляду на дешевшій допоміжній моделі

Прохід огляду LLM куратора — це звичайний слот допоміжного завдання — `auxiliary.curator` — поряд з Vision, Compression, Session Search тощо. «Auto» означає «використовувати мою основну чат‑модель»; перевизнач слот, щоб прив’язати конкретного провайдера + модель для проходу огляду.

**Найпростіший варіант — `hermes model`:**

⟦HOLD_1⟩

Той самий вибірник доступний у веб‑дашборді у вкладці **Models**.

**Прямий `config.yaml` (еквівалент):**

⟦HOLD_2⟩

Залишивши `provider: auto` (за замовчуванням), ти направляєш прохід огляду через будь‑яку свою основну чат‑модель, що відповідає поведінці всіх інших допоміжних завдань.

:::note Legacy config
Попередні випуски використовували одноразовий блок `curator.auxiliary.{provider,model}`. Цей шлях усе ще працює, але виводить рядок журналу про застарілість — будь ласка, мігруй до `auxiliary.curator`, зазначеного вище, щоб куратор користувався тією ж інфраструктурою (`hermes model`, вкладка **Models** у дашборді, `base_url`, `api_key`, `timeout`, `extra_body`) що й будь‑яке інше допоміжне завдання.
:::
## CLI

```bash
hermes curator status         # last run, counts, pinned list, LRU top 5
hermes curator run            # trigger a review now (blocks until the LLM pass finishes)
hermes curator run --background  # fire-and-forget: start the LLM pass in a background thread
hermes curator run --dry-run  # preview only — report without any mutations
hermes curator backup         # take a manual snapshot of ~/.hermes/skills/
hermes curator rollback       # restore from the newest snapshot
hermes curator rollback --list     # list available snapshots
hermes curator rollback --id <ts>  # restore a specific snapshot
hermes curator rollback -y         # skip the confirmation prompt
hermes curator pause          # stop runs until resumed
hermes curator resume
hermes curator pin <skill>    # never auto-transition this skill
hermes curator unpin <skill>
hermes curator restore <skill>  # move an archived skill back to active
```
## Резервні копії та відкат

Перед кожним реальним запуском куратора Hermes створює tar.gz‑знімок `~/.hermes/skills/` у `~/.hermes/skills/.curator_backups/<utc-iso>/skills.tar.gz`. Якщо під час запуску архівується або консолідується щось, чого ти не хотів змінювати, можна скасувати весь процес однією командою:

```bash
hermes curator rollback        # restore newest snapshot (with confirmation)
hermes curator rollback -y     # skip the prompt
hermes curator rollback --list # see all snapshots with reason + size
```

Сам відкат можна скасувати: перед заміною дерева навичок Hermes робить ще один знімок з міткою `pre-rollback to <target-id>`, тому помилковий відкат можна відмінити, прокрутивши вперед до цього знімка за допомогою `--id`.

Ти також можеш робити ручні знімки у будь‑який час за допомогою `hermes curator backup --reason "before-refactor"`. Рядок `--reason` потрапляє у `manifest.json` знімка і відображається у `--list`.

Знімки очищуються до `curator.backup.keep` (за замовчуванням 5), щоб обмежити використання диска:

```yaml
curator:
  backup:
    enabled: true
    keep: 5
```

Встанови `curator.backup.enabled: false`, щоб вимкнути автоматичне створення знімків. Ручна команда `hermes curator backup` все ще працює, коли резервні копії вимкнені, лише якщо спочатку встановити `enabled: true` — прапорець керує обома шляхами симетрично, тому немає можливості випадково пропустити попередній знімок перед запуском під час змінних запусків.

`hermes curator status` також виводить п’ять найрідше використовуваних навичок — швидкий спосіб побачити, які, ймовірно, наступними стануть застарілими.

Ті ж підкоманди доступні як слеш‑команда `/curator` всередині запущеної сесії (CLI або платформи шлюзу).
## Що означає «agent-created»

Навичка вважається створеною агентом, якщо її назва **не** міститься у:

- `~/.hermes/skills/.bundled_manifest` (навички, скопійовані з репозиторію під час встановлення), і
- `~/.hermes/skills/.hub/lock.json` (навички, встановлені за допомогою `hermes skills install`).

Все інше в `~/.hermes/skills/` — це вільна зона для куратора. Це включає:

- Навички, які агент зберігав через `skill_manage(action="create")` під час розмови.
- Навички, які ти створив вручну, написавши `SKILL.md`.
- Навички, додані через зовнішні каталоги навичок, на які ти вказав Hermes.

:::warning Твої навички, написані вручну, виглядають так само, як і навички, збережені агентом
Провенанс тут **бінарний** (bundled/hub проти всього іншого). Куратор не може розрізнити навичку, написану вручну і яку ти використовуєш у приватних робочих процесах, від навички, яку цикл самонавчання зберіг під час сесії. Обидві потрапляють у bucket «agent-created».
:::

Перед першим реальним проходом (за замовчуванням — 7 днів після встановлення) знайди час, щоб:

1. Запустити `hermes curator run --dry-run`, щоб точно побачити, що запропонує куратор.
2. Використати `hermes curator pin <name>`, щоб відокремити все, що не хочеш, щоб було змінено.
3. Або встановити `curator.enabled: false` у `config.yaml`, якщо ти хочеш сам керувати бібліотекою.

Архіви завжди можна відновити за допомогою `hermes curator restore <name>`, проте простіше зафіксувати їх заздалегідь, ніж потім шукати консольдацію.

Якщо ти хочеш захистити конкретну навичку від будь‑яких змін — наприклад, навичку, написану вручну, яку ти використовуєш — використай `hermes curator pin <name>`. Дивись наступний розділ.
## Прикріплення навички

Прикріплення захищає навичку від видалення — і автоматичними проходами архіву куратора, і викликом інструмента `skill_manage(action="delete")` агента. Після того, як навичка прикріплена:

- **Куратор** пропускає її під час автоматичних переходів (`active → stale → archived`), і її проход LLM‑перегляду інструктується залишити її без змін.
- **Інструмент `skill_manage` агента** відмовляє у `delete`, вказуючи користувачу на `hermes curator unpin <name>`. Патчі та правки все ще проходять, тому агент може покращувати вміст прикріпленої навички, коли з’являються підводні камені, без необхідності виконувати послідовність прикріпити/відкріпити/повторно‑прикріпити.

Прикріпити та відкріпити можна за допомогою:

```bash
hermes curator pin <skill>
hermes curator unpin <skill>
```

Прапорець зберігається як `"pinned": true` у записі навички у файлі `~/.hermes/skills/.usage.json`, тому він зберігається між сесіями.

Тільки **створені агентом** навички можна прикріпити — вбудовані та встановлені з хаба навички ніколи не підлягають мутації куратора, і `hermes curator pin` відмовить з пояснювальним повідомленням, якщо спробувати.

Якщо потрібна ще сильніша гарантія, ніж «без видалення» — наприклад, повне замороження вмісту навички, коли агент все ще читає її — відредагуйте файл `~/.hermes/skills/<name>/SKILL.md` безпосередньо у вашому редакторі. Прикріплення захищає лише інструмент‑викликане видалення, а не ваш власний доступ до файлової системи.
## Теле́метрія використання

Куратор підтримує sidecar у `~/.hermes/skills/.usage.json` з одним записом на кожен інструмент:

```json
{
  "my-skill": {
    "use_count": 12,
    "view_count": 34,
    "last_used_at": "2026-04-24T18:12:03Z",
    "last_viewed_at": "2026-04-23T09:44:17Z",
    "patch_count": 3,
    "last_patched_at": "2026-04-20T22:01:55Z",
    "created_at": "2026-03-01T14:20:00Z",
    "state": "active",
    "pinned": false,
    "archived_at": null
  }
}
```

Лічильники збільшуються, коли:

- `view_count`: агент викликає `skill_view` для інструмента.
- `use_count`: інструмент завантажується у prompt розмови.
- `patch_count`: `skill_manage patch/edit/write_file/remove_file` виконується над інструментом.

Вбудовані та встановлені через hub інструменти явно виключені з записів телеметрії.
## Звіти за запуском

Кожен запуск куратора записує каталог з міткою часу у `~/.hermes/logs/curator/`:

```
~/.hermes/logs/curator/
└── 20260429-111512/
    ├── run.json      # machine-readable: full fidelity, stats, LLM output
    └── REPORT.md     # human-readable summary
```

`REPORT.md` — це швидкий спосіб побачити, що зробив конкретний запуск: які **skill** перейшли, що сказав LLM‑рецензент, які **skill** він виправив. Зручно для аудиту без необхідності grep‑ити `agent.log`.

### Карта перейменувань у підсумку

Якщо під час запуску кілька **skill** були об’єднані під однією «umbrella» (або об’єднані майже дублікати), у підсумковому виведенні, що показується в кінці запуску, користувачеві показується явна карта перейменувань, що містить кожну пару `old-name → new‑name`, яку застосував куратор. Це додається до рядків переходу кожного **skill**, тому коли хвиля перейменувань відбувається, ти можеш одразу їх помітити без порівняння JSON‑звіту. Підказка також з’являється під `hermes curator pin`, щоб ти міг одразу закріпити нову назву, якщо хочеш зафіксувати нову мітку.
## Відновлення заархівованого skill

Якщо куратор заархівував щось, що ти все ще хочеш:

```bash
hermes curator restore <skill-name>
```

Це переміщує skill назад з `~/.hermes/skills/.archive/` у активне дерево і скидає його стан до `active`. Відновлення відхиляється, якщо з того часу був встановлений пакетний або hub‑встановлений skill під тим самим ім’ям (може затінювати upstream).
## Вимкнення за середовищем

Curator увімкнено за замовчуванням. Щоб його вимкнути:

- **Лише для одного профілю:** відредагуй `~/.hermes/config.yaml` (або конфігурацію активного профілю) і встанови `curator.enabled: false`.
- **Лише для одного запуску:** `hermes curator pause` — пауза зберігається між сесіями; використай `resume`, щоб знову увімкнути.

Curator також відмовляється запускатися, якщо не минуло `min_idle_hours`, тому на активній машині розробника він природно працює лише під час тихих періодів.
