# Вбудовані плагіни

Hermes постачається з невеликим набором плагінів, включених у репозиторій. Вони розташовані у `<repo>/plugins/<name>/` і завантажуються автоматично разом із плагінами, встановленими користувачем у `~/.hermes/plugins/`. Вони використовують той самий інтерфейс плагінів, що й сторонні плагіни — hooks, інструменти, slash‑команди — лише підтримуються у дереві.

Дивись сторінку [Plugins](/user-guide/features/plugins) для загального огляду системи плагінів і [Build a Hermes Plugin](/guides/build-a-hermes-plugin), щоб написати свій власний.
## Як працює виявлення

`PluginManager` сканує чотири джерела у зазначеному порядку:

1. **Bundled** — `<repo>/plugins/<name>/` (те, що документовано на цій сторінці)
2. **User** — `~/.hermes/plugins/<name>/`
3. **Project** — `./.hermes/plugins/<name>/` (вимагає `HERMES_ENABLE_PROJECT_PLUGINS=1`)
4. **Pip entry points** — `hermes_agent.plugins`

При колізії імен перевагу мають пізніші джерела — плагін користувача з назвою `disk-cleanup` замінить вбудований.

`plugins/memory/` та `plugins/context_engine/` навмисно виключені зі сканування вбудованих плагінів. Ці каталоги використовують власні шляхи виявлення, оскільки провайдери пам’яті та контекстні рушії є одно-вибірковими провайдерами, які налаштовуються через `hermes memory setup` / `context.engine` у конфігурації.
## Пакетні плагіни — це опція

Пакетні плагіни постачаються вимкненими. Discovery знаходить їх (вони з’являються у `hermes plugins list` та інтерактивному інтерфейсі `hermes plugins`), але жоден не завантажується, доки ти явно не ввімкнеш їх:

```bash
hermes plugins enable disk-cleanup
```

Або через `~/.hermes/config.yaml`:

```yaml
plugins:
  enabled:
    - disk-cleanup
```

Це той самий механізм, який використовується для плагінів, встановлених користувачем. Пакетні плагіни ніколи не вмикаються автоматично — ні при свіжій інсталяції, ні для існуючих користувачів, які оновлюються до новішої версії Hermes. Ти завжди явно їх вмикаєш.

Щоб вимкнути пакетний плагін знову:

```bash
hermes plugins disable disk-cleanup
# or: remove it from plugins.enabled in config.yaml
```
## Поточна поставка

Репозиторій постачається з цими вбудованими плагінами у каталозі `plugins/`. Усі вони опціональні — увімкни їх за допомогою `hermes plugins enable <name>`.

| Плагін | Тип | Призначення |
|---|---|---|
| `disk-cleanup` | hooks + slash command | Автоматичне відстеження тимчасових файлів і їх очищення в кінці сесії |
| `security-guidance` | hooks | Пошук шаблонів небезпечного коду у `write_file`/`patch` та додавання попередження про безпеку (або блокування) — 25 правил (форк Apache‑2.0 патернів `claude-plugins-official` від Anthropic) |
| `observability/langfuse` | hooks | Трасування ходів / викликів LLM / інструментів у [Langfuse](https://langfuse.com) |
| `spotify` | backend (7 tools) | Нативне відтворення Spotify, черга, пошук, плейлисти, альбоми, бібліотека |
| `google_meet` | standalone | Приєднання до викликів Meet, транскрипція живих субтитрів, опціональний реальний двосторонній аудіо |
| `image_gen/openai` | image backend | Бекенд генерації зображень OpenAI `gpt-image-2` (альтернатива FAL) |
| `image_gen/openai-codex` | image backend | Генерація зображень OpenAI через OAuth Codex |
| `image_gen/xai` | image backend | Бекенд xAI `grok-2-image` |
| `hermes-achievements` | dashboard tab | Колекційні бейджі у стилі Steam, створені на основі історії твоїх реальних сесій Hermes |
| `kanban/dashboard` | dashboard tab | UI дошки Kanban для диспетчера багатьох агентів — завдання, коментарі, розгалуження, перемикання дошок. Дивись [Kanban Multi-Agent](./kanban.md). |

Постачальники пам'яті (`plugins/memory/*`) та контекстні движки (`plugins/context_engine/*`) перераховані окремо у розділі [Memory Providers](./memory-providers.md) — вони керуються через `hermes memory` та `hermes plugins` відповідно. Повний опис кожного з двох довготривалих плагінів на основі hooks наведено нижче.
### disk-cleanup

Автоматично відстежує та видаляє епемерні файли, створені під час сесій — тестові скрипти, тимчасові виводи, журнали cron, застарілі профілі Chrome — без потреби, щоб агент пам’ятав викликати інструмент.

**Як це працює:**

| Hook | Поведінка |
|---|---|
| `post_tool_call` | Коли `write_file` / `terminal` / `patch` створює файл, що відповідає `test_*`, `tmp_*` або `*.test.*` всередині `HERMES_HOME` або `/tmp/hermes-*`, він тихо відстежується як `test` / `temp` / `cron-output`. |
| `on_session_end` | Якщо під час ходу були автоматично відстежені тестові файли, виконується безпечне `quick` очищення та записується однорядковий підсумок. Інакше залишається тихим. |

**Правила видалення:**

| Category | Threshold | Confirmation |
|---|---|---|
| `test` | кожен кінець сесії | Ніколи |
| `temp` | >7 днів від моменту відстеження | Ніколи |
| `cron-output` | >14 днів від моменту відстеження | Ніколи |
| порожні каталоги під `HERMES_HOME` | завжди | Ніколи |
| `research` | >30 днів, поза 10 найновішими | Завжди (лише глибоке) |
| `chrome-profile` | >14 днів від моменту відстеження | Завжди (лише глибоке) |
| файли >500 MB | ніколи автоматично | Завжди (лише глибоке) |

**Slash‑команда** — `/disk-cleanup` доступна як у CLI, так і в сесіях шлюзу:

```
/disk-cleanup status                     # breakdown + top-10 largest
/disk-cleanup dry-run                    # preview without deleting
/disk-cleanup quick                      # run safe cleanup now
/disk-cleanup deep                       # quick + list items needing confirmation
/disk-cleanup track <path> <category>    # manual tracking
/disk-cleanup forget <path>              # stop tracking (does not delete)
```

**Стан** — все зберігається в `$HERMES_HOME/disk-cleanup/`:

| File | Contents |
|---|---|
| `tracked.json` | Відстежені шляхи з категорією, розміром та міткою часу |
| `tracked.json.bak` | Резервна копія атомарного запису вищезазначеного |
| `cleanup.log` | Додаток‑лог аудиту кожного відстеження / пропуску / відхилення / видалення |

**Безпека** — очистка торкається лише шляхів під `HERMES_HOME` або `/tmp/hermes-*`. Windows‑монтовані (`/mnt/c/...`) відхиляються. Відомі верхньорівневі каталоги стану (`logs/`, `memories/`, `sessions/`, `cron/`, `cache/`, `skills/`, `plugins/`, `disk-cleanup/` самі) ніколи не видаляються, навіть коли порожні — нова інсталяція не буде знищена після першого завершення сесії.

**Увімкнення:** `hermes plugins enable disk-cleanup` (або постав галочку в `hermes plugins`).

**Вимкнення:** `hermes plugins disable disk-cleanup`.
### security-guidance

Швидкі попередження про безпеку під час запису файлів за допомогою шаблонного співставлення. Коли виклики агента `write_file` / `patch` / `skill_manage` містять вміст, що відповідає відомому небезпечному шаблону коду — `pickle.load`, `yaml.load` без `SafeLoader`, `eval(`, `os.system`, `subprocess(...,  shell=True)`, JS `child_process.exec`, React `dangerouslySetInnerHTML`, raw `.innerHTML =` / `.outerHTML =` / `document.write`, Node `crypto.createCipher`, AES ECB mode, TLS verification disabled, XXE‑вразливі `xml.etree` / `minidom` парсери, `<script src="//..." >` без SRI, `torch.load` без `weights_only=True`, ін’єкція `${{ github.event.* }}` у GitHub Actions — плагін додає блок `⚠️ Security guidance` до результату інструмента.

Файл все одно записується. Модель читає попередження у наступному повідомленні інструмента і може або виправити код, або задокументувати, чому конструкція безпечна в цьому контексті. Шаблонне співставлення має ненульовий рівень хибнопозитивних спрацьовувань, тому за замовчуванням використовується попередження (не блокування).

**Покриття:** 25 правил загалом, що охоплюють небезпечну десеріалізацію, ін’єкцію команд, XSS‑сінки, криптографічні пастки, XXE, постачальницькі (SRI) та ін’єкції у CI/CD‑конвеєри. Дані шаблонів — дослівний Apache‑2.0 форк [Anthropic's `claude-plugins-official`](https://github.com/anthropics/claude-plugins-official/tree/main/plugins/security-guidance/hooks) — дивись файли плагіна `LICENSE` та `NOTICE` для атрибуції.

**Режими:**

| Env var | Effect |
|---|---|
| (unset) | **режим попередження** (за замовчуванням) — файл записується, попередження додається до результату |
| `SECURITY_GUIDANCE_BLOCK=1` | **режим блокування** — запис відхиляється, попередження повертається як причина блокування |
| `SECURITY_GUIDANCE_DISABLE=1` | вимикач — плагін завантажується, але нічого не робить |

**Увімкнення:** `hermes plugins enable security-guidance` (або постав галочку в `hermes plugins`).

**Вимкнення:** `hermes plugins disable security-guidance`.

**Що ще не робить (поки):** оригінальний плагін Anthropic має ще два рівні — LLM‑перегляд diff на кожному ході агента, що змінював файли, та агентський перегляд під час коміту, що відстежує потік даних між файлами. Жоден з них не портовано. Агент вже може запускати ці перегляди за запитом через `delegate_task`.
### observability/langfuse

Трасуйте повороти Hermes, виклики LLM та інвокації інструментів у [Langfuse](https://langfuse.com) — відкриту платформу спостереження за LLM. Один span на кожен поворот, одне генерування на кожний API‑виклик, одне спостереження інструмента на кожний виклик інструмента. Підсумкові дані, підрахунок токенів за типом та оцінки вартості беруться з канонічних чисел `agent.usage_pricing` Hermes, тому панель Langfuse бачить той самий розподіл (input / output / `cache_read_input_tokens` / `cache_creation_input_tokens` / `reasoning_tokens`), який з’являється у `hermes logs`.

Плагін працює у режимі **fail‑open**: відсутність SDK, відсутність облікових даних або транзиторна помилка Langfuse — усе це перетворюється у тихий **no‑op** у хуці. Цикл агента ніколи не постраждає.

**Налаштування:**

```bash
pip install langfuse
hermes plugins enable observability/langfuse
```

Або позначте галочку в інтерактивному UI `hermes plugins`. Потім розмістіть облікові дані у `~/.hermes/.env`:

```bash
HERMES_LANGFUSE_PUBLIC_KEY=pk-lf-...
HERMES_LANGFUSE_SECRET_KEY=sk-lf-...
HERMES_LANGFUSE_BASE_URL=https://cloud.langfuse.com   # or your self-hosted URL
```

**Як це працює:**

| Hook | Поведінка |
|---|---|
| `pre_api_request` / `pre_llm_call` | Відкриває (або повторно використовує) кореневий span “Hermes turn” для поточного повороту. Запускає дочірнє спостереження `generation` для цього API‑виклику з серіалізованими останніми повідомленнями як вхідними даними. |
| `post_api_request` / `post_llm_call` | Закриває генерацію, додає `usage_details`, `cost_details`, `finish_reason`, вихід асистента + виклики інструментів. Якщо викликів інструментів немає і вміст не порожній, закриває поворот. |
| `pre_tool_call` | Запускає дочірнє спостереження `tool` з очищеними `args`. |
| `post_tool_call` | Закриває спостереження інструмента з очищеним `result`. Пакети `read_file` підсумовуються (head + tail + кількість пропущених рядків), щоб великий файл залишався в межах `HERMES_LANGFUSE_MAX_CHARS`. |

Ключі групування сесій беруться з ідентифікатора сесії Hermes (або ID завдання для під‑агентів) через `langfuse.propagate_attributes`, тому все в одній сесії `hermes chat` живе під однією сесією Langfuse.

**Перевірка:**

```bash
hermes plugins list                 # observability/langfuse should show "enabled"
hermes chat -q "hello"              # check the Langfuse UI for a "Hermes turn" trace
```

**Опційне налаштування** (у `.env`):

| Variable | Default | Призначення |
|---|---|---|
| `HERMES_LANGFUSE_ENV` | — | Тег середовища для трас (`production`, `staging`, …) |
| `HERMES_LANGFUSE_RELEASE` | — | Тег релізу/версії |
| `HERMES_LANGFUSE_SAMPLE_RATE` | `1.0` | Швидкість вибірки, передана SDK (0.0–1.0) |
| `HERMES_LANGFUSE_MAX_CHARS` | `12000` | Обрізка кожного поля для вмісту повідомлення / аргументів інструмента / результатів інструмента |
| `HERMES_LANGFUSE_DEBUG` | `false` | Детальне логування плагіна у `agent.log` |

Приймаються як змінні середовища з префіксом Hermes, так і стандартні SDK‑змінні (`LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`, `LANGFUSE_BASE_URL`) — пріоритет має префікс Hermes, якщо обидва встановлені.

**Продуктивність:** клієнт Langfuse кешується після першого виклику хука. Якщо облікові дані або SDK відсутні, це рішення також кешується — подальші хуки швидко повертаються без повторної перевірки змінних середовища чи перезавантаження конфігурації.

**Вимкнення:** `hermes plugins disable observability/langfuse`. Модуль плагіна все ще виявляється, але код модуля не виконується, доки його знову не ввімкнути.
### google_meet

Дозволяє агенту **приєднуватися, транскрибувати та брати участь у викликах Google Meet** — робити нотатки під час зустрічі, підсумовувати обговорення після, слідкувати за конкретними пунктами та (за бажанням) озвучувати відповіді у виклик за допомогою TTS.

**Що додає:**

- Безголовий віртуальний учасник, який приєднується до URL Meet за допомогою автоматизації браузера
- Живе транскрибування аудіо зустрічі через налаштований STT‑провайдер
- Набір інструментів `meet_summarize` / `meet_speak` / `meet_followup`, які агент викликає для дій на основі почутого
- Артефакти після зустрічі (транскрипт, нотатки з атрибуцією спікерів, список дій), збережені у `~/.hermes/cache/google_meet/<meeting_id>/`

**Налаштування:**

```bash
hermes plugins enable google_meet
# Prompts you to sign in via the plugin's OAuth flow on first use —
# needs a Google account with Meet access. Host approval may be required
# if the meeting enforces "only invited participants can join".
```

Використання в чаті:

> "Join meet.google.com/abc-defg-hij and take notes. After the call, send me a summary with action items."

Агент ініціює приєднання до зустрічі, передає транскрипцію у свій контекст під час зустрічі та створює структурований підсумок, коли зустріч закінчується (або коли ти попросиш його зупинитися).

**Коли використовувати:** регулярні стендапи, коли потрібен бот для транскрибування + підсумовування для асинхронних учасників; інтерв’ю у стилі deposition, коли потрібні структуровані нотатки; будь‑який випадок, коли інакше доводилося б користуватися Fireflies / Otter / Grain. Якщо не хочеш, щоб ШІ слухав — не вмикай.

**Вимкнення:** `hermes plugins disable google_meet`. Усі кешовані транскрипти та записи залишаються в `~/.hermes/cache/google_meet/`, доки їх не видалиш.
### hermes-achievements

Додає **вкладку досягнень у стилі Steam на панель керування** — 60+ колекційних, багаторівневих значків, згенерованих з реальної історії твоїх **session** Hermes. Досягнення інструментального ланцюга, шаблони налагодження, streak‑и кодування у вібраціях, використання **skill/memory**, різноманітність моделей/провайдерів, особливості способу життя (вихідні та нічні сесії). Спочатку створений [@PCinkusz](https://github.com/PCinkusz) як зовнішній плагін; інтегровано в дерево, щоб залишатися синхронізованим зі змінами функцій Hermes.

**Як це працює:**

- Сканує всю історію сесій у `~/.hermes/state.db` на бекенді панелі керування
- Статистика по **session** кешується за відбитком `(started_at, last_active)`, тому лише нові або змінені сесії повторно аналізуються під час наступних сканувань
- Перший скан запускається у фоні — панель керування ніколи не блокується в очікуванні, навіть при базах даних з тисячами сесій
- Стан розблокування зберігається у `$HERMES_HOME/plugins/hermes-achievements/state.json`

**Прогрес рівнів:** Copper → Silver → Gold → Diamond → Olympian. Кожна картка містить розділ «What counts», у якому перераховано точну метрику, що відстежується.

**Стани досягнень:**

| Стан | Опис |
|---|---|
| Unlocked | Принаймні один рівень досягнуто |
| Discovered | Відоме досягнення, прогрес видно, ще не отримано |
| Secret | Приховано, доки Hermes не виявить перший пов’язаний сигнал у твоїй історії |

**API** — маршрути монтуються під `/api/plugins/hermes-achievements/`:

| Endpoint | Призначення |
|---|---|
| `GET /achievements` | Повний каталог зі станом розблокування кожного значка (повертає заповнювач, поки виконується перший холодний скан) |
| `GET /scan-status` | Стан фонового сканера: `idle` / `running` / `failed`, тривалість останнього запуску, кількість запусків |
| `GET /recent-unlocks` | Двадцять останніх розблокованих значків, найновіші першими |
| `GET /sessions/{id}/badges` | Значки, здобуті переважно в одній конкретній сесії |
| `POST /rescan` | Ручний синхронний рескан (блокує; використовується, коли користувач натискає кнопку рескану) |
| `POST /reset-state` | Очистити історію розблокувань і кешований знімок |

**Файли стану** — розташовані у `$HERMES_HOME/plugins/hermes-achievements/`:

| Файл | Вміст |
|---|---|
| `state.json` | Історія розблокувань: які значки ти отримав і коли. Стабільна між оновленнями Hermes. |
| `scan_snapshot.json` | Пакет даних останнього завершеного сканування (подається одразу при завантаженні панелі) |
| `scan_checkpoint.json` | Кеш статистики по сесіях, індексований відбитком (дозволяє швидкі «теплі» рескани) |

**Примітки щодо продуктивності:**

- Холодний скан на ~8 000 сесій займає кілька хвилин. Він запускається у фоні під час першого запиту до панелі; UI бачить заповнювач і опитує `/scan-status`.
- **Інкрементальні результати під час холодного сканування** — сканер публікує частковий знімок кожні ~250 сесій, тож кожне оновлення панелі показує більше розблокованих значків у процесі сканування. Ніякого довгого очікування нулів.
- Теплий рескан повторно використовує статистику по сесіях для кожної сесії, чий відбиток `started_at` + `last_active` збігається з контрольною точкою — завершується за секунди навіть на великих історіях.
- TTL знімка в пам’яті — 120 s; застарілі запити одразу отримують старий знімок і запускають фонове оновлення. Ти ніколи не чекаєш на спінер лише через закінчення TTL.

**Увімкнення:** Нічого вмикати не потрібно — `hermes-achievements` є лише плагіном панелі керування (без хуків життєвого циклу, без інструментів, видимих моделлю). Він автоматично реєструється як вкладка в `hermes dashboard` під час першого запуску. Конфіг `plugins.enabled` контролює лише плагіни життєвого циклу/інструментів; плагіни панелі виявляються виключно через їх `dashboard/manifest.json`.

**Вимкнення:** Видали або перейменуй `plugins/hermes-achievements/dashboard/manifest.json`, або перевизначи його користувацьким плагіном з тим же ім’ям у `~/.hermes/plugins/hermes-achievements/`, який не постачає панель. Файли стану плагіна у `$HERMES_HOME/plugins/hermes-achievements/` залишаються — переустановка зберігає твою історію розблокувань.
## Додавання вбудованого плагіна

Вбудовані плагіни пишуться точно так само, як будь‑який інший Hermes плагін — дивись [Build a Hermes Plugin](/guides/build-a-hermes-plugin). Єдині відмінності:

- Каталог розташовується у `<repo>/plugins/<name>/` замість `~/.hermes/plugins/<name>/`
- Джерело маніфесту виводиться як `bundled` у `hermes plugins list`
- Користувацькі плагіни з тим самим ім’ям переважають вбудовану версію

Плагін є хорошим кандидатом для вбудовування, коли:

- У нього немає необов’язкових залежностей (або вони вже включені у `pip install .[all]` deps)
- Поведінка корисна більшості користувачів і реалізується за принципом «відмовитися, а не підключати» (opt‑out)
- Логіка прив’язана до хуків життєвого циклу, які агент інакше довелося б пам’ятати викликати
- Він доповнює базову можливість без розширення видимої інструментальної поверхні моделі

Контр‑приклади — речі, які мають залишатися плагінами, що встановлюються користувачем, а не вбудованими: інтеграції сторонніх сервісів з API‑ключами, вузькоспеціалізовані робочі процеси, великі дерева залежностей, будь‑що, що суттєво змінює поведінку агента за замовчуванням.
