# Kanban — Спільна робота профілів багатагентних агентів

> **Хочеш покроковий огляд?** Прочитай [підручник Kanban](./kanban-tutorial) — чотири історії користувачів (solo dev, fleet farming, role pipeline with retry, circuit breaker) з скріншотами панелі для кожної. Ця сторінка — довідковий матеріал; підручник — розповідна частина.

Hermes Kanban — це надійна дошка завдань, спільна для всіх твоїх профілів Hermes, яка дозволяє кільком іменованим агентам співпрацювати над роботою без крихких підагентських роїв у процесі. Кожне завдання — це рядок у `~/.hermes/kanban.db`; кожна передача — це рядок, який може читати і записувати будь‑хто; кожен worker — це окремий процес ОС зі своєю ідентичністю.

### Два інтерфейси: модель працює через інструменти, ти — через CLI

Дошка має два входи, обидва підключені до одного `~/.hermes/kanban.db`:

- **Агенти керують дошкою через спеціальний набір інструментів `kanban_*`** — `kanban_show`, `kanban_list`, `kanban_complete`, `kanban_block`, `kanban_heartbeat`, `kanban_comment`, `kanban_create`, `kanban_link`, `kanban_unblock`. Диспетчер створює кожного worker‑а з цими інструментами вже у його схемі; профілі оркестраторів також можуть явно ввімкнути набір інструментів `kanban`. Модель читає і маршрутизує завдання, викликаючи інструменти безпосередньо, *не* запускаючи `hermes kanban`. Дивись [How workers interact with the board](#how-workers-interact-with-the-board) нижче.
- **Ти (і скрипти, і cron) керуєш дошкою через `hermes kanban …`** у CLI, `/kanban …` як slash‑команду або через панель. Це призначено для людей і автоматизації — місць, де немає моделі виклику інструментів за кадром.

Обидва інтерфейси працюють через один шар `kanban_db`, тому читання бачать узгоджений стан, а записи не можуть розбігатися. Решта цієї сторінки показує приклади CLI, бо їх легко скопіювати‑вставити, проте кожна команда CLI має еквівалент виклику інструмента, який використовує модель.

Це форма, яка охоплює навантаження, які `delegate_task` не може виконати:

- **Тріаж досліджень** — паралельні дослідники + аналітик + письменник, людина в циклі.
- **Заплановані операції** — регулярні щоденні брифінги, що створюють журнал протягом тижнів.
- **Цифрові двійники** — постійні іменовані помічники (`inbox-triage`, `ops-review`), які накопичують пам'ять з часом.
- **Інженерні конвеєри** — розбивка → паралельна реалізація у робочих дереві → рев’ю → ітерація → PR.
- **Робота флоту** — один спеціаліст, що керує N об’єктами (50 соціальних акаунтів, 12 моніторингових сервісів).

Для повного обґрунтування дизайну, порівняльного аналізу проти Cline Kanban / Paperclip / NanoClaw / Google Gemini Enterprise та восьми канонічних шаблонів співпраці, дивись `docs/hermes-kanban-v1-spec.pdf` у репозиторії.
## Kanban vs. `delegate_task`

Вони схожі, але це не один і той самий примітив.

| | `delegate_task` | Kanban |
|---|---|---|
| Shape | RPC‑виклик (fork → join) | Durable message queue + state machine |
| Parent | Блокує, доки дочірній агент не поверне результат | Fire‑and‑forget після `create` |
| Child identity | Анонімний субагент | Іменований профіль з постійною пам’яттю |
| Resumability | Немає — failed = failed | Block → unblock → re‑run; crash → reclaim |
| Human in the loop | Не підтримується | Коментар / розблокування в будь‑який момент |
| Agents per task | Один виклик = один субагент | N агентів протягом життєвого циклу завдання (retry, review, follow‑up) |
| Audit trail | Втрачається при стисканні контексту | Durable rows in SQLite назавжди |
| Coordination | Ієрархічна (caller → callee) | Peer‑to‑peer — будь‑який профіль читає/записує будь‑яке завдання |

**Однорядкове розрізнення:** `delegate_task` — це функціональний виклик; Kanban — це робоча черга, де кожна передача — це рядок, який будь‑який профіль (або людина) може бачити та редагувати.

**Використовуй `delegate_task`, коли** батьківському агенту потрібна коротка відповідь з міркуванням перед продовженням, без залучення людей, а результат повертається у контекст батька.

**Використовуй Kanban, коли** робота перетинає межі агентів, має переживати перезапуски, може потребувати вводу людини, може бути підхоплена іншим ролем або має бути доступна для перегляду після завершення.

Вони співіснують: kanban‑робітник може викликати `delegate_task` всередині свого виконання.
## Core concepts

- **Board** — окрема черга завдань зі своєю SQLite‑базою, каталогом `workspaces` та циклом диспетчера. Одна інсталяція може мати багато `boards` (наприклад, по одній на проєкт, репозиторій чи домен); дивись [Boards (multi-project)](#boards-multi-project) нижче. Користувачі однопроєктних установок залишаються на `default` `board` і ніколи не бачать слово «board» поза цим розділом документації.
- **Task** — рядок з назвою, необов’язковим тілом, одним виконавцем (ім’я `profile`), статусом (`triage | todo | ready | running | blocked | done | archived`), необов’язковим простором імен `tenant`, необов’язковим ключем ідемпотентності (дедуплікація для повторних автоматизацій).
- **Link** — рядок `task_links`, що фіксує залежність *батько → дитина*. Диспетчер підвищує `todo → ready`, коли всі батьки мають статус `done`.
- **Comment** — протокол між агентами. Агенти та люди додають коментарі; коли працівник (повторно) запускається, він читає всю гілку коментарів як частину свого контексту.
- **Workspace** — каталог, у якому працює працівник. Три типи:
  - `scratch` (за замовчуванням) — новий тимчасовий каталог під `~/.hermes/kanban/workspaces/<id>/` (або `~/.hermes/kanban/boards/<slug>/workspaces/<id>/` на недефолтних `boards`). **Видаляється після завершення завдання** — `scratch` епемерний за задумом, тому каталог стирається в момент, коли працівник (або `hermes kanban complete <id>`) позначає завдання як виконане. Якщо потрібно зберегти вихідні дані працівника, використай `worktree:` або `dir:<path>` замість. При першому створенні `scratch`‑workspace в інсталяції диспетчер записує попередження і генерує подію `tip_scratch_workspace` на завданні (видима через `hermes kanban show <id>`).
  - `dir:<path>` — існуючий спільний каталог (сховище Obsidian, каталог поштових операцій, папка облікового запису). **Повинен бути абсолютним шляхом.** Відносні шляхи типу `dir:../tenants/foo/` відхиляються під час диспетчиризації, бо вони розв’язувались би від поточної робочої директорії диспетчера, що є неоднозначним і вектором втечі. Шлях довірений — це твій комп’ютер, твоя файлова система, працівник працює під твоїм uid. Це модель загрози «trusted‑local‑user»; kanban за задумом однопоточний. **Зберігається після завершення.**
  - `worktree` — git‑worktree у `.worktrees/<id>/` для завдань кодування. Використовуй `worktree:<path>`, щоб зафіксувати точний шлях. На боці працівника `git worktree add` створює його, використовуючи `--branch`, якщо вказано. **Зберігається після завершення.**
- **Dispatcher** — довгоживущий цикл, який кожні N секунд (за замовчуванням 60) звільняє застарілі заявки, відновлює впавших працівників (PID зник, але TTL ще не вичерпано), підвищує готові завдання, атомарно захоплює їх і запускає призначені `profiles`. За замовчуванням працює **всередині шлюзу** (`kanban.dispatch_in_gateway: true`). Один диспетчер обробляє всі `boards` за один тик; працівники запускаються з зафіксованою змінною `HERMES_KANBAN_BOARD`, щоб вони не бачили інші `boards`. Після `kanban.failure_limit` послідовних невдач запуску того ж завдання (за замовчуванням 2) диспетчер автоматично блокує його, вказуючи останню помилку як причину — запобігає надмірному створенню (thrashing) завдань, у яких відсутній `profile`, неможливо змонтувати `workspace` тощо.
- **Tenant** — необов’язковий рядковий простір імен *всередині* `board`. Одна спеціалізована флотилія може обслуговувати кілька бізнесів (`--tenant business-a`) з ізоляцією даних за шляхом `workspace` та префіксом ключа пам’яті. `Tenant` — м’який фільтр; `board` — жорстка межа ізоляції.
## Boards (multi-project)

Boards дозволяють розділяти несумісні потоки роботи — по одному на проєкт, репозиторій або домен — у ізольовані черги. При новій інсталяції існує лише одна дошка під назвою `default` (DB у `~/.hermes/kanban.db` для зворотної сумісності). Користувачі, які працюють лише з одним потоком, ніколи не повинні знати про дошки; ця функція включається за бажанням.

Ізоляція на рівні дошки є абсолютною:

- Окрема SQLite‑DB на дошку (`~/.hermes/kanban/boards/<slug>/kanban.db`).
- Окремі каталоги `workspaces/` і `logs/`.
- Робітники, запущені для завдання, бачать **лише** завдання своєї дошки — диспетчер встановлює `HERMES_KANBAN_BOARD` у дочірньому середовищі, і кожен інструмент `kanban_*`, до якого має доступ робітник, читає його.
- Зв’язування завдань між дошками заборонено (спрощує схему; якщо дійсно потрібні крос‑проєктні посилання, використай вільний текст і шукай їх за ідентифікатором вручну).

### Managing boards from the CLI

```bash
# See what's on disk. Fresh installs show only "default".
hermes kanban boards list

# Create a new board.
hermes kanban boards create atm10-server \
    --name "ATM10 Server" \
    --description "Minecraft modded server ops" \
    --icon 🎮 \
    --switch                   # optional: make it the active board

# Operate on a specific board without switching.
hermes kanban --board atm10-server list
hermes kanban --board atm10-server create "Restart ATM server" --assignee ops

# Change which board is "current" for subsequent calls.
hermes kanban boards switch atm10-server
hermes kanban boards show             # who's active right now?

# Rename the display name (the slug is immutable — it's the directory name).
hermes kanban boards rename atm10-server "ATM10 (Prod)"

# Archive (default) — moves the board's dir to boards/_archived/<slug>-<ts>/.
# Recoverable by moving the dir back.
hermes kanban boards rm atm10-server

# Hard delete — `rm -rf` the board dir. No recovery.
hermes kanban boards rm atm10-server --delete
```

Порядок визначення дошки (спочатку найвища пріоритетність):

1. Явний `--board <slug>` у виклику CLI.
2. Змінна середовища `HERMES_KANBAN_BOARD` (встановлюється диспетчером під час створення робітника, тому робітники не бачать інші дошки).
3. `~/.hermes/kanban/current` — slug, збережений командою `hermes kanban boards switch`.
4. `default`.

Slug‑и перевіряються: малі літери, цифри, дефіси та підкреслення, 1‑64 символи, мають починатися з літери або цифри. Введення у верхньому регістрі автоматично переводиться у нижній. Будь‑що інше (слеші, пробіли, крапки, `..`) відхиляється на рівні CLI, тому трюки типу path‑traversal не можуть створити назву дошки.

### Managing boards from the dashboard

`hermes dashboard` → вкладка **Kanban** показує перемикач дошок у верхній частині, як тільки існує більше однієї дошки (або будь‑яка дошка має завдання). Користувачі з однією дошкою бачать лише маленьку кнопку `+ New board`; перемикач приховано, доки це не потрібно.

- **Board dropdown** — вибір активної дошки. Твій вибір зберігається у `localStorage` браузера, тому він зберігається між перезавантаженнями без зміни вказівника `current` у CLI, який може бути відкритий у термінальній сесії.
- **+ New board** — відкриває модальне вікно, у якому потрібно ввести slug, відображувану назву, опис і іконку. Є опція автоматично перемкнутись на нову дошку.
- **Archive** — показується лише для дошок, відмінних від `default`. Після підтвердження переміщує каталог дошки до `boards/_archived/`.

Усі API‑ендпоінти дашборду приймають параметр `?board=<slug>` для обмеження за дошкою. WebSocket подій прив’язаний до дошки під час підключення; перемикання в UI відкриває новий WS для нової дошки.
## Швидкий старт

Команди нижче — це **ти** (людина), яка налаштовує дошку та створює завдання. Після того, як завдання призначено, диспетчер створює профіль‑виконавця, і далі **модель керує завданням через виклики інструменту `kanban_*`, а не через CLI‑команди** — дивись [Як працівники взаємодіють з дошкою](#how-workers-interact-with-the-board).

```bash
# 1. Create the board (you)
hermes kanban init

# 2. Start the gateway (hosts the embedded dispatcher)
hermes gateway start

# 3. Create a task (you — or an orchestrator agent via kanban_create)
hermes kanban create "research AI funding landscape" --assignee researcher

# 4. Watch activity live (you)
hermes kanban watch

# 5. See the board (you)
hermes kanban list
hermes kanban stats
```

Коли диспетчер підхоплює `t_abcd` і створює профіль `researcher`, перше, що робить модель працівника, — це виклик `kanban_show()`, щоб прочитати його завдання. Вона не виконує `hermes kanban show t_abcd`.

### Диспетчер, вбудований у шлюз (за замовчуванням)

Диспетчер працює всередині процесу шлюзу. Нічого встановлювати, жодних окремих сервісів не потрібно — якщо шлюз запущений, готові завдання підхоплюються на наступному тикі (за замовчуванням 60 сек).

```yaml
# config.yaml
kanban:
  dispatch_in_gateway: true        # default
  dispatch_interval_seconds: 60    # default
```

Перевизначте прапорець конфігурації під час виконання через `HERMES_KANBAN_DISPATCH_IN_GATEWAY=0` для налагодження. Стандартний нагляд за шлюзом застосовується: запустіть `hermes gateway start` безпосередньо або підключіть шлюз як юніт systemd користувача (див. документацію шлюзу). Без запущеного шлюзу завдання зі статусом `ready` залишаються, доки шлюз не з’явиться — `hermes kanban create` попереджає про це під час створення.

Запуск `hermes kanban daemon` як окремого процесу **застарілий**; використовуйте шлюз. Якщо дійсно неможливо запустити шлюз (політика безголового хоста забороняє довгоживучі сервіси тощо), «escape hatch» `--force` зберігає старий автономний демон живим протягом одного циклу випуску, але запуск одночасно вбудованого у шлюз диспетчера **і** автономного демона проти того ж `kanban.db` викликає конфлікти претензій і не підтримується.

### Ідемпотентне створення (для автоматизації / вебхуків)

```bash
# First call creates the task. Any subsequent call with the same key
# returns the existing task id instead of duplicating.
hermes kanban create "nightly ops review" \
    --assignee ops \
    --idempotency-key "nightly-ops-$(date -u +%Y-%m-%d)" \
    --json
```

### Масові CLI‑дієслова

Усі дієслова життєвого циклу приймають кілька ідентифікаторів, тому можна очистити пакет за одну команду:

```bash
hermes kanban complete t_abc t_def t_hij --result "batch wrap"
hermes kanban archive  t_abc t_def t_hij
hermes kanban unblock  t_abc t_def
hermes kanban block    t_abc "need input" --ids t_def t_hij
```
## Як працівники взаємодіють з дошкою

**Працівники не викликають `hermes kanban`.** Коли диспетчер створює працівника, він встановлює `HERMES_KANBAN_TASK=t_abcd` у середовищі дочірнього процесу, і ця змінна середовища активує спеціальний **kanban toolset** у схемі моделі. Той самий toolset також доступний профілям оркестраторів, які вмикають `kanban` у своїй конфігурації toolsets. Ці інструменти читають і змінюють дошку безпосередньо через шар Python `kanban_db`, так само, як це робить CLI. Запущений працівник викликає їх, як будь‑який інший інструмент; він ніколи не бачить і не потребує CLI `hermes kanban`.

| Tool | Purpose | Required params |
|---|---|---|
| `kanban_show` | Читати поточне завдання (заголовок, тіло, попередні спроби, батьківські передачі, коментарі, повністю попередньо сформований `worker_context`). За замовчуванням — ідентифікатор завдання з середовища. | — |
| `kanban_list` | Перелік підсумків завдань з фільтрами за `assignee`, `status`, `tenant`, видимістю архіву та лімітом. Призначено для оркестраторів, які шукають роботу на дошці. | — |
| `kanban_complete` | Завершити з структурованою передачею `summary` + `metadata`. | принаймні один з `summary` / `result` |
| `kanban_block` | Ескалувати для людського вводу з `reason`. | `reason` |
| `kanban_heartbeat` | Сигналізувати живість під час довгих операцій. Чистий побічний ефект. | — |
| `kanban_comment` | Додати стійку нотатку до потоку завдання. | `task_id`, `body` |
| `kanban_create` | (Оркестратори) розподілити на дочірні завдання з `assignee`, необов’язковими `parents`, `skills` тощо. | `title`, `assignee` |
| `kanban_link` | (Оркестратори) додати залежність `parent_id → child_id` після створення. | `parent_id`, `child_id` |
| `kanban_unblock` | (Оркестратори) перемістити заблоковане завдання назад у `ready`. | `task_id` |

Типовий хід працівника виглядає так:

```
# Model's tool calls, in order:
kanban_show()                                     # no args — uses HERMES_KANBAN_TASK
# (model reads the returned worker_context, does the work via terminal/file tools)
kanban_heartbeat(note="halfway through — 4 of 8 files transformed")
# (more work)
kanban_complete(
    summary="migrated limiter.py to token-bucket; added 14 tests, all pass",
    metadata={"changed_files": ["limiter.py", "tests/test_limiter.py"], "tests_run": 14},
)
```

**Оркестратор**‑працівник розподіляє завдання інакше:

```
kanban_show()
kanban_create(
    title="research ICP funding 2024-2026",
    assignee="researcher-a",
    body="focus on seed + series A, North America, AI-adjacent",
)
# → returns {"task_id": "t_r1", ...}
kanban_create(title="research ICP funding — EU angle", assignee="researcher-b", body="…")
# → returns {"task_id": "t_r2", ...}
kanban_create(
    title="synthesize findings into launch brief",
    assignee="writer",
    parents=["t_r1", "t_r2"],                     # promotes to ready when both complete
    body="one-pager, 300 words, neutral tone",
)
kanban_complete(summary="decomposed into 2 research tasks + 1 writer; linked dependencies")
```

Інструменти «(Оркестратори)» — `kanban_list`, `kanban_create`, `kanban_link`, `kanban_unblock` та `kanban_comment` на зовнішніх завданнях — доступні через той самий toolset; конвенція (забезпечена навичкою `kanban-orchestrator`) полягає в тому, що профілі працівників не розподіляють і не маршрутизують несумісну роботу, а профілі оркестраторів не виконують реалізаційну роботу. Працівники, створені диспетчером, залишаються прив’язаними до завдання для деструктивних операцій життєвого циклу і не можуть змінювати несуміжні завдання.
### Чому інструменти замість виклику `hermes kanban`

Три причини:

1. **Портативність бекенду.** Робітники, чиї термінальні інструменти вказують на віддалений бекенд (Docker / Modal / Singularity / SSH), запускали б `hermes kanban complete` *всередині* контейнера, де `hermes` не встановлено і `~/.hermes/kanban.db` не змонтовано. Інструменти kanban працюють у власному процесі агента на Python і завжди мають доступ до `~/.hermes/kanban.db` незалежно від бекенду терміналу.
2. **Відсутність крихкості при quoting у shell.** Передача `--metadata '{"files": [...]}'` через shlex + argparse є прихованою пасткою. Структуровані аргументи інструменту обходять це повністю.
3. **Кращі помилки.** Результати інструменту — це структурований JSON, який модель може аналізувати, а не рядки stderr, які треба розбирати.

**Нульовий вплив схеми на звичайні сесії.** Звичайна `hermes chat` сесія не містить інструментів `kanban_*` у своїй схемі, якщо активний профіль явно не вмикає набір інструментів `kanban` для роботи оркестратора. Робітники, створені диспетчером, отримують інструменти у межах завдання, бо встановлено `HERMES_KANBAN_TASK`; профілі оркестраторів отримують ширший набір через конфігурацію. Немає зайвих інструментів для користувачів, які ніколи не користуються kanban.

Навички `kanban-worker` і `kanban-orchestrator` навчають модель, який інструмент викликати, коли і в якому порядку.
### Рекомендовані докази передачі

`kanban_complete(summary=..., metadata={...})` навмисно гнучка:
резюме — це зрозумілий людям підсумок, а `metadata` — це машинозчитувана передача, яку нижчі агенти, рецензенти або панелі інструментів можуть використовувати без парсингу тексту.

Для інженерних та ревізійних завдань віддавай перевагу цій необов’язковій формі метаданих:

```json
{
  "changed_files": ["path/to/file.py"],
  "verification": ["pytest tests/hermes_cli/test_kanban_db.py -q"],
  "dependencies": ["parent task id or external issue, if any"],
  "blocked_reason": null,
  "retry_notes": "what failed before, if this was a retry",
  "residual_risk": ["what was not tested or still needs human review"]
}
```

Ці ключі — це конвенція, а не вимога схеми. Корисна властивість полягає в тому, що кожен виконавець залишає достатньо доказів для наступного читача, щоб швидко відповісти на чотири питання:

1. Що змінилося?
2. Як це було підтверджено?
3. Що може розблокувати або повторити це, якщо воно не вдасться?
4. Який ризик все ще навмисно залишено відкритим?

Тримай секрети, сирі логи, токени, матеріали OAuth та непов’язані транскрипти поза `metadata`. Зберігай вказівники та резюме замість цього. Якщо завдання не має файлів або тестів, вкажи це явно в `summary` і використай `metadata` для доказів, які існують, наприклад, URL‑джерела, ідентифікатори задач або кроки ручного перегляду.
### Навичка worker

Будь‑який профіль, який має виконувати завдання kanban, повинен завантажити навичку `kanban-worker`. Вона навчає worker повному життєвому циклу у **викликах інструментів**, а не у командах CLI:

1. При запуску виклич `kanban_show()`, щоб прочитати заголовок + текст + батьківські handoffs + попередні спроби + повний ланцюжок коментарів.
2. `cd $HERMES_KANBAN_WORKSPACE` (через інструмент terminal) і працюй у цьому каталозі.
3. Викликай `kanban_heartbeat(note="...")` кожні кілька хвилин під час тривалих операцій. **Якщо твоя робота може тривати довше 1 години, викликай `kanban_heartbeat` принаймні раз на годину** — диспетчер відбирає завдання, які працювали довше `kanban.dispatch_stale_timeout_seconds` (за замовчуванням 4 год) без heartbeat за останню годину, припускаючи, що worker впав без очистки. Відбір безпечний (завдання повертається до `ready` для повторного розподілу без збільшення лічильника помилок), але ти втрачаєш прогрес поточного запуску.
4. Заверши за допомогою `kanban_complete(summary="...", metadata={...})` або `kanban_block(reason="...")`, якщо застряг.

Останній виклик `kanban_complete` / `kanban_block` є частиною протоколу worker. Якщо процес worker завершується зі статусом 0, коли завдання ще `running`, диспетчер розглядає це як порушення протоколу, генерує подію `protocol_violation` і автоматично блокує завдання на наступному тіку замість того, щоб перезапустити його в тому ж циклі. Це зазвичай означає, що модель вивела просту текстову відповідь і завершилася, не використовуючи інтерфейс інструмента Kanban.

`kanban-worker` — це вбудована навичка, синхронізована в кожному профілі під час встановлення та оновлення — окремого кроку встановлення Skills Hub немає. Перевір, чи вона присутня в тому профілі, який ти використовуєш для kanban‑worker’ів (`researcher`, `writer`, `ops` тощо):

```bash
hermes -p <your-worker-profile> skills list | grep kanban-worker
```

Якщо вбудована копія відсутня, віднови її для цього профілю:

```bash
hermes -p <your-worker-profile> skills reset kanban-worker --restore
```

Диспетчер також автоматично додає `--skills kanban-worker` при створенні кожного worker, тож worker завжди має доступ до бібліотеки шаблонів, навіть якщо конфігурація default skills у профілю її не містить.
### Прикріплення додаткових навичок до конкретного завдання

Іноді одне завдання потребує спеціального контексту, який профіль виконавця не містить за замовчуванням — перекладацька робота, що потребує навички `translation`, ревізія, що потребує `github-code-review`, аудит безпеки, що потребує `security-pr-audit`. Замість того, щоб щоразу редагувати профіль виконавця, можна прикріпити навички безпосередньо до завдання.

**Від оркеструвального агента** (звичний випадок — один агент направляє роботу іншому), використай масив `skills` інструмента `kanban_create`:

```
kanban_create(
    title="translate README to Japanese",
    assignee="linguist",
    skills=["translation"],
)

kanban_create(
    title="audit auth flow",
    assignee="reviewer",
    skills=["security-pr-audit", "github-code-review"],
)
```

**Від людини (CLI / slash‑команда)**, повторюй `--skill` для кожної навички:

```bash
hermes kanban create "translate README to Japanese" \
    --assignee linguist \
    --skill translation

hermes kanban create "audit auth flow" \
    --assignee reviewer \
    --skill security-pr-audit \
    --skill github-code-review
```

**З панелі управління**, введи навички, розділені комами, у поле **skills** форми швидкого створення.

Ці навички є **додатковими** до вбудованого `kanban-worker` — диспетчер генерує один прапорець `--skills <name>` для кожної (і для вбудованої), тому воркер запускається зі всіма завантаженими навичками. Імена навичок мають збігатися з навичками, які фактично встановлені у профілі виконавця (виконай `hermes skills list`, щоб переглянути доступні); встановлення під час виконання недоступне.
### Навичка оркестратора

**Правильно налаштований оркестратор не виконує роботу сам.** Він розбиває ціль користувача на завдання, пов’язує їх, призначає кожне одному з профілів, які ти налаштував, і відступає. Навичка `kanban-orchestrator` кодує це у вигляді шаблонів викликів інструментів: правила запобігання спокусі, підказка виявлення профілю на Кроці‑0 (диспетчер безшумно відхиляє невідомі імена виконавців, тому оркестратор має прив’язати кожну картку до профілів, які дійсно існують на твоїй машині) і плейбук розкладання, орієнтований на `kanban_create` / `kanban_link` / `kanban_comment`.

Канонічний хід оркестратора (два паралельних дослідника передають роботу письменнику):

```
# Goal from user: "draft a launch post on the ICP funding landscape"
kanban_create(title="research ICP funding, NA angle",  assignee="researcher-a", body="…")  # → t_r1
kanban_create(title="research ICP funding, EU angle",  assignee="researcher-b", body="…")  # → t_r2
kanban_create(
    title="synthesize ICP funding research into launch post draft",
    assignee="writer",
    parents=["t_r1", "t_r2"],        # promoted to 'ready' when both researchers complete
    body="one-pager, neutral tone, cite sources inline",
)                                     # → t_w1
# Optional: add cross-cutting deps discovered later without re-creating tasks
kanban_link(parent_id="t_r1", child_id="t_followup")
kanban_complete(
    summary="decomposed into 2 parallel research tasks → 1 synthesis task; writer starts when both researchers finish",
)
```

`kanban-orchestrator` — це вбудована навичка. Вона синхронізується в кожен профіль під час
встановлення та оновлення, тому окремий крок встановлення Skills Hub не потрібен. Перевір, чи вона
присутня у твоєму профілі оркестратора:

```bash
hermes -p orchestrator skills list | grep kanban-orchestrator
```

Якщо вбудована копія відсутня, віднови її для цього профілю:

```bash
hermes -p orchestrator skills reset kanban-orchestrator --restore
```

Для найкращих результатів поєднай її з профілем, набори інструментів якого обмежені операціями над дошкою (`kanban`, `gateway`, `memory`), щоб оркестратор буквально не міг виконувати завдання впровадження, навіть якщо спробує.
## Дашборд (GUI)

CLI та слеш‑команда `/kanban` достатні, щоб запустити дошку без графічного інтерфейсу, але візуальна дошка часто є правильним інтерфейсом для людей‑в‑циклі: триаж, нагляд за різними профілями, читання ланцюжків коментарів і перетягування карток між колонками. Hermes постачається це як **вбудований плагін дашборда** у `plugins/kanban/` — не є ядровою функцією, не окремим сервісом — згідно з моделлю, описаною в [Extending the Dashboard](./extending-the-dashboard).

Відкрий його за допомогою:

```bash
hermes kanban init      # one-time: create kanban.db if not already present
hermes dashboard        # "Kanban" tab appears in the nav, after "Skills"
```
### Що дає плагін

- Вкладка **Kanban** показує один стовпець на кожен статус: `triage`, `todo`, `ready`, `running`, `blocked`, `done` (плюс `archived`, коли перемикач увімкнено).
  - `triage` — це колонка‑парковка для грубих ідей. За замовчуванням (`kanban.auto_decompose: true`) диспетчер автоматично запускає **decomposer** для задач, що потрапляють сюди — профіль оркестратора читає грубу ідею, дивиться на ваш список профілів (з описами) і розподіляє задачу у маленький граф дочірніх задач, спрямованих до найкращих спеціалістів. Оригінальна задача залишається живою як батько кожної дочірньої, щоб оркестратор прокинувся і оцінив завершення, коли все закінчиться. Перемкни **Orchestration: Auto/Manual**‑перемикач у верхній частині сторінки (або встанови `kanban.auto_decompose: false`), щоб перейти в ручний режим, коли задачі `triage` залишаються на місці, доки ти не натиснеш **⚗ Decompose** на картці або не виконаєш `hermes kanban decompose <id>`. Для задач, які не потребують розгалуження (або для налаштувань без профілю оркестратора), кнопка **✨ Specify** робить переписування специфікації однієї задачі (заголовок + тіло з метою, підходом, критеріями прийняття) за допомогою тієї ж LLM‑механіки. Дивись [Auto vs Manual orchestration](#auto-vs-manual-orchestration) нижче.
- Карти показують ідентифікатор задачі, заголовок, бейдж пріоритету, тег орендаря, призначений профіль, кількість коментарів/посилань, **чіп прогресу** (`N/M` дочірніх завершено, коли у задачі є залежності) та «створено N тому». Прапорець у кожній картці дозволяє багатовибір.
- **Смужки за профілем всередині Running** — прапорець у панелі інструментів перемикає підгрупування колонки **Running** за виконавцем.
- **Live‑оновлення через WebSocket** — плагін стежить за таблицею `task_events` у режимі лише‑додання з коротким інтервалом опитування; дошка відображає зміни миттєво, коли будь‑який профіль (CLI, gateway або інша вкладка дашборду) діє. Перезавантаження дебаунсуються, тому сплеск подій викликає одне повторне отримання даних.
- **Drag‑drop** картки між колонками для зміни статусу. При падінні надсилається `PATCH /api/plugins/kanban/tasks/:id`, який проходить через той самий код `kanban_db`, що й CLI — три поверхні ніколи не розходяться. Переміщення у деструктивні статуси (`done`, `archived`, `blocked`) запитують підтвердження. На сенсорних пристроях використовується fallback на вказівник, тож дошка працює на планшеті.
- **Inline‑створення** — натисни `+` у заголовку будь‑якої колонки, щоб ввести заголовок, виконавця, пріоритет і (за бажанням) батьківську задачу з випадаючого списку над усіма існуючими задачами. Натисни **Enter**, щоб створити задачу, **Shift+Enter** — вставити новий рядок у поле заголовка, або **Escape** — скасувати. Створення з колонки **Triage** автоматично паркує нову задачу у `triage`.
- **Багатовибір з масовими діями** — **Shift/Ctrl**‑клік по картці або позначка її прапорця додає її до вибору. Появляється панель масових дій у верхній частині з пакетними переходами статусу, архівуванням та переназначенням (через випадаючий список профілів або «(unassign)»). Деструктивні пакети спочатку підтверджуються. Часткові помилки за ідентифікатором повідомляються без переривання решти.
- **Клік по картці** (без **Shift/Ctrl**) відкриває бокову панель (Escape або клік поза межами закриває) з:
  - **Редагованим заголовком** — клікни заголовок, щоб перейменувати.
  - **Редагованим виконавцем / пріоритетом** — клікни метарядок, щоб переписати.
  - **Редагованим описом** — за замовчуванням markdown‑рендеринг (заголовки, жирний, курсив, inline‑code, fenced‑code, `http(s)`/`mailto:` посилання, марковані списки) з кнопкою «edit», що замінює його на textarea. Рендерер markdown — це крихітний, XSS‑безпечний рендерер: кожна підстановка виконується над HTML‑екранованим вводом, лише `http(s)`/`mailto:` посилання пропускаються, а `target="_blank"` + `rel="noopener noreferrer"` завжди встановлюються.
  - **Редактор залежностей** — список чіпів батьків і дітей, кожен з `×` для відв’язування, плюс випадаючі списки над усіма іншими задачами для додавання нового батька або дитини. Спроби створити цикл відхиляються на сервері з чітким повідомленням.
  - **Рядок дії статусу** (→ triage / → ready / → running / block / unblock / complete / archive) з підтверджувальними запитами для деструктивних переходів. Для карток у колонці **Triage** рядок також пропонує дві дії, керовані LLM: **⚗ Decompose** розпилює задачу у граф дочірніх задач, спрямованих до спеціалістів за описом (шлях оркестратора), і **✨ Specify** робить переписування специфікації однієї задачі. **Decompose** повертається до стилю **Specify**, коли LLM вирішує, що задача не виграє від розгалуження, тому це строгий надмножина. Обидві доступні через CLI (`hermes kanban decompose <id>` / `specify <id>` / `--all`), будь‑яку платформу gateway (`/kanban decompose <id>`), і програмно через `POST /api/plugins/kanban/tasks/:id/decompose` та `…/specify`. Налаштуй моделі у `auxiliary.kanban_decomposer` та `auxiliary.triage_specifier` у `config.yaml`.
  - Розділ результату (також markdown‑рендеринг), гілка коментарів з **Enter**‑для‑відправки, останні 20 подій.
- **Фільтри панелі інструментів** — пошук вільним текстом, випадаючий список орендаря (за замовчуванням `dashboard.kanban.default_tenant` з `config.yaml`), випадаючий список виконавця, перемикач «показати архівовані», перемикач «смужки за профілем» та кнопка **Nudge dispatcher**, щоб не чекати наступного 60‑секундного тіку.

Візуально ціль — знайомий макет Linear / Fusion: темна тема, заголовки колонок з лічильниками, кольорові крапки статусу, чіпи для пріоритету та орендаря. Плагін читає лише CSS‑змінні теми (`--color-*`, `--radius`, `--font-mono`, …), тому автоматично підлаштовується під активну тему дашборду.
### Auto vs Manual orchestration

Дошка kanban має два способи обробки задачі, яку ти кидаєш у колонку **Triage**:

**Auto (за замовчуванням)** — `kanban.auto_decompose: true`. Диспетчер, вбудований у шлюз, запускає **decomposer** на кожному тіку, обмежений `kanban.auto_decompose_per_tick` (за замовчуванням 3 задачі за тик), щоб масове навантаження задач triage не споживало зайві ресурси допоміжного LLM. Decomposer читає загальну ідею, дивиться на твої встановлені профілі + їхні описи і просить LLM створити JSON‑граф задач: які задачі створювати, кому їх призначати і які залежності між ними. Оригінальна задача triage стає батьківською для кожного листа в графі, тому вона залишається активною, доки весь граф не завершиться — а потім повертається у стан `ready`, щоб її виконавець (профіль оркестратора) міг оцінити завершення і додати нові задачі, якщо робота ще не завершена. Це потік «кидай однорядкове повідомлення, йди далі».

**Manual** — `kanban.auto_decompose: false`. Задачі triage залишаються в triage, доки ти не втрутишся. Натисни кнопку **⚗ Decompose** на картці, запусти `hermes kanban decompose <id>` (або `--all`), або використай `/kanban decompose <id>` у чаті. Це відповідає поведінці дошки до використання decomposer‑а, корисно, коли потрібен повний контроль над тим, що і коли запускається.

Перемикай між двома режимами за допомогою **пілки** **Orchestration: Auto/Manual** у верхній частині сторінки kanban (emerald = Auto, muted gray = Manual) або редагуючи `config.yaml` безпосередньо. Обидва режими співіснують з `hermes kanban specify` — це все ще доступно як переписування однозадачної специфікації, коли не потрібен fan‑out.

Рішення маршрутизації decomposer‑а залежать від описів профілів, що є примітивом маркування per‑profile, який ти задаєш за допомогою `hermes profile create --description "..."`, `hermes profile describe <name> --text "..."`, `hermes profile describe <name> --auto` (LLM‑генерує з встановлених навичок профілю + моделі) або редактора per‑profile у розширеній панелі **Orchestration settings** на дашборді. Профілі без опису все одно з’являються у списку — їх можна маршрутизувати за назвою, просто менш точно. Decomposer НІКОЛИ не залишає дочірню задачу з `assignee=None`: коли LLM вибирає невідомий профіль, дитина маршрутується до `kanban.default_assignee` (або активного профілю за замовчуванням, якщо це не задано).

Налаштування (всі під `kanban:` у `~/.hermes/config.yaml`):

| Key | Default | Purpose |
|---|---|---|
| `auto_decompose` | `true` | Диспетчер автоматично запускає decomposer кожен тик. |
| `auto_decompose_per_tick` | `3` | Обмеження кількості розкладів за один тик диспетчера. Надлишок переноситься на наступний тик. |
| `orchestrator_profile` | `""` | Профіль, який відповідає за розкладання. Порожньо = повернутись до активного профілю за замовчуванням. |
| `default_assignee` | `""` | Куди потрапляє дочірня задача, коли LLM вибирає невідомий профіль. Порожньо = повернутись до активного профілю за замовчуванням. |

І два допоміжних слоти LLM:

| Key | Purpose |
|---|---|
| `auxiliary.kanban_decomposer` | Модель, що створює граф задач (викликається Decompose). Встанови `provider`/`model`, щоб перевизначити основну чат‑модель. |
| `auxiliary.profile_describer` | Модель, що автоматично генерує описи профілів (викликається `hermes profile describe --auto`). |

### REST surface

All routes are mounted under `/api/plugins/kanban/` and protected by the dashboard's ephemeral session token:

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/board?tenant=<name>&include_archived=…` | Повна дошка, згрупована за колонкою статусу, плюс орендарі + виконавці для випадаючих списків фільтра |
| `GET` | `/tasks/:id` | Завдання + коментарі + події + посилання |
| `POST` | `/tasks` | Створення (обгортає `kanban_db.create_task`, приймає `triage: bool` і `parents: [id, …]`) |
| `PATCH` | `/tasks/:id` | Статус / виконавець / пріоритет / заголовок / тіло / результат |
| `POST` | `/tasks/bulk` | Застосувати той самий патч (status / archive / assignee / priority) до кожного `id` у `ids`. Помилки по окремим `id` повідомляються без переривання обробки інших |
| `POST` | `/tasks/:id/comments` | Додати коментар |
| `POST` | `/tasks/:id/specify` | Запустити triage‑specifier — допоміжний LLM заповнює тіло завдання і переводить його з `triage` у `todo`. Повертає `{ok, task_id, reason, new_title}`; `ok=false` з людським поясненням причини «not in triage» / відсутній aux‑клієнт / помилка LLM — це 200, а не 4xx |
| `POST` | `/tasks/:id/decompose` | Запустити kanban‑decomposer — допоміжний LLM створює граф завдань, а helper атомарно створює дочірні елементи + зв’язує корінь + перемикає `triage → todo`. Повертає `{ok, task_id, reason, fanout, child_ids, new_title}`. Така ж конвенція 200‑on‑LLM‑error, як у `/specify`. |
| `GET` | `/profiles` | Список встановлених профілів з їх описами (використовується редактором опису профілів у dashboard та picker‑ом оркестратора) |
| `PATCH` | `/profiles/:name` | Встановити або очистити опис профілю (користувачем — `description_auto: false`). Повертає `{ok, profile, description}` |
| `POST` | `/profiles/:name/describe-auto` | Згенерувати опис профілю за допомогою `auxiliary.profile_describer`. Зберігає з `description_auto: true`, щоб dashboard міг показати бейдж «review» |
| `GET` | `/orchestration` | Прочитати налаштування kanban‑orchestration (`orchestrator_profile`, `default_assignee`, `auto_decompose`) плюс *розв’язані* ефективні значення після запасних (fallback) |
| `PUT` | `/orchestration` | Оновити один або кілька з трьох ключів оркестрації у `config.yaml`. Перевіряє, що непорожні назви профілів дійсно існують |
| `POST` | `/links` | Додати залежність (`parent_id` → `child_id`) |
| `DELETE` | `/links?parent_id=…&child_id=…` | Видалити залежність |
| `POST` | `/dispatch?max=…&dry_run=…` | Підштовхнути диспетчера — пропустити 60‑секундну затримку |
| `GET` | `/config` | Прочитати налаштування `dashboard.kanban` з `config.yaml` — `default_tenant`, `lane_by_profile`, `include_archived_by_default`, `render_markdown` |
| `WS` | `/events?since=<event_id>` | Потік у реальному часі рядків `task_events` |

Every handler is a thin wrapper — the plugin is ~700 lines of Python (router + WebSocket tail + bulk batcher + config reader) and adds no new business logic. A tiny `_conn()` helper auto-initializes `kanban.db` on every read and write, so a fresh install works whether the user opened the dashboard first, hit the REST API directly, or ran `hermes kanban init`.
### Конфігурація Dashboard

Будь‑який з цих ключів у `dashboard.kanban` у `~/.hermes/config.yaml` змінює типові параметри вкладки — плагін читає їх під час завантаження через `GET /config`:

```yaml
dashboard:
  kanban:
    default_tenant: acme              # preselects the tenant filter
    lane_by_profile: true             # default for the "lanes by profile" toggle
    include_archived_by_default: false
    render_markdown: true             # set false for plain <pre> rendering
```

Кожен ключ є необов’язковим і повертається до зазначеного типового значення.
### Модель безпеки

HTTP‑middleware автентифікації дашборду [явно пропускає `/api/plugins/`](./extending-the-dashboard#backend-api-routes) — маршрути плагінів не автентифіковані за задумом, оскільки дашборд за замовчуванням прив’язується до `localhost`. Це означає, що REST‑поверхня kanban доступна будь‑якому процесу на хості.

WebSocket робить ще один крок: він вимагає епемерний токен сесії дашборду як параметр запиту `?token=…` (браузери не можуть встановити `Authorization` у запиті на оновлення), використовуючи той самий шаблон, що й у браузерному PTY‑мосту.

Якщо запустити `hermes dashboard --host 0.0.0.0`, кожен маршрут плагіна — включно з kanban — стане доступним з мережі. **Не роби цього на спільному хості.** Дошка містить тіла задач, коментарі та шляхи робочих просторів; зловмисник, який отримав доступ до цих маршрутів, отримує право читання всього вашого простору співпраці і може також створювати / переназначати / архівувати задачі.

Задачі у `~/.hermes/kanban.db` навмисно не прив’язані до профілю (це примітив координації). Якщо відкрити дашборд за допомогою `hermes -p <profile> dashboard`, дошка все одно показуватиме задачі, створені будь‑яким іншим профілем на хості. Один користувач володіє всіма профілями, але це варто знати, якщо існує кілька персон.
### Live updates

`task_events` — це таблиця SQLite лише для додавання (append‑only) з монотонним `id`. Endpoint WebSocket зберігає останній отриманий ідентифікатор події кожного клієнта та надсилає нові рядки, коли вони з’являються. Коли надходить велика кількість подій, frontend перезавантажує (дуже недорогий) endpoint `board` — простіше і правильніше, ніж намагатися патчити локальний стан для кожного типу події. Режим WAL означає, що цикл читання ніколи не блокує транзакції `BEGIN IMMEDIATE` диспетчера.
### Розширення

Плагін використовує стандартний контракт плагіну Hermes dashboard — дивись [Розширення Dashboard](./extending-the-dashboard) для повної довідки щодо маніфесту, слотів оболонки, слотів у межах сторінки та SDK плагіну. Додаткові стовпці, власне оформлення картки, макети, відфільтровані за орендарем, або повні заміни `tab.override` — усе це можна реалізувати без форкування плагіну.

Щоб вимкнути без видалення: додай `dashboard.plugins.kanban.enabled: false` до `config.yaml` (або видали `plugins/kanban/dashboard/manifest.json`).
### Межа області

GUI навмисно тонка. Усе, що робить плагін, доступно з CLI; плагін лише робить це зручним для людей. Автоматичне призначення, бюджети, ворота управління та перегляди організаційної схеми залишаються у просторі користувача — профіль роутера, інший плагін або повторне використання `tools/approval.py` — точно так, як зазначено в розділі «out-of-scope» специфікації дизайну.
## CLI command reference

This is the surface **you** (or scripts, cron, the dashboard) use to drive the board. Workers running inside the dispatcher use the `kanban_*` [tool surface](#how-workers-interact-with-the-board) for the same operations — the CLI here and the tools there both route through `kanban_db`, so the two surfaces agree by construction.

```
hermes kanban init                                     # create kanban.db + print daemon hint
hermes kanban create "<title>" [--body ...] [--assignee <profile>]
                                [--parent <id>]... [--tenant <name>]
                                [--workspace scratch|worktree|worktree:<path>|dir:<path>]
                                [--branch <name>]
                                [--priority N] [--triage] [--idempotency-key KEY]
                                [--max-runtime 30m|2h|1d|<seconds>]
                                [--max-retries N]
                                [--skill <name>]...
                                [--json]
hermes kanban list [--mine] [--assignee P] [--status S] [--tenant T] [--archived]
        [--workflow-template-id <id>] [--current-step-key <key>]
        [--sort created|created-desc|priority|priority-desc|status|assignee|title|updated]
        [--json]
hermes kanban show <id> [--json]
hermes kanban assign <id> <profile>                    # or 'none' to unassign
hermes kanban link <parent_id> <child_id>
hermes kanban unlink <parent_id> <child_id>
hermes kanban claim <id> [--ttl SECONDS]
hermes kanban comment <id> "<text>" [--author NAME]

# Bulk verbs — accept multiple ids:
hermes kanban complete <id>... [--result "..."]
hermes kanban block <id> "<reason>" [--ids <id>...]
hermes kanban unblock <id>...
hermes kanban archive <id>...

hermes kanban tail <id>                                # follow a single task's event stream
hermes kanban watch [--assignee P] [--tenant T]        # live stream ALL events to the terminal
        [--kinds completed,blocked,…] [--interval SECS]
hermes kanban heartbeat <id> [--note "..."]            # worker liveness signal for long ops
hermes kanban runs <id> [--json]                       # attempt history (one row per run)
hermes kanban assignees [--json]                       # profiles on disk + per-assignee task counts
hermes kanban dispatch [--dry-run] [--max N]           # one-shot pass
        [--failure-limit N] [--json]
hermes kanban daemon --force                           # DEPRECATED — standalone dispatcher (use `hermes gateway start` instead)
        [--failure-limit N] [--pidfile PATH] [-v]
hermes kanban stats [--json]                           # per-status + per-assignee counts
hermes kanban log <id> [--tail BYTES]                  # worker log from ~/.hermes/kanban/logs/
hermes kanban notify-subscribe <id>                    # gateway bridge hook (used by /kanban in the gateway)
        --platform <name> --chat-id <id> [--thread-id <id>] [--user-id <id>]
hermes kanban notify-list [<id>] [--json]
hermes kanban notify-unsubscribe <id>
        --platform <name> --chat-id <id> [--thread-id <id>]
hermes kanban context <id>                             # what a worker sees
hermes kanban specify [<id> | --all] [--tenant T]      # flesh out a triage-column idea
        [--author NAME] [--json]                       #   into a full spec and promote to todo
hermes kanban gc [--event-retention-days N]            # workspaces + old events + old logs
        [--log-retention-days N]
```

All commands are also available as a slash command in the interactive CLI and in the messaging gateway (see [`/kanban` slash command](#kanban-slash-command) below).

`--max-retries` is a per-task circuit‑breaker override for the dispatcher. `--max-retries 1` blocks the task on the first non‑successful attempt, while `--max-retries 3` allows two retries and blocks on the third failure. Omit it to use `kanban.failure_limit` from `config.yaml`, then the built‑in default.

### Concurrency, scheduling, and child promotion config

| Config key | Default | What it does |
|------------|---------|--------------|
| `kanban.max_in_progress` | unset (unlimited) | Caps the number of simultaneously running tasks. When the board already has N running, the dispatcher skips spawning more — useful for slow workers (local LLMs, resource‑constrained hosts) so they finish what they have before more pile up and time out. Invalid or below‑1 values log a warning and behave as unlimited. |
| `kanban.auto_promote_children` | `true` | After `decompose_triage_task()` produces children with no parent‑blocker dependencies, they're automatically promoted to `ready` so the dispatcher can pick them up. Set to `false` to require manual review — children stay in `todo` until you promote them. |
| `kanban.default_workdir` | unset | Board‑level default working directory applied to new tasks when neither `--workspace` nor the task itself overrides it. Per‑task `workspace:` still wins. |

```yaml
kanban:
  max_in_progress: 2
  auto_promote_children: false
  default_workdir: ~/work/active-project
```

### Scheduled task starts (`scheduled_at`)

Set `scheduled_at` on a task to delay dispatch until a specific time. The dispatcher skips ready tasks whose `scheduled_at` is in the future and picks them up on the first tick after that timestamp.

```bash
hermes kanban create "nightly backup audit" \
  --assignee ops --scheduled-at "2026-06-01T03:00:00Z"
```

### Respawn guard

The dispatcher refuses to re‑spawn a ready task when it hit a quota/auth/429 error on the previous run (`blocker_auth`), or completed a run successfully within the guard window (`recent_success`), or a recent task comment links to a GitHub PR (`active_pr`). This prevents repeat worker storms on the same bug or task while a human catches up. See the `respawn_guarded` row in the [event reference](#event-reference).

### Drag‑to‑delete and bulk delete (dashboard)

The dashboard exposes a **trash drop zone** on the kanban page — drag any card into it to delete the task (cascades through `task_events`, child links, and subscriptions). A confirmation prompt protects against accidents. Bulk delete is also reachable via `DELETE /api/plugins/kanban/tasks` with a JSON body `{"ids": ["t_abc", "t_def", ...]}`.

### Worker visibility endpoints

The dashboard plugin API now exposes three read‑only endpoints for external monitors:

| Endpoint | Returns |
|----------|----------|
| `GET /api/plugins/kanban/workers/active` | Currently spawned workers with PID, profile, task id, started‑at, last heartbeat |
| `GET /api/plugins/kanban/runs/{id}` | Single‑run detail — task id, status, started/ended, exit code, log path |
| `GET /api/plugins/kanban/inspect` | Combined dispatcher snapshot — backlog, in‑progress count vs. `max_in_progress`, recent events |

All three are gated by the same dashboard plugin auth as the rest of the kanban plugin API.

### Kanban Swarm topology helper

`hermes kanban swarm` creates a durable **Kanban Swarm v1** graph in one shot: a completed root/blackboard card, N parallel worker cards, a verifier card gated on all workers, and a synthesizer card gated on the verifier. Shared swarm context (the “blackboard”) is stored as structured JSON comments on the root card so any worker can read it.

```bash
hermes kanban swarm "Design a multi-region failover plan" \
  --workers researcher,architect,sre \
  --verifier reviewer --synthesizer writer
```

The resulting graph dispatches normally — workers run in parallel, the verifier wakes after they all finish, the synthesizer wakes after the verifier marks the work clean.
## `/kanban` slash command {#kanban-slash-command}

Кожен `hermes kanban <action>` дієслово також доступний як `/kanban <action>` — і всередині інтерактивної сесії `hermes chat`, **і** з будь‑якої платформи шлюзу (Telegram, Discord, Slack, WhatsApp, Signal, Matrix, Mattermost, email, SMS). Обидві поверхні викликають один і той самий entry point `hermes_cli.kanban.run_slash()`, який повторно використовує дерево аргументів `hermes kanban`, тому поверхня аргументів, прапорці та формат виводу ідентичні між CLI, `/kanban` та `hermes kanban`. Тобі не потрібно залишати чат, щоб керувати дошкою.

```
/kanban list
/kanban show t_abcd
/kanban create "write launch post" --assignee writer --parent t_research
/kanban comment t_abcd "looks good, ship it"
/kanban unblock t_abcd
/kanban dispatch --max 3
/kanban specify t_abcd                  # flesh out a triage one-liner into a real spec
/kanban specify --all --tenant engineering  # sweep every triage task in one tenant
```

Цитуй багатослівні аргументи так само, як у shell — `run_slash` розбирає решту рядка за допомогою `shlex.split`, тому працюють і `"..."`, і `'...'`.

### Mid-run usage: `/kanban` обходить захисний механізм запущеного агента

Шлюз зазвичай ставить у чергу slash‑команди та повідомлення користувачів, доки агент ще мислить — це запобігає випадковому запуску другого ходу, коли перший ще в процесі. **`/kanban` явно виключений з цього захисного механізму**. Дошка зберігається у `~/.hermes/kanban.db`, а не у стані запущеного агента, тому читання (`list`, `show`, `context`, `tail`, `watch`, `stats`, `runs`) і запис (`comment`, `unblock`, `block`, `assign`, `archive`, `create`, `link`, …) виконуються одразу, навіть посеред ходу.

Саме в цьому полягає вся суть розділення:

- Робітник блокується в очікуванні піра → ти надсилаєш `/kanban unblock t_abcd` зі свого телефону, і диспетчер підхоплює піра на наступному тику. Заблокований робітник не переривається — просто перестає бути заблокованим.
- Ти помічаєш картку, якій потрібен людський контекст → `/kanban comment t_xyz "use the 2026 schema, not 2025"` з’являється у темі завдання, і *наступний* запуск цього завдання прочитає його у `kanban_show()`.
- Ти хочеш дізнатися, що робить твій флот, не зупиняючи оркестратор → `/kanban list --mine` або `/kanban stats` переглядає дошку, не торкаючись основної розмови.

### Auto-subscribe on `/kanban create` (лише шлюз)

Коли ти створюєш завдання через шлюз за допомогою `/kanban create "…"`, вихідний чат (платформа + chat id + thread id) автоматично підписується на події терміналу завдання (`completed`, `blocked`, `gave_up`, `crashed`, `timed_out`). Ти отримуєш одне повідомлення на кожну подію терміналу — включно з першим рядком підсумку результату робітника у `completed` — без необхідності опитувати чи запам’ятовувати id завдання.

```
you> /kanban create "transcribe today's podcast" --assignee transcriber
bot> Created t_9fc1a3  (ready, assignee=transcriber)
     (subscribed — you'll be notified when t_9fc1a3 completes or blocks)

… ~8 minutes later …

bot> ✓ t_9fc1a3 completed by transcriber
     transcribed 42 minutes, saved to podcast/2026-05-04.md
```

Підписки автоматично видаляються, коли завдання переходить у `done` або `archived`. Якщо ти скриптуєш створення з `--json` (машинний вивід), авто‑підписка пропускається — передбачається, що скриптові виклики хочуть керувати підписками явно через `/kanban notify-subscribe`.

### Output truncation in messaging

Платформи шлюзу мають практичні обмеження довжини повідомлення. Якщо `/kanban list`, `/kanban show` або `/kanban tail` генерують більше ~3800 символів виводу, відповідь обрізається з футером `… (truncated; use \`hermes kanban …\` in your terminal for full output)`. У CLI такого обмеження немає.

### Autocomplete

В інтерактивному CLI, ввівши `/kanban ` і натиснувши Tab, ти циклічно перебираєш вбудований список підкоманд (`list`, `ls`, `show`, `create`, `assign`, `link`, `unlink`, `claim`, `comment`, `complete`, `block`, `unblock`, `archive`, `tail`, `dispatch`, `context`, `init`, `gc`). Інші дієслова, зазначені у довідці CLI вище (`watch`, `stats`, `runs`, `log`, `assignees`, `heartbeat`, `notify-subscribe`, `notify-list`, `notify-unsubscribe`, `daemon`), також працюють — вони просто ще не включені у список підказок автодоповнення.
## Шаблони співпраці

The board supports these eight patterns without any new primitives:

| Шаблон | Форма | Приклад |
|---|---|---|
| **P1 Fan-out** | N однорангових, однакова роль | "research 5 angles in parallel" |
| **P2 Pipeline** | ланцюжок ролей: scout → editor → writer | daily brief assembly |
| **P3 Voting / quorum** | N однорангових + 1 агрегатор | 3 researchers → 1 reviewer picks |
| **P4 Long-running journal** | один профіль + спільна директорія + cron | Obsidian vault |
| **P5 Human-in-the-loop** | блокування працівника → коментарі користувача → розблокування | ambiguous decisions |
| **P6 `@mention`** | inline routing from prose | `@reviewer look at this` |
| **P7 Thread-scoped workspace** | `/kanban here` у темі | per-project gateway threads |
| **P8 Fleet farming** | один профіль, N суб’єктів | 50 social accounts |
| **P9 Triage specifier** | нечітка ідея → `triage` → `hermes kanban specify` розширює тіло → `todo` | "turn this one-liner into a spec'd task" |

For worked examples of each, see `docs/hermes-kanban-v1-spec.pdf`.
## Багатокористувацьке використання

Коли один флот спеціалістів обслуговує кілька компаній, позначай кожне завдання тегом **tenant**:

```bash
hermes kanban create "monthly report" \
    --assignee researcher \
    --tenant business-a \
    --workspace dir:~/tenants/business-a/data/
```

Працівники отримують `$HERMES_TENANT` і просторово розмежовують записи пам’яті, додаючи префікс. Дошка, диспетчер і визначення профілів спільні; лише дані мають область видимості.
## Сповіщення шлюзу

Коли ти запускаєш `/kanban create …` з шлюзу (Telegram, Discord, Slack тощо), чат, з якого була виконана команда, автоматично підписується на нове завдання. Фонова служба сповіщень шлюзу опитує `task_events` кожні кілька секунд і надсилає одне повідомлення на кожну подію терміналу (`completed`, `blocked`, `gave_up`, `crashed`, `timed_out`) у цей чат. Завершені завдання також надсилають перший рядок `--result` робітника, щоб ти бачив результат без необхідності виконувати `/kanban show`.

Ти можеш керувати підписками явно з CLI — це корисно, коли скрипт / cron‑завдання хоче сповістити чат, з якого воно не було запущено:

```bash
hermes kanban notify-subscribe t_abcd \
    --platform telegram --chat-id 12345678 --thread-id 7
hermes kanban notify-list
hermes kanban notify-unsubscribe t_abcd \
    --platform telegram --chat-id 12345678 --thread-id 7
```

Підписка автоматично видаляє себе, коли завдання досягає стану `done` або `archived`; очищення не потрібне.
## Запуски — один рядок на спробу

Завдання — це логічна одиниця роботи; **run** — це одна спроба його виконати. Коли диспетчер бере готове завдання, він створює рядок у `task_runs` і встановлює `tasks.current_run_id` на нього. Коли ця спроба завершується — completed, blocked, crashed, timed out, spawn‑failed, reclaimed — рядок run закривається з `outcome`, а вказівник завдання очищається. Завдання, яке було спробовано три рази, має три рядки `task_runs`.

Чому дві таблиці, а не просто змінювати завдання: потрібна **повна історія спроб** для реальних постмортемів («другий рецензент схвалив, третій — злив»), і потрібне чисте місце для метаданих кожної спроби — які файли змінилися, які тести пройшли, які зауваження залишив рецензент. Це факти run, а не факти завдання.

Запуски також містять **структуровану передачу**. Коли виконавець завершує завдання (через `kanban_complete(...)`) він може передати:

- `summary` (tool param) / `--summary` (CLI) — людська передача; зберігається в run; дочірні процеси бачать її у своєму `build_worker_context`.
- `metadata` (tool param) / `--metadata` (CLI) — довільний JSON‑словник у run; діти бачать його серіалізованим разом із `summary`.
- `result` (tool param) / `--result` (CLI) — короткий рядок журналу, що зберігається в рядку завдання (застаріле поле, залишене для зворотної сумісності).

Дочірні процеси читають останній завершений run, беручи `summary` + `metadata` для кожного батька. Виконавці, які повторно запускаються, читають попередні спроби свого власного завдання (outcome, summary, error), щоб не повторювати шлях, який вже завершився помилкою.

```
# What a worker actually does — a tool call, from inside the agent loop:
kanban_complete(
    summary="implemented token bucket, keys on user_id with IP fallback, all tests pass",
    metadata={"changed_files": ["limiter.py", "tests/test_limiter.py"], "tests_run": 14},
    result="rate limiter shipped",
)
```

Ту ж саму передачу можна отримати з CLI, коли ти (людина) маєш завершити завдання, яке виконавець не може — напр., завдання, яке було залишено, або те, що ти позначив як виконане вручну з панелі управління:

```bash
hermes kanban complete t_abcd \
    --result "rate limiter shipped" \
    --summary "implemented token bucket, keys on user_id with IP fallback, all tests pass" \
    --metadata '{"changed_files": ["limiter.py", "tests/test_limiter.py"], "tests_run": 14}'

# Review the attempt history on a retried task:
hermes kanban runs t_abcd
#   #  OUTCOME       PROFILE           ELAPSED  STARTED
#   1  blocked       worker               12s  2026-04-27 14:02
#        → BLOCKED: need decision on rate-limit key
#   2  completed     worker                8m   2026-04-27 15:18
#        → implemented token bucket, keys on user_id with IP fallback
```

Запуски доступні на панелі управління (розділ Run History у боковій панелі, один кольоровий рядок на спробу) і через REST API (`GET /api/plugins/kanban/tasks/:id` повертає масив `runs[]`). `PATCH /api/plugins/kanban/tasks/:id` з `{status: "done", summary, metadata}` передає обидва параметри ядру, тому кнопка панелі «mark done» еквівалентна CLI. Рядки `task_events` містять `run_id`, до якого вони належать, тому UI може групувати їх за спробами, а подія `completed` вбудовує підсумковий рядок у своєму навантаженні (обмежено 400 символами), щоб нотифікатори шлюзу могли відображати структуровану передачу без другого SQL‑запиту.

**Попередження про масове завершення.** `hermes kanban complete a b c --summary X` відхиляється — структурована передача прив’язана до run, тому копіювання одного і того ж `summary` до N завдань майже завжди помилкове. Масове завершення *без* `--summary` / `--metadata` все ще працює для типового випадку «я завершив купу адміністративних завдань».

**Повернені запуски при зміні статусу.** Якщо ти перетягнеш працююче завдання з `running` у боковій панелі (назад до `ready` або одразу до `todo`), або заархівуєш завдання, яке ще працювало, поточний run закривається з `outcome='reclaimed'`, а не залишається без батька. Рядок `task_runs` завжди перебуває в термінальному стані, коли `tasks.current_run_id` дорівнює `NULL`, і навпаки — ця інваріантність зберігається у CLI, панелі, диспетчері та нотифікаторі.

**Синтетичні запуски для завершень без претензії.** Завершення або блокування завдання, яке ніколи не було претендоване (наприклад, людина закриває `ready` завдання з панелі, додаючи `summary`, або користувач CLI виконує `hermes kanban complete <ready-task> --summary X`), інакше втратило б передачу. Замість цього ядро вставляє рядок run нульової тривалості (`started_at == ended_at`), що містить `summary` / `metadata` / `reason`, щоб історія спроб залишалась повною. Подія `completed` / `blocked` має `run_id`, що вказує на цей рядок.

**Живе оновлення бокової панелі.** Коли потік подій WebSocket панелі повідомляє про нові події для завдання, яке користувач переглядає, бокова панель перезавантажується сама (через лічильник подій, переданий у список залежностей `useEffect`). Тепер не потрібно закривати і відкривати панель, щоб побачити новий рядок run або оновлений результат.

### Сумісність у майбутньому

Два nullable‑стовпці в `tasks` зарезервовані для маршрутизації робочих процесів v2: `workflow_template_id` (який шаблон належить цьому завданню) і `current_step_key` (який крок у цьому шаблоні активний). Ядро v1 ігнорує їх для маршрутизації, але дозволяє клієнтам записувати їх, щоб випуск v2 міг додати механізм маршрутизації без додаткової міграції схеми.
## Посилання на події

Кожен перехід додає рядок до `task_events`. Кожен рядок містить необов’язковий `run_id`, щоб UI могли групувати події за спробою. Типи групуються в три кластери, що спрощує фільтрацію (`hermes kanban watch --kinds completed,gave_up,timed_out`):

**Життєвий цикл** (що змінилося в завданні як у логічній одиниці):

| Kind | Payload | When |
|---|---|---|
| `created` | `{assignee, status, parents, tenant}` | Завдання вставлено. `run_id` — `NULL`. |
| `promoted` | — | `todo → ready`, бо всі батьки досягли `done`. `run_id` — `NULL`. |
| `claimed` | `{lock, expires, run_id}` | Диспетчер атомарно забрав `ready`‑завдання для запуску. |
| `completed` | `{result_len, summary?}` | Робітник записав `--result` / `--summary` і завдання перейшло в `done`. `summary` — це передача першого рядка (обмеження 400 симв); повна версія зберігається в рядку запуску. Якщо `complete_task` викликано для ніколи не забраного завдання з полями передачі, синтезується запуск нульової тривалості, щоб `run_id` все ще вказував на щось. |
| `blocked` | `{reason}` | Робітник або людина перевели завдання в `blocked`. Синтезується запуск нульової тривалості, коли викликано для ніколи не забраного завдання з `--reason`. |
| `unblocked` | — | `blocked → ready`, вручну або через `/unblock`. `run_id` — `NULL`. |
| `archived` | — | Приховано в дошці за замовчуванням. Якщо завдання ще виконувалося, переносить `run_id` запуску, який був відновлений як побічний ефект. |

**Редагування** (зміни, здійснені людиною, що не є переходами):

| Kind | Payload | When |
|---|---|---|
| `assigned` | `{assignee}` | Змінено виконавця (включаючи скасування). |
| `edited` | `{fields}` | Оновлено заголовок або тіло. |
| `reprioritized` | `{priority}` | Змінено пріоритет. |
| `status` | `{status}` | Перетягуванням у дашборді записано статус безпосередньо (наприклад, `todo → ready`). Переносить `run_id` запуску, який був відновлений при перетягуванні з `running`; інакше `run_id` — `NULL`. |

**Телеметрія робітника** (про процес виконання, а не про логічне завдання):

| Kind | Payload | When |
|---|---|---|
| `spawned` | `{pid}` | Диспетчер успішно запустив процес робітника. |
| `heartbeat` | `{note?}` | Робітник викликав `hermes kanban heartbeat $TASK`, щоб сигналізувати про живучість під час довгих операцій. |
| `reclaimed` | `{stale_lock}` | Термін TTL блокування минув без завершення; завдання повертається до `ready`. |
| `crashed` | `{pid, claimer}` | PID робітника більше не живий, хоча TTL ще не сплив. |
| `timed_out` | `{pid, elapsed_seconds, limit_seconds, sigkill}` | Перевищено `max_runtime_seconds`; диспетчер послав SIGTERM (потім SIGKILL через 5 сек грації) і поставив у чергу знову. |
| `stale` | `{elapsed_seconds, last_heartbeat_at, heartbeat_age_seconds, timeout_seconds, pid, terminated}` | Завдання працювало довше, ніж `kanban.dispatch_stale_timeout_seconds` (за замовчуванням 4 год) **І** не отримано `kanban_heartbeat` за останню годину. Диспетчер посилає SIGTERM локальному робітнику (якщо є), скидає завдання в `ready` для повторного диспетчирингу. Не збільшує лічильник помилок (stale — це виявлення відсутності на боці диспетчера, а не помилка робітника). Робітники, що виконують довгі операції, мають викликати `kanban_heartbeat` принаймні раз на годину, щоб уникнути цього. |
| `respawn_guarded` | `{reason}` | Диспетчер відмовився повторно запускати це готове завдання в цьому такті. Причини: `blocker_auth` (останньою помилкою була помилка квоти/автентифікації/429 — чекати, поки вікно ліміту скидиться), `recent_success` (завершений запуск був за останню годину — чекати перегляду перед повторним запуском), `active_pr` (у недавньому коментарі з’явився URL GitHub PR — попередній робітник вже відкрив PR). Завдання залишається в `ready`; наступний такт отримає ще одну можливість запуску. Якщо підляжаща умова зберігається, звичайний ланцюговий вимикач `consecutive_failures` автоматично заблокує через `gave_up` після `failure_limit` помилок. |
| `spawn_failed` | `{error, failures}` | Одна спроба запуску зазнала невдачі (відсутній PATH, робочий простір не змонтовано тощо). Лічильник збільшується; завдання повертається до `ready` для повторної спроби. |
| `protocol_violation` | `{pid, claimer, exit_code}` | Робітник успішно завершився, коли завдання ще було в `running`, зазвичай тому, що він відповів без виклику `kanban_complete` або `kanban_block`. Диспетчер також випускає `gave_up` і одразу автоматично блокує замість повторної спроби. |
| `gave_up` | `{failures, effective_limit, limit_source, error}` | Ланцюговий вимикач спрацював після N послідовних невдалих спроб. Завдання автоматично блокується з останньою помилкою. Ефективне обмеження визначається як `max_retries` завдання, потім `failure_limit` диспетчера / `kanban.failure_limit`, потім вбудоване значення за замовчуванням. |

`hermes kanban tail <id>` показує ці події для окремого завдання. `hermes kanban watch` транслює їх по всій дошці.
## Out of scope

Kanban навмисно працює в режимі одного хоста. `~/.hermes/kanban.db` — це локальний файл SQLite, а диспетчер створює воркери на тому ж комп'ютері. Запуск спільної дошки на двох хостах не підтримується — немає примітиву координації для «worker X на host A, worker Y на host B», і шлях виявлення збоїв передбачає, що PID‑и локальні для хоста. Якщо потрібна багатохостова робота, запусти незалежну дошку на кожному хості та використай `delegate_task` / чергу повідомлень для їх з’єднання.
