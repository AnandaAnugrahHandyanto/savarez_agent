# Пакетна обробка

Пакетна обробка дозволяє запускати **Hermes Agent** на сотнях чи тисячах запитів паралельно, генеруючи структуровані дані траєкторій. Це в основному використовується для **генерації даних для навчання** — створення траєкторій у форматі ShareGPT зі статистикою використання інструментів, які можна застосовувати для донастройки або оцінки.

## Огляд

Запускатор пакетної обробки (`batch_runner.py`) обробляє набір даних у форматі JSONL, виконуючи кожен запит у повній сесії агента з доступом до інструментів. Кожен запит отримує власне ізольоване середовище. Результатом є структуровані дані траєкторії з повною історією розмови, статистикою викликів інструментів та метриками покриття міркувань.

## Швидкий старт

```bash
# Basic batch run
python batch_runner.py \
    --dataset_file=data/prompts.jsonl \
    --batch_size=10 \
    --run_name=my_first_run \
    --model=anthropic/claude-sonnet-4.6 \
    --num_workers=4

# Resume an interrupted run
python batch_runner.py \
    --dataset_file=data/prompts.jsonl \
    --batch_size=10 \
    --run_name=my_first_run \
    --resume

# List available toolset distributions
python batch_runner.py --list_distributions
```

:::tip Predictable cost at scale
Пакетні запуски створюють багато одночасних сесій агента, кожна з яких робить виклики моделей та інструментів. Підписка [Nous Portal](/user-guide/features/tool-gateway) об’єднує доступ до моделей, веб‑пошуку, генерації зображень, TTS та хмарних браузерів в одному рахунку — це корисно, коли потрібна стабільна вартість за траєкторію без постійного регулювання лімітів по п’яти обліковим записам постачальників. Налаштуй за допомогою `hermes setup --portal`, а потім вкажи `--model` на модель Nous.
:::

## Формат набору даних

Вхідний набір даних — це файл JSONL (по одному JSON‑об’єкту в рядку). Кожен запис має містити поле `prompt`:

```jsonl
{"prompt": "Write a Python function that finds the longest palindromic substring"}
{"prompt": "Create a REST API endpoint for user authentication using Flask"}
{"prompt": "Debug this error: TypeError: cannot unpack non-iterable NoneType object"}
```

Опціонально можна додати:
- `image` або `docker_image`: образ контейнера, який використовується для пісочниці цього запиту (працює з бекендами Docker, Modal та Singularity)
- `cwd`: перевизначення робочого каталогу для термінальної сесії завдання

## Параметри конфігурації

| Параметр | За замовчуванням | Опис |
|-----------|-------------------|------|
| `--dataset_file` | (обов’язково) | Шлях до JSONL‑набору даних |
| `--batch_size` | (обов’язково) | Кількість запитів у пакеті |
| `--run_name` | (обов’язково) | Назва цього запуску (використовується для каталогу виводу та контрольних точок) |
| `--distribution` | `"default"` | Розподіл набору інструментів для вибірки |
| `--model` | `claude-sonnet-4.6` | Модель, яку слід використовувати |
| `--base_url` | `https://openrouter.ai/api/v1` | Базовий URL API |
| `--api_key` | (змінна оточення) | Ключ API для моделі |
| `--max_turns` | `10` | Максимальна кількість ітерацій викликів інструментів на запит |
| `--num_workers` | `4` | Кількість паралельних процесів |
| `--resume` | `false` | Відновити з контрольної точки |
| `--verbose` | `false` | Увімкнути докладний лог |
| `--max_samples` | all | Обробити лише перші N зразків набору даних |
| `--max_tokens` | model default | Максимальна кількість токенів у відповіді моделі |

### Маршрутизація провайдерів (OpenRouter)

| Параметр | Опис |
|-----------|------|
| `--providers_allowed` | Список провайдерів, розділений комами, які дозволені (наприклад, `"anthropic,openai"`) |
| `--providers_ignored` | Список провайдерів, розділений комами, які ігноруються (наприклад, `"together,deepinfra"`) |
| `--providers_order` | Бажаний порядок провайдерів, розділений комами |
| `--provider_sort` | Сортування за `"price"`, `"throughput"` або `"latency"` |

### Керування міркуваннями

| Параметр | Опис |
|-----------|------|
| `--reasoning_effort` | Рівень зусиль: `none`, `minimal`, `low`, `medium`, `high`, `xhigh` |
| `--reasoning_disabled` | Повністю вимкнути токени міркувань/думання |

### Додаткові параметри

| Параметр | Опис |
|-----------|------|
| `--ephemeral_system_prompt` | Системний промпт, що використовується під час виконання, але **НЕ** зберігається у траєкторіях |
| `--log_prefix_chars` | Кількість символів, що показуються у попередньому перегляді логів (за замовчуванням: 100) |
| `--prefill_messages_file` | Шлях до JSON‑файлу з попередньо заповненими повідомленнями для few‑shot підготовки |

## Розподіли набору інструментів

Кожен запит отримує випадково вибраний набір інструментів з **розподілу**. Це забезпечує, що дані для навчання охоплюють різноманітні комбінації інструментів. Використай `--list_distributions`, щоб переглянути всі доступні розподіли.

У поточній реалізації розподіли задають ймовірність для **кожного окремого інструменту**. Селектор випадково включає або виключає кожен інструмент незалежно, а потім гарантує, що принаймні один інструмент увімкнено. Це відрізняється від вручну створених таблиць готових комбінацій.

## Формат виводу

Увесь вивід записується у `data/<run_name>/`:

```text
data/my_run/
├── trajectories.jsonl    # Combined final output (all batches merged)
├── batch_0.jsonl         # Individual batch results
├── batch_1.jsonl
├── ...
├── checkpoint.json       # Resume checkpoint
└── statistics.json       # Aggregate tool usage stats
```

### Формат траєкторії

Кожен рядок у `trajectories.jsonl` — це JSON‑об’єкт:

```json
{
  "prompt_index": 42,
  "conversations": [
    {"from": "human", "value": "Write a function..."},
    {"from": "gpt", "value": "I'll create that function...",
     "tool_calls": [...]},
    {"from": "tool", "value": "..."},
    {"from": "gpt", "value": "Here's the completed function..."}
  ],
  "metadata": {
    "batch_num": 2,
    "timestamp": "2026-01-15T10:30:00",
    "model": "anthropic/claude-sonnet-4.6"
  },
  "completed": true,
  "partial": false,
  "api_calls": 3,
  "toolsets_used": ["terminal", "file"],
  "tool_stats": {
    "terminal": {"count": 2, "success": 2, "failure": 0},
    "read_file": {"count": 1, "success": 1, "failure": 0}
  },
  "tool_error_counts": {
    "terminal": 0,
    "read_file": 0
  }
}
```

Поле `conversations` використовує формат, схожий на ShareGPT, з полями `from` та `value`. Статистика інструментів нормалізується, включаючи всі можливі інструменти з нульовими значеннями за замовчуванням, що забезпечує узгоджену схему для сумісності з наборами даних HuggingFace.

## Контрольні точки

Запускатор пакетної обробки має надійний механізм контрольних точок для стійкості до збоїв:

- **Файл контрольної точки:** Зберігається після завершення кожного пакету, фіксує, які індекси запитів вже оброблені
- **Відновлення за вмістом:** При `--resume` запускатор сканує існуючі файли пакетів і порівнює завершені запити за їх фактичним текстовим вмістом (а не лише за індексами), що дозволяє відновитися навіть при зміні порядку набору даних
- **Неуспішні запити:** Позначені як завершені лише успішно оброблені запити — неуспішні будуть повторені при відновленні
- **Об’єднання пакетів:** Після завершення всі файли пакетів (включаючи з попередніх запусків) об’єднуються в один `trajectories.jsonl`

### Як працює відновлення

1. Сканувати всі файли `batch_*.jsonl` для знайдених завершених запитів (за збігом вмісту)
2. Відфільтрувати набір даних, виключивши вже завершені запити
3. Переформувати пакети з залишкових запитів
4. Обробити лише залишкові запити
5. Об’єднати всі файли пакетів (старі + нові) у фінальний результат

## Фільтрація якості

Запускатор пакетної обробки застосовує автоматичну фільтрацію якості:

- **Фільтр без міркувань:** Зразки, у яких жодна відповідь асистента не містить міркувань (відсутність `<REASONING_SCRATCHPAD>` або токенів нативного мислення), відкидаються
- **Фільтр пошкоджених записів:** Записи з вигаданими назвами інструментів (яких немає у валідному списку) відфільтровуються під час фінального об’єднання
- **Статистика міркувань:** Відстежується відсоток ходів з/без міркувань протягом усього запуску

## Статистика

Після завершення запускатор виводить розгорнуту статистику:

- **Використання інструментів:** Кількість викликів, успішність/невдачі по кожному інструменту
- **Покриття міркувань:** Відсоток ходів асистента з міркуваннями
- **Відкинуті зразки:** Кількість зразків, відфільтрованих через відсутність міркувань
- **Тривалість:** Загальний час обробки

Статистика також зберігається у `statistics.json` для програмного аналізу.

## Випадки використання

### Генерація даних для навчання

Генеруй різноманітні траєкторії використання інструментів для донастройки:

```bash
python batch_runner.py \
    --dataset_file=data/coding_prompts.jsonl \
    --batch_size=20 \
    --run_name=coding_v1 \
    --model=anthropic/claude-sonnet-4.6 \
    --num_workers=8 \
    --distribution=default \
    --max_turns=15
```

### Оцінка моделей

Оцінюй, наскільки добре модель використовує інструменти за стандартизованими запитами:

```bash
python batch_runner.py \
    --dataset_file=data/eval_suite.jsonl \
    --batch_size=10 \
    --run_name=eval_gpt4 \
    --model=openai/gpt-4o \
    --num_workers=4 \
    --max_turns=10
```

### Контейнерні образи per‑prompt

Для бенчмарків, що вимагають специфічних середовищ, кожен запит може вказувати власний образ контейнера:

```jsonl
{"prompt": "Install numpy and compute eigenvalues of a 3x3 matrix", "image": "python:3.11-slim"}
{"prompt": "Compile this Rust program and run it", "image": "rust:1.75"}
{"prompt": "Set up a Node.js Express server", "image": "node:20-alpine", "cwd": "/app"}
```

Запускатор пакетної обробки перевіряє доступність Docker‑образів перед запуском кожного запиту.
