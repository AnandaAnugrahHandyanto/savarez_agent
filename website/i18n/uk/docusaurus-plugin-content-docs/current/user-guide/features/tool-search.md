# Пошук інструментів

Коли у тебе багато серверів MCP або підключених інструментів‑плагінів, не‑ядерних, їх JSON‑схеми можуть займати значну частину контекстного вікна на кожному кроці — навіть коли лише кілька з них актуальні для того, що користувач дійсно запитав.

**Пошук інструментів** — це прогресивний шар розкриття Hermes, який можна ввімкнути для вирішення цієї проблеми. Коли він активований, інструменти MCP і плагінів замінюються у масиві інструментів, видимих моделі, трьома мостовими інструментами, а модель завантажує схему кожного конкретного інструмента за запитом.

:::info Вбудовані інструменти Hermes ніколи не відкладатимуться
Інструменти, що становлять ядро можливостей Hermes (`terminal`,
`read_file`, `write_file`, `patch`, `search_files`, `todo`, `memory`,
`browser_*`, `web_search`, `web_extract`, `clarify`, `execute_code`,
`delegate_task`, `session_search`, `send_message` та решта
`_HERMES_CORE_TOOLS`) *завжди* завантажуються безпосередньо. Відкладатися можуть лише інструменти MCP та не‑ядерні плагіни.
:::

## Як це працює

Коли Пошук інструментів активується на кроці, модель бачить три нових інструменти замість відкладених:

```
tool_search(query, limit?)     — search the deferred-tool catalog
tool_describe(name)            — load the full schema for one tool
tool_call(name, arguments)     — invoke a deferred tool
```

Типова взаємодія виглядає так:

```
Model: tool_search("create a github issue")
  → { matches: [{ name: "mcp_github_create_issue", ... }, ...] }
Model: tool_describe("mcp_github_create_issue")
  → { parameters: { type: "object", properties: { ... } } }
Model: tool_call("mcp_github_create_issue", { title: "...", body: "..." })
  → { ok: true, issue_number: 42 }
```

Коли модель викликає `tool_call`, Hermes **розгортає міст** і диспетчеризує підлягаючий інструмент точно так, ніби модель викликала його безпосередньо. Хуки до виклику інструмента, захисні бар’єри, запити на схвалення та хуки після виклику працюють з реальним ім’ям інструмента — а не з `tool_call`. Канал активності в CLI та шлюзі також розгортається, тому ти бачиш підлягаючий інструмент, а не міст.

## Коли це активується?

За замовчуванням Пошук інструментів працює в режимі `auto`: він активується лише тоді, коли схеми відкладених інструментів споживали б принаймні 10 % контекстного вікна активної моделі. Нижче цього порогу збірка масиву інструментів є чистим пропуском і не створює накладних витрат.

Це рішення переоцінюється щоразу, коли будується масив інструментів, тому:

- Сесія з лише кількома інструментами MCP і довгою контекстною моделлю ніколи не активує Пошук інструментів.
- Сесія з багатьма підключеними серверами MCP (зазвичай 15 + інструментів) починає його активувати.
- Видалення серверів MCP під час сесії правильно повертає прямий доступ на наступній збірці.

## Налаштування

```yaml
tools:
  tool_search:
    enabled: auto       # auto (default), on, or off
    threshold_pct: 10   # percentage of context — only used in auto mode
    search_default_limit: 5
    max_search_limit: 20
```

| Ключ | За замовчуванням | Значення |
| --- | --- | --- |
| `enabled` | `auto` | `auto` активується вище порогу; `on` завжди активується, якщо є хоча б один відкладений інструмент; `off` вимикає функціональність повністю. |
| `threshold_pct` | `10` | Відсоток довжини контексту, при якому вмикається режим `auto`. Діапазон 0–100. |
| `search_default_limit` | `5` | Кількість результатів, що повертаються, коли модель викликає `tool_search` без параметра `limit`. |
| `max_search_limit` | `20` | Жорстка верхня межа, яку модель може запросити через `limit`. Діапазон 1–50. |

Також можна переключити застарілу булеву форму:

```yaml
tools:
  tool_search: true   # equivalent to {enabled: auto}
```

## Коли НЕ слід використовувати

Пошук інструментів вимагає фіксованих токенних витрат на крок (три схеми мостових інструментів, ~300 токенів) і принаймні одного додаткового раунду (search → describe → call) заради економії на відкладених схемах. Це однозначна вигода, коли у тебе багато інструментів і використовується лише кілька за крок; це накладні витрати, коли інструментів мало.

Режим `auto` за замовчуванням самостійно керує цим. Якщо встановити `enabled: on` без умов, очікуй невелику вартість на крок для невеликих наборів інструментів.

## Компроміси, які залишаються

Вони випливають з інваріанту цілісності кешу підказок — це властивість будь‑якого прогресивного розкриття, а не специфічна реалізація:

- **Один додатковий раунд на «холодних» інструментах.** При першому запиті моделі до відкладеного інструмента вона виконує один‑два зайві виклики, щоб знайти і завантажити схему. Заощадження токенів на статичній стороні реальні, але частина їх повертається під час виконання.
- **Немає вигоди кешу для відкладених схем.** Завантажений результат `tool_describe` потрапляє в історію розмови (тому кешується на наступних кроках), проте він ніколи не користується префіксом кешу системного підказника.
- **Залежність від якості моделі.** Пошук інструментів передбачає, що модель може сформулювати розумний запит пошуку для потрібного інструмента. Менші моделі роблять це гірше; опубліковані дані Anthropic (49 % → 74 % на Opus 4 з/без пошуку інструментів) демонструють перевагу, але й ~26 % точності залишаються помилками пошуку.
- **Редагування набору інструментів анулює кеш.** Додавання або видалення інструмента під час сесії змінює опис мостових інструментів (включаючи кількість відкладених) і каталог, тому кеш підказок анулюється. Це той самий компроміс, що й при будь‑якому редагуванні набору інструментів.

## Деталі реалізації

- **Пошук:** BM25 за токенізованою назвою інструмента + описом + іменами параметрів. При відсутності позитивних результатів BM25 повертає буквальний підрядковий збіг за назвою інструмента, що захищає від випадків дегенерації нульового IDF (наприклад, пошук `"github"` у каталозі, де кожна назва інструмента містить «github»).
- **Каталог без стану між кроками.** Він перебудовується з поточного списку `tool-defs` при кожній збірці — без `Map`, прив’язаного до сесії. Це запобігає багу, коли збережений каталог відстає від живого реєстру інструментів.
- **Каталог обмежений наборами інструментів сесії.** `tool_search`, `tool_describe` і `tool_call` бачать і викликають лише інструменти, які були фактично надані сесії. Підагент, kanban‑worker або шлюз‑сесія, обмежені підмножиною наборів інструментів, не можуть використати міст для виявлення чи виклику інструмента поза цією підмножиною — відкладений каталог є лише відкладеною частиною власних увімкнених/вимкнених наборів інструментів, а не всього реєстру процесу.
- **Немає JS‑пісочниці.** Hermes використовує простіший режим «структурованих інструментів» (search / describe / call як звичайні функції). JS‑пісочниця «code mode», яку пропонують інші реалізації, має великий поверхневий простір; ми її пропускаємо.
