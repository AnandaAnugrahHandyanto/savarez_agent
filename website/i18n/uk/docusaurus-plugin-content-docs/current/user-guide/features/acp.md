# Інтеграція ACP Editor

Hermes Agent може працювати як сервер ACP, дозволяючи редакторам, сумісним з ACP, спілкуватися з Hermes через stdio та відображати:

- повідомлення чату
- активність інструментів
- відмінності у файлах
- команди терміналу
- запити на підтвердження
- потокові фрагменти роздумів/відповідей

ACP підходить, коли потрібно, щоб Hermes поводився як агент, вбудований у редактор, а не як окремий CLI‑інструмент або бот для обміну повідомленнями.
## Що Hermes надає в режимі ACP

Hermes працює з підготовленим набором інструментів `hermes‑acp`, призначеним для робочих процесів редактора. Він включає:

- інструменти роботи з файлами: `read_file`, `write_file`, `patch`, `search_files`
- інструменти терміналу: `terminal`, `process`
- інструменти веб/браузера
- пам’ять, todo, пошук у сесії
- навички
- `execute_code` та `delegate_task`
- vision

Навмисно виключені речі, які не підходять типовому UX редактора, такі як доставка повідомлень та управління cron‑завданнями.
## Встановлення

Встанови Hermes звичайно, а потім додай ACP‑extra:

```bash
pip install -e '.[acp]'
```

Це встановлює залежність `agent-client-protocol` і додає:

- `hermes acp`
- `hermes-acp`
- `python -m acp_adapter`

Для встановлень через реєстр Zed Zed запускає Hermes через офіційний запис реєстру ACP. Цей запис використовує дистрибутив `uvx`, який виконує:

```bash
uvx --from 'hermes-agent[acp]==<version>' hermes-acp
```

Переконайся, що `uv` доступний у `PATH` перед використанням шляху встановлення з реєстру.
## Запуск сервера ACP

Будь‑яка з наведених нижче команд запускає Hermes у режимі ACP:

```bash
hermes acp
```

```bash
hermes-acp
```

```bash
python -m acp_adapter
```

Hermes записує логи у `stderr`, тому `stdout` залишається вільним для трафіку ACP JSON‑RPC.

Для неінтерактивних перевірок:

```bash
hermes acp --version
hermes acp --check
```

### Інструменти браузера (необов’язково)

Інструменти браузера (`browser_navigate`, `browser_click` тощо) залежать від npm‑пакету `agent-browser` та Chromium, які не входять до Python‑wheel. Встанови їх за допомогою:

```bash
hermes acp --setup-browser           # interactive (prompts before ~400 MB download)
hermes acp --setup-browser --yes     # accept the download non-interactively
```

Це автономна команда. Потік `terminal‑auth` реєстру Zed (`hermes acp --setup`) також пропонує ініціалізацію браузера як наступне питання після вибору моделі, тому більшість користувачів ніколи не потребують запуску `--setup-browser` безпосередньо.

Що виконує ця команда:

- Встановлює Node.js 22 LTS у `~/.hermes/node/`, якщо його немає
- `npm install -g agent-browser @askjo/camofox-browser` у цей префікс (sudo не потрібен — `npm`‑овий `--prefix` вказує на користувацький записуваний каталог, яким керує Hermes)
- Встановлює Playwright Chromium або використовує виявлений системний Chrome/Chromium, якщо він доступний

Ініціалізація ідемпотентна — повторний запуск виконується швидко і пропускає вже виконані дії.
## Налаштування редактора

### VS Code

Встанови розширення [ACP Client](https://marketplace.visualstudio.com/items?itemName=formulahendry.acp-client).

Щоб підключитися:

1. Відкрий панель ACP Client у **Activity Bar**.
2. Вибери **Hermes Agent** зі вбудованого списку агентів.
3. Підключись і розпочни чат.

Якщо хочеш визначити Hermes вручну, додай його через налаштування VS Code у `acp.agents`:

```json
{
  "acp.agents": {
    "Hermes Agent": {
      "command": "hermes",
      "args": ["acp"]
    }
  }
}
```

### Zed

Zed v0.221.x і новіші встановлюють зовнішні агенти через офіційний **ACP Registry**.

1. Відкрий панель **Agent**.
2. Натисни **Add Agent** або виконай команду `zed: acp registry`.
3. Знайди **Hermes Agent**.
4. Встанови його і запусти новий потік зовнішнього агента Hermes.

Передумови:

- Спочатку налаштуй облікові дані провайдера Hermes за допомогою `hermes model` або вкажи їх у `~/.hermes/.env` / `~/.hermes/config.yaml`.
- Встанови `uv`, щоб запускатор реєстру міг виконати `uvx --from 'hermes-agent[acp]==<version>' hermes-acp`.

Для локальної розробки, доки запис у реєстрі недоступний, використай власний сервер агента в налаштуваннях Zed:

```json
{
  "agent_servers": {
    "hermes-agent": {
      "type": "custom",
      "command": "hermes",
      "args": ["acp"]
    }
  }
}
```

### JetBrains

Використай плагін, сумісний з ACP, і вкажи йому:

```text
/path/to/hermes-agent/acp_registry
```
## Реєстр маніфест

Офіційна копія метаданих ACP Registry Hermes знаходиться за адресою:

```text
acp_registry/agent.json
acp_registry/icon.svg
```

PR у upstream‑реєстрі копіює ці файли у кореневу директорію `hermes-agent/` в `agentclientprotocol/registry`.

Запис у реєстрі використовує дистрибутив `uvx`, який вказує безпосередньо на випуск `hermes-agent` у PyPI:

```text
uvx --from 'hermes-agent[acp]==<version>' hermes-acp
```

CI реєстру перевіряє, що закріплена версія існує в PyPI, тому `version` у маніфесті та `package` у `uvx` повинні завжди збігатися з `pyproject.toml`. `scripts/release.py` автоматично підтримує їх у синхронному стані.
## Конфігурація та облікові дані

ACP‑режим використовує ту ж конфігурацію Hermes, що й CLI:

- `~/.hermes/.env`
- `~/.hermes/config.yaml`
- `~/.hermes/skills/`
- `~/.hermes/state.db`

Визначення провайдера користується звичайним runtime‑резолвером Hermes, тому ACP успадковує поточний налаштований провайдер та облікові дані. Hermes також надає метод автентифікації в терміналі (`--setup`) для клієнтів реєстру під час першого запуску; він відкриває інтерактивне налаштування моделі/провайдера Hermes.
## Поведінка сесії

ACP‑сесії відстежуються менеджером сесій у пам’яті адаптера ACP, доки сервер працює.

Кожна сесія зберігає:

- ідентифікатор сесії
- робочий каталог
- обрану модель
- поточну історію розмови
- подію скасування

Базовий `AIAgent` все ще використовує звичайні шляхи збереження/логування Hermes, проте `list/load/resume/fork` ACP прив’язані до процесу запущеного в даний момент ACP‑сервера.
## Поведінка робочого каталогу

ACP‑сесії прив’язують cwd редактора до ідентифікатора завдання Hermes, тому інструменти роботи з файлами та терміналом виконуються відносно робочого простору редактора, а не cwd процесу сервера.
## Approvals

Небезпечні термінальні команди можуть бути перенаправлені назад до редактора у вигляді запитів на підтвердження. Параметри підтвердження ACP простіші, ніж у CLI‑потоку:

- allow once
- allow always
- deny

При тайм‑ауті або помилці місток підтвердження відхиляє запит.

### Session‑scoped edit auto‑approval

ACP додає третій рівень між *allow once* і *allow always*: **Allow for session**. Вибір цього варіанту у запиті дозволу редактора записує підтвердження лише в межах поточної сесії ACP — кожен наступний відповідний виклик інструменту в цій сесії виконується без запиту, але нова сесія ACP (або перезапуск редактора) скидає налаштування і знову запитує вперше.

| Option | Editor label | Scope | Persisted across restarts |
|---|---|---|---|
| `allow_once` | Allow once | This one tool call | No |
| `allow_session` | Allow for session | All matching calls in this ACP session | No — cleared when the session ends |
| `allow_always` | Allow always | All future sessions | Yes (written to the Hermes permanent allowlist) |
| `deny` | Deny | This one tool call | No |

`allow_session` — правильний варіант за замовчуванням для робочого процесу редактора, коли ти довіряєш агенту протягом виконання завдання, але не хочеш створювати довготривале запис у allowlist. Компроміс безпеки очевидний: чим ширший спектр, тим рідше редактор буде переривати тебе, і тим більше шкоди може завдати неправильно працюючий агент (або ін’єкція запиту), перш ніж ти це помітиш. Починай з `allow_once` для незнайомих команд; переходь до `allow_session`, коли побачиш, що агент правильно виконує той самий шаблон кілька разів; залишай `allow_always` лише для дійсно ідемпотентних команд, яким ти довіряєш назавжди (наприклад, `git status`).

Місток ACP відображає ці варіанти на внутрішню семантику підтверджень Hermes — `allow_always` записує постійний запис у allowlist так само, як це робить CLI, тоді як `allow_session` впливає лише на кеш підтверджень у процесі поточної сесії ACP.
## Устранення проблем

### ACP‑агент не з’являється в редакторі

Перевір:

- У Zed відкрий реєстр ACP за допомогою `zed: acp registry` і пошукай **Hermes Agent**.
- Для ручної/локальної розробки переконайся, що власна команда `agent_servers` вказує на `hermes acp`.
- Hermes встановлений і доступний у твоєму PATH.
- Додаток ACP встановлений (`pip install -e '.[acp]'`).
- `uv` встановлений, якщо запускати з офіційного запису реєстру Zed.

### ACP запускається, але одразу помилкує

Спробуй такі перевірки:

```bash
hermes acp --version
hermes acp --check
hermes doctor
hermes status
```

### Відсутні облікові дані

Режим ACP використовує існуючу конфігурацію провайдера Hermes. Налаштуй облікові дані за допомогою:

```bash
hermes model
```

або відредагувавши `~/.hermes/.env`. Клієнти реєстру також можуть ініціювати термінальний процес автентифікації Hermes, який запускає той самий інтерактивний налаштування провайдера/моделі.

### Запускатор реєстру Zed не може знайти `uv`

Встанови `uv` згідно офіційної документації з встановлення `uv`, потім повтори спробу запуску Hermes Agent з Zed.
