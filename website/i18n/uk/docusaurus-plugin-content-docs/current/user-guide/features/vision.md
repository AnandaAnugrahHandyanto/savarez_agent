# Vision & Image Paste

Hermes Agent підтримує **мультимодальне бачення** — ти можеш вставляти зображення з буфера обміну безпосередньо в CLI і просити агента проаналізувати, описати або працювати з ними. Зображення надсилаються моделі як блоки вмісту, закодовані у base64, тому будь‑яка модель, що підтримує зір, може їх обробляти.

:::tip
Підписники Portal отримують моделі з можливістю зору (Claude, GPT‑5, Gemini) в тому ж каталозі — без додаткових облікових даних. Дивись [Nous Portal](/integrations/nous-portal).
:::
## Як це працює

1. Скопіюй зображення у буфер обміну (скріншот, зображення в браузері тощо)
2. Прикріпи його одним із способів нижче
3. Введи своє питання та натисни **Enter**
4. Зображення з’явиться у вигляді бейджу `[📎 Image #1]` над полем вводу
5. Після відправки зображення надсилається моделі як блок візуального контенту

Ти можеш прикріпити кілька зображень перед відправкою — кожне отримує свій бейдж. Натисни `Ctrl+C`, щоб очистити всі прикріплені зображення.

Зображення зберігаються у `~/.hermes/images/` у форматі PNG з іменами файлів, що містять мітку часу.
## Методи вставки

Як ти додаєш зображення, залежить від середовища терміналу. Не всі методи працюють скрізь — ось повний огляд:

### Команда `/paste`

**Найнадійніший явний запасний (варіант) прикріплення зображення.**

```
/paste
```

Введи `/paste` і натисни Enter. Hermes перевіряє буфер обміну на наявність зображення та прикріплює його. Це найбезпечніший варіант, коли твій термінал переписує `Cmd+V`/`Ctrl+V`, або коли ти скопіював лише зображення і немає текстового навантаження у вигляді bracketed‑paste для аналізу.

### Ctrl+V / Cmd+V

Hermes тепер обробляє вставку як багаторівневий процес:
- спочатку вставка звичайного тексту
- резервний варіант native clipboard / OSC52 тексту, якщо термінал не передав текст чисто
- прикріплення зображення, коли буфер обміну або вставлене навантаження розпізнається як зображення або шлях до зображення

Це означає, що тимчасові шляхи скріншотів macOS та URI зображень `file://...` можуть одразу прикріплюватися замість того, щоб залишатися у редакторі як сирий текст.

:::warning
Якщо у твоєму буфері обміну **тільки зображення** (без тексту), термінали все одно не можуть безпосередньо передати бінарні байти зображення. Використовуй `/paste` як явний запасний (варіант) прикріплення зображення.
:::

### `/terminal-setup` для VS Code / Cursor / Windsurf

Якщо ти запускаєш TUI у локальному інтегрованому терміналі сімейства VS Code на macOS, Hermes може встановити рекомендовані прив’язки `workbench.action.terminal.sendSequence` для кращої підтримки багаторядкових вводів та паритету undo/redo:

```text
/terminal-setup
```

Це особливо корисно, коли `Cmd+Enter`, `Cmd+Z` або `Shift+Cmd+Z` перехоплюються IDE. Запусти це лише на локальній машині — не всередині SSH‑сесії.
## Сумісність платформи

| Середовище | `/paste` | Cmd/Ctrl+V | `/terminal-setup` | Примітки |
|---|:---:|:---:|:---:|---|
| **macOS Terminal / iTerm2** | ✅ | ✅ | n/a | Найкращий досвід — рідний буфер обміну + відновлення шляху до скріншоту |
| **Apple Terminal** | ✅ | ✅ | n/a | Якщо Cmd+←/→/⌫ переписується, використай запасні варіанти Ctrl+A / Ctrl+E / Ctrl+U |
| **Linux X11 desktop** | ✅ | ✅ | n/a | Потрібен `xclip` (`apt install xclip`) |
| **Linux Wayland desktop** | ✅ | ✅ | n/a | Потрібен `wl-paste` (`apt install wl-clipboard`) |
| **WSL2 (Windows Terminal)** | ✅ | ✅ | n/a | Використовує `powershell.exe` — додаткове встановлення не потрібно |
| **VS Code / Cursor / Windsurf (local)** | ✅ | ✅ | ✅ | Рекомендовано для кращої паритету Cmd+Enter / undo / redo |
| **VS Code / Cursor / Windsurf (SSH)** | ❌² | ❌² | ❌³ | Запусти `/terminal-setup` на локальній машині замість цього |
| **SSH terminal (any)** | ❌² | ❌² | n/a | Віддалений буфер обміну недоступний |

² Дивись [SSH & Remote Sessions](#ssh--remote-sessions) нижче
³ Команда записує локальні прив’язки клавіш IDE і не повинна виконуватись з віддаленого хоста
## Налаштування для конкретної платформи

### macOS

**Немає необхідності в налаштуванні.** Hermes використовує `osascript` (вбудований у macOS) для читання буфера обміну. Для підвищення швидкості, за бажанням встанови `pngpaste`:

```bash
brew install pngpaste
```

### Linux (X11)

Встанови `xclip`:

```bash
# Ubuntu/Debian
sudo apt install xclip

# Fedora
sudo dnf install xclip

# Arch
sudo pacman -S xclip
```

### Linux (Wayland)

Сучасні Linux‑десктопи (Ubuntu 22.04+, Fedora 34+) часто використовують Wayland за замовчуванням. Встанови `wl-clipboard`:

```bash
# Ubuntu/Debian
sudo apt install wl-clipboard

# Fedora
sudo dnf install wl-clipboard

# Arch
sudo pacman -S wl-clipboard
```

:::tip Як перевірити, чи ти працюєш у Wayland
```bash
echo $XDG_SESSION_TYPE
# "wayland" = Wayland, "x11" = X11, "tty" = no display server
```
:::

### WSL2

**Додаткове налаштування не потрібне.** Hermes автоматично виявляє WSL2 (через `/proc/version`) і використовує `powerscript.exe` для доступу до буфера обміну Windows через .NET `System.Windows.Forms.Clipboard`. Це вбудовано в інтероп Windows у WSL2 — `powerscript.exe` доступний за замовчуванням.

Дані буфера обміну передаються у вигляді base64‑закодованого PNG через stdout, тому перетворення шляхів файлів або створення тимчасових файлів не потрібно.

:::info Примітка щодо WSLg
Якщо ти використовуєш WSLg (WSL2 з підтримкою GUI), Hermes спочатку пробує шлях PowerShell, а потім переходить до `wl-paste`. Місток буфера обміну WSLg підтримує лише формат BMP для зображень — Hermes автоматично конвертує BMP у PNG за допомогою Pillow (якщо встановлено) або команди `convert` з ImageMagick.
:::

#### Перевірка доступу до буфера обміну у WSL2

```bash
# 1. Check WSL detection
grep -i microsoft /proc/version

# 2. Check PowerShell is accessible
which powershell.exe

# 3. Copy an image, then check
powershell.exe -NoProfile -Command "Add-Type -AssemblyName System.Windows.Forms; [System.Windows.Forms.Clipboard]::ContainsImage()"
# Should print "True"
```
## SSH & Remote Sessions

**Вставка зображення з буфера обміну не працює повністю через SSH.** Коли ти підключаєшся по SSH до віддаленої машини, Hermes CLI працює на віддаленому хості. Інструменти буфера обміну (`xclip`, `wl-paste`, `powershell.exe`, `osascript`) читають буфер обміну машини, на якій вони запущені — тобто віддаленого сервера, а не твоєї локальної машини. Твоє локальне зображення в буфері обміну тому недоступне з віддаленої сторони.

Текст іноді все ж може передаватися через вставку в терміналі або OSC52, але доступ до зображень у буфері обміну та локальні тимчасові шляхи скріншотів залишаються прив'язаними до машини, на якій запущений Hermes.

### Обхідні шляхи для SSH

1. **Завантажити файл зображення** — Збережи зображення локально, завантаж його на віддалений сервер за допомогою `scp`, файлового провідника VSCode (перетягування) або будь‑якого іншого методу передачі файлів. Потім посилайся на нього за шляхом. *(Команда `/attach <filepath>` запланована для майбутнього випуску.)*

2. **Використати URL** — Якщо зображення доступне в інтернеті, просто встав URL у своє повідомлення. Агент може використати `vision_analyze` для перегляду будь‑якого URL‑зображення безпосередньо.

3. **Перенаправлення X11** — Підключайся з `ssh -X` для перенаправлення X11. Це дозволяє `xclip` на віддаленій машині отримати доступ до твого локального буфера X11. Потрібен запущений X‑сервер локально (XQuartz на macOS, вбудований у Linux‑десктопах X11). Повільно для великих зображень.

4. **Використати платформу обміну повідомленнями** — Надсилай зображення Hermes через Telegram, Discord, Slack або WhatsApp. Ці платформи нативно підтримують завантаження зображень і не підлягають обмеженням буфера обміну/терміналу.
## Чому термінали не можуть вставляти зображення

Це поширене джерело плутанини, тому ось технічне пояснення:

Термінали — це **текстові** інтерфейси. Коли ти натискаєш `Ctrl+V` (або `Cmd+V`), емулятор терміналу:

1. Читає буфер обміну на предмет **текстового вмісту**
2. Обгортає його у [bracketed paste](https://en.wikipedia.org/wiki/Bracketed-paste) escape‑послідовності
3. Надсилає його застосунку через текстовий потік терміналу

Якщо буфер обміну містить лише зображення (без тексту), термінал не має чого надсилати. Не існує стандартної escape‑послідовності терміналу для бінарних даних зображення. Термінал просто нічого не робить.

Саме тому Hermes використовує окрему перевірку буфера обміну — замість отримання даних зображення через подію вставки в терміналі, він викликає інструменти ОС (`osascript`, `powershell.exe`, `xclip`, `wl-paste`) безпосередньо через `subprocess`, щоб прочитати буфер обміну незалежно.
## Підтримувані моделі

Вставка зображення працює з будь‑якою моделлю, що підтримує зір. Зображення надсилається у вигляді base64‑закодованого data URL у форматі вмісту OpenAI Vision:

```json
{
  "type": "image_url",
  "image_url": {
    "url": "data:image/png;base64,..."
  }
}
```

Більшість сучасних моделей підтримують цей формат, включаючи GPT‑4 Vision, Claude (з зіром), Gemini та open‑source мультимодальні моделі, які обслуговуються через OpenRouter.
## Маршрутизація зображень (Vision‑Capable vs Text‑Only Models)

Коли користувач додає зображення — з буфера CLI, через шлюз (Telegram/Discord фото) або будь‑якої іншої точки входу — Hermes маршрутизує його залежно від того, чи підтримує твоя поточна модель зору:

| Твоя модель | Що відбувається з зображенням |
|---|---|
| **Vision‑capable** (GPT‑4V, Claude with vision, Gemini, Qwen‑VL, MiMo‑VL тощо) | Надсилається як **реальні пікселі** у рідному форматі зображення провайдера, зазначеному вище. Без шару текстового резюме. |
| **Text‑only** (DeepSeek V3, менші open‑source моделі, старі chat‑only кінцеві точки) | Маршрутується через допоміжний інструмент `vision_analyze` — допоміжна модель зору описує зображення, і текстовий опис вставляється в розмову. |

Ти не налаштовуєш це — Hermes читає можливості твоєї поточної моделі з метаданих провайдера і автоматично вибирає правильний шлях. Практичний ефект: можна перемикатися між моделями з підтримкою зору та без неї під час сесії, і обробка зображень «просто працює» без зміни робочого процесу. Text‑only моделі отримують зв’язний контекст про зображення замість пошкодженого мультимодального навантаження, яке вони мали б відхилити.

Яку допоміжну модель зору використовувати для шляху текстового опису, можна налаштувати у `auxiliary.vision` — дивись [Auxiliary Models](/user-guide/configuration#auxiliary-models).

### `vision_analyze` має таку ж подвійну поведінку

Інструмент `vision_analyze` сам діє за тим же принципом маршрутизації. Коли активна основна модель підтримує зору **і** її провайдер підтримує вміст зображень у результатах інструменту (на даний момент стеки Anthropic, OpenAI, Azure‑OpenAI та Gemini 3.x), `vision_analyze` обходить допоміжний описувач і повертає сирі пікселі зображення у вигляді мультимодального результату інструменту. Основна модель бачить зображення нативно на наступному кроці — без виклику допоміжного інструменту, без втрати інформації про текстове резюме, без додаткової затримки.

Для Text‑only основних моделей (або провайдерів, чиї канали результатів інструменту не передають зображення), `vision_analyze` повертається до старого шляху: він запитує налаштовану допоміжну модель зору, щоб описати зображення, і повертає опис як простий текст. У будь‑якому випадку підпис інструменту залишається тим же — інструмент вирішує, який шлях обрати під час виконання, виходячи з активної моделі.
