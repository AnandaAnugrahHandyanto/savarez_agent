# Honcho Memory

[Honcho](https://github.com/plastic-labs/honcho) — це AI‑нативний бекенд пам’яті, який додає діалектичне мислення та глибоке моделювання користувача поверх вбудованої системи пам’яті Hermes. Замість простого сховища ключ‑значення Honcho підтримує поточну модель користувача — його уподобання, стиль спілкування, цілі та шаблони — шляхом роздумів над розмовами після їх завершення.

:::info Honcho is a Memory Provider Plugin
Honcho інтегровано в систему [Memory Providers](./memory-providers.md). Усі функції нижче доступні через уніфікований інтерфейс провайдера пам’яті.
:::
## Що додає Honcho

| Capability | Built-in Memory | Honcat |
|-----------|----------------|--------|
| Персистентність між сесіями | ✔ File-based MEMORY.md/USER.md | ✔ Server-side with API |
| Профіль користувача | ✔ Manual agent curation | ✔ Automatic dialectic reasoning |
| Підсумок сесії | — | ✔ Session-scoped context injection |
| Ізоляція багатагентних | — | ✔ Per‑peer profile separation |
| Режими спостереження | — | ✔ Unified or directional observation |
| Висновки (отримані інсайти) | — | ✔ Server‑side reasoning about patterns |
| Пошук по історії | ✔ FTS5 session search | ✔ Semantic search over conclusions |

**Dialectic reasoning**: Після кожного обміну (регульованого `dialecticCadence`) Honcho аналізує діалог і отримує інсайти про уподобання, звички та цілі користувача. Ці інсайти накопичуються з часом, надаючи агенту все глибше розуміння, що виходить за межі того, що користувач явно сказав. Діалектичне міркування підтримує багатократний прохід (1–3 проходи) з автоматичним вибором холодної/теплої підказки — запити холодного старту орієнтовані на загальні факти про користувача, а теплі запити пріоритетно використовують контекст, прив’язаний до сесії.

**Session‑scoped context**: Базовий контекст тепер включає підсумок сесії разом із представленням користувача та карткою співрозмовника. Це дає агенту усвідомлення того, що вже обговорювалося в поточній сесії, зменшуючи повторення та забезпечуючи безперервність.

**Multi‑agent profiles**: Коли кілька інстанцій Hermes спілкуються з одним користувачем (наприклад, помічник‑програміст і персональний помічник), Honcho підтримує окремі профілі «peer». Кожен peer бачить лише свої спостереження та висновки, запобігаючи перехресному забрудненню контексту.
## Налаштування

```bash
hermes memory setup    # select "honcho" from the provider list
```

Або налаштуй вручну:

```yaml
# ~/.hermes/config.yaml
memory:
  provider: honcho
```

```bash
echo 'HONCHO_API_KEY=***' >> ~/.hermes/.env
```

Отримай API‑ключ за адресою [honcho.dev](https://honcho.dev).
## Архітектура

### Двошаровий інжекційний контекст

Кожен хід (у режимі `hybrid` або `context`), Honcho збирає два шари контексту, які інжектуються в системний підказник:

1. **Базовий контекст** — підсумок сесії, представлення користувача, картка користувача‑пір, самопредставлення ШІ та ідентифікаційна картка ШІ. Оновлюється згідно `contextCadence`. Це шар «хто цей користувач».
2. **Діалектичне доповнення** — синтезоване LLM‑міркування про поточний стан і потреби користувача. Оновлюється згідно `dialecticCadence`. Це шар «що важливо саме зараз».

Обидва шари конкатенуються та обрізаються до бюджету `contextTokens` (за потреби).

### Вибір холодного/теплого підказника

Діалектика автоматично обирає між двома стратегіями підказника:

- **Холодний старт** (ще немає базового контексту): Загальний запит — «Хто ця особа? Які її уподобання, цілі та стиль роботи?»
- **Тепла сесія** (базовий контекст існує): Запит у межах сесії — «Виходячи з того, що вже обговорювалося в цій сесії, який контекст про цього користувача є найрелевантнішим?»

Це відбувається автоматично, залежно від того, чи заповнено базовий контекст.

### Три ортогональні налаштувальні ручки

Вартість і глибина контролюються трьома незалежними ручками:

| Ручка | Керує | За замовчуванням |
|------|----------|-------------------|
| `contextCadence` | Кількість ходів між викликами API `context()` (оновлення базового шару) | `1` |
| `dialecticCadence` | Кількість ходів між викликами LLM `peer.chat()` (оновлення діалектичного шару) | `2` (рекомендовано 1–5) |
| `dialecticDepth` | Кількість проходів `.chat()` за одне діалектичне виклику (1–3) | `1` |

Ці ручки ортогональні — можна часто оновлювати базовий контекст і рідко запускати діалектику, або ж виконувати глибоку багатопрохідну діалектику при низькій частоті. Приклад: `contextCadence: 1, dialecticCadence: 5, dialecticDepth: 2` оновлює базовий контекст кожен хід, запускає діалектику кожні 5 ходів, і кожен запуск діалектики робить 2 проходи.

### Глибина діалектики (багатопрохідна)

Коли `dialecticDepth` > 1, кожен діалектичний виклик виконує кілька проходів `.chat()`:

- **Прохід 0**: Холодний або теплий підказник (див. вище)
- **Прохід 1**: Самоаудит — виявляє прогалини у початковій оцінці та синтезує докази з недавніх сесій
- **Прохід 2**: Узгодження — перевіряє протиріччя між попередніми проходами та створює фінальний синтез

Кожен прохід використовує пропорційний рівень міркування (легший на ранніх проходах, базовий рівень для головного проходу). Рівні можна перевизначити per‑pass за допомогою `dialecticDepthLevels` — напр., `["minimal", "medium", "high"]` для трьохпрохідного запуску.

Проходи можуть завершитися раніше, якщо попередній прохід повернув сильний сигнал (довгий, структурований вивід), тому глибина 3 не завжди означає 3 виклики LLM.

### Попереднє прогрівання на старті сесії

При ініціалізації сесії Honcho запускає діалектичний виклик у фоні з повною налаштованою `dialecticDepth` і передає результат безпосередньо до складання контексту першого ходу. Однопрохідне прогрівання на холодному пірі часто повертає скудний вивід — багатопрохідна глибина виконує цикл аудит/узгодження ще до того, як користувач щось скаже. Якщо прогрівання не встигло завершитися до першого ходу, перший хід переходить у синхронний виклик з обмеженим тайм‑аутом.

### Адаптивний рівень міркування за запитом

Автоматично інжектована діалектика масштабує `dialecticReasoningLevel` залежно від довжини запиту: +1 рівень при ≥120 символах, +2 при ≥400, обмежено `reasoningLevelCap` (за замовчуванням `"high"`). Вимкнути можна за допомогою `reasoningHeuristic: false`, щоб зафіксувати кожен автоматичний виклик на `dialecticReasoningLevel`. Доступні рівні: `minimal`, `low`, `medium`, `high`, `max`.
## Параметри конфігурації

Honcho налаштовується у `~/.honcho/config.json` (глобально) або `$HERMES_HOME/honcho.json` (локально для профілю). Майстер налаштувань виконує це за тебе.

### Повна довідка по конфігурації

| Ключ | За замовчуванням | Опис |
|-----|-------------------|------|
| `contextTokens` | `null` (без обмеження) | Бюджет токенів для автоматично впровадженого контексту за хід. Встанови ціле число (наприклад 1200), щоб обмежити. Обрізає на межах слів |
| `contextCadence` | `1` | Мінімальна кількість ходів між викликами API `context()` (оновлення базового шару) |
| `dialecticCadence` | `2` | Мінімальна кількість ходів між викликами LLM `peer.chat()` (діалектичний шар). Рекомендовано 1–5. У режимі `tools` не має значення — виклики моделі здійснюються явно |
| `dialecticDepth` | `1` | Кількість проходів `.chat()` за одне діалектичне викликання. Обмежено 1–3 |
| `dialecticDepthLevels` | `null` | Необов’язковий масив рівнів міркувань за проходом, напр. `["minimal", "low", "medium"]`. Перезаписує пропорційні значення за замовчуванням |
| `dialecticReasoningLevel` | `'low'` | Базовий рівень міркувань: `minimal`, `low`, `medium`, `high`, `max` |
| `dialecticDynamic` | `true` | Якщо `true`, модель може перепризначати рівень міркувань для кожного виклику через параметр інструмента |
| `dialecticMaxChars` | `600` | Максимальна кількість символів діалектичного результату, що впроваджується у системний запит |
| `recallMode` | `'hybrid'` | `hybrid` (авто‑впровадження + інструменти), `context` (лише впровадження), `tools` (лише інструменти) |
| `writeFrequency` | `'async'` | Коли здійснюється запис повідомлень: `async` (фоновий потік), `turn` (синхронно), `session` (пакетно в кінці) або ціле число N |
| `saveMessages` | `true` | Чи зберігати повідомлення у API Honcho |
| `observationMode` | `'directional'` | `directional` (все увімкнено) або `unified` (спільний пул). Перезаписується об’єктом `observation` для детального керування |
| `messageMaxChars` | `25000` | Максимальна кількість символів у повідомленні, що надсилається через `add_messages()`. При перевищенні розбивається на частини |
| `dialecticMaxInputChars` | `10000` | Максимальна кількість символів у вхідному запиті діалектики для `peer.chat()` |
| `sessionStrategy` | `'per-directory'` | `per-directory`, `per-repo`, `per-session` або `global` |

**Стратегія сесії** визначає, як сесії Honcho співвідносяться з твоєю роботою:
- `per-session` — кожен запуск `hermes` отримує нову сесію. Чистий старт, пам’ять через інструменти. Рекомендовано новачкам.
- `per-directory` — одна сесія Honcho на робочий каталог. Контекст накопичується між запусками.
- `per-repo` — одна сесія на git‑репозиторій.
- `global` — одна спільна сесія для всіх каталогів.

**Режим recall** визначає, як пам’ять потрапляє у розмови:
- `hybrid` — контекст автоматично впроваджується у системний запит **і** інструменти доступні (модель вирішує, коли їх запитувати).
- `context` — лише авто‑впровадження, інструменти приховані.
- `tools` — лише інструменти, без авто‑впровадження. Агент має явно викликати `honcho_reasoning`, `honcho_search` тощо.

**Налаштування за режимом recall:**

| Налаштування | `hybrid` | `context` | `tools` |
|---------------|----------|-----------|--------|
| `writeFrequency` | записує повідомлення | записує повідомлення | записує повідомлення |
| `contextCadence` | контролює оновлення базового контексту | контролює оновлення базового контексту | не має значення — без впровадження |
| `dialecticCadence` | контролює авто‑виклики LLM | контролює авто‑виклики LLM | не має значення — виклики моделі явно |
| `dialecticDepth` | багаторазовий прохід за викликанням | багаторазовий прохід за викликанням | не має значення — виклики моделі явно |
| `contextTokens` | обмежує впровадження | обмежує впровадження | не має значення — без впровадження |
| `dialecticDynamic` | дозволяє перепризначення моделлю | N/A (без інструментів) | дозволяє перепризначення моделлю |

У режимі `tools` модель повністю контролює процес — вона викликає `honcho_reasoning`, коли хоче, з будь‑яким `reasoning_level`. Параметри cadence та бюджету застосовуються лише до режимів з авто‑впровадженням (`hybrid` та `context`).
## Спостереження (Directional vs. Unified)

Honcho моделює розмову як обмін повідомленнями між партнерами. Кожен партнер має два перемикачі спостереження, які 1:1 відповідають `SessionPeerConfig` Honcho:

| Перемикач | Ефект |
|--------|--------|
| `observeMe` | Honcho створює представлення цього партнера на основі його власних повідомлень |
| `observeOthers` | Цей партнер спостерігає повідомлення іншого партнера (забезпечує крос‑партнерське міркування) |

Два партнери × два перемикачі = чотири прапорці. `observationMode` — це скорочений пресет:

| Пресет | Прапорці користувача | Прапорці ШІ | Семантика |
|--------|-----------|----------|-----------|
| `"directional"` (за замовчуванням) | я: увімкнено, інші: увімкнено | я: увімкнено, інші: увімкнено | Повне взаємне спостереження. Дозволяє крос‑партнерську діалектику — «що ШІ знає про користувача, базуючись на тому, що сказав користувач і на відповіді ШІ». |
| `"unified"` | я: увімкнено, інші: вимкнено | я: вимкнено, інші: увімкнено | Семантика спільного пулу — ШІ спостерігає лише повідомлення користувача, користувач лише самостійно моделює себе. Пул з одним спостерігачем. |

Перевизначити пресет можна за допомогою явного блоку `observation` для керування окремими партнерами:

```json
"observation": {
  "user": { "observeMe": true,  "observeOthers": true },
  "ai":   { "observeMe": true,  "observeOthers": false }
}
```

Типові шаблони:

| Намір | Конфігурація |
|--------|--------|
| Повне спостереження (більшість користувачів) | `"observationMode": "directional"` |
| ШІ не має повторно моделювати користувача за власними відповідями | `"ai": {"observeMe": true, "observeOthers": false}` |
| Сильна персона, яку ШІ‑партнер не повинен оновлювати через самоспостереження | `"ai": {"observeMe": false, "observeOthers": true}` |

Перемикачі на боці сервера, встановлені через [Honcho dashboard](https://app.honcho.dev), мають пріоритет над локальними значеннями за замовчуванням — Hermes синхронізує їх під час ініціалізації сесії.
## Інструменти

Коли Honcho активний як постачальник пам'яті, доступні п'ять інструментів:

| Tool | Призначення |
|------|--------------|
| `honcho_profile` | Прочитати або оновити картку співрозмовника — передай `card` (список фактів) для оновлення, не вказуй параметр для читання |
| `honcho_search` | Семантичний пошук по контексту — сирі уривки, без синтезу LLM |
| `honcho_context` | Повний контекст сесії — підсумок, представлення, картка, останні повідомлення |
| `honcho_reasoning` | Синтезована відповідь від LLM Honcho — передай `reasoning_level` (minimal/low/medium/high/max) для керування глибиною |
| `honcho_conclude` | Створити або видалити висновки — передай `conclusion` для створення, `delete_id` для видалення (лише PII) |
## CLI Commands

Підкоманда `hermes honcho` **реєструється лише коли Honcho є активним провайдером пам'яті** (`memory.provider: honcho` у `config.yaml`). Запусти `hermes memory setup` і спочатку обери Honcho; підкоманда з'явиться при наступному виклику.

```bash
hermes honcho status          # Connection status, config, and key settings
hermes honcho setup           # Redirects to `hermes memory setup`
hermes honcho strategy        # Show or set session strategy (per-session/per-directory/per-repo/global)
hermes honcho peer            # Show or update peer names + dialectic reasoning level
hermes honcho mode            # Show or set recall mode (hybrid/context/tools)
hermes honcho tokens          # Show or set token budget for context and dialectic
hermes honcho identity        # Seed or show the AI peer's Honcho identity
hermes honcho sync            # Sync Honcho config to all existing profiles
hermes honcho peers           # Show peer identities across all profiles
hermes honcho sessions        # List known Honcho session mappings
hermes honcho map             # Map current directory to a Honcho session name
hermes honcho enable          # Enable Honcho for the active profile
hermes honcho disable         # Disable Honcho for the active profile
hermes honcho migrate         # Step-by-step migration guide from openclaw-honcho
```
## Перехід з `hermes honcho`

Якщо ти раніше користувався автономним `hermes honcho setup`:

1. Твоя існуюча конфігурація (`honcho.json` або `~/.honcho/config.json`) зберігається
2. Твої серверні дані (пам'ять, висновки, профілі користувачів) залишаються недоторканими
3. Встанови `memory.provider: honcho` у `config.yaml`, щоб повторно активувати

Не потрібно повторно входити в систему чи виконувати налаштування. Запусти `hermes memory setup` і вибери «honcho» — майстер виявить твою існуючу конфігурацію.
