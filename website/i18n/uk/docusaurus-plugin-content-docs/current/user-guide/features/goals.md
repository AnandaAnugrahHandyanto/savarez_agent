# Постійні цілі (`/goal`)

`/goal` дає Hermes **постійну ціль**, яка зберігається між ходами. Після кожного ходу легка модель‑суддя перевіряє, чи задоволена ціль останньою відповіддю асистента. Якщо ні, Hermes автоматично подає запит продовження назад у ту ж сесію і продовжує працювати — доки ціль не буде досягнута, ти не **призупиниш** або не **очистиш** її, або не вичерпається ліміт ходів.

Це наш варіант **циклу Ральфа**, безпосередньо натхненний [/goal](/goal) у [Codex CLI 0.128.0](https://github.com/openai/codex) від Еріка Траута (OpenAI). Основна ідея — тримати ціль живою між ходами і не зупинятись, доки вона не буде досягнута — їхня. Реалізація тут незалежна та адаптована до архітектури Hermes.
## Коли це використовувати

Використовуй `/goal` для завдань, коли хочеш, щоб Hermes ітерував самостійно, не перепитуючи тебе кожен хід:

- «Виправити всі помилки lint у `src/` і переконатися, що `ruff check` проходить»
- «Перенести функціональність X з репозиторію Y, включно з тестами, і отримати зелений статус CI»
- «Дослідити, чому ідентифікатори сесій іноді відхиляються під час компресії в процесі виконання, і підготувати звіт»
- «Створити невеликий CLI для перейменування файлів за їх EXIF‑датами, потім протестувати його в папці `photos/`»

Завдання, у яких агент виконує один хід і зупиняється, не потребують `/goal`. Завдання, де *треба було б сказати «продовжуй» тричі*, саме тут проявляють свою користь.
## Quick start

```
/goal Fix every failing test in tests/hermes_cli/ and make sure scripts/run_tests.sh passes for that directory
```

Що ти побачиш:

1. **Goal accepted** — `⊙ Goal set (20-turn budget): <your goal>`
2. **Turn 1 runs** — Hermes starts working as if you'd sent the goal as a normal message.
3. **Judge runs** — after the turn, the judge model decides `done` or `continue`.
4. **Loop fires if needed** — if `continue`, you'll see `↻ Continuing toward goal (1/20): <judge's reason>` and Hermes takes the next step automatically.
5. **Terminates** — eventually you see either `✓ Goal achieved: <reason>` or `⏸ Goal paused — N/20 turns used`.
## Команди

| Команда | Що вона робить |
|---|---|
| `/goal <text>` | Встановити (або замінити) постійну ціль. Одразу запускає перший хід, тому не потрібно надсилати окреме повідомлення. |
| `/goal` або `/goal status` | Показати поточну ціль, її статус і використані ходи. |
| `/goal pause` | Зупинити цикл автоматичного продовження без очищення цілі. |
| `/goal resume` | Відновити цикл (скидає лічильник ходів до нуля). |
| `/goal clear` | Повністю видалити ціль. |

Працює ідентично в CLI та на всіх платформах шлюзу (Telegram, Discord, Slack, Matrix, Signal, WhatsApp, SMS, iMessage, Webhook, API server та веб‑дашборд).
## Додавання критеріїв під час виконання: `/subgoal`

Поки ціль активна, ти можеш додати додаткові критерії прийняття за допомогою `/subgoal <текст>` без скидання циклу. Кожен виклик додає один пронумерований пункт до списку підцілей цілі; **промпт продовження**, який агент бачить на наступному кроці, включає оригінальну ціль плюс блок «Додаткові критерії, які користувач додав під час циклу», а **промпт судді** переписується так, щоб вердикт враховував кожну підціль — ціль не позначається як завершена, доки не буде виконано оригінальне завдання **і** всі підцілі.

| Command | What it does |
|---|---|
| `/subgoal <text>` | Додати новий критерій до активної цілі. Потрібна активна `/goal`. |
| `/subgoal` (no args) | Показати поточний пронумерований список підцілей. |
| `/subgoal remove <N>` | Видалити N‑ту підціль (нумерація з 1). |
| `/subgoal clear` | Видалити всі підцілі, зберігши оригінальну ціль без змін. |

Підцілі зберігаються разом із ціллю в `SessionDB.state_meta`, тому вони виживають після `/resume`. Встановлення нової `/goal <text>` замінює ціль і очищає список підцілей; `/goal clear` робить те ж саме.

Використовуй це, коли запускаєш цикл («виправити неуспішні тести») і під час виконання помічаєш, що також потрібно «додати регресійний тест для багу, який ти щойно виправив» — `/subgoal add a regression test` уточнює критерії успіху без переривання запущеного циклу.
## Деталі поведінки

### Суддя

Після кожного ходу Hermes викликає допоміжну модель з:

- Текстом поточної цілі
- Останньою фінальною відповіддю агента (останні ~4 KB тексту)
- Системним запитом, який наказує судді відповісти строгим JSON: `{"done": <bool>, "reason": "<one-sentence rationale>"}`

Суддя навмисно консервативний: він позначає ціль `done` лише коли відповідь **явно** підтверджує, що ціль завершена, коли фінальний результат чітко створений, або коли ціль недосяжна/заблокована (розглядається як DONE з причиною блокування, щоб не витрачати бюджет на неможливі завдання).

### Семантика «fail-open»

Якщо суддя помилиться (збої мережі, некоректна відповідь, недоступний допоміжний клієнт), Hermes трактує вердикт як `continue` — зламаний суддя ніколи не зупиняє прогрес. **Бюджет ходів** є реальним захистом.

### Бюджет ходів

За замовчуванням — 20 ходів продовження (`goals.max_turns` у `config.yaml`). Коли бюджет вичерпується, Hermes автоматично паузить і повідомляє, як діяти далі:

```
⏸ Goal paused — 20/20 turns used. Use /goal resume to keep going, or /goal clear to stop.
```

`/goal resume` скидає лічильник до нуля, тож можна продовжувати роботу невеликими порціями.

### Повідомлення користувача завжди мають пріоритет

Будь‑яке реальне повідомлення, яке ти надсилаєш, поки ціль активна, має пріоритет над циклом продовження. У CLI твоє повідомлення потрапляє в `_pending_input` перед чергою продовження; у шлюзі воно проходить через FIFO‑адаптер тим же шляхом. Суддя запускається знову після твого ходу — тому, якщо твоє повідомлення випадково завершує ціль, суддя це зафіксує і зупинить процес.

### Безпека під час виконання (шлюз)

Поки агент вже працює, команди `/goal status`, `/goal pause` і `/goal clear` безпечні — вони змінюють лише стан контрольної площини і не переривають поточний хід. Встановлення **нової** цілі під час виконання (`/goal <new text>`) відхиляється з повідомленням, що спочатку треба виконати `/stop`, щоб стара задача продовження не змагалася з новою.

### Стійкість

Стан цілі зберігається в `SessionDB.state_meta` під ключем `goal:<session_id>`. Це означає, що `/resume` продовжує саме там, де ти зупинився — встанови ціль, закрий ноутбук, повернись завтра, виконай `/resume`, і ціль залишиться в тому ж стані (активна, паузована або завершена).

### Кеш підказок

Підказка продовження — це просте повідомлення ролі користувача, додане до історії. Вона **не** змінює системний запит, не міняє набір інструментів і не впливає на розмову таким чином, який би анулював кеш підказок Hermes. Виконання цілі у 20 ходів коштує так само кеш‑виходно, як і 20 ходів звичайної розмови.
## Конфігурація

Додай до `~/.hermes/config.yaml`:

```yaml
goals:
  # Max continuation turns before Hermes auto-pauses and asks you to
  # /goal resume. Default 20. Lower this if you want tighter loops;
  # raise it for long-running refactors.
  max_turns: 20
```

### Вибір моделі судді

Суддя використовує допоміжне завдання `goal_judge`. За замовчуванням воно підключається до твоєї основної моделі (див. [Auxiliary Models](/user-guide/configuration#auxiliary-models)). Якщо ти хочеш перенаправити суддю на дешеву швидку модель, щоб знизити витрати, додай перевизначення:

```yaml
auxiliary:
  goal_judge:
    provider: openrouter
    model: google/gemini-3-flash-preview
```

Виклик судді невеликий (~200 токенів у виводі) і виконується один раз за хід, тому дешева швидка модель зазвичай є правильним вибором.
## Приклад покрокового проходу

```
You: /goal Create four files /tmp/note_{1..4}.txt, one per turn, each containing its number as text

  ⊙ Goal set (20-turn budget): Create four files /tmp/note_{1..4}.txt, one per turn, each containing its number as text

Hermes: Creating /tmp/note_1.txt now.
  💻 echo "1" > /tmp/note_1.txt   (0.1s)
  I've created /tmp/note_1.txt with the content "1". I'll continue with the remaining files on the next turn as you specified.

  ↻ Continuing toward goal (1/20): Only 1 of 4 files has been created; 3 files remain.

Hermes: [Continuing toward your standing goal]
  💻 echo "2" > /tmp/note_2.txt   (0.1s)
  Created /tmp/note_2.txt. Two more to go.

  ↻ Continuing toward goal (2/20): 2 of 4 files created; 2 remain.

Hermes: [Continuing toward your standing goal]
  💻 echo "3" > /tmp/note_3.txt   (0.1s)
  Created /tmp/note_3.txt.

  ↻ Continuing toward goal (3/20): 3 of 4 files created; 1 remains.

Hermes: [Continuing toward your standing goal]
  💻 echo "4" > /tmp/note_4.txt   (0.1s)
  All four files have been created: /tmp/note_1.txt through /tmp/note_4.txt, each containing its number.

  ✓ Goal achieved: All four files were created with the specified content, completing the goal.

You: _
```

Чотири ходи, один виклик `/goal`, нуль підказок «продовжуй» від тебе.
## Коли суддя помиляється

Ні один суддя не досконалий. Є два типи помилок, на які варто звертати увагу:

**False negative — суддя каже «продовжуй», хоча ціль вже досягнута.** Це виявляє ліміт ходів. Ти побачиш `⏸ Goal paused` і зможеш виконати `/goal clear` або просто надіслати нове повідомлення.

**False positive — суддя каже «завершено», хоча робота ще триває.** Ти побачиш `✓ Goal achieved`, але знаєш краще. Надішли наступне повідомлення, щоб продовжити, або точніше встанови ціль: `/goal <більш конкретний текст>`. Системний промпт судді навмисно консервативний, щоб false positives траплялися рідше, ніж false negatives.

Якщо verdict судді здається тобі непереконливим, текст причини в рядку `↻ Continuing toward goal` або `✓ Goal achieved` підкаже, що саме суддя бачив. Зазвичай цього достатньо, щоб діагностувати, чи був текст цілі неоднозначним, чи відповідь моделі була.
## Атрибуція

`/goal` — це реалізація Hermes патерну **Ralph loop**. Дизайн, орієнтований на користувача — підтримувати ціль живою протягом ходів, не зупинятись, доки вона не досягнута, з елементами керування `create`/`pause`/`resume`/`clear` — був популяризований і випущений у [Codex CLI 0.128.0](https://github.com/openai/codex) Еріком Траутом у команді OpenAI Codex. Наше впровадження є незалежним (центральний реєстр `CommandDef`, збереження у `SessionDB.state_meta`, допоміжний клієнт‑суддя, продовження у вигляді adapter‑FIFO на стороні шлюзу), проте ідея їхня. Кредит там, де він належить.
