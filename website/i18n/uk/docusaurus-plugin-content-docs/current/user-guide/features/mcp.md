# MCP (Model Context Protocol)

MCP дозволяє Hermes Agent підключатися до зовнішніх серверів інструментів, щоб агент міг користуватися інструментами, які знаходяться поза межами самого Hermes — GitHub, базами даних, файловими системами, стеком браузерів, внутрішніми API та іншими.

Якщо ти коли‑небудь хотів, щоб Hermes використовував інструмент, який вже існує десь інше, MCP, як правило, є найчистішим способом це зробити.
## Що дає MCP

- Доступ до зовнішніх екосистем інструментів без необхідності спочатку писати власний інструмент Hermes
- Локальні stdio‑сервери та віддалені HTTP‑MCP‑сервери в одній конфігурації
- Автоматичне виявлення інструментів та їх реєстрація під час запуску
- Утилітні обгортки для ресурсів MCP та prompt‑ів, коли це підтримується сервером
- Фільтрація за сервером, щоб ти міг надати лише ті інструменти MCP, які дійсно потрібні Hermes‑у.
## Швидкий старт

1. Встанови підтримку MCP (вже включено, якщо ти використовував стандартний скрипт встановлення):

```bash
cd ~/.hermes/hermes-agent
uv pip install -e ".[mcp]"
```

2. Додай сервер MCP у `~/.hermes/config.yaml`:

```yaml
mcp_servers:
  filesystem:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/projects"]
```

3. Запусти Hermes:

```bash
hermes chat
```

4. Попроси Hermes використати можливість, підкріплену MCP.

Наприклад:

```text
List the files in /home/user/projects and summarize the repo structure.
```

Hermes виявить інструменти сервера MCP і використає їх, як будь‑який інший інструмент.
## Каталог: одно‑клікове встановлення схвалених Nous MCP

Hermes постачає відібраний каталог серверів MCP, які персонал Nous переглянув
і злив. За замовчуванням вони вимкнені — встановлюй лише те, що дійсно
потрібно.

```bash
hermes mcp                # interactive picker (default)
hermes mcp catalog        # plain-text list, scriptable
hermes mcp install n8n    # install a catalog entry by name
```

Пікер показує кожен запис із його поточним статусом:

```
n8n          available              Manage and inspect n8n workflows from Hermes
linear       enabled                Linear issue/project management (remote OAuth)
github       installed (disabled)   GitHub repo + PR tools
```

Натисни `Enter` на рядку, щоб встановити (і пройти через необхідні облікові дані),
увімкнути, вимкнути або видалити. Записи каталогу зберігаються у
`optional-mcps/` у репозиторії hermes-agent — наявність у цьому каталозі означає
схвалення Nous. Спільнотний рівень подачі відсутній; записи додаються шляхом
злиття PR.

Записи каталогу можуть вимагати:

- **API key** — Hermes запитує під час встановлення і записує значення у
  `~/.hermes/.env`. Не‑секретні значення (базові URL) потрапляють у той самий файл.
- **OAuth** (віддалений MCP) — записується як `auth: oauth` у твоїй конфігурації; клієнт MCP відкриває браузер при першому підключенні.
- **OAuth** (постачальник‑третьої сторони, наприклад Google/GitHub) — Hermes підказує
  `hermes auth <provider>`, якщо ти ще не автентифікувався.

### Вибір інструментів під час встановлення

Після налаштування облікових даних Hermes опитує сервер MCP, щоб отримати список усіх
інструментів, які він надає, і показує чек‑лист:

```
Select tools for 'linear' (SPACE toggle, ENTER confirm)
  [x] find_issues       Find issues matching a query
  [x] get_issue         Get a single issue
  [x] create_issue      Create a new issue
  [ ] delete_workspace  Delete a Linear workspace
  ...
```

Попередньо відмічені рядки беруться з:

1. **Твій попередній вибір**, якщо ти вже встановив цей запис раніше (перевстановлення
   зберігає те, що було — значення за замовчуванням у маніфесті не перезаписують його)
2. **`tools.default_enabled` у маніфесті**, якщо запис його оголошує (деякі
   записи каталогу попередньо обрізають мутуючі або рідко‑використовувані інструменти)
3. **Все**, якщо жодна з умов не підходить

Підтверди чек‑лист клавішею `Enter`. Тільки відмічені інструменти потрапляють у
`mcp_servers.<name>.tools.include`. Якщо ти вибрав усе, фільтр не записується
(найчистіша форма конфігурації, ідентична поведінка).

**Якщо опитування не вдається** (сервер недоступний, OAuth ще не завершений,
служба бекенду не працює), встановлення все одно успішне: застосовується
`tools.default_enabled` з маніфесту (якщо оголошено), або фільтр не записується
(якщо ні). Перезапусти `hermes mcp configure <name>`, коли сервер стане доступним,
щоб уточнити налаштування.

### Модель довіри

Встановлення запису каталогу виконує все, що вказано в маніфесті — `git clone`,
команди `bootstrap` (`pip install`, `npm install` тощо) і, зрештою,
власний код сервера MCP. Маніфести проходять рев’ю PR у репозиторії hermes-agent,
тому Nous переглянув кожен запис перед його випуском — **але все одно варто
прочитати маніфест перед встановленням**, особливо поле `source:` з посиланням
на репозиторій, команди `install.bootstrap:` та будь‑які виклики `transport.command:`.

Маніфести розташовані за адресою
[`optional-mcps/<name>/manifest.yaml`](https://github.com/NousResearch/hermes-agent/tree/main/optional-mcps)
на GitHub. Пікер також виводить URL `source:` під час встановлення, щоб ти міг швидко
перевірити вихідний репозиторій.

### Сумісність версій маніфесту

Маніфести фіксують `manifest_version`. Каталог є forward‑compatible: якщо PR додає запис
з новішою `manifest_version`, ніж розуміє твій встановлений Hermes, пікер покаже
попередження (`⚠ '<name>' requires a newer Hermes`) для цього запису замість того,
щоб тихо його приховати. Запусти `hermes update`, щоб встановити останню версію Hermes,
коли побачиш таке повідомлення.

### Підстановка змінних середовища `${ENV_VAR}` під час виконання

У `transport.command`, `transport.args`, `transport.url` та `headers` запису
каталогу `${VAR}` підставляються під час підключення до сервера зі змінних
середовища (включаючи все, що є у `~/.hermes/.env`). Це корисно, коли запис
хоче посилатися на значення, налаштоване користувачем в іншому місці — напр.
`${HOME}/foo` або `${MY_PROVIDER_TOKEN}`.

Зверни увагу, що це відрізняється від `${INSTALL_DIR}` у маніфестах каталогу,
яке підставляється під час встановлення шляхом, куди каталог клонув запис.

### Оновлення вибору інструментів пізніше

```bash
hermes mcp configure linear
```

Відкриває той самий чек‑лист з твоїм поточним вибором, попередньо відміченим. Використовуй це,
коли хочеш увімкнути більше інструментів або коли сервер додав нові інструменти,
які ти хочеш підключити.

### Оновлення маніфесту каталогу

MCP ніколи не оновлюються автоматично. Перезапусти `hermes mcp install <name>`,
щоб оновити після оновлення Hermes, якщо змінилася версія маніфесту.

Щоб додати MCP до каталогу, відкрий PR проти
[`optional-mcps/`](https://github.com/NousResearch/hermes-agent/tree/main/optional-mcps).
## Два типи серверів MCP

### Stdio‑сервери

Stdio‑сервери працюють як локальні підпроцеси та спілкуються через stdin/stdout.

```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
```

Використовуй stdio‑сервери, коли:
- сервер встановлений локально
- потрібен низькозатримковий доступ до локальних ресурсів
- ти слідуєш документації серверів MCP, яка показує `command`, `args` і `env`

### HTTP‑сервери

HTTP‑сервери MCP — це віддалені кінцеві точки, до яких Hermes підключається безпосередньо.

```yaml
mcp_servers:
  remote_api:
    url: "https://mcp.example.com/mcp"
    headers:
      Authorization: "Bearer ***"
```

Використовуй HTTP‑сервери, коли:
- сервер MCP розміщений в іншому місці
- твоя організація надає внутрішні кінцеві точки MCP
- ти не хочеш, щоб Hermes створював локальний підпроцес для цієї інтеграції

### HTTP‑сервери з автентифікацією OAuth

Більшість хостованих серверів MCP (Linear, Sentry, Atlassian, Asana, Figma, Stripe, …) вимагають OAuth 2.1 замість статичного bearer‑токену. Встанови `auth: oauth`, і Hermes самостійно виконає виявлення, динамічну реєстрацію клієнта, PKCE, обмін токенами, оновлення та підвищення рівня автентифікації через MCP Python SDK.

```yaml
mcp_servers:
  linear:
    url: "https://mcp.linear.app/mcp"
    auth: oauth
```

При першому підключенні Hermes виводить URL авторизації, відкриває твій браузер, якщо це можливо, і чекає на OAuth‑колбек на локальному loopback‑порті. Токени кешуються у `~/.hermes/mcp-tokens/<server>.json` з правами 0o600; подальші запуски тихо їх використовують, доки оновлення не зазнає невдачі.

**Віддалені / безголові хости.** Коли Hermes працює на іншій машині, ніж твій браузер, loopback‑колбек не може дістатися до твого ноутбука. Є два способи завершити процес:

- **Вставка (без налаштувань):** у інтерактивному терміналі Hermes виводить «Or paste the redirect URL here…» разом з URL авторизації. Відкрий URL у браузері, підтвердь, скопіюй повний URL, на який перейшов браузер (перенаправлення покаже помилку з’єднання — це очікувано), і встав його у запит. Працюють і прості рядки запиту `?code=…&state=…`.
- **SSH‑перенаправлення порту:** `ssh -N -L <port>:127.0.0.1:<port> user@host` у окремому терміналі, після чого перенаправлення працюватиме звичайно.

Дивись [OAuth over SSH / Remote Hosts](../../guides/oauth-over-ssh.md#mcp-servers) для повного покрокового опису, включаючи сервери без DCR (наприклад, Slack), попередньо зареєстровані `client_id`/`client_secret`, налаштування scope та повторну автентифікацію через `hermes mcp login <server>`.

**Підводний камінь — гонка автоперезавантаження конфігурації.** Якщо ти редагуєш `~/.hermes/config.yaml` під час роботи активної сесії Hermes, CLI автоматично перезавантажує з’єднання MCP з тайм‑аутом у 30 секунд. Це недостатньо для інтерактивного OAuth‑потоку. Додай запис, а потім запусти `hermes mcp login <server>` у новому терміналі — він чекатиме повних 5 хвилин, щоб ти завершив автентифікацію.
## Базова довідка про конфігурацію

Hermes читає конфігурацію MCP з `~/.hermes/config.yaml` у розділі `mcp_servers`.

### Загальні ключі

| Ключ | Тип | Опис |
|---|---|---|
| `command` | string | Виконуваний файл для stdio MCP сервера |
| `args` | list | Аргументи для stdio сервера |
| `env` | mapping | Змінні середовища, передані stdio серверу |
| `url` | string | HTTP MCP кінцева точка |
| `headers` | mapping | HTTP‑заголовки для віддалених серверів |
| `timeout` | number | Тайм‑аут виклику інструменту |
| `connect_timeout` | number | Тайм‑аут початкового підключення |
| `enabled` | bool | Якщо `false`, Hermes пропускає сервер повністю |
| `supports_parallel_tool_calls` | bool | Якщо `true`, інструменти з цього сервера можуть виконуватись одночасно |
| `tools` | mapping | Фільтрація інструментів та політика їх використання на сервері |

### Мінімальний приклад stdio

```yaml
mcp_servers:
  filesystem:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"]
```

### Мінімальний приклад HTTP

```yaml
mcp_servers:
  company_api:
    url: "https://mcp.internal.example.com"
    headers:
      Authorization: "Bearer ***"
```
## Вбудовані пресети

Для відомих серверів MCP команда `hermes mcp add` приймає прапорець `--preset`, який заповнює дані транспорту, щоб тобі не доводилося шукати команду та аргументи. Пресет лише задає типові значення — усе інше (змінні середовища, заголовки, фільтрація), що передано в тому ж рядку команди, має перевагу.

| Пресет | Що він підключає |
|---|---|
| `codex` | MCP‑сервер CLI **Codex** (`codex mcp-server` через stdio). Потрібен CLI `codex`, який має бути в `PATH`. |

```bash
# Add Codex CLI as an MCP server in one line
hermes mcp add codex --preset codex
```

Це записує еквівалент:

```yaml
mcp_servers:
  codex:
    command: "codex"
    args: ["mcp-server"]
```

Ти можеш вибрати будь‑яку локальну назву (`hermes mcp add my-codex --preset codex` підходить); пресет лише надає типові значення для `command`/`args`.
## Як Hermes реєструє інструменти MCP

Hermes додає префікс до інструментів MCP, щоб вони не конфліктували з вбудованими іменами:

```text
mcp_<server_name>_<tool_name>
```

Приклади:

| Сервер | Інструмент MCP | Зареєстроване ім'я |
|---|---|---|
| `filesystem` | `read_file` | `mcp_filesystem_read_file` |
| `github` | `create-issue` | `mcp_github_create_issue` |
| `my-api` | `query.data` | `mcp_my_api_query_data` |

На практиці зазвичай не потрібно викликати префіксоване ім'я вручну — Hermes бачить інструмент і обирає його під час звичайного роздумування.
## Інструменти утиліти MCP

Коли підтримується, Hermes також реєструє утиліти навколо ресурсів та підказок MCP:

- `list_resources`
- `read_resource`
- `list_prompts`
- `get_prompt`

Ці інструменти реєструються для кожного сервера з тим самим шаблоном префікса, наприклад:

- `mcp_github_list_resources`
- `mcp_github_get_prompt`

### Важливо

Ці інструменти утиліти тепер враховують можливості:
- Hermes реєструє утиліти ресурсів лише якщо сесія MCP фактично підтримує операції з ресурсами
- Hermes реєструє утиліти підказок лише якщо сесія MCP фактично підтримує операції з підказками

Тому сервер, який надає викликаємі інструменти, але не має ресурсів/підказок, не отримає цих додаткових обгорток.
## Фільтрація на рівні сервера

Ти можеш керувати тим, які інструменти кожен сервер MCP додає до Hermes, забезпечуючи тонке управління простором імен твоїх інструментів.

### Вимкнути сервер повністю

```yaml
mcp_servers:
  legacy:
    url: "https://mcp.legacy.internal"
    enabled: false
```

Якщо `enabled: false`, Hermes пропускає сервер повністю і навіть не намагається встановити з’єднання.

### Білий список інструментів сервера

```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
    tools:
      include: [create_issue, list_issues]
```

Реєструються лише інструменти цього сервера MCP.

### Чорний список інструментів сервера

```yaml
mcp_servers:
  stripe:
    url: "https://mcp.stripe.com"
    tools:
      exclude: [delete_customer]
```

Реєструються всі інструменти сервера, крім виключених.

### Правило пріоритету

Якщо присутні обидва:

```yaml
tools:
  include: [create_issue]
  exclude: [create_issue, delete_issue]
```

`include` має перевагу.

### Фільтрувати також утилітарні інструменти

Ти можеш окремо вимкнути додані Hermes обгортки утиліт:

```yaml
mcp_servers:
  docs:
    url: "https://mcp.docs.example.com"
    tools:
      prompts: false
      resources: false
```

Це означає:
- `tools.resources: false` вимикає `list_resources` та `read_resource`
- `tools.prompts: false` вимикає `list_prompts` та `get_prompt`

### Повний приклад

```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
    tools:
      include: [create_issue, list_issues, search_code]
      prompts: false

  stripe:
    url: "https://mcp.stripe.com"
    headers:
      Authorization: "Bearer ***"
    tools:
      exclude: [delete_customer]
      resources: false

  legacy:
    url: "https://mcp.legacy.internal"
    enabled: false
```
## Що станеться, якщо все буде відфільтровано?

Якщо у твоїй конфігурації фільтруються всі викликаємі інструменти та вимикаються або пропускаються всі підтримувані утиліти, Hermes не створює порожній runtime MCP toolset для цього сервера.

Це зберігає список інструментів чистим.
## Поведінка під час виконання

### Час виявлення

Hermes виявляє сервери MCP під час запуску та реєструє їх інструменти у звичайному реєстрі інструментів.

### Динамічне виявлення інструментів

Сервери MCP можуть повідомляти Hermes про зміну доступних інструментів під час виконання, надсилаючи сповіщення `notifications/tools/list_changed`. Коли Hermes отримує це сповіщення, він автоматично повторно отримує список інструментів сервера та оновлює реєстр — без необхідності вручну виконувати `/reload-mcp`.

Це корисно для серверів MCP, можливості яких змінюються динамічно (наприклад, сервер, який додає інструменти, коли завантажується нова схема бази даних, або видаляє інструменти, коли сервіс вимикається).

Оновлення захищене блокуванням, тому швидкі послідовні сповіщення від одного сервера не викликають перекриття оновлень. Сповіщення про зміни підказок та ресурсів (`prompts/list_changed`, `resources/list_changed`) отримуються, але ще не обробляються.

### Перезавантаження

Якщо ти змінив конфігурацію MCP, використай:

```text
/reload-mcp
```

Це перезавантажує сервери MCP з конфігурації та оновлює список доступних інструментів. Для змін інструментів під час виконання, які надсилає сам сервер, дивись розділ [Dynamic Tool Discovery](#dynamic-tool-discovery) вище.

### Набори інструментів

Кожен налаштований сервер MCP також створює runtime toolset, коли він додає принаймні один зареєстрований інструмент:

```text
mcp-<server>
```

Це полегшує роботу з серверами MCP на рівні набору інструментів.
## Модель безпеки

### Фільтрація середовища Stdio

Для серверів stdio Hermes не передає сліпо все твоє середовище оболонки.

Передаються лише явно налаштовані `env` та безпечний базовий набір. Це зменшує випадкове витікання секретів.

### Контроль експозиції на рівні конфігурації

Нова підтримка фільтрації також є контролем безпеки:
- вимкнути небезпечні інструменти, які ти не хочеш, щоб модель бачила
- надати лише мінімальний білий список для чутливого сервера
- вимкнути обгортки ресурсів/промптів, коли ти не хочеш, щоб ця поверхня була доступна
## Приклади використання

### GitHub‑сервер з мінімальним інтерфейсом управління issue

```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
    tools:
      include: [list_issues, create_issue, update_issue]
      prompts: false
      resources: false
```

Використовуй його так:

```text
Show me open issues labeled bug, then draft a new issue for the flaky MCP reconnection behavior.
```

### Stripe‑сервер з видаленими небезпечними діями

```yaml
mcp_servers:
  stripe:
    url: "https://mcp.stripe.com"
    headers:
      Authorization: "Bearer ***"
    tools:
      exclude: [delete_customer, refund_payment]
```

Використовуй його так:

```text
Look up the last 10 failed payments and summarize common failure reasons.
```

### Сервер файлової системи для кореня одного проєкту

```yaml
mcp_servers:
  project_fs:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/my-project"]
```

Використовуй його так:

```text
Inspect the project root and explain the directory layout.
```
## Усунення проблем

### MCP сервер не підключається

Перевір:

```bash
# Verify MCP deps are installed (already included in standard install)
cd ~/.hermes/hermes-agent && uv pip install -e ".[mcp]"

node --version
npx --version
```

Потім перевір свою конфігурацію та перезапусти Hermes.

### Інструменти не відображаються

Можливі причини:
- сервер не зміг підключитися
- виявлення інструментів не вдалося
- твоя конфігурація фільтра виключила інструменти
- можливість утиліти відсутня на цьому сервері
- сервер вимкнено за допомогою `enabled: false`

Якщо ти навмисно фільтруєш, це очікувана поведінка.

### Чому не з’явилися утиліти ресурсу або підказки?

Тому що Hermes тепер реєструє ці обгортки лише коли обидві умови істинні:
1. твоя конфігурація дозволяє їх
2. сесія сервера фактично підтримує цю можливість

Це навмисно і забезпечує достовірність списку інструментів.
## Паралельні виклики інструментів

За замовчуванням інструменти MCP виконуються послідовно — по одному за раз. Якщо ваш сервер MCP надає інструменти, які безпечно запускати одночасно (наприклад, запити лише на читання, незалежні API‑виклики), ви можете увімкнути паралельне виконання:

```yaml
mcp_servers:
  docs:
    command: "docs-server"
    supports_parallel_tool_calls: true
```

Коли `supports_parallel_tool_calls` має значення `true`, Hermes може виконувати кілька інструментів з цього сервера одночасно в одному пакеті викликів інструментів, так само, як це робиться для вбудованих інструментів лише для читання (`web_search`, `read_file` тощо).

:::caution
Увімкнюйте паралельні виклики лише для серверів MCP, інструменти яких безпечно виконувати одночасно. Якщо інструменти читають і записують спільний стан, файли, бази даних або зовнішні ресурси, перегляньте умови гонок читання/запису перед увімкненням цього налаштування.
:::
## Підтримка вибірки MCP

Сервери MCP можуть запитувати інференс LLM у Hermes через протокол `sampling/createMessage`. Це дозволяє серверу MCP попросити Hermes згенерувати текст від його імені — корисно для серверів, яким потрібні можливості LLM, але у них немає власного доступу до моделі.

Вибірка **увімкнена за замовчуванням** для всіх серверів MCP (коли SDK MCP це підтримує). Налаштуй її для кожного сервера у розділі `sampling`:

```yaml
mcp_servers:
  my_server:
    command: "my-mcp-server"
    sampling:
      enabled: true            # Enable sampling (default: true)
      model: "openai/gpt-4o"  # Override model for sampling requests (optional)
      max_tokens_cap: 4096     # Max tokens per sampling response (default: 4096)
      timeout: 30              # Timeout in seconds per request (default: 30)
      max_rpm: 10              # Rate limit: max requests per minute (default: 10)
      max_tool_rounds: 5       # Max tool-use rounds in sampling loops (default: 5)
      allowed_models: []       # Allowlist of model names the server may request (empty = any)
      log_level: "info"        # Audit log level: debug, info, or warning (default: info)
```

Обробник вибірки включає обмежувач швидкості за **ковзним вікном**, тайм‑аутів для кожного запиту та обмеження глибини циклу інструментів, щоб запобігти неконтрольованому використанню. Метрики (кількість запитів, помилки, використані токени) відстежуються для кожного екземпляра сервера.

Щоб вимкнути вибірку для конкретного сервера:

```yaml
mcp_servers:
  untrusted_server:
    url: "https://mcp.example.com"
    sampling:
      enabled: false
```
## Запуск Hermes як MCP‑серверу

Окрім підключення **до** MCP‑серверів, Hermes може також **бути** MCP‑сервером. Це дозволяє іншим агентам, сумісним з MCP (Claude Code, Cursor, Codex або будь‑якому MCP‑клієнту), використовувати можливості обміну повідомленнями Hermes — отримувати список розмов, читати історію повідомлень і надсилати повідомлення на всіх підключених платформах.

### Коли це використовувати

- Ти хочеш, щоб Claude Code, Cursor або інший код‑агент надсилав і читав повідомлення Telegram/Discord/Slack через Hermes
- Ти хочеш один MCP‑сервер, який одночасно підключає всі платформи обміну повідомленнями Hermes
- У тебе вже працює шлюз Hermes з підключеними платформами

### Швидкий старт

```bash
hermes mcp serve
```

Це запускає stdio MCP‑сервер. MCP‑клієнт (не ти) керує життєвим циклом процесу.

### Налаштування MCP‑клієнта

Додай Hermes до конфігурації свого MCP‑клієнта. Наприклад, у `~/.claude/claude_desktop_config.json` для Claude Code:

```json
{
  "mcpServers": {
    "hermes": {
      "command": "hermes",
      "args": ["mcp", "serve"]
    }
  }
}
```

Або, якщо ти встановив Hermes у певному місці:

```json
{
  "mcpServers": {
    "hermes": {
      "command": "/home/user/.hermes/hermes-agent/venv/bin/hermes",
      "args": ["mcp", "serve"]
    }
  }
}
```

### Доступні інструменти

MCP‑сервер надає 10 інструментів, що відповідають поверхні канального мосту OpenClaw плюс специфічний для Hermes браузер каналів:

| Інструмент | Опис |
|------|-------------|
| `conversations_list` | Список активних розмов обміну повідомленнями. Фільтруй за платформою або шукай за назвою. |
| `conversation_get` | Отримати докладну інформацію про одну розмову за ключем сесії. |
| `messages_read` | Прочитати недавню історію повідомлень розмови. |
| `attachments_fetch` | Витягнути нетекстові вкладення (зображення, медіа) з конкретного повідомлення. |
| `events_poll` | Опитувати нові події розмови з моменту позиції курсора. |
| `events_wait` | Довге опитування / блокування, доки не надійде наступна подія (майже реальний час). |
| `messages_send` | Надіслати повідомлення через платформу (наприклад `telegram:123456`, `discord:#general`). |
| `channels_list` | Список доступних цілей обміну повідомленнями на всіх платформах. |
| `permissions_list_open` | Список запитів на підтвердження, що спостерігаються під час цієї сесії мосту. |
| `permissions_respond` | Дозволити або відхилити запит на підтвердження. |

### Система подій

MCP‑сервер включає живий міст подій, який опитує базу даних сесій Hermes для нових повідомлень. Це дає MCP‑клієнтам майже реальний час усвідомлення вхідних розмов:

```
# Poll for new events (non-blocking)
events_poll(after_cursor=0)

# Wait for next event (blocks up to timeout)
events_wait(after_cursor=42, timeout_ms=30000)
```

Типи подій: `message`, `approval_requested`, `approval_resolved`

Черга подій зберігається в пам’яті і запускається, коли міст підключається. Старіші повідомлення доступні через `messages_read`.

### Параметри

```bash
hermes mcp serve              # Normal mode
hermes mcp serve --verbose    # Debug logging on stderr
```

### Як це працює

MCP‑сервер читає дані розмов безпосередньо зі сховища сесій Hermes (`~/.hermes/sessions/sessions.json` та SQLite‑бази даних). Фоновий потік опитує базу даних для нових повідомлень і підтримує чергу подій у пам’яті. Для надсилання повідомлень використовується та сама інфраструктура `send_message`, що й у самому агенті Hermes.

Шлюз НЕ потребує запуску для операцій читання (перелік розмов, читання історії, опитування подій). Він Потрібен для операцій надсилання, оскільки адаптери платформ потребують активних з’єднань.

### Поточні обмеження

- Вбудований `hermes mcp serve` сьогодні надає **лише stdio** MCP‑сервер. Якщо потрібен HTTP MCP‑сервер, запусти окремий адаптер — або, що набагато частіше, використай **клієнтську** сторону MCP Hermes, яка вже підтримує і stdio, і HTTP (`url` + `headers` у `mcp_servers.yaml` / `config.yaml`; див. [HTTP servers](#http-servers) вище).
- Опитування подій з інтервалом ~200 мс через оптимізоване за mtime DB‑опитування (пропускає роботу, коли файли не змінилися)
- Протокол push‑повідомлень `claude/channel` ще не реалізований
- Надсилання лише тексту (без медіа/вкладень через `messages_send`)
