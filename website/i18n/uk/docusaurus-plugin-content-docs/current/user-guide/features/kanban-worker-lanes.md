# Канбан робочі смуги

**Робоча смуга** — це клас процесу, до якого диспетчер канбану може направляти завдання. Кожна смуга має ідентифікатор (рядок виконавця), механізм створення та контракт щодо того, що вона повинна робити із завданням після створення.

Ця сторінка є контрактом. Вона існує для двох аудиторій:

- **Оператори**, які обирають, які смуги підключити до дошки (які профілі створювати, які виконавці використовувати).
- **Автори плагінів / інтеграцій**, які хочуть додати нову форму смуги (CLI‑робітник, що обгортає Codex / Claude Code / OpenCode, контейнеризований робітник‑рев’ю, сервіс, що не є Hermes, який отримує завдання через API).

Якщо ти пишеш сам код робітника — агент, який працює *всередині* смуги — навичка [`kanban-worker`](https://github.com/NousResearch/hermes-agent/blob/main/skills/devops/kanban-worker/SKILL.md) містить детальніший процедурний опис.
## Ієрархія

```text
Hermes Kanban  =  canonical task lifecycle + audit trail
Worker lane    =  implementation executor for one assigned card
Reviewer       =  human or human-proxy that gates "done"
GitHub PR      =  upstreamable artifact (optional, for code lanes)
```

Hermes Kanban володіє правдою стану життєвого циклу — `ready` → `running` → `blocked` / `done` / `archived`. Робочі lanes виконують роботу, але ніколи не володіють цією правдою; все, що вони роблять, повертається через ядро kanban за допомогою інструментів `kanban_*` (або, для зовнішніх робітників, не пов’язаних з Hermes, через API). Рецензенти контролюють перехід від «code change written» до «task done».
## Що забезпечує lane

Щоб lane — це робітник kanban, інтеграція повинна надавати три речі:

### 1. Рядок `assignee`

Диспетчер порівнює `task.assignee` або з іменем профілю Hermes (типова форма lane), або з зареєстрованим не‑spawnable‑ідентифікатором (форма lane плагіна — див. [Adding an external CLI worker lane](#adding-an-external-cli-worker-lane) нижче). Завдання, чиїй `assignee` не розв’язується, залишаються у `ready` з подією `skipped_nonspawnable`, щоб оператор дошки міг їх виправити; вони не відкидаються мовчки і не виконуються довільним запасним варіантом.

### 2. Механізм `spawn`

Для lane‑ів профілів Hermes диспетчерський `_default_spawn` запускає

```
hermes -p <assignee> chat -q <prompt>
```

(або еквівалентну форму модуля, коли shim `hermes` відсутній у `$PATH`) всередині закріпленої робочої області завдання, встановлюючи такі змінні середовища:

| Змінна | Призначення |
|---|---|
| `HERMES_KANBAN_TASK` | ідентифікатор завдання, над яким працює робітник |
| `HERMES_KANBAN_DB` | абсолютний шлях до файлу SQLite конкретної дошки |
| `HERMES_KANBAN_BOARD` | slug дошки |
| `HERMES_KANBAN_WORKSPACES_ROOT` | корінь дерева робочих областей дошки |
| `HERMES_KANBAN_WORKSPACE` | абсолютний шлях до *цієї* робочої області завдання |
| `HERMES_KANBAN_RUN_ID` | ідентифікатор поточного запуску (для gate життєвого циклу) |
| `HERMES_KANBAN_CLAIM_LOCK` | рядок блокування claim (`<host>:<pid>:<uuid>`) |
| `HERMES_PROFILE` | ім’я профілю самого робітника (для атрибуції автора `kanban_comment`) |
| `HERMES_TENANT` | простір імен tenant, якщо у завдання є такий |

Для lane‑ів, що не є Hermes (зареєстрованих через плагін), плагін постачає власну callable `spawn_fn`, яка отримує `task`, `workspace` і `board` і повертає необов’язковий PID для виявлення краху.

### 3. Завершувач життєвого циклу

Кожен claim має завершитися **рівно одним** із:

- `kanban_complete(summary=..., metadata=...)` — завдання успішне, статус змінюється на `done`.
- `kanban_block(reason=...)` — завдання чекає на людський ввід, статус змінюється на `blocked`. Диспетчер перезапускає його, коли виконується `kanban_unblock`.
- Процес робітника завершується без виклику інструменту. Ядро його збирає і генерує `crashed` (PID помер), `gave_up` (спрацьовує breaker послідовних помилок) або `timed_out` (перевищено `max_runtime`). Це шлях помилки; здорові робітники не закінчуються тут.

Ядро kanban забезпечує, щоб саме один із цих варіантів завершував кожен запуск. Робітник, який не викликає жодного з них і завершується нормально, розглядається як **зламаний**.
## Вихідні дані та конвенція «review‑required»

Для більшості завдань, що змінюють код, робота не вважається справді *завершеною* в момент, коли виконавець закінчує — потрібен людський рецензент. Ядро канбану не примушує розрізняти це («завдання, що змінює код» — нечітке поняття, і примусове блокування замість завершення для кожного кодувальника порушило б потоки, де огляд не потрібен). Це конвенція, накладена зверху:

- **Блокувати замість завершувати**, з префіксом `reason` `review-required: `, щоб панель інструментів / `hermes kanban show` відображала рядок як такий, що очікує перегляду.
- **Спочатку кидати структуровані метадані у `kanban_comment`**, оскільки `kanban_block` передає лише зрозумілий людині `reason`. Коментарі — це стійкий канал анотацій: усі поля, важливі для аудиту (changed_files, tests_run, diff_path або PR‑url, decisions), розміщуються саме там.
- **Рецензент або схвалює і розблоковує**, що повторно запускає виконавця з потоком коментарів для подальших уточнень; або просить внести зміни іншим коментарем, який наступний запуск виконавця бачить у контексті `kanban_show`.

Навичка [`kanban-worker`](https://github.com/NousResearch/hermes-agent/blob/main/skills/devops/kanban-worker/SKILL.md) містить приклади як для `kanban_complete` (справді термінальні завдання — виправлення типографічних помилок, зміни документації, написання досліджень), так і для шаблону блокування `review-required`.
## Журнали та аудит‑трасування

Диспетчер записує stdout/stderr кожного worker‑a у `<board-root>/logs/<task_id>.log`. Журнали можна аудиту­вати через метадані kanban:

- рядки `task_runs` містять `log_path`, код виходу (за наявності), підсумок та метадані.
- рядки `task_events` містять кожен перехід стану (`promoted`, `claimed`, `heartbeat`, `completed`, `blocked`, `gave_up`, `crashed`, `timed_out`, `reclaimed`, `claim_extended`).
- `kanban_show` повертає і те, і інше, тому оглядач (або наступний worker) — читаючи завдання, отримує повну історію без потреби у доступі до панелі.

Панель візуалізує історію запусків із підсумками, блоками метаданих та бейджами статусу виходу. Користувачі CLI можуть виконати `hermes kanban tail <task_id>`, щоб слідкувати в реальному часі, або `hermes kanban runs <task_id>` для перегляду історичного списку спроб.
## Існуючі форми lane

### Hermes profile lane (default)

Форма, яку сьогодні приймає кожен kanban‑worker: виконавець — це ім’я профілю, диспетчер запускає `hermes -p <profile>`, worker автоматично завантажує навичку [`kanban-worker`](https://github.com/NousResearch/hermes-agent/blob/main/skills/devops/kanban-worker/SKILL.md) плюс блок системної підказки `KANBAN_GUIDANCE` і використовує інструменти `kanban_*` для завершення запуску. Ніякого налаштування, окрім визначення профілю.

Коли створюєш профілі для свого флоту, обирай назви, що відповідають *ролі*, яку оркестратор має маршрутизувати. Оркестратор (коли він є) виявляє назви твоїх профілів за допомогою `hermes profile list` — немає фіксованого списку, який система передбачає (дивись навичку [`kanban-orchestrator`](https://github.com/NousResearch/hermes-agent/blob/main/skills/devops/kanban-orchestrator/SKILL.md) для сторони оркестратора контракту).

### Orchestrator profile lane

Спеціалізація profile lane: оркестратор — це Hermes profile, чиї інструменти включають `kanban`, але виключають `terminal` / `file` / `code` / `web` для реалізації. Його завдання — розбивати високорівневу мету на дочірні завдання за допомогою `kanban_create` + `kanban_link` і крок назад. Навичка оркестратора кодує правила проти спокуси.
## Додавання зовнішньої CLI‑worker lane

Підключення інструменту CLI, який не є частиною Hermes (Codex CLI, Claude Code CLI, OpenCode CLI, локальний запуск моделі кодування тощо) як kanban‑worker lane *ще не є випробуваним шляхом*. Функція створення процесу диспетчера є підключуваною (`spawn_fn` — параметр у `dispatch_once`), і плагін може зареєструвати власну `spawn_fn` для не‑Hermes виконавця, проте супровідна інтеграція — обгортання коду виходу CLI у виклики `kanban_complete` / `kanban_block`, відображення конвенцій робочого простору/пісочниці CLI у змінну оточення диспетчера `HERMES_KANBAN_WORKSPACE`, обробка автентифікації та політики для кожного CLI — все ще потребує індивідуального проєктування.

Якщо ти розглядаєш можливість додати CLI‑lane, відкрий issue, у якому опишеш конкретний CLI та робочий процес, який хочеш реалізувати. Контракт вище визначає обмеження, яким має відповідати будь‑яка така lane; форма реалізації (один плагін на CLI чи універсальний плагін‑раннер CLI, параметризований конфігурацією) залишається відкритим питанням.

Історичний issue щодо цього — [#19931](https://github.com/NousResearch/hermes-agent/issues/19931), а закритий, не злитий PR, специфічний для Codex — [#19924](https://github.com/NousResearch/hermes-agent/pull/19924) — вони описують початкову пропозицію архітектури, проте раннер не був впроваджений.
## Режими відмов, які обробляє диспетчер

Щоб авторам lane не доводилося повторно реалізовувати їх:

- **Stale claim TTL** — робітник, який отримав claim і потім ніколи не надсилає heartbeat / не завершує / не блокує, буде звільнений після `DEFAULT_CLAIM_TTL_SECONDS` (за замовчуванням 15 хв) — але лише якщо процес робітника дійсно завершився. Живий робітник (повільна модель, що проводить 20+ хв у виклику LLM без інструментів) отримує *продовжений* claim замість того, щоб його вбити; лише мертвий PID звільняється.
- **Crashed worker** — робітник, чий локальний PID на хості зник, виявляється за допомогою `detect_crashed_workers` і завершується; задача збільшує `consecutive_failures` і може автоматично блокуватись, коли спрацьовує breaker.
- **Run-level retry** — коли задача повторно запускається (після блокування, після краху, після звільнення), робітник може використати параметр `expected_run_id` у завершувальних інструментах, щоб швидко завершитися, якщо його власний запуск вже був замінений.
- **Per-task max runtime** — `task.max_runtime_seconds` жорстко обмежує **wall‑clock** час кожного запуску, незалежно від живості PID. Це ловить справді «завислі» робітники, які продовження живого PID інакше тримало б їх у роботі.
- **Stranded-task detection** — готова задача, чиїй виконавець ніколи не створює claim протягом `kanban.stranded_threshold_seconds` (за замовчуванням 30 хв), з’являється у `hermes kanban diagnostics` як попередження `stranded_in_ready`. Серйозність підвищується до помилки при 2× порогу і до критичної при 6×. Це ловить помилково вказаних виконавців, видалені профілі та недоступні зовнішні пулі робітників в одному сигналі — без урахування ідентичності, без дозволеного списку на кожній дошці.
