# Веб‑дашборд

Веб‑дашборд — це UI у браузері для керування твоєю установкою Hermes Agent. Замість редагування YAML‑файлів або запуску CLI‑команд ти можеш налаштовувати параметри, керувати API‑ключами та стежити за сесіями через чистий веб‑інтерфейс.

:::tip
Аутентифікація в режимі Hosted‑mode використовує OAuth Nous Portal; якщо ти також хочеш, щоб дашборд спілкувався з реальним бекендом, `hermes setup --portal` підключає модель і шлюз інструментів. Дивись [Nous Portal](/integrations/nous-portal).
:::
## Quick Start

```bash
hermes dashboard
```

Це запускає локальний веб‑сервер і відкриває `http://127.0.0.1:9119` у твоєму браузері. Панель керування працює повністю на твоєму комп’ютері — жодні дані не залишають localhost.

### Options

| Flag | Default | Description |
|------|---------|-------------|
| `--port` | `9119` | Порт, на якому запускати веб‑сервер |
| `--host` | `127.0.0.1` | Адреса прив’язки |
| `--no-open` | — | Не відкривати браузер автоматично |
| `--insecure` | off | Дозволити прив’язку до не‑localhost хостів (**DANGEROUS** — відкриває API‑ключі в мережі; використай разом з firewall та сильною автентифікацією) |
| `--tui` | off | Надати доступ до вкладки Chat у браузері (вбудований `hermes --tui` через PTY/WebSocket). Або встанови `HERMES_DASHBOARD_TUI=1`. |

```bash
# Custom port
hermes dashboard --port 8080

# Bind to all interfaces (use with caution on shared networks)
hermes dashboard --host 0.0.0.0

# Start without opening browser
hermes dashboard --no-open

# Enable the in-browser Chat tab
hermes dashboard --tui
```
## Передумови

Типова установка `hermes-agent` не включає HTTP‑стек або допоміжний PTY‑інструмент — це додаткові extras. **Веб‑дашборд** потребує FastAPI і Uvicorn (`web` extra). Вкладка **Chat** також потребує `ptyprocess` для запуску вбудованого TUI у псевдо‑терміналі (`pty` extra на POSIX). Встанови обидва так:

```bash
pip install 'hermes-agent[web,pty]'
```

`web` extra підтягує FastAPI/Uvicorn; `pty` extra підтягує `ptyprocess` (POSIX) або `pywinpty` (рідний Windows — зауваж, що вбудований TUI все одно вимагає WSL). `pip install hermes-agent[all]` включає обидва extras і є найпростішим шляхом, якщо ти також хочеш обмін повідомленнями/голосом тощо.

Коли ти запускаєш `hermes dashboard` без необхідних залежностей, він підкаже, що саме треба встановити. Якщо фронтенд ще не зібраний і доступний `npm`, він збирається автоматично під час першого запуску.

Вкладка Chat навмисно вимкнена для простого запуску `hermes dashboard`. Запусти дашборд з `hermes dashboard --tui` або встанови `HERMES_DASHBOARD_TUI=1`, коли потрібна вбудована браузерна чат‑панель.
## Pages

### Статус

На головній сторінці показано живий огляд вашої інсталяції:

- **Agent version** та дата випуску
- **Gateway status** — running/stopped, PID, підключені платформи та їх стан
- **Active sessions** — кількість сесій, активних за останні 5 хвилин
- **Recent sessions** — список 20 найостанніших сесій з моделлю, кількістю повідомлень, використанням токенів та попереднім переглядом розмови

Сторінка статусу автоматично оновлюється кожні 5 секунд.

### Чат

Вкладка **Chat** вбудовує повний Hermes TUI (той самий інтерфейс, який ти отримуєш від `hermes --tui`) безпосередньо у браузері. Все, що можна робити в термінальному TUI — slash‑команди, вибір моделі, картки викликів інструментів, потокове markdown, підказки clarify/sudo/approval, темізація шкіри — працює ідентично тут, бо дашборд запускає реальний бінарник TUI і рендерить його ANSI‑вивід через [xterm.js](https://xtermjs.org/) з WebGL‑рендерером для піксель‑точного розташування клітинок.

**Як це працює:**

- `/api/pty` відкриває WebSocket, автентифікований токеном сесії дашборда
- Сервер спауниє `hermes --tui` за POSIX‑псевдо‑терміналом
- Натискання клавіш передаються до PTY; ANSI‑вивід повертається у браузер
- WebGL‑рендерер xterm.js малює кожну клітинку у сітку цілих пікселів; відстеження миші (SGR 1006), широкі символи (Unicode 11) та гліфи‑рамки рендеряться нативно
- Зміна розміру вікна браузера змінює розмір TUI через аддон `@xterm/addon-fit`

**Відновити існуючу сесію:** у вкладці **Sessions** натисни іконку відтворення (▶) поруч із будь‑якою сесією. Це перейде до `/chat?resume=<id>` і запустить TUI з `--resume`, завантажуючи повну історію.

**Передумови:**

- Node.js (те ж саме, що й для `hermes --tui`; пакет TUI збирається при першому запуску)
- `ptyprocess` — встановлюється разом з extra `pty` (`pip install 'hermes-agent[web,pty]'`, або `[all]` охоплює обидва)
- POSIX‑ядро (Linux, macOS або WSL2). Термінальна панель `/chat` вимагає POSIX PTY — у нативному Windows‑Python немає еквіваленту, тому на чистій Windows‑установці решта дашборда (сесії, завдання, метрики, редактор конфігурації) працює, а вкладка `/chat` покаже банер з рекомендацією використати WSL2 для цієї функції.

Закрий вкладку браузера, і PTY буде акуратно завершений на сервері. При повторному відкритті створюється нова сесія.

### Конфіг

Редактор у вигляді форми для `config.yaml`. Всі 150+ полів конфігурації автоматично виявляються з `DEFAULT_CONFIG` і організовані у вкладки‑категорії:

- **model** — модель за замовчуванням, провайдер, базовий URL, налаштування міркувань
- **terminal** — бекенд (local/docker/ssh/modal), таймаут, уподобання оболонки
- **display** — шкіра, прогрес інструменту, відображення відновлення, налаштування спінера
- **agent** — максимальна кількість ітерацій, таймаут шлюзу, рівень сервісу
- **delegation** — обмеження підагентів, зусилля міркувань
- **memory** — вибір провайдера, налаштування інжекції контексту
- **approvals** — режим схвалення небезпечних команд (ask/yolo/deny)
- І ще — кожен розділ `config.yaml` має відповідні поля форми

Поля з відомими допустимими значеннями (бекенд терміналу, шкіра, режим схвалення тощо) відображаються у випадаючих списках. Булеві значення — у вигляді перемикачів. Усе інше — текстове поле.

**Дії:**

- **Save** — одразу записує зміни у `config.yaml`
- **Reset to defaults** — повертає всі поля до значень за замовчуванням (не зберігає, доки не натиснеш **Save**)
- **Export** — завантажує поточний конфіг у вигляді JSON
- **Import** — завантажує JSON‑файл конфігурації, замінюючи поточні значення

:::tip
Зміни конфігурації набувають чинності у наступній сесії агента або після перезапуску шлюзу. Веб‑дашборд редагує той самий файл `config.yaml`, який читає `hermes config set` і шлюз.
:::

### API‑ключі

Керування файлом `.env`, у якому зберігаються API‑ключі та облікові дані. Ключі згруповані за категоріями:

- **LLM Providers** — OpenRouter, Anthropic, OpenAI, DeepSeek тощо
- **Tool API Keys** — Browserbase, Firecrawl, Tavily, ElevenLabs тощо
- **Messaging Platforms** — Telegram, Discord, Slack bot tokens тощо
- **Agent Settings** — не‑секретні змінні оточення, наприклад `API_SERVER_ENABLED`

Для кожного ключа показано:
- Чи встановлений він зараз (з редагованим попереднім переглядом значення)
- Опис його призначення
- Посилання на сторінку реєстрації/ключа провайдера
- Поле вводу для встановлення або оновлення значення
- Кнопка видалення

Рідко використовувані ключі приховані за замовчуванням під перемикачем.

### Сесії

Переглядай та досліджуй усі сесії агента. Кожен рядок показує назву сесії, іконку платформи‑джерела (CLI, Telegram, Discord, Slack, cron), назву моделі, кількість повідомлень, кількість викликів інструментів та час останньої активності. Активні сесії позначені пульсуючою міткою.

- **Search** — повнотекстовий пошук по всьому вмісту повідомлень за допомогою FTS5. Результати показують підсвічені фрагменти та автоматично скролять до першого підходящого повідомлення при розгортанні.
- **Expand** — натисни сесію, щоб завантажити її повну історію повідомлень. Повідомлення кольорово позначені за роллю (user, assistant, system, tool) і рендеряться як Markdown з підсвічуванням синтаксису.
- **Tool calls** — повідомлення‑асистенти з викликами інструментів показують згортні блоки з назвою функції та JSON‑аргументами.
- **Delete** — видали сесію та її історію повідомлень за допомогою іконки смітника.

### Логи

Переглядай файли логів агента, шлюзу та помилок з фільтрацією та живим підключенням.

- **File** — перемикай між логами `agent`, `errors` та `gateway`
- **Level** — фільтруй за рівнем логування: ALL, DEBUG, INFO, WARNING або ERROR
- **Component** — фільтруй за компонентом‑джерелом: all, gateway, agent, tools, cli або cron
- **Lines** — вибери кількість рядків для відображення (50, 100, 200 або 500)
- **Auto-refresh** — перемикач живого підключення, яке опитує нові рядки кожні 5 секунд
- **Color-coded** — рядки логів кольорово позначені за серйозністю (червоний — помилки, жовтий — попередження, тьмяний — debug)

### Аналітика

Аналітика використання та витрат, обчислена з історії сесій. Вибери період (7, 30 або 90 днів), щоб побачити:

- **Summary cards** — загальна кількість токенів (вхідних/вихідних), відсоток кеш‑хітів, загальна оцінена або фактична вартість, та кількість сесій зі середньоденною середньою
- **Daily token chart** — стовпчаста діаграма, що показує використання вхідних та вихідних токенів за день, з підказками при наведенні, що розкладають дані та вартість
- **Daily breakdown table** — дата, кількість сесій, вхідні токени, вихідні токени, кеш‑хіт, вартість за кожен день
- **Per-model breakdown** — таблиця, що показує кожну використану модель, кількість сесій, використання токенів та оцінену вартість

### Cron

Створюй та керуй запланованими cron‑завданнями, які виконують підказки агента за розкладом.

- **Create** — заповни назву (необов’язково), підказку, вираз cron (наприклад `0 9 * * *`) та ціль доставки (local, Telegram, Discord, Slack або email)
- **Job list** — кожне завдання показує назву, попередній перегляд підказки, вираз розкладу, бейдж стану (enabled/paused/error), ціль доставки, час останнього запуску та час наступного запуску
- **Pause / Resume** — перемикай завдання між активним та паузованим станами
- **Trigger now** — негайно виконай завдання поза його звичайним розкладом
- **Delete** — назавжди видали cron‑завдання

### Навички

Переглядай, шукай та вмикай/вимикай навички та набори інструментів. Навички завантажуються з `~/.hermes/skills/` і згруповані за категоріями.

- **Search** — фільтруй навички та набори інструментів за назвою, описом або категорією
- **Category filter** — натискай «пілки» категорій, щоб звузити список (наприклад MLOps, MCP, Red Teaming, AI)
- **Toggle** — вмикай або вимикай окремі навички перемикачем. Зміни набувають чинності у наступній сесії.
- **Toolsets** — окремий розділ показує вбудовані набори інструментів (операції з файлами, веб‑перегляд тощо) з їх активним/неактивним статусом, вимогами до налаштування та списком включених інструментів

:::warning Security
Веб‑дашборд читає та записує ваш файл `.env`, у якому містяться API‑ключі та секрети. За замовчуванням він прив’язується до `127.0.0.1` — доступний лише з вашого локального комп’ютера. Якщо прив’язати його до `0.0.0.0`, будь‑хто у вашій мережі зможе переглядати та змінювати ваші облікові дані. Дашборд не має власної автентифікації.
:::
## `/reload` Slash Command

Дашборд PR також додає slash‑команду `/reload` до інтерактивного CLI. Після зміни API‑ключів через веб‑дашборд (або шляхом прямого редагування `.env`) використай `/reload` у активній CLI‑сесії, щоб застосувати зміни без перезапуску:

```
You → /reload
  Reloaded .env (3 var(s) updated)
```

Ця команда повторно читає `~/.hermes/.env` у середовище процесу, що працює. Це корисно, коли ти додав новий ключ провайдера через дашборд і хочеш використати його одразу.
## REST API

Веб‑дашборд надає REST API, який споживає фронтенд. Ви також можете викликати ці кінцеві точки безпосередньо для автоматизації:

### GET /api/status

Повертає версію агента, статус **gateway**, стани платформи та кількість активних сесій.

### GET /api/sessions

Повертає 20 найновіших сесій з метаданими (модель, кількість токенів, часові мітки, попередній перегляд).

### GET /api/config

Повертає поточний вміст `config.yaml` у форматі JSON.

### GET /api/config/defaults

Повертає значення конфігурації за замовчуванням.

### GET /api/config/schema

Повертає схему, що описує кожне поле конфігурації — тип, опис, категорію та варіанти вибору, якщо застосовано. Фронтенд використовує її для відображення правильного віджету вводу для кожного поля.

### PUT /api/config

Зберігає нову конфігурацію. Тіло запиту: `{"config": {...}}`.

### GET /api/env

Повертає всі відомі змінні середовища з їхнім статусом встановлення/відсутності, прихованими значеннями, описами та категоріями.

### PUT /api/env

Встановлює змінну середовища. Тіло запиту: `{"key": "VAR_NAME", "value": "secret"}`.

### DELETE /api/env

Видаляє змінну середовища. Тіло запиту: `{"key": "VAR_NAME"}`.

### GET /api/sessions/\{session_id\}

Повертає метадані однієї сесії.

### GET /api/sessions/\{session_id\}/messages

Повертає повну історію повідомлень сесії, включаючи виклики **tool** та часові мітки.

### GET /api/sessions/search

Повнотекстовий пошук по вмісту повідомлень. Параметр запиту: `q`. Повертає ідентифікатори відповідних сесій з підсвіченими фрагментами.

### DELETE /api/sessions/\{session_id\}

Видаляє сесію та її історію повідомлень.

### GET /api/logs

Повертає рядки журналу. Параметри запиту: `file` (agent/errors/gateway), `lines` (кількість), `level`, `component`.

### GET /api/analytics/usage

Повертає використання токенів, витрати та аналітику сесій. Параметр запиту: `days` (за замовчуванням 30). Відповідь містить щоденні розбивки та агрегати за моделями.

### GET /api/cron/jobs

Повертає всі налаштовані **cron**‑jobs зі станом, розкладом та історією запусків.

### POST /api/cron/jobs

Створює новий **cron**‑job. Тіло запиту: `{"prompt": "...", "schedule": "0 9 * * *", "name": "...", "deliver": "local"}`.

### POST /api/cron/jobs/\{job_id\}/pause

Призупиняє **cron**‑job.

### POST /api/cron/jobs/\{job_id\}/resume

Відновлює призупинений **cron**‑job.

### POST /api/cron/jobs/\{job_id\}/trigger

Негайно запускає **cron**‑job поза його розкладом.

### DELETE /api/cron/jobs/\{job_id\}

Видаляє **cron**‑job.

### GET /api/skills

Повертає всі **skills** з їхньою назвою, описом, категорією та статусом увімкнення.

### PUT /api/skills/toggle

Увімкнути або вимкнути **skill**. Тіло запиту: `{"name": "skill-name", "enabled": true}`.

### GET /api/tools/toolsets

Повертає всі **toolsets** з їхньою міткою, описом, списком **tools** та статусом активності/налаштування.
## OAuth Authentication (gated mode)

Коли панель керування прив’язана до публічної адреси — будь‑якої, окрім `127.0.0.1` / `localhost` — Hermes Agent активує шлюз автентифікації на базі OAuth. Кожен запит має містити перевірений cookie сесії, інакше його перенаправляють через повний OAuth‑процес через Nous Portal.

Це призначено для розгорнутих у хмарі середовищ (зазвичай Fly.io), де панель доступна через публічний інтернет. Панелі, що належать оператору і прив’язані до loopback, не підлягають зміні.

### When the gate engages

| Flags | Auth gate | Use case |
|-------|-----------|----------|
| `hermes dashboard` (default — binds to `127.0.0.1`) | OFF | Локальна розробка |
| `hermes dashboard --host 0.0.0.0` | **ON** | Продакшн / розгортання на Fly.io |
| `hermes dashboard --host 192.168.1.10 --insecure` | OFF | Довірена LAN; користувач обирає автентифікацію за допомогою токену сесії (legacy) |

Шлюз увімкнено **тільки** коли:

1. Хост прив’язки не `127.0.0.1`, `::1`, `localhost` або `0.0.0.0` **І**
2. Прапорець `--insecure` **не** встановлений.

Встановлення `--insecure` зберігає існуючу поведінку з одним процесом токену сесії — без OAuth‑танцю, без необхідності у плагінах провайдерів. Використовуй лише в мережах, яким довіряєш усім клієнтам.

### Fail-closed semantics

Якщо шлюз мав би активуватись, але **не** зареєстровано `DashboardAuthProvider` (немає плагіна Nous, немає кастомного плагіна), `hermes dashboard` відмовляє у прив’язці з явним повідомленням про помилку. Не існує «default-deny, але приймати все» запасного варіанту — неправильно налаштована панель у режимі шлюзу ніколи не запуститься.

### Default provider: Nous Research

Вбудований плагін `plugins/dashboard_auth/nous` **завжди встановлюється** та автозавантажується. Він автоматично реєструє `DashboardAuthProvider` з назвою `nous`, коли налаштовано client ID.

#### Configuration

Плагін читає налаштування з двох джерел, причому змінна середовища має пріоритет, якщо задана не порожня:

**`config.yaml`** — канонічне джерело:

```yaml
dashboard:
  oauth:
    client_id: agent:01HXYZ…             # required to engage the gate
    portal_url: https://portal.nousresearch.com  # optional; defaults to production
```

**Environment variables** — перевизначення оператором:

| Env var | Overrides | Format | Provisioned by |
|---------|-----------|--------|----------------|
| `HERMES_DASHBOARD_OAUTH_CLIENT_ID` | `dashboard.oauth.client_id` | `agent:{instance_id}` | Nous Portal під час provisioning на Fly.io |
| `HERMES_DASHBOARD_PORTAL_URL` | `dashboard.oauth.portal_url` | URL (default: `https://portal.nousresearch.com`) | Portal — перевизначається лише для staging або кастомного розгортання |

Відповідно до конвенції Hermes Agent (`~/.hermes/.env` використовується лише для API‑ключів / секретів), **`config.yaml` є рекомендованим місцем для встановлення цих значень** у локальній розробці, on‑prem та будь‑якому розгортанні, яким ти керуєш безпосередньо. Шлях через змінні середовища існує, щоб ін’єкція секретів платформи Fly.io могла передавати `client_id` під час кожного розгортання без потреби редагувати `config.yaml` у образі — це його головна мета.

Порожні значення змінних середовища трактуються як не встановлені, тому provisioned‑but‑not‑populated Fly secret не може випадково затінити дійсний запис у `config.yaml`.

Якщо жодне джерело не надає `client_id`, плагін повідомляє конкретну причину, а помилка прив’язки з fail‑closed інформує точно, що треба виправити:

```
Refusing to bind dashboard to 0.0.0.0 — the OAuth auth gate engages on
non-loopback binds, but no auth providers are registered.

Bundled providers reported these issues:
  • nous: HERMES_DASHBOARD_OAUTH_CLIENT_ID is not set (and
    dashboard.oauth.client_id in config.yaml is empty). The Nous Portal
    provisions this env var (shape 'agent:{instance_id}') when it
    deploys a Hermes Agent instance — set it to your provisioned
    client id (either as an env var or under dashboard.oauth.client_id
    in config.yaml), or pass --insecure to skip the OAuth gate entirely.

Or pass --insecure to skip the auth gate (NOT recommended on untrusted
networks).
```

### Public URL override

За замовчуванням панель формує OAuth callback URL з запиту — `X-Forwarded-Host` + `X-Forwarded-Proto` + `X-Forwarded-Prefix` (коли uvicorn налаштовано з `proxy_headers=True`, що `start_server` вмикає під час шлюзу). Це працює «з коробки» на Fly.io, який правильно встановлює всі три заголовки.

Для розгортань за зворотними проксі, які не завжди передають ці заголовки (ручні nginx‑настройки, on‑prem ingress, кастомні домени Fly з частковими ланцюжками проксі), встанови `dashboard.public_url` (або `HERMES_DASHBOARD_PUBLIC_URL`) у **повний публічний URL**, за яким доступна панель:

```yaml
dashboard:
  public_url: "https://dashboard.example.com/hermes"
```

Коли встановлено, OAuth callback URL стає `<public_url>/auth/callback` буквально — `X-Forwarded-Prefix` ігнорується на цьому шляху, бо оператор явно вказав публічний URL. Це навмисно: додавання префікса ще раз подвоїло б його у випадку, коли префікс вже включений у `public_url`.

Той самий пріоритет, що й у інших налаштувань панелі — змінна середовища переважає `config.yaml`:

| Surface | Override path | When to use |
|---------|---------------|-------------|
| `dashboard.public_url` у `config.yaml` | `HERMES_DASHBOARD_PUBLIC_URL` | Локальна розробка / on‑prem (канонічно) |
| `HERMES_DASHBOARD_PUBLIC_URL` env var | — | Секрети платформи Fly.io / CI |
| (unset) | — | За замовчуванням — формувати з `X-Forwarded-*` заголовків |

Валідація відхиляє значення без схеми `http://` / `https://`, без хоста, або що містять лапки, кутові дужки, пробіли чи контрольні символи. Некоректне значення тихо переходить до формування з заголовків, щоб процес входу продовжився, а не перенаправляв користувача на шкідливий URL.

> **Note:** `public_url` переважає лише OAuth callback URL. Прапорець `Secure` для cookie все ще керується `request.url.scheme` (X-Forwarded-Proto при `proxy_headers`), тому `http://` `public_url` у TLS‑термінованому публічному розгортанні призведе до не‑Secure cookie. Це підводний камінь оператора — поєднуй `public_url` з правильним TLS‑термінуванням вище.

### OAuth flow

Провайдер реалізує [контракт OAuth Nous Portal v1](https://github.com/NousResearch/nous-account-service/blob/main/docs/agent-dashboard-oauth-contract.md) — grant типу authorization‑code з PKCE (S256):

1. Користувач заходить на `/` без cookie сесії → шлюз перенаправляє на `/login`.
2. На сторінці входу показується кнопка «Continue with Nous Research» → `/auth/login?provider=nous`.
3. Сервер зберігає стан PKCE у короткоживучому cookie, перенаправляє користувача до `https://portal.nousresearch.com/oauth/authorize?…`.
4. Користувач автентифікується в Portal, повертається на `/auth/callback?code=…&state=…`.
5. Сервер обмінює код на access token через `POST /api/oauth/token`, перевіряє підпис JWT проти JWKS Portal (`/.well-known/jwks.json`) і встановлює cookie `hermes_session_at`.
6. Користувач перенаправляється на `/` (або на початковий deep‑link шлях через параметр запиту `next=`).

Access token має TTL 15 хвилин. **У контракті v1 немає refresh token** — коли токен спливає, SPA‑обгортка fetch виявляє 401 і виконує повне перенаправлення назад до `/login` для повторного запуску процесу.

### Cookies set

| Name | Lifetime | Notes |
|------|----------|-------|
| `hermes_session_at` | TTL токену (15 хв) | HttpOnly, SameSite=Lax, Secure‑when‑HTTPS |
| `hermes_session_pkce` | 10 хв | HttpOnly; містить PKCE‑verifier + підказку провайдера під час циклу |
| `hermes_session_rt` | не використовується у v1 | Зарезервовано для майбутньої сумісності; не записується, коли `refresh_token` порожній |

Усі три мають `Path=/` та `SameSite=Lax`. Прапорець `Secure` встановлюється, коли панель доступна через HTTPS (визначається за схемою URL запиту — враховує `X-Forwarded-Proto` від TLS‑термінатора Fly при `proxy_headers=True`).

### Logout

Віджет у боковій панелі показує `Logged in as <user_id…> via nous` з іконкою виходу. При натисканні виконується POST до `/auth/logout`, який стирає всі cookie автентифікації панелі та перенаправляє назад до `/login`.

### Audit log

Кожен старт входу, успіх, невдача та помилка верифікації сесії записуються у вигляді JSON‑рядка до `$HERMES_HOME/logs/dashboard-auth.log`. Чутливі поля (`access_token`, `refresh_token`, `code`, `code_verifier`, `state`, заголовок `Authorization`) перед записом редагуються.

### Custom providers

Щоб підключити не‑Nous OAuth‑провайдера (наприклад Google, GitHub, кастомний OIDC), створи плагін, який реєструє `DashboardAuthProvider`:

```python
# ~/.hermes/plugins/dashboard-auth-myidp/__init__.py
from hermes_cli.dashboard_auth import DashboardAuthProvider, Session, LoginStart

class MyIdPProvider(DashboardAuthProvider):
    name = "myidp"
    display_name = "My Identity Provider"

    def start_login(self, *, redirect_uri): ...
    def complete_login(self, *, code, state, code_verifier, redirect_uri): ...
    def verify_session(self, *, access_token): ...
    def refresh_session(self, *, refresh_token): ...
    def revoke_session(self, *, refresh_token): ...

def register(ctx):
    ctx.register_dashboard_auth_provider(MyIdPProvider())
```

Сторінка входу перелічує всі зареєстровані провайдери; можна мати кілька провайдерів, і користувач обирає один на `/login`.

## Development

Якщо ти вносиш внесок у фронтенд веб‑дашборду:

```bash
# Terminal 1: start the backend API
hermes dashboard --no-open

# Terminal 2: start the Vite dev server with HMR
cd web/
npm install
npm run dev
```

Сервер розробки Vite за адресою `http://localhost:5173` проксіює запити `/api` до FastAPI‑бекенду за адресою `http://127.0.0.1:9119`.

Фронтенд побудований на React 19, TypeScript, Tailwind CSS v4 та компонентах shadcn/ui-style. Продукційні збірки записуються у `hermes_cli/web_dist/`, який FastAPI‑сервер подає як статичну SPA.
## Автоматичне збирання при оновленні

Коли ти запускаєш `hermes update`, веб‑фронтенд автоматично перебудовується, якщо доступний `npm`. Це тримає дашборд синхронізованим з оновленнями коду. Якщо `npm` не встановлено, оновлення пропускає збірку фронтенду, і `hermes dashboard` збере його під час першого запуску.
