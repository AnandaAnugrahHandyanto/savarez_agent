# Постійна пам'ять

Hermes Agent має обмежену, куровану пам'ять, яка зберігається між сесіями. Це дозволяє йому пам’ятати твої уподобання, проєкти, середовище та те, чого він навчився.

## Як це працює

Два файли складають пам'ять агента:

| Файл | Призначення | Ліміт символів |
|------|-------------|----------------|
| **MEMORY.md** | Особисті нотатки агента — факти про середовище, конвенції, вивчене | 2 200 символів (~800 токенів) |
| **USER.md** | Профіль користувача — твої уподобання, стиль спілкування, очікування | 1 375 символів (~500 токенів) |

Обидва зберігаються в `~/.hermes/memories/` і інжектуються у системний промпт як заморожений знімок на початку сесії. Агент керує власною пам'яттю за допомогою інструмента `memory` — він може додавати, замінювати або видаляти записи.

:::info
Ліміти символів тримають пам'ять сфокусованою. Коли пам'ять заповнена, агент консолідує або замінює записи, щоб звільнити місце для нової інформації.
:::

## Як пам'ять виглядає в системному промпті

На початку кожної сесії записи пам'яті завантажуються з диска і рендеряться у системний промпт як заморожений блок:

```
══════════════════════════════════════════════
MEMORY (your personal notes) [67% — 1,474/2,200 chars]
══════════════════════════════════════════════
User's project is a Rust web service at ~/code/myapi using Axum + SQLx
§
This machine runs Ubuntu 22.04, has Docker and Podman installed
§
User prefers concise responses, dislikes verbose explanations
```

Формат включає:
- Заголовок, що показує, який сховок (MEMORY або USER PROFILE)
- Відсоток використання та кількість символів, щоб агент знав ємність
- Окремі записи, розділені роздільниками `§` (символ розділу)
- Записи можуть бути багаторядковими

**Шаблон замороженого знімка:** Інжекція системного промпту здійснюється один раз на початку сесії і більше не змінюється протягом сесії. Це навмисно — зберігає кеш префіксу LLM для продуктивності. Коли агент додає/видаляє записи пам'яті під час сесії, зміни одразу зберігаються на диск, але не з’являються в системному промпті до наступного запуску сесії. Відповіді інструмента завжди показують актуальний стан.

## Дії інструмента Memory

Агент використовує інструмент `memory` з такими діями:

- **add** — Додати новий запис пам'яті
- **replace** — Замінити існуючий запис оновленим вмістом (використовує пошук підрядка через `old_text`)
- **remove** — Видалити запис, який більше не актуальний (використовує пошук підрядка через `old_text`)

Дії `read` немає — вміст пам'яті автоматично інжектується в системний промпт на початку сесії. Агент бачить свою пам'ять як частину контексту розмови.

### Пошук підрядка

Дії `replace` та `remove` використовують короткий унікальний підрядок — не потрібно вказувати повний текст запису. Параметр `old_text` має бути унікальним підрядком, який ідентифікує саме один запис:

```python
# If memory contains "User prefers dark mode in all editors"
memory(action="replace", target="memory",
       old_text="dark mode",
       content="User prefers light mode in VS Code, dark mode in terminal")
```

Якщо підрядок підходить до декількох записів, повертається помилка з проханням уточнити збіг.

## Пояснення двох цілей

### `memory` — особисті нотатки агента

Для інформації, яку агент має запам’ятати про середовище, робочі процеси та отримані уроки:

- Факти про середовище (OS, інструменти, структура проєкту)
- Конвенції та налаштування проєкту
- Дивні особливості інструментів та знайдені обхідні шляхи
- Записи діарії завершених завдань
- Навички та техніки, які спрацювали

### `user` — профіль користувача

Для інформації про ідентичність, уподобання та стиль спілкування користувача:

- Ім’я, роль, часовий пояс
- Переваги спілкування (коротко vs детально, формати)
- Дратівливі моменти та те, чого слід уникати
- Привички у робочому процесі
- Технічний рівень навичок

## Що зберігати, а що пропускати

### Зберігати (проактивно)

Агент зберігає автоматично — не потрібно просити. Він зберігає, коли навчається:

- **Уподобання користувача:** “Я віддаю перевагу TypeScript над JavaScript” → зберегти в `user`
- **Факти про середовище:** “Цей сервер працює на Debian 12 з PostgreSQL 16” → зберегти в `memory`
- **Виправлення:** “Не використовуйте `sudo` для Docker‑команд, користувач у групі docker” → зберегти в `memory`
- **Конвенції:** “Проєкт використовує табуляції, ширину рядка 120 симв., docstring‑и у стилі Google” → зберегти в `memory`
- **Завершена робота:** “Міграція бази даних з MySQL на PostgreSQL 2026‑01‑15” → зберегти в `memory`
- **Явні запити:** “Пам’ятай, що ротація мого API‑ключа відбувається щомісяця” → зберегти в `memory`

### Пропускати

- **Тривіальна/очевидна інформація:** “Користувач запитав про Python” — занадто загальне, щоб бути корисним
- **Легко відновлювані факти:** “Python 3.12 підтримує вкладені f‑string” — можна швидко знайти в інтернеті
- **Сирі дампи даних:** великі блоки коду, лог‑файли, таблиці даних — занадто великі для пам'яті
- **Ефемерні дані сесії:** тимчасові шляхи файлів, одноразовий контекст налагодження
- **Інформація, вже присутня у файлах контексту:** вміст SOUL.md та AGENTS.md

## Управління ємністю

Пам'ять має суворі ліміти символів, щоб системні промпти залишалися обмеженими:

| Сховок | Ліміт | Типові записи |
|--------|------|---------------|
| memory | 2 200 символів | 8‑15 записів |
| user   | 1 375 символів | 5‑10 записів |

### Що відбувається, коли пам'ять заповнена

Коли ти намагаєшся додати запис, який перевищить ліміт, інструмент повертає помилку:

```json
{
  "success": false,
  "error": "Memory at 2,100/2,200 chars. Adding this entry (250 chars) would exceed the limit. Replace or remove existing entries first.",
  "current_entries": ["..."],
  "usage": "2,100/2,200"
}
```

Тоді агент повинен:
1. Прочитати поточні записи (показані у відповіді помилки)
2. Визначити записи, які можна видалити або консолідувати
3. Використати `replace`, щоб об’єднати пов’язані записи у коротші варіанти
4. Потім `add` новий запис

**Краща практика:** коли пам'ять перевищує 80 % ємності (видно у заголовку системного промпту), консолідуй записи перед додаванням нових. Наприклад, об’єднай три окремі “проєкт використовує X” записи в один комплексний опис проєкту.

### Практичні приклади хороших записів пам'яті

**Компактні, інформативні записи працюють найкраще:**

```
# Good: Packs multiple related facts
User runs macOS 14 Sonoma, uses Homebrew, has Docker Desktop and Podman. Shell: zsh with oh-my-zsh. Editor: VS Code with Vim keybindings.

# Good: Specific, actionable convention
Project ~/code/api uses Go 1.22, sqlc for DB queries, chi router. Run tests with 'make test'. CI via GitHub Actions.

# Good: Lesson learned with context
The staging server (10.0.1.50) needs SSH port 2222, not 22. Key is at ~/.ssh/staging_ed25519.

# Bad: Too vague
User has a project.

# Bad: Too verbose
On January 5th, 2026, the user asked me to look at their project which is
located at ~/code/api. I discovered it uses Go version 1.22 and...
```

## Запобігання дублюванню

Система пам'яті автоматично відхиляє точні дублікати записів. Якщо спробувати додати вже існуючий вміст, вона повертає успішну відповідь з повідомленням “no duplicate added”.

## Безпекове сканування

Записи пам'яті скануються на ін’єкції та ексфільтрацію перед прийняттям, оскільки вони інжектуються в системний промпт. Вміст, що відповідає шаблонам загроз (prompt injection, credential exfiltration, SSH backdoors) або містить невидимі Unicode‑символи, блокується.

## Пошук у сесії

Окрім MEMORY.md та USER.md, агент може шукати у своїх минулих розмовах за допомогою інструмента `session_search`:

- Всі CLI та messaging сесії зберігаються у SQLite (`~/.hermes/state.db`) з повнотекстовим пошуком FTS5
- Запити повертають реальні повідомлення з БД — без підсумовування LLM, без скорочень
- Агент може знайти те, про що говорив тижнями раніше, навіть якщо це не в активній пам'яті
- Агент також може прокручувати вперед/назад у будь‑якій знайденій сесії

```bash
hermes sessions list    # Browse past sessions
```

Дивись [Session Search Tool](/user-guide/sessions#session-search-tool) для трьох форм виклику (discovery / scroll / browse) та формату відповіді.

### session_search vs memory

| Функція | Постійна пам'ять | Пошук у сесії |
|--------|------------------|--------------|
| **Ємність** | ~1 300 токенів загалом | Необмежено (всі сесії) |
| **Швидкість** | Миттєво (у системному промпті) | ~20 ms запит FTS5, ~1 ms прокрутка |
| **Вартість** | Токени у кожному промпті | Безкоштовно — без викликів LLM |
| **Випадок використання** | Ключові факти завжди в контексті | Пошук конкретних минулих розмов |
| **Керування** | Ручне, куроване агентом | Автоматичне — всі сесії зберігаються |
| **Токен‑витрати** | Фіксовано за сесію (~1 300 токенів) | За запитом (шукається лише при потребі) |

**Пам'ять** — це для критичних фактів, які мають бути завжди в контексті. **Пошук у сесії** — для запитів типу “чи обговорювали ми X минулого тижня?”, коли агент потребує згадати деталі минулих розмов.

## Конфігурація

```yaml
# In ~/.hermes/config.yaml
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200   # ~800 tokens
  user_char_limit: 1375     # ~500 tokens
```

## Зовнішні провайдери пам'яті

Для глибшої, постійної пам'яті, що виходить за межі MEMORY.md та USER.md, Hermes постачається з 8 плагінами зовнішніх провайдерів пам'яті — включаючи Honcho, OpenViking, Mem0, Hindsight, Holographic, RetainDB, ByteRover та Supermemory.

Зовнішні провайдери працюють **паралельно** зі вбудованою пам'яттю (ніколи її не замінюючи) і додають можливості, такі як графи знань, семантичний пошук, автоматичне видобування фактів та крос‑сесійне моделювання користувачів.

```bash
hermes memory setup      # pick a provider and configure it
hermes memory status     # check what's active
```

Дивись посібник [Memory Providers](./memory-providers.md) для повних деталей про кожного провайдера, інструкцій з налаштування та порівняння.
