# Проксі підписки

Проксі підписки — це локальний HTTP‑сервер, який дозволяє зовнішнім застосункам — OpenViking, Karakeep, Open WebUI, будь‑якому, що спілкується сумісно з OpenAI‑комплітаціями чату — використовувати твою підписку провайдера, керовану Hermes, як їхню LLM‑точку. Проксі додає правильні облікові дані (оновлюючи їх автоматично), тому застосунок ніколи не потребує статичного API‑ключа.

Це відрізняється від [API‑сервера](./api-server.md):

| | API‑сервер | Проксі підписки |
|---|---|---|
| Що обслуговує | Твій агент (повний набір інструментів, пам'ять, навички) | Сира модельна інференція |
| Випадок використання | «Використовувати Hermes як бекенд чату» | «Використовувати мою підписку Portal з іншого застосунку» |
| Авторизація | Твій `API_SERVER_KEY` | Будь‑який bearer (проксі додає справжній) |
| Виклики інструментів | Так — агент запускає інструменти | Ні — лише прозорий прохід |

Використовуй API‑сервер, коли потрібен **агент** як бекенд. Використовуй проксі, коли потрібна лише **модель** через твою підписку.

## Швидкий старт

### 1. Увійди в свого провайдера (одноразово)

```bash
hermes auth add nous
```

Це відкриє браузер для OAuth‑потоку Nous Portal. Hermes зберігає refresh‑токен у `~/.hermes/auth.json` — у тому самому місці, де живуть усі логіни провайдерів Hermes.

### 2. Запусти проксі

```bash
hermes proxy start
```

```
Starting Hermes proxy for Nous Portal
  Listening on:  http://127.0.0.1:8645/v1
  Forwarding to: (resolved per-request from your subscription)
  Use any bearer token in the client — the proxy attaches your real credential.
```

Залиш його працювати у передньому плані. Використовуй `tmux`, `nohup` або unit systemd, якщо хочеш, щоб він продовжував працювати після виходу з системи.

### 3. Налаштуй застосунок на використання проксі

Будь‑яка конфігурація сумісного з OpenAI застосунку приймає той самий трійник:

```
Base URL:   http://127.0.0.1:8645/v1
API key:    anything (e.g. "sk-unused")
Model:      Hermes-4-70B    # or Hermes-4.3-36B, Hermes-4-405B
```

Проксі ігнорує заголовок `Authorization` від твого застосунку і додає твої реальні облікові дані Portal до запиту вгору. Оновлення відбуваються автоматично, коли bearer наближається до закінчення терміну дії.

## Доступні провайдери

```bash
hermes proxy providers
```

Наразі постачаються: `nous` (Nous Portal) та `xai` (xAI / Grok). Інші OAuth‑провайдери можна додати, реалізувавши інтерфейс `UpstreamAdapter` у `hermes_cli/proxy/adapters/`.

## Перевірка стану

```bash
hermes proxy status
```

```
Hermes proxy upstream adapters

  [nous    ] Nous Portal — ready (bearer expires 2026-05-15T06:43:21Z)
```

Якщо бачиш `not logged in`, запусти `hermes auth add nous`. Якщо бачиш `credentials need attention`, твій refresh‑токен був відкликаний (рідко — стається, коли ти вийшов із веб‑інтерфейсу Portal) — просто знову виконай `hermes auth add nous`.

## Дозволені шляхи

Проксі пересилає лише ті шляхи, які реально обслуговує upstream. Для Nous Portal:

| Шлях | Призначення |
|------|-------------|
| `/v1/chat/completions` | Chat completions (стрімінг + без стрімінгу) |
| `/v1/completions` | Legacy text completions |
| `/v1/embeddings` | Embeddings |
| `/v1/models` | Список моделей |

Інші шляхи (`/v1/images/generations`, `/v1/audio/speech` тощо) повертають 404 з чіткою помилкою, що вказує на дозволені шляхи. Це запобігає випадковим запитам клієнтів до upstream.

## Налаштування OpenViking для використання Portal

[OpenViking](https://github.com/volcengine/OpenViking) — це база контексту, якій потрібен LLM‑провайдер для її VLM (vision/language model, використовується для витягування пам'яті) та модель embeddings. За допомогою проксі можна вказати його `vlm.api_base` на твій локальний проксі:

Відредагуй `~/.openviking/ov.conf`:

```json
{
  "vlm": {
    "provider": "openai",
    "model": "Hermes-4-70B",
    "api_base": "http://127.0.0.1:8645/v1",
    "api_key": "unused-proxy-attaches-real-creds"
  }
}
```

Потім запусти проксі в терміналі поряд з `openviking-server`:

```bash
# Terminal 1
hermes proxy start

# Terminal 2
openviking-server
```

VLM‑виклики OpenViking тепер проходять через твою підписку Portal. Модель embeddings все ще потребує власного провайдера — Portal обслуговує `/v1/embeddings`, але вибір моделі залежить від того, що підтримує твій тариф; перевір `portal.nousresearch.com/models`.

## Налаштування Karakeep (або будь‑якого застосунку закладок/резюме)

[Karakeep](https://karakeep.app/) приймає API, сумісне з OpenAI, для резюмування закладок. У його конфігурації:

```bash
# Karakeep .env
OPENAI_API_BASE_URL=http://127.0.0.1:8645/v1
OPENAI_API_KEY=any-non-empty-string
INFERENCE_TEXT_MODEL=Hermes-4-70B
```

Той самий шаблон працює для Open WebUI, LobeChat, NextChat або будь‑якого іншого клієнта, сумісного з OpenAI.

## Виставлення у LAN

За замовчуванням проксі прив'язується до `127.0.0.1` (лише localhost). Щоб інші машини у мережі могли його використовувати:

```bash
hermes proxy start --host 0.0.0.0 --port 8645
```

⚠ **Увага:** будь‑хто у твоїй мережі тепер може користуватися твоєю підпискою Portal. Проксі не має власної автентифікації — він приймає будь‑який bearer. Використовуй брандмауер, VPN або зворотний проксі з належною автентифікацією, якщо відкриваєш його за межами довіреної мережі.

## Обмеження швидкості

Обмеження RPM/TPM твого тарифу Portal застосовуються до всього проксі. Проксі не розподіляє і не пулить запити — це один bearer з повним лімітом твоєї підписки. Моніторинг використання доступний на [portal.nousresearch.com](https://portal.nousresearch.com).

## Архітектура

Проксі навмисно мінімальний. На кожен запит:

1. Отримати `POST /v1/chat/completions` від твого застосунку
2. Дізнатися поточну облікову дану адаптера (оновити, якщо закінчується)
3. Перешліфувати тіло запиту без змін, з `Authorization: Bearer <minted-key>`
4. Потоково передати відповідь назад без змін (SSE зберігається)

Ніяких трансформацій. Ніякого логування тіл запитів. Ніякого циклу агента. Проксі — це прозорий прохід, що додає облікові дані.

## Майбутнє: більше OAuth‑провайдерів

Система адаптерів розширювана. Додавання нового провайдера (наприклад, HuggingFace, чат‑ендпоінт GitHub Copilot, Anthropic через OAuth) вимагає реалізації `UpstreamAdapter` у `hermes_cli/proxy/adapters/<provider>.py` та реєстрації його в `adapters/__init__.py`. Провайдери, які не сумісні з OpenAI на рівні протоколу (наприклад Anthropic Messages API), потребуватимуть шару трансформації, що виходить за межі поточної реалізації.
