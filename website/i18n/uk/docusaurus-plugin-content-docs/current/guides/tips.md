# Поради та найкращі практики

Колекція швидких практичних порад, які одразу підвищують твою ефективність у роботі з Hermes Agent. Кожен розділ охоплює окремий аспект — переглянь заголовки та переходи до потрібного.

:::tip Не впевнений, яку модель обрати?
Запусти `hermes setup --portal` — ти отримаєш понад 300 моделей, включаючи Claude, GPT‑5 та Gemini, в рамках однієї підписки. Дивись [Nous Portal](/integrations/nous-portal).
:::

---
## Отримання найкращих результатів

### Будь конкретним щодо того, чого ти хочеш

Неоднозначні підказки дають неоднозначні результати. Замість «виправ код», скажи «виправ TypeError у `api/handlers.py` на рядку 47 — функція `process_request()` отримує `None` від `parse_body()`». Чим більше контексту ти надаєш, тим менше ітерацій потрібно.

### Надай контекст заздалегідь

Навантажуй свій запит релевантними деталями: шляхи до файлів, повідомлення про помилки, очікувана поведінка. Одне добре сформульоване повідомлення краще трьох раундів уточнень. Вставляй трасування помилок безпосередньо — агент зможе їх розпарсити.

### Використовуй файли контексту для повторюваних інструкцій

Якщо ти часто повторюєш одні й ті ж інструкції («використовуй табуляції, а не пробіли», «ми користуємося pytest», «API знаходиться за `/api/v2`»), розмісти їх у файлі `AGENTS.md`. Агент читає його автоматично в кожній сесії — без додаткових зусиль після налаштування.

### Дай агенту можливість користуватися своїми інструментами

Не намагайся контролювати кожен крок. Скажи «знайди і виправ тест, що падає» замість «відкрий `tests/test_foo.py`, подивись на рядок 42, потім…». Агент має пошук файлів, доступ до терміналу та виконання коду — дай йому можливість досліджувати та ітерувати.

### Використовуй навички для складних робочих процесів

Перш ніж писати довгу підказку, як щось зробити, перевір, чи вже існує навичка для цього. Введи `/skills`, щоб переглянути доступні навички, або просто викликай одну безпосередньо, наприклад `/axolotl` чи `/github-pr-workflow`.
## Поради для досвідчених користувачів CLI

### Багаторядковий ввід

Натисни **Alt+Enter**, **Ctrl+J** або **Shift+Enter**, щоб вставити новий рядок без відправки. `Shift+Enter` працює лише тоді, коли термінал передає його як окремий клавішний сигнал (за замовчуванням у Kitty / foot / WezTerm / Ghostty; iTerm2 / Alacritty / VS Code terminal після ввімкнення протоколу клавіатури Kitty). Інші два варіанти працюють у будь‑якому терміналі.

### Виявлення вставки

CLI автоматично розпізнає багаторядкові вставки. Просто встав кодовий блок або трасування помилки — він не надсилатиме кожен рядок окремим повідомленням. Вставка буферизується і надсилається як одне повідомлення.

### Переривання та перенаправлення

Натисни **Ctrl+C** один раз, щоб перервати агента під час відповіді. Потім можеш ввести нове повідомлення, щоб перенаправити його. Подвійне натискання **Ctrl+C** протягом 2 секунд примусово завершує роботу. Це безцінно, коли агент починає йти не тим шляхом.

### Відновлення сесій за допомогою `-c`

Забув щось зі своєї останньої сесії? Запусти `hermes -c`, щоб відновити саме там, де зупинився, з повною історією розмови. Також можна відновити за назвою: `hermes -r "my research project"`.

### Вставка зображення з буфера обміну

Натисни **Ctrl+V**, щоб вставити зображення з буфера обміну безпосередньо в чат. Агент використовує зір для аналізу скріншотів, діаграм, повідомлень про помилки або макетів інтерфейсу — не потрібно спочатку зберігати файл.

### Автодоповнення слеш‑команд

Введи `/` і натисни **Tab**, щоб побачити всі доступні команди. Це включає вбудовані команди (`/compress`, `/model`, `/title`) та всі встановлені **skill**. Не потрібно нічого запам’ятовувати — автодоповнення Tab покриє тебе.

:::tip
Використовуй `/verbose`, щоб перемикатися між режимами відображення виводу інструменту: **off → new → all → verbose**. Режим «all» чудово підходить для спостереження за діями агента; «off» — найчистіший для простих запитань‑відповідей.
:::
## Файли контексту

### AGENTS.md: Мозок твого проєкту

Створи `AGENTS.md` у корені проєкту з рішеннями архітектури, конвенціями коду та інструкціями, специфічними для проєкту. Цей файл автоматично включається в кожну сесію, тому агент завжди знає правила твого проєкту.

```markdown
# Project Context
- This is a FastAPI backend with SQLAlchemy ORM
- Always use async/await for database operations
- Tests go in tests/ and use pytest-asyncio
- Never commit .env files
```

### SOUL.md: Налаштування особистості

Хочеш, щоб Hermes мав стабільний голос за замовчуванням? Відредагуй `~/.hermes/SOUL.md` (або `$HERMES_HOME/SOUL.md`, якщо використовуєш власний домашній каталог Hermes). Hermes тепер автоматично створює стартовий SOUL і використовує цей глобальний файл як джерело особистості для всіх інстанцій.

Для повного посібника дивись [Use SOUL.md with Hermes](/guides/use-soul-with-hermes).

```markdown
# Soul
You are a senior backend engineer. Be terse and direct.
Skip explanations unless asked. Prefer one-liners over verbose solutions.
Always consider error handling and edge cases.
```

Використовуй `SOUL.md` для стійкої особистості. Використовуй `AGENTS.md` для інструкцій, специфічних для проєкту.

### .cursorrules Compatibility

Вже маєш файл `.cursorrules` або `.cursor/rules/*.mdc`? Hermes читає їх також. Не потрібно дублювати свої конвенції коду — вони завантажуються автоматично з робочого каталогу.

### Discovery

Hermes завантажує `AGENTS.md` верхнього рівня з поточного робочого каталогу під час запуску сесії. Файли `AGENTS.md` у підкаталогах виявляються ліниво під час викликів інструментів (через `subdirectory_hints.py`) і включаються до результатів інструменту — вони не завантажуються заздалегідь у системний підказник.

:::tip
Тримай файли контексту сфокусованими та стислими. Кожен символ враховується у твоєму токен‑бюджеті, оскільки вони включаються в кожне окреме повідомлення.
:::
## Пам'ять та навички

### Пам'ять vs. Навички: що куди відносити

**Пам'ять** — це факти: твоє середовище, уподобання, розташування проєктів і те, що агент дізнався про тебе. **Навички** — це процедури: багатокрокові робочі процеси, інструкції, специфічні для інструменту, та повторно використовувані рецепти. Використовуй пам'ять для «що», навички — для «як».

### Коли створювати навички

Якщо ти виявив завдання, яке займає 5 + кроків і ти будеш виконувати його знову, попроси агента створити навичку для нього. Скажи «збережи те, що ти щойно зробив, як навичку під назвою `deploy‑staging`». Наступного разу просто введи `/deploy-staging`, і агент завантажить повну процедуру.

### Управління обсягом пам'яті

Пам'ять навмисно обмежена (~2 200 символів для `MEMORY.md`, ~1 375 символів для `USER.md`). Коли вона заповнюється, агент консолідує записи. Ти можеш допомогти, сказавши «очисти свою пам'ять» або «заміни стару нотатку про Python 3.9 — тепер у нас 3.12».

### Дай агенту запам'ятати

Після продуктивної сесії скажи «запам'ятай це на наступний раз», і агент збереже ключові висновки. Можеш бути конкретним: «збережи в пам'ять, що наш CI використовує GitHub Actions з робочим процесом `deploy.yml`».

:::warning
Пам'ять — це заморожений знімок: зміни, внесені під час сесії, не з'являються в системному запиті, доки не розпочнеться наступна сесія. Агент записує дані на диск одразу, але кеш запиту не оновлюється посередині сесії.
:::
## Продуктивність та вартість

### Не порушуй кеш підказок

Більшість провайдерів LLM кешують префікс системної підказки. Якщо ти підтримуєш стабільність системної підказки (однакові файли контексту, однакова пам'ять), наступні повідомлення у сесії отримують **влучення в кеш**, що значно дешевше. Уникай зміни моделі або системної підказки під час сесії.

### Використовуй /compress перед досягненням лімітів

Довгі сесії накопичують токени. Коли ти помічаєш, що відповіді сповільнюються або обрізаються, запусти `/compress`. Це підсумовує історію розмови, зберігаючи ключовий контекст і різко зменшуючи кількість токенів. Використовуй `/usage`, щоб перевірити поточний стан.

### Делегуй для паралельної роботи

Потрібно дослідити три теми одночасно? Попроси агента використати `delegate_task` з паралельними підзадачами. Кожен підагент працює незалежно зі своїм контекстом, і лише підсумкові резюме повертаються — це значно зменшує використання токенів у головній розмові.

### Використовуй execute_code для пакетних операцій

Замість запуску термінальних команд по одній, попроси агента написати скрипт, який виконає все одразу. «Write a Python script to rename all `.jpeg` files to `.jpg` and run it» дешевше і швидше, ніж перейменовувати файли поодинці.

### Обери правильну модель

Використовуй `/model`, щоб переключати моделі під час сесії. Використовуй модель‑фронтир (Claude Sonnet/Opus, GPT-4o) для складних роздумів і архітектурних рішень. Перемикайся на швидшу модель для простих завдань, таких як форматування, перейменування чи генерація шаблонного коду.

:::tip
Запускай `/usage` періодично, щоб бачити споживання токенів. Запускай `/insights` для ширшого огляду шаблонів використання за останні 30 днів.
:::
## Поради щодо обміну повідомленнями

### Встановити домашній канал

Використай `/sethome` у улюбленому чаті Telegram або Discord, щоб позначити його як домашній канал. Результати **cron**‑завдань та вивід запланованих задач будуть доставлені сюди. Без цього агент не має куди надсилати проактивні повідомлення.

### Використовуй /title для організації сесій

Назви свої сесії за допомогою `/title auth-refactor` або `/title research-llm-quantization`. Іменовані сесії легко знайти за допомогою `hermes sessions list` і відновити командою `hermes -r "auth-refactor"`. Без назви сесії накопичуються і їх неможливо розрізнити.

### DM‑парування для доступу команди

Замість ручного збирання ID користувачів для allowlist‑ів увімкни DM‑парування. Коли учасник команди надсилає боту DM, він отримує одноразовий код парування. Ти підтверджуєш його командою `hermes pairing approve telegram XKGH5N7P` — просто і безпечно.

### Режими відображення прогресу інструменту

Використай `/verbose`, щоб керувати кількістю видимих дій інструменту. У платформах обміну повідомленнями менше зазвичай краще — залишай режим «new», щоб бачити лише нові виклики інструменту. У CLI режим «all» дає задовільний живий перегляд усіх дій агента.

:::tip
У платформах обміну повідомленнями сесії автоматично скидаються після часу бездіяльності (за замовчуванням: 24 години) або щодня о 4 ранку. За потреби налаштуй це в `~/.hermes/config.yaml`, якщо потрібні довші сесії.
:::
## Безпека

### Використовуй Docker для ненадійного коду

Коли працюєш з ненадійними репозиторіями або запускаєш незнайомий код, використай Docker або Daytona як бекенд терміналу. Встанови `TERMINAL_BACKEND=docker` у своєму `.env`. Шкідливі команди всередині контейнера не можуть нашкодити твоїй хост‑системі.

```bash
# In your .env:
TERMINAL_BACKEND=docker
TERMINAL_DOCKER_IMAGE=hermes-sandbox:latest
```

### Уникай підводних каменів кодування в Windows

У Windows деякі кодування за замовчуванням (наприклад, `cp125x`) не можуть представити всі символи Unicode, що може спричинити `UnicodeEncodeError` під час запису файлів у тестах або скриптах.

- Віддавай перевагу відкриттю файлів з явним кодуванням UTF‑8:

```python
with open("results.txt", "w", encoding="utf-8") as f:
    f.write("✓ All good\n")
```

- У PowerShell ти також можеш переключити поточну сесію на UTF‑8 для консолі та виводу нативних команд:

```powershell
$OutputEncoding = [Console]::OutputEncoding = [Text.UTF8Encoding]::new($false)
```

Це залишає PowerShell і дочірні процеси у UTF‑8 і допомагає уникнути помилок, притаманних лише Windows.

### Переглядай перед вибором «Always»

Коли агент ініціює схвалення небезпечної команди (`rm -rf`, `DROP TABLE` тощо), ти отримуєш чотири варіанти: **once**, **session**, **always**, **deny**. Подумай двічі, перш ніж вибирати «always» — це назавжди додає шаблон до білосписку. Починай з «session», доки не відчуєш впевненості.

### Схвалення команд — твоя система безпеки

Hermes перевіряє кожну команду проти підготовленого списку небезпечних шаблонів перед виконанням. Це включає рекурсивне видалення, SQL‑дропи, передавання `curl` у shell тощо. Не вимикай це у продакшені — це існує з вагомих причин.

:::warning
Коли працюєш у бекенді контейнера (Docker, Singularity, Modal, Daytona), перевірка небезпечних команд **пропускається**, оскільки контейнер є межою безпеки. Переконайся, що образи контейнерів належним чином захищені.
:::

### Використовуй білосписки для ботів обміну повідомленнями

Ніколи не встановлюй `GATEWAY_ALLOW_ALL_USERS=true` у бота з доступом до терміналу. Завжди використовуйте платформо‑специфічні білосписки (`TELEGRAM_ALLOWED_USERS`, `DISCORD_ALLOWED_USERS`) або DM‑парування для контролю, хто може взаємодіяти з твоїм агентом.

```bash
# Recommended: explicit allowlists per platform
TELEGRAM_ALLOWED_USERS=123456789,987654321
DISCORD_ALLOWED_USERS=123456789012345678

# Or use cross-platform allowlist
GATEWAY_ALLOWED_USERS=123456789,987654321
```
