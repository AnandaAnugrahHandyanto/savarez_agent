# Вирішення проблем з Cron

Коли завдання cron не працює так, як очікується, виконуй ці перевірки послідовно. Більшість проблем належать до однієї з чотирьох категорій: таймінг, доставка, дозволи або завантаження навичок.

---
## Роботи не запускаються

### Перевірка 1: Переконайся, що робота існує та активна

```bash
hermes cron list
```

Знайди роботу і підтвердь, що її стан `[active]` (а не `[paused]` чи `[completed]`). Якщо показано `[completed]`, лічильник повторень може бути вичерпано — відредагуй роботу, щоб скинути його.

### Перевірка 2: Переконайся, що розклад правильний

Неправильно сформований розклад без повідомлення про помилку переходить у одноразовий режим або взагалі відхиляється. Перевір свій вираз:

| Твій вираз | Має оцінюватись як |
|----------------|-------------------|
| `0 9 * * *` | 9:00 ранку щодня |
| `0 9 * * 1` | 9:00 ранку кожного понеділка |
| `every 2h` | Кожні 2 години від зараз |
| `30m` | 30 хвилин від зараз |
| `2025-06-01T09:00:00` | 1 червня 2025 о 9:00 UTC |

Якщо робота запускається один раз і потім зникає зі списку, це одноразовий розклад (`30m`, `1d` або ISO‑таймстамп) — очікувана поведінка.

### Перевірка 3: Чи працює шлюз?

Cron‑роботи запускаються фоновим потоком‑таймером шлюзу, який тикає кожні 60 секунд. Звичайна CLI‑сесія чату **не** запускає cron‑роботи автоматично.

Якщо ти очікуєш автоматичний запуск робіт, потрібен запущений шлюз (`hermes gateway` у передньому плані або `hermes gateway start` для встановленої служби). Для одноразового налагодження можна вручну запустити тикер командою `hermes cron tick`.

### Перевірка 4: Перевір системний час і часовий пояс

Роботи використовують локальний часовий пояс. Якщо годинник твоєї машини неправильний або встановлений інший часовий пояс, роботи спрацюватимуть у неправильний час. Перевір:

```bash
date
hermes cron list   # Compare next_run times with local time
```

## Помилки доставки

### Перевірка 1: Переконайся, що ціль доставки вказана правильно

Цілі доставки чутливі до регістру і вимагають налаштованої платформи. Неправильно налаштована ціль без повідомлення відкидає відповідь.

| Ціль | Вимоги |
|--------|----------|
| `telegram` | `TELEGRAM_BOT_TOKEN` у `~/.hermes/.env` |
| `discord` | `DISCORD_BOT_TOKEN` у `~/.hermes/.env` |
| `slack` | `SLACK_BOT_TOKEN` у `~/.hermes/.env` |
| `whatsapp` | налаштований шлюз WhatsApp |
| `signal` | налаштований шлюз Signal |
| `matrix` | налаштований homeserver Matrix |
| `email` | SMTP‑настройки у `config.yaml` |
| `sms` | налаштований провайдер SMS |
| `local` | права запису у `~/.hermes/cron/output/` |
| `origin` | надсилає у чат, де створено завдання |

Інші підтримувані платформи: `mattermost`, `homeassistant`, `dingtalk`, `feishu`, `wecom`, `weixin`, `bluebubbles`, `qqbot` та `webhook`. Також можна вказати конкретний чат у форматі `platform:chat_id` (наприклад, `telegram:-1001234567890`).

Якщо доставка не вдається, завдання все одно виконується — просто нікуди не надсилає. Перевір `hermes cron list` для оновленого поля `last_error` (за наявності).

### Перевірка 2: Використання `[SILENT]`

Якщо твій cron‑завдання не генерує вивід або агент відповідає `[SILENT]`, доставка придушується. Це навмисно для моніторингових завдань — але переконайся, що підказка випадково не придушує все.

Підказка типу «відповідай `[SILENT]`, якщо нічого не змінилося» також без повідомлень поглинає непорожні відповіді. Перевір свою умовну логіку.

### Перевірка 3: Права токенів платформ

Кожен бот у платформі обміну повідомленнями потребує певних прав для отримання повідомлень. Якщо доставка беззвучно не відбувається:

- **Telegram**: бот має бути адміністратором у цільовій групі/каналі
- **Discord**: бот повинен мати право надсилати повідомлення у цільовому каналі
- **Slack**: бот має бути доданим до робочого простору і мати область `chat:write`

### Перевірка 4: Обгортка відповіді

За замовчуванням відповіді cron обгортаються заголовком і підвалом (`cron.wrap_response: true` у `config.yaml`). Деякі платформи або інтеграції можуть не коректно обробляти це. Щоб вимкнути:

```yaml
cron:
  wrap_response: false
```

## Помилки завантаження навичок

### Перевірка 1: Перевір, чи встановлені навички

```bash
hermes skills list
```

Навички мають бути встановлені, перш ніж їх можна приєднати до cron‑завдань. Якщо якась навичка відсутня, спочатку встанови її за допомогою `hermes skills install <skill-name>` або через `/skills` у CLI.

### Перевірка 2: Перевірка назви навички та назви папки навички

Назви навичок чутливі до регістру і повинні збігатися з назвою папки встановленої навички. Якщо твоє завдання вказує `ai-funding-daily-report`, а назва папки навички інша, підтвердь точну назву за допомогою `hermes skills list`.

### Перевірка 3: Навички, які потребують інтерактивних інструментів

Cron‑завдання виконуються з вимкненими наборами інструментів `cronjob`, `messaging` та `clarify`. Це запобігає рекурсивному створенню cron‑завдань, прямому надсиланню повідомлень (доставка здійснюється планувальником) та інтерактивним підказкам. Якщо навичка покладається на ці набори інструментів, вона не працюватиме в контексті cron.

Перевір документацію навички, щоб переконатися, що вона працює в неінтерактивному (headless) режимі.

### Перевірка 4: Порядок завантаження кількох навичок

При використанні кількох навичок вони завантажуються у зазначеному порядку. Якщо навичка A залежить від контексту, отриманого від навички B, переконайся, що B завантажується першою:

```bash
/cron add "0 9 * * *" "..." --skill context-skill --skill target-skill
```

У цьому прикладі `context-skill` завантажується перед `target-skill`.
## Помилки та збої завдань

### Перевірка 1: Переглянь недавній вивід завдання

Якщо завдання виконувалося і завершилось помилкою, ти можеш побачити контекст помилки у:

1. Чаті, де завдання доставляє результат (якщо доставка була успішною)
2. `~/.hermes/logs/agent.log` для повідомлень планувальника (або `errors.log` для попереджень)
3. Метаданих `last_run` завдання через `hermes cron list`

### Перевірка 2: Типові шаблони помилок

**"No such file or directory" для скриптів**
Шлях `script` має бути абсолютним (або відносним до каталогу конфігурації Hermes). Перевір:
```bash
ls ~/.hermes/scripts/your-script.py   # Must exist
hermes cron edit <job_id> --script ~/.hermes/scripts/your-script.py
```

**"Skill not found" під час виконання завдання**
Skill має бути встановлений на машині, де працює планувальник. Якщо ти перемикаєшся між машинами, skills не синхронізуються автоматично — перевстанови їх за допомогою `hermes skills install <skill-name>`.

**Завдання виконується, але нічого не доставляє**
Швидше за все проблема з ціллю доставки (див. **Delivery Failures** вище) або тихо придушена відповідь (`[SILENT]`).

**Завдання зависає або перевищує час**
Планувальник використовує тайм‑аут на основі бездіяльності (за замовчуванням 600 с, налаштовується через змінну середовища `HERMES_CRON_TIMEOUT`, `0` — без обмежень). Агент може працювати стільки, скільки активно викликає інструменти — таймер спрацьовує лише після тривалої бездіяльності. Довгі завдання слід виконувати у скриптах, які збирають дані та доставляють лише результат.

### Перевірка 3: Конкуренція за блокування

Планувальник використовує файлове блокування, щоб запобігти накладанню тиків. Якщо запущено два екземпляри шлюзу (або CLI‑сесія конфліктує з шлюзом), завдання можуть бути затримані або пропущені.

Завершити дублікати процесів шлюзу:
```bash
ps aux | grep hermes
# Kill duplicate processes, keep only one
```

### Перевірка 4: Права доступу до `jobs.json`

Завдання зберігаються у `~/.hermes/cron/jobs.json`. Якщо цей файл недоступний для читання/запису твоїм користувачем, планувальник завершиться тихо:
```bash
ls -la ~/.hermes/cron/jobs.json
chmod 600 ~/.hermes/cron/jobs.json   # Your user should own it
```
## Проблеми продуктивності

### Повільний запуск завдання

Кожне cron‑завдання створює нову сесію AIAgent, що може включати автентифікацію провайдера та завантаження моделі. Для розкладів, чутливих до часу, додай буферний час (наприклад, `0 8 * * *` замість `0 9 * * *`).

### Забагато одночасних завдань

Планувальник виконує завдання послідовно в кожному тикі. Якщо кілька завдань заплановано на один і той самий час, вони виконуються одне за іншим. Розподіли розклади (наприклад, `0 9 * * *` і `5 9 * * *` замість обох у `0 9 * * *`), щоб уникнути затримок.

### Великий вивід скрипту

Скрипти, які виводять мегабайти даних, сповільнюватимуть агента і можуть перевищити ліміти токенів. Фільтруй/підсумовуй на рівні скрипту — виводь лише те, що агент потребує для роздумів.
## Команди діагностики

```bash
hermes cron list                    # Show all jobs, states, next_run times
hermes cron run <job_id>            # Schedule for next tick (for testing)
hermes cron edit <job_id>           # Fix configuration issues
hermes logs                         # View recent Hermes logs
hermes skills list                  # Verify installed skills
```

## Отримання додаткової допомоги

Якщо ти пройшов цей посібник, а проблема залишилася:

1. Запусти завдання за допомогою `hermes cron run <job_id>` (виконується на наступному тику шлюзу) і стеж за помилками у виводі чату
2. Перевір `~/.hermes/logs/agent.log` на предмет повідомлень планувальника та `~/.hermes/logs/errors.log` на предмет попереджень
3. Відкрий issue на [github.com/NousResearch/hermes-agent](https://github.com/NousResearch/hermes-agent) з:
   - ідентифікатором завдання та розкладом
   - ціллю доставки
   - очікуваним результатом і тим, що сталося
   - відповідними повідомленнями про помилки з журналів
