# Запуск локальних LLM на Mac

Цей посібник проведе тебе через запуск локального LLM‑серверу на macOS з API, сумісним з OpenAI. Ти отримуєш повну приватність, нульові витрати на API та несподівано хорошу продуктивність на Apple Silicon.

Ми розглядаємо два бекенди:

| Backend | Встановлення | Найкраще для | Формат |
|---------|--------------|--------------|--------|
| **llama.cpp** | `brew install llama.cpp` | Найшвидший час до першого токену, квантизований KV‑кеш для низької пам’яті | GGUF |
| **omlx** | [omlx.ai](https://omlx.ai) | Найшвидша генерація токенів, нативна оптимізація Metal | MLX (safetensors) |

Обидва надають кінцеву точку `/v1/chat/completions`, сумісну з OpenAI. Hermes працює з будь‑яким — просто вкажи `http://localhost:8080` або `http://localhost:8000`.

:::info Apple Silicon only
Цей посібник орієнтований на Mac з Apple Silicon (M1 та новіші). Intel‑Mac працюватимуть з llama.cpp, але без прискорення GPU — очікуй значно нижчу продуктивність.
:::

## Вибір моделі

Для старту радимо **Qwen3.5-9B** — це потужна модель розуміння, яка комфортно вміщується в 8 ГБ+ уніфікованої пам’яті з квантизацією.

| Variant | Size on disk | RAM needed (128K context) | Backend |
|---------|-------------|---------------------------|---------|
| Qwen3.5-9B‑Q4_K_M (GGUF) | 5.3 GB | ~10 GB з квантизованим KV‑кешем | llama.cpp |
| Qwen3.5-9B‑mlx‑lm‑mxfp4 (MLX) | ~5 GB | ~12 GB | omlx |

**Правило пам’яті:** розмір моделі + KV‑кеш. 9B Q4 модель — ~5 GB. KV‑кеш при 128K контексті з Q4 квантизацією додає ~4‑5 GB. При типі кешу f16 це зростає до ~16 GB. Прапорці квантизації KV‑кешу в llama.cpp — ключовий трюк для систем з обмеженою пам’яттю.

Для більших моделей (27B, 35B) потрібна 32 GB+ уніфікованої пам’яті. 9B — оптимальний варіант для машин з 8‑16 GB.

## Варіант A: llama.cpp

llama.cpp — найпортативніший локальний runtime LLM. На macOS він використовує Metal для прискорення GPU «з коробки».

### Встановлення

```bash
brew install llama.cpp
```

Це встановить глобальну команду `llama-server`.

### Завантаження моделі

Тобі потрібна модель у форматі GGUF. Найпростіше отримати її з Hugging Face через `huggingface-cli`:

```bash
brew install huggingface-cli
```

Потім завантаж:

```bash
huggingface-cli download unsloth/Qwen3.5-9B-GGUF Qwen3.5-9B-Q4_K_M.gguf --local-dir ~/models
```

:::tip Gated models
Деякі моделі на Hugging Face вимагають автентифікації. Спочатку виконай `huggingface-cli login`, якщо отримав помилку 401 або 404.
:::

### Запуск сервера

```bash
llama-server -m ~/models/Qwen3.5-9B-Q4_K_M.gguf \
  -ngl 99 \
  -c 131072 \
  -np 1 \
  -fa on \
  --cache-type-k q4_0 \
  --cache-type-v q4_0 \
  --host 0.0.0.0
```

Що робить кожен прапорець:

| Flag | Purpose |
|------|---------|
| `-ngl 99` | Переносить усі шари на GPU (Metal). Використай велике число, щоб нічого не залишалося на CPU. |
| `-c 131072` | Розмір вікна контексту (128K токенів). Зменшуй, якщо бракує пам’яті. |
| `-np 1` | Кількість паралельних слотів. Тримай 1 для однокористувацького використання — більше слотів розподіляє твою пам’ять. |
| `-fa on` | Flash attention. Зменшує використання пам’яті та прискорює інференс довгих контекстів. |
| `--cache-type-k q4_0` | Квантизує кеш ключів до 4‑біт. **Це головний економічний трюк.** |
| `--cache-type-v q4_0` | Квантизує кеш значень до 4‑біт. Разом з попереднім зменшує пам’ять KV‑кешу приблизно на 75 % порівняно з f16. |
| `--host 0.0.0.0` | Слухає на всіх інтерфейсах. Використай `127.0.0.1`, якщо не потрібен мережевий доступ. |

Сервер готовий, коли ти бачиш:

```
main: server is listening on http://0.0.0.0:8080
srv  update_slots: all slots are idle
```

### Оптимізація пам’яті для обмежених систем

Прапорці `--cache-type-k q4_0 --cache-type-v q4_0` — найважливіша оптимізація для систем з обмеженою пам’яттю. Ось їх вплив при 128K контексті:

| KV cache type | KV cache memory (128K ctx, 9B model) |
|---------------|--------------------------------------|
| f16 (default) | ~16 GB |
| q8_0 | ~8 GB |
| **q4_0** | **~4 GB** |

На Mac з 8 GB використай KV‑кеш `q4_0` і обери меншу модель, що все ще вміщується у мінімальний контекст Hermes — 64K. На 16 GB можна комфортно працювати з 128K контекстом. На 32 GB+ можна запускати більші моделі або кілька паралельних слотів.

Якщо пам’яті все ще не вистачає, зменш контекст, залишаючись принаймні на 64K, або переключись на меншу модель чи іншу квантизацію (Q3_K_M замість Q4_K_M).

### Перевірка

```bash
curl -s http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen3.5-9B-Q4_K_M.gguf",
    "messages": [{"role": "user", "content": "Hello!"}],
    "max_tokens": 50
  }' | jq .choices[0].message.content
```

### Отримання назви моделі

Якщо забув назву моделі, запитай endpoint моделей:

```bash
curl -s http://localhost:8080/v1/models | jq '.data[].id'
```

## Варіант B: MLX via omlx

[omlx](https://omlx.ai) — нативний macOS‑додаток, що керує та сервісує MLX‑моделі. MLX — це власний фреймворк Apple для машинного навчання, оптимізований саме під уніфіковану пам’ять Apple Silicon.

### Встановлення

Завантаж та встанови з [omlx.ai](https://omlx.ai). Додаток надає GUI для керування моделями та вбудований сервер.

### Завантаження моделі

Використай додаток omlx, щоб переглянути та завантажити моделі. Шукай `Qwen3.5-9B-mlx-lm-mxfp4` і завантаж її. Моделі зберігаються локально (зазвичай у `~/.omlx/models/`).

### Запуск сервера

omlx сервісує моделі за `http://127.0.0.1:8000` за замовчуванням. Запусти сервіс через UI додатку або, якщо доступний, через CLI.

### Перевірка

```bash
curl -s http://127.0.0.1:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen3.5-9B-mlx-lm-mxfp4",
    "messages": [{"role": "user", "content": "Hello!"}],
    "max_tokens": 50
  }' | jq .choices[0].message.content
```

### Список доступних моделей

omlx може одночасно сервісувати кілька моделей:

```bash
curl -s http://127.0.0.1:8000/v1/models | jq '.data[].id'
```

## Benchmarks: llama.cpp vs MLX

Обидва бекенди протестовані на одній машині (Apple M5 Max, 128 GB уніфікованої пам’яті) з тією ж моделлю (Qwen3.5-9B) при порівнянних рівнях квантизації (Q4_K_M для GGUF, mxfp4 для MLX). П’ять різноманітних запитів, три запуску кожного, бекенди тестувалися послідовно, щоб уникнути конкуренції ресурсів.

### Результати

| Metric | llama.cpp (Q4_K_M) | MLX (mxfp4) | Winner |
|--------|-------------------|-------------|--------|
| **TTFT (avg)** | **67 ms** | 289 ms | llama.cpp (4.3× швидше) |
| **TTFT (p50)** | **66 ms** | 286 ms | llama.cpp (4.3× швидше) |
| **Generation (avg)** | 70 tok/s | **96 tok/s** | MLX (37 % швидше) |
| **Generation (p50)** | 70 tok/s | **96 tok/s** | MLX (37 % швидше) |
| **Total time (512 tokens)** | 7.3 s | **5.5 s** | MLX (25 % швидше) |

### Що це означає

- **llama.cpp** переважає у обробці запиту — його flash attention + квантизований KV‑кеш дає перший токен за ~66 ms. Якщо ти створюєш інтерактивні застосунки, де важлива реакція (чат‑боти, автодоповнення), це суттєва перевага.
- **MLX** генерує токени ~37 % швидше після старту. Для пакетних навантажень, довгих генерацій або будь‑яких задач, де важливий загальний час завершення, MLX завершує швидше.
- Обидва бекенди **надзвичайно стабільні** — варіація між запусками практично відсутня. Ти можеш покластися на ці цифри.

### Який варіант обрати?

| Use case | Recommendation |
|----------|---------------|
| Інтерактивний чат, інструменти з низькою затримкою | llama.cpp |
| Довгі генерації, масова обробка | MLX (omlx) |
| Обмежена пам’ять (8‑16 GB) | llama.cpp (квантизований KV‑кеш неперевершений) |
| Сервісування кількох моделей одночасно | omlx (вбудована підтримка мульти‑моделей) |
| Максимальна сумісність (Linux теж) | llama.cpp |

## Підключення до Hermes

Коли твій локальний сервер запущений:

```bash
hermes model
```

Вибери **Custom endpoint** і слідуй підказкам. Тебе попросять ввести базовий URL та назву моделі — використай значення з того бекенду, який ти налаштував вище.

## Тайм‑ауты

Hermes автоматично виявляє локальні кінцеві точки (localhost, LAN IP) і пом’якшує тайм‑аут стрімінгу. Ніяких налаштувань не потрібно для більшості випадків.

Якщо все ж отримуєш тайм‑аут (наприклад, дуже великі контексти на повільному обладнанні), можеш перевизначити тайм‑аут читання стріму:

```bash
# In your .env — raise from the 120s default to 30 minutes
HERMES_STREAM_READ_TIMEOUT=1800
```

| Timeout | Default | Local auto-adjustment | Env var override |
|---------|---------|----------------------|------------------|
| Stream read (socket‑level) | 120 s | Підвищено до 1800 s | `HERMES_STREAM_READ_TIMEOUT` |
| Stale stream detection | 180 s | Вимкнено повністю | `HERMES_STREAM_STALE_TIMEOUT` |
| API call (non‑streaming) | 1800 s | Без змін | `HERMES_API_TIMEOUT` |

Тайм‑аут читання стріму — найчастіша причина проблем: це дедлайн на рівні сокету для отримання наступного фрагмента даних. Під час поповнення великого контексту локальні моделі можуть не видавати вихідні дані кілька хвилин, обробляючи запит. Автовиявлення обробляє це прозоро.
