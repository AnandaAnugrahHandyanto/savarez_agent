# Довідка щодо Slash‑команд

Hermes має два інтерфейси slash‑команд, обидва керуються центральним `COMMAND_REGISTRY` у `hermes_cli/commands.py`:

- **Інтерактивні CLI slash‑команди** — викликаються `cli.py`, з автодоповненням із реєстру
- **Messaging slash‑команди** — викликаються `gateway/run.py`, з текстом довідки та меню платформ, згенерованими з реєстру

Встановлені **skill** також доступні як динамічні slash‑команди на обох інтерфейсах. Це включає вбудовані **skill**, такі як `/plan`, яка відкриває режим планування та зберігає markdown‑плани у `.hermes/plans/` відносно активної робочої теки/каталогу бекенду.
## Дозволи та розподіл admin/користувач

Кожна платформа обміну повідомленнями, яка підтримує білий список per‑user (Telegram, Discord, Slack, Matrix, Mattermost, Signal, …), також підтримує двоступеневий розподіл slash‑команд: **admin** отримують усі зареєстровані команди, **звичайні користувачі** — лише ті назви, які вказані в `user_allowed_commands` (плюс завжди дозволений базовий `/help` та `/whoami`). Налаштуй `allow_admin_from` і `user_allowed_commands` (а також еквіваленти per‑group `group_allow_admin_from` / `group_user_allowed_commands`) у блоці `extra:` файлу `~/.hermes/gateway-config.yaml`.

Дивись документацію per‑platform для прикладів — структура ідентична на всіх платформах:

- [Telegram](../user-guide/messaging/telegram.md#slash-command-access-control)
- [Discord](../user-guide/messaging/discord.md)
- [Slack](../user-guide/messaging/slack.md)
- [Matrix](../user-guide/messaging/matrix.md)
- [Mattermost](../user-guide/messaging/mattermost.md)
- [Signal](../user-guide/messaging/signal.md)

Якщо `allow_admin_from` не встановлено для певної області, ця область залишатиметься у режимі необмеженої зворотної сумісності — кожен дозволений користувач зможе виконувати будь‑яку команду.
## Інтерактивні CLI slash‑команди

Введи `/` у CLI, щоб відкрити меню автодоповнення. Вбудовані команди не чутливі до регістру.
### Сесія

| Команда | Опис |
|---------|------|
| `/new [name]` (alias: `/reset`) | Запустити нову сесію (новий ID сесії + історія). Необов’язковий `[name]` задає початкову назву сесії — напр., `/new my-experiment` відкриває нову сесію вже з назвою `my-experiment`, що полегшує пошук пізніше за допомогою `/resume` або `/sessions`. Додай `now`, `--yes` або `-y`, щоб пропустити модальне підтвердження — напр., `/reset now`, `/new --yes my-experiment`. |
| `/clear` | Очистити екран і запустити нову сесію |
| `/history` | Показати історію розмови |
| `/save` | Зберегти поточну розмову |
| `/retry` | Повторити останнє повідомлення (надіслати агенту ще раз) |
| `/undo` | Видалити останню взаємодію користувач‑асистент |
| `/title` | Встановити назву для поточної сесії (використання: `/title My Session Name`) |
| `/compress [focus topic]` | Ручне стискання контексту розмови (очищення пам’яті + підсумок). Необов’язкова тема фокусу звужує, що саме зберігається у підсумку. |
| `/rollback` | Переглянути або відновити контрольні точки файлової системи (використання: `/rollback [number]`) |
| `/snapshot [create\|restore <id>\|prune]` (alias: `/snap`) | Створити або відновити знімки стану конфігурації/стану Hermes. `create [label]` зберігає знімок, `restore <id>` повертає до нього, `prune [N]` видаляє старі знімки, або просто вивести список без аргументів. |
| `/stop` | Припинити всі запущені фонові процеси |
| `/queue <prompt>` (alias: `/q`) | Додати запит у чергу на наступний хід (не перериває поточну відповідь агента). |
| `/steer <prompt>` | Вставити нотатку під час виконання, яка з’явиться у агента **після наступного виклику інструменту** — без переривання, без нового ходу користувача. Текст додається до вмісту останнього результату інструменту після його завершення, надаючи агенту новий контекст без порушення циклу викликів інструментів. Використовуй для корекції напрямку під час завдання (наприклад, “зосередитися на модулі auth”, коли агент виконує тести). |
| `/goal <text>` | Встановити постійну мету, до якої Hermes працює протягом ходів — наш варіант циклу Ralph. Після кожного ходу допоміжна модель‑суддя вирішує, чи досягнуто мети; якщо ні, Hermes продовжує автоматично. Підкоманди: `/goal status`, `/goal pause`, `/goal resume`, `/goal clear`. Бюджет за замовчуванням — 20 ходів (`goals.max_turns`); будь‑яке реальне повідомлення користувача перериває цикл, а стан зберігається після `/resume`. Дивись [Persistent Goals](/user-guide/features/goals) для повного огляду. |
| `/subgoal <text>` | Додати користувацький критерій до активної мети під час циклу. Промпт продовження передає всі підцілі агенту дослівно, і суддя враховує їх у своїй оцінці DONE/CONTINUE — тому мета не вважається виконаною, доки не виконано оригінальну мету **і** всі підцілі. Підкоманди: `/subgoal` (список), `/subgoal remove <N>`, `/subgoal clear`. Потрібна активна `/goal`. |
| `/resume [name]` | Відновити раніше названу сесію |
| `/sessions` (TUI alias: `/switch`) | Класичний CLI: переглянути та відновити попередні сесії у інтерактивному вибірнику. TUI: відкрити живий перемикач сесій для поточних відкритих TUI‑сесій. Використай `/sessions new` у TUI, щоб одразу запустити іншу живу сесію. |
| `/redraw` | Примусово перерисувати весь UI (виправляє зсуви терміналу після зміни розміру tmux, артефакти виділення мишкою тощо) |
| `/status` | Показати інформацію про сесію — модель, провайдер, профіль, ID сесії, робочий каталог, назва, час створення/оновлення, загальна кількість токенів, стан запущеного агента — а потім локальний блок **Session recap** (кількість останніх ходів користувача/асистента, кількість результатів інструментів, топ‑інструменти, останні змінені файли, останній запит користувача та остання відповідь асистента). Підсумок обчислюється локально з пам’яті розмови; жодних викликів LLM, жодного впливу на кеш промптів. |
| `/agents` (alias: `/tasks`) | Показати активних агентів та запущені завдання у поточній сесії. |
| `/background <prompt>` (alias: `/bg`, `/btw`) | Запустити запит у окремій фоновій сесії. Агент обробляє твій запит незалежно — твоя поточна сесія залишається вільною для іншої роботи. Результати з’являються у панелі після завершення завдання. Дивись [CLI Background Sessions](/user-guide/cli#background-sessions). |
| `/branch [name]` (alias: `/fork`) | Відгалужити поточну сесію (дослідити інший шлях) |
| `/handoff <platform>` | **Лише CLI.** Передати поточну сесію на платформу обміну повідомленнями (Telegram, Discord, Slack, WhatsApp, Signal, Matrix). Шлюз підхоплює її одразу, створює нову гілку на платформах, що підтримують гілки (теми Telegram, гілки текстових каналів Discord, гілки повідомлень Slack), прив’язує ціль до твого CLI `session_id`, щоб відтворити повний ролевий транскрипт, і створює синтетичний хід користувача, щоб агент підтвердив роботу в новому місці. CLI коректно завершується з підказкою `/resume`; локально можна відновити будь‑коли за допомогою `/resume <title>`. Відхилено посеред ходу. Потрібно, щоб шлюз був запущений і був налаштований домашній канал для цільової платформи (`/sethome` у чаті призначення). Дивись [Cross-Platform Handoff](/user-guide/sessions#cross-platform-handoff). |

| `/skin` | Показати або змінити шкіру/тему відображення |
| `/statusbar` (alias: `/sb`) | Перемикати панель стану контексту/моделі вкл./викл. |
| `/voice [on\|off\|tts\|status]` | Перемикати голосовий режим CLI та відтворення мови. Запис використовує `voice.record_key` (за замовчуванням: `Ctrl+B`). |
| `/yolo` | Перемикати YOLO режим — пропускати всі запити підтвердження небезпечних команд. |
| `/footer [on\|off\|status]` | Перемикати нижній колонтитул метаданих шлюзу у фінальних відповідях (показує модель, кількість інструментів, час). |
| `/busy [queue\|steer\|interrupt\|status]` | Тільки CLI: керувати тим, що робить натискання Enter, коли Hermes працює — поставити нове повідомлення в чергу, спрямувати під час ходу або перервати одразу. |

### Швидкі команди

Команди, визначені користувачем, прив’язують коротку команду зі слешем до команди оболонки або іншої команди зі слешем. Налаштуй їх у `~/.hermes/config.yaml`:

```yaml
quick_commands:
  status:
    type: exec
    command: systemctl status hermes-agent
  deploy:
    type: exec
    command: scripts/deploy.sh
  inbox:
    type: alias
    target: /gmail unread
```

Потім вводь `/status`, `/deploy` або `/inbox` у CLI або на платформі обміну повідомленнями. Швидкі команди розв’язуються під час відправки і можуть не з’являтися у кожній вбудованій таблиці автодоповнення/допомоги.

Швидкі команди не підтримують скорочення підказок, що складаються лише з рядка. Розмісти довші багаторазові підказки у skill або використай `type: alias`, щоб послатися на існуючу команду зі слешем.
### Користувацькі аліаси моделей

Визначай свої власні короткі назви для моделей, якими часто користуєшся, а потім викликай їх за допомогою `/model <alias>` у CLI або будь‑якій платформі обміну повідомленнями. Аліаси працюють однаково в обох випадках, у режимі лише для сесії (за замовчуванням) та з перемикачем `--global`.

Підтримуються два формати конфігурації:

**Повна форма** — вкажи точну модель, провайдера та, за потреби, базовий URL. Додай це у `~/.hermes/config.yaml`:

```yaml
model_aliases:
  fav:
    model: claude-sonnet-4.6
    provider: anthropic
  grok:
    model: grok-4
    provider: x-ai
  ollama-qwen:
    model: qwen3-coder:30b
    provider: custom
    base_url: http://localhost:11434/v1
```

**Коротка форма** — `provider/model` в одному рядку. Встановлюй з оболонки без редагування YAML:

```bash
hermes config set model.aliases.fav anthropic/claude-opus-4.6
hermes config set model.aliases.grok x-ai/grok-4
```

Потім у чаті:

```
/model fav            # session-only
/model grok --global  # also persists current-model change to config.yaml
```

Користувацькі аліаси мають пріоритет над вбудованими короткими назвами, тому створення аліасу `sonnet`, `kimi`, `opus` тощо перекриє вбудований. Назви аліасів не залежать від регістру.
### Розв’язання аліасів

Команди підтримують підстановку префікса: введення `/h` розв’язується в `/help`, `/mod` — в `/model`. Якщо префікс неоднозначний (відповідає кільком командам), перевагу отримує перший збіг у порядку реєстру. Повні назви команд і зареєстровані аліаси завжди мають пріоритет над підстановкою префікса.
## Команди слеш‑повідомлень

Шлюз обміну повідомленнями підтримує такі вбудовані команди в чатах Telegram, Discord, Slack, WhatsApp, Signal, Email, Home Assistant та Teams:

| Команда | Опис |
|---------|------|
| `/start` | Команда протоколу платформи. Багато чат‑платформ (Telegram, Discord, …) надсилають `/start` автоматично під час першого відкриття розмови з ботом. Hermes підтверджує ping безшумно — без відповіді агента, без споживання сесії — тому початкові рукостискання не витрачають хід. Ти також можеш надіслати її явно, щоб переконатися, що шлюз доступний. |
| `/new` | Розпочати нову розмову. |
| `/reset` | Скинути історію розмови. |
| `/status` | Показати інформацію про сесію, після чого слідує локальний блок **Session recap** (кількість недавніх ходів, топ‑інструменти, файли, останній запит + відповідь). |
| `/stop` | Припинити всі запущені фонові процеси та перервати працюючого агента. |
| `/model [provider:model]` | Показати або змінити модель. Підтримує перемикання провайдерів (`/model zai:glm-5`), кастомні кінцеві точки (`/model custom:model`), іменовані кастомні провайдери (`/model custom:local:qwen`), авто‑виявлення (`/model custom`) та користувацькі псевдоніми (`/model fav`, `/model grok` — див. [Custom model aliases](#custom-model-aliases)). Використай `--global`, щоб зберегти зміну в `config.yaml`. **Примітка:** `/model` може перемикати лише між вже налаштованими провайдерами. Щоб додати новий провайдер або задати API‑ключі, використай `hermes model` у терміналі (поза чат‑сесією). |
| `/codex-runtime [auto\|codex_app_server\|on\|off]` | Перемкнути необов’язковий [Codex app‑server runtime](../user-guide/features/codex-app-server-runtime). Зберігається в `model.openai_runtime` у `config.yaml` і очищає кешованого агента, тож наступне повідомлення підхопить новий runtime. Діє з наступної сесії. |
| `/personality [name]` | Встановити накладку персональності для сесії. |
| `/fast [normal\|fast\|status]` | Перемкнути швидкий режим — OpenAI Priority Processing / Anthropic Fast Mode. |
| `/retry` | Повторити останнє повідомлення. |
| `/undo` | Видалити останній обмін. |
| `/sethome` (alias: `/set-home`) | Позначити поточний чат як домашній канал платформи для доставок. |
| `/compress [focus topic]` | Ручне стискання контексту розмови. Додаткова тема фокусування звужує, що саме зберігається у підсумку. |
| `/topic [off\|help\|session-id]` | **Лише Telegram DM.** Керування режимом теми для багатосесійних DM, яким керує користувач. `/topic` вмикає його або показує статус; `/topic off` вимикає і очищає прив’язки; `/topic help` показує інструкції; `/topic <session-id>` всередині теми відновлює попередню сесію. Див. [Multi‑session DM mode](/user-guide/messaging/telegram#multi-session-dm-mode-topic). |
| `/title [name]` | Встановити або показати назву сесії. |
| `/resume [name]` | Відновити раніше названу сесію. |
| `/usage` | Показати використання токенів, оцінку вартості (вхід/вихід), стан вікна контексту, тривалість сесії та — коли доступно від активного провайдера — розділ **Account limits** з залишковою квотою/кредитами, отриманими в режимі реального часу з API провайдера. |
| `/insights [days]` | Показати аналітику використання. |
| `/reasoning [level\|show\|hide]` | Змінити інтенсивність міркувань або перемкнути їх відображення. |
| `/voice [on\|off\|tts\|join\|channel\|leave\|status]` | Керувати голосовими відповідями в чаті. `join`/`channel`/`leave` керують режимом голосових каналів Discord. |
| `/rollback [number]` | Перелічити або відновити контрольні точки файлової системи. |
| `/background <prompt>` | Запустити запит у окремій фоновой сесії. Результати повертаються в той самий чат після завершення завдання. Див. [Messaging Background Sessions](/user-guide/messaging/#background-sessions). |
| `/queue <prompt>` (alias: `/q`) | Додати запит у чергу на наступний хід без переривання поточного. |
| `/steer <prompt>` | Вставити повідомлення після наступного виклику інструменту без переривання — модель підхопить його на наступній ітерації, а не як новий хід. |

Модель‑суддя перевіряє після кожного ходу; якщо мета не досягнута, Hermes автоматично продовжує, доки не буде досягнуто, ти не зупиниш/не очистиш її, або не вичерпаєш бюджет ходів (за замовчуванням 20). Підкоманди: `/goal status`, `/goal pause`, `/goal resume`, `/goal clear`. Безпечно запускати посеред роботи агента для статусу/паузи/очищення; встановлення нової мети вимагає спочатку `/stop`. Див. | `/deny` | Відхилити очікуючу небезпечну команду. |
| `/update` | Оновити Hermes Agent до останньої версії. |
| `/restart` | Коректно перезапустити шлюз після завершення активних запусків. Коли шлюз знову підключиться, він надсилає підтвердження в чат/тему запитувача. |

| `/<skill-name>` | Викликати будь‑яку встановлену навичку за назвою. |
## Примітки

- `/skin`, `/snapshot`, `/gquota`, `/reload`, `/tools`, `/toolsets`, `/browser`, `/config`, `/cron`, `/skills`, `/platforms`, `/paste`, `/image`, `/statusbar`, `/plugins`, `/busy`, `/indicator`, `/redraw`, `/clear`, `/history`, `/save`, `/copy`, `/handoff` і `/quit` — це **CLI‑only** команди.
- `/verbose` **CLI‑only за замовчуванням**, але його можна ввімкнути для платформ обміну повідомленнями, встановивши `display.tool_progress_command: true` у `config.yaml`. При ввімкненні він циклічно змінює режим `display.tool_progress` і зберігає його в конфігурації.
- `/sethome`, `/update`, `/restart`, `/approve`, `/deny`, `/topic` і `/commands` — **messaging‑only** команди.
- `/status`, `/background`, `/queue`, `/steer`, `/voice`, `/reload-mcp`, `/reload-skills`, `/rollback`, `/debug`, `/fast`, `/footer`, `/curator`, `/kanban`, `/sessions` і `/yolo` працюють **у CLI та в шлюзі обміну повідомленнями**.
- `/voice join`, `/voice channel` і `/voice leave` мають сенс лише в Discord.
- У TUI `/sessions` показує активні сесії у поточному процесі TUI. Використовуй `/resume [name]` або `hermes --tui --resume <id-or-title>` для збережених або закритих транскриптів.
## Підтвердження команд, що руйнують

CLI запитує підтвердження перед виконанням слеш‑команд, які викидають незбережений стан сесії. Поточний набір деструктивних команд:

| Команда | Що вона знищує |
|---------|------------------|
| `/clear` | Очищає екран і запускає нову сесію — поточний ідентифікатор сесії та історія в пам’яті зникають. |
| `/new` / `/reset` | Запускає нову сесію (новий ідентифікатор сесії + порожня історія). |
| `/undo` | Видаляє останню взаємодію користувач‑асистент з історії. |
| `/exit --delete` / `/quit --delete` | Виходить **і** назавжди видаляє SQLite‑історію поточної сесії та транскрипти на диску. |

Для кожної з цих команд CLI відкриває модальне вікно з трьома варіантами: **Approve Once** (підтвердити цього разу), **Always Approve** (підтвердити і зберегти `approvals.destructive_slash_confirm: false`, щоб майбутні деструктивні команди виконувалися без запиту), або **Cancel**.

**Inline skip:** додай `now`, `--yes` або `-y`, щоб обійти модальне вікно для одного виклику — напр. `/reset now`, `/new --yes my-session`, `/clear -y`, `/undo -y`. Корисно, коли модальне вікно не відображається правильно у твоєму терміналі (див. [issue #30768](https://github.com/NousResearch/hermes-agent/issues/30768) для native Windows PowerShell) або при скриптуванні CLI.

Встанови `approvals.destructive_slash_confirm: false` у `~/.hermes/config.yaml`, щоб глобально вимкнути запити; встанови назад `true`, щоб знову їх увімкнути. Дивись [Security — Destructive slash command confirmation](../user-guide/security.md#dangerous-command-approval) для контексту.
