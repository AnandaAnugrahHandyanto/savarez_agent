# Среда выполнения инструментов

Инструменты Hermes — это саморегистрирующиеся функции, сгруппированные в наборы инструментов и исполняемые через центральную систему реестра/диспатча.

Основные файлы:

- `tools/registry.py`
- `model_tools.py`
- `toolsets.py`
- `tools/terminal_tool.py`
- `tools/environments/*`

## Модель регистрации инструментов

Каждый модуль инструмента вызывает `registry.register(...)` во время импорта.

`model_tools.py` отвечает за импорт/обнаружение модулей инструментов и построение списка схем, используемых моделью.

### Как работает `registry.register()`

Каждый файл инструмента в `tools/` вызывает `registry.register()` на уровне модуля, объявляя себя. Сигнатура функции:

```python
registry.register(
    name="terminal",               # Unique tool name (used in API schemas)
    toolset="terminal",            # Toolset this tool belongs to
    schema={...},                  # OpenAI function-calling schema (description, parameters)
    handler=handle_terminal,       # The function that executes when the tool is called
    check_fn=check_terminal,       # Optional: returns True/False for availability
    requires_env=["SOME_VAR"],     # Optional: env vars needed (for UI display)
    is_async=False,                # Whether the handler is an async coroutine
    description="Run commands",    # Human-readable description
    emoji="💻",                    # Emoji for spinner/progress display
)
```

Каждый вызов создаёт `ToolEntry`, сохраняемый в словаре‑одиночке `ToolRegistry._tools`, ключом которого является имя инструмента. Если происходит конфликт имён между наборами инструментов, выводится предупреждение, и победителем становится более поздняя регистрация.

### Обнаружение: `discover_builtin_tools()`

При импорте `model_tools.py` вызывается `discover_builtin_tools()` из `tools/registry.py`. Эта функция сканирует каждый файл `tools/*.py`, используя парсинг AST, чтобы найти модули, содержащие вызовы `registry.register()` на верхнем уровне, после чего импортирует их:

```python
# tools/registry.py (simplified)
def discover_builtin_tools(tools_dir=None):
    tools_path = Path(tools_dir) if tools_dir else Path(__file__).parent
    for path in sorted(tools_path.glob("*.py")):
        if path.name in {"__init__.py", "registry.py", "mcp_tool.py"}:
            continue
        if _module_registers_tools(path):  # AST check for top-level registry.register()
            importlib.import_module(f"tools.{path.stem}")
```

Такое автоматическое обнаружение означает, что новые файлы инструментов подхватываются автоматически — не требуется поддерживать ручной список. Проверка AST совпадает только с вызовами `registry.register()` на верхнем уровне (не внутри функций), поэтому вспомогательные модули в `tools/` не импортируются.

Каждый импорт инициирует вызовы `registry.register()` модуля. Ошибки в необязательных инструментах (например, отсутствие `fal_client` для генерации изображений) перехватываются и логируются — они не мешают загрузке остальных инструментов.

После обнаружения основных инструментов также находятся инструменты MCP и плагины:

1. **Инструменты MCP** — `tools.mcp_tool.discover_mcp_tools()` читает конфигурацию сервера MCP и регистрирует инструменты с внешних серверов.
2. **Плагин‑инструменты** — `hermes_cli.plugins.discover_plugins()` загружает пользовательские/проектные/pip‑плагины, которые могут регистрировать дополнительные инструменты.

## Проверка доступности инструмента (`check_fn`)

Каждый инструмент может опционально предоставить `check_fn` — вызываемую функцию, возвращающую `True`, если инструмент доступен, и `False` в противном случае. Типичные проверки включают:

- **Наличие API‑ключа** — например, `lambda: bool(os.environ.get("SERP_API_KEY"))` для веб‑поиска
- **Запущенный сервис** — например, проверка конфигурации сервера Honcho
- **Установленный бинарник** — например, проверка наличия `playwright` для браузерных инструментов

Когда `registry.get_definitions()` формирует список схем для модели, она вызывает `check_fn()` каждого инструмента:

```python
# Simplified from registry.py
if entry.check_fn:
    try:
        available = bool(entry.check_fn())
    except Exception:
        available = False   # Exceptions = unavailable
    if not available:
        continue            # Skip this tool entirely
```

Ключевые поведения:
- Результаты проверки **кешируются в пределах одного вызова** — если несколько инструментов используют одну и ту же `check_fn`, она выполняется лишь один раз.
- Исключения в `check_fn()` рассматриваются как «недоступно» (fail‑safe).
- Метод `is_toolset_available()` проверяет, прошла ли проверка `check_fn` наборов инструментов, и используется для отображения в UI и разрешения наборов.

## Разрешение наборов инструментов

Наборы инструментов — это именованные группы инструментов. Hermes разрешает их через:

- явные списки включённых/исключённых наборов
- предустановки платформ (`hermes-cli`, `hermes-telegram` и др.)
- динамические наборы MCP
- специально курируемые наборы, такие как `hermes-acp`

### Как `get_tool_definitions()` фильтрует инструменты

Главная точка входа — `model_tools.get_tool_definitions(enabled_toolsets, disabled_toolsets, quiet_mode)`:

1. **Если передан `enabled_toolsets`** — включаются только инструменты из указанных наборов. Каждое имя набора разрешается через `resolve_toolset()`, который разворачивает составные наборы в отдельные имена инструментов.
2. **Если передан `disabled_toolsets`** — начинается с **ВСЕХ** наборов, после чего вычитаются отключённые.
3. **Если ни то, ни другое** — включаются все известные наборы.
4. **Фильтрация реестра** — полученный набор имён передаётся в `registry.get_definitions()`, где применяется фильтрация `check_fn` и возвращаются схемы в формате OpenAI.
5. **Динамическое исправление схем** — после фильтрации схемы `execute_code` и `browser_navigate` корректируются так, чтобы ссылаться только на инструменты, действительно прошедшие фильтрацию (предотвращает галлюцинации модели о недоступных инструментах).

### Устаревшие имена наборов

Старые имена наборов с суффиксом `_tools` (например, `web_tools`, `terminal_tools`) сопоставляются с их современными именами через `_LEGACY_TOOLSET_MAP` для обратной совместимости.

## Диспатч

Во время выполнения инструменты вызываются через центральный реестр, за исключением некоторых инструментов уровня агента (например, память, todo, поиск‑сессий), которые обрабатываются в цикле агента.

### Поток диспатча: вызов модели → обработка

Когда модель возвращает `tool_call`, происходит следующий поток:

```
Model response with tool_call
    ↓
run_agent.py agent loop
    ↓
model_tools.handle_function_call(name, args, task_id, user_task)
    ↓
[Agent-loop tools?] → handled directly by agent loop (todo, memory, session_search, delegate_task)
    ↓
[Plugin pre-hook] → invoke_hook("pre_tool_call", ...)
    ↓
registry.dispatch(name, args, **kwargs)
    ↓
Look up ToolEntry by name
    ↓
[Async handler?] → bridge via _run_async()
[Sync handler?]  → call directly
    ↓
Return result string (or JSON error)
    ↓
[Plugin post-hook] → invoke_hook("post_tool_call", ...)
```

### Обёртка ошибок

Выполнение любого инструмента оборачивается обработкой ошибок на двух уровнях:

1. **`registry.dispatch()`** — перехватывает любые исключения из обработчика и возвращает `{"error": "Tool execution failed: ExceptionType: message"}` в виде JSON.
2. **`handle_function_call()`** — оборачивает весь диспатч вторичным `try/except`, возвращая `{"error": "Error executing tool_name: message"}`.

Это гарантирует, что модель всегда получает корректно сформированную JSON‑строку, а не необработанное исключение.

### Инструменты уровня цикла агента

Четыре инструмента перехватываются до диспатча реестра, потому что им нужен состояние агента (TodoStore, MemoryStore и т.д.):

- `todo` — планирование/отслеживание задач
- `memory` — постоянные записи памяти
- `session_search` — поиск по кросс‑сессиям
- `delegate_task` — создание подпроцессов‑агентов

Схемы этих инструментов всё равно регистрируются в реестре (для `get_tool_definitions`), но их обработчики возвращают заглушку‑ошибку, если диспатч всё же попадает к ним напрямую.

### Асинхронный мост

Когда обработчик инструмента асинхронный, `_run_async()` переводит его в синхронный путь диспатча:

- **CLI‑путь (без запущенного цикла событий)** — использует постоянный event loop, чтобы кешировать асинхронные клиенты.
- **Gateway‑путь (цикл уже запущен)** — создаёт одноразовый поток с `asyncio.run()`.
- **Рабочие потоки (параллельные инструменты)** — используют постоянные циклы, хранящиеся в thread‑local storage.

## Поток одобрения DANGEROUS_PATTERNS

Инструмент терминала интегрирует систему одобрения опасных команд, определённую в `tools/approval.py`:

1. **Обнаружение шаблонов** — `DANGEROUS_PATTERNS` — список кортежей `(regex, description)`, покрывающих разрушительные операции:
   - рекурсивные удаления (`rm -rf`)
   - форматирование файловой системы (`mkfs`, `dd`)
   - разрушительные SQL‑операции (`DROP TABLE`, `DELETE FROM` без `WHERE`)
   - перезапись системных конфигураций (`> /etc/`)
   - управление сервисами (`systemctl stop`)
   - удалённый запуск кода (`curl | sh`)
   - форк‑бомбы, убийства процессов и т.п.

2. **Проверка** — перед выполнением любой команды терминала `detect_dangerous_command(command)` сравнивает её со всеми шаблонами.

3. **Запрос одобрения** — если найдено совпадение:
   - **CLI‑режим** — интерактивный запрос предлагает пользователю одобрить, отклонить или разрешить навсегда.
   - **Gateway‑режим** — асинхронный колл‑бек отправляет запрос в мессенджер.
   - **Умное одобрение** — при желании вспомогательная LLM может автоматически одобрять низко‑рисковые команды, совпадающие с шаблоном (например, `rm -rf node_modules/` безопасна, но попадает под «рекурсивное удаление»).

4. **Состояние сессии** — одобрения отслеживаются в рамках сессии. После одобрения «рекурсивного удаления» в данной сессии последующие `rm -rf` не вызывают повторный запрос.

5. **Постоянный список разрешённых** — опция «разрешить навсегда» записывает шаблон в `config.yaml` в поле `command_allowlist`, сохраняющееся между сессиями.

## Среды выполнения терминала/рантайма

Система терминала поддерживает несколько бекендов:

- local
- docker
- ssh
- singularity
- modal
- daytona

Также поддерживает:

- переопределения cwd для отдельной задачи
- управление фоновыми процессами
- PTY‑режим
- колл‑беки одобрения для опасных команд

## Параллелизм

Вызовы инструментов могут выполняться последовательно или параллельно в зависимости от набора инструментов и требований взаимодействия.
