# Разрешение среды выполнения провайдера

Hermes имеет общий разрешатель среды выполнения провайдера, используемый в:

- CLI
- gateway
- cron jobs
- ACP
- auxiliary model calls

Основная реализация:

- `hermes_cli/runtime_provider.py` — разрешение учётных данных, `_resolve_custom_runtime()`
- `hermes_cli/auth.py` — реестр провайдеров, `resolve_provider()`
- `hermes_cli/model_switch.py` — общий конвейер переключения `/model` (CLI + gateway)
- `agent/auxiliary_client.py` — маршрутизация вспомогательных моделей
- `providers/` — ABC + точки входа реестра (`ProviderProfile`, `register_provider`, `get_provider_profile`, `list_providers`)
- `plugins/model-providers/<name>/` — плагины для каждого провайдера (встроенные), которые объявляют `api_mode`, `base_url`, `env_vars`, `fallback_models` и регистрируют себя в реестре при первом обращении. Пользовательские плагины в `$HERMES_HOME/plugins/model-providers/<name>/` переопределяют встроенные плагины с тем же именем.

`get_provider_profile()` в `providers/` возвращает `ProviderProfile` для заданного идентификатора провайдера. `runtime_provider.py` вызывает эту функцию во время разрешения, чтобы получить канонический `base_url`, приоритетный список `env_vars`, `api_mode` и `fallback_models`, не дублируя эти данные в нескольких файлах. Добавление нового плагина в `plugins/model-providers/<your-provider>/` (или `$HERMES_HOME/plugins/model-providers/<your-provider>/`), который вызывает `register_provider()`, достаточно, чтобы `runtime_provider.py` его обнаружил — в самом разрешателе не требуется добавлять ветку.

Если ты пытаешься добавить нового первоклассного инференс‑провайдера, прочитай [Adding Providers](./adding-providers.md) и руководство [Model Provider Plugin guide](./model-provider-plugin.md) рядом с этой страницей.
## Приоритет разрешения провайдера

На высоком уровне разрешение провайдера использует:

1. явный запрос CLI/время выполнения
2. конфигурацию модели/провайдера в `config.yaml`
3. переменные окружения
4. специфичные для провайдера значения по умолчанию или авторазрешение

Этот порядок имеет значение, потому что Hermes рассматривает сохранённый выбор модели/провайдера как источник истины для обычных запусков. Это предотвращает тихое переопределение конечной точки устаревшим экспортом оболочки, выбранным пользователем последним в `hermes model`.

## Вывод разрешения среды выполнения

Разрешатель среды выполнения возвращает данные, такие как:

- `provider`
- `api_mode`
- `base_url`
- `api_key`
- `source`
- метаданные, специфичные для провайдера, такие как информация об истечении срока действия и обновлении
## Почему это важно

Этот резольвер — основная причина, по которой Hermes может использовать общую логику аутентификации/выполнения между:

- `hermes chat`
- обработкой сообщений шлюза
- заданиями cron, запускаемыми в новых сессиях
- сессиями редактора ACP
- вспомогательными задачами модели
## OpenRouter и пользовательские базовые URL, совместимые с OpenAI

Hermes содержит логику, предотвращающую утечку неверного API‑ключа на пользовательский эндпоинт, когда существует несколько ключей провайдеров (например, `OPENROUTER_API_KEY` и `OPENAI_API_KEY`).

API‑ключ каждого провайдера привязан к своему базовому URL:

- `OPENROUTER_API_KEY` отправляется только на эндпоинты `openrouter.ai`
- `OPENAI_API_KEY` используется для пользовательских эндпоинтов и как запасной (вариант)

Hermes также различает:

- реальный пользовательский эндпоинт, выбранный пользователем
- запасный путь OpenRouter, используемый, когда пользовательский эндпоинт не настроен

Это различие особенно важно для:

- локальных серверов моделей
- API, совместимых с OpenAI, но не являющихся OpenRouter
- переключения провайдеров без повторного запуска настройки
- пользовательских эндпоинтов, сохранённых в конфигурации, которые должны продолжать работать, даже если `OPENAI_BASE_URL` не экспортирован в текущем shell
## Путь к нативному Anthropic

Anthropic больше не доступен только «через OpenRouter».

Когда разрешение провайдера выбирает `anthropic`, Hermes использует:

- `api_mode = anthropic_messages`
- нативный Anthropic Messages API
- `agent/anthropic_adapter.py` для перевода

Разрешение учётных данных для нативного Anthropic теперь отдаёт предпочтение обновляемым учётным данным Claude Code перед скопированными токенами окружения, если оба источника присутствуют. На практике это означает:

- Файлы учётных данных Claude Code рассматриваются как предпочтительный источник, когда они содержат обновляемую аутентификацию
- Ручные значения `ANTHROPIC_TOKEN` / `CLAUDE_CODE_OAUTH_TOKEN` по‑прежнему работают как явные переопределения
- Hermes проверяет возможность обновления учётных данных Anthropic перед вызовами нативного Messages API
- Hermes всё ещё один раз повторяет запрос при получении 401 после пересборки клиента Anthropic, как запасной (вариант) путь
## Путь OpenAI Codex

Codex использует отдельный путь Responses API:

- `api_mode = codex_responses`
- выделенная поддержка разрешения учётных данных и хранилища аутентификации
## Вспомогательная маршрутизация моделей

Вспомогательные задачи, такие как:

- vision
- web extraction summarization
- context compression summaries
- skills hub operations
- MCP helper operations
- memory flushes

могут использовать собственный роутинг провайдера/модели, а не основную разговорную модель.

Когда вспомогательная задача настроена с провайдером `main`, Hermes разрешает её через тот же общий путь выполнения, что и обычный чат. На практике это означает:

- пользовательские конечные точки, задаваемые переменными окружения, по‑прежнему работают
- пользовательские конечные точки, сохранённые через `hermes model` / `config.yaml`, также работают
- вспомогательная маршрутизация может отличать реальный сохранённый пользовательский endpoint от запасного (варианта) OpenRouter
## Запасные (вариант) модели

Hermes поддерживает настроенную цепочку запасных (вариант) провайдеров — список записей `(provider, model)`, которые пробуются по порядку, когда у основной модели возникают ошибки. Устаревший одноэлементный словарь `fallback_model` всё ещё принимается для обратной совместимости (и мигрируется при первой записи).

### Как это работает внутри

1. **Хранение**: `AIAgent.__init__` сохраняет словарь `fallback_model` и устанавливает `_fallback_activated = False`.

2. **Точки срабатывания**: `_try_activate_fallback()` вызывается из трёх мест в основном цикле повторных попыток в `run_agent.py`:
   - После максимального количества попыток при неверных ответах API (None choices, отсутствие контента)
   - При ошибках клиента, не подлежащих повтору (HTTP 401, 403, 404)
   - После максимального количества попыток при транзиентных ошибках (HTTP 429, 500, 502, 503)

3. **Поток активации** (`_try_activate_fallback`):
   - Сразу возвращает `False`, если уже активировано или не сконфигурировано
   - Вызывает `resolve_provider_client()` из `auxiliary_client.py` для создания нового клиента с правильной аутентификацией
   - Определяет `api_mode`: `codex_responses` для openai-codex, `anthropic_messages` для anthropic, `chat_completions` для всего остального
   - Меняет «на месте»: `self.model`, `self.provider`, `self.base_url`, `self.api_mode`, `self.client`, `self._client_kwargs`
   - Для запасного (варианта) anthropic: создаёт нативный клиент Anthropic вместо совместимого с OpenAI
   - Переоценивает кэширование подсказок (включено для моделей Claude на OpenRouter)
   - Устанавливает `_fallback_activated = True` — предотвращает повторный запуск
   - Сбрасывает счётчик повторов до 0 и продолжает цикл

4. **Поток конфигурации**:
   - CLI: `cli.py` читает `CLI_CONFIG["fallback_model"]` → передаёт в `AIAgent(fallback_model=…)`
   - Шлюз: `gateway/run.py._load_fallback_model()` читает `config.yaml` → передаёт в `AIAgent`
   - Валидация: оба ключа `provider` и `model` должны быть непустыми, иначе запасной (вариант) отключён

### Что НЕ поддерживает запасные (вариант) модели

- **Делегирование субагентов** (`tools/delegate_tool.py`): субагенты наследуют провайдера родителя, но не конфигурацию запасного (варианта) провайдера
- **Вспомогательные задачи**: используют собственную независимую цепочку автоопределения провайдера (см. раздел «Auxiliary model routing» выше)

Cron‑задачи **поддерживают** запасные (вариант) модели: `run_job()` читает `fallback_providers` (или устаревший `fallback_model`) из `config.yaml` и передаёт его в `AIAgent(fallback_model=…)`, соответствуя шаблону `_load_fallback_model()` шлюза. См. [Cron Internals](./cron-internals.md).

### Покрытие тестами

Поведение запасных (вариант) моделей проверяется в нескольких наборах тестов:

- `tests/run_agent/test_fallback_credential_isolation.py` — изоляция учётных данных между основной и запасной моделью
- `tests/hermes_cli/test_fallback_cmd.py` — команда CLI `/fallback`
- `tests/gateway/test_fallback_eviction.py` — удаление провайдеров‑неудач из шлюза
