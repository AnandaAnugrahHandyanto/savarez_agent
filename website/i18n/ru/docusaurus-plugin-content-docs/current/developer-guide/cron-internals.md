# Внутреннее устройство Cron

Подсистема cron обеспечивает выполнение запланированных задач — от простых однократных задержек до повторяющихся задач по cron‑выражениям с внедрением skill и кроссплатформенной доставкой.

## Ключевые файлы

| Файл | Назначение |
|------|------------|
| `cron/jobs.py` | Модель задачи, хранилище, атомарное чтение/запись в `jobs.json` |
| `cron/scheduler.py` | Цикл планировщика — обнаружение задач, готовых к выполнению, их запуск, отслеживание повторов |
| `tools/cronjob_tools.py` | Регистрация и обработчик инструмента `cronjob`, ориентированного на модель |
| `gateway/run.py` | Интеграция с шлюзом — тик cron в длительно работающем цикле |
| `hermes_cli/cron.py` | CLI‑подкоманды `hermes cron` |

## Модель планирования

Поддерживаются четыре формата расписания:

| Формат | Пример | Поведение |
|--------|--------|-----------|
| **Относительная задержка** | `30m`, `2h`, `1d` | Однократное выполнение, срабатывает после указанной длительности |
| **Интервал** | `every 2h`, `every 30m` | Повторяющееся, срабатывает через регулярные интервалы |
| **Cron‑выражение** | `0 9 * * *` | Стандартный 5‑полевая синтаксис cron (минута, час, день, месяц, день недели) |
| **ISO‑таймстамп** | `2025-01-15T09:00:00` | Однократное выполнение, срабатывает в точное время |

Поверхность, ориентированная на модель, представляет собой один инструмент `cronjob` с действиями: `create`, `list`, `update`, `pause`, `resume`, `run`, `remove`.

## Хранилище задач

Задачи сохраняются в `~/.hermes/cron/jobs.json` с атомарной семантикой записи (запись во временный файл, затем переименование). Каждая запись задачи содержит:

```json
{
  "id": "a1b2c3d4e5f6",
  "name": "Daily briefing",
  "prompt": "Summarize today's AI news and funding rounds",
  "schedule": {
    "kind": "cron",
    "expr": "0 9 * * *",
    "display": "0 9 * * *"
  },
  "skills": ["ai-funding-daily-report"],
  "deliver": "telegram:-1001234567890",
  "repeat": {
    "times": null,
    "completed": 42
  },
  "state": "scheduled",
  "enabled": true,
  "next_run_at": "2025-01-16T09:00:00Z",
  "last_run_at": "2025-01-15T09:00:00Z",
  "last_status": "ok",
  "created_at": "2025-01-01T00:00:00Z",
  "model": null,
  "provider": null,
  "script": null
}
```

### Состояния жизненного цикла задачи

| Состояние | Значение |
|-----------|----------|
| `scheduled` | Активна, будет выполнена в следующее запланированное время |
| `paused` | Приостановлена — не будет выполнена, пока не возобновлена |
| `completed` | Исчерпан счётчик повторов или однократная задача уже выполнена |
| `running` | В данный момент исполняется (промежуточное состояние) |

### Обратная совместимость

У старых задач может быть единственное поле `skill` вместо массива `skills`. Планировщик нормализует это при загрузке — одиночный `skill` преобразуется в `skills: [skill]`.

## Выполнение планировщика

### Цикл тика

Планировщик работает на периодическом тике (по умолчанию: каждые 60 секунд):

```text
tick()
  1. Acquire scheduler lock (prevents overlapping ticks)
  2. Load all jobs from jobs.json
  3. Filter to due jobs (next_run <= now AND state == "scheduled")
  4. For each due job:
     a. Set state to "running"
     b. Create fresh AIAgent session (no conversation history)
     c. Load attached skills in order (injected as user messages)
     d. Run the job prompt through the agent
     e. Deliver the response to the configured target
     f. Update run_count, compute next_run
     g. If repeat count exhausted → state = "completed"
     h. Otherwise → state = "scheduled"
  5. Write updated jobs back to jobs.json
  6. Release scheduler lock
```

### Интеграция с шлюзом

В режиме шлюза планировщик запускается в отдельном фоновом потоке (`_start_cron_ticker` в `gateway/run.py`), который каждые 60 секунд вызывает `scheduler.tick()` параллельно с обработкой сообщений.

В режиме CLI задачи cron срабатывают только при выполнении команд `hermes cron` или во время активных CLI‑сессий.

### Изоляция свежей сессии

Каждая задача cron запускается в полностью новой сессии агента:

- Нет истории диалога из предыдущих запусков
- Нет памяти о прошлых выполнениях cron (если только не сохранено в памяти/файлах)
- Промпт должен быть автономным — задачи cron не могут задавать уточняющие вопросы
- Набор инструментов `cronjob` отключён (защита от рекурсии)

## Задачи с поддержкой skill

Задача cron может привязывать один или несколько skill через поле `skills`. Во время выполнения:

1. Skill загружаются в указанном порядке
2. Содержание каждого `SKILL.md` внедряется как контекст
3. Промпт задачи добавляется как инструкция задачи
4. Агент обрабатывает объединённый контекст skill + промпт

Это позволяет использовать переиспользуемые, протестированные рабочие процессы без вставки полных инструкций в промпты cron. Например:

```
Create a daily funding report → attach "ai-funding-daily-report" skill
```

### Задачи со скриптом

Задачи могут также привязывать Python‑скрипт через поле `script`. Скрипт запускается *до* каждого хода агента, а его stdout внедряется в промпт как контекст. Это позволяет реализовать сбор данных и паттерны обнаружения изменений:

```python
# ~/.hermes/scripts/check_competitors.py
import requests, json
# Fetch competitor release notes, diff against last run
# Print summary to stdout — agent analyzes and reports
```

Таймаут скрипта по умолчанию — 120 секунд. Функция `_get_script_timeout()` определяет лимит через трёхуровневую цепочку:

1. **Переопределение на уровне модуля** — `_SCRIPT_TIMEOUT` (для тестов/монкипатчей). Используется только если отличается от значения по умолчанию.
2. **Переменная окружения** — `HERMES_CRON_SCRIPT_TIMEOUT`
3. **Конфигурация** — `cron.script_timeout_seconds` в `config.yaml` (читается через `load_config()`)
4. **Значение по умолчанию** — 120 секунд

### Восстановление провайдера

`run_job()` передаёт пользователю настроенные запасные (fallback) провайдеры и пул учётных данных в экземпляр `AIAgent`:

- **Запасные провайдеры** — читаются `fallback_providers` (список) или `fallback_model` (устаревший dict) из `config.yaml`, соответствуя шаблону `_load_fallback_model()` шлюза. Передаются как `fallback_model=` в `AIAgent.__init__`, который нормализует оба формата в цепочку запасных провайдеров.
- **Пул учётных данных** — загружается через `load_pool(provider)` из `agent.credential_pool` с использованием разрешённого имени провайдера во время выполнения. Передаётся только если пул содержит учётные данные (`pool.has_credentials()`). Позволяет выполнять ротацию ключей того же провайдера при ошибках 429/лимита запросов.

Это повторяет поведение шлюза — без него агенты cron не смогут восстановиться после ограничения скорости.

## Модель доставки

Результаты задач cron могут доставляться на любую поддерживаемую платформу:

| Цель | Синтаксис | Пример |
|------|-----------|--------|
| Исходный чат | `origin` | Доставить в чат, где была создана задача |
| Локальный файл | `local` | Сохранить в `~/.hermes/cron/output/` |
| Telegram | `telegram` или `telegram:<chat_id>` | `telegram:-1001234567890` |
| Discord | `discord` или `discord:#channel` | `discord:#engineering` |
| Slack | `slack` | Доставить в домашний канал Slack |
| WhatsApp | `whatsapp` | Доставить в домашний чат WhatsApp |
| Signal | `signal` | Доставить в Signal |
| Matrix | `matrix` | Доставить в домашнюю комнату Matrix |
| Mattermost | `mattermost` | Доставить в домашний чат Mattermost |
| Email | `email` | Доставить по электронной почте |
| SMS | `sms` | Доставить по SMS |
| Home Assistant | `homeassistant` | Доставить в разговор HA |
| DingTalk | `dingtalk` | Доставить в DingTalk |
| Feishu | `feishu` | Доставить в Feishu |
| WeCom | `wecom` | Доставить в WeCom |
| Weixin | `weixin` | Доставить в Weixin (WeChat) |
| BlueBubbles | `bluebubbles` | Доставить в iMessage через BlueBubbles |
| QQ Bot | `qqbot` | Доставить в QQ (Tencent) через Official API v2 |

Для тем Telegram используйте формат `telegram:<chat_id>:<thread_id>` (например, `telegram:-1001234567890:17585`).

### Обёртка ответа

По умолчанию (`cron.wrap_response: true`) доставки cron оборачиваются:

- Заголовком, идентифицирующим имя задачи cron и её задачу
- Подвалом, указывающим, что агент не видит доставленное сообщение в диалоге

Префикс `[SILENT]` в ответе cron полностью подавляет доставку — полезно для задач, которым нужно только записать файлы или выполнить побочные эффекты.

### Изоляция сессии

Доставки cron **не** отражаются в истории диалога сессии шлюза. Они существуют только в собственной сессии задачи cron. Это предотвращает нарушения чередования сообщений в целевом чате.

## Защита от рекурсии

В сессиях, запущенных cron, набор инструментов `cronjob` отключён. Это предотвращает:

- Создание новых задач cron из запланированной задачи
- Рекурсивное планирование, которое могло бы взорвать использование токенов
- Случайную модификацию расписания задач изнутри задачи

## Блокировка

Планировщик использует межпроцессную файловую блокировку (`fcntl.flock` на Unix, `msvcrt.locking` на Windows), чтобы предотвратить двойное выполнение одного и того же батча задач в результате перекрывающихся тиков — даже между тикером шлюза в процессе и отдельным вызовом `hermes cron` / ручным вызовом `tick()`. Если блокировку получить не удалось, `tick()` сразу возвращает `0`.

## CLI‑интерфейс

CLI `hermes cron` предоставляет прямое управление задачами:

```bash
hermes cron list                    # Show all jobs
hermes cron create                  # Interactive job creation (alias: add)
hermes cron edit <job_id>           # Edit job configuration
hermes cron pause <job_id>          # Pause a running job
hermes cron resume <job_id>         # Resume a paused job
hermes cron run <job_id>            # Trigger immediate execution
hermes cron remove <job_id>         # Delete a job
```
