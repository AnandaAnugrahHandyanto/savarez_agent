# Добавление провайдеров

Hermes уже может работать с любой совместимой с OpenAI конечной точкой через путь пользовательского провайдера. Добавляй встроенный провайдер только если нужен первоклассный UX для этого сервиса:

- аутентификация, специфичная для провайдера, или обновление токена
- курированный каталог моделей
- записи меню настройки / `hermes model`
- псевдонимы провайдера для синтаксиса `provider:model`
- не‑OpenAI‑совместимая форма API, требующая адаптера

Если провайдер представляет собой просто «другой совместимый с OpenAI базовый URL и API‑ключ», может быть достаточно именованного пользовательского провайдера.
## Ментальная модель

Встроенный провайдер должен согласовываться на нескольких уровнях:

1. `hermes_cli/auth.py` решает, как находить учётные данные.
2. `hermes_cli/runtime_provider.py` преобразует их в данные выполнения:
   - `provider`
   - `api_mode`
   - `base_url`
   - `api_key`
   - `source`
3. `run_agent.py` использует `api_mode`, чтобы решить, как формировать и отправлять запросы.
4. `hermes_cli/models.py` и `hermes_cli/main.py` делают провайдер видимым в CLI. (`hermes_cli/setup.py` автоматически делегирует `main.py` — изменений здесь не требуется.)
5. `agent/auxiliary_client.py` и `agent/model_metadata.py` поддерживают работу вспомогательных задач и бюджетирование токенов.

Важная абстракция — `api_mode`.

- Большинство провайдеров используют `chat_completions`.
- Codex использует `codex_responses`.
- Anthropic использует `anthropic_messages`.
- Новый протокол, не относящийся к OpenAI, обычно подразумевает добавление нового адаптера и новой ветки `api_mode`.
## Выбери путь реализации сначала

### Путь A — провайдер, совместимый с OpenAI

Используй, когда провайдер принимает запросы в стиле стандартных chat‑completions.

Типичные задачи:

- добавить метаданные аутентификации
- добавить каталог моделей / алиасы
- добавить разрешение времени выполнения
- добавить привязку меню CLI
- добавить значения по умолчанию для вспомогательных моделей
- добавить тесты и пользовательскую документацию

Обычно не требуется новый адаптер или новый `api_mode`.

### Путь B — нативный провайдер

Используй, когда провайдер не ведёт себя как chat‑completions OpenAI.

Примеры в репозитории на данный момент:

- `codex_responses`
- `anthropic_messages`

Этот путь включает всё из Пути A, а также:

- адаптер провайдера в `agent/`
- ветки `run_agent.py` для построения запросов, диспетчеризации, извлечения использования, обработки прерываний и нормализации ответов
- тесты адаптера
## Контрольный список файлов

### Требуется для каждого встроенного провайдера

1. `hermes_cli/auth.py`
2. `hermes_cli/models.py`
3. `hermes_cli/runtime_provider.py`
4. `hermes_cli/main.py`
5. `agent/auxiliary_client.py`
6. `agent/model_metadata.py`
7. tests
8. документация для пользователей в `website/docs/`

:::tip
`hermes_cli/setup.py` **не** требует изменений. Мастер установки делегирует выбор провайдера/модели функции `select_provider_and_model()` в `main.py` — любой провайдер, добавленный там, автоматически доступен в `hermes setup`.
:::

### Дополнительно для нативных / не‑OpenAI провайдеров

10. `agent/<provider>_adapter.py`
11. `run_agent.py`
12. `pyproject.toml`, если требуется SDK провайдера
## Fast path: Simple API-key providers

Если твой **provider** — это просто совместимая с OpenAI конечная точка, аутентификация которой происходит с помощью единственного API‑ключа, тебе не нужно трогать `auth.py`, `runtime_provider.py`, `main.py` или любые другие файлы из полного чек‑листа ниже.

Все, что нужно:

1. Каталог плагина в `plugins/model-providers/<your-provider>/`, содержащий:
   - `__init__.py` — вызывает `register_provider(profile)` на уровне модуля
   - `plugin.yaml` — манифест (name, kind: model-provider, version, description)
2. И всё. Плагины‑**provider** автоматически загружаются при первом вызове `get_provider_profile()` или `list_providers()` — как встроенные плагины (этот репозиторий), так и пользовательские плагины в `$HERMES_HOME/plugins/model-providers/` будут обнаружены.

Когда ты добавляешь плагин и он вызывает `register_provider()`, происходит автоматическое подключение следующего:

1. Запись `PROVIDER_REGISTRY` в `auth.py` (разрешение учётных данных, поиск переменных окружения)
2. `api_mode` устанавливается в `chat_completions`
3. `base_url` берётся из конфигурации или объявленной переменной окружения
4. `env_vars` проверяются в порядке приоритета для поиска API‑ключа
5. Список `fallback_models` регистрируется для **provider**
6. Флаг CLI `--provider` принимает идентификатор **provider**
7. Меню `hermes model` включает этот **provider**
8. Мастер `hermes setup` автоматически делегирует `main.py`
9. Синтаксис псевдонима `provider:model` работает
10. Runtime‑разрешатель возвращает правильный `base_url` и `api_key`
11. Флаг CLI `--provider <name>` принимает идентификатор **provider**
12. Активация запасной (вариант) модели может плавно переключиться на **provider**

Пользовательские плагины в `$HERMES_HOME/plugins/model-providers/<name>/` переопределяют встроенные плагины с тем же именем (принцип «последний записавший выигрывает» в `register_provider()`) — так что сторонние разработчики могут monkey‑patch или заменять любой встроенный профиль без изменения репозитория.

См. `plugins/model-providers/nvidia/` или `plugins/model-providers/gmi/` как шаблон, а также полный [Model Provider Plugin guide](/developer-guide/model-provider-plugin) для справки по полям, идиомам хуков и сквозным примерам.
## Полный путь: OAuth и сложные провайдеры

Воспользуйся полным чек‑листом ниже, если твоему провайдеру требуется любое из следующего:

- OAuth или обновление токена (Nous Portal, Codex, Google Gemini, Qwen Portal, Copilot)
- Не‑OpenAI‑API‑формат, требующий нового адаптера (Anthropic Messages, Codex Responses)
- Обнаружение пользовательского конечного пункта или многорегиональное зондирование (z.ai, Kimi)
- Курированный статический каталог моделей или живой запрос `/models`
- Специфичные для провайдера пункты меню `hermes model` с индивидуальными потоками аутентификации
## Шаг 1: Выбери один канонический идентификатор провайдера

Выбери единственный идентификатор провайдера и используй его везде.

Примеры из репозитория:

- `openai-codex`
- `kimi-coding`
- `minimax-cn`

Тот же самый идентификатор должен появляться в:

- `PROVIDER_REGISTRY` в `hermes_cli/auth.py`
- `_PROVIDER_LABELS` в `hermes_cli/models.py`
- `_PROVIDER_ALIASES` как в `hermes_cli/auth.py`, так и в `hermes_cli/models.py`
- параметр CLI `--provider` в `hermes_cli/main.py`
- ветки настройки/выбора модели
- значения по умолчанию auxiliary‑model
- тесты

Если идентификатор различается между этими файлами, провайдер будет полу‑подключён: аутентификация может работать, тогда как `/model`, настройка или разрешение во время выполнения тихо его пропустят.
## Шаг 2: Добавить метаданные аутентификации в `hermes_cli/auth.py`

Для провайдеров, использующих API‑ключ, добавьте запись `ProviderConfig` в `PROVIDER_REGISTRY` со следующими полями:

- `id`
- `name`
- `auth_type="api_key"`
- `inference_base_url`
- `api_key_env_vars`
- необязательный `base_url_env_var`

Также добавьте псевдонимы в `_PROVIDER_ALIASES`.

Используйте существующие провайдеры в качестве шаблонов:

- простой путь API‑ключа: Z.AI, MiniMax
- путь API‑ключа с определением конечной точки: Kimi, Z.AI
- нативное разрешение токена: Anthropic
- путь OAuth / auth‑store: Nous, OpenAI Codex

Вопросы, на которые нужно ответить здесь:

- Какие переменные окружения должен проверять Hermes и в каком порядке приоритета?
- Нужны ли провайдеру переопределения базового URL?
- Требуется ли проверка конечной точки или обновление токена?
- Что должно говорить сообщение об ошибке аутентификации, когда учётные данные отсутствуют?

Если провайдеру требуется что‑то больше, чем просто «поиск API‑ключа», добавьте отдельный резолвер учётных данных вместо того, чтобы встраивать логику в несвязанные ветки.
## Шаг 3: Добавление каталога моделей и алиасов в `hermes_cli/models.py`

Обнови каталог провайдеров, чтобы провайдер корректно отображался в меню и поддерживал синтаксис `provider:model`.

Типичные изменения:

- `_PROVIDER_MODELS`
- `_PROVIDER_LABELS`
- `_PROVIDER_ALIASES`
- порядок отображения провайдеров внутри `list_available_providers()`
- `provider_model_ids()`, если провайдер поддерживает динамический запрос `/models`

Если провайдер предоставляет живой список моделей, отдай ему предпочтение и оставь `_PROVIDER_MODELS` в качестве запасного варианта.

Этот файл также отвечает за работу таких вводов:

```text
anthropic:claude-sonnet-4-6
kimi:model-name
```

Если алиасы отсутствуют здесь, провайдер может успешно аутентифицироваться, но всё равно будет ошибаться при разборе `/model`.
## Шаг 4: Разрешение данных выполнения в `hermes_cli/runtime_provider.py`

`resolve_runtime_provider()` — общий путь, используемый CLI, gateway, cron, ACP и вспомогательными клиентами.

Добавь ветку, которая возвращает словарь как минимум со следующими полями:

```python
{
    "provider": "your-provider",
    "api_mode": "chat_completions",  # or your native mode
    "base_url": "https://...",
    "api_key": "...",
    "source": "env|portal|auth-store|explicit",
    "requested_provider": requested_provider,
}
```

Если провайдер совместим с OpenAI, `api_mode` обычно должен оставаться `chat_completions`.

Будь внимателен к приоритету API‑ключей. Hermes уже содержит логику, предотвращающую утечку ключа OpenRouter к несвязанным конечным точкам. Новый провайдер должен быть столь же явным относительно того, какой ключ используется для какого базового URL.
## Шаг 5: Подключить CLI в `hermes_cli/main.py`

Провайдер не будет обнаружен, пока он не появится в интерактивном потоке `hermes model`.

Обнови следующее в `hermes_cli/main.py`:

- словарь `provider_labels`
- список `providers` в `select_provider_and_model()`
- диспетчер провайдера (`if selected_provider == …`)
- варианты выбора аргумента `--provider`
- варианты входа/выхода, если провайдер поддерживает такие потоки
- функцию `_model_flow_<provider>()` либо переиспользуй `_model_flow_api_key_provider()`, если она подходит

:::tip
`hermes_cli/setup.py` менять не нужно — он вызывает `select_provider_and_model()` из `main.py`, поэтому твой новый провайдер появится автоматически и в `hermes model`, и в `hermes setup`.
:::
## Шаг 6: Сохранить работу вспомогательных вызовов

Два файла имеют значение:

### `agent/auxiliary_client.py`

Добавь недорогую / быструю aux‑модель по умолчанию в `_API_KEY_PROVIDER_AUX_MODELS`, если это прямой провайдер API‑ключа.

Вспомогательные задачи включают:

- суммирование визуального контента
- суммирование извлечения из веба
- сжатие контекста в виде резюме
- резюме поиска по сессии
- сбросы памяти

Если у провайдера нет разумного aux‑значения по умолчанию, побочные задачи могут падать или неожиданно использовать дорогую основную модель.

### `agent/model_metadata.py`

Добавь длины контекста для моделей провайдера, чтобы планирование токенов, пороги сжатия и ограничения оставались разумными.
## Шаг 7: Если провайдер нативный, добавить адаптер и поддержку `run_agent.py`

Если провайдер не использует обычные завершения чата, вынеси специфичную для провайдера логику в `agent/<provider>_adapter.py`.

Сохраняй `run_agent.py` сфокусированным на оркестрации. Он должен вызывать вспомогательные функции адаптера, а не формировать полезную нагрузку провайдера «вручную» по всему файлу.

Нативный провайдер обычно требует работы в следующих местах:

### Новый файл адаптера

Типичные обязанности:

- создать SDK / HTTP‑клиент
- разрешить токены
- преобразовать сообщения в стиле OpenAI в формат запроса провайдера
- при необходимости преобразовать схемы инструментов
- нормализовать ответы провайдера обратно в то, что ожидает `run_agent.py`
- извлечь данные об использовании и причине завершения

### `run_agent.py`

Найди `api_mode` и проверь каждый пункт переключения. Как минимум, убедись в следующем:

- `__init__` выбирает новый `api_mode`
- построение клиента работает для провайдера
- `_build_api_kwargs()` умеет формировать запросы
- `_interruptible_api_call()` направляет вызов к правильному клиенту
- пути прерывания / восстановления клиента работают
- проверка ответа принимает форму ответа провайдера
- извлечение причины завершения корректно
- извлечение использования токенов корректно
- активация запасного (варианта) модели может плавно переключиться на нового провайдера
- пути генерации сводки и сброса памяти продолжают работать

Также ищи в `run_agent.py` `self.client.`. Любой путь кода, который предполагает наличие стандартного клиента OpenAI, может сломаться, когда нативный провайдер использует другой объект клиента или `self.client = None`.

### Кеширование подсказок и специфичные для провайдера поля запросов

Кеширование подсказок и специфичные для провайдера поля запросов легко регрессировать.

Примеры, уже присутствующие в репозитории:

- у Anthropic есть нативный путь кеширования подсказок
- у OpenRouter есть поля маршрутизации провайдера
- не каждый провайдер должен получать каждую опцию со стороны запроса

Когда добавляешь нативного провайдера, дважды проверь, что Hermes отправляет только те поля, которые провайдер действительно понимает.
## Шаг 8: Тесты

По крайней мере, затронь тесты, которые проверяют подключение провайдеров.

Типичные места:

- `tests/hermes_cli/test_runtime_provider_resolution.py`
- `tests/cli/test_cli_provider_resolution.py`
- `tests/hermes_cli/test_model_switch_custom_providers.py` (и соседние `tests/hermes_cli/test_model_switch_*.py`)
- `tests/hermes_cli/test_setup_model_provider.py`
- `tests/run_agent/test_provider_parity.py`
- `tests/run_agent/test_run_agent.py`
- `tests/test_<provider>_adapter.py` для нативного провайдера

Для примеров, предназначенных только для документации, набор файлов может отличаться. Главное — покрыть:

- разрешение аутентификации
- меню CLI / выбор провайдера
- разрешение провайдера в рантайме
- путь выполнения агента
- разбор `provider:model`
- любые специфичные для адаптера преобразования сообщений

Запусти тесты с отключённым xdist:

```bash
source venv/bin/activate
python -m pytest tests/hermes_cli/test_runtime_provider_resolution.py tests/cli/test_cli_provider_resolution.py tests/hermes_cli/test_setup_model_provider.py tests/run_agent/test_provider_parity.py -n0 -q
```

Для более глубоких изменений запусти полный набор перед отправкой:

```bash
source venv/bin/activate
python -m pytest tests/ -n0 -q
```
## Шаг 9: Живая проверка

После тестов запусти настоящий smoke‑test.

```bash
source venv/bin/activate
python -m hermes_cli.main chat -q "Say hello" --provider your-provider --model your-model
```

Также протестируй интерактивные сценарии, если ты менял меню:

```bash
source venv/bin/activate
python -m hermes_cli.main model
python -m hermes_cli.main setup
```

Для нативных провайдеров проверь хотя бы один вызов инструмента, а не только текстовый ответ.
## Шаг 10: Обновление пользовательской документации

Если провайдер предназначен для выпуска в качестве полноценной опции, обнови также пользовательскую документацию:

- `website/docs/getting-started/quickstart.md`
- `website/docs/user-guide/configuration.md`
- `website/docs/reference/environment-variables.md`

Разработчик может безупречно подключить провайдер, но при этом пользователи всё равно не смогут обнаружить необходимые переменные окружения или порядок настройки.

## Общие подводные камни

### 1. Добавление провайдера в auth, но не в разбор модели

Это приводит к корректному разрешению учётных данных, тогда как вводы `/model` и `provider:model` завершаются ошибкой.

### 2. Забвение, что `config["model"]` может быть строкой или словарём

Большая часть кода выбора провайдера должна нормализовать обе формы.

### 3. Предположение, что нужен встроенный провайдер

Если сервис совместим с OpenAI, пользовательский провайдер может уже решить задачу с меньшими затратами на обслуживание.

### 4. Забвение вспомогательных путей

Основной путь чата может работать, тогда как суммирование, сброс памяти или вспомогательные функции зрения падают, потому что aux‑маршрутизация никогда не обновлялась.

### 5. Ветви нативного провайдера, скрытые в `run_agent.py`

Ищи `api_mode` и `self.client.`. Не предполагай, что очевидный путь запроса — единственный.

### 6. Отправка параметров, предназначенных только для OpenRouter, другим провайдерам

Поля, такие как маршрутизация провайдера, относятся только к тем провайдерам, которые их поддерживают.

### 7. Обновление `hermes model`, но не `hermes setup`

Оба потока должны знать о провайдере.
## Хорошие цели поиска при реализации

Если ты ищешь все места, где **provider** взаимодействует, ищи следующие символы:

- `PROVIDER_REGISTRY`
- `_PROVIDER_ALIASES`
- `_PROVIDER_MODELS`
- `resolve_runtime_provider`
- `_model_flow_`
- `select_provider_and_model`
- `api_mode`
- `_API_KEY_PROVIDER_AUX_MODELS`
- `self.client.`
