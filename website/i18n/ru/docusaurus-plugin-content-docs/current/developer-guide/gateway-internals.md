# Внутреннее устройство шлюза

Шлюз обмена сообщениями — это долгоживущий процесс, который соединяет Hermes с более чем 20 внешними платформами обмена сообщениями через единый архитектурный подход.
## Ключевые файлы

| Файл | Назначение |
|------|------------|
| `gateway/run.py` | `GatewayRunner` — главный цикл, слеш‑команды, диспетчеризация сообщений (большой файл; см. git для текущего количества строк) |
| `gateway/session.py` | `SessionStore` — постоянное хранение диалогов и построение ключа сессии |
| `gateway/delivery.py` | Доставка исходящих сообщений на целевые платформы/каналы |
| `gateway/pairing.py` | Поток сопряжения DM для авторизации пользователя |
| `gateway/channel_directory.py` | Сопоставление ID чатов с читаемыми именами для доставки через cron |
| `gateway/hooks.py` | Обнаружение хуков, их загрузка и диспетчеризация событий жизненного цикла |
| `gateway/mirror.py` | Кросс‑сессионное зеркалирование сообщений для `send_message` |
| `gateway/status.py` | Управление блокировкой токенов для экземпляров шлюза, ограниченных профилем |
| `gateway/builtin_hooks/` | Точка расширения для всегда‑зарегистрированных хуков (не поставляются) |
| `gateway/platforms/` | Адаптеры платформ (по одной на каждую платформу обмена сообщениями) |

## Поток сообщений

Когда сообщение поступает с любой платформы:

1. **Platform adapter** получает необработанное событие, нормализует его в `MessageEvent`.
2. **Base adapter** проверяет активную защиту сессии:
   - Если агент запущен для этой сессии → ставит сообщение в очередь, устанавливает событие прерывания.
   - Если `/approve`, `/deny`, `/stop` → обход защиты (выполняется inline).
3. **GatewayRunner._handle_message()** получает событие:
   - Определяет ключ сессии через `_session_key_for_source()` (формат: `agent:main:{platform}:{chat_type}:{chat_id}`).
   - Проверяет авторизацию (см. раздел Authorization ниже).
   - Проверяет, является ли сообщение slash‑командой → передаёт обработчику команд.
   - Проверяет, запущен ли уже агент → перехватывает команды вроде `/stop`, `/status`.
   - Иначе → создаёт экземпляр `AIAgent` и запускает диалог.
4. **Response** отправляется обратно через адаптер платформы.

### Формат ключа сессии

Ключи сессий кодируют полный контекст маршрутизации:

```
agent:main:{platform}:{chat_type}:{chat_id}
```

Например: `agent:main:telegram:private:123456789`.

Платформы, поддерживающие ветки (темы форумов Telegram, ветки Discord, ветки Slack), могут включать идентификаторы веток в часть `chat_id`. **Никогда не создавай ключи сессий вручную** — всегда используй `build_session_key()` из `gateway/session.py`.

### Двухуровневая защита сообщений

Когда агент активно работает, входящие сообщения проходят через два последовательных уровня защиты:

1. **Уровень 1 — Base adapter** (`gateway/platforms/base.py`): проверяет `_active_sessions`. Если сессия активна, ставит сообщение в очередь `_pending_messages` и устанавливает событие прерывания. Это перехватывает сообщения *до* того, как они попадут в gateway runner.

2. **Уровень 2 — Gateway runner** (`gateway/run.py`): проверяет `_running_agents`. Перехватывает специфические команды (`/stop`, `/new`, `/queue`, `/status`, `/approve`, `/deny`) и направляет их соответствующим образом. Всё остальное вызывает `running_agent.interrupt()`.

Команды, которые должны достичь runner, пока агент заблокирован (например, `/approve`), отправляются **inline** через `await self._message_handler(event)` — они обходят систему фоновых задач, чтобы избежать гонок.
## Авторизация

Шлюз использует многоуровневую проверку авторизации, выполняемую последовательно:

1. **Флаг разрешения всем пользователям на платформе** (например, `TELEGRAM_ALLOW_ALL_USERS`) — если установлен, все пользователи на этой платформе авторизованы
2. **Белый список платформы** (например, `TELEGRAM_ALLOWED_USERS`) — идентификаторы пользователей, разделённые запятыми
3. **DM‑сопряжение** — аутентифицированные пользователи могут сопрягать новых пользователей с помощью кода сопряжения
4. **Глобальный флаг разрешения всем** (`GATEWAY_ALLOW_ALL_USERS`) — если установлен, все пользователи на всех платформах авторизованы
5. **По умолчанию: отказ** — неавторизованные пользователи отклоняются

### Поток DM‑сопряжения

```text
Admin: /pair
Gateway: "Pairing code: ABC123. Share with the user."
New user: ABC123
Gateway: "Paired! You're now authorized."
```

Состояние сопряжения сохраняется в `gateway/pairing.py` и сохраняется при перезапусках.
## Диспетчеризация слеш‑команд

Все слеш‑команды в шлюзе проходят через один и тот же конвейер разрешения:

1. `resolve_command()` из `hermes_cli/commands.py` сопоставляет ввод с каноническим именем (обрабатывает псевдонимы, совпадения по префиксу)
2. Каноническое имя проверяется против `GATEWAY_KNOWN_COMMANDS`
3. Обработчик в `_handle_message()` диспатчит выполнение по каноническому имени
4. Некоторые команды ограничены конфигурацией (`gateway_config_gate` на `CommandDef`)

### Защита от выполнения команд во время работы агента

Команды, которые **не должны** выполняться, пока агент обрабатывает запрос, отклоняются сразу:

```python
if _quick_key in self._running_agents:
    if canonical == "model":
        return "⏳ Agent is running — wait for it to finish or /stop first."
```

Команды‑обходы (`/stop`, `/new`, `/approve`, `/deny`, `/queue`, `/status`) имеют специальную обработку.
## Источники конфигурации

Шлюз читает конфигурацию из нескольких источников:

| Источник | Что предоставляет |
|--------|-----------------|
| `~/.hermes/.env` | API‑ключи, токены ботов, учётные данные платформ |
| `~/.hermes/config.yaml` | Настройки модели, конфигурация инструментов, параметры отображения |
| Переменные окружения | Переопределяют любые из вышеуказанных |

В отличие от CLI (которая использует `load_cli_config()` с жёстко заданными значениями по умолчанию), шлюз читает `config.yaml` напрямую через загрузчик YAML. Это означает, что ключи конфигурации, присутствующие в словаре значений по умолчанию CLI, но отсутствующие в файле конфигурации пользователя, могут вести себя по‑разному в CLI и шлюзе.
## Адаптеры платформ

Каждая мессенджер‑платформа имеет адаптер в `gateway/platforms/`:

```text
gateway/platforms/
├── base.py              # BaseAdapter — shared logic for all platforms
├── telegram.py          # Telegram Bot API (long polling or webhook)
├── discord.py           # Discord bot via discord.py
├── slack.py             # Slack Socket Mode
├── whatsapp.py          # WhatsApp Business Cloud API
├── signal.py            # Signal via signal-cli REST API
├── matrix.py            # Matrix via mautrix (optional E2EE)
├── mattermost.py        # Mattermost WebSocket API
├── email.py             # Email via IMAP/SMTP
├── sms.py               # SMS via Twilio
├── dingtalk.py          # DingTalk WebSocket
├── feishu.py            # Feishu/Lark WebSocket or webhook
├── wecom.py             # WeCom (WeChat Work) callback
├── weixin.py            # Weixin (personal WeChat) via iLink Bot API
├── bluebubbles.py       # Apple iMessage via BlueBubbles macOS server
├── qqbot/               # QQ Bot (Tencent QQ) via Official API v2 (sub-package: adapter.py, crypto.py, keyboards.py, …)
├── yuanbao.py           # Yuanbao (Tencent) DM/group adapter
├── feishu_comment.py    # Feishu document/drive comment-reply handler
├── msgraph_webhook.py   # Microsoft Graph change-notification webhook (Teams, Outlook, etc.)
├── webhook.py           # Inbound/outbound webhook adapter
├── api_server.py        # REST API server adapter
└── homeassistant.py     # Home Assistant conversation integration
```

Адаптеры реализуют общий интерфейс:
- `connect()` / `disconnect()` — управление жизненным циклом
- `send_message()` — отправка исходящих сообщений
- `on_message()` — нормализация входящих сообщений → `MessageEvent`

### Блокировки токенов

Адаптеры, которые подключаются с уникальными учётными данными, вызывают `acquire_scoped_lock()` в `connect()` и `release_scoped_lock()` в `disconnect()`. Это предотвращает одновременное использование одного и того же токена бота двумя профилями.
## Путь доставки

Исходящие доставки (`gateway/delivery.py`) обрабатывают:

- **Прямой ответ** — отправить ответ обратно в исходный чат
- **Доставка в домашний канал** — направлять выводы cron‑задач и результаты фоновых процессов в настроенный домашний канал
- **Явная доставка к цели** — инструмент `send_message`, указывающий `telegram:-1001234567890`, или CLI [`hermes send`](/guides/pipe-script-output), оборачивающий тот же инструмент для скриптов оболочки
- **Кроссплатформенная доставка** — доставлять в другую платформу, отличную от той, где было получено сообщение

Доставки cron‑задач НЕ зеркалируются в историю сессии шлюза — они существуют только в своей отдельной cron‑сессии. Это осознанный выбор дизайна, позволяющий избежать нарушений чередования сообщений.
## Hooks

Gateway hooks — это Python‑модули, реагирующие на события жизненного цикла:

### События hook‑ов шлюза

| Event | Когда срабатывает |
|-------|--------------------|
| `gateway:startup` | Запуск процесса шлюза |
| `session:start` | Начало новой сессии разговора |
| `session:end` | Завершение сессии или её истечение |
| `session:reset` | Пользователь сбрасывает сессию командой `/new` |
| `agent:start` | Агент начинает обработку сообщения |
| `agent:step` | Агент завершает одну итерацию вызова инструмента |
| `agent:end` | Агент заканчивает работу и возвращает ответ |
| `command:*` | Выполняется любая слеш‑команда |

Hook‑ы обнаруживаются в `gateway/builtin_hooks/` (точка расширения — в поставляемой версии пустая; `_register_builtin_hooks()` — заглушка) и `~/.hermes/hooks/` (установленные пользователем). Каждый hook представляет собой каталог с манифестом `HOOK.yaml` и файлом `handler.py`.
## Интеграция провайдера памяти

Когда включён плагин провайдера памяти (например, Honcho):

1. Gateway создаёт `AIAgent` для каждого сообщения с идентификатором сессии
2. `MemoryManager` инициализирует провайдера с контекстом сессии
3. Инструменты провайдера (например, `honcho_profile`, `viking_search`) маршрутизируются через:

```text
AIAgent._invoke_tool()
  → self._memory_manager.handle_tool_call(name, args)
    → provider.handle_tool_call(name, args)
```

4. При завершении или сбросе сессии вызывается `on_session_end()` для очистки и окончательной выгрузки данных

### Жизненный цикл очистки памяти

Когда сессия сбрасывается, возобновляется или истекает:
1. Встроенные памяти сбрасываются на диск
2. Хук провайдера памяти `on_session_end()` вызывается
3. Временный `AIAgent` выполняет ход разговора, использующий только память
4. Затем контекст удаляется или архивируется
## Обслуживание в фоновом режиме

- **Cron ticking** — проверяет расписание задач и запускает ожидающие задачи
- **Session expiry** — удаляет заброшенные сессии после истечения тайм‑аута
- **Memory flush** — проактивно очищает память перед истечением сессии
- **Cache refresh** — обновляет списки моделей и статус провайдеров
## Управление процессами

Шлюз работает как долгоживущий процесс, управляемый через:

- `hermes gateway start` / `hermes gateway stop` — ручное управление
- `systemctl` (Linux) или `launchctl` (macOS) — управление сервисом
- PID‑файл в `~/.hermes/gateway.pid` — отслеживание процесса в рамках профиля

**Профильный vs глобальный**: `start_gateway()` использует PID‑файлы, привязанные к профилю. `hermes gateway stop` останавливает только шлюз текущего профиля. `hermes gateway stop --all` использует глобальное сканирование `ps aux` для завершения всех процессов шлюза (используется при обновлениях).
