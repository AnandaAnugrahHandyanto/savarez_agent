# Внутреннее устройство цикла агента

Ядром оркестрации является класс `AIAgent` из `run_agent.py` — большой файл (~4 400 строк), который обрабатывает всё: от сборки подсказки до вызова инструментов и переключения провайдеров.

## Основные обязанности

`AIAgent` отвечает за:

- Сборку эффективной системной подсказки и схем инструментов через `prompt_builder.py`
- Выбор правильного провайдера/режима API (`chat_completions`, `codex_responses`, `anthropic_messages`)
- Выполнение прерываемых вызовов модели с поддержкой отмены
- Выполнение вызовов инструментов (последовательно или параллельно через пул потоков)
- Поддержание истории диалога в формате сообщений OpenAI
- Обработку сжатия, повторных попыток и переключения на запасной (вариант) модель
- Отслеживание бюджета итераций для родительских и дочерних агентов
- Сброс постоянной памяти перед потерей контекста

## Две точки входа

```python
# Simple interface — returns final response string
response = agent.chat("Fix the bug in main.py")

# Full interface — returns dict with messages, metadata, usage stats
result = agent.run_conversation(
    user_message="Fix the bug in main.py",
    system_message=None,           # auto-built if omitted
    conversation_history=None,      # auto-loaded from session if omitted
    task_id="task_abc123"
)
```

`chat()` — тонкая оболочка вокруг `run_conversation()`, которая извлекает поле `final_response` из словаря результата.

## Режимы API

Hermes поддерживает три режима выполнения API, определяемые выбором провайдера, явными аргументами и эвристикой базового URL:

| Режим API | Используется для | Тип клиента |
|----------|------------------|-------------|
| `chat_completions` | Совместимые с OpenAI эндпоинты (OpenRouter, кастомные, большинство провайдеров) | `openai.OpenAI` |
| `codex_responses` | OpenAI Codex / Responses API | `openai.OpenAI` с форматом Responses |
| `anthropic_messages` | Нативный Anthropic Messages API | `anthropic.Anthropic` через адаптер |

Режим определяет, как форматируются сообщения, как структурируются вызовы инструментов, как парсятся ответы и как работает кэширование/стриминг. Все три режима сходятся к единому внутреннему формату сообщений (словарям в стиле OpenAI `role`/`content`/`tool_calls`) до и после вызовов API.

**Порядок разрешения режима:**
1. Явный аргумент конструктора `api_mode` (наивысший приоритет)
2. Специфическое определение провайдера (например, провайдер `anthropic` → `anthropic_messages`)
3. Эвристика базового URL (например, `api.anthropic.com` → `anthropic_messages`)
4. По умолчанию — `chat_completions`

## Жизненный цикл хода

Каждая итерация цикла агента следует этой последовательности:

```text
run_conversation()
  1. Generate task_id if not provided
  2. Append user message to conversation history
  3. Build or reuse cached system prompt (prompt_builder.py)
  4. Check if preflight compression is needed (>50% context)
  5. Build API messages from conversation history
     - chat_completions: OpenAI format as-is
     - codex_responses: convert to Responses API input items
     - anthropic_messages: convert via anthropic_adapter.py
  6. Inject ephemeral prompt layers (budget warnings, context pressure)
  7. Apply prompt caching markers if on Anthropic
  8. Make interruptible API call (_interruptible_api_call)
  9. Parse response:
     - If tool_calls: execute them, append results, loop back to step 5
     - If text response: persist session, flush memory if needed, return
```

### Формат сообщения

Все сообщения используют совместимый с OpenAI формат внутри:

```python
{"role": "system", "content": "..."}
{"role": "user", "content": "..."}
{"role": "assistant", "content": "...", "tool_calls": [...]}
{"role": "tool", "tool_call_id": "...", "content": "..."}
```

Содержание рассуждений (от моделей, поддерживающих расширенное мышление) хранится в `assistant_msg["reasoning"]` и при желании выводится через `reasoning_callback`.

### Правила чередования сообщений

Цикл агента строго соблюдает чередование ролей сообщений:

- После системного сообщения: `User → Assistant → User → Assistant → …`
- При вызове инструмента: `Assistant (с tool_calls) → Tool → Tool → … → Assistant`
- **Никогда** два сообщения ассистента подряд
- **Никогда** два сообщения пользователя подряд
- **Только** роль `tool` может иметь последовательные записи (параллельные результаты инструмента)

Провайдеры проверяют эти последовательности и отклоняют некорректные истории.

## Прерываемые вызовы API

Запросы к API оборачиваются в `_interruptible_api_call()`, который выполняет реальный HTTP‑запрос в фоновом потоке, одновременно отслеживая событие прерывания:

```text
┌────────────────────────────────────────────────────┐
│  Main thread                  API thread           │
│                                                    │
│   wait on:                     HTTP POST           │
│    - response ready     ───▶   to provider         │
│    - interrupt event                               │
│    - timeout                                       │
└────────────────────────────────────────────────────┘
```

При прерывании (пользователь отправляет новое сообщение, команда `/stop` или сигнал):
- Поток API прерывается (ответ отбрасывается)
- Агент может обработать новый ввод или корректно завершиться
- Частичный ответ не вставляется в историю диалога

## Выполнение инструментов

### Последовательно vs Параллельно

Когда модель возвращает вызовы инструментов:

- **Один вызов** → выполняется напрямую в основном потоке
- **Несколько вызовов** → выполняются параллельно через `ThreadPoolExecutor`
  - Исключение: инструменты, помеченные как интерактивные (например, `clarify`), принудительно выполняются последовательно
  - Результаты возвращаются в оригинальном порядке вызовов, независимо от порядка завершения

### Поток выполнения

```text
for each tool_call in response.tool_calls:
    1. Resolve handler from tools/registry.py
    2. Fire pre_tool_call plugin hook
    3. Check if dangerous command (tools/approval.py)
       - If dangerous: invoke approval_callback, wait for user
    4. Execute handler with args + task_id
    5. Fire post_tool_call plugin hook
    6. Append {"role": "tool", "content": result} to history
```

### Инструменты уровня агента

Некоторые инструменты перехватываются `run_agent.py` *до* попадания в `handle_function_call()`:

| Инструмент | Почему перехвачен |
|------------|-------------------|
| `todo` | Читает/записывает локальное состояние задач агента |
| `memory` | Записывает в файлы постоянной памяти с ограничениями по количеству символов |
| `session_search` | Запрашивает историю сессии через базу данных сессий агента |
| `delegate_task` | Создаёт подагента(ов) с изолированным контекстом |

Эти инструменты изменяют состояние агента напрямую и возвращают синтетические результаты без прохождения через реестр.

## Точки обратного вызова

`AIAgent` поддерживает платформенно‑специфичные callbacks, позволяющие отображать прогресс в реальном времени в CLI, шлюзе и интеграциях ACP:

| Callback | Когда вызывается | Используется в |
|----------|------------------|----------------|
| `tool_progress_callback` | До/после каждого выполнения инструмента | CLI‑спиннер, сообщения о прогрессе шлюза |
| `thinking_callback` | Когда модель начинает/прекращает думать | Индикатор CLI «thinking…» |
| `reasoning_callback` | Когда модель возвращает содержимое рассуждений | Отображение рассуждений в CLI, блоки рассуждений шлюза |
| `clarify_callback` | При вызове инструмента `clarify` | Запрос ввода в CLI, интерактивное сообщение шлюза |
| `step_callback` | После каждого полного хода агента | Отслеживание шагов шлюзом, прогресс ACP |
| `stream_delta_callback` | Каждый токен стрима (если включён) | Отображение стрима в CLI |
| `tool_gen_callback` | При парсинге вызова инструмента из стрима | Предпросмотр инструмента в спиннере CLI |
| `status_callback` | Изменения состояния (thinking, executing и т.п.) | Обновления статуса в ACP |

## Бюджет и поведение запасного (вариант) режима

### Бюджет итераций

Агент отслеживает итерации через `IterationBudget`:

- По умолчанию — 90 итераций (настраивается через `agent.max_turns`)
- Каждый агент имеет собственный бюджет. Подагенты получают независимые бюджеты, ограниченные `delegation.max_iterations` (по умолчанию — 50) — суммарное количество итераций родителя и подагентов может превышать лимит родителя
- При достижении 100 % агент останавливается и возвращает сводку выполненной работы

### Запасная (вариант) модель

Когда основная модель терпит неудачу (429 rate limit, 5xx server error, 401/403 auth error):

1. Проверяется список `fallback_providers` в конфигурации
2. Поочерёдно пробуются все запасные провайдеры
3. При успехе разговор продолжается с новым провайдером
4. При 401/403 сначала пытаются обновить учётные данные, затем переходят к следующему запасному провайдеру

Система запасных моделей также покрывает вспомогательные задачи независимо — зрение, сжатие и веб‑извлечение каждый имеют собственную цепочку запасных провайдеров, настраиваемую в секции конфигурации `auxiliary.*`.

## Сжатие и постоянство

### Когда срабатывает сжатие

- **Preflight** (до вызова API): если диалог превышает 50 % окна контекста модели
- **Автосжатие шлюза**: если диалог превышает 85 % (более агрессивно, запускается между ходами)

### Что происходит во время сжатия

1. Память сначала сбрасывается на диск (чтобы избежать потери данных)
2. Средние ходы диалога суммируются в компактное резюме
3. Сохраняются последние N сообщений без изменений (`compression.protect_last_n`, по умолчанию — 20)
4. Пары сообщений «вызов инструмента / результат» сохраняются вместе (не разделяются)
5. Генерируется новый идентификатор родословной сессии (сжатие создаёт «дочернюю» сессию)

### Постоянство сессии

После каждого хода:
- Сообщения сохраняются в хранилище сессий (SQLite через `hermes_state.py`)
- Изменения памяти сбрасываются в `MEMORY.md` / `USER.md`
- Сессия может быть возобновлена позже через `/resume` или `hermes chat --resume`

## Ключевые файлы исходного кода

| Файл | Назначение |
|------|------------|
| `run_agent.py` | Класс `AIAgent` — полный цикл агента |
| `agent/prompt_builder.py` | Сборка системной подсказки из памяти, навыков, файлов контекста, личности |
| `agent/context_engine.py` | Абстрактный класс `ContextEngine` — подключаемое управление контекстом |
| `agent/context_compressor.py` | Движок по умолчанию — алгоритм потерьного суммирования |
| `agent/prompt_caching.py` | Маркеры кэширования подсказок Anthropic и метрики кэша |
| `agent/auxiliary_client.py` | Вспомогательный клиент LLM для побочных задач (зрение, суммирование) |
| `model_tools.py` | Сборка схем инструментов, диспетчер `handle_function_call()` |
