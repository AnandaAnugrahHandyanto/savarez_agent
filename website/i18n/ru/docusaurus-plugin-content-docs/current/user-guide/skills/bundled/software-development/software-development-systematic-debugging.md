# Систематическая отладка
## Обзор

Случайные исправления тратят время и создают новые баги. Быстрые патчи скрывают основные проблемы.

**Основной принцип:** ВСЕГДА находить коренную причину перед попыткой исправлений. Исправление симптомов — провал.

**Нарушение буквы этого процесса — нарушение духа отладки.**
## Железный закон

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

Если ты не завершил фазу 1, ты не можешь предлагать исправления.
## Когда использовать

Используй для **любых** технических проблем:
- Ошибки тестов
- Ошибки в продакшене
- Неожиданное поведение
- Проблемы с производительностью
- Ошибки сборки
- Проблемы интеграции

**Используй это **особенно**, когда:**
- На тебя давит время (в экстренных ситуациях хочется угадывать)
- «Просто быстрое исправление» кажется очевидным
- Ты уже попробовал несколько решений
- Предыдущее исправление не сработало
- Ты не полностью понимаешь проблему

**Не пропускай, если:**
- Проблема кажется простой (у простых багов тоже есть коренные причины)
- Ты спешишь (спешка гарантирует переделку)
- Кто‑то требует исправить **сейчас** (системный подход быстрее, чем хаотичное исправление)

## Phase 1: Исследование корневой причины

**ПРЕД ТЕМ как пытаться ВНУТРЬ любые исправления:**

### 1. Внимательно читай сообщения об ошибках

- Не пропускай ошибки или предупреждения
- Они часто содержат точное решение
- Читай трассировки стека полностью
- Обращай внимание на номера строк, пути к файлам, коды ошибок

**Действие:** Используй `read_file` для соответствующих исходных файлов. Используй `search_files`, чтобы найти строку ошибки в кодовой базе.

### 2. Воспроизводи последовательно

- Можешь ли ты вызвать её надёжно?
- Каковы точные шаги?
- Происходит ли это каждый раз?
- Если не воспроизводится → собирай больше данных, не угадывай

**Действие:** Используй инструмент `terminal`, чтобы запустить падающий тест или вызвать баг:

```bash
# Run specific failing test
pytest tests/test_module.py::test_name -v

# Run with verbose output
pytest tests/test_module.py -v --tb=long
```

### 3. Проверь последние изменения

- Что изменилось и могло вызвать проблему?
- `git diff`, последние коммиты
- Новые зависимости, изменения конфигурации

**Действие:**

```bash
# Recent commits
git log --oneline -10

# Uncommitted changes
git diff

# Changes in specific file
git log -p --follow src/problematic_file.py | head -100
```

### 4. Сбор доказательств в многокомпонентных системах

**КОГДА система имеет несколько компонентов (API → service → database, CI → build → deploy):**

**ПРЕД ТЕМ как предлагать исправления, добавь диагностическое инструментирование:**

Для КАЖДОЙ границы компонента:
- Логируй, какие данные входят в компонент
- Логируй, какие данные выходят из компонента
- Проверяй распространение окружения/конфигурации
- Проверяй состояние на каждом уровне

Запусти один раз, чтобы собрать доказательства, показывающие, ГДЕ происходит сбой.
ЗАТЕМ проанализируй доказательства, чтобы определить проблемный компонент.
ЗАТЕМ исследуй именно этот компонент.

### 5. Трассировка потока данных

**КОГДА ошибка находится глубоко в стеке вызовов:**

- Откуда берётся неправильное значение?
- Какая функция вызвала эту функцию с плохим значением?
- Продолжай трассировать вверх, пока не найдёшь источник
- Исправляй в источнике, а не на уровне симптома

**Действие:** Используй `search_files`, чтобы проследить ссылки:

```python
# Find where the function is called
search_files("function_name(", path="src/", file_glob="*.py")

# Find where the variable is set
search_files("variable_name\\s*=", path="src/", file_glob="*.py")
```

### Чек‑лист завершения Phase 1

- [ ] Сообщения об ошибках полностью прочитаны и поняты
- [ ] Проблема воспроизведена последовательно
- [ ] Последние изменения выявлены и проверены
- [ ] Сбор доказательств (логи, состояние, поток данных) выполнен
- [ ] Проблема изолирована до конкретного компонента/кода
- [ ] Сформирована гипотеза о корневой причине

**СТОП:** Не переходи к Phase 2, пока не поймёшь, ПОЧЕМУ это происходит.
## Phase 2: Анализ шаблона

**Найди шаблон перед исправлением:**

### 1. Найти рабочие примеры

- Найди похожий рабочий код в той же кодовой базе
- Что работает и схоже с тем, что сломано?

**Действие:** Используй `search_files`, чтобы найти сопоставимые шаблоны:

```python
search_files("similar_pattern", path="src/", file_glob="*.py")
```

### 2. Сравнить с референсами

- Если реализуешь шаблон, прочитай референсную реализацию ПОЛНОСТЬЮ
- Не пролистывай — читай каждую строку
- Пойми шаблон полностью перед применением

### 3. Выявить различия

- Что отличается между рабочим и неработающим?
- Перечисли каждое различие, даже самое мелкое
- Не предполагай, что это не имеет значения

### 4. Понять зависимости

- Какие другие компоненты нужны?
- Какие настройки, конфигурация, окружение?
- Какие предположения он делает?

## Фаза 3: Гипотеза и тестирование

**Научный метод:**

### 1. Сформулировать одну гипотезу

- Чётко сформулируй: «Я считаю, что X является причиной, потому что Y»
- Запиши её
- Будь конкретен, а не расплывчатой

### 2. Тестировать минимально

- Внеси НАИМЕНЬШЕЕ возможное изменение, чтобы проверить гипотезу
- Одна переменная за раз
- Не исправляй сразу несколько вещей

### 3. Проверить перед тем как продолжать

- Сработало? → Фаза 4
- Не сработало? → Сформулировать НОВУЮ гипотезу
- НЕ добавляй новые исправления поверх

### 4. Когда не знаешь

- Скажи «Я не понимаю X»
- Не притворяйся, что знаешь
- Попроси пользователя о помощи
- Исследуй дальше

## Фаза 4: Реализация

**Исправляй коренную причину, а не симптом:**

### 1. Создай падающий тест

- Самый простой способ воспроизведения
- Автоматический тест, если возможно
- ДОЛЖЕН быть до исправления
- Используй навык `test-driven-development`

### 2. Реализуй единственное исправление

- Устрани выявленную коренную причину
- ОДНО изменение за раз
- Без улучшений «пока я здесь»
- Без объединённого рефакторинга

### 3. Проверь исправление

```bash
# Run the specific regression test
pytest tests/test_module.py::test_regression -v

# Run full suite — no regressions
pytest tests/ -q
```

### 4. Если исправление не работает — правило трёх

- **СТОП.**
- Счёт: сколько исправлений ты попробовал?
- Если &lt; 3: вернись к Фазе 1, проанализируй заново с новой информацией
- **Если ≥ 3: СТОП и поставь под вопрос архитектуру (шаг 5 ниже)**
- НЕ пытайся выполнить исправление #4 без обсуждения архитектуры

### 5. Если 3 и более исправлений не удались: поставь под вопрос архитектуру

**Шаблон, указывающий на проблему архитектуры:**
- Каждое исправление раскрывает новое общее состояние/связь в другом месте
- Для реализации исправлений требуется «масштабный рефакторинг»
- Каждое исправление создаёт новые симптомы в других частях

**СТОП и поставь под вопрос основы:**
- Является ли этот шаблон фундаментально правильным?
- Мы «привязались к нему из‑за инерции»?
- Нужно ли рефакторить архитектуру вместо того, чтобы продолжать исправлять симптомы?

**Обсуди с пользователем, прежде чем пытаться делать дальнейшие исправления.**

Это НЕ провал гипотезы — это неверная архитектура.
## Красные флаги — ОСТАНОВИСЬ и следуй процессу

Если ты замечаешь, что думаешь:
- «Быстрое исправление на сейчас, разберусь позже»
- «Просто попробую изменить X и посмотрю, сработает ли»
- «Добавлю несколько изменений, запущу тесты»
- «Пропущу тест, проверю вручную»
- «Наверное, это X, давай поправим это»
- «Я не полностью понимаю, но это может сработать»
- «Шаблон говорит X, но я адаптирую его иначе»
- «Вот основные проблемы: [список исправлений без расследования]»
- Предлагаешь решения, не проследив поток данных
- **«Ещё одна попытка исправления» (когда уже попытался 2 и более раз)**
- **Каждое исправление выявляет новую проблему в другом месте**

**Все это значит: ОСТАНОВИСЬ. Вернись к Фазе 1.**

**Если 3 и более попыток исправления не удались:** Сомневайся в архитектуре (шаг 5 Фазы 4).
## Общие рационализации

| Оправдание | Реальность |
|------------|------------|
| «Проблема простая, процесс не нужен» | Простые проблемы тоже имеют коренные причины. Процесс ускоряет решение простых багов. |
| «Экстренно, нет времени на процесс» | Систематическая отладка **быстрее**, чем угадывание и проверка. |
| «Сначала попробую это, потом разберусь» | Первый фикс задаёт шаблон. Делай правильно с самого начала. |
| «Напишу тест после того, как убедюсь, что фикс работает» | Непроверенные фиксы не держатся. Тест сначала подтверждает его работоспособность. |
| «Несколько фиксов сразу экономят время» | Нельзя изолировать, что сработало. Появляются новые баги. |
| «Ссылка слишком длинная, я адаптирую шаблон» | Частичное понимание гарантирует баги. Прочитай полностью. |
| «Я вижу проблему, позвольте мне её исправить» | Видеть симптомы ≠ понимать коренную причину. |
| «Ещё одна попытка исправления» (после 2+ неудач) | 3+ неудачи = архитектурная проблема. Сомневайся в шаблоне, а не фиксируй снова.
## Быстрая справка

| Этап | Ключевые действия | Критерии успеха |
|------|-------------------|----------------|
| **1. Коренная причина** | Изучить ошибки, воспроизвести их, проверить изменения, собрать доказательства, отследить поток данных | Понять что и почему |
| **2. Паттерн** | Найти работающие примеры, сравнить, выявить различия | Понять, что отличается |
| **3. Гипотеза** | Сформировать теорию, провести минимальное тестирование, меняя по одной переменной | Подтверждена или новая гипотеза |
| **4. Реализация** | Создать регрессионный тест, исправить коренную причину, проверить | Ошибка устранена, все тесты проходят |
## Интеграция Hermes Agent

### Инструменты расследования

Используй эти инструменты Hermes на этапе Phase 1:

- **`search_files`** — поиск строк ошибок, трассировка вызовов функций, поиск шаблонов
- **`read_file`** — чтение исходного кода с номерами строк для точного анализа
- **`terminal`** — запуск тестов, проверка истории git, воспроизведение багов
- **`web_search`/`web_extract`** — исследование сообщений об ошибках, документации библиотек

### С delegate_task

Для сложной многокомпонентной отладки запускай подагенты‑исследователи:

```python
delegate_task(
    goal="Investigate why [specific test/behavior] fails",
    context="""
    Follow systematic-debugging skill:
    1. Read the error message carefully
    2. Reproduce the issue
    3. Trace the data flow to find root cause
    4. Report findings — do NOT fix yet

    Error: [paste full error]
    File: [path to failing code]
    Test command: [exact command]
    """,
    toolsets=['terminal', 'file']
)
```

### С test-driven-development

При исправлении багов:
1. Напиши тест, который воспроизводит баг (RED)
2. Систематически отлаживай, чтобы найти причину
3. Исправь причину (GREEN)
4. Тест подтверждает исправление и предотвращает регрессию
## Реальное влияние

Из сеансов отладки:
- Систематический подход: 15–30 минут на исправление
- Подход случайных исправлений: 2–3 часа безрезультатных попыток
- Доля исправлений с первой попытки: 95 % против 40 %
- Внедрено новых багов: почти ноль против частых случаев

**Никаких коротких путей. Никаких догадок. Систематический подход всегда побеждает.**
