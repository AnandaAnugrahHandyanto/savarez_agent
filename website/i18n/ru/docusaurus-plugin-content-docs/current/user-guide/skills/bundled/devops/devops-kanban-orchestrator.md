# Kanban Orchestrator — Playbook по декомпозиции

> **Основной жизненный цикл рабочего процесса** (включая шаблон fan‑out `kanban_create` и правило «декомпозировать, а не выполнять») автоматически внедряется в каждый процесс kanban через блок системного подсказки `KANBAN_GUIDANCE`. Этот навык — более подробный playbook для профиля оркестратора, чья основная задача — маршрутизация.
## Профили настраиваются пользователем — это не фиксированный список

Настройки Hermes сильно различаются. Некоторые пользователи используют один профиль, который делает всё; некоторые запускают небольшой набор (`docker‑worker`, `cron‑worker`); некоторые создают курируемую команду специалистов, названную ими самими. **Нет списка специалистов по умолчанию** — навык‑оркестратор не знает, какие профили существуют на этой машине.

Прежде чем распределять задачи, необходимо опираться на те профили, которые действительно существуют. Диспетчер молча не создаёт задачи с неизвестными именами исполнителей — он не автокорректирует, не предлагает варианты, не переходит к запасному (варианту). Поэтому задача, назначенная `researcher` в настройке, где есть только `docker‑worker`, будет вечно находиться в состоянии `ready`.

**Шаг 0: обнаружить доступные профили перед планированием.**

Используй один из вариантов:

- `hermes profile list` — выводит таблицу профилей, настроенных на этой машине. Запусти её через свой терминальный инструмент, если он у тебя есть; иначе спроси пользователя.
- `kanban_list(assignee="<some-name>")` — проверка одного имени. Возвращает пустой список (а не ошибку) для неизвестного исполнителя, так что это лишь подтверждает имя, которое ты уже рассматриваешь.
- **Просто спроси пользователя.** «Какие профили у тебя настроены?» — хороший первый вопрос, когда цель требует более одного специалиста.

Сохрани результат в своей рабочей памяти на оставшуюся часть диалога. Переспрашивать каждый ход — пустая трата вызова инструмента.
## Когда использовать доску (а не просто выполнять работу)

Создавай задачи Kanban, когда выполняется любое из следующих условий:

1. **Требуются несколько специалистов.** Research + analysis + writing — это три профиля.
2. **Работа должна выжить после сбоя или перезапуска.** Длительная, периодическая или важная.
3. **Пользователь может захотеть вмешаться.** Human‑in‑the‑loop на любом этапе.
4. **Несколько подзадач могут выполняться параллельно.** Распараллеливание для ускорения.
5. **Ожидается проверка / итерация.** Профиль проверяющего циклически обрабатывает вывод drafter.
6. **Важно наличие аудита.** Строки доски сохраняются в SQLite навсегда.

Если *ни одно* из этих условий не применимо — это небольшая одноразовая задача рассуждения — используй `delegate_task` вместо этого или ответь пользователю напрямую.
## Правила против искушения

Your job description says "route, don't execute." The rules that enforce that:

- **Не выполняй работу сам.** Твой ограниченный набор инструментов обычно даже не включает терминал/файлы/код/веб для реализации. Если ты начинаешь «просто быстро поправить» — остановись и создай задачу для нужного специалиста.
- **Для любой конкретной задачи создай задачу в Kanban и назначь её.** Каждый раз.
- **Разделяй запросы с несколькими направлениями перед созданием карточек.** Запрос пользователя может содержать несколько независимых потоков работы. Сначала выдели эти направления, затем создай по одной карточке на каждое направление вместо объединения несвязанных задач в одну карточку исполнителя.
- **Выполняй независимые направления параллельно.** Если две карточки не зависят друг от друга, оставь их несвязанными, чтобы диспетчер мог разветвить их. Связывай только истинные зависимости данных.
- **Никогда не создавай зависимую работу как независимые готовые карточки.** Если карточка должна ждать другую карточку, передай `parents=[...]` в оригинальном вызове `kanban_create`. Не создавай её сначала и не связывай позже, и не полагайся на текст вроде «ждать T1» в теле.
- **Если ни один специалист не подходит под доступные профили, спроси пользователя, какой профиль создать или какой существующий профиль использовать.** Не придумывай названия профилей; диспетчер тихо отбрасывает неизвестных исполнителей.
- **Разбивай, направляй и подводи итоги — вот вся работа.**
## Руководство по декомпозиции

### Шаг 1 — Понять цель

Задавай уточняющие вопросы, если цель неясна. Спросить — дешево; дорого будет запустить неправильный флот.

### Шаг 2 — Набросить граф задач

Прежде чем что‑то создавать, набросай граф вслух (в своём ответе пользователю). Рассматривай каждый конкретный рабочий поток как кандидат‑карту:

1. Выдели **lanes** из запроса.
2. Сопоставь каждую **lane** с одним из профилей, обнаруженных на Шаге 0. Если **lane** не подходит ни к одному существующему профилю, спроси у пользователя, какой использовать, или создай новый.
3. Определи, является ли каждая **lane** независимой или зависит от другой **lane**.
4. Создай независимые **lanes** как параллельные карты без ссылок‑родителей.
5. Создай карты синтеза/ревью/интеграции со ссылками‑родителями на **lanes**, от которых они зависят. Карта‑ребёнок, созданная с незавершёнными родителями, начинается в `todo`; диспетчер переводит её в `ready` только после завершения всех родителей.

Примеры подсказок, которые следует разветвлять (используя имена профилей‑заполнителей — подставь те, что есть у пользователя):

- «Build an app» → одна карта для профиля, ориентированного на дизайн/продукт, одна‑две карты для инженерных профилей для реализации, плюс позже карта интеграции/ревью, если у пользователя есть профиль ревьюера.
- «Fix blockers and check model variants» → одна карта реализации для исправления блокеров и одна карта исследования/проверки конфигураций/источников. Финальная карта ревьюера может зависеть от обеих.
- «Research docs and implement» → карта исследования docs может работать параллельно с картой обнаружения кода; реализация ждёт только если действительно нужны эти выводы.
- «Analyze this screenshot and find the related code» → одна карта для профиля с возможностями зрения для визуального анализа, а другая ищет кодовую базу.

Слова вроде «also», «finally» или «and» автоматически не означают зависимость. Чаще они означают «убедись, что это покрыто перед тем, как отчитаться». Связывай задачи только тогда, когда одна карта не может начаться, пока не появятся результаты другой.

Покажи граф пользователю перед созданием карт. Позволь ему исправить его — включая то, какое именно имя профиля должно владеть каждой **lane**.

### Шаг 3 — Создать задачи и связать их

Используй имена профилей из Шага 0. Пример ниже использует заполнители `<profile-A>`, `<profile-B>`, `<profile-C>` — замени их теми, что действительно есть у пользователя.

```python
t1 = kanban_create(
    title="research: Postgres cost vs current",
    assignee="<profile-A>",  # whichever profile handles research on this setup
    body="Compare estimated infrastructure costs, migration costs, and ongoing ops costs over a 3-year window. Sources: AWS/GCP pricing, team time estimates, current Postgres bills from peers.",
    tenant=os.environ.get("HERMES_TENANT"),
)["task_id"]

t2 = kanban_create(
    title="research: Postgres performance vs current",
    assignee="<profile-A>",  # same profile, run in parallel
    body="Compare query latency, throughput, and scaling characteristics at our expected data volume (~500GB, 10k QPS peak). Sources: benchmark papers, public case studies, pgbench results if easy.",
)["task_id"]

t3 = kanban_create(
    title="synthesize migration recommendation",
    assignee="<profile-B>",  # whichever profile does synthesis/analysis
    body="Read the findings from T1 (cost) and T2 (performance). Produce a 1-page recommendation with explicit trade-offs and a go/no-go call.",
    parents=[t1, t2],
)["task_id"]

t4 = kanban_create(
    title="draft decision memo",
    assignee="<profile-C>",  # whichever profile drafts user-facing prose
    body="Turn the analyst's recommendation into a 2-page memo for the CTO. Match the tone of previous decision memos in the team's knowledge base.",
    parents=[t3],
)["task_id"]
```

`parents=[...]` ограничивает продвижение — дочерние задачи остаются в `todo`, пока каждый родитель не перейдёт в `done`, после чего автоматически продвигаются в `ready`. Ручная координация не нужна; диспетчер и движок зависимостей справятся.

Если граф задач имеет зависимости, сначала создай карты‑родители, получи их идентификаторы и включи эти `id` в список `parents` дочерней карты при вызове `kanban_create`. Не создавай все карты параллельно и потом связывай их; это создаёт окно, в котором диспетчер может захватить дочернюю задачу до появления её входов.

### Шаг 4 — Завершить свою собственную задачу

Если ты был запущен как задача сам (например, профилю‑планировщику был назначен `T0: "investigate Postgres migration"`), пометь её выполненной с резюме того, что ты создал:

```python
kanban_complete(
    summary="decomposed into T1-T4: 2 research lanes in parallel, 1 synthesis on their outputs, 1 prose draft on the recommendation",
    metadata={
        "task_graph": {
            "T1": {"assignee": "<profile-A>", "parents": []},
            "T2": {"assignee": "<profile-A>", "parents": []},
            "T3": {"assignee": "<profile-B>", "parents": ["T1", "T2"]},
            "T4": {"assignee": "<profile-C>", "parents": ["T3"]},
        },
    },
)
```

### Шаг 5 — Отчитаться пользователю

Скажи ему, что ты создал, простым prose, указав реальные профили, которые использовал:

> Я поставил в очередь 4 задачи:
> - **T1** (`<profile-A>`): сравнение стоимости
> - **T2** (`<profile-A>`): сравнение производительности, параллельно с T1
> - **T3** (`<profile-B>`): синтезирует T1 + T2 в рекомендацию
> - **T4** (`<profile-C>`): превращает T3 в мемо CTO
>
> Диспетчер сейчас возьмёт T1 и T2. T3 начнётся, когда обе завершатся. Ты получишь ping от шлюза, когда T4 завершится. Используй дашборд или `hermes kanban tail <id>`, чтобы следить за процессом.
## Общие шаблоны

**Fan-out + fan-in (research → synthesize):** N карточек типа `research` без родителей, одна карточка `synthesize`, у которой все они указаны как родители.

**Parallel implementation + validation:** одна карточка `implementer` вносит изменение, пока карточка `explorer/researcher` проверяет конфигурацию, документацию или сопоставление исходного кода. Карточка `reviewer` может зависеть от обеих. Не заставляй `implementer` выполнять несвязанные проверки только потому, что пользователь упомянул их в одном предложении.

**Pipeline with gates:** `planner → implementer → reviewer`. Для каждого этапа `parents=[previous_task]`. `Reviewer` блокирует или завершает задачу; если `reviewer` блокирует, оператор разблокирует её с обратной связью и перезапускает.

**Same‑profile queue:** N задач, все назначены одному и тому же профилю, без зависимостей между ними. `Dispatcher` сериализует — этот профиль обрабатывает их в порядке приоритета, накапливая опыт в своей памяти.

**Human‑in‑the‑loop:** Любая задача может вызвать `kanban_block()`, чтобы ждать ввода. `Dispatcher` перезапускает задачу после `/unblock`. Ветка комментариев сохраняет полный контекст.
## Подводные камни

**Придумывание имён профилей, которых не существует.** Диспетчер молча не создаёт задачи для неизвестных исполнителей — карточка просто остаётся в `ready` навечно. Всегда назначай задачу профилю, найденному на Шаге 0; если не уверен, спроси пользователя.

**Объединение независимых дорожек в одну карточку.** Если пользователь просит два независимых результата, создай две карточки. Пример: «исправить блокирующие проблемы и проверить варианты модели» — это не одна задача исправления; создай карточку *fixer/engineer* для исправлений и карточку *explorer/researcher* для проверки вариантов, затем при желании поставь проверку обеих карточек в шлюз инструментов.

**Чрезмерное связывание из‑за формулировки.** «Наконец проверить X» может оставаться параллельно с реализацией, если X — статическая конфигурация, документация или поиск источников. Связывай её с реализацией только тогда, когда проверка зависит от результата реализации.

**Забывание ссылок‑зависимостей.** Если граф задач выглядит так: `research -> implement -> review`, не создавай все задачи как независимые карточки в состоянии `ready`. Используй ссылки‑родители, чтобы `implement`/`review` не могли запуститься до появления их входных данных.

**Переназначение vs. новая задача.** Если ревьюер блокирует задачу с пометкой «нужны изменения», создай НОВУЮ задачу, связанную с задачей ревьюера — не перезапускай ту же задачу с суровым видом. Новая задача назначается профилю оригинального исполнителя.

**Порядок аргументов для ссылок.** `kanban_link(parent_id=..., child_id=...)` — сначала родитель. Перемешивание их приводит к понижению неверной задачи до `todo`.

**Не предсоздавай весь граф, если его форма зависит от промежуточных результатов.** Если структура T3 зависит от того, что найдут T1 и T2, позволь T3 существовать как задачу «синтезировать выводы», первая часть которой — прочитать передачи от родителей и спланировать дальнейшие шаги. Оркестраторы могут создавать оркестраторы.

**Наследование арендатора.** Если в окружении установлен `HERMES_TENANT`, передавай `tenant=os.environ.get("HERMES_TENANT")` в каждый вызов `kanban_create`, чтобы дочерние задачи оставались в том же пространстве имён.
## Восстановление зависших воркеров

Когда профиль воркера постоянно падает, галлюцинирует или блокируется из‑за собственных ошибок (обычно: неверная модель, отсутствующий skill, сломанные учётные данные), панель Kanban помечает задачу значком ⚠ и открывает раздел **Recovery** в выдвижном меню. Три основных действия:

1. **Reclaim** (или `hermes kanban reclaim <task_id>`) — немедленно прервать работающего воркера и вернуть задачу в состояние `ready`. Текущее время жизни claim ≈ 15 мин; это быстрый путь выхода.
2. **Reassign** (или `hermes kanban reassign <task_id> <new-profile> --reclaim`) — переключить задачу на другой профиль (который существует в этой конфигурации) и позволить диспетчеру подобрать её с новым воркером.
3. **Change profile model** — панель выводит подсказку для копирования‑вставки `hermes -p <profile> model`, так как конфигурация профиля хранится на диске; отредактируй её в терминале, затем выполни **Reclaim**, чтобы повторить попытку с новой моделью.

Предупреждения о галлюцинациях появляются в задачах, где claim воркера `kanban_complete(created_cards=[...])` включал идентификаторы карточек, которых не существует или которые не были созданы профилем воркера (шлюз блокирует завершение), либо где свободно‑форматное резюме ссылается на идентификаторы `t_<hex>`, которые не разрешаются (советующий скан текста, не блокирующий). Оба случая генерируют события аудита, которые сохраняются даже после действий по восстановлению — след остаётся для отладки.
