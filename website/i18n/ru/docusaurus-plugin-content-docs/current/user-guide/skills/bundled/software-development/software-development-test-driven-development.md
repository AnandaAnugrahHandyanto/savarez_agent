# Разработка через тестирование (TDD)
## Общее описание

Сначала напиши тест. Посмотри, как он падает. Затем напиши минимальный код, чтобы он прошёл.

**Основной принцип:** Если ты не видел, как тест падает, ты не знаешь, проверяет ли он нужное.

**Нарушение буквы правил — нарушение духа правил.**
## Когда использовать

**Всегда:**
- Новые функции
- Исправления ошибок
- Рефакторинг
- Изменения поведения

**Исключения (спроси пользователя сначала):**
- Одноразовые прототипы
- Сгенерированный код
- Файлы конфигурации

Думаешь «пропустить TDD хотя бы один раз»? Стоп. Это рационализация.
## Железный закон

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

Писать код перед тестом? Удали его. Начни заново.

**Без исключений:**
- Не оставляй его как «reference»
- Не «адаптируй» его во время написания тестов
- Не смотри на него
- Удалить — значит удалить

Реализуй всё заново, опираясь только на тесты. Точка.
## Цикл Red‑Green‑Refactor

### RED — Написать падающий тест

Напиши один минимальный тест, показывающий, что должно произойти.

**Хороший тест:**
```python
def test_retries_failed_operations_3_times():
    attempts = 0
    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise Exception('fail')
        return 'success'

    result = retry_operation(operation)

    assert result == 'success'
    assert attempts == 3
```
Чёткое название, проверяет реальное поведение, один аспект.

**Плохой тест:**
```python
def test_retry_works():
    mock = MagicMock()
    mock.side_effect = [Exception(), Exception(), 'success']
    result = retry_operation(mock)
    assert result == 'success'  # What about retry count? Timing?
```
Неоднозначное название, проверяет мок, а не реальный код.

**Требования:**
- По одному поведению на тест
- Чёткое описательное название («and» в названии? Раздели его)
- Реальный код, а не моки (если только это действительно невозможно)
- Название описывает поведение, а не реализацию

### Verify RED — Убедиться, что тест падает

**ОБЯЗАТЕЛЬНО. Никогда не пропускай.**

```bash
# Use terminal tool to run the specific test
pytest tests/test_feature.py::test_specific_behavior -v
```

Проверь:
- Тест падает (не ошибки из‑за опечаток)
- Сообщение об ошибке соответствует ожиданиям
- Падение происходит из‑за отсутствующей функции

**Тест проходит сразу?** Ты проверяешь уже существующее поведение. Исправь тест.

**Тест выдаёт ошибку?** Исправь ошибку, запусти снова, пока он не упадёт корректно.

### GREEN — Минимальный код

Напиши самый простой код, чтобы тест прошёл. Ничего больше.

**Хорошо:**
```python
def add(a, b):
    return a + b  # Nothing extra
```

**Плохо:**
```python
def add(a, b):
    result = a + b
    logging.info(f"Adding {a} + {b} = {result}")  # Extra!
    return result
```

Не добавляй функции, не рефакторь другой код и не «улучшаешь» за пределами теста.

**Обман разрешён в GREEN:**
- Жёстко задать возвращаемые значения
- Копировать‑вставлять
- Дублировать код
- Пропускать граничные случаи

Мы исправим это в REFACTOR.

### Verify GREEN — Убедиться, что тест прошёл

**ОБЯЗАТЕЛЬНО.**

```bash
# Run the specific test
pytest tests/test_feature.py::test_specific_behavior -v

# Then run ALL tests to check for regressions
pytest tests/ -q
```

Проверь:
- Тест проходит
- Остальные тесты тоже проходят
- Вывод чистый (без ошибок, предупреждений)

**Тест падает?** Исправляй код, а не тест.

**Другие тесты падают?** Сразу исправляй регрессии.

### REFACTOR — Очистка

После того как GREEN прошёл:
- Убирай дублирование
- Улучшаешь названия
- Выноси вспомогательные функции
- Упрощай выражения

Поддерживай зелёный статус тестов всё время. Не добавляй новое поведение.

**Если тесты падают во время рефакторинга:** откатывай сразу. Делай более мелкие шаги.

### Повторить

Следующий падающий тест для следующего поведения. По одному циклу за раз.
## Почему порядок имеет значение

**«Я напишу тесты позже, чтобы проверить, что всё работает»**

Тесты, написанные после кода, сразу проходят. Немедленный проход ничего не доказывает:
- Может проверять не то, что нужно
- Может проверять реализацию, а не поведение
- Может упустить крайние случаи, о которых ты забыл
- Ты никогда не увидишь, как он ловит ошибку

Тест‑первый заставляет увидеть провал теста, доказывая, что он действительно что‑то проверяет.

**«Я уже вручную протестировал все крайние случаи»**

Ручное тестирование — ad‑hoc. Ты считаешь, что протестировал всё, но:
- Нет записи о том, что было протестировано
- Нельзя повторно запустить, когда код меняется
- Легко забыть случаи под давлением
- «Это работало, когда я пробовал» ≠ всесторонне

Автоматические тесты систематичны. Они запускаются одинаково каждый раз.

**«Удалять X часов работы — пустая трата»**

Ошибка «затраченных средств». Время уже ушло. Твой выбор сейчас:
- Удалить и переписать с TDD (высокая уверенность)
- Оставить и добавить тесты позже (низкая уверенность, вероятно баги)

«Трата» — это сохранение кода, которому нельзя доверять.

**«TDD догматичен, быть прагматичным значит адаптироваться»**

TDD — это прагматичность:
- Находит баги до коммита (быстрее, чем отладка после)
- Предотвращает регрессии (тесты сразу ловят поломки)
- Документирует поведение (тесты показывают, как использовать код)
- Позволяет рефакторинг (меняй свободно, тесты поймают поломки)

«Прагматичные» сокращения = отладка в продакшене = медленнее.

**«Тесты после достигают тех же целей — это дух, а не ритуал»**

Нет. Тесты‑после отвечают на вопрос «Что это делает?». Тесты‑сначала отвечают на вопрос «Что это должно делать?».

Тесты‑после предвзяты твоей реализацией. Ты тестируешь то, что построил, а не то, что требуется. Тесты‑сначала заставляют обнаружить крайние случаи до реализации.
## Распространённые оправдания

| Оправдание | Реальность |
|------------|------------|
| «Слишком простое, чтобы тестировать» | Простой код ломается. Тест занимает 30 секунд. |
| «Я протестирую позже» | Тесты, проходящие сразу, ничего не доказывают. |
| «Тесты после достигают тех же целей» | Тесты‑после = «что это делает?». Тесты‑сначала = «что это должно делать?». |
| «Уже протестировано вручную» | Ад‑хок ≠ систематический. Нет записи, нельзя пере‑запустить. |
| «Удалять X часов — пустая трата» | Ошибка «затраты упущены». Сохранение непроверенного кода — технический долг. |
| «Сохранить как справку, писать тесты позже» | Ты всё равно адаптируешь его. Это тестирование после. Удалить — значит удалить. |
| «Нужно сначала исследовать» | Хорошо. Выбрось исследование, начни с TDD. |
| «Трудный тест = неясный дизайн» | Слушай тест. Трудно протестировать = трудно использовать. |
| «TDD замедлит меня» | TDD быстрее, чем отладка. Прагматичный подход = тестировать сначала. |
| «Ручное тестирование быстрее» | Ручное не доказывает граничные случаи. Ты будешь пере‑тестировать каждое изменение. |
| «В существующем коде нет тестов» | Ты его улучшaешь. Добавляй тесты для кода, к которому прикасаешься. |
## Красные флажки — ОСТАНОВИСЬ и начни заново

Если ты замечаешь, что делаешь что‑то из перечисленного, удали код и начни заново с TDD:

- Код до теста
- Тест после реализации
- Тест проходит сразу при первом запуске
- Не можешь объяснить, почему тест провалился
- Тесты добавлены «позже»
- Убеждаешь себя «только в этот раз»
- «Я уже проверил это вручную»
- «Тесты после реализации достигают той же цели»
- «Сохраняю как справку» или «адаптирую существующий код»
- «Уже потратил X часов, удалять — пустая трата»
- «TDD — догма, я практичен»
- «Это иначе, потому что…»

**Все это значит: удали код. Начни заново с TDD.**
## Контрольный список проверки

Перед тем как пометить работу завершённой:

- [ ] У каждой новой функции или метода есть тест
- [ ] Отслеживал каждый провал теста перед реализацией
- [ ] Каждый тест проваливался по ожидаемой причине (отсутствие функции, а не опечатка)
- [ ] Написал минимальный код, позволяющий пройти каждый тест
- [ ] Все тесты проходят
- [ ] Вывод чистый (без ошибок и предупреждений)
- [ ] Тесты используют реальный код (моки только в случае крайней необходимости)
- [ ] Покрыты граничные случаи и ошибки

Не можешь поставить галочки во всех пунктах? Ты пропустил TDD. Начни заново.
## Когда застрял

| Проблема | Решение |
|---------|----------|
| Не знаешь, как тестировать | Напиши желаемый API. Сначала напиши утверждение. Спроси пользователя. |
| Тест слишком сложный | Дизайн слишком сложный. Упрощай интерфейс. |
| Нужно мокировать всё | Код слишком связан. Используй внедрение зависимостей. |
| Настройка теста огромна | Выдели вспомогательные функции. Всё ещё сложно? Упрощай дизайн. |
## Интеграция Hermes Agent

### Запуск тестов

Используй инструмент `terminal` для запуска тестов на каждом этапе:

```python
# RED — verify failure
terminal("pytest tests/test_feature.py::test_name -v")

# GREEN — verify pass
terminal("pytest tests/test_feature.py::test_name -v")

# Full suite — verify no regressions
terminal("pytest tests/ -q")
```

### С delegate_task

При распределении подагентов для реализации обеспечь применение TDD в цели:

```python
delegate_task(
    goal="Implement [feature] using strict TDD",
    context="""
    Follow test-driven-development skill:
    1. Write failing test FIRST
    2. Run test to verify it fails
    3. Write minimal code to pass
    4. Run test to verify it passes
    5. Refactor if needed
    6. Commit

    Project test command: pytest tests/ -q
    Project structure: [describe relevant files]
    """,
    toolsets=['terminal', 'file']
)
```

### С systematic-debugging

Обнаружена ошибка? Напиши падающий тест, воспроизводящий её. Следуй циклу TDD. Тест подтверждает исправление и предотвращает регрессию.

Никогда не исправляй ошибки без теста.
## Тестирование антипаттернов

- **Тестирование поведения моков вместо реального поведения** — моки должны проверять взаимодействия, а не заменять тестируемую систему
- **Тестирование деталей реализации** — проверяй поведение/результаты, а не внутренние вызовы методов
- **Только «счастливый путь»** — всегда тестируй граничные случаи, ошибки и предельные условия
- **Хрупкие тесты** — тесты должны проверять поведение, а не структуру; рефакторинг не должен их ломать
## Финальное правило

```
Production code → test exists and failed first
Otherwise → not TDD
```

Никаких исключений без явного согласия пользователя.
