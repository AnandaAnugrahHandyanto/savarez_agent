# Спайк

Используй этот **skill**, когда пользователь хочет **проверить идею** перед тем, как переходить к реальной реализации — проверить осуществимость, сравнить подходы или выявить неизвестные, на которые никакое исследование не даст ответа. Спайки по задумке одноразовые: выбрасывай их, как только они отработают свою задачу.

Загружай этот **skill**, когда пользователь говорит что‑то вроде «let me try this», «I want to see if X works», «spike this out», «before I commit to Y», «quick prototype of Z», «is this even possible?», или «compare A vs B».
## Когда НЕ использовать это

- Ответ можно получить из документации или просмотрев код — просто проведи исследование, не создавай
- Работа предназначена для production‑пути — используй `writing-plans` / `plan` вместо этого
- Идея уже проверена — переходи сразу к реализации
## Если у пользователя установлен полный GSD‑system

Если `gsd-spike` отображается как соседний skill (установлен через `npx get-shit-done-cc --hermes`), отдавай предпочтение **`gsd-spike`**, когда пользователь хочет использовать полный GSD‑workflow: постоянное состояние в `.planning/spikes/`, отслеживание MANIFEST между сессиями, формат вердикта Given/When/Then и шаблоны коммитов, интегрирующиеся с остальной частью GSD. Этот skill — облегчённая автономная версия для пользователей, у которых нет (или они не хотят) полного GSD‑system.
## Core method

Независимо от масштаба, каждый спайк следует этому циклу:

```
decompose  →  research  →  build  →  verdict
   ↑__________________________________________↓
                  iterate on findings
```

### 1. Декомпозиция

Разбей идею пользователя на **2‑5 независимых вопросов о реализуемости**. Каждый вопрос — это один спайк. Представь их в виде таблицы с формулировкой Given/When/Then:

| # | Спайк | Проверяется (Given/When/Then) | Риск |
|---|-------|-------------------------------|------|
| 001 | websocket-streaming | Given a WS connection, when LLM streams tokens, then client receives chunks &lt; 100 ms | High |
| 002a | pdf-parse-pdfjs | Given a multi-page PDF, when parsed with pdfjs, then structured text is extractable | Medium |
| 002b | pdf-parse-camelot | Given a multi-page PDF, when parsed with camelot, then structured text is extractable | Medium |

**Типы спайков:**
- **standard** — один подход, отвечающий на один вопрос
- **comparison** — один и тот же вопрос, разные подходы (общий номер, суффикс‑буква `a`/`b`/`c`)

**Хорошие вопросы спайка:** конкретная реализуемость с наблюдаемым выводом.
**Плохие вопросы спайка:** слишком общие, без наблюдаемого вывода или просто «прочитать документацию про X».

**Сортировать по риску.** Спайк, который с наибольшей вероятностью «убьёт» идею, выполняется первым. Нет смысла прототипировать лёгкие части, если тяжёлая часть не работает.

**Пропустить декомпозицию** только если пользователь уже точно знает, что именно он хочет реализовать, и сообщает об этом. Тогда возьми его идею как один спайк.

### 2. Выравнивание (для идей с несколькими спайками)

Представь таблицу спайков. Спроси: «Собрать всё в этом порядке или скорректировать?» Позволь пользователю убрать, переупорядочить или переформулировать пункты перед тем, как ты начнёшь писать код.

### 3. Исследование (для каждого спайка, перед построением)

Спайки не свободны от исследований — ты исследуешь достаточно, чтобы выбрать правильный подход, а затем строишь. Для каждого спайка:

1. **Кратко опиши.** 2‑3 предложения: что это за спайк, почему он важен, основной риск.
2. **Выяви конкурирующие подходы**, если есть реальный выбор:

   | Подход | Инструмент/Библиотека | Плюсы | Минусы | Статус |
   |--------|----------------------|------|--------|--------|
   | ... | ... | ... | ... | maintained / abandoned / beta |

3. **Выбери один.** Объясни, почему. Если 2 + подхода заслуживают внимания, создай быстрые варианты в рамках спайка.
4. **Пропусти исследование** для чистой логики без внешних зависимостей.

Используй инструменты Hermes для шага исследования:

- `web_search("python websocket streaming libraries 2025")` — найти кандидаты
- `web_extract(urls=["https://websockets.readthedocs.io/..."])` — прочитать реальные документы (возвращает markdown)
- `terminal("pip show websockets | grep Version")` — проверить, что установлено в виртуальном окружении проекта

Для библиотек без страниц документации клонируй и читай их `README.md` / `examples/` через `read_file`. Context7 MCP (если пользователь его настроил) тоже хороший источник — `mcp_*_resolve-library-id`, затем `mcp_*_query-docs`.

### 4. Сборка

По одному каталогу на спайк. Делай его автономным.

<!-- ascii-guard-ignore -->
```
spikes/
├── 001-websocket-streaming/
│   ├── README.md
│   └── main.py
├── 002a-pdf-parse-pdfjs/
│   ├── README.md
│   └── parse.js
└── 002b-pdf-parse-camelot/
    ├── README.md
    └── parse.py
```
<!-- ascii-guard-ignore-end -->

**Ставь в приоритет то, с чем пользователь может взаимодействовать.** Спайки проваливаются, когда единственный вывод — строка лога «it works». Пользователь хочет *почувствовать* работу спайка. Приоритетные варианты, в порядке предпочтения:

1. Выполняемый CLI, принимающий ввод и выводящий наблюдаемый результат
2. Минимальная HTML‑страница, демонстрирующая поведение
3. Маленький веб‑сервер с одной конечной точкой
4. Юнит‑тест, проверяющий вопрос с явно видимыми утверждениями

**Глубина важнее скорости.** Никогда не объявляй «работает», основываясь лишь на одном успешном запуске. Тестируй граничные случаи. Следуй неожиданным находкам. Вердикт надёжен только при честном исследовании.

**Избегай**, если только спайк не требует этого: сложного управления пакетами, сборочных инструментов/бандлеров, Docker, файлов окружения, систем конфигураций. Всё хардкодь — это спайк.

**Сборка одного спайка** — типичная последовательность инструментов:

```
terminal("mkdir -p spikes/001-websocket-streaming")
write_file("spikes/001-websocket-streaming/README.md", "# 001: websocket-streaming\n\n...")
write_file("spikes/001-websocket-streaming/main.py", "...")
terminal("cd spikes/001-websocket-streaming && python3 main.py")
# Observe output, iterate.
```

**Параллельные спайки‑сравнения (002a / 002b) — делегировать.** Когда два подхода можно выполнять параллельно и оба требуют реального инженерного труда (а не 10‑строчных прототипов), разветвляй с помощью `delegate_task`:

```
delegate_task(tasks=[
    {"goal": "Build 002a-pdf-parse-pdfjs: ...", "toolsets": ["terminal", "file", "web"]},
    {"goal": "Build 002b-pdf-parse-camelot: ...", "toolsets": ["terminal", "file", "web"]},
])
```

Каждый суб‑агент возвращает свой вердикт; ты пишешь сравнение.

### 5. Вердикт

Каждый `README.md` спайка заканчивается:

```markdown
## Verdict: VALIDATED | PARTIAL | INVALIDATED

### What worked
- ...

### What didn't
- ...

### Surprises
- ...

### Recommendation for the real build
- ...
```

**VALIDATED** = основной вопрос получен «да», с доказательствами.
**PARTIAL** = работает при ограничениях X, Y, Z — документируй их.
**INVALIDATED** = не работает, по этой причине. Это тоже успешный спайк.
## Сравнительные спайки

Когда два подхода отвечают на один и тот же вопрос (002a / 002b), собирай их **один за другим**, а затем проведи сравнение «лицом к лицу» в конце:

```markdown
## Head-to-head: pdfjs vs camelot

| Dimension | pdfjs (002a) | camelot (002b) |
|-----------|--------------|----------------|
| Extraction quality | 9/10 structured | 7/10 table-only |
| Setup complexity | npm install, 1 line | pip + ghostscript |
| Perf on 100-page PDF | 3s | 18s |
| Handles rotated text | no | yes |

**Winner:** pdfjs for our use case. Camelot if we need table-first extraction later.
```
## Режим Frontier (выбор следующего спайка)

Если спайки уже существуют и пользователь спрашивает «что мне спайкнуть дальше?», пройди по существующим каталогам и ищи:

- **Integration risks** — два подтверждённых спайка, которые затрагивают один и тот же ресурс, но тестировались независимо
- **Data handoffs** — вывод спайка A предполагался совместимым с вводом спайка B; это никогда не подтверждалось
- **Gaps in the vision** — возможности, предположенные, но не проверенные
- **Alternative approaches** — альтернативные варианты для PARTIAL или INVALIDATED спайков

Предложи 2‑4 кандидата в виде Given/When/Then. Позволь пользователю выбрать.
