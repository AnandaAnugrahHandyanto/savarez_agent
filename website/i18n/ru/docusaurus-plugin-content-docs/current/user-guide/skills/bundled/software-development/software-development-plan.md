# Режим планирования

Используй этот навык, когда пользователь хочет получить план вместо выполнения.

## Основное поведение

На этом ходу ты только планируешь.

- Не реализуй код.
- Не редактируй файлы проекта, кроме markdown‑файла плана.
- Не запускай изменяющие команды терминала, не делай commit, push и не выполняй внешние действия.
- При необходимости можешь просматривать репозиторий или другой контекст только с помощью команд/инструментов в режиме чтения.
- Твой результат — markdown‑план, сохранённый в активном рабочем пространстве под `.hermes/plans/`.

## Требования к выводу

Напиши markdown‑план, который будет конкретным и выполнимым.

Включи, если это уместно:
- Цель
- Текущий контекст / предположения
- Предлагаемый подход
- Пошаговый план
- Файлы, которые, вероятно, изменятся
- Тесты / проверка
- Риски, компромиссы и открытые вопросы

Если задача связана с кодом, укажи точные пути к файлам, вероятные цели тестов и шаги проверки.

## Место сохранения

Сохрани план с помощью `write_file` по пути:
- `.hermes/plans/YYYY-MM-DD_HHMMSS-<slug>.md`

Считай, что путь относительный к активному рабочему каталогу / бэкенд‑рабочему пространству. Инструменты файлов Hermes учитывают бэкенд, поэтому использование относительного пути сохраняет план вместе с пространством в локальном, docker, ssh, modal и daytona бэкендах.

Если среда выполнения предоставляет конкретный целевой путь, используй именно его.
Если нет — создай разумное имя файла с меткой времени в `.hermes/plans/`.

## Стиль взаимодействия

- Если запрос достаточно ясен, сразу запиши план.
- Если к `/plan` нет явной инструкции, выведи задачу из текущего контекста разговора.
- Если запрос действительно недостаточно уточнён, задай короткий уточняющий вопрос вместо догадок.
- После сохранения плана коротко ответь, что ты спланировал, и укажи путь к сохранённому файлу.
