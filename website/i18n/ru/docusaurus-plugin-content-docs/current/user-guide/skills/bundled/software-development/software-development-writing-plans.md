# Планы реализации

## Обзор

Пишите подробные планы внедрения, исходя из того, что исполнитель не знаком с кодовой базой и может выбрать неудачное решение. Опишите всё, что ему понадобится: какие файлы править, полный код, команды для проверки, какие документы посмотреть и как убедиться в результате. Дробите задачи на маленькие шаги. DRY. YAGNI. TDD. Частые коммиты.

Считайте, что исполнитель умеет программировать, но почти ничего не знает ни об используемом наборе инструментов, ни о предметной области. Считайте также, что он не слишком хорошо проектирует тесты.

**Основной принцип:** хороший план делает реализацию очевидной. Если человеку приходится что-то додумывать, план неполный.

## Когда использовать

**Всегда используйте перед:**
- Реализацией многошаговых функций
- Декомпозицией сложных требований
- Делегированием субагентам через `subagent-driven-development`

**Не пропускайте, если:**
- Кажется, что задача простая (предположения приводят к ошибкам)
- Вы планируете реализовывать всё сами (вашему будущему себе тоже нужна опора)
- Вы работаете в одиночку (документация всё равно важна)

## Размер задач

**Каждая задача = 2-5 минут сосредоточенной работы.**

Каждый шаг — это одно действие:
- "Написать падающий тест" — шаг
- "Запустить и убедиться, что он падает" — шаг
- "Реализовать минимальный код, чтобы тест прошёл" — шаг
- "Запустить тесты и убедиться, что они проходят" — шаг
- "Закоммитить" — шаг

**Слишком крупно:**
```markdown
### Задача 1: построить систему аутентификации
[50 lines of code across 5 files]
```

**Правильный размер:**
```markdown
### Задача 1: создать модель User с полем email
[10 lines, 1 file]

### Задача 2: добавить поле password hash в User
[8 lines, 1 file]

### Задача 3: создать утилиту хеширования паролей
[15 lines, 1 file]
```

### Структура задачи

Каждая задача оформляется так:

````markdown
### Задача N: [Название задачи]

**Цель:** Что эта задача должна сделать (одно предложение)

**Файлы:**
- Создать: `exact/path/to/new_file.py`
- Изменить: `exact/path/to/existing.py:45-67` (номера строк, если известны)
- Тест: `tests/path/to/test_file.py`

**Step 1: Написать падающий тест**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

**Step 2: Запустить тест, чтобы убедиться в падении**

Запустить: `pytest tests/path/test.py::test_specific_behavior -v`
Ожидается: FAIL — "функция не определена"

**Step 3: Написать минимальную реализацию**

```python
def function(input):
    return expected
```

**Step 4: Запустить тест, чтобы убедиться в прохождении**

Запустить: `pytest tests/path/test.py::test_specific_behavior -v`
Ожидается: PASS

**Step 5: Закоммитить**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

Стройте задачи по порядку:

## Принципы

### DRY (Don't Repeat Yourself)

**Плохо:** копировать валидацию в 3 местах
**Хорошо:** вынести функцию валидации и использовать везде

### YAGNI (You Aren't Gonna Need It)

**Плохо:** добавлять "гибкость" на будущее
**Хорошо:** реализовывать только то, что нужно сейчас

```python
# Плохо — нарушение YAGNI
class User:
    def __init__(self, name, email):
        self.name = name
        self.email = email
        self.preferences = {}  # Пока не нужно!
        self.metadata = {}     # Пока не нужно!

# Хорошо — YAGNI
class User:
    def __init__(self, name, email):
        self.name = name
        self.email = email
```

### TDD (разработка через тестирование)

Каждая задача, которая создаёт код, должна включать полный цикл TDD:
1. Написать падающий тест
2. Запустить и убедиться в падении
3. Написать минимальный код
4. Запустить и убедиться в прохождении

Подробности см. в навыке `test-driven-development`.

### Частые коммиты

Делайте коммит после каждой задачи:
```bash
git add [files]
git commit -m "type: description"
```

## Типичные ошибки

### Расплывчатые задачи

**Плохо:** "Добавить аутентификацию"
**Хорошо:** "Создать модель User с полями email и password_hash"

### Неполный код

**Плохо:** "Шаг 1: добавить функцию валидации"
**Хорошо:** "Шаг 1: добавить функцию валидации" и сразу после этого полный код функции

### Нет проверки

**Плохо:** "Шаг 3: проверить, что всё работает"
**Хорошо:** "Шаг 3: запустить `pytest tests/test_auth.py -v`, ожидается: 3 passed"

### Нет путей к файлам

**Плохо:** "Создать файл модели"
**Хорошо:** "Создать: `src/models/user.py`"

## Передача к выполнению

После сохранения плана предложите способ его выполнения:

**"План готов и сохранён. Готовы выполнить его через `subagent-driven-development` — я буду запускать нового субагента на каждую задачу с двухэтапной проверкой (сначала соответствие спецификации, затем качество кода). Продолжать?"**

При выполнении используйте навык `subagent-driven-development`:
- свежий `delegate_task` на каждую задачу с полным контекстом
- проверка соответствия спецификации после каждой задачи
- проверка качества кода после прохождения спецификации
- переход к следующей задаче только после одобрения обеих проверок
