# p5.js Production Pipeline
## Когда использовать

Используй, когда пользователи запрашивают : p5.js‑скетчи, креативный кодинг, генеративное искусство, интерактивные визуализации, анимации на canvas, визуальное искусство в браузере, визуализацию данных, эффекты шейдеров или любой проект на p5.js.
## Что внутри

Производственный конвейер для интерактивного и генеративного визуального искусства с использованием p5.js. Создаёт скетчи в браузере, генеративное искусство, визуализацию данных, интерактивные эксперименты, 3D‑сцены, аудио‑реактивные визуализации и моушн‑графику — экспортируемые в HTML, PNG, GIF, MP4 или SVG. Охватывает: 2D/3D‑рендеринг, шум и системы частиц, поля потоков, шейдеры (GLSL), манипуляцию пикселями, кинетическую типографику, сцены WebGL, аудио‑анализ, взаимодействие мышью/клавиатурой и безголовый экспорт в высоком разрешении.
## Творческий стандарт

Это визуальное искусство, отрисованное в браузере. Холст — это среда; алгоритм — это кисть.

**Прежде чем написать единую строку кода**, сформулируй творческую концепцию. Что передаёт эта работа? Что заставит зрителя остановить прокрутку? Чем она отличается от примера учебника по коду? Запрос пользователя — это отправная точка — интерпретируй его с творческим стремлением.

**Первый рендеринг — неподлежащий компромиссу.** Вывод должен сразу же поражать визуально. Если он выглядит как упражнение из учебника p5.js, стандартная конфигурация или «AI‑сгенерированный креативный код», — это ошибка. Пересмотри подход перед публикацией.

**Выходи за пределы референсного словаря.** Шумовые функции, системы частиц, цветовые палитры и шейдерные эффекты в примерах — это лишь стартовый словарь. Для каждого проекта комбинируй, накладывай и изобретай. Каталог — это палитра красок — ты пишешь картину.

**Будь проактивно креативным.** Если пользователь просит «систему частиц», предоставь систему частиц с возникающим стайным поведением, шлейфами‑призраками, глубинным туманом, меняющим палитру, и фоновым шумовым полем, которое «дышит». Добавь как минимум одну визуальную деталь, о которой пользователь не просил, но которая ему понравится.

**Плотно, слоисто, продумано.** Каждый кадр должен вознаграждать зрителя. Никаких плоских белых фонов. Всегда композиционная иерархия. Всегда осознанный цвет. Всегда микродетали, которые появляются только при близком рассмотрении.

**Единство эстетики важнее количества функций.** Все элементы должны служить единому визуальному языку — общая цветовая температура, согласованный словарь толщины штрихов, гармоничные скорости движения. Эскиз с десятью несвязанными эффектами хуже, чем один с тремя, которые сочетаются между собой.

### Примечание к версии

**p5.js 1.x** (1.11.3) — вариант по умолчанию: стабильный, хорошо документированный, самая широкая совместимость библиотек. Используй его, если только проект не требует возможностей 2.x.

**p5.js 2.x** (2.2+) добавляет: `async setup()` вместо `preload()`, цветовые режимы OKLCH/OKLAB, `splineVertex()`, API шейдеров `.modify()`, переменные шрифты, `textToContours()`, события указателя. Требуется для p5.brush. См. `references/core-api.md` § p5.js 2.0.
## Конвейер

Каждый проект следует одному и тому же 6‑этапному пути:

```
CONCEPT → DESIGN → CODE → PREVIEW → EXPORT → VERIFY
```

1. **CONCEPT** — Формулирование творческого видения: настроение, цветовой мир, словарь движений, что делает его уникальным
2. **DESIGN** — Выбор режима, размера холста, модели взаимодействия, цветовой системы, формата экспорта. Привязка концепции к техническим решениям
3. **CODE** — Написание единственного HTML‑файла с встроенным p5.js. Структура: глобальные переменные → `preload()` → `setup()` → `draw()` → вспомогательные функции → классы → обработчики событий
4. **PREVIEW** — Открытие в браузере, проверка визуального качества. Тестирование на целевом разрешении. Проверка производительности
5. **EXPORT** — Захват вывода: `saveCanvas()` для PNG, `saveGif()` для GIF, `saveFrames()` + ffmpeg для MP4, Puppeteer для безголового пакетного режима
6. **VERIFY** — Соответствует ли вывод концепции? Выглядит ли он визуально впечатляюще при предполагаемом размере отображения? Стоит ли оформить его в рамку?

### Правила вариаций per‑project

Никогда не используй конфигурации по умолчанию. Для каждого проекта:
- **Собственная цветовая палитра** — никогда не `fill(255, 0, 0)`. Всегда разработанная палитра из 3‑7 цветов.
- **Собственная лексика толщины линий** — тонкие акценты (0.5), средняя структура (1‑2), жирное выделение (3‑5).
- **Обработка фона** — никогда не `background(0)` или `background(255)`. Всегда текстурный, градиентный или многослойный.
- **Разнообразие движения** — разные скорости для разных элементов. Основные 1×, второстепенные 0.3×, фоновые 0.1×.
- **Как минимум один вымышленный элемент** — пользовательское поведение частиц, оригинальное применение шума, уникальная реакция взаимодействия.

### Проектно‑специфическое изобретение

Для каждого проекта придумай хотя бы один из:
- Собственная цветовая палитра, соответствующая настроению (не пресет).
- Новая комбинация шумовых полей (например, curl noise + domain warp + feedback).
- Уникальное поведение частиц (особые силы, кастомные следы, особый спаун‑инг).
- Механика взаимодействия, которой пользователь не просил, но которая поднимает работу.
- Техника композиции, создающая визуальную иерархию.

### Философия проектирования параметров

Параметры должны возникать из алгоритма, а не из общего меню. Спроси: «Какие свойства *этой* системы должны быть настраиваемыми?»

**Хорошие параметры** раскрывают характер алгоритма:
- **Количества** — сколько частиц, ветвей, ячеек (контролируют плотность).
- **Масштабы** — частота шума, размер элементов, интервал (контролируют текстуру).
- **Скорости** — скорость, темп роста, затухание (контролируют энергию).
- **Пороги** — когда меняется поведение? (контролируют драматизм).
- **Соотношения** — пропорции, баланс сил (контролируют гармонию).

**Плохие параметры** — универсальные элементы, не связанные с алгоритмом:
- «color1», «color2», «size» — бессмысленны без контекста.
- Переключатели для несвязанных эффектов.
- Параметры, меняющие только внешний вид, а не поведение.

Каждый параметр должен менять то, как алгоритм *думает*, а не только то, как он *выглядит*. Параметр «turbulence», меняющий количество октав шума, хорош. Ползунок «particle size», меняющий только радиус в `ellipse()`, поверхностен.
## Workflow

### Шаг 1: Креативное видение

Перед тем как писать код, сформулируй:

- **Настроение / атмосфера**: Что должен чувствовать зритель? Созерцательное? Заряженное? Тревожное? Игривое?
- **Визуальная история**: Что происходит со временем (или при взаимодействии)? Строится? Разрушается? Трансформируется? Осциллирует?
- **Цветовой мир**: Тёплый/холодный? Монохромный? Дополнительные? Какой доминирующий оттенок? Какой акцент?
- **Язык форм**: Органические изгибы? Острые геометрические формы? Точки? Линии? Смешанные?
- **Словарь движений**: Медленное скольжение? Взрывной всплеск? Дыхательный пульс? Механическая точность?
- **Что делает ЭТО особенным**: Что единственное делает этот скетч уникальным?

Сопоставь запрос пользователя с эстетическими выборами. «Расслабляющий генеративный фон» требует совсем другого, чем «глитч‑визуализация данных».

### Шаг 2: Технический дизайн

- **Режим** — какой из 7 режимов из таблицы выше
- **Размер холста** — альбомный 1920x1080, портретный 1080x1920, квадратный 1080x1080 или адаптивный `windowWidth/windowHeight`
- **Рендерер** — `P2D` (по умолчанию) или `WEBGL` (для 3D, шейдеров, продвинутых режимов смешивания)
- **Частота кадров** — 60fps (интерактивно), 30fps (фон) или `noLoop()` (статическое генеративное)

### Шаг 3: Код скетча

Для **интерактивного генеративного искусства** (исследование seed‑ов, настройка параметров): начни с `templates/viewer.html`. - **Соответствует ли видению?** Сравни результат с креативной концепцией. Если выглядит шаблонно, вернись к Шагу 1
- **Проверка разрешения**: Является ли изображение чётким при целевом размере отображения? Нет ли артефактов aliasing?
- **Проверка производительности**: Поддерживает ли оно 60fps в браузере? (Минимум 30fps для анимаций)
- **Проверка цветов**: Хорошо ли сочетаются цвета? Протестируй на светлых и тёмных мониторах
- **Граничные случаи**: Что происходит у краёв холста? При изменении размера? После 10‑минутного непрерывного запуска?
## Критические замечания по реализации

### Производительность — сначала отключите FES

Friendly Error System (FES) добавляет до 10‑кратного оверхеда. Отключайте её в каждом продакшн‑скетче:

```javascript
p5.disableFriendlyErrors = true;  // BEFORE setup()

function setup() {
  pixelDensity(1);  // prevent 2x-4x overdraw on retina
  createCanvas(1920, 1080);
}
```

В горячих циклах (частицы, пиксельные операции) используйте `Math.*` вместо обёрток p5 — измеримо быстрее:

```javascript
// In draw() or update() hot paths:
let a = Math.sin(t);          // not sin(t)
let r = Math.sqrt(dx*dx+dy*dy); // not dist() — or better: skip sqrt, compare magSq
let v = Math.random();        // not random() — when seed not needed
let m = Math.min(a, b);       // not min(a, b)
```

Никогда не вызывайте `console.log()` внутри `draw()`. Никогда не манипулируйте DOM в `draw()`. См. `references/troubleshooting.md` § Performance.

### Посев случайности — всегда

Каждый генеративный скетч должен быть воспроизводим. Один и тот же seed — один и тот же результат.

```javascript
function setup() {
  randomSeed(CONFIG.seed);
  noiseSeed(CONFIG.seed);
  // All random() and noise() calls now deterministic
}
```

Не используйте `Math.random()` для генеративного контента — только для производительно‑критичного невизуального кода. Для визуальных элементов всегда используйте `random()`. Если нужен случайный seed: `CONFIG.seed = floor(random(99999))`.

### Поддержка платформ генеративного искусства (fxhash / Art Blocks)

Для платформ генеративного искусства замените PRNG p5 на детерминированный random платформы:

```javascript
// fxhash convention
const SEED = $fx.hash;              // unique per mint
const rng = $fx.rand;               // deterministic PRNG
$fx.features({ palette: 'warm', complexity: 'high' });

// In setup():
randomSeed(SEED);   // for p5's noise()
noiseSeed(SEED);

// Replace random() with rng() for platform determinism
let x = rng() * width;  // instead of random(width)
```

См. `references/export-pipeline.md` § Platform Export.

### Цветовой режим — используйте HSB

HSB (Hue, Saturation, Brightness) значительно проще в работе, чем RGB, для генеративного искусства:

```javascript
colorMode(HSB, 360, 100, 100, 100);
// Now: fill(hue, sat, bri, alpha)
// Rotate hue: fill((baseHue + offset) % 360, 80, 90)
// Desaturate: fill(hue, sat * 0.3, bri)
// Darken: fill(hue, sat, bri * 0.5)
```

Никогда не хардкодьте «сырые» значения RGB. Определите объект палитры, генерируйте вариации процедурно. См. `references/color-systems.md`.

### Шум — многократные октавы, а не «сырой»

«Сырой» `noise(x, y)` выглядит как гладкие пятна. Наслаивайте октавы для естественной текстуры:

```javascript
function fbm(x, y, octaves = 4) {
  let val = 0, amp = 1, freq = 1, sum = 0;
  for (let i = 0; i < octaves; i++) {
    val += noise(x * freq, y * freq) * amp;
    sum += amp;
    amp *= 0.5;
    freq *= 2;
  }
  return val / sum;
}
```

Для плавных органических форм используйте **domain warping**: подавайте вывод шума обратно как координаты входа шума. См. `references/visual-effects.md`.

### `createGraphics()` для слоёв — не опционально

Однопроходный рендеринг выглядит плоско. Используйте буферы вне экрана для композиции:

```javascript
let bgLayer, fgLayer, trailLayer;
function setup() {
  createCanvas(1920, 1080);
  bgLayer = createGraphics(width, height);
  fgLayer = createGraphics(width, height);
  trailLayer = createGraphics(width, height);
}
function draw() {
  renderBackground(bgLayer);
  renderTrails(trailLayer);   // persistent, fading
  renderForeground(fgLayer);  // cleared each frame
  image(bgLayer, 0, 0);
  image(trailLayer, 0, 0);
  image(fgLayer, 0, 0);
}
```

### Производительность — векторизуйте, где возможно

Вызовы `draw` в p5.js дорогие. Для тысяч частиц:

```javascript
// SLOW: individual shapes
for (let p of particles) {
  ellipse(p.x, p.y, p.size);
}

// FAST: single shape with beginShape()
beginShape(POINTS);
for (let p of particles) {
  vertex(p.x, p.y);
}
endShape();

// FASTEST: pixel buffer for massive counts
loadPixels();
for (let p of particles) {
  let idx = 4 * (floor(p.y) * width + floor(p.x));
  pixels[idx] = r; pixels[idx+1] = g; pixels[idx+2] = b; pixels[idx+3] = 255;
}
updatePixels();
```

См. `references/troubleshooting.md` § Performance.

### Instance Mode для нескольких скетчей

Глобальный режим загрязняет `window`. Для продакшна используйте instance mode:

```javascript
const sketch = (p) => {
  p.setup = function() {
    p.createCanvas(800, 800);
  };
  p.draw = function() {
    p.background(0);
    p.ellipse(p.mouseX, p.mouseY, 50);
  };
};
new p5(sketch, 'canvas-container');
```

Требуется при встраивании нескольких скетчей на одну страницу или интеграции с фреймворками.

### Подводные камни WebGL Mode

- `createCanvas(w, h, WEBGL)` — начало координат в центре, а не в левом‑верхнем углу
- Ось Y инвертирована (положительный Y вверх в WEBGL, вниз в P2D)
- `translate(-width/2, -height/2)` — чтобы получить координаты, похожие на P2D
- `push()`/`pop()` вокруг каждой трансформации — стек матриц переполняется без предупреждения
- `texture()` перед `rect()`/`plane()` — не после
- Пользовательские шейдеры: `createShader(vert, frag)` — тестируйте в разных браузерах

### Экспорт — соглашение о привязках клавиш

Каждый скетч должен включать эти привязки в `keyPressed()`:

```javascript
function keyPressed() {
  if (key === 's' || key === 'S') saveCanvas('output', 'png');
  if (key === 'g' || key === 'G') saveGif('output', 5);
  if (key === 'r' || key === 'R') { randomSeed(millis()); noiseSeed(millis()); }
  if (key === ' ') CONFIG.paused = !CONFIG.paused;
}
```

### Безголовый экспорт видео — используйте `noLoop()`

Для безголового рендеринга через Puppeteer скетч **должен** вызывать `noLoop()` в `setup`. Без этого цикл `draw` p5 будет работать свободно, а скриншоты будут медленными — скетч «обгонит» их, и вы получите пропущенные/дублирующиеся кадры.

```javascript
function setup() {
  createCanvas(1920, 1080);
  pixelDensity(1);
  noLoop();                    // capture script controls frame advance
  window._p5Ready = true;      // signal readiness to capture script
}
```

Скрипт `scripts/export-frames.js` обнаруживает `_p5Ready` и вызывает `redraw()` один раз на каждый захват, обеспечивая точное соответствие 1:1 кадр‑к‑кадру. См. `references/export-pipeline.md` § Deterministic Capture.

Для многосценарных видео используйте архитектуру per‑clip: один HTML‑файл на сцену, рендерите независимо, склейте с помощью `ffmpeg -f concat`. См. `references/export-pipeline.md` § Per-Clip Architecture.

### Рабочий процесс агента

При создании скетчей p5.js:

1. **Напиши HTML‑файл** — один самодостаточный файл, весь код инлайн
2. **Открой в браузере** — `open sketch.html` (macOS) или `xdg-open sketch.html` (Linux)
3. **Локальные ресурсы** (шрифты, изображения) требуют сервера: `python3 -m http.server 8080` в каталоге проекта, затем открой `http://localhost:8080/sketch.html`
4. **Экспорт PNG/GIF** — добавь шорткаты в `keyPressed()` как показано выше, укажи пользователю, какую клавишу нажимать
5. **Безголовый экспорт** — `node scripts/export-frames.js sketch.html --frames 300` для автоматического захвата кадров (скетч должен использовать `noLoop()` + `_p5Ready`)
6. **Рендеринг MP4** — `bash scripts/render.sh sketch.html output.mp4 --duration 30`
7. **Итеративная доработка** — редактируй HTML‑файл, пользователь обновляет страницу, чтобы увидеть изменения
8. **Загрузка справок по требованию** — используй `skill_view(name="p5js", file_path="references/...")` для загрузки конкретных файлов справки по мере необходимости во время реализации

## Creative Divergence (использовать только по запросу пользователя на экспериментальный/креативный/уникальный вывод)

Если пользователь просит креативный, экспериментальный, удивительный или нетрадиционный результат, выбери стратегию, которая лучше всего подходит, и обоснуй её шаги **до** генерации кода.

- **Conceptual Blending** — когда пользователь называет два объекта для комбинирования или хочет гибридную эстетику
- **SCAMPER** — когда пользователь хочет изменить известный паттерн генеративного искусства
- **Distance Association** — когда пользователь задаёт единую концепцию и хочет её исследовать («создай что‑то о времени»)

### Conceptual Blending
1. Назови две отдельные визуальные системы (например, физика частиц + рукописный текст)
2. Сопоставь их элементы (частицы = капли чернил, силы = давление пера, поля = буквенные формы)
3. Смешай избирательно — оставь только те соответствия, которые дают интересные возникающие визуалы
4. Реализуй смесь как единую систему, а не как две системы рядом

### SCAMPER Transformation
Возьми известный генеративный паттерн (поле потоков, система частиц, L‑system, клеточный автомат) и систематически трансформируй его:
- **Substitute** — замени круги текстовыми символами, линии — градиентами
- **Combine** — объединить два паттерна (поле потоков + вороной)
- **Adapt** — применить 2‑D паттерн к 3‑D проекции
- **Modify** — увеличить масштаб, искривить пространство координат
- **Purpose** — использовать физическую симуляцию для типографии, алгоритм сортировки — для цвета
- **Eliminate** — убрать сетку, убрать цвет, убрать симметрию
- **Reverse** — запустить симуляцию в обратном направлении, инвертировать пространство параметров

### Distance Association
1. Зафиксируй концепцию пользователя (например, «одиночество»)
2. Сгенерируй ассоциации на трёх уровнях удалённости:
   - **Близко** (очевидно): пустая комната, одинокая фигурка, тишина
   - **Средне** (интересно): одна рыба в стае, плывущая в обратном направлении; телефон без уведомлений; зазор между вагонами метро
   - **Далеко** (абстрактно): простые числа, асимптотические кривые, цвет 3 утра
3. Разработай ассоциации средней удалённости — они достаточно конкретны для визуализации, но достаточно неожиданны, чтобы быть интересными.
