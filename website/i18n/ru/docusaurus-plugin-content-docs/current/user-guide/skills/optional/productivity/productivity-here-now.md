# here.now

here.now позволяет агентам публиковать веб‑сайты и хранить приватные файлы в облачных Drives.

Используй here.now для двух задач:

- **Sites**: публикация веб‑сайтов и файлов по `{slug}.here.now`.
- **Drives**: хранение приватных файлов агента в облачных папках.

## Требования

- Обязательные бинарные файлы: `curl`, `file`, `jq`
- Необязательная переменная окружения: `$HERENOW_API_KEY`
- Необязательная переменная токена Drive: `$HERENOW_DRIVE_TOKEN`
- Необязательный файл учётных данных: `~/.herenow/credentials`
- Пути вспомогательных скриптов навыка:
  - `${HERMES_SKILL_DIR}/scripts/publish.sh` для публикации сайтов
  - `${HERMES_SKILL_DIR}/scripts/drive.sh` для приватного хранения в Drive

## Создание сайта

```bash
PUBLISH="${HERMES_SKILL_DIR}/scripts/publish.sh"
bash "$PUBLISH" {file-or-dir} --client hermes
```

Выводит живой URL (например `https://bright-canvas-a7k2.here.now/`).

Под капотом это трёхшаговый процесс: create/update → upload files → finalize. Сайт не становится живым, пока не завершится финализация.

Без API‑ключа создаётся **анонимный сайт**, который истекает через 24 часа.
С сохранённым API‑ключом сайт становится постоянным.

**Структура файлов:** Для HTML‑сайтов размести `index.html` в корне директории, которую публикуешь, а не внутри подпапки. Содержимое директории становится корнем сайта. Например, публикуй `my-site/`, где есть `my-site/index.html` — не публикуй родительскую папку, содержащую `my-site/`.

Можно также публиковать «сырые» файлы без HTML. Одинарные файлы получают богатый авто‑просмотрщик (изображения, PDF, видео, аудио). При публикации нескольких файлов генерируется авто‑созданный список каталогов с навигацией по папкам и галереей изображений.

## Обновление существующего сайта

```bash
PUBLISH="${HERMES_SKILL_DIR}/scripts/publish.sh"
bash "$PUBLISH" {file-or-dir} --slug {slug} --client hermes
```

Скрипт автоматически загружает `claimToken` из `.herenow/state.json` при обновлении анонимных сайтов. Передай `--claim-token {token}`, чтобы переопределить.

Аутентифицированные обновления требуют сохранённого API‑ключа.

## Использование Drive

Используй Drive, когда пользователю нужно приватное облачное хранилище для файлов агента: документы, контекст, память, планы, ресурсы, медиа, исследования, код и всё, что должно сохраняться без публикации как веб‑сайт.

У каждой учётной записи, вошедшей в систему, есть Drive по умолчанию под названием `My Drive`.

```bash
DRIVE="${HERMES_SKILL_DIR}/scripts/drive.sh"
bash "$DRIVE" default
bash "$DRIVE" ls "My Drive"
bash "$DRIVE" put "My Drive" notes/today.md --from ./notes/today.md
bash "$DRIVE" cat "My Drive" notes/today.md
bash "$DRIVE" share "My Drive" --perms write --prefix notes/ --ttl 7d
```

Используй токены Drive с ограниченным диапазоном для передачи агент‑агенту. Если получаешь блок `herenow_drive`, используй его `token` как `Authorization: Bearer <token>` к `api_base`, учитывай `pathPrefix`, если он присутствует, и сохраняй ETag при записях. `pathPrefix` со значением `null` означает доступ ко всему Drive. Если навык доступен, предпочитай `drive.sh`; иначе вызывай перечисленные операции API напрямую.

## Хранение API‑ключа

Скрипт публикации читает API‑ключ из следующих источников (первый совпавший выигрывает):

1. флаг `--api-key {key}` (только для CI/скриптов — избегай в интерактивном использовании)
2. переменная окружения `$HERENOW_API_KEY`
3. файл `~/.herenow/credentials` (рекомендовано для агентов)

Чтобы сохранить ключ, запиши его в файл учётных данных:

```bash
mkdir -p ~/.herenow && echo "{API_KEY}" > ~/.herenow/credentials && chmod 600 ~/.herenow/credentials
```

**ВАЖНО**: После получения API‑ключа сразу сохраняй его — запусти команду выше сам. Не проси пользователя делать это вручную. Избегай передачи ключа через флаги CLI (например `--api-key`) в интерактивных сессиях; файл учётных данных — предпочтительный способ хранения.

Никогда не коммить учётные данные или локальные файлы состояния (`~/.herenow/credentials`, `.herenow/state.json`) в систему контроля версий.

## Получение API‑ключа

Чтобы перейти от анонимных (24 ч) к постоянным сайтам:

1. Спроси у пользователя его адрес электронной почты.
2. Запроси одноразовый код входа:

```bash
curl -sS https://here.now/api/auth/agent/request-code \
  -H "content-type: application/json" \
  -d '{"email": "user@example.com"}'
```

3. Скажи пользователю: «Проверьте почту, вам должно прийти письмо с кодом входа от here.now, вставьте его сюда».
4. Проверь код и получи API‑ключ:

```bash
curl -sS https://here.now/api/auth/agent/verify-code \
  -H "content-type: application/json" \
  -d '{"email":"user@example.com","code":"ABCD-2345"}'
```

5. Сохрани полученный `apiKey` сам (не проси пользователя делать это):

```bash
mkdir -p ~/.herenow && echo "{API_KEY}" > ~/.herenow/credentials && chmod 600 ~/.herenow/credentials
```

## Файл состояния

После каждой операции создания/обновления сайта скрипт пишет в `.herenow/state.json` в рабочей директории:

```json
{
  "publishes": {
    "bright-canvas-a7k2": {
      "siteUrl": "https://bright-canvas-a7k2.here.now/",
      "claimToken": "abc123",
      "claimUrl": "https://here.now/claim?slug=bright-canvas-a7k2&token=abc123",
      "expiresAt": "2026-02-18T01:00:00.000Z"
    }
  }
}
```

Перед созданием или обновлением сайтов можешь проверить этот файл, чтобы найти ранее использованные slug. Рассматривай `.herenow/state.json` только как внутренний кэш.
Никогда не представляй этот локальный путь как URL и не используй его как источник правды для режима аутентификации, срока действия или claim URL.

## Что говорить пользователю

Для опубликованных сайтов:

- Всегда делись `siteUrl` из текущего запуска скрипта.
- Читай и учитывай строки `publish_result.*` из stderr скрипта, чтобы определить режим аутентификации.
- Когда `publish_result.auth_mode=authenticated`: скажи пользователю, что сайт **постоянный** и сохранён в его аккаунте. claim URL не нужен.
- Когда `publish_result.auth_mode=anonymous`: скажи пользователю, что сайт **истекает через 24 часа**. Поделись claim URL (если `publish_result.claim_url` не пуст и начинается с `https://`), чтобы он мог сохранить его навсегда. Предупреди, что токены запроса выдаются только один раз и не могут быть восстановлены.
- Никогда не проси пользователя проверять `.herenow/state.json` для получения claim URL или статуса аутентификации.

Для Drive:

- Не описывай файлы Drive как публичные URL.
- Сообщай пользователю, что содержимое Drive приватно, если только не поделено токеном с ограниченным диапазоном.
- При предоставлении доступа другому агенту предпочитай токен с узким `pathPrefix` и коротким TTL.

## Параметры publish.sh

| Flag                   | Description                                  |
| ---------------------- | -------------------------------------------- |
| `--slug {slug}`        | Обновить существующий сайт вместо создания |
| `--claim-token {token}`| Переопределить токен запроса для анонимных обновлений |
| `--title {text}`       | Заголовок просмотрщика (для не‑HTML сайтов) |
| `--description {text}` | Описание просмотрщика |
| `--ttl {seconds}`      | Установить срок действия (только для аутентифицированных) |
| `--client {name}`      | Имя агента для атрибуции (например `hermes`) |
| `--base-url {url}`     | Базовый URL API (по умолчанию `https://here.now`) |
| `--allow-nonherenow-base-url` | Разрешить отправку аутентификации на нестандартный `--base-url` |
| `--api-key {key}`      | Переопределить API‑ключ (предпочтительно файл учётных данных) |
| `--spa`                | Включить SPA‑маршрутизацию (отдавать index.html для неизвестных путей) |
| `--forkable`           | Разрешить другим форкать этот сайт |
