# Миграция OpenClaw → Hermes

Используй этот **skill**, когда пользователь хочет перенести свою настройку OpenClaw в Hermes Agent с минимальной ручной очисткой.
## Команда CLI

Для быстрой, неинтерактивной миграции используй встроенную команду CLI:

```bash
hermes claw migrate              # Full interactive migration
hermes claw migrate --dry-run    # Preview what would be migrated
hermes claw migrate --preset user-data   # Migrate without secrets
hermes claw migrate --overwrite  # Overwrite existing conflicts
hermes claw migrate --source /custom/path/.openclaw  # Custom source
```

Команда CLI запускает тот же скрипт миграции, который описан ниже. Используй этот skill (через агента), когда нужна интерактивная, управляемая миграция с предварительным просмотром dry‑run и разрешением конфликтов по каждому элементу.

**Первичная настройка:** Мастер `hermes setup` автоматически обнаруживает `~/.openclaw` и предлагает миграцию до начала конфигурации.
## Что делает этот навык

Он использует `scripts/openclaw_to_hermes.py` для:

- импорта `SOUL.md` в домашний каталог Hermes как `SOUL.md`;
- преобразования файлов OpenClaw `MEMORY.md` и `USER.md` в записи памяти Hermes;
- объединения шаблонов одобрения команд OpenClaw в `command_allowlist` Hermes;
- миграции совместимых с Hermes настроек обмена сообщениями, таких как `TELEGRAM_ALLOWED_USERS` и `MESSAGING_CWD`;
- копирования навыков OpenClaw в `~/.hermes/skills/openclaw-imports/`;
- по желанию — копирования файла инструкций рабочего пространства OpenClaw в выбранное рабочее пространство Hermes;
- зеркального копирования совместимых ресурсов рабочего пространства, таких как `workspace/tts/`, в `~/.hermes/tts/`;
- архивирования несекретных документов, не имеющих прямого назначения в Hermes;
- создания структурированного отчёта со списком перенесённых элементов, конфликтов, пропущенных элементов и причин.

## Протокол взаимодействия с пользователем

Hermes CLI поддерживает инструмент `clarify` для интерактивных запросов, но он ограничен:

- один выбор за раз
- до 4 заранее определённых вариантов
- автоматической свободной опцией `Other`

Он **не** поддерживает настоящие множественные чекбоксы в одном запросе.

Для каждого вызова `clarify`:

- всегда включай непустой `question`
- включай `choices` только для реальных выбираемых запросов
- ограничивай `choices` 2‑4 простыми строковыми вариантами
- никогда не выводи заполнители или усечённые варианты, такие как `...`
- никогда не добавляй лишние пробелы или стили к вариантам
- никогда не включай фиктивные поля формы в вопрос, например `enter directory here`, пустые строки для заполнения или подчёркивания вроде `_____`
- для открытых вопросов о пути запрашивай только простое предложение; пользователь вводит ответ в обычном CLI‑prompt под панелью

Если вызов `clarify` возвращает ошибку, изучи текст ошибки, исправь payload и попробуй ещё раз один раз с корректным `question` и чистыми вариантами.

Когда `clarify` доступен и сухой прогон выявляет требующее решения действие пользователя, твоё **следующее действие должно быть вызовом инструмента `clarify`**.
Не заканчивай ход обычным сообщением ассистента, например:

- «Позволь мне показать варианты»
- «Что ты хочешь сделать?»
- «Вот варианты»

Если требуется решение пользователя, собирай его через `clarify` прежде, чем писать дальнейший текст.
Если остаётся несколько нерешённых решений, не вставляй пояснительное сообщение ассистента между ними. После получения одного ответа `clarify` твоё следующее действие обычно будет следующий требуемый вызов `clarify`.

Относись к `workspace-agents` как к нерешённому решению, когда сухой прогон сообщает:

- `kind="workspace-agents"`
- `status="skipped"`
- `reason` содержит `No workspace target was provided`

В этом случае необходимо спросить об инструкциях рабочего пространства перед выполнением. Не игнорируй это как решение пропустить.

Из‑за этого ограничения используй упрощённый поток принятия решений:

1. Для конфликтов `SOUL.md` используй `clarify` с вариантами, например:
   - `keep existing`
   - `overwrite with backup`
   - `review first`
2. Если сухой прогон показывает один или несколько элементов `kind="skill"` со `status="conflict"`, используй `clarify` с вариантами, например:
   - `keep existing skills`
   - `overwrite conflicting skills with backup`
   - `import conflicting skills under renamed folders`
3. Для инструкций рабочего пространства используй `clarify` с вариантами, например:
   - `skip workspace instructions`
   - `copy to a workspace path`
   - `decide later`
4. Если пользователь выбирает копировать инструкции рабочего пространства, задай последующий открытый вопрос `clarify`, запрашивая **абсолютный путь**.
5. Если пользователь выбирает `skip workspace instructions` или `decide later`, продолжай без `--workspace-target`.
6. Для режима миграции используй `clarify` с этими 3 вариантами:
   - `user-data only`
   - `full compatible migration`
   - `cancel`
7. `user-data only` означает: мигрировать пользовательские данные и совместимую конфигурацию, но **не** импортировать разрешённые секреты.
8. `full compatible migration` означает: мигрировать те же совместимые пользовательские данные плюс разрешённые секреты, если они присутствуют.
9. Если `clarify` недоступен, задай тот же вопрос обычным текстом, но всё равно ограничь ответ вариантами `user-data only`, `full compatible migration` или `cancel`.

**Ворота выполнения**:

- Не выполняй, пока остаётся нерешённый пропуск `workspace-agents` из‑за `No workspace target was provided`.
- Единственные допустимые способы его разрешить:
  - пользователь явно выбирает `skip workspace instructions`
  - пользователь явно выбирает `decide later`
  - пользователь предоставляет путь к рабочему пространству после выбора `copy to a workspace path`
- Отсутствие целевого рабочего пространства в сухом прогоне само по себе не даёт разрешения на выполнение.
- Не выполняй, пока любое требуемое решение `clarify` остаётся нерешённым.

Используй эти точные формы payload для `clarify` как шаблон по умолчанию:

- `{"question":"Your existing SOUL.md conflicts with the imported one. What should I do?","choices":["keep existing","overwrite with backup","review first"]}`
- `{"question":"One or more imported OpenClaw skills already exist in Hermes. How should I handle those skill conflicts?","choices":["keep existing skills","overwrite conflicting skills with backup","import conflicting skills under renamed folders"]}`
- `{"question":"Choose migration mode: migrate only user data, or run the full compatible migration including allowlisted secrets?","choices":["user-data only","full compatible migration","cancel"]}`
- `{"question":"Do you want to copy the OpenClaw workspace instructions file into a Hermes workspace?","choices":["skip workspace instructions","copy to a workspace path","decide later"]}`
- `{"question":"Please provide an absolute path where the workspace instructions should be copied."}`
**План команды**

1. **SOUL.md** – если выбран вариант `keep existing`, флаг `--overwrite` **не** добавляется.
2. **Перезапись с резервной копией** – при выборе `overwrite with backup` добавляем флаг `--overwrite`.
3. **Обзор перед выполнением** – если выбран `review first`, процесс останавливается до выполнения и предлагается просмотреть соответствующие файлы.
4. **Конфликтующие навыки** –
   - При выборе `keep existing skills` добавляем `--skill-conflict skip`.
   - При выборе `overwrite conflicting skills with backup` добавляем `--skill-conflict overwrite`.
   - При выборе `import conflicting skills under renamed folders` добавляем `--skill-conflict rename`.
5. **Наборы данных** –
   - При выборе `user-data only` используем `--preset user-data` и **не** добавляем `--migrate-secrets`.
   - При выборе `full compatible migration` используем `--preset full --migrate-secrets`.
6. **Целевой рабочий каталог** – флаг `--workspace-target` добавляется **только**, если пользователь явно указал абсолютный путь к рабочему пространству.
7. **Пропуск инструкций рабочего пространства** – если выбран `skip workspace instructions` или `decide later`, флаг `--workspace-target` не добавляется.

Перед запуском команды я ещё раз озвучу полученный план, чтобы убедиться, что он полностью соответствует выбранным пользователем опциям.
## Правила отчётности после выполнения

После выполнения рассматривай JSON‑вывод скрипта как единственный источник правды.

1. Основывай все подсчёты на `report.summary`.
2. Добавляй элемент в раздел **Successfully Migrated** только если его `status` точно равно `migrated`.
3. Не заявляй, что конфликт был решён, если в отчёте этот элемент не помечен как `migrated`.
4. Не говори, что `SOUL.md` был перезаписан, если элемент с `kind="soul"` имеет `status!="migrated"`.
5. Если `report.summary.conflict > 0`, включи раздел конфликтов вместо молчаливого предположения об успехе.
6. Если подсчёты и перечисленные элементы не совпадают, исправь список, чтобы он соответствовал отчёту, перед тем как отвечать.
7. Включи путь `output_dir` из отчёта, когда он доступен, чтобы пользователь мог проверить `report.json`, `summary.md`, резервные копии и архивированные файлы.
8. При переполнении памяти или профиля пользователя не говори, что записи были архивированы, если отчёт явно не указывает путь архива. Если существует `details.overflow_file`, укажи, что полный список переполнения был экспортирован туда.
9. Если навык был импортирован в переименованную папку, укажи конечное назначение и упомяни `details.renamed_from`.
10. Если присутствует `report.skill_conflict_mode`, используй его как источник правды для выбранной политики конфликта импортируемого навыка.
11. Если элемент имеет `status="skipped"`, не описывай его как перезаписанный, сохранённый, мигрированный или решённый.
12. Если `kind="soul"` имеет `status="skipped"` с причиной `Target already matches source`, скажи, что он был оставлен без изменений, и не упоминай резервную копию.
13. Если переименованный импортированный навык имеет пустой `details.backup`, не подразумевай, что существующий навык Hermes был переименован или сохранён. Скажи только, что импортированная копия была помещена в новое место, и укажи `details.renamed_from` как предшествующую папку, которая осталась на месте.
## Presets миграции

Предпочитай эти два preset'а в обычном использовании:

- `user-data`
- `full`

`user-data` включает:

- `soul`
- `workspace-agents`
- `memory`
- `user-profile`
- `messaging-settings`
- `command-allowlist`
- `skills`
- `tts-assets`
- `archive`

`full` включает всё из `user-data` плюс:

- `secret-settings`

Скрипт‑помощник всё ещё поддерживает категориальный `--include` / `--exclude`, но рассматривай это как продвинутый запасной (вариант), а не как поведение по умолчанию.

## Важные правила

1. Выполняй пробный запуск перед тем, как писать, если только пользователь явно не попросил действовать сразу.
2. По умолчанию не переноси секреты. Токены, блобы аутентификации, учётные данные устройств и сырые конфигурации шлюза должны оставаться вне Hermes, если только пользователь явно не запросит миграцию секретов.
3. Не перезаписывай без‑молвно непустые цели Hermes, если только пользователь явно не захочет этого. Скрипт‑помощник сохранит резервные копии, когда перезапись включена.
4. Всегда предоставляй пользователю отчёт о пропущенных элементах. Этот отчёт является частью миграции, а не необязательной добавкой.
5. Предпочитай основной рабочий каталог OpenClaw (`~/.openclaw/workspace/`) вместо `workspace.default/`. Используй рабочий каталог по умолчанию только как запасной вариант, когда основные файлы отсутствуют.
6. Даже в режиме миграции секретов переноси секреты только в чистый пункт назначения Hermes. Неподдерживаемые блобы аутентификации всё равно должны быть отмечены как пропущенные.
7. Если пробный запуск показывает большую копию ресурсов, конфликтующий `SOUL.md` или переполненные записи памяти, укажи на это отдельно перед выполнением.
8. По умолчанию выбирай «только пользовательские данные», если пользователь не уверен.
9. Включай `workspace-agents` только когда пользователь явно указал путь к целевому рабочему каталогу.
10. Рассматривай параметры уровня категории `--include` / `--exclude` как продвинутый способ обхода, а не как обычный поток.
11. Не заканчивай сводку пробного запуска расплывчатым «Что бы ты хотел сделать?», если доступен `clarify`. Используй структурированные последующие подсказки.
12. Не используй открытый запрос `clarify`, когда подходит реальный выбор. Сначала предлагай варианты выбора, а свободный ввод оставляй только для абсолютных путей или запросов просмотра файлов.
13. После пробного запуска никогда не останавливайся после резюме, если остаётся нерешённое решение. Сразу вызывай `clarify` для самого приоритетного блокирующего решения.
14. Порядок приоритетов для последующих вопросов:
    - конфликт `SOUL.md`
    - конфликты импортированных навыков
    - режим миграции
    - пункт назначения инструкций рабочего каталога
15. Не обещай представить варианты позже в том же сообщении. Представляй их, действительно вызывая `clarify`.
16. После ответа о режиме миграции явно проверь, остаётся ли нерешённым `workspace-agents`. Если да, твой следующий шаг должен быть вызов `clarify` для инструкций рабочего каталога.
17. После любого ответа `clarify`, если остаётся другое требуемое решение, не описывай только что принятое решение. Сразу задавай следующий необходимый вопрос.

- заполненные файлы памяти Hermes преобразованными знаниями OpenClaw
- навыки OpenClaw, доступные в `~/.hermes/skills/openclaw-imports/`
- отчёт о миграции, показывающий любые конфликты, пропуски или неподдерживаемые данные
