# Память Honcho для Hermes

Honcho предоставляет AI‑нативное кросс‑сессионное моделирование пользователя. Он узнаёт, кто пользователь, в разных разговорах, и даёт каждому профилю Hermes собственную peer‑идентичность, одновременно предоставляя единый общий вид пользователя.
## Когда использовать

- Настройка Honcho (облако или самостоятельный хостинг)
- Устранение неполадок, когда память не работает / узлы не синхронизируются
- Создание многопрофильных конфигураций, где каждый агент имеет свой собственный узел Honcho
- Настройка параметров наблюдения, воспоминаний, глубины диалектики или частоты записи
- Понимание, что делают 5 инструментов Honcho и когда их использовать
- Настройка бюджетов контекста и внедрение сводки сессии

## Архитектура

### Внедрение базового контекста

Когда Honcho внедряет контекст в системный запрос (в режимах `hybrid` или `context`), он собирает блок базового контекста в следующем порядке:

1. **Сводка сессии** — короткое резюме текущей сессии (размещается первым, чтобы модель сразу имела разговорную непрерывность)
2. **Представление пользователя** — накопленная Honcho модель пользователя (предпочтения, факты, паттерны)
3. **Карточка AI‑пира** — идентификационная карточка AI‑пира этого профиля Hermes

Сводка сессии генерируется автоматически Honcho в начале каждого хода (когда существует предыдущая сессия). Она даёт модели «тёплый» старт без повторного воспроизведения полной истории.

### Выбор холодного / тёплого запроса

Honcho автоматически выбирает одну из двух стратегий запроса:

| Условие | Стратегия | Что происходит |
|-----------|----------|----------------|
| Нет предыдущей сессии или пустое представление | **Cold start** | Лёгкий вводный запрос; пропускает внедрение сводки; стимулирует модель узнать о пользователе |
| Существующее представление и/или история сессии | **Warm start** | Полное внедрение базового контекста (сводка → представление → карточка); более богатый системный запрос |

Настраивать это не требуется — выбор происходит автоматически в зависимости от состояния сессии.

### Пиры

Honcho моделирует разговоры как взаимодействия между **пирами**. Hermes создаёт два пира на каждую сессию:

- **Пир пользователя** (`peerName`): представляет человека. Honcho формирует представление пользователя из наблюдаемых сообщений.
- **AI‑пир** (`aiPeer`): представляет данный экземпляр Hermes. Каждый профиль получает собственного AI‑пира, чтобы агенты развивали независимые взгляды.

### Наблюдение

У каждого пира есть два переключателя наблюдения, определяющих, что Honcho будет изучать:

| Переключатель | Что делает |
|---------------|------------|
| `observeMe` | Наблюдаются собственные сообщения пира (строится самопредставление) |
| `observeOthers` | Наблюдаются сообщения других пиров (строится кросс‑пировое понимание) |

По умолчанию все четыре переключателя **включены** (полное двунаправленное наблюдение).

Настройка per‑peer в `honcho.json`:

```json
{
  "observation": {
    "user": { "observeMe": true, "observeOthers": true },
    "ai":   { "observeMe": true, "observeOthers": true }
  }
}
```

Или используйте короткие пресеты:

| Пресет | Пользователь | AI | Сценарий |
|--------|--------------|----|----------|
| `"directional"` (по умолчанию) | me:on, others:on | me:on, others:on | Мульти‑агент, полная память |
| `"unified"` | me:on, others:off | me:off, others:on | Один агент, моделирование только пользователя |

Изменения настроек в [панели Honcho](https://app.honcho.dev) синхронно применяются при инициализации сессии — серверные конфиги имеют приоритет над локальными значениями по умолчанию.

### Сессии

Сессии Honcho определяют, куда попадают сообщения и наблюдения. Варианты стратегии:

| Стратегия | Поведение |
|-----------|-----------|
| `per-directory` (по умолчанию) | Одна сессия на рабочий каталог |
| `per-repo` | Одна сессия на корень git‑репозитория |
| `per-session` | Новая сессия Honcho при каждом запуске Hermes |
| `global` | Одна общая сессия для всех каталогов |

Ручное переопределение: `hermes honcho map my-project-name`

### Режимы воспоминания

Как агент получает доступ к памяти Honcho:

| Режим | Автоматически внедрять контекст? | Доступные инструменты? | Сценарий |
|------|----------------------------------|------------------------|----------|
| `hybrid` (по умолчанию) | Да | Да | Агент решает, когда использовать инструменты, а когда полагаться на авто‑контекст |
| `context` | Да | Нет (скрыты) | Минимальные токены, без вызовов инструментов |
| `tools` | Нет | Да | Агент полностью контролирует доступ к памяти |
## Три ортогональных регулятора

Диалектическое поведение Honcho контролируется тремя независимыми измерениями. Каждое из них можно настраивать, не влияя на остальные:

### Cadence (когда)

Определяет **как часто** происходят вызовы диалектики и контекста.

| Ключ | По умолчанию | Описание |
|-----|--------------|----------|
| `contextCadence` | `1` | Минимальное количество ходов между вызовами API контекста |
| `dialecticCadence` | `2` | Минимальное количество ходов между вызовами диалектического API. Рекомендуется 1–5 |
| `injectionFrequency` | `every-turn` | `every-turn` или `first-turn` для базовой инъекции контекста |

Более высокие значения cadence заставляют диалектический LLM работать реже. `dialecticCadence: 2` означает, что движок срабатывает каждый второй ход. Установка значения `1` заставляет его срабатывать каждый ход.

### Depth (сколько)

Определяет **сколько раундов** диалектического рассуждения Honcho выполняет за запрос.

| Ключ | По умолчанию | Диапазон | Описание |
|-----|--------------|----------|----------|
| `dialecticDepth` | `1` | 1‑3 | Количество раундов диалектического рассуждения за запрос |
| `dialecticDepthLevels` | — | array | Необязательные переопределения уровня рассуждения для каждого раунда (см. ниже) |

`dialecticDepth: 2` означает, что Honcho выполняет два раунда диалектического синтеза. Первый раунд генерирует начальный ответ; второй уточняет его.

`dialecticDepthLevels` позволяет задать уровень рассуждения для каждого раунда независимо:

```json
{
  "dialecticDepth": 3,
  "dialecticDepthLevels": ["low", "medium", "high"]
}
```

Если `dialecticDepthLevels` не указано, раунды используют **пропорциональные уровни**, полученные из `dialecticReasoningLevel` (базового):

| Depth | Уровни проходов |
|-------|-----------------|
| 1 | [base] |
| 2 | [minimal, base] |
| 3 | [minimal, base, low] |

Это делает ранние проходы дешёвыми, а полную глубину использует только на финальном синтезе.

**Depth при старте сессии.** Предварительный прогрев при старте сессии запускает полностью сконфигурированный `dialecticDepth` в фоне до первого хода. Однопроходный прогрев на холодном пире часто возвращает скудный вывод — многопроходный прогрев выполняет цикл аудита/согласования до того, как пользователь заговорит. Ход 1 использует результат прогрева напрямую; если прогрев не успел завершиться, ход 1 переходит к синхронному вызову с ограниченным тайм‑аутом.

### Level (насколько интенсивно)

Определяет **интенсивность** каждого раунда диалектического рассуждения.

| Ключ | По умолчанию | Описание |
|-----|--------------|----------|
| `dialecticReasoningLevel` | `low` | `minimal`, `low`, `medium`, `high`, `max` |
| `dialecticDynamic` | `true` | При `true` модель может передать `reasoning_level` в `honcho_reasoning`, переопределяя значение по умолчанию для конкретного вызова. `false` = всегда использовать `dialecticReasoningLevel`, переопределения модели игнорируются |

Более высокие уровни дают более богатый синтез, но требуют больше токенов на бекэнде Honcho.
## Мульти‑профильная настройка

Каждый профиль Hermes получает собственного собеседника Honcho AI, при этом все они используют одно и то же рабочее пространство (контекст пользователя). Это означает:

- Все профили видят одинаковое представление пользователя
- Каждый профиль формирует свою собственную AI‑идентичность и наблюдения
- Выводы, записанные одним профилем, видны другим через общее рабочее пространство

### Создание профиля с Honcho peer

```bash
hermes profile create coder --clone
# creates host block hermes.coder, AI peer "coder", inherits config from default
```

Что делает `--clone` для Honcho:
1. Создаёт блок хоста `hermes.coder` в `honcho.json`
2. Устанавливает `aiPeer: "coder"` (имя профиля)
3. Наследует `workspace`, `peerName`, `writeFrequency`, `recallMode` и т.д. из настроек по умолчанию
4. Активно создаёт собеседника в Honcho, чтобы он существовал до первой отправки сообщения

### Заполнение существующих профилей

```bash
hermes honcho sync    # creates host blocks for all profiles that don't have one yet
```

### Конфигурация для каждого профиля

Переопределить любую настройку в блоке хоста:

```json
{
  "hosts": {
    "hermes.coder": {
      "aiPeer": "coder",
      "recallMode": "tools",
      "dialecticDepth": 2,
      "observation": {
        "user": { "observeMe": true, "observeOthers": false },
        "ai": { "observeMe": true, "observeOthers": true }
      }
    }
  }
}
```
## Инструменты

У агента есть 5 двунаправленных Honcho‑инструментов (скрыты в режиме `context` recall):

| Инструмент | Вызов LLM? | Стоимость | Когда использовать |
|------------|------------|-----------|----------------------|
| `honcho_profile` | Нет | минимальная | Быстрый фактический снимок в начале разговора или для быстрых запросов имени/роли/предпочтений |
| `honcho_search` | Нет | низкая | Получить конкретные прошлые факты для самостоятельного рассуждения — сырые выдержки, без синтеза |
| `honcho_context` | Нет | низкая | Полный снимок контекста сессии: резюме, представление, карточка, последние сообщения |
| `honcho_reasoning` | Да | средне‑высокая | Вопрос на естественном языке, синтезированный диалектическим движком Honcho |
| `honcho_conclude` | Нет | минимальная | Записать или удалить постоянный факт; передай `peer: "ai"` для самопознания ИИ |

### `honcho_profile`
Чтение или обновление карточки собеседника — отобранные ключевые факты (имя, роль, предпочтения, стиль общения). Передай `card: [...]` для обновления; опусти параметр для чтения. Вызова LLM нет.

### `honcho_search`
Семантический поиск по сохранённому контексту для конкретного собеседника. Возвращает сырые выдержки, ранжированные по релевантности, без синтеза. По умолчанию 800 токенов, максимум 2000. Полезно, когда нужны конкретные прошлые факты для собственного рассуждения, а не синтезированный ответ.

### `honcho_context`
Полный снимок контекста сессии от Honcho — резюме сессии, представление собеседника, карточка собеседника и последние сообщения. Вызова LLM нет. Используй, когда хочешь увидеть всё, что Honcho знает о текущей сессии и собеседнике, одним запросом.

### `honcho_reasoning`
Вопрос на естественном языке, отвечаемый диалектическим движком рассуждений Honcho (вызов LLM на бекенде Honcho). Более высокая стоимость, более высокое качество. Передай `reasoning_level` для управления глубиной: `minimal` (быстро/дешево) → `low` → `medium` → `high` → `max` (тщательно). Если параметр опущен, используется настроенный по умолчанию уровень (`low`). Используй для синтезированного понимания паттернов, целей или текущего состояния пользователя.

### `honcho_conclude`
Записать или удалить постоянный вывод о собеседнике. Передай `conclusion: "..."` для создания. Передай `delete_id: "..."` для удаления вывода (для удаления персональных данных — Honcho со временем сам исправляет неверные выводы, поэтому удаление требуется только для ПИИ). Ты ДОЛЖЕН передать ровно один из этих параметров.

### Двунаправленное таргетирование собеседников

Все 5 инструментов принимают необязательный параметр `peer`:
- `peer: "user"` (по умолчанию) — работает с пользователем
- `peer: "ai"` — работает с AI‑собеседником этого профиля
- `peer: "<explicit-id>"` — любой ID собеседника в рабочем пространстве

Примеры:
```
honcho_profile                        # read user's card
honcho_profile peer="ai"              # read AI peer's card
honcho_reasoning query="What does this user care about most?"
honcho_reasoning query="What are my interaction patterns?" peer="ai" reasoning_level="medium"
honcho_conclude conclusion="Prefers terse answers"
honcho_conclude conclusion="I tend to over-explain code" peer="ai"
honcho_conclude delete_id="abc123"    # PII removal
```
## Шаблоны использования агента

Руководство для Hermes, когда память Honcho активна.

### При начале разговора

```
1. honcho_profile                  → fast warmup, no LLM cost
2. If context looks thin → honcho_context  (full snapshot, still no LLM)
3. If deep synthesis needed → honcho_reasoning  (LLM call, use sparingly)
```

Не вызывай `honcho_reasoning` на каждом ходе. Автовнедрение уже обрабатывает обновление текущего контекста. Используй инструмент рассуждения только тогда, когда действительно нужен синтезированный инсайт, которого нет в базовом контексте.

### Когда пользователь делится чем‑то, что нужно запомнить

```
honcho_conclude conclusion="<specific, actionable fact>"
```

Хорошие выводы: «Предпочитает примеры кода вместо текстовых объяснений», «Работает над асинхронным проектом на Rust до апреля 2026 г.»
Плохие выводы: «Пользователь сказал что‑то о Rust» (слишком расплывчато), «Пользователь кажется техническим» (уже в представлении).

### Когда пользователь спрашивает о прошлом контексте / тебе нужно вспомнить детали

```
honcho_search query="<topic>"       → fast, no LLM, good for specific facts
honcho_context                       → full snapshot with summary + messages
honcho_reasoning query="<question>"  → synthesized answer, use when search isn't enough
```

### Когда использовать `peer: "ai"`

Используй целевой AI‑пир для построения и запроса собственного самопознания агента:
- `honcho_conclude conclusion="I tend to be verbose when explaining architecture" peer="ai"` — самокоррекция
- `honcho_reasoning query="How do I typically handle ambiguous requests?" peer="ai"` — самоаудит
- `honcho_profile peer="ai"` — просмотр собственной карточки идентификации

### Когда НЕ вызывать инструменты

В режимах `hybrid` и `context` базовый контекст (представление пользователя + карточка + резюме сессии) автоматически внедряется перед каждым ходом. Не повторяй запрос того, что уже внедрено. Вызывай инструменты только когда:
- Нужно получить то, чего нет во внедрённом контексте;
- Пользователь явно просит тебя вспомнить или проверить память;
- Ты формируешь вывод о чём‑то новом.

### Осведомлённость о каденции

`honcho_reasoning` со стороны инструмента стоит столько же, сколько диалектика автовнедрения. После явного вызова инструмента каденция автовнедрения сбрасывается — это предотвращает двойную оплату за один ход.
## Справочник конфигурации

Файл конфигурации: `$HERMES_HOME/honchrome.json` (локальный профиль) или `~/.honcho/config.json` (глобальный).

### Основные параметры

| Параметр | По умолчанию | Описание |
|-----|---------|-------------|
| `apiKey` | -- | API‑ключ ([получить](https://app.honcho.dev)) |
| `baseUrl` | -- | Базовый URL для самохостинга Honcho |
| `peerName` | -- | Идентичность пользователя‑пира |
| `aiPeer` | host key | Идентичность AI‑пира |
| `workspace` | host key | Идентификатор общего рабочего пространства |
| `recallMode` | `hybrid` | `hybrid`, `context` или `tools` |
| `observation` | all on | Булевы флаги `observeMe`/`observeOthers` для каждого пира (все включены) |
| `writeFrequency` | `async` | `async`, `turn`, `session` или целое N |
| `sessionStrategy` | `per-directory` | `per-directory`, `per-repo`, `per-session`, `global` |
| `messageMaxChars` | `25000` | Максимум символов в сообщении (при превышении разбивается на части) |

### Параметры диалектики

| Параметр | По умолчанию | Описание |
|-----|---------|-------------|
| `dialecticReasoningLevel` | `low` | `minimal`, `low`, `medium`, `high`, `max` |
| `dialecticDynamic` | `true` | Автоматически повышать уровень рассуждения в зависимости от сложности запроса; `false` — фиксированный уровень |
| `dialecticDepth` | `1` | Количество диалектических раундов на запрос (1‑3) |
| `dialecticDepthLevels` | -- | Необязательный массив уровней для каждого раунда, например `["low", "high"]` |
| `dialecticMaxInputChars` | `10000` | Максимум символов во входных данных диалектического запроса |

### Бюджет контекста и инъекция

| Параметр | По умолчанию | Описание |
|-----|---------|-------------|
| `contextTokens` | uncapped | Максимум токенов для совмещённой базовой инъекции контекста (резюме + представление + карточка). Чтобы установить ограничение, укажи целое число; иначе оставь без ограничения. |
| `injectionFrequency` | `every-turn` | `every-turn` или `first-turn` |
| `contextCadence` | `1` | Минимальное количество ходов между вызовами API контекста |
| `dialecticCadence` | `2` | Минимальное количество ходов между вызовами диалектического LLM (рекомендовано 1–5) |

Бюджет `contextTokens` применяется во время инъекции. Если суммарное резюме + представление + карточка превышают бюджет, Honcho сначала обрезает резюме, затем представление, сохраняя карточку. Это предотвращает рост контекста в длительных сессиях.

### Санитизация памяти‑контекста

Honcho очищает блок `memory-context` перед инъекцией, чтобы избежать внедрения подсказок и некорректного содержимого:

- Удаляет XML/HTML‑теги из пользовательских выводов
- Нормализует пробелы и управляющие символы
- Обрезает отдельные выводы, превышающие `messageMaxChars`
- Экранирует последовательности‑разделители, которые могут нарушить структуру системного подсказочного текста

Это исправление устраняет крайние случаи, когда сырые пользовательские выводы с разметкой или специальными символами могут испортить инъецируемый блок контекста.
## Устранение неполадок

### «Honcls not configured»
Запусти `hermes honcho setup`. Убедись, что в `~/.hermes/config.yaml` указано `memory.provider: honcho`.

### Память не сохраняется между сессиями
Проверь `hermes honcho status` — убедись, что `saveMessages: true` и `writeFrequency` не установлен в `session` (что записывает только при выходе).

### Профиль не получает собственного пира
Используй `--clone` при создании: `hermes profile create <name> --clone`. Для уже существующих профилей: `hermes honcho sync`.

### Изменения наблюдений в панели не отражаются
Конфигурация наблюдений синхронизируется с сервером при каждой инициализации сессии. Запусти новую сессию после изменения настроек в UI Honcho.

### Сообщения обрезаются
Сообщения, превышающие `messageMaxChars` (по умолчанию 25 k), автоматически разбиваются с маркерами `[continued]`. Если это происходит часто, проверь, не увеличивают ли результаты инструмента или содержимое навыка размер сообщения.

### Внедрение контекста слишком велико
Если появляются предупреждения о превышении бюджета контекста, снизь `contextTokens` или уменьшите `dialecticDepth`. Сводка сессии обрезается первой, когда бюджет ограничен.

### Сводка сессии отсутствует
Сводка сессии требует хотя бы одного предыдущего хода в текущей сессии Honcho. При холодном старте (новая сессия, без истории) сводка опускается, и Honcho использует стратегию подсказки для холодного старта.
