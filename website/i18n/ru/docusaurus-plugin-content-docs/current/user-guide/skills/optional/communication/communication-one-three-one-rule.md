# Правило коммуникации 1‑3‑1

Структурированный формат принятия решений, когда задача имеет несколько жизнеспособных подходов и пользователю нужна чёткая рекомендация. Генерирует лаконичное формулирование проблемы, три варианта с компромиссами и практический план для выбранного пути.

## Когда использовать

- Пользователь явно просит ответ в формате «1‑3‑1».
- Пользователь говорит «дай варианты» или «каковы мои варианты» для технического решения.
- Задача имеет несколько жизнеспособных подходов с существенными компромиссами (архитектура, инструменты, стратегия миграции).
- Пользователю нужен документ, который он может передать команде или заинтересованным сторонам.

Не используйте для простых вопросов с единственным очевидным ответом, отладочных сессий или задач, где пользователь уже определился с подходом.

## Процедура

1. **Проблема** (одно предложение)
   - Сформулируй основное решение или желаемый результат в одном лаконичном предложении.
   - Сосредоточься на *что*, а не на *как* — без деталей реализации, названий инструментов или конкретных технологий.
   - Делай коротко. Если нужен союз «и», ты описываешь две проблемы.

2. **Варианты** (ровно три)
   - Представь три различных, жизнеспособных подхода, помеченных A, B, C.
   - Каждый вариант получает краткое описание, плюсы и минусы.
   - Варианты должны представлять действительно разные стратегии, а не незначительные вариации одного подхода.

3. **Рекомендация** (один вариант)
   - Укажи, какой вариант ты рекомендуешь и почему, исходя из контекста и приоритетов пользователя.
   - Будь прямым — это твоё профессиональное суждение, а не уклончивый ответ.

4. **Определение готовности**
   - Перечисли конкретные критерии успеха для рекомендованного варианта.
   - Это конкретные, проверяемые результаты — а не расплывчатые цели.
   - Если пользователь выбирает другой вариант, обнови этот раздел в соответствии с ним.

5. **План реализации**
   - Конкретные шаги для выполнения рекомендованного варианта.
   - Укажи конкретные команды, инструменты или действия, где это уместно.
   - Если пользователь выбирает другой вариант, обнови этот раздел в соответствии с ним.

## Проверка

- Ответ содержит ровно одно предложение Проблемы.
- Ответ содержит ровно три Варианта (A, B, C) с плюсами и минусами для каждого.
- Ответ содержит единую Рекомендацию, выбирающую один вариант с обоснованием.
- Определение готовности и План реализации соответствуют выбранному варианту.
- Если пользователь выбирает другой вариант, Рекомендация, Определение готовности и План реализации обновляются соответственно.

## Пример

User: «Дай мне 1‑3‑1 для добавления логики повторных попыток в наш API‑клиент.»

Agent:

**Problem:** API‑вызовы к внешним сервисам иногда падают под нагрузкой и требуют корректного повторного выполнения.

**Options:**

- **Option A: Экспоненциальный backoff с jitter в общей утилите.**
  **Pros:** Простой, хорошо известный шаблон; одна реализация, общая для всех вызывающих.
  **Cons:** Нет circuit‑breaking; вызывающие всё равно блокируются во время повторов.

- **Option B: Библиотека повторов (tenacity) как декоратор для каждого вызова эндпоинта.**
  **Pros:** Настраиваемо per‑endpoint; встроенная поддержка circuit‑breaker и rate‑limiting.
  **Cons:** Добавляет зависимость; паттерн декоратора может скрывать поток управления.

- **Option C: Асинхронный retry через очередь с обработкой dead‑letter.**
  **Pros:** Не блокирует; неудачные вызовы отслеживаются и могут быть переотправлены; хорошо масштабируется.
  **Cons:** Значительное увеличение сложности; требуется бекенд очереди; избыточно для сервисов с небольшим объёмом.

**Recommendation:** Option A. Объём вызовов не оправдывает инфраструктуру очередей, а общая утилита с экспоненциальным backoff + jitter покрывает сценарий отказов без новых зависимостей.

**Definition of Done:**

- Все внешние API‑вызовы проходят через утилиту retry.
- Логируются попытки повторов с указанием эндпоинта и номера попытки.
- Тесты покрывают исчерпание повторов, мгновенный успех и сценарий «временный‑затем‑успех».
