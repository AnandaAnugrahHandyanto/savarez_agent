# Сессии

Hermes Agent автоматически сохраняет каждый разговор как сессию. Сессии позволяют возобновлять разговор, выполнять поиск по нескольким сессиям и полностью управлять историей разговоров.
## Как работают сессии

Каждый разговор — будь то из CLI, Telegram, Discord, Slack, WhatsApp, Signal, Matrix, Teams или любой другой платформы обмена сообщениями — сохраняется как сессия с полной историей сообщений. Сессии отслеживаются в:

1. **SQLite‑база данных** (`~/.hermes/state.db`) — структурированные метаданные сессии с полнотекстовым поиском FTS5, плюс полная история сообщений

SQLite‑база данных хранит:
- ID сессии, платформа‑источник, ID пользователя
- **Заголовок сессии** (уникальное, человекочитаемое имя)
- Название модели и её конфигурацию
- Снимок системного промпта
- Полную историю сообщений (роль, содержание, вызовы инструментов, результаты инструментов)
- Счётчики токенов (входящие/исходящие)
- Метки времени (`started_at`, `ended_at`)
- ID родительской сессии (для разделения сессий, вызванного сжатием)

### Что учитывается в контексте

Hermes сохраняет историю сессии, чтобы можно было возобновлять разговоры, но не пересылает каждый байт, с которым когда‑либо работал. На каждом шаге модель видит выбранный системный промпт, текущее окно разговора и любой контент, который Hermes явно вставляет для этого шага.

Медиа‑вложения обрабатываются как ввод, ограниченный текущим шагом:

- Изображения могут быть прикреплены нативно к следующему вызову модели, либо предварительно проанализированы в текстовое описание, если активная модель не поддерживает нативный зрительный ввод.
- Аудио транскрибируется в текст, когда включён режим speech‑to‑text.
- Текстовые документы могут включать извлечённый текст; другие типы документов обычно представляются сохранённым локальным путём и короткой заметкой.
- Пути к вложениям и извлечённый/полученный текст могут появляться в транскрипте, но сырые байты изображения, аудио или бинарного файла не копируются повторно в будущие подсказки.

Например, если пользователь отправил изображение и попросил Hermes сделать из него мем, Hermes может один раз проанализировать изображение с помощью зрительного модуля и запустить скрипт обработки изображения. Последующие шаги автоматически не включают оригинальный JPEG в контекст. Они содержат только то, что было записано в разговоре: запрос пользователя, короткое описание изображения, путь к локальному кэшу или окончательный ответ ассистента.

Самая частая причина роста контекста — не медиа‑файл, а объёмный текст: вставленные транскрипты, полные логи, большие выводы инструментов, длинные диффы, повторяющиеся отчёты о статусе и детальные дампы доказательств. Предпочитай резюме, пути к файлам, фокусированные выдержки и поиск через инструменты, а не копирование больших артефактов в чат.

:::tip
Используй `/compress`, когда сессия становится длинной, `/new` для создания новой ветки и `hermes sessions prune` только когда действительно нужно удалить старые завершённые сессии из хранилища. Сжатие уменьшает активный контекст; это не удаление данных для конфиденциальности. Передай имя в `/new` (например, `/new payments-refactor`), чтобы задать начальный заголовок новой сессии — это удобно для последующего поиска с помощью `/resume <name>` или в списке `/sessions`.
:::

## Возобновление сессии CLI

Возобнови предыдущие беседы из CLI с помощью `--continue` или `--resume`:

### Продолжить последнюю сессию

```bash
# Resume the most recent CLI session
hermes --continue
hermes -c

# Or with the chat subcommand
hermes chat --continue
hermes chat -c
```

Это ищет самую последнюю `cli`‑сессию в базе данных SQLite и загружает её полную историю разговора.

### Возобновить по названию

Если ты задал сессии название (см. [Названия сеансов](#session-naming) ниже), можешь возобновить её по имени:

```bash
# Resume a named session
hermes -c "my project"

# If there are lineage variants (my project, my project #2, my project #3),
# this automatically resumes the most recent one
hermes -c "my project"   # → resumes "my project #3"
```

### Возобновить конкретную сессию

```bash
# Resume a specific session by ID
hermes --resume 20250305_091523_a1b2c3d4
hermes -r 20250305_091523_a1b2c3d4

# Resume by title
hermes --resume "refactoring auth"

# Or with the chat subcommand
hermes chat --resume 20250305_091523_a1b2c3d4
```

Идентификаторы сессий отображаются при выходе из CLI‑сессии и их можно найти с помощью `hermes sessions list`.

### Краткое содержание разговора при возобновлении

Когда ты возобновляешь сессию, Hermes выводит компактное резюме предыдущего разговора в стилизованной панели перед приглашением ввода:

<img className="docs-terminal-figure" src="/img/docs/session-recap.svg" alt="Stylized preview of the Previous Conversation recap panel shown when resuming a Hermes session." />
<p className="docs-figure-caption">Режим возобновления показывает компактную панель с недавними репликами пользователя и ассистента перед возвратом к живому приглашению.</p>

Резюме:
- Показывает **сообщения пользователя** (золотой `●`) и **ответы ассистента** (зелёный `◆`);
- **Обрезает** длинные сообщения (300 символов для пользователя, 200 символов / 3 строки для ассистента);
- **Сворачивает вызовы инструментов** до количества с названиями инструментов (например, `[3 tool calls: terminal, web_search]`);
- **Скрывает** системные сообщения, результаты инструментов и внутреннее рассуждение;
- **Ограничивает** вывод последними 10 обменами с индикатором «... N earlier messages ...»;
- Использует **тусклое оформление**, чтобы отличать его от активного разговора.

Чтобы отключить резюме и оставить поведение в виде однострочного вывода, задай в `~/.hermes/config.yaml`:

```yaml
display:
  resume_display: minimal   # default: full
```

:::tip
Идентификаторы сессий имеют формат `YYYYMMDD_HHMMSS_<hex>` — сессии CLI/TUI используют 6‑символьный hex‑суффикс (например, `20250305_091523_a1b2c3`), сессии шлюза — 8‑символьный суффикс (например, `20250305_091523_a1b2c3d4`). Можно возобновлять по ID (полному или уникальному префиксу) или по названию — оба варианта работают с `-c` и `-r`.
:::
## Кроссплатформенный переход

Используй `/handoff <platform>` из CLI‑сессии, чтобы перенести живой разговор в домашний канал мессенджера. Агент продолжит ровно там, где остановился CLI — тот же идентификатор сессии, полный роле‑aware транскрипт, вызовы инструментов и всё остальное.

```bash
# Inside a CLI session
/handoff telegram
```

Что происходит:

1. CLI проверяет, что `<platform>` включена и у неё установлен домашний канал (выполни `/sethome` в целевом чате один раз, чтобы настроить его).
2. CLI помечает сессию как ожидающую и **блокирующе опрашивает шлюз**. Отказывает, если агент находится в середине хода — подожди, пока завершится текущий ответ.
3. Наблюдатель шлюза захватывает переход и запрашивает у адаптера назначения новый поток:
   - **Telegram** — открывает новую тему форума (DM‑темы, если в Bot API 9.4+ включён режим Topics в чате, либо тему форума в супергруппе).
   - **Discord** — создаёт поток с автоархивом на 1440 минут под домашним текстовым каналом.
   - **Slack** — публикует стартовое сообщение и использует его `ts` как якорь темы.
   - **WhatsApp / Signal / Matrix / SMS** — нативных тем нет, переходит напрямую в домашний канал.
4. Шлюз привязывает ключ назначения к твоему существующему идентификатору CLI‑сессии, затем формирует синтетический пользовательский ход, прося агента подтвердить и подвести итог. Ответ появляется в новом потоке.
5. Когда шлюз подтверждает успех, CLI выводит подсказку `/resume` и корректно завершает работу:

   ```
   ↻ Handoff complete. The session is now active on telegram.
     Resume it on this CLI later with: /resume my-session-title
   ```

6. С этого момента разговор живёт на платформе. Отвечай в новом потоке — любой уполномоченный в этом канале участник использует ту же сессию, и любое последующее реальное сообщение пользователя в потоке автоматически присоединяется, потому что ключ сессии потока не содержит `user_id`.

**Возврат в CLI:** когда захочешь вернуться к работе на компьютере, просто выполни `/resume <title>` (или `hermes -r "<title>"` из оболочки) и продолжи там, где остановилась платформа.

**Режимы отказа:**
- Не настроен домашний канал → CLI отказывает с подсказкой `/sethome`.
- Платформа не включена / шлюз не запущен → CLI истекает через 60 с с чётким сообщением, а твоя CLI‑сессия остаётся нетронутой.
- Не удалось создать поток (отсутствие прав, отключён режим тем) → переходит напрямую в домашний канал и всё равно завершает процесс; изоляция потока отсутствует, но сам переход работает.
- `adapter.send` не удалось (лимит запросов, временная ошибка API) → переход помечается как неудачный с указанием причины; запись очищается, и ты можешь повторить попытку.

**Ограничение, о котором стоит знать:** для платформ без поддержки потоков и с групповыми домашними каналами синтетический ход используется как сессия в стиле DM. Это работает для домашних каналов‑личных сообщений (обычная настройка), но не идеально для действительно общих групповых чатов. Потоки поддерживаются в Telegram, Discord и Slack — наиболее распространённый случай, поэтому большинство конфигураций никогда не сталкиваются с этой проблемой.
## Именование сессий

Давай сессиям читаемые человеком названия, чтобы их было легко находить и возобновлять.

### Автоматически сгенерированные названия

Hermes автоматически генерирует короткое описательное название (3–7 слов) для каждой сессии после первого обмена. Это происходит в фоновом потоке с использованием быстрой вспомогательной модели, поэтому не добавляет задержки. Ты увидишь автоматически сгенерированные названия при просмотре сессий с помощью `hermes sessions list` или `hermes sessions browse`.

Автоматическое присвоение названия срабатывает только один раз за сессию и пропускается, если ты уже задал название вручную.

### Установка названия вручную

Используй слеш‑команду `/title` внутри любой чат‑сессии (CLI или gateway):

```
/title my research project
```

Название применяется сразу. Если сессия ещё не создана в базе данных (например, ты выполнил `/title` до отправки первого сообщения), оно ставится в очередь и применяется, как только сессия начнётся.

Ты также можешь переименовать существующие сессии из командной строки:

```bash
hermes sessions rename 20250305_091523_a1b2c3d4 "refactoring auth module"
```

### Правила названий

- **Уникальность** — два сессии не могут иметь одинаковое название
- **Максимум 100 символов** — сохраняет вывод списка чистым
- **Очистка** — управляющие символы, нулевой ширины и RTL‑перекрытия автоматически удаляются
- **Обычный Unicode допускается** — эмодзи, CJK, символы с диакритиками работают без проблем

### Авто‑наследование при сжатии

Когда контекст сессии сжимается (вручную через `/compress` или автоматически), Hermes создаёт новую продолжительную сессию. Если у оригинала было название, новая сессия автоматически получает нумерованное название:

```
"my project" → "my project #2" → "my project #3"
```

Когда ты возобновляешь по имени (`hermes -c "my project"`), автоматически выбирается самая последняя сессия в цепочке.

### `/title` в мессенджерах

Команда `/title` работает во всех платформах шлюза (Telegram, Discord, Slack, WhatsApp):

- `/title My Research` — установить название сессии
- `/title` — показать текущее название
## Команды управления сессиями

Hermes предоставляет полный набор команд управления сессиями через `hermes sessions`:

### Список сессий

```bash
# List recent sessions (default: last 20)
hermes sessions list

# Filter by platform
hermes sessions list --source telegram

# Show more sessions
hermes sessions list --limit 50
```

Когда у сессий есть заголовки, вывод показывает заголовки, превью и относительные метки времени:

```
Title                  Preview                                  Last Active   ID
────────────────────────────────────────────────────────────────────────────────────────────────
refactoring auth       Help me refactor the auth module please   2h ago        20250305_091523_a
my project #3          Can you check the test failures?          yesterday     20250304_143022_e
—                      What's the weather in Las Vegas?          3d ago        20250303_101500_f
```

Если у сессий нет заголовков, используется более простой формат:

```
Preview                                            Last Active   Src    ID
──────────────────────────────────────────────────────────────────────────────────────
Help me refactor the auth module please             2h ago        cli    20250305_091523_a
What's the weather in Las Vegas?                    3d ago        tele   20250303_101500_f
```

### Экспорт сессий

```bash
# Export all sessions to a JSONL file
hermes sessions export backup.jsonl

# Export sessions from a specific platform
hermes sessions export telegram-history.jsonl --source telegram

# Export a single session
hermes sessions export session.jsonl --session-id 20250305_091523_a1b2c3d4
```

Экспортированные файлы содержат один JSON‑объект на строку с полной метаинформацией о сессии и всеми сообщениями.

### Удалить сессию

```bash
# Delete a specific session (with confirmation)
hermes sessions delete 20250305_091523_a1b2c3d4

# Delete without confirmation
hermes sessions delete 20250305_091523_a1b2c3d4 --yes
```

### Переименовать сессию

```bash
# Set or change a session's title
hermes sessions rename 20250305_091523_a1b2c3d4 "debugging auth flow"

# Multi-word titles don't need quotes in the CLI
hermes sessions rename 20250305_091523_a1b2c3d4 debugging auth flow
```

Если заголовок уже используется другой сессией, будет показана ошибка.

### Очистка старых сессий

```bash
# Delete ended sessions older than 90 days (default)
hermes sessions prune

# Custom age threshold
hermes sessions prune --older-than 30

# Only prune sessions from a specific platform
hermes sessions prune --source telegram --older-than 60

# Skip confirmation
hermes sessions prune --older-than 30 --yes
```

:::info
Очистка удаляет только **завершённые** сессии (сессии, которые были явно завершены или автоматически сброшены). Активные сессии никогда не удаляются.
:::

### Статистика сессий

```bash
hermes sessions stats
```

Вывод:

```
Total sessions: 142
Total messages: 3847
  cli: 89 sessions
  telegram: 38 sessions
  discord: 15 sessions
Database size: 12.4 MB
```

Для более глубокого анализа — использования токенов, оценки стоимости, разбивки по инструментам и паттернов активности — используй [`hermes insights`](/reference/cli-commands#hermes-insights).
## Инструмент поиска по сессиям

У агента есть встроенный инструмент `session_search`, который выполняет полнотекстовый поиск по всем прошлым разговорам с помощью движка SQLite FTS5 — и позволяет агенту прокручивать любую найденную сессии. Без вызовов LLM, без суммирования, без усечения. Каждый вариант возвращает реальные сообщения из БД.

### Три формы вызова

Инструмент выводит, что тебе нужно, из переданных аргументов. Параметр `mode` отсутствует.

**1. Обнаружение — передай `query`:**

```python
session_search(query="auth refactor", limit=3)
```

Запускает FTS5, удаляет дубли по иерархии сессий, возвращает топ‑N сессий. Каждый результат содержит:

- `session_id`, `title`, `when`, `source`
- `snippet` — выдержка с подсветкой FTS5
- `bookend_start` — первые 3 сообщения пользователя + ассистента в сессии (цель/запуск)
- `messages` — ±5 сообщений вокруг совпадения FTS5, с пометкой якорного сообщения (хит в контексте)
- `bookend_end` — последние 3 сообщения пользователя + ассистента в сессии (результат/решения)
- `match_message_id`, `messages_before`, `messages_after`

Начальные и конечные части + окно вместе восстанавливают цель → совпадение → результат без необходимости платить за весь транскрипт. Типичное время выполнения: 15–50 мс на реальной БД сессий.

**2. Прокрутка — передай `session_id` + `around_message_id`:**

```python
session_search(session_id="20260510_174648_805cc2", around_message_id=590803, window=10)
```

Возвращает окно ±`window` сообщений, центрированное на якорном сообщении. Без FTS5, без bookends — только срез. Используется после вызова обнаружения, когда требуется больше контекста, чем стандартные ±5 сообщений.

- Чтобы прокручивать **вперёд**: передай `messages[-1].id` обратно как `around_message_id`
- Чтобы прокручивать **назад**: передай `messages[0].id` обратно как `around_message_id`
- Граничное сообщение появляется в обоих окнах как ориентир
- Когда `messages_before` или `messages_after` меньше `window`, ты находишься в начале или в конце сессии

Типичное время выполнения: 1–2 мс за вызов прокрутки.

**3. Обзор — без аргументов:**

```python
session_search()
```

Возвращает недавние сессии в хронологическом порядке (заголовки, превью, метки времени). Полезно, когда пользователь спрашивает «над чем я работал», не называя тему.

### Синтаксис запросов FTS5

Режим ключевых слов поддерживает стандартный синтаксис запросов FTS5:

- Простые ключевые слова: `docker deployment` (по умолчанию FTS5 использует AND)
- Фразы: `"exact phrase"`
- Булевы операции: `docker OR kubernetes`, `python NOT java`
- Префикс: `deploy*`

### Необязательные параметры

- `sort` — `newest` или `oldest`, поверх ранжирования FTS5. По умолчанию сортировка только по релевантности (подходит для исследовательского воспоминания). Используй `newest` для вопросов типа «где мы оставили X», `oldest` — для «как начиналось X».
- `role_filter` — список ролей через запятую, которые следует включить. По умолчанию при обнаружении используется `user,assistant` (вывод инструмента обычно является шумом). Передай `user,assistant,tool`, чтобы включить вывод инструмента (отладка поведения инструмента), или `tool`, чтобы искать только вывод инструмента.

### Когда используется

Агент получает подсказку использовать поиск по сессиям автоматически:

> *«Когда пользователь ссылается на что‑то из прошлого разговора или ты подозреваешь, что релевантный контекст существует, используй session_search, чтобы вспомнить его, прежде чем просить их повторить.»*

Типичные триггеры: «мы делали это раньше», «помнишь, когда», «в прошлый раз», «как я уже говорил», или любая ссылка на проект/человека/концепцию, которой нет в текущем окне.
## Отслеживание сессий по платформам

### Сессии шлюза

На платформах обмена сообщениями сессии идентифицируются детерминированным ключом, построенным из источника сообщения:

| Тип чата | Формат ключа по умолчанию | Поведение |
|-----------|---------------------------|-----------|
| Telegram DM | `agent:main:telegram:dm:<chat_id>` | Одна сессия на каждый личный чат |
| Discord DM | `agent:main:discord:dm:<chat_id>` | Одна сессия на каждый личный чат |
| WhatsApp DM | `agent:main:whatsapp:dm:<canonical_identifier>` | Одна сессия на каждого пользователя DM (алиасы LID/телефон сворачиваются в одну личность, когда существует сопоставление) |
| Group chat | `agent:main:<platform>:group:<chat_id>:<user_id>` | По пользователю внутри группы, если платформа предоставляет идентификатор пользователя |
| Group thread/topic | `agent:main:<platform>:group:<chat_id>:<thread_id>` | Общая сессия для всех участников ветки (по умолчанию). По пользователю при `thread_sessions_per_user: true`. |
| Channel | `agent:main:<platform>:channel:<chat_id>:<user_id>` | По пользователю внутри канала, если платформа предоставляет идентификатор пользователя |

Когда Hermes не может получить идентификатор участника для общего чата, он переходит к одной общей сессии для этой комнаты.

### Общие vs изолированные групповые сессии

По умолчанию Hermes использует `group_sessions_per_user: true` в `config.yaml`. Это означает:

- Alice и Bob могут одновременно общаться с Hermes в одном канале Discord, не разделяя историю транскриптов
- Длительная задача одного пользователя, насыщенная инструментами, не загрязняет контекстное окно другого пользователя
- Обработка прерываний также остаётся по пользователю, потому что ключ `running-agent` совпадает с изолированным ключом сессии

Если вместо этого нужен один общий «мозг комнаты», установи:

```yaml
group_sessions_per_user: false
```

Это вернёт группы/каналы к единой общей сессии на комнату, что сохраняет общий контекст разговора, но также делит затраты токенов, состояние прерываний и рост контекста.

### Политики сброса сессий

Сессии шлюза автоматически сбрасываются согласно настраиваемым политикам:

- **idle** — сброс после N минут бездействия
- **daily** — сброс в определённый час каждый день
- **both** — сброс происходит первым из событий (бездействие или ежедневный сброс)
- **none** — автоматический сброс не выполняется

Перед автоматическим сбросом сессии агент получает возможность сохранить любые важные воспоминания или инструменты из разговора.

Сессии с **активными фоновыми процессами** никогда не сбрасываются автоматически, независимо от политики.
## Места хранения

| Что | Путь | Описание |
|------|------|-------------|
| SQLite‑база данных | `~/.hermes/state.db` | Все метаданные сессий + сообщения с поддержкой FTS5 |
| Сообщения шлюза | `~/.hermes/state.db` | SQLite — каноничное хранилище всех сообщений сессий |
| Индекс маршрутизации шлюза | `~/.hermes/sessions/sessions.json` | Сопоставляет ключи сессий активным идентификаторам (метаданные источника, флаги истечения) |

SQLite‑база данных использует режим WAL для одновременных чтений и единственного писателя, что хорошо подходит для многоплатформенной архитектуры шлюза.

:::note Legacy JSONL transcripts
Сессии, созданные до того, как `state.db` стал каноничным, могут оставлять
файлы `*.jsonl` в `~/.hermes/sessions/`. Они больше не записываются и не
читаются Hermes. Их можно безопасно удалить после проверки, что соответствующая сессия присутствует в `state.db`.
:::

### Схема базы данных

Ключевые таблицы в `state.db`:

- **sessions** — метаданные сессии (id, source, user_id, model, title, timestamps, token counts). У заголовков есть уникальный индекс (NULL‑заголовки допускаются, только ненулевые должны быть уникальны).
- **messages** — полная история сообщений (role, content, tool_calls, tool_name, token_count)
- **messages_fts** — виртуальная таблица FTS5 для полнотекстового поиска по содержимому сообщений
## Истечение сессии и очистка

### Автоматическая очистка

- Сессии шлюза автоматически сбрасываются в соответствии с настроенной политикой сброса
- Перед сбросом агент сохраняет память и навыки из истекающей сессии
- Авто‑обрезка по согласованию: когда `sessions.auto_prune` равно `true`, завершённые сессии старше `sessions.retention_days` (по умолчанию 90) удаляются при запуске CLI/шлюза
- После обрезки, когда действительно удалены строки, `state.db` проходит `VACUUM` для освобождения места на диске (SQLite не уменьшает файл при обычном `DELETE`)
- Обрезка выполняется не чаще, чем раз в `sessions.min_interval_hours` (по умолчанию 24); метка времени последнего запуска хранится внутри самого `state.db`, поэтому она общая для всех процессов Hermes в одном `HERMES_HOME`

По умолчанию **выключено** — история сессий ценна для восстановления через `session_search`, и её тихое удаление может удивить пользователей. Включи в `~/.hermes/config.yaml`:

```yaml
sessions:
  auto_prune: true          # opt in — default is false
  retention_days: 90        # keep ended sessions this many days
  vacuum_after_prune: true  # reclaim disk space after a pruning sweep
  min_interval_hours: 24    # don't re-run the sweep more often than this
```

Активные сессии никогда не удаляются автоматически, независимо от их возраста.

### Ручная очистка

```bash
# Prune sessions older than 90 days
hermes sessions prune

# Delete a specific session
hermes sessions delete <session_id>

# Export before pruning (backup)
hermes sessions export backup.jsonl
hermes sessions prune --older-than 30 --yes
```

:::tip
База данных растёт медленно (обычно : 10‑15 МБ для сотен сессий), а история сессий обеспечивает восстановление через `session_search` в прошлых разговорах, поэтому авто‑обрезка по умолчанию отключена. Включи её, если у тебя тяжёлая нагрузка шлюза/cron и `state.db` заметно влияет на производительность (наблюдаемый сбой: `state.db` 384 МБ с ~1000 сессиями замедляет вставки FTS5 и список `/resume`). Используй `hermes sessions prune` для одноразовой очистки без включения автоматической обрезки.
:::
