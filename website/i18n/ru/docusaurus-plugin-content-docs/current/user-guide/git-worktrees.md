# Git Worktrees

Hermes Agent часто используется в больших, длительно живущих репозиториях. Когда нужно:

- Запустить **несколько агентов параллельно** в одном проекте, или
- Держать экспериментальные рефакторинги изолированными от основной ветки,

Git **worktrees** — самый безопасный способ дать каждому агенту собственный чек‑аут без дублирования всего репозитория.

Эта страница показывает, как комбинировать worktrees с Hermes, чтобы каждая сессия имела чистый, изолированный рабочий каталог.

## Почему использовать Worktrees с Hermes?

Hermes рассматривает **текущий рабочий каталог** как корень проекта:

- CLI: каталог, из которого запускается `hermes` или `hermes chat`
- Шлюзы обмена сообщениями: каталог, задаваемый переменной `MESSAGING_CWD`

Если запустить несколько агентов в **одном и том же чек‑ауте**, их изменения могут конфликтовать:

- Один агент может удалить или перезаписать файлы, которые использует другой.
- Труднее понять, какие изменения относятся к какому эксперименту.

С worktrees каждый агент получает:

- **Свою ветку и рабочий каталог**
- **Свою историю Checkpoint Manager** для `/rollback`

См. также: [Checkpoints and /rollback](./checkpoints-and-rollback.md).

## Быстрый старт: создание Worktree

Из главного репозитория (содержащего `.git/`) создайте новый worktree для ветки‑фичи:

```bash
# From the main repo root
cd /path/to/your/repo

# Create a new branch and worktree in ../repo-feature
git worktree add ../repo-feature feature/hermes-experiment
```

Это создаст:

- Новый каталог: `../repo-feature`
- Новую ветку: `feature/hermes-experiment`, проверенную в этом каталоге

Теперь можно перейти в новый worktree и запустить Hermes:

```bash
cd ../repo-feature

# Start Hermes in the worktree
hermes
```

Hermes:

- Будет видеть `../repo-feature` как корень проекта.
- Будет использовать этот каталог для файлов контекста, правок кода и инструментов.
- Будет вести **отдельную историю контрольных точек** для `/rollback`, ограниченную этим worktree.

## Запуск нескольких агентов параллельно

Можно создать несколько worktree, каждый со своей веткой:

```bash
cd /path/to/your/repo

git worktree add ../repo-experiment-a feature/hermes-a
git worktree add ../repo-experiment-b feature/hermes-b
```

В отдельных терминалах:

```bash
# Terminal 1
cd ../repo-experiment-a
hermes

# Terminal 2
cd ../repo-experiment-b
hermes
```

Каждый процесс Hermes:

- Работает в своей ветке (`feature/hermes-a` vs `feature/hermes-b`).
- Записывает контрольные точки в репозиторий‑теневую хеш‑структуру (полученную из пути worktree).
- Может использовать `/rollback` независимо, не влияя на другой процесс.

Это особенно полезно, когда:

- Выполняются пакетные рефакторинги.
- Пробуются разные подходы к одной задаче.
- Пары CLI + gateway‑сессий работают с одним и тем же upstream‑репозиторием.

## Безопасное удаление Worktrees

Когда эксперимент завершён:

1. Реши, сохранять работу или удалить её.
2. Если нужно сохранить:
   - Слей ветку в основную ветку как обычно.
3. Удали worktree:

```bash
cd /path/to/your/repo

# Remove the worktree directory and its reference
git worktree remove ../repo-feature
```

Примечания:

- `git worktree remove` откажется удалять worktree с несохранёнными изменениями, если не указать принудительный флаг.
- Удаление worktree **не** удаляет ветку автоматически; ветку можно удалить или оставить обычными командами `git branch`.
- Данные контрольных точек Hermes в `~/.hermes/checkpoints/` не удаляются автоматически при удалении worktree, но обычно занимают очень мало места.

## Лучшие практики

- **Один worktree на каждый эксперимент Hermes**
  - Создавай отдельную ветку/worktree для каждого значительного изменения.
  - Это делает диффы сфокусированными, а PR‑ы небольшими и удобными для ревью.
- **Названия веток отражают эксперимент**
  - например `feature/hermes-checkpoints-docs`, `feature/hermes-refactor-tests`.
- **Часто коммить**
  - Делай git‑коммиты для крупных вех.
  - Используй [checkpoints and /rollback](./checkpoints-and-rollback.md) как страховку для правок, сделанных инструментами, между коммитами.
- **Не запускай Hermes из корня «голого» репозитория, когда используешь worktrees**
  - Предпочитай каталоги worktree, чтобы каждый агент имел чёткую область.

## Использование `hermes -w` (автоматический режим Worktree)

Hermes имеет встроенный флаг `-w`, который **автоматически создаёт временный git worktree** со своей веткой. Не нужно вручную настраивать worktrees — просто `cd` в репозиторий и запусти:

```bash
cd /path/to/your/repo
hermes -w
```

Hermes:

- Создаст временный worktree в `.worktrees/` внутри репозитория.
- Проверит изолированную ветку (например `hermes/hermes-<hash>`).
- Запустит полную CLI‑сессию внутри этого worktree.

Это самый простой способ получить изоляцию worktree. Можно также комбинировать с единственным запросом:

```bash
hermes -w -q "Fix issue #123"
```

Для параллельных агентов открой несколько терминалов и запусти `hermes -w` в каждом — каждый запуск получит собственный worktree и ветку автоматически.

## Итоги

- Используй **git worktrees**, чтобы каждая сессия Hermes имела собственный чистый чек‑аут.
- Используй **ветки** для фиксации высокоуровневой истории экспериментов.
- Используй **контрольные точки + `/rollback`** для восстановления после ошибок внутри каждого worktree.

Эта комбинация даёт тебе:

- Жёсткие гарантии, что разные агенты и эксперименты не будут мешать друг другу.
- Быстрые циклы итераций с лёгким откатом плохих правок.
- Чистые, удобные для ревью pull‑request‑ы.
