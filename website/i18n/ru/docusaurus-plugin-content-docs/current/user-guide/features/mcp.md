# MCP (Протокол контекста модели)

MCP позволяет Hermes Agent подключаться к внешним серверам инструментов, чтобы агент мог использовать инструменты, находящиеся за пределами Hermes — GitHub, базы данных, файловые системы, браузерные стеки, внутренние API и многое другое.

Если ты когда‑нибудь хотел, чтобы Hermes использовал уже существующий где‑то инструмент, MCP обычно — самый чистый способ сделать это.
## Что дает MCP

- Доступ к внешним экосистемам инструментов без необходимости сначала писать собственный инструмент Hermes
- Локальные stdio‑серверы и удалённые HTTP‑серверы MCP в одной конфигурации
- Автоматическое обнаружение и регистрация инструментов при запуске
- Утилитарные обёртки для ресурсов MCP и подсказок, когда это поддерживается сервером
- Фильтрация по серверу, позволяющая раскрывать только те инструменты MCP, которые ты действительно хочешь, чтобы Hermes видел
## Быстрый старт

1. Установи поддержку MCP (уже включена, если ты использовал стандартный скрипт установки):

```bash
cd ~/.hermes/hermes-agent
uv pip install -e ".[mcp]"
```

2. Добавь сервер MCP в `~/.hermes/config.yaml`:

```yaml
mcp_servers:
  filesystem:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/projects"]
```

3. Запусти Hermes:

```bash
hermes chat
```

4. Попроси Hermes использовать возможность, реализованную через MCP.

Например:

```text
List the files in /home/user/projects and summarize the repo structure.
```

Hermes обнаружит инструменты сервера MCP и будет использовать их как любые другие инструменты.
## Каталог: установка в один клик для одобренных Nous MCP

Hermes поставляет отобранный каталог серверов MCP, которые сотрудники Nous проверили
и объединили. По умолчанию они отключены — устанавливай только то, что действительно
нужно.

```bash
hermes mcp                # interactive picker (default)
hermes mcp catalog        # plain-text list, scriptable
hermes mcp install n8n    # install a catalog entry by name
```

Панель выбора показывает каждую запись с её текущим статусом:

```
n8n          available              Manage and inspect n8n workflows from Hermes
linear       enabled                Linear issue/project management (remote OAuth)
github       installed (disabled)   GitHub repo + PR tools
```

Нажми `Enter` на строке, чтобы установить (и пройти процесс ввода необходимых учётных данных),
включить, отключить или удалить. Записи каталога хранятся в
`optional-mcps/` репозитория hermes-agent — наличие в этой директории означает
одобрение Nous. Уровня подачи сообществом нет; записи добавляются путём
слияния PR.

Записи каталога могут требовать:

- **API key** — Hermes запрашивает её во время установки и записывает значение в
  `~/.hermes/.env`. Не секретные значения (базовые URL) также попадают в тот же файл.
- **OAuth** (удалённый MCP) — записывается как `auth: oauth` в твоей конфигурации; клиент MCP
  открывает браузер при первом подключении.
- **OAuth** (провайдер третьей стороны, например Google/GitHub) — Hermes предлагает выполнить
  `hermes auth <provider>`, если ты ещё не прошёл аутентификацию.

### Выбор инструментов во время установки

После настройки учётных данных Hermes опрашивает сервер MCP, чтобы перечислить все
инструменты, которые он предоставляет, и показывает чек‑лист:

```
Select tools for 'linear' (SPACE toggle, ENTER confirm)
  [x] find_issues       Find issues matching a query
  [x] get_issue         Get a single issue
  [x] create_issue      Create a new issue
  [ ] delete_workspace  Delete a Linear workspace
  ...
```

Строки, отмеченные по умолчанию, берутся из:

1. **Твоего предыдущего выбора**, если ты уже устанавливал эту запись (переустановки
   сохраняют выбранные инструменты — значения по умолчанию из манифеста их не переопределяют)
2. **`tools.default_enabled` манифеста**, если запись его объявляет (некоторые
   записи каталога предварительно отсекают мутирующие или редко используемые инструменты)
3. **Всего**, если ни одно из вышеуказанных не применимо

Подтверди чек‑лист клавишей `Enter`. В `mcp_servers.<name>.tools.include` попадут только
отмеченные инструменты. Если ты выберешь всё, фильтр не будет записан
(самая чистая форма конфигурации, идентичное поведение).

**Если опрос не удался** (сервер недоступен, OAuth ещё не завершён,
служба‑поставщик не запущена), установка всё равно считается успешной: применяется
`tools.default_enabled` из манифеста напрямую (если объявлен), либо фильтр не записывается
(если нет). После того как сервер станет доступен, запусти
`hermes mcp configure <name>` заново, чтобы уточнить настройки.

### Модель доверия

Установка записи каталога выполняет всё, что указано в манифесте — `git clone`,
команды `bootstrap` записи (`pip install`, `npm install` и т.п.), и
в конечном итоге собственный код сервера MCP. Манифесты проходят проверку PR
в репозитории hermes-agent, поэтому Nous проверил каждую запись перед её
публикацией — **но всё равно стоит прочитать манифест перед установкой**, особенно
поле `source:` с репозиторием, команды `install.bootstrap:` и любые вызовы
`transport.command:`.

Манифесты находятся по пути
[`optional-mcps/<name>/manifest.yaml`](https://github.com/NousResearch/hermes-agent/tree/main/optional-mcps)
на GitHub. Панель выбора также выводит URL из `source:` манифеста во время установки,
чтобы ты мог быстро проверить исходный репозиторий.

### Совместимость версий манифеста

Манифесты фиксируют `manifest_version`. Каталог совместим вперёд: если PR добавит запись
с более новой `manifest_version`, чем понимает установленный у тебя Hermes, панель
выдаст предупреждение (`⚠ '<name>' requires a newer Hermes`) для этой записи вместо
тихого её скрытия. Запусти `hermes update`, чтобы установить последнюю версию Hermes,
когда увидишь такое сообщение.

### Подстановка переменных окружения `${ENV_VAR}` во время выполнения

Внутри `transport.command`, `transport.args`, `transport.url` и `headers` записи
манифеста плейсхолдеры `${VAR}` подставляются в момент подключения к серверу
из переменных окружения (включая всё из `~/.hermes/.env`). Это удобно, когда запись
каталога должна сослаться на значение, заданное пользователем где‑то ещё — например
`${HOME}/foo` или `${MY_PROVIDER_TOKEN}`.

Обрати внимание, что это отличается от `${INSTALL_DIR}` в манифестах каталога,
который подставляется во время установки в путь, куда каталог клонировал репозиторий записи.

### Обновление выбора инструментов позже

```bash
hermes mcp configure linear
```

Повторно открывает тот же чек‑лист с текущим выбором, отмеченным заранее. Используй
это, когда нужно включить дополнительные инструменты или когда сервер добавил новые
инструменты, которые ты хочешь активировать.

### Обновление манифеста каталога

MCP никогда не обновляются автоматически. Запусти `hermes mcp install <name>` заново,
чтобы обновить после обновления Hermes, если версия манифеста изменилась.

Чтобы добавить MCP в каталог, открой PR против
[`optional-mcps/`](https://github.com/NousResearch/hermes-agent/tree/main/optional-mcps).
## Два типа серверов MCP

### Stdio‑серверы

Stdio‑серверы работают как локальные подпроцессы и общаются через stdin/stdout.

```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
```

Используй stdio‑серверы, когда:
- сервер установлен локально;
- нужен низкозадержочный доступ к локальным ресурсам;
- ты следуешь документации MCP‑сервера, где указаны `command`, `args` и `env`.

### HTTP‑серверы

HTTP‑серверы MCP — это удалённые конечные точки, к которым Hermes подключается напрямую.

```yaml
mcp_servers:
  remote_api:
    url: "https://mcp.example.com/mcp"
    headers:
      Authorization: "Bearer ***"
```

Используй HTTP‑серверы, когда:
- сервер MCP размещён где‑то ещё;
- в твоей организации открыты внутренние конечные точки MCP;
- ты не хочешь, чтобы Hermes запускал локальный подпроцесс для этой интеграции.

### HTTP‑серверы с аутентификацией OAuth

Большинство размещённых серверов MCP (Linear, Sentry, Atlassian, Asana, Figma, Stripe, …) требуют OAuth 2.1 вместо статического токена‑носителя. Установи `auth: oauth`, и Hermes выполнит обнаружение, динамическую регистрацию клиента, PKCE, обмен токенов, их обновление и повышенную аутентификацию через MCP Python SDK.

```yaml
mcp_servers:
  linear:
    url: "https://mcp.linear.app/mcp"
    auth: oauth
```

При первом подключении Hermes выводит URL для авторизации, открывает твой браузер, если это возможно, и ждёт обратного вызова OAuth на локальном loopback‑порту. Токены кэшируются в `~/.hermes/mcp-tokens/<server>.json` с правами 0o600; последующие запуски используют их тихо, пока не произойдёт ошибка обновления.

**Удалённые / безголовые хосты.** Когда Hermes работает на машине, отличной от той, где открыт браузер, обратный вызов loopback не может достичь твоего ноутбука. Есть два способа завершить поток:

- **Вставка вручную (без настройки):** в интерактивном терминале Hermes выводит «Or paste the redirect URL here…» рядом с URL‑ом авторизации. Открой URL в браузере, одобри запрос, скопируй полный URL, на который перенаправил браузер (перенаправление покажет ошибку соединения — это ожидаемо), вставь его в приглашение. Работают и простые строки запроса `?code=…&state=…`.
- **SSH‑перенаправление порта:** `ssh -N -L <port>:127.0.0.1:<port> user@host` в отдельном терминале, затем позволь перенаправлению пройти обычным образом.

См. [OAuth over SSH / Remote Hosts](../../guides/oauth-over-ssh.md#mcp-servers) для полного руководства, включая серверы без DCR (например, Slack), предварительно зарегистрированные `client_id`/`client_secret`, настройку областей и повторную аутентификацию через `hermes mcp login <server>`.

**Подводный камень — гонка авто‑перезагрузки конфигурации.** Когда ты редактируешь `~/.hermes/config.yaml` из работающей сессии Hermes, CLI автоматически перезагружает соединения MCP с таймаутом 30 с. Это недостаточно для интерактивного OAuth‑потока. Добавь запись, затем запусти `hermes mcp login <server>` в новом терминале — он будет ждать полные 5 минут, пока ты завершишь аутентификацию.
## Справочник базовой конфигурации

Hermes читает конфигурацию MCP из `~/.hermes/config.yaml` в разделе `mcp_servers`.

### Общие ключи

| Ключ | Тип | Значение |
|---|---|---|
| `command` | string | Исполняемый файл для stdio MCP‑сервера |
| `args` | list | Аргументы для stdio‑сервера |
| `env` | mapping | Переменные окружения, передаваемые stdio‑серверу |
| `url` | string | HTTP‑конечная точка MCP |
| `headers` | mapping | HTTP‑заголовки для удалённых серверов |
| `timeout` | number | Таймаут вызова инструмента |
| `connect_timeout` | number | Таймаут начального соединения |
| `enabled` | bool | Если `false`, Hermes полностью пропускает сервер |
| `supports_parallel_tool_calls` | bool | Если `true`, инструменты с этого сервера могут выполняться параллельно |
| `tools` | mapping | Фильтрация инструментов и политика их использования для конкретного сервера |

### Минимальный пример stdio

```yaml
mcp_servers:
  filesystem:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"]
```

### Минимальный пример HTTP

```yaml
mcp_servers:
  company_api:
    url: "https://mcp.internal.example.com"
    headers:
      Authorization: "Bearer ***"
```
## Встроенные пресеты

Для известных серверов MCP команда `hermes mcp add` принимает флаг `--preset`, который заполняет параметры транспорта, чтобы тебе не пришлось искать команду и аргументы вручную. Пресет задаёт только значения по умолчанию — всё остальное (переменные окружения, заголовки, фильтрация), что ты укажешь в той же командной строке, будет иметь приоритет.

| Пресет | Что он подключает |
|---|---|
| `codex` | MCP‑сервер CLI Codex (`codex mcp-server` через stdio). Требует наличие CLI `codex` в PATH. |

```bash
# Add Codex CLI as an MCP server in one line
hermes mcp add codex --preset codex
```

Это эквивалентно следующему:

```yaml
mcp_servers:
  codex:
    command: "codex"
    args: ["mcp-server"]
```

Можно использовать любое локальное имя (`hermes mcp add my-codex --preset codex` подходит); пресет только задаёт значения по умолчанию для `command`/`args`.
## Как Hermes регистрирует инструменты MCP

Hermes добавляет префикс к инструментам MCP, чтобы они не конфликтовали со встроенными именами:

```text
mcp_<server_name>_<tool_name>
```

Примеры:

| Сервер | Инструмент MCP | Зарегистрированное имя |
|---|---|---|
| `filesystem` | `read_file` | `mcp_filesystem_read_file` |
| `github` | `create-issue` | `mcp_github_create_issue` |
| `my-api` | `query.data` | `mcp_my_api_query_data` |

На практике обычно не требуется вызывать имя с префиксом вручную — Hermes видит инструмент и выбирает его в процессе обычного рассуждения.
## Утилиты MCP

Если поддерживается, Hermes также регистрирует утилиты для работы с ресурсами MCP и подсказками:

- `list_resources`
- `read_resource`
- `list_prompts`
- `get_prompt`

Эти утилиты регистрируются для каждого сервера с тем же шаблоном префикса, например:

- `mcp_github_list_resources`
- `mcp_github_get_prompt`

### Важно

Эти утилиты теперь учитывают возможности:
- Hermes регистрирует утилиты ресурсов только если сессия MCP действительно поддерживает операции с ресурсами
- Hermes регистрирует утилиты подсказок только если сессия MCP действительно поддерживает операции с подсказками

Поэтому сервер, который предоставляет вызываемые инструменты, но не имеет ресурсов или подсказок, не получит эти дополнительные обёртки.
## Фильтрация по серверу

Ты можешь управлять тем, какие инструменты каждый сервер MCP вносит в Hermes, позволяя тонко настраивать пространство имён твоих инструментов.

### Отключить сервер полностью

```yaml
mcp_servers:
  legacy:
    url: "https://mcp.legacy.internal"
    enabled: false
```

Если `enabled: false`, Hermes полностью игнорирует сервер и даже не пытается установить соединение.

### Белый список инструментов сервера

```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
    tools:
      include: [create_issue, list_issues]
```

Регистрируются только инструменты этого сервера MCP.

### Чёрный список инструментов сервера

```yaml
mcp_servers:
  stripe:
    url: "https://mcp.stripe.com"
    tools:
      exclude: [delete_customer]
```

Регистрируются все инструменты сервера, кроме исключённых.

### Правило приоритета

Если оба присутствуют:

```yaml
tools:
  include: [create_issue]
  exclude: [create_issue, delete_issue]
```

`include` имеет приоритет.

### Также фильтровать утилитарные инструменты

Ты также можешь отдельно отключить добавленные Hermes обёртки утилит:

```yaml
mcp_servers:
  docs:
    url: "https://mcp.docs.example.com"
    tools:
      prompts: false
      resources: false
```

Это означает:
- `tools.resources: false` отключает `list_resources` и `read_resource`
- `tools.prompts: false` отключает `list_prompts` и `get_prompt`

### Полный пример

```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
    tools:
      include: [create_issue, list_issues, search_code]
      prompts: false

  stripe:
    url: "https://mcp.stripe.com"
    headers:
      Authorization: "Bearer ***"
    tools:
      exclude: [delete_customer]
      resources: false

  legacy:
    url: "https://mcp.legacy.internal"
    enabled: false
```
## Что происходит, если всё отфильтровать?

Если твоя конфигурация отфильтрует все вызываемые инструменты и отключит или опустит все поддерживаемые утилиты, Hermes не создаёт пустой набор runtime MCP‑toolset для этого сервера.

Это сохраняет список инструментов чистым.
## Поведение во время выполнения

### Время обнаружения

Hermes обнаруживает серверы MCP при запуске и регистрирует их инструменты в обычный реестр инструментов.

### Динамическое обнаружение инструментов

Серверы MCP могут уведомлять Hermes об изменении доступных инструментов во время выполнения, отправляя уведомление `notifications/tools/list_changed`. Когда Hermes получает это уведомление, он автоматически повторно запрашивает список инструментов сервера и обновляет реестр — без необходимости вручную выполнять `/reload-mcp`.

Это полезно для серверов MCP, чьи возможности меняются динамически (например, сервер, который добавляет инструменты при загрузке новой схемы базы данных, или удаляет инструменты, когда сервис выходит из сети).

Обновление защищено блокировкой, поэтому быстрые последовательные уведомления от одного и того же сервера не вызывают перекрывающихся обновлений. Уведомления об изменении подсказок и ресурсов (`prompts/list_changed`, `resources/list_changed`) получаются, но пока не обрабатываются.

### Перезагрузка

Если ты изменил конфигурацию MCP, используй:

```text
/reload-mcp
```

Это перезагружает серверы MCP из конфигурации и обновляет список доступных инструментов. Для изменений инструментов во время выполнения, инициированных самим сервером, смотри раздел [Dynamic Tool Discovery](#dynamic-tool-discovery) выше.

### Наборы инструментов

Каждый настроенный сервер MCP также создаёт **runtime набор инструментов**, когда он вносит хотя бы один зарегистрированный инструмент:

```text
mcp-<server>
```

Это упрощает работу с серверами MCP на уровне наборов инструментов.
## Модель безопасности

### Фильтрация среды stdio

Для серверов stdio Hermes не передаёт слепо всю твою оболочечную среду.

Передаётся только явно настроенный `env` и безопасный базовый набор. Это снижает риск случайной утечки секретов.

### Управление раскрытием на уровне конфигурации

Новая поддержка фильтрации также служит средством безопасности:
- отключи опасные инструменты, которые ты не хочешь показывать модели;
- открой только минимальный белый список для чувствительного сервера;
- отключи обёртки ресурсов/подсказок, если не хочешь раскрывать эту поверхность.
## Примеры сценариев использования

### GitHub‑сервер с минимальным интерфейсом управления задачами

```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
    tools:
      include: [list_issues, create_issue, update_issue]
      prompts: false
      resources: false
```

Используй так:

```text
Show me open issues labeled bug, then draft a new issue for the flaky MCP reconnection behavior.
```

### Stripe‑сервер со снятыми опасными действиями

```yaml
mcp_servers:
  stripe:
    url: "https://mcp.stripe.com"
    headers:
      Authorization: "Bearer ***"
    tools:
      exclude: [delete_customer, refund_payment]
```

Используй так:

```text
Look up the last 10 failed payments and summarize common failure reasons.
```

### Файловый сервер для корня единственного проекта

```yaml
mcp_servers:
  project_fs:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/my-project"]
```

Используй так:

```text
Inspect the project root and explain the directory layout.
```
## Устранение неполадок

### MCP‑сервер не подключается

Проверь:

```bash
# Verify MCP deps are installed (already included in standard install)
cd ~/.hermes/hermes-agent && uv pip install -e ".[mcp]"

node --version
npx --version
```

Затем проверь конфигурацию и перезапусти Hermes.

### Инструменты не отображаются

Возможные причины:
- сервер не смог подключиться
- обнаружение не удалось
- твоя конфигурация фильтра исключила инструменты
- возможность утилиты отсутствует на этом сервере
- сервер отключён с `enabled: false`

Если ты намеренно применяешь фильтрацию, это ожидаемо.

### Почему не появились утилиты ресурсов или подсказок?

Потому что Hermes теперь регистрирует эти обёртки только когда оба условия истинны:
1. твоя конфигурация их разрешает
2. сессия сервера действительно поддерживает эту возможность

Это преднамеренно и обеспечивает достоверность списка инструментов.
## Параллельные вызовы инструментов

По умолчанию инструменты MCP выполняются последовательно — по одному. Если твой MCP‑сервер предоставляет инструменты, безопасные для одновременного выполнения (например, запросы только для чтения, независимые вызовы API), ты можешь включить параллельное исполнение:

```yaml
mcp_servers:
  docs:
    command: "docs-server"
    supports_parallel_tool_calls: true
```

Когда `supports_parallel_tool_calls` имеет значение `true`, Hermes может выполнять несколько инструментов с этого сервера одновременно в рамках одной партии вызовов инструментов, так же как это делается для встроенных инструментов только для чтения (`web_search`, `read_file` и т.д.).

:::caution
Включай параллельные вызовы только для MCP‑серверов, чьи инструменты безопасны при одновременном запуске. Если инструменты читают и записывают общие состояния, файлы, базы данных или внешние ресурсы, проверь условия гонки чтения/записи перед включением этой настройки.
:::
## Поддержка sampling MCP

Серверы MCP могут запрашивать вывод LLM у Hermes через протокол `sampling/createMessage`. Это позволяет серверу MCP попросить Hermes сгенерировать текст от его имени — полезно для серверов, которым нужны возможности LLM, но у них нет собственного доступа к модели.

Sampling **включён по умолчанию** для всех серверов MCP (когда SDK MCP поддерживает его). Настраивай его для каждого сервера в разделе `sampling`:

```yaml
mcp_servers:
  my_server:
    command: "my-mcp-server"
    sampling:
      enabled: true            # Enable sampling (default: true)
      model: "openai/gpt-4o"  # Override model for sampling requests (optional)
      max_tokens_cap: 4096     # Max tokens per sampling response (default: 4096)
      timeout: 30              # Timeout in seconds per request (default: 30)
      max_rpm: 10              # Rate limit: max requests per minute (default: 10)
      max_tool_rounds: 5       # Max tool-use rounds in sampling loops (default: 5)
      allowed_models: []       # Allowlist of model names the server may request (empty = any)
      log_level: "info"        # Audit log level: debug, info, or warning (default: info)
```

Обработчик sampling включает ограничитель скорости скользящего окна, тайм‑ауты для каждого запроса и ограничения глубины цикла инструментов, чтобы предотвратить неконтролируемое использование. Метрики (количество запросов, ошибки, использованные токены) отслеживаются для каждого экземпляра сервера.

Чтобы отключить sampling для конкретного сервера:

```yaml
mcp_servers:
  untrusted_server:
    url: "https://mcp.example.com"
    sampling:
      enabled: false
```
## Запуск Hermes в качестве MCP‑сервера

Помимо подключения **к** MCP‑серверам, Hermes также может **являться** MCP‑сервером. Это позволяет другим агентам с поддержкой MCP (Claude Code, Cursor, Codex или любому MCP‑клиенту) использовать возможности обмена сообщениями Hermes — просматривать списки разговоров, читать историю сообщений и отправлять сообщения на всех подключённых платформах.

### Когда использовать это

- Ты хочешь, чтобы Claude Code, Cursor или другой кодирующий агент отправлял и читал сообщения Telegram/Discord/Slack через Hermes.
- Тебе нужен один MCP‑сервер, который одновременно соединяет все подключённые к Hermes платформы обмена сообщениями.
- У тебя уже запущен шлюз Hermes с подключёнными платформами.

### Быстрый старт

```bash
hermes mcp serve
```

Это запускает stdio MCP‑сервер. MCP‑клиент (не ты) управляет жизненным циклом процесса.

### Конфигурация MCP‑клиента

Добавь Hermes в конфигурацию своего MCP‑клиента. Например, в `~/.claude/claude_desktop_config.json` для Claude Code:

```json
{
  "mcpServers": {
    "hermes": {
      "command": "hermes",
      "args": ["mcp", "serve"]
    }
  }
}
```

Или если ты установил Hermes в определённое место:

```json
{
  "mcpServers": {
    "hermes": {
      "command": "/home/user/.hermes/hermes-agent/venv/bin/hermes",
      "args": ["mcp", "serve"]
    }
  }
}
```

### Доступные инструменты

MCP‑сервер предоставляет 10 инструментов, соответствующих поверхности мостов каналов OpenClaw плюс специфичный для Hermes браузер каналов:

| Инструмент | Описание |
|------|-------------|
| `conversations_list` | Вывести список активных разговоров. Фильтровать по платформе или искать по имени. |
| `conversation_get` | Получить подробную информацию об одном разговоре по ключу сессии. |
| `messages_read` | Прочитать недавнюю историю сообщений для разговора. |
| `attachments_fetch` | Извлечь нетекстовые вложения (изображения, медиа) из конкретного сообщения. |
| `events_poll` | Опросить новые события разговора с позиции курсора. |
| `events_wait` | Длинный опрос / блокировка до появления следующего события (почти в реальном времени). |
| `messages_send` | Отправить сообщение через платформу (например, `telegram:123456`, `discord:#general`). |
| `channels_list` | Вывести список доступных целей обмена сообщениями на всех платформах. |
| `permissions_list_open` | Вывести список ожидающих одобрения запросов, обнаруженных во время этой сессии моста. |
| `permissions_respond` | Разрешить или отклонить ожидающий запрос одобрения. |

### Система событий

MCP‑сервер включает живой мост событий, который опрашивает базу данных сессий Hermes на предмет новых сообщений. Это даёт MCP‑клиентам почти мгновенное представление о входящих разговорах:

```
# Poll for new events (non-blocking)
events_poll(after_cursor=0)

# Wait for next event (blocks up to timeout)
events_wait(after_cursor=42, timeout_ms=30000)
```

Типы событий: `message`, `approval_requested`, `approval_resolved`

Очередь событий хранится в памяти и запускается при подключении моста. Более старые сообщения доступны через `messages_read`.

### Параметры

```bash
hermes mcp serve              # Normal mode
hermes mcp serve --verbose    # Debug logging on stderr
```

### Как это работает

MCP‑сервер читает данные разговоров напрямую из хранилища сессий Hermes (`~/.hermes/sessions/sessions.json` и SQLite‑базы). Фоновый поток опрашивает базу данных на предмет новых сообщений и поддерживает очередь событий в памяти. Для отправки сообщений используется та же инфраструктура `send_message`, что и у самого агента Hermes.

Шлюз **не требуется** для операций чтения (список разговоров, чтение истории, опрос событий). Он **нужен** для операций отправки, поскольку адаптерам платформ требуются активные соединения.

### Текущие ограничения

- Встроенный `hermes mcp serve` сегодня предоставляет только **stdio‑only** MCP‑сервер. Если нужен HTTP‑MCP‑сервер, запусти отдельный адаптер — или, что гораздо чаще, используй **клиентскую** сторону MCP Hermes, которая уже поддерживает как stdio, так и HTTP (`url` + `headers` в `mcp_servers.yaml` / `config.yaml`; см. [HTTP servers](#http-servers) выше).
- Опрос событий с интервалом ~200 мс через оптимизированный по mtime опрос БД (пропускает работу, когда файлы не изменились).
- Протокол push‑уведомлений `claude/channel` пока не реализован.
- Отправка только текста (без медиа/вложений через `messages_send`).
