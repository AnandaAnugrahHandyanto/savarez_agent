# Канбан‑полосы работников

**Полоса работника** — это класс процесса, к которому диспетчер канбана может направлять задачи. Каждая полоса имеет идентификатор (строка `assignee`), механизм создания и контракт о том, что она должна делать с задачей после создания.

Эта страница представляет контракт. Она предназначена для двух аудиторий:

- **Операторы**, выбирающие, какие полосы подключать к доске (какие профили создавать, какие `assignee` использовать).
- **Авторы плагинов / интеграций**, желающие добавить новую форму полосы (CLI‑работник, оборачивающий Codex / Claude Code / OpenCode, контейнеризованный работник‑ревью, сервис, не относящийся к Hermes, который получает задачи через API).

Если ты пишешь сам код работника — агент, который работает *внутри* полосы — смотри навык [`kanban-worker`](https://github.com/NousResearch/hermes-agent/blob/main/skills/devops/kanban-worker/SKILL.md) для более подробного описания процедур.
## Иерархия

```text
Hermes Kanban  =  canonical task lifecycle + audit trail
Worker lane    =  implementation executor for one assigned card
Reviewer       =  human or human-proxy that gates "done"
GitHub PR      =  upstreamable artifact (optional, for code lanes)
```

Hermes Kanban владеет истинным состоянием жизненного цикла — `ready` → `running` → `blocked` / `done` / `archived`. Рабочие lanes выполняют работу, но не владеют этим состоянием; всё, что они делают, возвращается через ядро kanban с помощью инструментов `kanban_*` (или, для внешних работников, не являющихся Hermes, через API). Рецензенты контролируют переход от «code change written» к «task done».
## Что предоставляет lane

Чтобы lane мог работать как kanban‑worker, интеграция должна предоставлять три вещи:

### 1. Строка assignee

Диспетчер сопоставляет `task.assignee` либо с именем профиля Hermes (форма lane по умолчанию), либо с зарегистрированным не‑spawnable‑идентификатором (форма lane плагина — см. [Adding an external CLI worker lane](#adding-an-external-cli-worker-lane) ниже). Задачи, для которых assignee не удалось разрешить, остаются в `ready` с событием `skipped_nonspawnable`, чтобы оператор доски мог их исправить; они не отбрасываются молча и не выполняются произвольным запасным (fallback) вариантом.

### 2. Механизм spawn

Для lane‑профилей Hermes диспетчерский `_default_spawn` запускает `hermes -p <assignee> chat -q <prompt>` (или эквивалентную форму модуля, когда shim `hermes` отсутствует в `$PATH`) внутри закреплённого рабочего пространства задачи, задавая следующие переменные окружения:

| Variable | Carries |
|---|---|
| `HERMES_KANBAN_TASK` | идентификатор задачи, над которой работает воркер |
| `HERMES_KANBAN_DB` | абсолютный путь к файлу SQLite доски |
| `HERMES_KANBAN_BOARD` | slug доски |
| `HERMES_KANBAN_WORKSPACES_ROOT` | корень дерева рабочих пространств доски |
| `HERMES_KANBAN_WORKSPACE` | абсолютный путь к *этому* рабочему пространству задачи |
| `HERMES_KANBAN_RUN_ID` | идентификатор текущего запуска (для gate жизненного цикла) |
| `HERMES_KANBAN_CLAIM_LOCK` | строка блокировки claim (`<host>:<pid>:<uuid>`) |
| `HERMES_PROFILE` | имя собственного профиля воркера (для атрибуции автора `kanban_comment`) |
| `HERMES_TENANT` | пространство имён tenant, если у задачи есть tenant |

Для lane, не относящихся к Hermes (зарегистрированных через плагин), плагин предоставляет собственный вызываемый объект `spawn_fn`, который получает `task`, `workspace` и `board` и возвращает необязательный pid для обнаружения падения.

### 3. Завершитель жизненного цикла

Каждое claim должно завершаться ровно одним из вариантов:

- `kanban_complete(summary=..., metadata=...)` — задача успешно завершена, статус меняется на `done`.
- `kanban_block(reason=...)` — задача ждёт ввода человека, статус меняется на `blocked`. Диспетчер перезапускает её, когда выполнится `kanban_unblock`.
- Процесс воркера завершается без вызова инструмента. Ядро «перехватывает» его и генерирует `crashed` (PID умер), `gave_up` (сработал предохранитель последовательных неудач) или `timed_out` (превышено `max_runtime`). Это путь ошибки; здоровые воркеры здесь не заканчивают работу.

Ядро kanban гарантирует, что ровно один из этих вариантов завершает каждый запуск. Воркер, который не вызывает ни один из них и завершается нормально, считается падением.
## Outputs и конвенция «review-required»

Для большинства задач, меняющих код, работа не считается действительно *завершённой* в момент, когда исполнитель заканчивает — её должен проверить человек‑рецензент. Ядро kanban не принуждает к такому различию («задача, меняющая код» — размытое понятие, и принудительное блокирование вместо завершения для каждого код‑исполнителя нарушило бы потоки, где ревью не требуется). Это конвенция, наложенная сверху:

- **Block instead of complete**, с префиксом `reason` `review-required: `, чтобы панель управления / `hermes kanban show` отображала строку как ожидающую ревью.
- **Сначала помещать структурированные метаданные в `kanban_comment`**, так как `kanban_block` передаёт только человекочитаемый `reason`. Комментарии — надёжный канал аннотаций: туда помещаются все поля, важные для аудита (changed_files, tests_run, diff_path или PR‑url, decisions).
- **Рецензент либо одобряет и разблокирует**, что перезапускает исполнителя с веткой комментариев для последующих обсуждений; либо запрашивает изменения через другой комментарий, который следующий запуск исполнителя видит как часть контекста `kanban_show`.

Навык [`kanban-worker`](https://github.com/NousResearch/hermes-agent/blob/main/skills/devops/kanban-worker/SKILL.md) содержит примеры как для `kanban_complete` (по‑истине конечные задачи — исправление опечаток, изменения документации, написание исследований), так и для шаблона блокировки «review-required».
## Журналы и аудит‑трасса

Диспетчер записывает stdout/stderr каждого рабочего процесса задачи в `<board-root>/logs/<task_id>.log`. Журналы доступны для аудита из метаданных kanban:

- строки `task_runs` содержат `log_path`, код выхода (если доступен), сводку и метаданные.
- строки `task_events` фиксируют каждое изменение состояния (`promoted`, `claimed`, `heartbeat`, `completed`, `blocked`, `gave_up`, `crashed`, `timed_out`, `reclaimed`, `claim_extended`).
- `kanban_show` возвращает оба набора, поэтому рецензент (или последующий воркер), просматривающий задачу, получает полную историю без необходимости доступа к дашборду.

Дашборд отображает историю запусков со сводками, блоками метаданных и значками статуса завершения. Пользователи CLI могут выполнить `hermes kanban tail <task_id>`, чтобы следить в реальном времени, или `hermes kanban runs <task_id>` для получения списка исторических попыток.
## Существующие формы lane

### Hermes profile lane (по умолчанию)

Форма, которую сегодня принимает каждый kanban‑worker: ответственный — это имя профиля, диспетчер запускает `hermes -p <profile>`, worker автоматически загружает навык [`kanban-worker`](https://github.com/NousResearch/hermes-agent/blob/main/skills/devops/kanban-worker/SKILL.md) плюс блок системного подсказки `KANBAN_GUIDANCE` и использует инструменты `kanban_*` для завершения выполнения. Никакой настройки, кроме определения профиля, не требуется.

Когда ты создаёшь профили для своего флота, выбирай имена, соответствующие *роли*, которую ты хочешь, чтобы оркестратор направлял. Оркестратор (если он есть) обнаруживает имена твоих профилей через `hermes profile list` — нет фиксированного списка, который система предполагает (см. навык [`kanban-orchestrator`](https://github.com/NousResearch/hermes-agent/blob/main/skills/devops/kanban-orchestrator/SKILL.md) для стороны оркестратора).

### Orchestrator profile lane

Специализация profile lane: оркестратор — это профиль Hermes, набор инструментов которого включает `kanban`, но исключает `terminal` / `file` / `code` / `web` для реализации. Его задача — разбивать высокоуровневую цель на дочерние задачи с помощью `kanban_create` + `kanban_link` и отступать назад. Навык оркестратора кодирует правила против искушения.
## Добавление внешнего CLI‑рабочего канала

Подключение не‑Hermes CLI‑инструмента (Codex CLI, Claude Code CLI, OpenCode CLI, локального исполнителя модели кода и т.п.) в качестве канбан‑рабочей полосы *ещё не является готовым решением*. Функция создания процесса диспетчера (`spawn_fn`) заменяема (параметр `spawn_fn` в `dispatch_once`), и плагин может зарегистрировать собственный `spawn_fn` для не‑Hermes‑получателя, но сопутствующая работа по интеграции — обёртка кода выхода CLI в вызовы `kanban_complete` / `kanban_block`, сопоставление конвенций рабочего пространства/песочницы CLI с переменной окружения диспетчера `HERMES_KANBAN_WORKSPACE`, обработка аутентификации и политик для каждого CLI — всё ещё требует проектирования под каждую интеграцию.

Если ты планируешь добавить CLI‑полосу, открой issue с описанием конкретного CLI и рабочего процесса, который хочешь реализовать. Приведённый выше контракт — это ограничения, которым любой такой канал должен соответствовать; форма реализации (один плагин на каждый CLI или универсальный плагин‑исполнитель CLI, параметризованный конфигурацией) остаётся открытой.

Исторический issue по этому вопросу: [#19931](https://github.com/NousResearch/hermes-agent/issues/19931), а закрытый, но не смёрженный PR, специфичный для Codex: [#19924](https://github.com/NousResearch/hermes-agent/pull/19924) — они описывают первоначальное предложение архитектуры, но не привели к появлению исполнителя.
## Режимы отказов, которые обрабатывает диспетчер

Чтобы авторам lane не приходилось реализовывать их заново:

- **Stale claim TTL** — worker, который захватил задачу и затем никогда не отправлял heartbeat / не завершал / не блокировал её, будет освобождён после `DEFAULT_CLAIM_TTL_SECONDS` (по умолчанию 15 минут) — но только если процесс действительно завершился. Живой worker (медленная модель, проводящая более 20 минут в одном вызове LLM без инструментов) получает *расширенный* claim вместо завершения; освобождается только мёртвый PID.
- **Crashed worker** — worker, чей локальный PID на хосте исчез, обнаруживается `detect_crashed_workers` и завершается; задача увеличивает `consecutive_failures` и может автоматически блокироваться, когда срабатывает предохранитель.
- **Run-level retry** — когда задача пере‑запускается (post‑block, post‑crash, post‑reclaim), worker может использовать параметр `expected_run_id` у завершающих инструментов, чтобы быстро завершиться, если его собственный запуск уже был заменён.
- **Per‑task max runtime** — `task.max_runtime_seconds` жёстко ограничивает реальное время выполнения каждой попытки, независимо от живости PID. Позволяет поймать действительно зависшие workers, которые иначе продолжали бы работать благодаря продлению живого PID.
- **Stranded‑task detection** — готовая задача, для которой исполнитель никогда не создал claim в течение `kanban.stranded_threshold_seconds` (по умолчанию 30 минут), появляется в `hermes kanban diagnostics` как предупреждение `stranded_in_ready`. Серьёзность повышается до ошибки при 2× порога и до критической при 6×. Позволяет обнаружить опечатки в именах исполнителей, удалённые профили и недоступные внешние пулы workers в одном сигнале — без учёта идентичности, без необходимости создавать allowlist для каждой доски.
