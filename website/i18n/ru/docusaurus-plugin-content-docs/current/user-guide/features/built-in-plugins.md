# Встроенные плагины

Hermes поставляется с небольшим набором плагинов, включённых в репозиторий. Они находятся в `<repo>/plugins/<name>/` и загружаются автоматически вместе с пользовательскими плагинами в `~/.hermes/plugins/`. Они используют тот же интерфейс плагинов, что и сторонние плагины — хуки, инструменты, слеш‑команды — только находятся в дереве проекта.

См. страницу [Плагины](/user-guide/features/plugins) для общей системы плагинов и [Build a Hermes Plugin](/guides/build-a-hermes-plugin), чтобы написать свой.
## Как работает обнаружение

`PluginManager` сканирует четыре источника в указанном порядке:

1. **Bundled** — `<repo>/plugins/<name>/` (что описано на этой странице)
2. **User** — `~/.hermes/plugins/<name>/`
3. **Project** — `./.hermes/plugins/<name>/` (требуется `HERMES_ENABLE_PROJECT_PLUGINS=1`)
4. **Pip entry points** — `hermes_agent.plugins`

При конфликте имён приоритет имеет более поздний источник — плагин пользователя с именем `disk-cleanup` заменит встроенный.

`plugins/memory/` и `plugins/context_engine/` намеренно исключены из сканирования встроенных плагинов. Эти каталоги используют собственные пути обнаружения, поскольку провайдеры памяти и движки контекста являются единственными выбираемыми провайдерами, настраиваемыми через `hermes memory setup` / `context.engine` в конфигурации.
## Встроенные плагины доступны по желанию

Встроенные плагины поставляются отключёнными. Discovery находит их (они отображаются в `hermes plugins list` и в интерактивном UI `hermes plugins`), но ни один не загружается, пока ты явно не включишь их:

```bash
hermes plugins enable disk-cleanup
```

Или через `~/.hermes/config.yaml`:

```yaml
plugins:
  enabled:
    - disk-cleanup
```

Это тот же механизм, который используют пользовательские плагины. Встроенные плагины никогда не включаются автоматически — ни при свежей установке, ни для существующих пользователей, обновляющихся до более новой версии Hermes. Ты всегда явно включаешь их.

Чтобы отключить встроенный плагин снова:

```bash
hermes plugins disable disk-cleanup
# or: remove it from plugins.enabled in config.yaml
```
## Текущие поставки

Репозиторий поставляет эти встроенные плагины в каталоге `plugins/`. Все они опциональны — включай их через `hermes plugins enable <name>`.

| Плагин | Тип | Назначение |
|---|---|---|
| `disk-cleanup` | hooks + slash command | Автоматически отслеживает временные файлы и удаляет их в конце сессии |
| `security-guidance` | hooks | Сопоставляет опасный код в `write_file`/`patch` и добавляет предупреждение о безопасности (или блокирует) — 25 правил (форк Apache‑2.0 шаблонов Anthropic `claude-plugins-official`) |
| `observability/langfuse` | hooks | Трассирует ходы, вызовы LLM и инструменты в [Langfuse](https://langfuse.com) |
| `spotify` | backend (7 tools) | Нативное воспроизведение Spotify, очередь, поиск, плейлисты, альбомы, библиотека |
| `google_meet` | standalone | Присоединяется к Meet‑звонкам, транскрибирует в реальном времени, опциональный двунаправленный аудио‑дуплекс |
| `image_gen/openai` | image backend | Бэкенд генерации изображений OpenAI `gpt-image-2` (альтернатива FAL) |
| `image_gen/openai-codex` | image backend | Генерация изображений OpenAI через Codex OAuth |
| `image_gen/xai` | image backend | Бэкенд xAI `grok-2-image` |
| `hermes-achievements` | dashboard tab | Коллекционные значки в стиле Steam, генерируемые из реальной истории сессий Hermes |
| `kanban/dashboard` | dashboard tab | UI Kanban‑доски для мульти‑агентного диспетчера — задачи, комментарии, fan‑out, переключение досок. См. [Kanban Multi-Agent](./kanban.md). |

Поставщики памяти (`plugins/memory/*`) и движки контекста (`plugins/context_engine/*`) перечислены отдельно в [Memory Providers](./memory-providers.md) — ими управляют через `hermes memory` и `hermes plugins` соответственно. Полные детали по каждому плагину для двух длительно работающих hook‑плагинов ниже.

### disk-cleanup

Автоматически отслеживает и удаляет временные файлы, созданные во время сессий — тестовые скрипты, временные выводы, логи cron, устаревшие профили Chrome — без необходимости агенту помнить вызвать инструмент.

**Как это работает:**

| Hook | Поведение |
|---|---|
| `post_tool_call` | Когда `write_file` / `terminal` / `patch` создаёт файл, соответствующий `test_*`, `tmp_*` или `*.test.*` внутри `HERMES_HOME` или `/tmp/hermes-*`, он тихо отслеживается как `test` / `temp` / `cron-output`. |
| `on_session_end` | Если во время хода были автоматически отслежены тестовые файлы, запускает безопасную быструю очистку `quick` и записывает однострочное резюме. Иначе остаётся тихим. |

**Правила удаления:**

| Категория | Порог | Подтверждение |
|---|---|---|
| `test` | каждый конец сессии | Никогда |
| `temp` | > 7 дней с момента отслеживания | Никогда |
| `cron-output` | > 14 дней с момента отслеживания | Никогда |
| пустые каталоги под `HERMES_HOME` | всегда | Никогда |
| `research` | > 30 дней, кроме 10 самых новых | Всегда (только глубокая) |
| `chrome-profile` | > 14 дней с момента отслеживания | Всегда (только глубокая) |
| файлы > 500 МБ | никогда автоматически | Всегда (только глубокая) |

**Slash‑команда** — `/disk-cleanup` доступна и в CLI, и в сессиях шлюза:

```
/disk-cleanup status                     # breakdown + top-10 largest
/disk-cleanup dry-run                    # preview without deleting
/disk-cleanup quick                      # run safe cleanup now
/disk-cleanup deep                       # quick + list items needing confirmation
/disk-cleanup track <path> <category>    # manual tracking
/disk-cleanup forget <path>              # stop tracking (does not delete)
```

**Состояние** — всё хранится в `$HERMES_HOME/disk-cleanup/`:

| Файл | Содержание |
|---|---|
| `tracked.json` | Отслеживаемые пути с категорией, размером и меткой времени |
| `tracked.json.bak` | Атомарный бэкап вышеуказанного |
| `cleanup.log` | Журнал аудита каждого отслеживания / пропуска / отклонения / удаления (только добавление) |

**Безопасность** — очистка касается только путей под `HERMES_HOME` или `/tmp/hermes-*`. Монтирования Windows (`/mnt/c/...`) отклоняются. Известные каталоги верхнего уровня состояния (`logs/`, `memories/`, `sessions/`, `cron/`, `cache/`, `skills/`, `plugins/`, `disk-cleanup/` сам) никогда не удаляются, даже если пусты — свежая установка не будет «выпотрошена» после первого завершения сессии.

**Включение:** `hermes plugins enable disk-cleanup` (или отметь галочку в `hermes plugins`).

**Отключение:** `hermes plugins disable disk-cleanup`.

### security-guidance

Быстрое сопоставление шаблонов и предупреждения о безопасности при записи файлов. Когда вызовы `write_file` / `patch` / `skill_manage` агента содержат код, совпадающий с известным опасным шаблоном — `pickle.load`, `yaml.load` без `SafeLoader`, `eval(`, `os.system`, `subprocess(..., shell=True)`, JS `child_process.exec`, React `dangerouslySetInnerHTML`, сырые `.innerHTML =` / `.outerHTML =` / `document.write`, Node `crypto.createCipher`, AES‑ECB, отключённая проверка TLS, уязвимые парсеры `xml.etree` / `minidom` (XXE), `<script src="//…">` без SRI, `torch.load` без `weights_only=True`, инъекция `${{ github.event.* }}` в GitHub Actions — плагин добавляет блок `⚠️ Security guidance` к результату инструмента.

Файл всё равно записывается. Модель читает предупреждение в следующем сообщении инструмента и может либо исправить код, либо задокументировать, почему конструкция безопасна в данном контексте. У сопоставления шаблонов есть ненулевая доля ложных срабатываний, поэтому по умолчанию используется **предупреждение** (не блокировка).

**Охват:** 25 правил, покрывающих небезопасную десериализацию, инъекции команд, XSS‑синк, крипто‑подводные камни, XXE, цепочку поставок (SRI) и инъекции в CI/CD. Данные шаблонов — дословный форк Apache‑2.0 [Anthropic `claude-plugins-official`](https://github.com/anthropics/claude-plugins-official/tree/main/plugins/security-guidance/hooks); см. файлы `LICENSE` и `NOTICE` плагина.

**Режимы:**

| Переменная окружения | Эффект |
|---|---|
| (не задана) | **режим предупреждения** (по умолчанию) — файл записан, к результату добавлено предупреждение |
| `SECURITY_GUIDANCE_BLOCK=1` | **режим блокировки** — запись отклонена, предупреждение возвращено как причина блокировки |
| `SECURITY_GUIDANCE_DISABLE=1` | выключатель — плагин загружается, но ничего не делает |

**Включение:** `hermes plugins enable security-guidance` (или отметь галочку в `hermes plugins`).

**Отключение:** `hermes plugins disable security-guidance`.

**Что пока не делает:** оригинальный плагин Anthropic имеет два дополнительных уровня — LLM‑diff‑ревью на каждый ход агента, затрагивающий файлы, и ревью при коммите, отслеживающее поток данных между файлами. Ни один из них не портирован. Агент уже может запускать эти ревью по запросу через `delegate_task`.

### observability/langfuse

Трассирует ходы Hermes, вызовы LLM и обращения к инструментам в [Langfuse](https://langfuse.com) — открытой платформе наблюдаемости LLM. Один span на ход, одно поколение на API‑вызов, одно наблюдение инструмента на каждый вызов инструмента. Суммы использования, подсчёты токенов по типу и оценки стоимости берутся из канонических чисел Hermes `agent.usage_pricing`, поэтому дашборд Langfuse видит тот же разбор (input / output / `cache_read_input_tokens` / `cache_creation_input_tokens` / `reasoning_tokens`), который выводится в `hermes logs`.

Плагин работает в режиме **fail‑open**: отсутствие SDK, учётных данных или временная ошибка Langfuse приводят к тихому no‑op в hook. Цикл агента не страдает.

**Настройка:**

```bash
pip install langfuse
hermes plugins enable observability/langfuse
```

Или отметь галочку в интерактивном UI `hermes plugins`. Затем помести учётные данные в `~/.hermes/.env`:

```bash
HERMES_LANGFUSE_PUBLIC_KEY=pk-lf-...
HERMES_LANGFUSE_SECRET_KEY=sk-lf-...
HERMES_LANGFUSE_BASE_URL=https://cloud.langfuse.com   # or your self-hosted URL
```

**Как это работает:**

| Hook | Поведение |
|---|---|
| `pre_api_request` / `pre_llm_call` | Открывает (или переиспользует) корневой span «Hermes turn» для текущего хода. Запускает дочернее наблюдение `generation` для этого API‑вызова с сериализованными последними сообщениями как входными данными. |
| `post_api_request` / `post_llm_call` | Закрывает `generation`, прикрепляет `usage_details`, `cost_details`, `finish_reason`, вывод ассистента + вызовы инструментов. Если нет вызовов инструментов и контент не пуст, закрывает ход. |
| `pre_tool_call` | Запускает дочернее наблюдение `tool` с санитизированными `args`. |
| `post_tool_call` | Закрывает наблюдение инструмента с санитизированным `result`. Пэйлоады `read_file` суммируются (head + tail + количество опущенных строк), чтобы огромный файл чтения оставался ниже `HERMES_LANGFUSE_MAX_CHARS`. |

Группировка сессий происходит по идентификатору сессии Hermes (или ID задачи для суб‑агентов) через `langfuse.propagate_attributes`, так что всё в одной сессии `hermes chat` живёт в одной сессии Langfuse.

**Проверка:**

```bash
hermes plugins list                 # observability/langfuse should show "enabled"
hermes chat -q "hello"              # check the Langfuse UI for a "Hermes turn" trace
```

**Опциональная настройка** (в `.env`):

| Переменная | По умолчанию | Назначение |
|---|---|---|
| `HERMES_LANGFUSE_ENV` | — | Тег окружения в трассах (`production`, `staging`, …) |
| `HERMES_LANGFUSE_RELEASE` | — | Тег релиза/версии |
| `HERMES_LANGFUSE_SAMPLE_RATE` | `1.0` | Доля выборки, передаваемая SDK (0.0–1.0) |
| `HERMES_LANGFUSE_MAX_CHARS` | `12000` | Обрезка каждого поля для содержимого сообщения / аргументов инструмента / результатов |
| `HERMES_LANGFUSE_DEBUG` | `false` | Подробный лог плагина в `agent.log` |

Принимаются как префиксные переменные Hermes, так и стандартные SDK‑переменные (`LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`, `LANGFUSE_BASE_URL`) — при конфликте выигрывают префиксные.

**Производительность:** клиент Langfuse кэшируется после первого вызова hook. Если учётные данные или SDK отсутствуют, это решение тоже кэшируется — последующие хуки быстро возвращаются без повторных проверок env‑переменных или перезагрузки конфигурации.

**Отключение:** `hermes plugins disable observability/langfuse`. Модуль плагина всё ещё обнаруживается, но код не исполняется, пока не включишь его снова.

### google_meet

Позволяет агенту **присоединяться, транскрибировать и участвовать в звонках Google Meet** — вести заметки, суммировать обсуждение после, выполнять последующие действия и (опционально) отвечать в звонке через TTS.

**Что добавляет:**

- Безголовый виртуальный участник, который присоединяется к Meet‑URL через автоматизацию браузера
- Живая транскрипция аудио встречи через настроенного провайдера STT
- Набор инструментов `meet_summarize` / `meet_speak` / `meet_followup`, которые агент вызывает для действий над услышанным
- Артефакты после встречи (транскрипт, заметки с указанием говорящего, задачи) сохраняются под `~/.hermes/cache/google_meet/<meeting_id>/`

**Настройка:**

```bash
hermes plugins enable google_meet
# Prompts you to sign in via the plugin's OAuth flow on first use —
# needs a Google account with Meet access. Host approval may be required
# if the meeting enforces "only invited participants can join".
```

**Использование из чата:**

> “Join meet.google.com/abc-defg-hij and take notes. After the call, send me a summary with action items.”

Агент инициирует присоединение к встрече, передаёт транскрипцию в свой контекст по мере разговора и выдаёт структурированное резюме, когда встреча заканчивается (или когда ты попросишь остановить).

**Когда использовать:** регулярные стендапы, где нужен бот‑транскрибатор + суммаризатор для асинхронных участников; интервью‑депозиции, где нужны структурированные заметки; любой случай, когда обычно используют Fireflies / Otter / Grain. Если не хочешь, чтобы ИИ слушал — не включай.

**Отключение:** `hermes plugins disable google_meet`. Кэшированные транскрипты и записи остаются в `~/.hermes/cache/google_meet/` до их удаления.

### hermes-achievements

Добавляет **вкладку достижений в стиле Steam на панель** — 60+ коллекционных, многоуровневых значков, генерируемых из реальной истории сессий Hermes. Достижения за цепочки инструментов, паттерны отладки, «vibe‑coding»‑стрейки, использование навыков/памяти, разнообразие моделей/провайдеров, особенности образа жизни (выходные и ночные сессии). Изначально создан [@PCinkusz](https://github.com/PCinkusz) как внешний плагин; теперь встроен, чтобы оставаться синхронным с изменениями Hermes.

**Как работает:**

- Сканирует всю базу `~/.hermes/state.db` истории сессий на бекенде дашборда
- Статистика по сессии кэшируется по отпечатку `(started_at, last_active)`, поэтому только новые или изменённые сессии переанализируются при последующих сканах
- Первый скан запускается в фоновом потоке — дашборд никогда не блокируется, даже при базе с тысячами сессий
- Состояние разблокировки сохраняется в `$HERMES_HOME/plugins/hermes-achievements/state.json`

**Прогресс уровней:** Copper → Silver → Gold → Diamond → Olympian. Каждая карта показывает раздел «What counts», где перечислены точные метрики.

**Состояния достижений:**

| Состояние | Значение |
|---|---|
| Unlocked | Достигнут хотя бы один уровень |
| Discovered | Достижение известно, прогресс виден, но ещё не получено |
| Secret | Скрыто, пока Hermes не обнаружит первый сигнал в истории |

**API** — маршруты монтируются под `/api/plugins/hermes-achievements/`:

| Endpoint | Назначение |
|---|---|
| `GET /achievements` | Полный каталог с состоянием разблокировки по каждому значку (возвращает заглушку, пока первый холодный скан работает) |
| `GET /scan-status` | Состояние фонового сканера: `idle` / `running` / `failed`, последняя длительность, количество запусков |
| `GET /recent-unlocks` | Двадцать последних разблокированных значков, новейшие первыми |
| `GET /sessions/{id}/badges` | Значки, полученные преимущественно в конкретной сессии |
| `POST /rescan` | Синхронный ручной рескан (блокирует; используется при нажатии кнопки рескан) |
| `POST /reset-state` | Очистить историю разблокировок и кэшированные снимки |

**Файлы состояния** — находятся в `$HERMES_HOME/plugins/hermes-achievements/`:

| Файл | Содержание |
|---|---|
| `state.json` | История разблокировок: какие значки получены и когда. Стабильно при обновлениях Hermes. |
| `scan_snapshot.json` | Последний завершённый снимок сканирования (отдаётся сразу при загрузке дашборда) |
| `scan_checkpoint.json` | Кэш статистики по сессиям, ключевой по отпечатку (ускоряет тёплые ресканы) |

**Замечания о производительности:**

- Холодный скан ~8 000 сессий занимает несколько минут. Он запускается в фоновом потоке при первом запросе дашборда; UI показывает заглушку и опрашивает `/scan-status`.
- **Инкрементальные результаты во время холодного скана** — сканер публикует частичный снимок каждые ~250 сессий, так что каждый обновлённый дашборд показывает всё больше разблокированных значков по мере прогресса. Нет долгого ожидания нулей.
- Тёплый рескан переиспользует статистику по сессиям, чей отпечаток `started_at` + `last_active` совпадает с контрольной точкой — завершается за секунды даже при больших историях.
- TTL снимка в памяти — 120 s; устаревшие запросы сразу получают старый снимок и запускают фоновой рефреш. Ты никогда не ждёшь спиннер, только потому что TTL истёк.

**Включение:** ничего включать не нужно — `hermes-achievements` это только плагин дашборда (без хуков жизненного цикла, без инструментов, видимых модели). Он автоматически регистрируется как вкладка в `hermes dashboard` при первом запуске. Конфиг `plugins.enabled` управляет только хуками/инструментами; плагины дашборда обнаруживаются через их `dashboard/manifest.json`.

**Отказ от использования:** удали или переименуй `plugins/hermes-achievements/dashboard/manifest.json`, либо переопредели его пользовательским плагином того же имени в `~/.hermes/plugins/hermes-achievements/`, который не поставляет дашборд. Файлы состояния плагина в `$HERMES_HOME/plugins/hermes-achievements/` сохраняются — переустановка сохраняет историю достижений.
## Добавление встроенного плагина

Встроенные плагины пишутся точно так же, как любые другие плагины Hermes — см. [Build a Hermes Plugin](/guides/build-a-hermes-plugin). Единственные различия:

- Каталог находится в `<repo>/plugins/<name>/` вместо `~/.hermes/plugins/<name>/`
- Источник манифеста отображается как `bundled` в `hermes plugins list`
- Пользовательские плагины с тем же именем переопределяют встроенную версию

Плагин является хорошим кандидатом для встраивания, когда:

- У него нет необязательных зависимостей (или они уже указаны в `pip install .[all]` deps)
- Поведение полезно большинству пользователей и реализовано как **opt‑out**, а не **opt‑in**
- Логика привязана к хукам жизненного цикла, которые агент иначе пришлось бы помнить вызвать
- Он дополняет базовую возможность, не расширяя видимую моделью поверхность инструментов

Контрпримеры — вещи, которые должны оставаться пользовательскими плагинами, а не быть встроенными: сторонние интеграции с API‑ключами, нишевые рабочие процессы, большие деревья зависимостей, всё, что существенно меняет поведение агента по умолчанию.
