# Поиск инструментов

Когда у тебя есть множество MCP‑серверов или подключённых к сессии плагинов‑инструментов, их JSON‑схемы могут занимать значительную часть окна контекста на каждом шаге — даже если только несколько из них действительно нужны для того, что пользователь запросил.

**Поиск инструментов** — это опциональный слой прогрессивного раскрытия Hermes для этой проблемы. При активации инструменты MCP и плагинов заменяются в массиве видимых модели инструментов тремя мостовыми инструментами, а модель загружает схему конкретного инструмента по запросу.

:::info Встроенные инструменты Hermes никогда не откладываются
Инструменты, составляющие базовый набор возможностей Hermes (`terminal`, `read_file`, `write_file`, `patch`, `search_files`, `todo`, `memory`, `browser_*`, `web_search`, `web_extract`, `clarify`, `execute_code`, `delegate_task`, `session_search`, `send_message` и остальные из `_HERMES_CORE_TOOLS`) *всегда* загружаются напрямую. Откладываться могут только инструменты MCP и плагины‑инструменты, не входящие в ядро.
:::

## Как это работает

Когда Поиск инструментов активируется на ход, модель видит три новых инструмента вместо отложенных:

```
tool_search(query, limit?)     — search the deferred-tool catalog
tool_describe(name)            — load the full schema for one tool
tool_call(name, arguments)     — invoke a deferred tool
```

Типичное взаимодействие выглядит так:

```
Model: tool_search("create a github issue")
  → { matches: [{ name: "mcp_github_create_issue", ... }, ...] }
Model: tool_describe("mcp_github_create_issue")
  → { parameters: { type: "object", properties: { ... } } }
Model: tool_call("mcp_github_create_issue", { title: "...", body: "..." })
  → { ok: true, issue_number: 42 }
```

Когда модель вызывает `tool_call`, Hermes **разворачивает мост** и передаёт вызов подлежащему инструменту так, как если бы модель вызвала его напрямую. Хуки до вызова инструмента, ограничения, запросы подтверждения и хуки после вызова работают с реальным именем инструмента — а не с `tool_call`. Лента активности в CLI и шлюзе также разворачивается, так что ты видишь подлежащий инструмент, а не мост.

## Когда он активируется?

По умолчанию Поиск инструментов работает в режиме `auto`: он активируется только тогда, когда отложенные схемы инструментов занимают как минимум 10 % окна контекста активной модели. Ниже этого порога сборка массива инструментов проходит без изменений и не добавляет накладных расходов.

Это решение переоценивается каждый раз при построении массива инструментов, поэтому:

- Сессия с несколькими MCP‑инструментами и моделью с длинным контекстом никогда не активирует Поиск инструментов.
- Сессия с множеством подключённых MCP‑серверов (обычно 15 + инструментов) начинает его активировать.
- Удаление MCP‑серверов в середине сессии корректно возвращает прямой доступ при следующей сборке.

## Конфигурация

```yaml
tools:
  tool_search:
    enabled: auto       # auto (default), on, or off
    threshold_pct: 10   # percentage of context — only used in auto mode
    search_default_limit: 5
    max_search_limit: 20
```

| Ключ | По умолчанию | Описание |
| --- | --- | --- |
| `enabled` | `auto` | `auto` активируется выше порога; `on` всегда активен, если есть хотя бы один откладываемый инструмент; `off` полностью отключает. |
| `threshold_pct` | `10` | Процент длины контекста, при котором включается режим `auto`. Диапазон 0–100. |
| `search_default_limit` | `5` | Количество результатов, возвращаемых, когда модель вызывает `tool_search` без указания `limit`. |
| `max_search_limit` | `20` | Жёсткий верхний предел, который модель может запросить через `limit`. Диапазон 1–50. |

Можно также переключить устаревшую булеву форму:

```yaml
tools:
  tool_search: true   # equivalent to {enabled: auto}
```

## Когда НЕ использовать

Поиск инструментов заменяет фиксированную стоимость токенов за ход (три схемы мостовых инструментов, ~300 токенов) и как минимум один дополнительный раунд‑трип (search → describe → call) на экономию от отложенных схем. Это явно выгодно, когда у тебя много инструментов и ты используешь их по‑мало за ход; это накладные расходы, когда инструментов мало.

Настройка `auto` по умолчанию решает эту задачу за тебя. Если установить `enabled: on` без условий, ожидай небольшие дополнительные затраты за ход при небольших наборах инструментов.

## Компромиссы, которые остаются

Они вытекают из инварианта целостности кэша подсказок — они присущи любой схеме прогрессивного раскрытия, а не только этой реализации:

- **Один дополнительный раунд‑трип для «холодных» инструментов.** При первом запросе отложенного инструмента модель тратит один‑два дополнительных вызова, чтобы найти и загрузить схему. Экономия токенов на статической стороне реальна, но часть её возвращается во время выполнения.
- **Отсутствие выгоды от кэша для отложенных схем.** Загруженный результат `tool_describe` попадает в историю диалога (поэтому кэшируется на последующих ходах), но он никогда не получает выгоду от префикса кэша системного подсказки.
- **Зависимость от качества модели.** Поиск инструментов предполагает, что модель может сформировать разумный поисковый запрос для нужного инструмента. Меньшие модели делают это хуже; опубликованные цифры Anthropic (49 % → 74 % на Opus 4 с и без поиска инструментов) показывают рост, но также и то, что ~26 % точности всё ещё теряется из‑за неудачного поиска.
- **Изменения набора инструментов инвалидируют кэш.** Добавление или удаление инструмента в середине сессии меняет описания мостовых инструментов (в которых указано количество отложенных инструментов) и каталог, поэтому кэш подсказок становится недействительным. Это тот же компромисс, что и при любом изменении набора инструментов.

## Детали реализации

- **Поиск:** BM25 по токенизированному названию инструмента + описанию + именам параметров. При отсутствии положительных результатов BM25 переходит к буквальному поиску подстроки в названии инструмента, что защищает от вырожденных случаев с нулевым IDF (например, поиск `"github"` в каталоге, где каждое название содержит «github»).
- **Каталог статичен между ходами.** Он перестраивается из текущего списка `tool-defs` при каждой сборке — без `Map`, привязанного к сессии. Это избавляет от класса багов, когда сохранённый каталог рассинхронизируется с живым реестром инструментов.
- **Каталог ограничен наборами инструментов сессии.** `tool_search`, `tool_describe` и `tool_call` видят и вызывают только те инструменты, которые действительно доступны в данной сессии. Под‑агент, kanban‑worker или шлюз‑сессия, ограниченная подмножеством наборов инструментов, не может использовать мост для обнаружения или вызова инструмента за пределами этого подмножества — отложенный каталог представляет собой отложенную часть включённых/исключённых наборов инструментов текущей сессии, а не весь реестр процесса.
- **Нет JS‑песочницы.** Hermes использует более простой режим «структурированных инструментов» (search / describe / call как обычные функции). JS‑песочница «режим кода», предлагаемая некоторыми другими реализациями, имеет большую поверхность атаки; мы её пропускаем.
