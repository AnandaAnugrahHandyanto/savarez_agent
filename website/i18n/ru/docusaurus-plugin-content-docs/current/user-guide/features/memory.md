# Постоянная память

Hermes Agent имеет ограниченную, курируемую память, которая сохраняется между сессиями. Это позволяет ему помнить твои предпочтения, твои проекты, твоё окружение и то, чему он научился.

## Как это работает

Память агента состоит из двух файлов:

| Файл | Назначение | Ограничение символов |
|------|------------|----------------------|
| **MEMORY.md** | Личные заметки агента — факты об окружении, конвенции, выученное | 2 200 символов (~800 токенов) |
| **USER.md** | Профиль пользователя — твои предпочтения, стиль общения, ожидания | 1 375 символов (~500 токенов) |

Оба находятся в `~/.hermes/memories/` и внедряются в системный запрос как замороженный снимок при запуске сессии. Агент управляет своей памятью через инструмент `memory` — он может добавлять, заменять или удалять записи.

:::info
Ограничения по символам сохраняют память сфокусированной. Когда память заполняется, агент консолидирует или заменяет записи, чтобы освободить место для новой информации.
:::

## Как память выглядит в системном запросе

В начале каждой сессии записи памяти загружаются с диска и отображаются в системном запросе как замороженный блок:

```
══════════════════════════════════════════════
MEMORY (your personal notes) [67% — 1,474/2,200 chars]
══════════════════════════════════════════════
User's project is a Rust web service at ~/code/myapi using Axum + SQLx
§
This machine runs Ubuntu 22.04, has Docker and Podman installed
§
User prefers concise responses, dislikes verbose explanations
```

Формат включает:
- Заголовок, указывающий, какое хранилище (MEMORY или USER PROFILE)
- Процент использования и количество символов, чтобы агент знал ёмкость
- Отдельные записи, разделённые делимитерами `§` (знак раздела)
- Записи могут быть многострочными

**Шаблон замороженного снимка:** Внедрение в системный запрос происходит один раз при старте сессии и не меняется в течение её. Это сделано намеренно — сохраняет кеш префикса LLM для производительности. Когда агент добавляет/удаляет записи памяти во время сессии, изменения сразу сохраняются на диск, но не появляются в системном запросе до следующего запуска сессии. Ответы инструмента всегда показывают актуальное состояние.

## Действия инструмента памяти

Агент использует инструмент `memory` с такими действиями:

- **add** — добавить новую запись памяти
- **replace** — заменить существующую запись обновлённым содержимым (использует поиск подстроки через `old_text`)
- **remove** — удалить запись, которая больше не актуальна (использует поиск подстроки через `old_text`)

Действия `read` нет — содержимое памяти автоматически внедряется в системный запрос при старте сессии. Агент видит свои воспоминания как часть контекста разговора.

### Поиск подстроки

Действия `replace` и `remove` используют короткий уникальный поиск подстроки — не требуется полный текст записи. Параметр `old_text` должен быть уникальной подстрокой, однозначно идентифицирующей одну запись:

```python
# If memory contains "User prefers dark mode in all editors"
memory(action="replace", target="memory",
       old_text="dark mode",
       content="User prefers light mode in VS Code, dark mode in terminal")
```

Если подстрока совпадает с несколькими записями, возвращается ошибка с просьбой уточнить совпадение.

## Два целевых хранилища

### `memory` — личные заметки агента

Для информации, которую агент должен помнить об окружении, рабочих процессах и извлечённых уроках:

- Факты об окружении (ОС, инструменты, структура проекта)
- Конвенции и конфигурация проекта
- Особенности инструментов и найденные обходные пути
- Записи дневника выполненных задач
- Навыки и техники, которые сработали

### `user` — профиль пользователя

Для информации о личности пользователя, его предпочтениях и стиле общения:

- Имя, роль, часовой пояс
- Предпочтения в общении (кратко vs подробно, форматы)
- Неприязни и вещи, которых следует избегать
- Привычки в рабочем процессе
- Уровень технической квалификации

## Что сохранять, а что пропускать

### Сохранять (проактивно)

Агент сохраняет автоматически — тебе не нужно просить. Он сохраняет, когда учится:

- **Предпочтения пользователя:** «Я предпочитаю TypeScript вместо JavaScript» → сохраняется в `user`
- **Факты об окружении:** «Этот сервер работает на Debian 12 с PostgreSQL 16» → сохраняется в `memory`
- **Исправления:** «Не используй `sudo` для команд Docker, пользователь находится в группе docker» → сохраняется в `memory`
- **Конвенции:** «Проект использует табуляцию, ширина строки 120 символов, docstring‑ы в стиле Google» → сохраняется в `memory`
- **Завершённая работа:** «Мигрировал базу данных из MySQL в PostgreSQL 15‑01‑2026» → сохраняется в `memory`
- **Явные запросы:** «Запомни, что ротация моего API‑ключа происходит ежемесячно» → сохраняется в `memory`

### Пропускать

- **Тривиальная/очевидная информация:** «Пользователь спросил про Python» — слишком расплывчато, чтобы быть полезным
- **Легко восстанавливаемые факты:** «Python 3.12 поддерживает вложенные f‑строки» — можно быстро найти в веб‑поиске
- **Сырые дампы данных:** большие блоки кода, файлы логов, таблицы данных — слишком объёмно для памяти
- **Эфемерные данные сессии:** временные пути файлов, одноразовый контекст отладки
- **Информация, уже присутствующая в файлах контекста:** содержимое `SOUL.md` и `AGENTS.md`

## Управление ёмкостью

Память имеет строгие ограничения по символам, чтобы системные запросы оставались ограниченными:

| Хранилище | Ограничение | Типичные записи |
|----------|-------------|-----------------|
| memory | 2 200 символов | 8‑15 записей |
| user | 1 375 символов | 5‑10 записей |

### Что происходит, когда память заполнена

Если попытаться добавить запись, превышающую лимит, инструмент возвращает ошибку:

```json
{
  "success": false,
  "error": "Memory at 2,100/2,200 chars. Adding this entry (250 chars) would exceed the limit. Replace or remove existing entries first.",
  "current_entries": ["..."],
  "usage": "2,100/2,200"
}
```

Тогда агент должен:
1. Прочитать текущие записи (показаны в ответе об ошибке)
2. Определить записи, которые можно удалить или консолидировать
3. Использовать `replace`, чтобы объединить связанные записи в более короткие версии
4. Затем `add` новую запись

**Лучший подход:** Когда память превышает 80 % ёмкости (это видно в заголовке системного запроса), консолидируй записи перед добавлением новых. Например, объединить три отдельные записи «проект использует X» в одну всестороннюю запись описания проекта.

### Практические примеры хороших записей памяти

**Компактные, информационно‑насыщенные записи работают лучше:**

```
# Good: Packs multiple related facts
User runs macOS 14 Sonoma, uses Homebrew, has Docker Desktop and Podman. Shell: zsh with oh-my-zsh. Editor: VS Code with Vim keybindings.

# Good: Specific, actionable convention
Project ~/code/api uses Go 1.22, sqlc for DB queries, chi router. Run tests with 'make test'. CI via GitHub Actions.

# Good: Lesson learned with context
The staging server (10.0.1.50) needs SSH port 2222, not 22. Key is at ~/.ssh/staging_ed25519.

# Bad: Too vague
User has a project.

# Bad: Too verbose
On January 5th, 2026, the user asked me to look at their project which is
located at ~/code/api. I discovered it uses Go version 1.22 and...
```

## Предотвращение дубликатов

Система памяти автоматически отклоняет точные дублирующие записи. Если попытаться добавить уже существующее содержание, возвращается успех с сообщением «no duplicate added».

## Проверка безопасности

Записи памяти сканируются на наличие паттернов инъекций и утечки данных перед принятием, поскольку они внедряются в системный запрос. Содержание, совпадающее с угрозами (prompt injection, credential exfiltration, SSH backdoors) или содержащее невидимые Unicode‑символы, блокируется.

## Поиск по сессии

Помимо `MEMORY.md` и `USER.md` агент может искать в своих прошлых разговорах с помощью инструмента `session_search`:

- Все CLI‑ и мессенджер‑сессии хранятся в SQLite (`~/.hermes/state.db`) с полнотекстовым поиском FTS5
- Запросы возвращают реальные сообщения из БД — без суммирования LLM, без усечения
- Агент может находить вещи, обсуждавшиеся недели назад, даже если они не находятся в активной памяти
- Агент также может прокручивать вперёд/назад внутри любой найденной сессии

```bash
hermes sessions list    # Browse past sessions
```

Смотри [Session Search Tool](/user-guide/sessions#session-search-tool) для трёх форм вызова (discovery / scroll / browse) и формата ответа.

### `session_search` vs `memory`

| Функция | Постоянная память | Поиск по сессии |
|--------|-------------------|-----------------|
| **Ёмкость** | ~1 300 токенов всего | Неограниченно (все сессии) |
| **Скорость** | Мгновенно (в системном запросе) | ~20 мс запрос FTS5, ~1 мс прокрутка |
| **Стоимость** | Стоимость токенов в каждом запросе | Бесплатно — без вызовов LLM |
| **Сценарий использования** | Ключевые факты всегда доступны | Поиск конкретных прошлых разговоров |
| **Управление** | Курируется агентом вручную | Автоматически — все сессии сохраняются |
| **Токен‑стоимость** | Фиксировано за сессию (~1 300 токенов) | По запросу (искать при необходимости) |

**Память** предназначена для критически важных фактов, которые всегда должны быть в контексте. **Поиск по сессии** используется для вопросов типа «обсуждали ли мы X на прошлой неделе?», когда агенту нужно вспомнить детали из прошлых разговоров.

## Конфигурация

```yaml
# In ~/.hermes/config.yaml
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200   # ~800 tokens
  user_char_limit: 1375     # ~500 tokens
```

## Внешние провайдеры памяти

Для более глубокой, постоянной памяти, выходящей за пределы `MEMORY.md` и `USER.md`, Hermes поставляется с 8 плагинами внешних провайдеров памяти — включая Honcho, OpenViking, Mem0, Hindsight, Holographic, RetainDB, ByteRover и Supermemory.

Внешние провайдеры работают **вместе** со встроенной памятью (никогда не заменяя её) и добавляют возможности, такие как графы знаний, семантический поиск, автоматическое извлечение фактов и кросс‑сессионное моделирование пользователя.

```bash
hermes memory setup      # pick a provider and configure it
hermes memory status     # check what's active
```

Смотри руководство [Memory Providers](./memory-providers.md) для полного описания каждого провайдера, инструкций по настройке и сравнения.
