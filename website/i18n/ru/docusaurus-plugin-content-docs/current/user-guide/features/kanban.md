# Kanban — Совместная работа профилей с несколькими агентами

> **Хочешь пошаговое руководство?** Читай [урок по Kanban](./kanban-tutorial) — четыре пользовательские истории (solo dev, fleet farming, role pipeline with retry, circuit breaker) со скриншотами дашборда для каждой. Эта страница — справочник; урок — повествование.

Hermes Kanban — надёжная доска задач, общая для всех твоих профилей Hermes, позволяющая нескольким именованным агентам совместно работать над задачами без хрупких подпроцессов‑роев. Каждая задача — строка в `~/.hermes/kanban.db`; каждая передача — строка, которую может читать и писать любой; каждый работник — полноценный процесс ОС со своей идентичностью.

### Две поверхности: модель взаимодействует через инструменты, ты — через CLI

Доска имеет два входа, оба работают с одной и той же `~/.hermes/kanban.db`:

- **Агенты управляют доской через специализированный набор инструментов `kanban_*`** — `kanban_show`, `kanban_list`, `kanban_complete`, `kanban_block`, `kanban_heartbeat`, `kanban_comment`, `kanban_create`, `kanban_link`, `kanban_unblock`. Диспетчер запускает каждого работника с этими инструментами уже в его схеме; профили‑оркестраторы также могут явно включать набор инструментов `kanban`. Модель читает и маршрутизирует задачи, вызывая инструменты напрямую, *не* вызывая `hermes kanban` через оболочку. См. [How workers interact with the board](#how-workers-interact-with-the-board) ниже.
- **Ты (и скрипты, и cron) управляешь доской через `hermes kanban …`** в CLI, `/kanban …` как slash‑команду или через дашборд. Это предназначено для людей и автоматизации — для мест, где нет модели вызова инструментов.

Обе поверхности проходят через один слой `kanban.db`, поэтому чтения видят согласованное состояние, а записи не могут дрейфовать. Остальная часть этой страницы показывает примеры CLI, потому что их легко скопировать‑вставить, но каждый глагол CLI имеет эквивалент вызова инструмента, который использует модель.

Это то, что покрывает нагрузки, недоступные `delegate_task`:

- **Триаж исследований** — параллельные исследователи + аналитик + писатель, человек в цикле.
- **Запланованные операции** — повторяющиеся ежедневные брифинги, формирующие журнал за недели.
- **Цифровые двойники** — постоянные именованные помощники (`inbox-triage`, `ops-review`), накапливающие память со временем.
- **Инженерные конвейеры** — разложить → реализовать в параллельных рабочих ветках → ревью → итерация → PR.
- **Работа флота** — один специалист управляет N объектами (50 соцсетей, 12 мониторимых сервисов).

Для полного обоснования дизайна, сравнительного анализа с Cline Kanban / Paperclip / NanoClaw / Google Gemini Enterprise и восьми каноническими паттернами сотрудничества смотри `docs/hermes-kanban-v1-spec.pdf` в репозитории.
## Kanban vs. `delegate_task`

Они выглядят похожими, но это не одинаковые примитивы.

| | `delegate_task` | Kanban |
|---|---|---|
| Форма | RPC‑вызов (fork → join) | Надёжная очередь сообщений + конечный автомат |
| Родитель | Блокирует до возврата дочернего | Fire‑and‑forget после `create` |
| Идентичность дочернего | Анонимный субагент | Именованный профиль с постоянной памятью |
| Возобновляемость | Нет — ошибка = ошибка | Block → unblock → повторный запуск; сбой → возврат |
| Человек в цикле | Не поддерживается | Комментарий / разблокировка в любой момент |
| Агентов на задачу | Один вызов = один субагент | N агентов за жизнь задачи (retry, review, follow‑up) |
| Аудит‑трейл | Утерян при сжатии контекста | Надёжные строки в SQLite навсегда |
| Координация | Иерархическая (caller → callee) | Пиринговая — любой профиль читает/пишет любую задачу |

**Однострочное различие:** `delegate_task` — это вызов функции; Kanban — очередь работы, где каждая передача представляет собой строку, видимую и редактируемую любым профилем (или человеком).

**Используй `delegate_task`, когда** родительскому агенту нужен короткий ответ рассуждения перед продолжением, людей нет, результат возвращается в контекст родителя.

**Используй Kanban, когда** работа пересекает границы агентов, должна пережить перезапуски, может потребовать ввода человека, может быть взята другим ролям или должна быть обнаружена позже.

Они сосуществуют: kanban‑worker может вызывать `delegate_task` внутри своего выполнения.
## Основные понятия

- **Board** — отдельная очередь задач со своей SQLite‑БД, каталогом рабочих пространств и циклом диспетчера. Одна установка может иметь множество board‑ов (например, по одному на проект, репозиторий или домен); см. [Boards (multi-project)](#boards-multi-project) ниже. Пользователи, работающие в рамках одного проекта, остаются на board `default` и никогда не видят слово «board» за пределами этого раздела документации.
- **Task** — строка с заголовком, необязательным телом, одним исполнителем (имя профиля), статусом (`triage | todo | ready | running | blocked | done | archived`), необязательным пространством имён арендатора, необязательным ключом идемпотентности (удаление дублей при повторных автоматизациях).
- **Link** — строка `task_links`, фиксирующая зависимость родитель → потомок. Диспетчер переводит `todo → ready`, когда все родители находятся в статусе `done`.
- **Comment** — протокол взаимодействия между агентами. Агенты и люди добавляют комментарии; когда рабочий процесс (re‑)создаётся, он читает всю ветку комментариев как часть своего контекста.
- **Workspace** — каталог, в котором работает рабочий процесс. Три типа:
  - `scratch` (по умолчанию) — свежий временный каталог под `~/.hermes/kanban/workspaces/<id>/` (или `~/.hermes/kanban/boards/<slug>/workspaces/<id>/` на нестандартных board‑ах). **Удаляется после завершения задачи** — `scratch` по замыслу эфемерен, поэтому каталог стирается в тот момент, когда рабочий процесс (или `hermes kanban complete <id>`) помечает задачу выполненной. Если нужно сохранить вывод рабочего процесса, используй `worktree:` или `dir:<path>` вместо этого. При первом создании `scratch`‑workspace в установке диспетчер выводит предупреждение и генерирует событие `tip_scratch_workspace` на задаче (видно через `hermes kanban show <id>`).
  - `dir:<path>` — существующий общий каталог (хранилище Obsidian, каталог почтовых операций, папка аккаунта). **Должен быть абсолютным путём.** Относительные пути вроде `dir:../tenants/foo/` отклоняются диспетчером, потому что они разрешаются относительно текущего рабочего каталога диспетчера, что неоднозначно и представляет вектор атаки «confused‑deputy». Путь считается надёжным — это твой компьютер, твоя файловая система, рабочий процесс запускается от твоего UID. Это модель угрозы «доверенный локальный пользователь»; kanban по замыслу однопользовательский. **Сохраняется после завершения.**
  - `worktree` — git‑worktree под `.worktrees/<id>/` для задач программирования. Используй `worktree:<path>`, чтобы зафиксировать точный целевой путь. На стороне рабочего процесса `git worktree add` создаёт его, используя `--branch`, если он указан. **Сохраняется после завершения.**
- **Dispatcher** — длительно работающий цикл, который каждые N секунд (по умолчанию 60): освобождает устаревшие заявки, освобождает крашнутые рабочие процессы (PID исчез, но TTL ещё не истёк), переводит готовые задачи, атомарно захватывает их и запускает назначенные профили. По умолчанию работает **внутри шлюза** (`kanban.dispatch_in_gateway: true`). Один диспетчер обрабатывает все board‑ы за тик; рабочие процессы запускаются с зафиксированным `HERMES_KANBAN_BOARD`, чтобы они не видели другие board‑ы. После `kanban.failure_limit` последовательных неудач запуска на одной задаче (по умолчанию 2) диспетчер автоматически блокирует её, указывая последнюю ошибку в качестве причины — это предотвращает «трешинг» задач, у которых отсутствует профиль, невозможно смонтировать рабочее пространство и т.п.
- **Tenant** — необязательное строковое пространство имён *внутри* board‑а. Один флот специалистов может обслуживать несколько бизнесов (`--tenant business-a`) с изоляцией данных по пути рабочего пространства и префиксу ключа памяти. Tenants выступают как мягкий фильтр; board‑ы — как жёсткая граница изоляции.
## Доски (мульти‑проектные)

Доски позволяют разделять несвязанные потоки работы — по одному на проект, репозиторий или домен — в изолированные очереди. При новой установке существует ровно одна доска под названием `default` (БД в `~/.hermes/kanban.db` для обратной совместимости). Пользователям, которым нужен только один поток работы, не нужно знать о досках; функция включается по желанию.

Изоляция на уровне доски абсолютна:

- Отдельная SQLite‑БД на каждую доску (`~/.hermes/kanban/boards/<slug>/kanban.db`).
- Отдельные каталоги `workspaces/` и `logs/`.
- Рабочие процессы, запущенные для задачи, видят **только** задачи своей доски — диспетчер задаёт переменную `HERMES_KANBAN_BOARD` в окружении дочернего процесса, и каждый инструмент `kanban_*`, к которому имеет доступ рабочий процесс, читает её.
- Связывание задач между досками запрещено (упрощает схему; если действительно нужны ссылки между проектами, используйте упоминания в свободном тексте и ищите их вручную по ID).

### Управление досками из CLI

```bash
# See what's on disk. Fresh installs show only "default".
hermes kanban boards list

# Create a new board.
hermes kanban boards create atm10-server \
    --name "ATM10 Server" \
    --description "Minecraft modded server ops" \
    --icon 🎮 \
    --switch                   # optional: make it the active board

# Operate on a specific board without switching.
hermes kanban --board atm10-server list
hermes kanban --board atm10-server create "Restart ATM server" --assignee ops

# Change which board is "current" for subsequent calls.
hermes kanban boards switch atm10-server
hermes kanban boards show             # who's active right now?

# Rename the display name (the slug is immutable — it's the directory name).
hermes kanban boards rename atm10-server "ATM10 (Prod)"

# Archive (default) — moves the board's dir to boards/_archived/<slug>-<ts>/.
# Recoverable by moving the dir back.
hermes kanban boards rm atm10-server

# Hard delete — `rm -rf` the board dir. No recovery.
hermes kanban boards rm atm10-server --delete
```

Порядок разрешения доски (сначала более высокий приоритет):

1. Явный `--board <slug>` в вызове CLI.
2. Переменная окружения `HERMES_KANBAN_BOARD` (устанавливается диспетчером при порождении рабочего процесса, поэтому рабочие процессы не видят другие доски).
3. `~/.hermes/kanban/current` — slug, сохранённый командой `hermes kanban boards switch`.
4. `default`.

Slug‑ы проверяются: строчные буквенно‑цифровые символы + дефисы + подчёркивания, 1‑64 символа, должны начинаться с буквенно‑цифрового символа. Ввод в верхнем регистре автоматически переводится в нижний. Всё остальное (слеши, пробелы, точки, `..`) отклоняется на уровне CLI, чтобы трюки с обходом путей не могли назвать доску.

### Управление досками из панели управления

`hermes dashboard` → вкладка **Kanban** показывает переключатель досок вверху, как только существует более одной доски (или у любой доски есть задачи). Пользователи с одной доской видят лишь небольшую кнопку `+ New board`; переключатель скрыт, пока он не нужен.

- **Выпадающий список досок** — выбирает активную доску. Ваш выбор сохраняется в `localStorage` браузера, поэтому он сохраняется между перезагрузками и не смещает указатель `current` CLI, если терминал остаётся открытым.
- **+ New board** — открывает модальное окно с запросом slug, отображаемого имени, описания и иконки. Есть опция автоматически переключиться на новую доску.
- **Archive** — показывается только для досок, отличных от `default`. После подтверждения перемещает каталог доски в `boards/_archived/`.

Все API‑конечные точки панели управления принимают параметр `?board=<slug>` для ограничения по доске. WebSocket событий привязывается к доске при установке соединения; переключение в UI открывает новый WS для выбранной доски.
## Быстрый старт

Команды ниже — **ты** (человек), настраивающий доску и создающий задачи. Как только задача назначена, диспетчер порождает назначенный профиль как воркер, и оттуда **модель управляет задачей через вызовы инструмента `kanban_*`, а не через команды CLI** — см. [Как воркеры взаимодействуют с доской](#how-workers-interact-with-the-board).

```bash
# 1. Create the board (you)
hermes kanban init

# 2. Start the gateway (hosts the embedded dispatcher)
hermes gateway start

# 3. Create a task (you — or an orchestrator agent via kanban_create)
hermes kanban create "research AI funding landscape" --assignee researcher

# 4. Watch activity live (you)
hermes kanban watch

# 5. See the board (you)
hermes kanban list
hermes kanban stats
```

Когда диспетчер подхватывает `t_abcd` и порождает профиль `researcher`, первая вещь, которую делает модель воркера, — вызывает `kanban_show()` для чтения своей задачи. Она не запускает `hermes kanban show t_abcd`.

### Диспетчер, встроенный в шлюз (по умолчанию)

Диспетчер работает внутри процесса шлюза. Ничего устанавливать не нужно, отдельный сервис не требуется — если шлюз запущен, готовые задачи подхватываются на следующем тике (по умолчанию 60 с).

```yaml
# config.yaml
kanban:
  dispatch_in_gateway: true        # default
  dispatch_interval_seconds: 60    # default
```

Переопределите флаг конфигурации во время выполнения через `HERMES_KANBAN_DISPATCH_IN_GATEWAY=0` для отладки. Применяется стандартная супервизия шлюза: запустите `hermes gateway start` напрямую или подключите шлюз как юнит systemd пользователя (см. документацию шлюза). Без запущенного шлюза задачи со статусом `ready` остаются где были, пока не появится шлюз — `hermes kanban create` предупреждает об этом при создании.

Запуск `hermes kanban daemon` как отдельного процесса **устарел**; используйте шлюз. Если действительно невозможно запустить шлюз (политика безголового хоста запрещает длительные сервисы и т.п.), флаг `--force` позволяет сохранить старый автономный демон на один цикл релиза, но запуск одновременно встроенного в шлюз диспетчера **и** автономного демона против одной и той же `kanban.db` приводит к конфликтам и не поддерживается.

### Идемпотентное создание (для автоматизации / вебхуков)

```bash
# First call creates the task. Any subsequent call with the same key
# returns the existing task id instead of duplicating.
hermes kanban create "nightly ops review" \
    --assignee ops \
    --idempotency-key "nightly-ops-$(date -u +%Y-%m-%d)" \
    --json
```

### Групповые глаголы CLI

Все глаголы жизненного цикла принимают несколько идентификаторов, так что можно очистить пакет за одну команду:

```bash
hermes kanban complete t_abc t_def t_hij --result "batch wrap"
hermes kanban archive  t_abc t_def t_hij
hermes kanban unblock  t_abc t_def
hermes kanban block    t_abc "need input" --ids t_def t_hij
```
## How workers interact with the board

**Workers do not shell out to `hermes kanban`.** When the dispatcher spawns a worker it sets `HERMES_KANBAN_TASK=t_abcd` in the child's env, and that env var flips on a dedicated **kanban toolset** in the model's schema. The same toolset is also available to orchestrator profiles that enable `kanban` in their toolsets config. These tools read and mutate the board directly via the Python `kanban_db` layer, same as the CLI does. A running worker calls these like any other tool; it never sees or needs the `hermes kanban` CLI.

| Инструмент | Назначение | Требуемые параметры |
|---|---|---|
| `kanban_show` | Чтение текущей задачи (заголовок, тело, предыдущие попытки, родительские передачи, комментарии, полностью предварительно отформатированный `worker_context`). По умолчанию берёт id задачи из переменной окружения. | — |
| `kanban_list` | Список сводок задач с фильтрами по `assignee`, `status`, `tenant`, видимости архивных и лимиту. Предназначен для оркестраторов, ищущих работу на доске. | — |
| `kanban_complete` | Завершение с `summary` + структурированной передачей `metadata`. | хотя бы один из `summary` / `result` |
| `kanban_block` | Эскалация для человеческого ввода с указанием `reason`. | `reason` |
| `kanban_heartbeat` | Сигнал о живости во время длительных операций. Чистый побочный эффект. | — |
| `kanban_comment` | Добавление долговечного примечания в ветку задачи. | `task_id`, `body` |
| `kanban_create` | (Orchestrators) разветвление в дочерние задачи с указанием `assignee`, опционально `parents`, `skills` и т.д. | `title`, `assignee` |
| `kanban_link` | (Orchestrators) добавить зависимость `parent_id → child_id` после создания. | `parent_id`, `child_id` |
| `kanban_unblock` | (Orchestrators) переместить заблокированную задачу обратно в `ready`. | `task_id` |

Типичный ход рабочего выглядит так:

```
# Model's tool calls, in order:
kanban_show()                                     # no args — uses HERMES_KANBAN_TASK
# (model reads the returned worker_context, does the work via terminal/file tools)
kanban_heartbeat(note="halfway through — 4 of 8 files transformed")
# (more work)
kanban_complete(
    summary="migrated limiter.py to token-bucket; added 14 tests, all pass",
    metadata={"changed_files": ["limiter.py", "tests/test_limiter.py"], "tests_run": 14},
)
```

**Orchestrator** вместо этого разветвляет задачи:

```
kanban_show()
kanban_create(
    title="research ICP funding 2024-2026",
    assignee="researcher-a",
    body="focus on seed + series A, North America, AI-adjacent",
)
# → returns {"task_id": "t_r1", ...}
kanban_create(title="research ICP funding — EU angle", assignee="researcher-b", body="…")
# → returns {"task_id": "t_r2", ...}
kanban_create(
    title="synthesize findings into launch brief",
    assignee="writer",
    parents=["t_r1", "t_r2"],                     # promotes to ready when both complete
    body="one-pager, 300 words, neutral tone",
)
kanban_complete(summary="decomposed into 2 research tasks + 1 writer; linked dependencies")
```

Инструменты, помеченные «(Orchestrators)» — `kanban_list`, `kanban_create`, `kanban_link`, `kanban_unblock` и `kanban_comment` для чужих задач — доступны через тот же набор инструментов; конвенция (принудительно задаваемая навыком `kanban-orchestrator`) состоит в том, что профили рабочих не разветвляют и не маршрутизируют несвязанные задачи, а профили оркестраторов не выполняют реализацию. Рабочие, порождённые диспетчером, остаются ограниченными задачей для деструктивных операций жизненного цикла и не могут изменять посторонние задачи.

### Почему инструменты, а не вызов `hermes kanban` через оболочку

Три причины:

1. **Портативность бекенда.** Рабочие, чей терминальный инструмент указывает на удалённый бекенд (Docker / Modal / Singularity / SSH), запускали бы `hermes kanban complete` *внутри* контейнера, где `hermes` не установлен и `~/.hermes/kanban.db` не смонтирован. Инструменты kanban работают в собственном Python‑процессе агента и всегда достигают `~/.hermes/kanban.db`, независимо от терминального бекенда.
2. **Отсутствие хрупкости кавычек оболочки.** Передача `--metadata '{"files": [...]}'` через shlex + argparse — скрытая ловушка. Структурированные аргументы инструмента полностью её обходят.
3. **Лучшие ошибки.** Результаты инструмента — структурированный JSON, который модель может анализировать, а не строки stderr, которые нужно парсить.

**Нулевой след схемы в обычных сессиях.** Обычная сессия `hermes chat` не содержит инструментов `kanban_*` в своей схеме, если только активный профиль явно не включил набор `kanban` для оркестрационной работы. Рабочие, порождённые диспетчером, получают инструменты, ограниченные задачей, потому что установлена `HERMES_KANBAN_TASK`; профили оркестраторов получают более широкий набор через конфиг. Нет лишних инструментов для пользователей, которые никогда не работают с kanban.

Навыки `kanban-worker` и `kanban-orchestrator` обучают модель, какой инструмент вызывать и в каком порядке.

### Рекомендуемая форма передачи доказательств

`kanban_complete(summary=..., metadata={...})` намеренно гибка: `summary` — человекочитаемое завершение, а `metadata` — машиночитаемая передача, которую downstream‑агенты, ревьюеры или дашборды могут использовать без парсинга текста.

Для инженерных и ревью‑задач предпочтительна следующая необязательная структура `metadata`:

```json
{
  "changed_files": ["path/to/file.py"],
  "verification": ["pytest tests/hermes_cli/test_kanban_db.py -q"],
  "dependencies": ["parent task id or external issue, if any"],
  "blocked_reason": null,
  "retry_notes": "what failed before, if this was a retry",
  "residual_risk": ["what was not tested or still needs human review"]
}
```

Эти ключи — конвенция, а не требование схемы. Главное свойство — каждый рабочий оставляет достаточно доказательств, чтобы следующий читатель быстро ответил на четыре вопроса:

1. Что изменилось?
2. Как это было проверено?
3. Что может разблокировать или повторить это при сбое?
4. Какой риск всё ещё намеренно оставлен открытым?

Не помещай секреты, сырые логи, токены, OAuth‑материалы и несвязанные транскрипты в `metadata`. Храни ссылки и сводки вместо этого. Если у задачи нет файлов или тестов, явно укажи это в `summary` и используй `metadata` для существующих доказательств, например URL исходного кода, идентификаторов задач или шагов ручного ревью.

### Навык рабочего

Любой профиль, который должен работать с kanban‑задачами, должен загрузить навык `kanban-worker`. Он обучает рабочего полному жизненному циклу **вызовов инструментов**, а не CLI‑команд:

1. При старте вызвать `kanban_show()` для чтения заголовка + тела + родительских передач + предыдущих попыток + полной ветки комментариев.
2. `cd $HERMES_KANBAN_WORKSPACE` (через терминальный инструмент) и выполнить работу там.
3. Вызывать `kanban_heartbeat(note="...")` каждые несколько минут во время длительных операций. **Если работа может длиться более часа, вызывай `kanban_heartbeat` хотя бы раз в час** — диспетчер возвращает задачи, которые работают дольше `kanban.dispatch_stale_timeout_seconds` (по умолчанию 4 ч) без heartbeat за последний час, полагая, что рабочий упал без очистки. Возврат безвреден (задача переходит в `ready` для повторного диспетчирования без увеличения счётчика неудач), но ты теряешь прогресс текущего запуска.
4. Завершить с `kanban_complete(summary="...", metadata={...})` или `kanban_block(reason="...")`, если застрял.

Этот финальный вызов `kanban_complete` / `kanban_block` является частью протокола рабочего. Если процесс рабочего завершается со статусом 0, пока задача всё ещё `running`, диспетчер считает это нарушением протокола, генерирует событие `protocol_violation` и автоматически блокирует задачу на следующем тике вместо повторного спауна в том же цикле. Обычно это означает, что модель выдала простой текстовый ответ и завершилась, не используя поверхность Kanban‑инструментов.

`kanban-worker` — встроенный навык, синхронизируемый в каждый профиль при установке и обновлении — отдельный шаг установки из Skills Hub не требуется. Убедись, что он присутствует в том профиле, который ты используешь для kanban‑рабочих (`researcher`, `writer`, `ops` и т.д.):

```bash
hermes -p <your-worker-profile> skills list | grep kanban-worker
```

Если встроенная копия отсутствует, восстанови её для этого профиля:

```bash
hermes -p <your-worker-profile> skills reset kanban-worker --restore
```

Диспетчер также автоматически передаёт `--skills kanban-worker` при спауне каждого рабочего, так что набор библиотек всегда доступен, даже если конфиг профиля по умолчанию не включает его.

### Привязка дополнительных навыков к конкретной задаче

Иногда отдельной задаче нужен специализированный контекст, которого нет в профиле исполнителя по умолчанию — переводческий заказ, требующий навыка `translation`, ревью‑задача, требующая `github-code-review`, аудит безопасности, требующий `security-pr-audit`. Вместо того чтобы каждый раз менять профиль исполнителя, прикрепи навыки непосредственно к задаче.

**От оркестратор‑агента** (обычный случай — один агент маршрутизирует работу другому) используй массив `skills` в инструменте `kanban_create`:

```
kanban_create(
    title="translate README to Japanese",
    assignee="linguist",
    skills=["translation"],
)

kanban_create(
    title="audit auth flow",
    assignee="reviewer",
    skills=["security-pr-audit", "github-code-review"],
)
```

**От человека (CLI / slash‑команда)**, укажи `--skill` для каждого навыка:

```bash
hermes kanban create "translate README to Japanese" \
    --assignee linguist \
    --skill translation

hermes kanban create "audit auth flow" \
    --assignee reviewer \
    --skill security-pr-audit \
    --skill github-code-review
```

**Из дашборда** введи навыки через запятую в поле **skills** формы быстрого создания.

Эти навыки **добавляются** к встроенному `kanban-worker` — диспетчер передаёт один флаг `--skills <name>` для каждого (и для встроенного), так что рабочий стартует со всеми ними загруженными. Имена навыков должны соответствовать тем, что действительно установлены в профиле исполнителя (см. `hermes skills list`); установка во время выполнения не поддерживается.

### Навык оркестратора

**Хорошо себя ведущий оркестратор не выполняет работу сам.** Он разбивает цель пользователя на задачи, связывает их, назначает каждой из профилей, которые ты настроил, и отстраняется. Навык `kanban-orchestrator` кодирует это как шаблоны вызовов инструментов: правила против искушения, запрос профиля на шаге 0 (диспетчер молча отказывает неизвестные имена исполнителей, поэтому оркестратор должен привязывать каждую карточку к реально существующим профилям), и план разложения, основанный на `kanban_create` / `kanban_link` / `kanban_comment`.

Канонический ход оркестратора (два параллельных исследователя передают задачу писателю):

```
# Goal from user: "draft a launch post on the ICP funding landscape"
kanban_create(title="research ICP funding, NA angle",  assignee="researcher-a", body="…")  # → t_r1
kanban_create(title="research ICP funding, EU angle",  assignee="researcher-b", body="…")  # → t_r2
kanban_create(
    title="synthesize ICP funding research into launch post draft",
    assignee="writer",
    parents=["t_r1", "t_r2"],        # promoted to 'ready' when both researchers complete
    body="one-pager, neutral tone, cite sources inline",
)                                     # → t_w1
# Optional: add cross-cutting deps discovered later without re-creating tasks
kanban_link(parent_id="t_r1", child_id="t_followup")
kanban_complete(
    summary="decomposed into 2 parallel research tasks → 1 synthesis task; writer starts when both researchers finish",
)
```

`kanban-orchestrator` — встроенный навык. Он синхронизируется в каждый профиль при установке и обновлении, отдельный шаг установки из Skills Hub не нужен. Убедись, что он присутствует в твоём профиле оркестратора:

```bash
hermes -p orchestrator skills list | grep kanban-orchestrator
```

Если встроенная копия отсутствует, восстанови её для этого профиля:

```bash
hermes -p orchestrator skills reset kanban-orchestrator --restore
```

Для наилучших результатов сочетай его с профилем, набор инструментов которого ограничен операциями над доской (`kanban`, `gateway`, `memory`), чтобы оркестратор буквально не мог выполнять задачи реализации, даже если попытается.
## Панель управления (GUI)

CLI и слеш‑команда `/kanban` достаточно, чтобы запускать доску без интерфейса, но визуальная доска часто является правильным интерфейсом для людей‑в‑цикле: триаж, надзор над профилями, чтение веток комментариев и перетаскивание карточек между колонками. Hermes поставляется с этим как **встроенный плагин панели** в `plugins/kanban/` — не ядро, не отдельный сервис — согласно модели, описанной в [Extending the Dashboard](./extending-the-dashboard).

Открой её с помощью:

```bash
hermes kanban init      # one-time: create kanban.db if not already present
hermes dashboard        # "Kanban" tab appears in the nav, after "Skills"
```

### Что даёт плагин

- Вкладка **Kanban**, показывающая одну колонку на каждый статус: `triage`, `todo`, `ready`, `running`, `blocked`, `done` (плюс `archived`, когда переключатель включён).
  - `triage` — колонка‑парковка для грубых идей. По умолчанию (`kanban.auto_decompose: true`) диспетчер автоматически запускает **декомпозер** для задач, попадающих сюда — профиль‑оркестратор читает грубую идею, смотрит ваш список профилей (с описаниями) и разветвляет задачу в небольшой граф дочерних задач, направленных к наиболее подходящим специалистам. Исходная задача остаётся живой как родитель всех дочерних, чтобы оркестратор проснулся и оценил завершение, когда всё закончится. Переключи таблетку **Orchestration: Auto/Manual** в верхней части страницы (или задай `kanban.auto_decompose: false`), чтобы перейти в ручной режим, где задачи триажа остаются на месте, пока ты не нажмёшь **⚗ Decompose** на карточке или не выполнит `hermes kanban decompose <id>`. Для задач, которым не нужен fan‑out (или для настроек без профиля‑оркестратора), кнопка **✨ Specify** делает переписывание спецификации одной задачи (заголовок + тело с целью, подходом, критериями приёмки) тем же механизмом LLM. См. [Auto vs Manual orchestration](#auto-vs-manual-orchestration) ниже.
- Карточки показывают id задачи, заголовок, бейдж приоритета, тег арендатора, назначенный профиль, количество комментариев/ссылок, **таблетку прогресса** (`N/M` дочерних выполнено, когда у задачи есть зависимости) и «создано N назад». Флажок на каждой карточке включает мультивыбор.
- **Полосы по профилям внутри Running** — чекбокс в тулбаре переключает подгруппировку колонки Running по исполнителю.
- **Live‑обновления через WebSocket** — плагин отслеживает только‑для‑добавления таблицу `task_events` с коротким интервалом опроса; доска отражает изменения мгновенно, как только любой профиль (CLI, gateway или другая вкладка панели) их делает. Перезагрузки дебаунсятся, так что всплеск событий приводит к единственному повторному запросу.
- **Перетаскивание** карточек между колонками меняет статус. При падении отправляется `PATCH /api/plugins/kanban/tasks/:id`, который проходит через тот же код `kanban_db`, что и CLI — три поверхности никогда не расходятся. Перемещения в деструктивные статусы (`done`, `archived`, `blocked`) требуют подтверждения. На сенсорных устройствах используется запасной (fallback) вариант на указатель, так что доска работает и на планшете.
- **Создание inline** — нажми `+` в заголовке любой колонки, чтобы ввести заголовок, исполнителя, приоритет и (по желанию) родительскую задачу из выпадающего списка над каждой существующей задачей. Нажми **Enter**, чтобы создать задачу, **Shift+Enter** — вставить перевод строки в поле заголовка, **Escape** — отменить. Создание из колонки **Triage** автоматически паркует новую задачу в триаже.
- **Мультивыбор с массовыми действиями** — shift/ctrl‑клик по карточке или отметка её чекбокса добавляют её в выбор. Появляется панель массовых действий вверху с пакетными переходами статусов, архивированием и переназначением (через выпадающий список профилей или «(unassign)»). Деструктивные пакеты требуют подтверждения. Частичные неудачи по отдельным id сообщаются без отмены остальных.
- **Клик по карточке** (без shift/ctrl) открывает боковой выдвижной блок (Escape или клик вне закрывает) с:
  - **Редактируемым заголовком** — кликни заголовок, чтобы переименовать.
  - **Редактируемым исполнителем / приоритетом** — кликни строку метаданных, чтобы изменить.
  - **Редактируемым описанием** — по умолчанию рендерится markdown (заголовки, жирный, курсив, inline‑code, fenced‑code, ссылки `http(s)` / `mailto:`, маркированные списки), с кнопкой «edit», которая заменяет его на textarea. Рендерер markdown — крошечный, XSS‑безопасный: каждый подстановочный процесс работает с HTML‑экранированным вводом, только ссылки `http(s)` / `mailto:` проходят, а `target="_blank"` + `rel="noopener noreferrer"` всегда задаются.
  - **Редактор зависимостей** — список чипов родителей и детей, каждый с `×` для отвязки, плюс выпадающие списки над каждой другой задачей для добавления нового родителя или ребёнка. Попытки создать цикл отклоняются сервером с чётким сообщением.
  - **Строка действий статуса** (→ triage / → ready / → running / block / unblock / complete / archive) с подтверждающими запросами для деструктивных переходов. Для карточек в колонке **Triage** строка также предлагает два действия, управляемые LLM: **⚗ Decompose** разветвляет задачу в граф дочерних задач, направленных к профильным специалистам (путь оркестратора), и **✨ Specify** делает переписывание спецификации одной задачи. Decompose переходит к стилю Specify, когда LLM решает, что задача не выигрывает от fan‑out, так что это строгий супермножеством. Оба доступны из CLI (`hermes kanban decompose <id>` / `specify <id>` / `--all`), из любого шлюза (`/kanban decompose <id>`), и программно через `POST /api/plugins/kanban/tasks/:id/decompose` и `…/specify`. Настройте модели в `auxiliary.kanban_decomposer` и `auxiliary.triage_specifier` в `config.yaml`.
  - Раздел результата (также markdown‑рендерится), ветка комментариев с отправкой по **Enter**, последние 20 событий.
- **Фильтры тулбара** — поиск по свободному тексту, выпадающий список арендаторов (по умолчанию `dashboard.kanban.default_tenant` из `config.yaml`), выпадающий список исполнителей, переключатель «show archived», переключатель «lanes by profile» и кнопка **Nudge dispatcher**, чтобы не ждать очередного 60‑секундного тика.

Визуально цель — знакомый макет Linear / Fusion: тёмная тема, заголовки колонок с подсчётами, цветные точки статуса, таблеточные чипы приоритета и арендатора. Плагин читает только CSS‑переменные темы (`--color-*`, `--radius`, `--font-mono`, …), поэтому автоматически переоформляется под любую активную тему панели.

### Auto vs Manual orchestration

Kanban‑доска имеет два способа обработки задачи, брошенной в колонку **Triage**:

**Auto (по умолчанию)** — `kanban.auto_decompose: true`. Встроенный в шлюз диспетчер запускает **декомпозер** каждый тик, ограниченный `kanban.auto_decompose_per_tick` (по умолчанию 3 задачи за тик), чтобы массовый поток триаж‑задач не «взорвал» вспомогательный LLM. Декомпозер читает грубую идею, смотрит ваши установленные профили + их описания и просит LLM сгенерировать JSON‑граф задач: какие задачи создать, кому их отправить и какие зависимости между ними. Исходная задача‑триаж становится родителем каждой листовой задачи в графе, остаётся живой, пока весь граф не завершится — затем продвигается обратно в `ready`, чтобы её исполнитель (профиль‑оркестратор) мог оценить завершение и добавить новые задачи, если работа не закончена. Это поток «бросил однострочник, ушёл».

**Manual** — `kanban.auto_decompose: false`. Задачи триажа остаются в триаже, пока ты не вмешаешься. Нажми кнопку **⚗ Decompose** на карточке, выполни `hermes kanban decompose <id>` (или `--all`), либо используй `/kanban decompose <id>` в чате. Это соответствует поведению доски до декомпозера, удобно, когда нужен полный контроль над тем, что и когда запускается.

Переключай режимы через таблетку **Orchestration: Auto/Manual** в верхней части kanban‑страницы (emerald = Auto, muted gray = Manual) или редактируя `config.yaml` напрямую. Оба режима сосуществуют с `hermes kanban specify` — он всё ещё доступен как переписывание спецификации одной задачи, когда не нужен fan‑out.

Решения маршрутизации декомпозера зависят от описаний профилей, что является примитивом маркировки профиля, задаваемым через `hermes profile create --description "..."`, `hermes profile describe <name> --text "..."`, `hermes profile describe <name> --auto` (LLM‑генерирует из установленных навыков профиля + модели) или редактором профилей в панели в расширенной панели **Orchestration settings**. Профили без описания всё равно появляются в списке — их можно маршрутизировать по имени, но менее точно. Декомпозер НИКОГДА не создаёт дочернюю задачу с `assignee=None`: когда LLM выбирает неизвестный профиль, задача направляется в `kanban.default_assignee` (или в активный профиль‑по‑умолчанию, если он не задан).

Настройки (все под `kanban:` в `~/.hermes/config.yaml`):

| Ключ | По‑умолчанию | Назначение |
|---|---|---|
| `auto_decompose` | `true` | Диспетчер автоматически запускает декомпозер каждый тик. |
| `auto_decompose_per_tick` | `3` | Ограничение количества декомпозиций за тик диспетчера. Остальное откладывается на следующий тик. |
| `orchestrator_profile` | `""` | Профиль, владеющий декомпозицией. Пусто = запасной вариант к активному профилю‑по‑умолчанию. |
| `default_assignee` | `""` | Куда попадает дочерняя задача, если LLM выбирает неизвестный профиль. Пусто = запасной вариант к активному профилю‑по‑умолчанию. |

И два вспомогательных LLM‑слота:

| Ключ | Назначение |
|---|---|
| `auxiliary.kanban_decomposer` | Модель, генерирующая граф задач (вызывается Decompose). Установи `provider`/`model`, чтобы переопределить основную чат‑модель. |
| `auxiliary.profile_describer` | Модель, автоматически генерирующая описания профилей (вызывается `hermes profile describe --auto`). |

### REST‑поверхность

Все маршруты монтируются под `/api/plugins/kanban/` и защищены эпhemerным токеном сессии панели:

| Метод | Путь | Назначение |
|---|---|---|
| `GET` | `/board?tenant=<name>&include_archived=…` | Полная доска, сгруппированная по колонкам статуса, плюс арендаторы + исполнители для выпадающих фильтров |
| `GET` | `/tasks/:id` | Задача + комментарии + события + ссылки |
| `POST` | `/tasks` | Создание (обёртка `kanban_db.create_task`, принимает `triage: bool` и `parents: [id, …]`) |
| `PATCH` | `/tasks/:id` | Статус / исполнитель / приоритет / заголовок / тело / результат |
| `POST` | `/tasks/bulk` | Применить один и тот же патч (status / archive / assignee / priority) ко всем id в `ids`. Ошибки по отдельным id сообщаются без отмены остальных |
| `POST` | `/tasks/:id/comments` | Добавить комментарий |
| `POST` | `/tasks/:id/specify` | Запустить триаж‑спецификатор — вспомогательный LLM заполняет тело задачи и переводит её из `triage` в `todo`. Возвращает `{ok, task_id, reason, new_title}`; `ok=false` с человекочитаемой причиной при «не в триаже», отсутствии aux‑клиента или ошибке LLM — 200, а не 4xx |
| `POST` | `/tasks/:id/decompose` | Запустить kanban‑декомпозер — вспомогательный LLM создаёт граф задач и атомарно создаёт детей + связывает корень + переключает `triage → todo`. Возвращает `{ok, task_id, reason, fanout, child_ids, new_title}`. Та же конвенция 200‑on‑LLM‑error, что и у `/specify`. |
| `GET` | `/profiles` | Список установленных профилей с их описаниями (используется редактором описаний профилей в панели и picker‑ом оркестратора). |
| `PATCH` | `/profiles/:name` | Установить или очистить описание профиля (пользовательское — `description_auto: false`). Возвращает `{ok, profile, description}`. |
| `POST` | `/profiles/:name/describe-auto` | Сгенерировать описание профиля через `auxiliary.profile_describer`. Сохраняет с `description_auto: true`, чтобы панель могла показать бейдж «review». |
| `GET` | `/orchestration` | Чтение настроек оркестрации kanban (`orchestrator_profile`, `default_assignee`, `auto_decompose`) плюс *разрешённые* эффективные значения после fallback‑ов. |
| `PUT` | `/orchestration` | Обновить один или несколько из трёх ключей оркестрации в `config.yaml`. Проверяет, что непустые имена профилей действительно существуют. |
| `POST` | `/links` | Добавить зависимость (`parent_id` → `child_id`) |
| `DELETE` | `/links?parent_id=…&child_id=…` | Удалить зависимость |
| `POST` | `/dispatch?max=…&dry_run=…` | Подтолкнуть диспетчер — пропустить 60‑секундное ожидание |
| `GET` | `/config` | Чтение предпочтений `dashboard.kanban` из `config.yaml` — `default_tenant`, `lane_by_profile`, `include_archived_by_default`, `render_markdown` |
| `WS` | `/events?since=<event_id>` | Live‑стрим строк `task_events` |

Каждый обработчик — тонкая обёртка; плагин ≈ 700 строк Python (router + WebSocket tail + bulk batcher + config reader) и не добавляет новой бизнес‑логики. Маленький помощник `_conn()` автоматически инициализирует `kanban.db` при каждом чтении и записи, так что свежая установка работает независимо от того, открыл ли пользователь панель первой, обратился к REST‑API напрямую или запустил `hermes kanban init`.

### Конфиг панели

Любой из этих ключей под `dashboard.kanban` в `~/.hermes/config.yaml` меняет значения по умолчанию вкладки — плагин читает их при загрузке через `GET /config`:

```yaml
dashboard:
  kanban:
    default_tenant: acme              # preselects the tenant filter
    lane_by_profile: true             # default for the "lanes by profile" toggle
    include_archived_by_default: false
    render_markdown: true             # set false for plain <pre> rendering
```

Каждый ключ опционален и падает к показанному значению по умолчанию.

### Модель безопасности

HTTP‑middleware панели [явно пропускает `/api/plugins/`](./extending-the-dashboard#backend-api-routes) — маршруты плагина не аутентифицируются по дизайну, потому что панель по умолчанию привязывается к localhost. Это значит, что kanban‑REST‑поверхность доступна любому процессу на хосте.

WebSocket делает один дополнительный шаг: требует эпhemerный токен сессии панели как параметр запроса `?token=…` (браузеры не могут установить `Authorization` в запросе апгрейда), используя тот же шаблон, что и in‑browser PTY bridge.

Если запустить `hermes dashboard --host 0.0.0.0`, каждый маршрут плагина — включая kanban — станет доступен из сети. **Не делай этого на общем хосте.** Доска содержит тела задач, комментарии и пути рабочего пространства; злоумышленник, получивший доступ к этим маршрутам, получает чтение всей вашей коллаборационной поверхности и может также создавать / переназначать / архивировать задачи.

Задачи в `~/.hermes/kanban.db` намеренно не зависят от профиля (это примитив координации). Если открыть панель с `hermes -p <profile> dashboard`, доска всё равно покажет задачи, созданные любым другим профилем на хосте. Один пользователь владеет всеми профилями, но это стоит знать, если существует несколько персон.

### Live‑обновления

`task_events` — это только‑для‑добавления таблица SQLite с монотонным `id`. WebSocket‑endpoint хранит последний полученный id каждого клиента и пушит новые строки по мере их появления. При всплеске событий фронтенд перезагружает (очень дешёвый) endpoint доски — проще и правильнее, чем пытаться патчить локальное состояние от каждого типа события. WAL‑режим гарантирует, что цикл чтения никогда не блокирует `BEGIN IMMEDIATE` транзакции диспетчера.

### Расширение

Плагин использует стандартный контракт плагинов панели Hermes — см. [Extending the Dashboard](./extending-the-dashboard) для полного справочника манифеста, слотов оболочки, слотов scoped‑по‑странице и Plugin SDK. Дополнительные колонки, кастомный chrome карточек, макеты с фильтрацией по арендаторам или полные замены `tab.override` полностью выражаемы без форка этого плагина.

Чтобы отключить без удаления: добавьте `dashboard.plugins.kanban.enabled: false` в `config.yaml` (или удалите `plugins/kanban/dashboard/manifest.json`).

### Границы области

GUI преднамеренно тонкое. Всё, что делает плагин, доступно из CLI; плагин лишь делает это удобным для людей. Авто‑назначения, бюджеты, ворота управления, представления орг‑структур остаются в пользовательском пространстве — профиль‑router, другой плагин или переиспользование `tools/approval.py` — точно как указано в разделе out‑of‑scope спецификации дизайна.
## Справочник команд CLI

Это поверхность, которую **ты** (или скрипты, cron, панель) используешь для управления доской. Работники, запущенные внутри диспетчера, используют `kanban_*` [интерфейс инструментов](#how-workers-interact-with-the-board) для тех же операций — CLI здесь и инструменты там оба маршрутизируются через `kanban_db`, поэтому две поверхности согласованы по конструкции.

```
hermes kanban init                                     # create kanban.db + print daemon hint
hermes kanban create "<title>" [--body ...] [--assignee <profile>]
                                [--parent <id>]... [--tenant <name>]
                                [--workspace scratch|worktree|worktree:<path>|dir:<path>]
                                [--branch <name>]
                                [--priority N] [--triage] [--idempotency-key KEY]
                                [--max-runtime 30m|2h|1d|<seconds>]
                                [--max-retries N]
                                [--skill <name>]...
                                [--json]
hermes kanban list [--mine] [--assignee P] [--status S] [--tenant T] [--archived]
        [--workflow-template-id <id>] [--current-step-key <key>]
        [--sort created|created-desc|priority|priority-desc|status|assignee|title|updated]
        [--json]
hermes kanban show <id> [--json]
hermes kanban assign <id> <profile>                    # or 'none' to unassign
hermes kanban link <parent_id> <child_id>
hermes kanban unlink <parent_id> <child_id>
hermes kanban claim <id> [--ttl SECONDS]
hermes kanban comment <id> "<text>" [--author NAME]

# Bulk verbs — accept multiple ids:
hermes kanban complete <id>... [--result "..."]
hermes kanban block <id> "<reason>" [--ids <id>...]
hermes kanban unblock <id>...
hermes kanban archive <id>...

hermes kanban tail <id>                                # follow a single task's event stream
hermes kanban watch [--assignee P] [--tenant T]        # live stream ALL events to the terminal
        [--kinds completed,blocked,…] [--interval SECS]
hermes kanban heartbeat <id> [--note "..."]            # worker liveness signal for long ops
hermes kanban runs <id> [--json]                       # attempt history (one row per run)
hermes kanban assignees [--json]                       # profiles on disk + per-assignee task counts
hermes kanban dispatch [--dry-run] [--max N]           # one-shot pass
        [--failure-limit N] [--json]
hermes kanban daemon --force                           # DEPRECATED — standalone dispatcher (use `hermes gateway start` instead)
        [--failure-limit N] [--pidfile PATH] [-v]
hermes kanban stats [--json]                           # per-status + per-assignee counts
hermes kanban log <id> [--tail BYTES]                  # worker log from ~/.hermes/kanban/logs/
hermes kanban notify-subscribe <id>                    # gateway bridge hook (used by /kanban in the gateway)
        --platform <name> --chat-id <id> [--thread-id <id>] [--user-id <id>]
hermes kanban notify-list [<id>] [--json]
hermes kanban notify-unsubscribe <id>
        --platform <name> --chat-id <id> [--thread-id <id>]
hermes kanban context <id>                             # what a worker sees
hermes kanban specify [<id> | --all] [--tenant T]      # flesh out a triage-column idea
        [--author NAME] [--json]                       #   into a full spec and promote to todo
hermes kanban gc [--event-retention-days N]            # workspaces + old events + old logs
        [--log-retention-days N]
```

Все команды также доступны как slash‑команда в интерактивном CLI и в шлюзе обмена сообщениями (см. slash‑команду [`/kanban`](#kanban-slash-command) ниже).

`--max-retries` — переопределение схемы circuit‑breaker для отдельной задачи в диспетчере. `--max-retries 1` блокирует задачу после первой неуспешной попытки, тогда как `--max-retries 3` позволяет две повторные попытки и блокирует на третьей неудаче. Опусти параметр, чтобы использовать `kanban.failure_limit` из `config.yaml`; тогда применяется встроенное значение по умолчанию.

### Конфигурация параллелизма, планирования и продвижения дочерних задач

| Ключ конфигурации | По умолчанию | Что делает |
|-------------------|--------------|------------|
| `kanban.max_in_progress` | unset (не ограничено) | Ограничивает количество одновременно запущенных задач. Когда на доске уже запущено N задач, диспетчер пропускает создание новых — полезно для медленных работников (локальные LLM, хосты с ограниченными ресурсами), чтобы они завершили текущие задачи, прежде чем появятся новые и истечёт время ожидания. Неверные значения или значения ниже 1 записывают предупреждение и ведут себя как неограниченные. |
| `kanban.auto_promote_children` | `true` | После того как `decompose_triage_task()` создаёт дочерние задачи без зависимостей‑блокировок, они автоматически продвигаются в `ready`, чтобы диспетчер мог их взять. Установи `false`, чтобы требовать ручного обзора — дочерние задачи остаются в `todo`, пока ты их не продвинешь. |
| `kanban.default_workdir` | unset | Значение рабочего каталога по умолчанию на уровне доски, применяемое к новым задачам, если не указано ни `--workspace`, ни собственное переопределение задачи. Параметр `workspace:` в задаче всё равно имеет приоритет. |

```yaml
kanban:
  max_in_progress: 2
  auto_promote_children: false
  default_workdir: ~/work/active-project
```

### Запуск запланированных задач (`scheduled_at`)

Установи `scheduled_at` у задачи, чтобы отложить её диспетчирование до конкретного времени. Диспетчер пропускает готовые задачи, у которых `scheduled_at` находится в будущем, и берёт их в работу на первом тике после этой метки времени.

```bash
hermes kanban create "nightly backup audit" \
  --assignee ops --scheduled-at "2026-06-01T03:00:00Z"
```

### Защита от повторного запуска

Диспетчер отказывается повторно запускать готовую задачу, если она столкнулась с ошибкой quota/auth/429 в предыдущем запуске (`blocker_auth`), или успешно завершилась в пределах окна защиты (`recent_success`), или недавний комментарий к задаче ссылается на Pull Request в GitHub (`active_pr`). Это предотвращает повторные штормы работников над одной ошибкой или задачей, пока человек не вмешается. См. строку `respawn_guarded` в [справочнике событий](#event-reference).

### Перетаскивание в корзину и массовое удаление (панель)

Панель отображает **зону сброса в корзину** на странице kanban — перетащи любую карточку в эту зону, чтобы удалить задачу (это каскадно удалит `task_events`, ссылки на дочерние задачи и подписки). Подтверждающий диалог защищает от случайных удалений. Массовое удаление также доступно через `DELETE /api/plugins/kanban/tasks` с JSON‑телом `{"ids": ["t_abc", "t_def", ...]}`.

### Эндпоинты видимости работников

Плагин API панели теперь предоставляет три только‑для‑чтения эндпоинта для внешних мониторингов:

| Эндпоинт | Возвращает |
|----------|------------|
| `GET /api/plugins/kanban/workers/active` | Текущие запущенные работники с PID, профилем, id задачи, временем старта и последним heartbeat |
| `GET /api/plugins/kanban/runs/{id}` | Детали отдельного запуска — id задачи, статус, время начала/завершения, код выхода, путь к журналу |
| `GET /api/plugins/kanban/inspect` | Снимок состояния диспетчера — очередь, количество запущенных задач vs. `max_in_progress`, последние события |

Все три защищены тем же механизмом аутентификации плагина панели, что и остальные эндпоинты API плагина kanban.

### Помощник топологии Kanban Swarm

`hermes kanban swarm` создаёт долговечный граф **Kanban Swarm v1** в один шаг: завершённую корневую/blackboard‑карточку, N параллельных карточек работников, карточку‑верификатор, блокированную всеми работниками, и карточку‑синтезатор, блокированную верификатором. Общий контекст роя («blackboard») хранится как структурированные JSON‑комментарии в корневой карточке, чтобы любой работник мог их прочитать.

```bash
hermes kanban swarm "Design a multi-region failover plan" \
  --workers researcher,architect,sre \
  --verifier reviewer --synthesizer writer
```

Получившийся граф диспетчеризуется как обычно — работники работают параллельно, верификатор просыпается после их завершения, синтезатор просыпается после того, как верификатор пометит работу как чистую.
## `/kanban` slash command {#kanban-slash-command}

Каждый глагол `hermes kanban <action>` также доступен как `/kanban <action>` — из интерактивной сессии `hermes chat` **и** с любой платформы‑gateway (Telegram, Discord, Slack, WhatsApp, Signal, Matrix, Mattermost, email, SMS). Оба интерфейса вызывают один и тот же входной пункт `hermes_cli.kanban.run_slash()`, который переиспользует дерево аргументов `hermes kanban`, поэтому поверхность аргументов, флаги и формат вывода одинаковы в CLI, `/kanban` и `hermes kanban`. Тебе не нужно покидать чат, чтобы управлять доской.

```
/kanban list
/kanban show t_abcd
/kanban create "write launch post" --assignee writer --parent t_research
/kanban comment t_abcd "looks good, ship it"
/kanban unblock t_abcd
/kanban dispatch --max 3
/kanban specify t_abcd                  # flesh out a triage one-liner into a real spec
/kanban specify --all --tenant engineering  # sweep every triage task in one tenant
```

Кавычки вокруг многословных аргументов используешь так же, как в оболочке — `run_slash` разбирает оставшуюся часть строки с помощью `shlex.split`, поэтому работают как `"..."`, так и `'...'`.

### Mid-run usage: `/kanban` bypasses the running-agent guard

Gateway обычно ставит slash‑команды и пользовательские сообщения в очередь, пока агент ещё думает — именно так предотвращается случайный запуск второго хода, пока первый ещё в полёте. **`/kanban` явно исключён из этого ограничения.** Доска хранится в `~/.hermes/kanban.db`, а не в состоянии работающего агента, поэтому чтения (`list`, `show`, `context`, `tail`, `watch`, `stats`, `runs`) и записи (`comment`, `unblock`, `block`, `assign`, `archive`, `create`, `link`, …) выполняются сразу, даже в середине хода.

Это и есть смысл разделения:

- Worker блокируется, ожидая пира → ты отправляешь `/kanban unblock t_abcd` со своего телефона, и диспетчер подхватывает пира на следующем тике. Заблокированный worker не прерывается — просто перестаёт быть заблокированным.
- Ты замечаешь карточку, которой нужен человеческий контекст → `/kanban comment t_xyz "use the 2026 schema, not 2025"` попадает в ветку задачи, и *следующий* запуск этой задачи прочитает её в `kanban_show()`.
- Хочешь узнать, что делает твой fleet, не останавливая orchestrator → `/kanban list --mine` или `/kanban stats` просматривают доску, не трогая основной разговор.

### Auto-subscribe on `/kanban create` (gateway only)

Когда ты создаёшь задачу через gateway с помощью `/kanban create "…"`, исходящий чат (платформа + chat id + thread id) автоматически подписывается на терминальные события задачи (`completed`, `blocked`, `gave_up`, `crashed`, `timed_out`). Ты получаешь одно сообщение на каждое терминальное событие — включая первую строку сводки результата worker‑а при `completed` — без необходимости опрашивать или запоминать id задачи.

```
you> /kanban create "transcribe today's podcast" --assignee transcriber
bot> Created t_9fc1a3  (ready, assignee=transcriber)
     (subscribed — you'll be notified when t_9fc1a3 completes or blocks)

… ~8 minutes later …

bot> ✓ t_9fc1a3 completed by transcriber
     transcribed 42 minutes, saved to podcast/2026-05-04.md
```

Подписки автоматически удаляются, как только задача переходит в `done` или `archived`. Если ты вызываешь `create` с `--json` (машинный вывод), авто‑подписка пропускается — предполагается, что скриптовые вызовы управляют подписками явно через `/kanban notify-subscribe`.

### Output truncation in messaging

Платформы‑gateway имеют практические ограничения на длину сообщения. Если `/kanban list`, `/kanban show` или `/kanban tail` генерируют более ~3800 символов вывода, ответ обрезается с футером `… (truncated; use \`hermes kanban …\` in your terminal for full output)`. В CLI такого ограничения нет.

### Autocomplete

В интерактивном CLI, вводя `/kanban ` и нажимая Tab, ты переключаешься по встроенному списку подкоманд (`list`, `ls`, `show`, `create`, `assign`, `link`, `unlink`, `claim`, `comment`, `complete`, `block`, `unblock`, `archive`, `tail`, `dispatch`, `context`, `init`, `gc`). Оставшиеся глаголы, перечисленные в справке CLI выше (`watch`, `stats`, `runs`, `log`, `assignees`, `heartbeat`, `notify-subscribe`, `notify-list`, `notify-unsubscribe`, `daemon`), тоже работают — они просто пока не отображаются в подсказке автодополнения.
## Паттерны сотрудничества

The board supports these eight patterns without any new primitives:

| Паттерн | Форма | Пример |
|---|---|---|
| **P1 Fan-out** | N подчинённых, одинаковая роль | «исследовать 5 углов параллельно» |
| **P2 Pipeline** | цепочка ролей: scout → editor → writer | сборка ежедневного брифа |
| **P3 Voting / quorum** | N подчинённых + 1 агрегатор | 3 исследователя → 1 рецензент выбирает |
| **P4 Long-running journal** | одинаковый профиль + общая директория + cron | хранилище Obsidian |
| **P5 Human-in-the-loop** | блоки рабочего → комментарии пользователя → разблокировка | неоднозначные решения |
| **P6 `@mention`** | встроенная маршрутизация из текста | `@reviewer look at this` |
| **P7 Thread-scoped workspace** | `/kanban here` в ветке | шлюз‑потоки для каждого проекта |
| **P8 Fleet farming** | один профиль, N субъектов | 50 социальных аккаунтов |
| **P9 Triage specifier** | грубая идея → `triage` → `hermes kanban specify` расширяет тело → `todo` | «превратить эту однострочку в задачу со спецификацией» |

For worked examples of each, see `docs/hermes-kanban-v1-spec.pdf`.
## Многоарендное использование

Когда один специализированный флот обслуживает несколько компаний, помечай каждую задачу тегом арендатора:

```bash
hermes kanban create "monthly report" \
    --assignee researcher \
    --tenant business-a \
    --workspace dir:~/tenants/business-a/data/
```

Работники получают `$HERMES_TENANT` и присваивают своим записям памяти пространство имён с префиксом. Доска, диспетчер и определения профилей общие; только данные ограничены конкретным арендатором.
## Уведомления шлюза

Когда ты запускаешь `/kanban create …` из шлюза (Telegram, Discord, Slack и т.д.), исходный чат автоматически подписывается на новую задачу. Фоновый процесс‑уведомитель шлюза опрашивает `task_events` каждые несколько секунд и отправляет одно сообщение о каждом терминальном событии (`completed`, `blocked`, `gave_up`, `crashed`, `timed_out`) в этот чат. Завершённые задачи также отправляют первую строку `--result` рабочего процесса, чтобы ты видел результат без необходимости выполнять `/kanban show`.

Подписки можно управлять явно из CLI — это полезно, когда скрипт / cron‑задача хочет уведомить чат, из которого она не была запущена:

```bash
hermes kanban notify-subscribe t_abcd \
    --platform telegram --chat-id 12345678 --thread-id 7
hermes kanban notify-list
hermes kanban notify-unsubscribe t_abcd \
    --platform telegram --chat-id 12345678 --thread-id 7
```

Подписка автоматически удаляется, как только задача переходит в состояние `done` или `archived`; очистка не требуется.
## Запуски — одна строка на попытку

Задача — логическая единица работы; **запуск** — одна попытка её выполнить. Когда диспетчер берёт готовую задачу, он создаёт строку в `task_runs` и указывает `tasks.current_run_id` на неё. Когда эта попытка заканчивается — выполнена, заблокирована, аварийно завершена, истёк тайм‑аут, не удалось запустить, возвращена — строка запуска закрывается полем `outcome`, а указатель задачи очищается. У задачи, которой пытались выполнить три раза, будет три строки в `task_runs`.

Почему две таблицы вместо простого изменения задачи: нужен **полный журнал попыток** для реальных постмортемов («вторая попытка ревьюера получила одобрение, третья — слияние»), и нужно чистое место для метаданных каждой попытки — какие файлы изменились, какие тесты запустились, какие замечания сделал ревьюер. Это факты запуска, а не факты задачи.

Запуски также являются местом, где живёт **структурированный handoff**. Когда воркер завершает задачу (через `kanban_complete(...)`) он может передать:

- `summary` (параметр инструмента) / `--summary` (CLI) — человеко‑ориентированный handoff; записывается в запуск; дочерние задачи видят его в своём `build_worker_context`.
- `metadata` (параметр инструмента) / `--metadata` (CLI) — произвольный JSON‑словарь в запуске; дочерние задачи видят его сериализованным рядом с `summary`.
- `result` (параметр инструмента) / `--result` (CLI) — короткая строка лога, записывается в строку задачи (унаследованное поле, оставлено для обратной совместимости).

Дочерние задачи читают `summary` + `metadata` последнего завершённого запуска каждого родителя. Перезапускающие воркеры читают предыдущие попытки своей задачи (`outcome`, `summary`, `error`), чтобы не повторять уже проваленный путь.

```
# What a worker actually does — a tool call, from inside the agent loop:
kanban_complete(
    summary="implemented token bucket, keys on user_id with IP fallback, all tests pass",
    metadata={"changed_files": ["limiter.py", "tests/test_limiter.py"], "tests_run": 14},
    result="rate limiter shipped",
)
```

Тот же handoff доступен из CLI, когда ты (человек) должен закрыть задачу, которую воркер не может завершить — например, задача была оставлена, или ты пометил её выполненной вручную из дашборда:

```bash
hermes kanban complete t_abcd \
    --result "rate limiter shipped" \
    --summary "implemented token bucket, keys on user_id with IP fallback, all tests pass" \
    --metadata '{"changed_files": ["limiter.py", "tests/test_limiter.py"], "tests_run": 14}'

# Review the attempt history on a retried task:
hermes kanban runs t_abcd
#   #  OUTCOME       PROFILE           ELAPSED  STARTED
#   1  blocked       worker               12s  2026-04-27 14:02
#        → BLOCKED: need decision on rate-limit key
#   2  completed     worker                8m   2026-04-27 15:18
#        → implemented token bucket, keys on user_id with IP fallback
```

Запуски отображаются в дашборде (раздел **Run History** в выдвижном меню, одна цветная строка на попытку) и через REST API (`GET /api/plugins/kanban/tasks/:id` возвращает массив `runs[]`). `PATCH /api/plugins/kanban/tasks/:id` с `{status: "done", summary, metadata}` пересылает оба поля ядру, поэтому кнопка дашборда «mark done» эквивалентна CLI. Строки `task_events` содержат `run_id`, к которому они относятся, чтобы UI мог группировать их по попыткам, а событие `completed` включает в полезную нагрузку `summary` первой строки (ограничено 400 символами), чтобы нотификаторы шлюза могли отобразить структурированный handoff без второго SQL‑запроса.

**Предупреждение при массовом закрытии.** `hermes kanban complete a b c --summary X` отклоняется — структурированный handoff привязан к отдельному запуску, поэтому копировать один и тот же `summary` в N задач почти всегда неправильно. Массовое закрытие *без* `--summary` / `--metadata` всё ещё работает для типичного случая «я завершил кучу административных задач».

**Возвращённые запуски при изменении статуса.** Если ты перетаскиваешь запущенную задачу из `running` в дашборде (назад в `ready` или сразу в `todo`), или архивируешь задачу, которая всё ещё выполнялась, текущий запуск завершается с `outcome='reclaimed'`, а не остаётся «сиротой». Строка `task_runs` всегда находится в терминальном состоянии, когда `tasks.current_run_id` равно `NULL`, и наоборот — этот инвариант сохраняется во всех интерфейсах: CLI, дашборд, диспетчер и нотификатор.

**Синтетические запуски для завершений без захвата.** Завершение или блокировка задачи, которая никогда не была захвачена (например, человек закрывает задачу `ready` из дашборда с `summary`, или пользователь CLI запускает `hermes kanban complete <ready-task> --summary X`) иначе привело бы к потере handoff. Вместо этого ядро вставляет строку запуска нулевой длительности (`started_at == ended_at`) с `summary` / `metadata` / `reason`, чтобы история попыток оставалась полной. У события `completed` / `blocked` поле `run_id` указывает на эту строку.

**Обновление выдвижного меню в реальном времени.** Когда поток событий WebSocket дашборда сообщает о новых событиях для текущей просматриваемой задачи, выдвижное меню перезагружается само (через счётчик событий задачи, включённый в список зависимостей `useEffect`). Теперь нет необходимости закрывать и открывать меню, чтобы увидеть новую строку запуска или обновлённый результат.

### Forward compatibility

Два nullable‑столбца в `tasks` зарезервированы для маршрутизации v2 workflow: `workflow_template_id` (какой шаблон относится к задаче) и `current_step_key` (какой шаг шаблона активен). Ядро v1 игнорирует их при маршрутизации, но позволяет клиентам записывать их, чтобы в выпуске v2 можно было добавить механизм маршрутизации без дополнительной миграции схемы.
## Справочник событий

Каждый переход добавляет строку в `task_events`. Каждая строка может содержать необязательный `run_id`, чтобы UI могли группировать события по попыткам. Типы группируются в три кластера, чтобы было удобно фильтровать (`hermes kanban watch --kinds completed,gave_up,timed_out`):

**Lifecycle** (что изменилось в задаче как логической единице):

| Kind | Payload | When |
|---|---|---|
| `created` | `{assignee, status, parents, tenant}` | Задача вставлена. `run_id` — `NULL`. |
| `promoted` | — | `todo → ready` потому что все родители достигли `done`. `run_id` — `NULL`. |
| `claimed` | `{lock, expires, run_id}` | Диспетчер атомарно захватил задачу `ready` для запуска. |
| `completed` | `{result_len, summary?}` | Работник записал `--result` / `--summary`, и задача перешла в `done`. `summary` — первая строка передачи (ограничение 400 симв); полная версия хранится в строке run. Если `complete_task` вызывается для задачи, которая никогда не была захвачена, но содержит поля передачи, синтезируется run нулевой длительности, чтобы `run_id` всё равно указывал на что‑то. |
| `blocked` | `{reason}` | Работник или человек перевёл задачу в `blocked`. При вызове для задачи, которая никогда не была захвачена, с `--reason` синтезируется run нулевой длительности. |
| `unblocked` | — | `blocked → ready`, вручную или через `/unblock`. `run_id` — `NULL`. |
| `archived` | — | Скрыта из доски по умолчанию. Если задача всё ещё выполнялась, сохраняет `run_id` того run, который был отозван как побочный эффект. |

**Edits** (человеческие изменения, не являющиеся переходами):

| Kind | Payload | When |
|---|---|---|
| `assigned` | `{assignee}` | Изменён исполнитель (включая снятие). |
| `edited` | `{fields}` | Обновлено название или тело. |
| `reprioritized` | `{priority}` | Изменён приоритет. |
| `status` | `{status}` | Перетаскивание в дашборде записало статус напрямую (например, `todo → ready`). Содержит `run_id` того run, который был отозван при перетаскивании из `running`; иначе `run_id` — `NULL`. |

**Worker telemetry** (о процессе выполнения, а не о логической задаче):

| Kind | Payload | When |
|---|---|---|
| `spawned` | `{pid}` | Диспетчер успешно запустил процесс работника. |
| `heartbeat` | `{note?}` | Работник вызвал `hermes kanban heartbeat $TASK`, чтобы сигнализировать о живости во время длительных операций. |
| `reclaimed` | `{stale_lock}` | TTL захвата истёк без завершения; задача возвращается в `ready`. |
| `crashed` | `{pid, claimer}` | PID работника больше не жив, но TTL ещё не истёк. |
| `timed_out` | `{pid, elapsed_seconds, limit_seconds, sigkill}` | Превышено `max_runtime_seconds`; диспетчер послал SIGTERM (затем SIGKILL через 5 с грации) и переочередил задачу. |
| `stale` | `{elapsed_seconds, last_heartbeat_at, heartbeat_age_seconds, timeout_seconds, pid, terminated}` | Задача работала дольше `kanban.dispatch_stale_timeout_seconds` (по умолчанию 4 ч) **и** за последний час не пришёл `kanban_heartbeat`. Диспетчер посылает SIGTERM локальному работнику (если есть), сбрасывает задачу в `ready` для повторного диспетчеринга. Счётчик отказов НЕ увеличивается (stale — это обнаружение отсутствия со стороны диспетчера, а не ошибка работника). Работники, выполняющие длительные операции, должны вызывать `kanban_heartbeat` минимум раз в час, чтобы этого избежать. |
| `respawn_guarded` | `{reason}` | Диспетчер отказался повторно запускать эту готовую задачу в текущем тикe. Причины: `blocker_auth` (последний отказ был ошибкой квоты/аутентификации/429 — ждать сброса окна ограничения), `recent_success` (за последний час завершился run — ждать ревью перед повторным запуском), `active_pr` (в недавнем комментарии появился URL PR — предыдущий работник уже открыл PR). Задача остаётся в `ready`; в следующем тике будет ещё одна попытка запуска. Если условие сохраняется, обычный автоматический выключатель `consecutive_failures` заблокирует задачу через `gave_up` после `failure_limit` отказов. |
| `spawn_failed` | `{error, failures}` | Одна попытка запуска не удалась (отсутствует PATH, рабочее пространство не монтируется, …). Счётчик увеличивается; задача возвращается в `ready` для повторной попытки. |
| `protocol_violation` | `{pid, claimer, exit_code}` | Работник завершился успешно, пока задача всё ещё была в `running`, обычно потому что он ответил, не вызвав `kanban_complete` или `kanban_block`. Диспетчер также генерирует `gave_up` и сразу блокирует задачу вместо повторной попытки. |
| `gave_up` | `{failures, effective_limit, limit_source, error}` | Сработал автоматический выключатель после N последовательных неуспешных попыток. Задача автоматически блокируется с последней ошибкой. Эффективный лимит определяется как `max_retries` задачи, затем `failure_limit` диспетчера / `kanban.failure_limit`, затем встроенное значение по умолчанию. |

`hermes kanban tail <id>` показывает эти события для одной задачи. `hermes kanban watch` транслирует их по всей доске.
## Не входит в область

Kanban преднамеренно одномашинный. `~/.hermes/kanban.db` — локальный файл SQLite, а диспетчер порождает воркеров на той же машине. Запуск совместной доски на двух хостах не поддерживается — нет примитива координации для «воркер X на хосте A, воркер Y на хосте B», и механизм обнаружения сбоев предполагает, что PID‑ы локальны для хоста. Если требуется мультихост, запусти независимую доску на каждом хосте и используй `delegate_task` / очередь сообщений для их соединения.
