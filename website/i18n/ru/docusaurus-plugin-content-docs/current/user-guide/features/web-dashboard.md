# Веб‑панель

Веб‑панель — это браузерный UI для управления установкой Hermes Agent. Вместо редактирования YAML‑файлов или запуска команд CLI ты можешь настраивать параметры, управлять API‑ключами и отслеживать сессии через чистый веб‑интерфейс.

:::tip
Hosted‑mode auth использует OAuth Nous Portal; если ты также хочешь, чтобы панель взаимодействовала с реальным бекендом, `hermes setup --portal` подключит модель и шлюз инструментов. Смотри [Nous Portal](/integrations/nous-portal).
:::
## Быстрый старт

```bash
hermes dashboard
```

Это запускает локальный веб‑сервер и открывает `http://127.0.0.1:9119` в твоём браузере. Панель управления работает полностью на твоём компьютере — никакие данные не покидают localhost.

### Параметры

| Флаг | По умолчанию | Описание |
|------|--------------|----------|
| `--port` | `9119` | Порт, на котором будет работать веб‑сервер |
| `--host` | `127.0.0.1` | Адрес привязки |
| `--no-open` | — | Не открывать браузер автоматически |
| `--insecure` | off | Разрешить привязку к не‑localhost хостам (**ОПАСНО** — раскрывает API‑ключи в сети; используйте вместе с файрволом и надёжной аутентификацией) |
| `--tui` | off | Отобразить вкладку чата в браузере (встроенный `hermes --tui` через PTY/WebSocket). Альтернативно задайте `HERMES_DASHBOARD_TUI=1`. |

```bash
# Custom port
hermes dashboard --port 8080

# Bind to all interfaces (use with caution on shared networks)
hermes dashboard --host 0.0.0.0

# Start without opening browser
hermes dashboard --no-open

# Enable the in-browser Chat tab
hermes dashboard --tui
```
## Предварительные требования

Стандартная установка `hermes-agent` не включает HTTP‑стек и вспомогательный PTY‑инструмент — они являются опциональными дополнениями. **Веб‑панель** требует FastAPI и Uvicorn (`web` extra). Вкладка **Chat** также нуждается в `ptyprocess` для запуска встроенного TUI через псевдотерминал (`pty` extra на POSIX). Установи оба дополнения так:

```bash
pip install 'hermes-agent[web,pty]'
```

`web` extra подтягивает FastAPI/Uvicorn; `pty` добавляет `ptyprocess` (POSIX) или `pywinpty` (нативный Windows — учти, что сам встроенный TUI всё равно требует WSL). `pip install hermes-agent[all]` включает оба дополнения и является самым простым способом, если тебе также нужны обмен сообщениями, голос и т.д.

Если запустить `hermes dashboard` без этих зависимостей, он подскажет, что установить. Если фронтенд ещё не собран и доступен `npm`, он будет автоматически построен при первом запуске.

Вкладка **Chat** намеренно отключена при обычном запуске `hermes dashboard`. Запусти панель с `hermes dashboard --tui` или установи `HERMES_DASHBOARD_TUI=1`, когда нужна встроенная панель чата в браузере.
## Страницы

### Статус

Главная страница отображает живой обзор твоей установки:

- **Agent version** → **Версия агента** и дата релиза
- **Gateway status** — запущен/остановлен, PID, подключённые платформы и их состояние
- **Active sessions** — количество активных сессий за последние 5 минут
- **Recent sessions** — список 20 самых последних сессий с моделью, количеством сообщений, использованием токенов и предварительным просмотром диалога

Страница статуса автоматически обновляется каждые 5 секунд.

### Чат

Вкладка **Chat** встраивает полный Hermes TUI (тот же интерфейс, что и при запуске `hermes --tui`) непосредственно в браузер. Всё, что можно делать в терминальном TUI — команды со слешем, выбор модели, карточки вызовов инструментов, потоковый markdown, запросы уточнения/sudo/одобрения, темизация скина — работает здесь идентично, потому что дашборд запускает реальный бинарный TUI и рендерит его ANSI‑вывод через [xterm.js](https://xtermjs.org/) с WebGL‑рендерером для пиксель‑точного расположения ячеек.

**Как это работает:**

- `/api/pty` открывает WebSocket, аутентифицированный токеном сессии дашборда
- Сервер запускает `hermes --tui` за POSIX‑псевдотерминалом
- Нажатия клавиш передаются в PTY; ANSI‑вывод возвращается в браузер
- WebGL‑рендерер xterm.js рисует каждую ячейку в целочисленной пиксельной сетке; отслеживание мыши (SGR 1006), широкие символы (Unicode 11) и символы‑рисунки отрисовываются нативно
- Изменение размера окна браузера меняет размер TUI через аддон `@xterm/addon-fit`

**Возобновить существующую сессию:** во вкладке **Sessions** нажми иконку воспроизведения (▶) рядом с любой сессией. Это перейдёт на `/chat?resume=<id>` и запустит TUI с `--resume`, загрузив полную историю.

**Предварительные требования:**

- Node.js (то же требование, что и для `hermes --tui`; пакет TUI собирается при первом запуске)
- `ptyprocess` — устанавливается через extra `pty` (`pip install 'hermes-agent[web,pty]'`, или `[all]` покрывает оба)
- POSIX‑ядро (Linux, macOS или WSL2).  Панель `/chat` конкретно нуждается в POSIX PTY — в нативном Windows‑Python эквивалента нет, поэтому на чистой Windows‑установке остальные части дашборда (сессии, задачи, метрики, редактор конфигурации) работают, но вкладка `/chat` покажет баннер с рекомендацией использовать WSL2 для этой функции.

Закрой вкладку браузера, и PTY будет корректно завершён на сервере. При повторном открытии будет создана новая сессия.

### Конфиг

Редактор на основе формы для `config.yaml`. Все более 150 полей конфигурации автоматически обнаруживаются из `DEFAULT_CONFIG` и организованы в табы‑категории:

- **model** — модель по умолчанию, провайдер, базовый URL, настройки рассуждений
- **terminal** — бекенд (local/docker/ssh/modal), таймаут, предпочтения оболочки
- **display** — скин, прогресс инструмента, отображение возобновления, настройки спиннера
- **agent** — максимальное количество итераций, таймаут шлюза, уровень сервиса
- **delegation** — ограничения подагентов, усилие рассуждений
- **memory** — выбор провайдера, настройки внедрения контекста
- **approvals** — режим одобрения опасных команд (ask/yolo/deny)
- И многое другое — у каждого раздела `config.yaml` есть соответствующие поля формы

Поля с известными допустимыми значениями (бекенд терминала, скин, режим одобрения и т.п.) отображаются выпадающими списками. Булевы значения — переключатели. Всё остальное — текстовый ввод.

**Действия:**

- **Save** — сразу записывает изменения в `config.yaml`
- **Reset to defaults** — возвращает все поля к значениям по умолчанию (не сохраняет, пока не нажмёшь **Save**)
- **Export** — скачивает текущий конфиг в виде JSON
- **Import** — загружает JSON‑файл конфигурации, заменяя текущие значения

:::tip
Изменения конфигурации вступают в силу при следующей сессии агента или перезапуске шлюза. Веб‑дашборд редактирует тот же файл `config.yaml`, который читают `hermes config set` и шлюз.
:::

### API‑ключи

Управляй файлом `.env`, где хранятся API‑ключи и учётные данные. Ключи сгруппированы по категориям:

- **LLM Providers** — OpenRouter, Anthropic, OpenAI, DeepSeek и др.
- **Tool API Keys** — Browserbase, Firecrawl, Tavily, ElevenLabs и др.
- **Messaging Platforms** — Telegram, Discord, Slack‑бот токены и др.
- **Agent Settings** — не‑секретные переменные окружения, такие как `API_SERVER_ENABLED`

Для каждого ключа отображается:
- Установлен ли он в данный момент (с редактируемым предварительным просмотром значения)
- Описание назначения
- Ссылка на страницу регистрации/получения ключа провайдера
- Поле ввода для установки или обновления значения
- Кнопка удаления

Продвинутые/реже используемые ключи скрыты по умолчанию за переключателем.

### Сессии

Просматривай и исследуй все сессии агента. Каждая строка показывает название сессии, иконку исходной платформы (CLI, Telegram, Discord, Slack, cron), имя модели, количество сообщений, количество вызовов инструментов и время последней активности. Активные сессии помечены пульсирующим бейджем.

- **Search** — полнотекстовый поиск по всему содержимому сообщений с использованием FTS5. Результаты показывают подсвеченные фрагменты и автоматически прокручивают к первому совпадению при раскрытии.
- **Expand** — кликни сессию, чтобы загрузить её полную историю сообщений. Сообщения цвет‑кодированы по роли (user, assistant, system, tool) и рендерятся как Markdown с подсветкой синтаксиса.
- **Tool calls** — сообщения‑ассистенты с вызовами инструментов отображаются в сворачиваемых блоках с именем функции и JSON‑аргументами.
- **Delete** — удаление сессии и её истории сообщений через иконку корзины.

### Логи

Просматривай файлы логов агента, шлюза и ошибок с фильтрацией и живым «tail».

- **File** — переключение между логами `agent`, `errors` и `gateway`
- **Level** — фильтр по уровню логов: ALL, DEBUG, INFO, WARNING или ERROR
- **Component** — фильтр по компоненту‑источнику: all, gateway, agent, tools, cli или cron
- **Lines** — выбор количества строк для отображения (50, 100, 200 или 500)
- **Auto-refresh** — переключатель живого «tail», который опрашивает новые строки каждые 5 секунд
- **Color-coded** — строки логов окрашены по серьёзности (красный — ошибки, жёлтый — предупреждения, тусклый — отладка)

### Аналитика

Аналитика использования и затрат, вычисляемая из истории сессий. Выбери период (7, 30 или 90 дней), чтобы увидеть:

- **Summary cards** — общее количество токенов (входящие/исходящие), процент попаданий в кэш, общая оценочная или фактическая стоимость и общее количество сессий со средним значением в день
- **Daily token chart** — stacked‑bar‑диаграмма, показывающая использование входных и выходных токенов по дням, с подсказками при наведении, раскрывающими детали и стоимость
- **Daily breakdown table** — дата, количество сессий, входящие токены, выходящие токены, процент кэш‑хитов и стоимость за каждый день
- **Per-model breakdown** — таблица, показывающая каждую использованную модель, количество её сессий, использование токенов и оценочную стоимость

### Cron

Создавай и управляй запланированными cron‑задачами, которые запускают запросы агента по расписанию.

- **Create** — укажи название (опционально), запрос, cron‑выражение (например `0 9 * * *`) и цель доставки (local, Telegram, Discord, Slack или email)
- **Job list** — каждая задача отображает название, предварительный просмотр запроса, выражение расписания, бейдж состояния (enabled/paused/error), цель доставки, время последнего запуска и время следующего запуска
- **Pause / Resume** — переключение задачи между активным и приостановленным состоянием
- **Trigger now** — немедленное выполнение задачи вне её обычного расписания
- **Delete** — окончательное удаление cron‑задачи

### Навыки

Просматривай, ищи и включай/выключай навыки и наборы инструментов. Навыки загружаются из `~/.hermes/skills/` и группируются по категориям.

- **Search** — фильтрация навыков и наборов инструментов по названию, описанию или категории
- **Category filter** — клик по «пилюлям» категории для сужения списка (например MLOps, MCP, Red Teaming, AI)
- **Toggle** — включить или отключить отдельный навык переключателем. Изменения вступают в силу в следующей сессии.
- **Toolsets** — отдельный раздел показывает встроенные наборы инструментов (операции с файлами, веб‑браузинг и др.) с их статусом (active/inactive), требованиями к настройке и списком включённых инструментов

:::warning Security
Веб‑дашборд читает и пишет твой файл `.env`, который содержит API‑ключи и секреты. По умолчанию он привязывается к `127.0.0.1` — доступен только с твоего локального компьютера. Если привязать его к `0.0.0.0`, любой в сети сможет просматривать и изменять твои учётные данные. У дашборда нет собственного механизма аутентификации.
:::
## Slash‑команда `/reload`

Dashboard PR также добавляет slash‑команду `/reload` в интерактивный CLI. После изменения API‑ключей через веб‑дашборд (или путём прямого редактирования файла `.env`), используй `/reload` в активной сессии CLI, чтобы применить изменения без перезапуска:

```
You → /reload
  Reloaded .env (3 var(s) updated)
```

Это перечитывает `~/.hermes/.env` в окружение запущенного процесса. Полезно, когда ты добавил новый ключ провайдера через дашборд и хочешь сразу им воспользоваться.
## REST API

Веб‑панель предоставляет REST API, которое использует фронтенд. Ты также можешь вызывать эти эндпоинты напрямую для автоматизации:

### GET /api/status

Возвращает версию агента, статус шлюза, состояния платформ и количество активных сессий.

### GET /api/sessions

Возвращает 20 самых последних сессий с метаданными (модель, количество токенов, метки времени, превью).

### GET /api/config

Возвращает текущее содержимое `config.yaml` в виде JSON.

### GET /api/config/defaults

Возвращает значения конфигурации по умолчанию.

### GET /api/config/schema

Возвращает схему, описывающую каждое поле конфигурации — тип, описание, категорию и варианты выбора, где применимо. Фронтенд использует её для отображения соответствующего виджета ввода для каждого поля.

### PUT /api/config

Сохраняет новую конфигурацию. Тело: `{"config": {...}}`.

### GET /api/env

Возвращает все известные переменные окружения с их статусом (установлена/не установлена), скрытыми значениями, описаниями и категориями.

### PUT /api/env

Устанавливает переменную окружения. Тело: `{"key": "VAR_NAME", "value": "secret"}`.

### DELETE /api/env

Удаляет переменную окружения. Тело: `{"key": "VAR_NAME"}`.

### GET /api/sessions/\{session_id\}

Возвращает метаданные одной сессии.

### GET /api/sessions/\{session_id\}/messages

Возвращает полную историю сообщений для сессии, включая вызовы инструментов и метки времени.

### GET /api/sessions/search

Полнотекстовый поиск по содержимому сообщений. Параметр запроса: `q`. Возвращает совпадающие идентификаторы сессий с выделенными фрагментами.

### DELETE /api/sessions/\{session_id\}

Удаляет сессию и её историю сообщений.

### GET /api/logs

Возвращает строки журналов. Параметры запроса: `file` (agent/errors/gateway), `lines` (количество), `level`, `component`.

### GET /api/analytics/usage

Возвращает использование токенов, стоимость и аналитику сессий. Параметр запроса: `days` (по умолчанию 30). Ответ включает ежедневные разбивки и агрегаты по моделям.

### GET /api/cron/jobs

Возвращает все настроенные cron‑задачи с их состоянием, расписанием и историей запусков.

### POST /api/cron/jobs

Создаёт новую cron‑задачу. Тело: `{"prompt": "...", "schedule": "0 9 * * *", "name": "...", "deliver": "local"}`.

### POST /api/cron/jobs/\{job_id\}/pause

Приостанавливает cron‑задачу.

### POST /api/cron/jobs/\{job_id\}/resume

Возобновляет приостановленную cron‑задачу.

### POST /api/cron/jobs/\{job_id\}/trigger

Немедленно запускает cron‑задачу вне её расписания.

### DELETE /api/cron/jobs/\{job_id\}

Удаляет cron‑задачу.

### GET /api/skills

Возвращает все навыки с их названием, описанием, категорией и статусом включения.

### PUT /api/skills/toggle

Включает или отключает навык. Тело: `{"name": "skill-name", "enabled": true}`.

### GET /api/tools/toolsets

Возвращает все наборы инструментов с их меткой, описанием, списком инструментов и статусом активности/настройки.
## OAuth Authentication (gated mode)

Когда панель привязана к публичному адресу — любому, кроме `127.0.0.1` / `localhost` — Hermes Agent включает OAuth‑основанный шлюз аутентификации. Каждый запрос должен содержать проверенный cookie‑сеанса, иначе он будет перенаправлен через полный OAuth‑круговорот через Nous Portal.

Это предназначено для размещённых развертываний (обычно Fly.io), где панель доступна из публичного интернета. Панели, управляемые оператором и привязанные к loopback, не затронуты.

### Когда шлюз включается

| Flags | Auth gate | Use case |
|-------|-----------|----------|
| `hermes dashboard` (default — binds to `127.0.0.1`) | OFF | Локальная разработка |
| `hermes dashboard --host 0.0.0.0` | **ON** | Продакшн / развертывание на Fly.io |
| `hermes dashboard --host 192.168.1.10 --insecure` | OFF | Доверенная LAN; пользователь выбирает устаревшую аутентификацию по токену сеанса |

Шлюз включён тогда и только тогда, когда:

1. Хост привязки не `127.0.0.1`, `::1`, `localhost` и не `0.0.0.0`, И
2. Флаг `--insecure` **не** установлен.

Установка `--insecure` сохраняет прежнее поведение с единственным процессом и токеном сеанса — без OAuth‑танца, без необходимости в плагинах провайдеров. Используй только в сетях, где доверяешь каждому клиенту.

### Семантика «fail‑closed»

Если шлюз должен включиться, но **нет** зарегистрированного `DashboardAuthProvider` (нет плагина Nous, нет пользовательского плагина), `hermes dashboard` отказывается привязываться с явным сообщением об ошибке. Нет «default‑deny, но принимать всё» запасного варианта — неправильно сконфигурированная панель с шлюзом никогда не запустится.

### Поставщик по умолчанию: Nous Research

Встроенный плагин `plugins/dashboard_auth/nous` **всегда установлен** и автоматически загружается. Он автоматически регистрирует `DashboardAuthProvider` с именем `nous`, когда сконфигурирован client ID.

#### Конфигурация

Плагин читает два источника, при наличии переменной окружения она имеет приоритет:

**`config.yaml`** — канонический источник:

```yaml
dashboard:
  oauth:
    client_id: agent:01HXYZ…             # required to engage the gate
    portal_url: https://portal.nousresearch.com  # optional; defaults to production
```

**Переменные окружения** — переопределения оператором:

| Env var | Overrides | Format | Provisioned by |
|---------|-----------|--------|----------------|
| `HERMES_DASHBOARD_OAUTH_CLIENT_ID` | `dashboard.oauth.client_id` | `agent:{instance_id}` | Nous Portal во время provisioning на Fly.io |
| `HERMES_DASHBOARD_PORTAL_URL` | `dashboard.oauth.portal_url` | URL (по умолчанию: `https://portal.nousresearch.com`) | Portal — переопределяется только для staging или кастомного развертывания |

Согласно конвенции Hermes Agent (`~/.hermes/.env` предназначен только для API‑ключей / секретов), **`config.yaml` — рекомендованное место для установки этих значений** в локальной разработке, on‑prem и любых развертываниях, которыми ты управляешь напрямую. Путь через переменные окружения существует, чтобы платформа Fly.io могла внедрять секреты `client_id` при каждом деплое без необходимости править `config.yaml` внутри образа — это его основная цель.

Пустые значения переменных окружения считаются неустановленными, поэтому provision‑секрет Fly, который не заполнен, не может случайно затмить корректную запись в `config.yaml`.

Если ни один источник не предоставляет `client_id`, плагин сообщает конкретную причину, а ошибка привязки панели в режиме «fail‑closed» точно указывает, что нужно исправить:

```
Refusing to bind dashboard to 0.0.0.0 — the OAuth auth gate engages on
non-loopback binds, but no auth providers are registered.

Bundled providers reported these issues:
  • nous: HERMES_DASHBOARD_OAUTH_CLIENT_ID is not set (and
    dashboard.oauth.client_id in config.yaml is empty). The Nous Portal
    provisions this env var (shape 'agent:{instance_id}') when it
    deploys a Hermes Agent instance — set it to your provisioned
    client id (either as an env var or under dashboard.oauth.client_id
    in config.yaml), or pass --insecure to skip the OAuth gate entirely.

Or pass --insecure to skip the auth gate (NOT recommended on untrusted
networks).
```

### Переопределение публичного URL

По умолчанию панель восстанавливает OAuth‑callback URL из запроса — `X-Forwarded-Host` + `X-Forwarded-Proto` + `X-Forwarded-Prefix` (когда uvicorn сконфигурирован с `proxy_headers=True`, что `start_server` включает под шлюзом). Это работает «из коробки» на Fly.io, где все три заголовка задаются корректно.

Для развертываний за обратными прокси, которые надёжно не передают эти заголовки (ручные nginx‑настройки, on‑prem ingress, кастомные домены Fly с частичными цепочками прокси), задай `dashboard.public_url` (или `HERMES_DASHBOARD_PUBLIC_URL`) как **полный публичный URL**, по которому достигается панель:

```yaml
dashboard:
  public_url: "https://dashboard.example.com/hermes"
```

Когда установлен, OAuth‑callback URL становится `<public_url>/auth/callback` дословно — `X-Forwarded-Prefix` игнорируется на этом пути, потому что оператор явно объявил публичный URL. Это намеренно: добавление префикса сверху привело бы к двойному префиксу в типичном случае, когда префикс уже включён в `public_url`.

Тот же порядок приоритета, что и у остальных настроек панели — переменные окружения выигрывают над `config.yaml`:

| Surface | Override path | When to use |
|---------|---------------|-------------|
| `dashboard.public_url` в `config.yaml` | `HERMES_DASHBOARD_PUBLIC_URL` | Локальная разработка / on‑prem (канонический) |
| `HERMES_DASHBOARD_PUBLIC_URL` env var | — | Секреты платформы Fly.io / CI |
| (unset) | — | По умолчанию — восстановление из заголовков `X-Forwarded-*` |

Валидация отклоняет значения без схемы `http://` / `https://`, без хоста, или содержащие кавычки, угловые скобки, пробелы или управляющие символы. Некорректное значение тихо падает к восстановлению из заголовков, чтобы поток входа продолжал работать, а не перенаправлял пользователя на вредоносный URL.

> **Note:** `public_url` переопределяет только OAuth‑callback URL. Флаг cookie `Secure` всё равно управляется `request.url.scheme` (X-Forwarded-Proto при `proxy_headers`), поэтому `http://` `public_url` в публичном деплое с TLS‑терминацией приведёт к не‑Secure cookie. Это подводный камень оператора — сочетай `public_url` с корректной TLS‑терминацией upstream.

### OAuth flow

Провайдер реализует [контракт OAuth Nous Portal v1](https://github.com/NousResearch/nous-account-service/blob/main/docs/agent-dashboard-oauth-contract.md) — grant типа authorization‑code с PKCE (S256):

1. Пользователь открывает `/` без cookie‑сеанса → шлюз перенаправляет на `/login`.
2. На странице входа появляется кнопка «Continue with Nous Research» → `/auth/login?provider=nous`.
3. Сервер сохраняет состояние PKCE в короткоживущий cookie, перенаправляет пользователя на `https://portal.nousresearch.com/oauth/authorize?…`.
4. Пользователь аутентифицируется в Portal, попадает на `/auth/callback?code=…&state=…`.
5. Сервер обменивает код на access‑token через `POST /api/oauth/token`, проверяет подпись JWT по JWKS Portal (`/.well-known/jwks.json`) и устанавливает cookie `hermes_session_at`.
6. Пользователь перенаправляется на `/` (или на исходный deep‑link путь через параметр `next=`).

Access‑token живёт 15 минут. **В контракте v1 нет refresh‑token** — по истечении срока SPA‑обёртка fetch обнаруживает 401 и полностью переходит на `/login` для повторного прохождения потока.

### Cookies set

| Name | Lifetime | Notes |
|------|----------|-------|
| `hermes_session_at` | Token TTL (15 min) | HttpOnly, SameSite=Lax, Secure‑when‑HTTPS |
| `hermes_session_pkce` | 10 min | HttpOnly; хранит PKCE‑verifier + подсказку провайдера во время круговорота |
| `hermes_session_rt` | unused in v1 | Зарезервировано для будущего; не записывается, когда `refresh_token` пуст |

Все три имеют `Path=/` и `SameSite=Lax`. Флаг `Secure` ставится, когда панель доступна по HTTPS (определяется схемой URL запроса — учитывает `X-Forwarded-Proto` от TLS‑терминатора Fly при `proxy_headers=True`).

### Logout

В боковой панели отображается `Logged in as <user_id…> via nous` с иконкой выхода. При клике отправляется POST на `/auth/logout`, который очищает все cookies аутентификации панели и перенаправляет обратно на `/login`.

### Audit log

Каждый старт входа, успех, неудача и ошибка проверки сеанса записываются как JSON‑строка в `$HERMES_HOME/logs/dashboard-auth.log`. Чувствительные поля (`access_token`, `refresh_token`, `code`, `code_verifier`, `state`, заголовок `Authorization`) перед записью редактируются.

### Custom providers

Чтобы подключить не‑Nous OAuth‑провайдера (например Google, GitHub, кастомный OIDC), создай плагин, регистрирующий `DashboardAuthProvider`:

```python
# ~/.hermes/plugins/dashboard-auth-myidp/__init__.py
from hermes_cli.dashboard_auth import DashboardAuthProvider, Session, LoginStart

class MyIdPProvider(DashboardAuthProvider):
    name = "myidp"
    display_name = "My Identity Provider"

    def start_login(self, *, redirect_uri): ...
    def complete_login(self, *, code, state, code_verifier, redirect_uri): ...
    def verify_session(self, *, access_token): ...
    def refresh_session(self, *, refresh_token): ...
    def revoke_session(self, *, refresh_token): ...

def register(ctx):
    ctx.register_dashboard_auth_provider(MyIdPProvider())
```

Страница входа перечисляет все зарегистрированные провайдеры; несколько провайдеров могут быть стэкированы, пользователь выбирает один на `/login`.

```bash
# Quick env-var path (Fly.io shape). HERMES_DASHBOARD_PORTAL_URL is
# optional — defaults to production.
HERMES_DASHBOARD_OAUTH_CLIENT_ID=agent:test \
  hermes dashboard --host 0.0.0.0

# Or the equivalent via config.yaml (recommended for local dev / on-prem):
#
#   dashboard:
#     oauth:
#       client_id: agent:test

# then just:
hermes dashboard --host 0.0.0.0

# Hit /api/status to see the gate state:
curl -s http://127.0.0.1:9119/api/status | jq '.auth_required, .auth_providers'
```

React‑страница статуса панели показывает те же поля под «Web server». Виджет AuthWidget в боковой панели отображает текущую личность после входа.
## CORS

Веб‑сервер ограничивает CORS только origin‑ами localhost:

- `http://localhost:9119` / `http://127.0.0.1:9119` (production)
- `http://localhost:3000` / `http://127.0.0.1:3000`
- `http://localhost:5173` / `http://127.0.0.1:5173` (Vite dev server)

Если запускать сервер на произвольном порту, соответствующий origin добавляется автоматически.
## Разработка

Если ты вносишь вклад в фронтенд веб‑дашборда:

```bash
# Terminal 1: start the backend API
hermes dashboard --no-open

# Terminal 2: start the Vite dev server with HMR
cd web/
npm install
npm run dev
```

Сервер разработки Vite по адресу `http://localhost:5173` проксирует запросы к `/api` в FastAPI‑бэкенд по адресу `http://127.0.0.1:9119`.

Фронтенд построен на React 19, TypeScript, Tailwind CSS v4 и компонентах shadcn/ui‑style. При продакшн‑сборке артефакты помещаются в `hermes_cli/web_dist/`, который FastAPI‑сервер обслуживает как статическое SPA.
## Автоматическая сборка при обновлении

Когда ты запускаешь `hermes update`, веб‑фронтенд автоматически пересобирается, если `npm` доступен. Это держит `dashboard` в синхронизации с обновлениями кода. Если `npm` не установлен, обновление пропускает сборку фронтенда, и `hermes dashboard` соберёт её при первом запуске.
