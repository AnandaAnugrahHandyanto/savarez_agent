# Прокси подписки

Прокси подписки — это локальный HTTP‑сервер, который позволяет внешним приложениям — OpenViking, Karakeep, Open WebUI и любому, кто использует совместимый с OpenAI API завершения чата — использовать твою подписку провайдера, управляемую Hermes, в качестве LLM‑конечного пункта. Прокси добавляет нужные учётные данные (обновляя их автоматически), так что приложению никогда не нужен статический API‑ключ.

Это отличается от [API‑сервера](./api-server.md):

| | API‑сервер | Прокси подписки |
|---|---|---|
| Что обслуживает | Твой агент (полный набор инструментов, память, навыки) | Прямой вывод модели |
| Сценарий использования | «Использовать Hermes как бекенд чата» | «Использовать мою подписку Portal из другого приложения» |
| Авторизация | Твой `API_SERVER_KEY` | Любой bearer‑токен (прокси добавляет реальный) |
| Вызовы инструментов | Да — агент запускает инструменты | Нет — только проброс |

Используй API‑сервер, когда нужен **агент** как бекенд. Используй прокси, когда нужна **модель** через твою подписку.

## Быстрый старт

### 1. Войди в провайдера (один раз)

```bash
hermes auth add nous
```

Это откроет браузер для OAuth‑потока Nous Portal. Hermes сохраняет токен обновления в `~/.hermes/auth.json` — том же месте, где хранятся все входы провайдеров Hermes.

### 2. Запусти прокси

```bash
hermes proxy start
```

```
Starting Hermes proxy for Nous Portal
  Listening on:  http://127.0.0.1:8645/v1
  Forwarding to: (resolved per-request from your subscription)
  Use any bearer token in the client — the proxy attaches your real credential.
```

Оставь его работать в форграунде. Используй `tmux`, `nohup` или unit `systemd`, если хочешь, чтобы он продолжал работать после выхода из системы.

### 3. Укажи приложению адрес прокси

Любая конфигурация совместимого с OpenAI приложения принимает один и тот же набор параметров:

```
Base URL:   http://127.0.0.1:8645/v1
API key:    anything (e.g. "sk-unused")
Model:      Hermes-4-70B    # or Hermes-4.3-36B, Hermes-4-405B
```

Прокси игнорирует заголовок `Authorization`, полученный от твоего приложения, и добавляет реальную учётную запись Portal к запросу upstream. Обновление токена происходит автоматически, когда срок действия bearer‑токена подходит к концу.

## Доступные провайдеры

```bash
hermes proxy providers
```

В текущей поставке: `nous` (Nous Portal) и `xai` (xAI / Grok). Можно добавить другие OAuth‑провайдеры, реализовав интерфейс `UpstreamAdapter` в `hermes_cli/proxy/adapters/`.

## Проверка статуса

```bash
hermes proxy status
```

```
Hermes proxy upstream adapters

  [nous    ] Nous Portal — ready (bearer expires 2026-05-15T06:43:21Z)
```

Если видишь `not logged in`, выполни `hermes auth add nous`. Если появляется `credentials need attention`, твой токен обновления был отозван (редко — происходит, если ты вышел из веб‑интерфейса Portal) — просто снова запусти `hermes auth add nous`.

## Разрешённые пути

Прокси пробрасывает только те пути, которые действительно обслуживает upstream. Для Nous Portal:

| Путь | Назначение |
|------|------------|
| `/v1/chat/completions` | Завершения чата (стриминг + без стриминга) |
| `/v1/completions` | Устаревшие текстовые завершения |
| `/v1/embeddings` | Эмбеддинги |
| `/v1/models` | Список моделей |

Другие пути (`/v1/images/generations`, `/v1/audio/speech` и т.д.) возвращают 404 с чёткой ошибкой, указывающей допустимые пути. Это препятствует случайным запросам к upstream.

## Настройка OpenViking для использования Portal

[OpenViking](https://github.com/volcengine/OpenViking) — база контекста, которой нужен LLM‑провайдер для её VLM (vision/language model, используемой для извлечения воспоминаний) и модель эмбеддингов. С прокси ты можешь указать её `vlm.api_base` на свой локальный прокси:

Отредактируй `~/.openviking/ov.conf`:

```json
{
  "vlm": {
    "provider": "openai",
    "model": "Hermes-4-70B",
    "api_base": "http://127.0.0.1:8645/v1",
    "api_key": "unused-proxy-attaches-real-creds"
  }
}
```

Затем запусти прокси в терминале рядом с `openviking-server`:

```bash
# Terminal 1
hermes proxy start

# Terminal 2
openviking-server
```

VLM‑вызовы OpenViking теперь проходят через твою подписку Portal. Модель эмбеддингов всё равно требует собственного провайдера — Portal обслуживает `/v1/embeddings`, но выбор модели зависит от твоего тарифного плана; смотри `portal.nousresearch.com/models`.

## Настройка Karakeep (или любого приложения‑закладки/резюмера)

[Karakeep](https://karakeep.app/) использует совместимый с OpenAI API для резюмирования закладок. В её конфигурации:

```bash
# Karakeep .env
OPENAI_API_BASE_URL=http://127.0.0.1:8645/v1
OPENAI_API_KEY=any-non-empty-string
INFERENCE_TEXT_MODEL=Hermes-4-70B
```

Тот же шаблон работает для Open WebUI, LobeChat, NextChat и любого другого клиента, совместимого с OpenAI.

## Экспонирование в LAN

По умолчанию прокси привязывается к `127.0.0.1` (только localhost). Чтобы другие машины в сети могли его использовать:

```bash
hermes proxy start --host 0.0.0.0 --port 8645
```

⚠ **Важно:** любой в твоей сети теперь может использовать твою подписку Portal. У прокси нет собственной аутентификации — он принимает любой bearer‑токен. Используй брандмауэр, VPN или обратный прокси с надёжной аутентификацией, если открываешь доступ за пределы доверённой сети.

## Ограничения по скорости

Ограничения RPM/TPM твоего тарифного плана Portal применяются ко всему прокси. Прокси не распределяет запросы и не использует пул — это один bearer‑токен с полной квотой твоей подписки. Мониторинг использования доступен на [portal.nousresearch.com](https://portal.nousresearch.com).

## Архитектура

Прокси преднамеренно прост. Для каждого запроса:

1. Приём `POST /v1/chat/completions` от твоего приложения
2. Поиск текущих учётных данных адаптера (обновление при необходимости)
3. Проброс тела запроса без изменений, с заголовком `Authorization: Bearer <minted-key>`
4. Потоковая передача ответа без изменений (SSE сохраняется)

Никаких трансформаций. Нет логирования тел запросов. Нет цикла агента. Прокси — это просто проходящий запрос, к которому добавляются учётные данные.

## Будущее: больше OAuth‑провайдеров

Система адаптеров расширяема. Добавление нового провайдера (например, HuggingFace, чат‑конечная точка GitHub Copilot, Anthropic через OAuth) требует реализации `UpstreamAdapter` в `hermes_cli/proxy/adapters/<provider>.py` и регистрации его в `adapters/__init__.py`. Провайдеры, которые не совместимы с OpenAI на уровне протокола (например, Anthropic Messages API), потребуют слой трансформации, что выходит за рамки текущей реализации.
