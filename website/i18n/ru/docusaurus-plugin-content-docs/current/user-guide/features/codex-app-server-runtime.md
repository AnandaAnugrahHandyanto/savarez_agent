# Среда выполнения Codex App-Server

Hermes может опционально передавать `openai/*` и `openai-codex/*` в [Codex CLI app-server](https://github.com/openai/codex) вместо собственного цикла инструментов. При включении команды терминала, правки файлов, изоляция и вызовы инструмента MCP выполняются внутри среды выполнения Codex — Hermes становится оболочкой вокруг неё (БД сессий, слеш‑команды, шлюз, память и обзор навыков).

Это **только при явном включении**. Поведение Hermes по умолчанию остаётся неизменным, пока ты не включишь флаг. Hermes никогда не перенаправляет тебя автоматически на эту среду выполнения.

:::tip
Не используешь OpenAI Codex? `hermes setup --portal` настраивает бэкенд без Codex с Claude/Gemini/и т.д. за один шаг. Смотри [Nous Portal](/integrations/nous-portal).
:::
## Почему

- Запуск агента OpenAI использует твою **подписку ChatGPT** (API‑ключ не нужен), применяя тот же поток аутентификации, что и у `Codex CLI`.
- Использование **наборов инструментов и песочницы Codex** — `shell` для терминала/чтения/записи/поиска, `apply_patch` для структурированных правок, `update_plan` для планирования, всё работает внутри песочницы `seatbelt/landlock`.
- **Нативные плагины Codex** — Linear, GitHub, Gmail, Calendar, Canva и др. — устанавливаются через `codex plugin`, автоматически мигрируют и активны в твоей сессии Hermes.
- **Более мощные инструменты Hermes** — `web_search`, `web_extract`, автоматизация браузера, зрительное восприятие, генерация изображений, навыки и TTS работают через обратный вызов MCP. Codex вызывает Hermes для инструментов, которых у него нет встроенно.
- **Память и подсказки навыков продолжают работать** — события Codex проецируются в форму сообщений Hermes, так что цикл самообучения видит обычный транскрипт.
## Какие инструменты действительно доступны модели

Это та часть, о которой большинство пользователей хотят знать сразу. Когда включён этот рантайм, модель, исполняющая твой ход, имеет три независимых источника инструментов:

### 1. Встроенный набор инструментов Codex (всегда включён)

Они поставляются вместе с `codex app-server` — без участия Hermes, без MCP, без плагинов. Все пять доступны с момента старта рантайма:

- **`shell`** — выполняет произвольные команды оболочки внутри песочницы. Так модель читает файлы (`cat`, `head`, `tail`), записывает их (`echo > foo`, heredocs), ищет их (`find`, `rg`, `grep`), навигирует по каталогам (`ls`, `cd`), запускает сборки, управляет процессами и делает всё, что ты обычно делаешь в bash.
- **`apply_patch`** — применяет структурированный мульти‑файловый дифф в формате патча Codex. Модель использует его для нетривиальных правок кода (добавление функции, рефакторинг в нескольких файлах); heredocs в `shell` всё ещё доступны для одноразовых записей.
- **`update_plan`** — внутренний трекер todo / плана Codex. Эквивалент инструмента Hermes `todo`, но полностью управляется внутри рантайма Codex.
- **`view_image`** — загружает локальный файл изображения в разговор, чтобы модель могла его увидеть.
- **`web_search`** — Codex имеет собственный встроенный веб‑поиск при соответствующей конфигурации. Hermes также раскрывает `web_search` (на базе Firecrawl) через обратный вызов ниже; модель выбирает тот, который ей удобнее.

То есть **всё, что ты делаешь через терминал — чтение/запись/поиск/нахождение/запуск — Codex делает нативно**. Профиль песочницы (`:workspace` по умолчанию, когда ты включаешь рантайм) контролирует, что доступно для записи.

### 2. Нативные плагины Codex (автоматически мигрированы из твоей установки `codex plugin`)

Когда ты включаешь рантайм, Hermes запрашивает у Codex RPC `plugin/list` и записывает запись `[plugins."<name>@openai-curated"]` для каждого установленного плагина. Сами плагины управляются Codex и авторизуются один раз через UI Codex.

Примеры (те, что в ветке OpenClaw выделены как «достойные YouTube‑видео»):

- **Linear** — поиск/обновление задач
- **GitHub** — поиск кода, просмотр PR, комментарии
- **Gmail** — чтение/отправка писем
- **Google Calendar** — создание/поиск событий
- **Outlook calendar/email** — аналогичная функциональность через коннектор Microsoft
- **Canva** — генерация дизайна
- …и любые другие, которые ты установил через `codex plugin marketplace add openai-curated` + `codex plugin install ...`

Что НЕ мигрировано:
- Плагины, которые ты ещё не установил — сначала установи их в Codex.
- Записи из маркетплейса приложений ChatGPT (`app/list`) — они уже включены в Codex благодаря твоей авторизации аккаунта.

### 3. Обратный вызов инструмента Hermes (сервер MCP, зарегистрированный в `~/.codex/config.toml`)

Hermes регистрирует себя как сервер MCP, чтобы Codex мог вызывать инструменты, которых нет в базовом наборе. Доступные через обратный вызов:

- **`web_search`** / **`web_extract`** — на базе Firecrawl; обычно чище, чем скрапинг для структурированного контента.
- **`browser_navigate` / `browser_click` / `browser_type` / `browser_press` / `browser_snapshot` / `browser_scroll` / `browser_back` / `browser_get_images` / `browser_console` / `browser_vision`** — полная автоматизация браузера через Camofox или Browserbase.
- **`vision_analyze`** — вызывает отдельную модель зрения для анализа изображения (отличается от `view_image` Codex, который лишь загружает изображение в разговор).
- **`image_generate`** — генерация изображений через цепочку плагинов Hermes `image_gen`.
- **`skill_view` / `skills_list`** — чтение из библиотеки навыков Hermes.
- **`text_to_speech`** — TTS через настроенного провайдера Hermes.

Когда модель запрашивает один из этих инструментов, Codex запускает подпроцесс `hermes_tools_mcp_server` через stdio MCP, вызов обрабатывается функцией `model_tools.handle_function_call()` (тот же путь, что и в стандартном рантайме Hermes), а результат возвращается Codex как любой другой ответ MCP.

### Что НЕ доступно в этом рантайме

Эти четыре инструмента Hermes требуют контекста работающего AIAgent (состояния mid‑loop) для диспатча, а без состояния MCP‑обратный вызов их не управляет. Переключайся обратно на стандартный рантайм (`/codex-runtime auto`), когда нужны любые из них:

- **`delegate_task`** — порождать суб‑агентов
- **`memory`** — постоянное хранилище памяти Hermes
- **`session_search`** — поиск по всем сессиям
- **`todo`** — хранилище задач Hermes (`update_plan` в Codex — эквивалент внутри рантайма)
## Функции рабочего процесса (`/goal`, kanban, cron)

### `/goal` (цикл Ralph)

**Работает в этом runtime.** Цели сохраняются в `state_meta` по ключу — идентификатору сессии, запрос продолжения передаётся обратно как обычное пользовательское сообщение через `run_conversation()`, а codex исполняет следующий ход нативно. Судья цели запускается через вспомогательный клиент (настраивается через `auxiliary.goal_judge` в `config.yaml`), независимо от того, какой runtime активен. Вердикт судьи «заблокировано, требуется ввод пользователя» — чистый способ выхода, если codex зависает на одобрениях.

**Важно знать:** каждый запрос продолжения — это новый ход codex, поэтому codex переоценивает политику одобрения команд с нуля. Если ты запускаешь длительную цель с множеством записей, ожидай больше запросов одобрения, чем при выполнении единственной задачи в сессии. Установи `default_permissions = ":workspace"` (Hermes делает это автоматически, когда ты включаешь runtime), чтобы простые записи в рабочее пространство не требовали подтверждения.

### Kanban (распределение рабочего дерева между несколькими агентами)

**Работает в этом runtime, с одной тонкой зависимостью.** Диспетчер kanban запускает каждого рабочего как отдельный подпроцесс `hermes chat -q`, который читает конфигурацию пользователя — поэтому, если в глобальном конфиге указано `model.openai_runtime: codex_app_server`, рабочие также будут запущены в codex‑runtime.

Что работает внутри рабочего codex‑runtime:
- Полный набор инструментов Codex (shell, apply_patch, update_plan, view_image, web_search) — рабочий действительно выполняет свою задачу нативно;
- Перенесённые плагины Codex — Linear, GitHub и др.;
- Обратный вызов Hermes‑tool для `browser_*`, vision, image_gen, skills, TTS.

Что также работает, потому что обратный вызов MCP их раскрывает:
- **`kanban_complete` / `kanban_block` / `kanban_comment` / `kanban_heartbeat`** — инструменты передачи управления рабочему. Они читают `HERMES_KANBAN_TASK` из переменной окружения (устанавливается диспетчером), правильно ограничивают доступ и записывают в SQLite‑БД доски, привязанную к `HERMES_KANBAN_DB`. Без этих функций в обратном вызове рабочий в этом runtime мог бы выполнять задачу, но не смог бы отчитаться, зависая до истечения таймаута диспетчера.
- **`kanban_show` / `kanban_list`** — запросы только для чтения доски, позволяющие рабочему проверять свой контекст.
- **`kanban_create` / `kanban_unblock` / `kanban_link`** — операции только для оркестратора. Доступны агентам‑оркестраторам, работающим в codex‑runtime и нуждающимся в распределении новых задач.

Инструменты kanban ограничены переменной окружения `HERMES_KANBAN_TASK`, которую задаёт диспетчер. Эта переменная передаётся в подпроцесс codex (codex наследует окружение) и оттуда в запущенный подпроцесс сервера MCP `hermes-tools`. Таким образом инструменты видят правильный идентификатор задачи и корректно ограничивают доступ. Для рабочих Codex app‑server Hermes также передаёт узкие переопределения песочницы, когда присутствует `HERMES_KANBAN_TASK`: сохраняет песочницу `workspace-write`, добавляет **каталог БД доски и каждый путь Kanban, закреплённый диспетчером**, как дополнительные записываемые корни (`HERMES_KANBAN_WORKSPACES_ROOT`, `HERMES_KANBAN_WORKSPACE`, наследованный `HERMES_KANBAN_ROOT` — дедуплицированы, сначала каталог БД), и по умолчанию отключает сеть. Это избавляет от хрупкого обходного пути `:danger-no-sandbox`, позволяя `kanban_complete` / `kanban_block` обновлять БД доски **и** рабочим записывать отчёты/артефакты в монтированные рабочие пространства, находящиеся вне каталога БД (например, `/media/.../kanban-workspaces/...` на отдельном диске — [issue #27941](https://github.com/NousResearch/hermes-agent/issues/27941)).

### Cron‑задачи

**Не тестировались специально.** Cron‑задачи запускаются через `cronjob` → `AIAgent.run_conversation`, тот же путь кода, что и в CLI. Если в конфигурации cron‑задачи указано `openai_runtime: codex_app_server`, она будет выполнена в codex. Применяются те же правила доступности инструментов — встроенные возможности codex + плагины + обратный вызов MCP работают, инструменты цикла агента (delegate_task, memory, session_search, todo) — нет. Если твоя cron‑задача зависит от этих инструментов, ограничь её профилем, использующим runtime по умолчанию.

## Требования

1. **Установлен Codex CLI:**
   ```bash
   npm i -g @openai/codex
   codex --version   # 0.130.0 or newer
   ```

2. **Вход в Codex OAuth.** Подпроцесс `codex` читает `~/.codex/auth.json`. Есть два способа его заполнить:
   ```bash
   codex login                  # writes tokens to ~/.codex/auth.json
   ```
   `hermes auth login codex` от Hermes записывает в `~/.hermes/auth.json` — это отдельная сессия. **Запусти `codex login` отдельно**, если ты ещё этого не сделал.

3. **(Опционально) Установи нужные плагины Codex.** Когда ты включаешь runtime, Hermes автоматически мигрирует любые уже установленные через Codex CLI курируемые плагины:
   ```bash
   codex plugin marketplace add openai-curated
   # then via codex's TUI, install Linear / GitHub / Gmail / etc.
   ```
   Hermes обнаружит их и запишет записи `[plugins."<name>@openai-curated"]` в `~/.codex/config.toml` автоматически.
## Включение

В сессии Hermes:

```
/codex-runtime codex_app_server
```

Эта команда:
- Проверяет, что установлен CLI `codex` (блокирует выполнение с подсказкой об установке, если его нет).
- Сохраняет `model.openai_runtime: codex_app_server` в ваш `config.yaml`.
- Переносит пользовательские MCP‑серверы из `~/.hermes/config.yaml` в `~/.codex/config.toml`.
- **Обнаруживает и переносит установленные нативные плагины Codex** (Linear, GitHub, Gmail, Calendar, Canva и др.) запросом к RPC `plugin/list` Codex.
- **Регистрирует собственные инструменты Hermes как MCP‑сервер**, чтобы подпроцесс codex мог вызывать инструменты, которые не поставляются с codex.
- **Записывает `default_permissions = ":workspace"`**, чтобы песочница разрешала записи внутри рабочей области без запроса подтверждения для каждой операции.
- Сообщает, что было перенесено. Вступает в силу в **следующей** сессии — текущий кэшированный агент сохраняет прежнее окружение, поэтому кэши подсказок остаются действительными.

Синонимы: `/codex-runtime on`, `/codex-runtime off`, `/codex-runtime auto`.

Чтобы проверить текущее состояние без изменения чего‑либо:
```
/codex-runtime
```

Также можно задать вручную в `~/.hermes/config.yaml`:
```yaml
model:
  openai_runtime: codex_app_server   # default is "auto" (= Hermes runtime)
```
## Цикл самоулучшения (память + подсказки навыков)

Фоновое самоулучшение Hermes срабатывает при достижении пороговых счётчиков:

- Каждый 10‑й запрос пользователя → форкнутый агент‑обзор смотрит на диалог и решает, сохранять ли что‑то в память.
- Каждая 10‑я итерация инструмента в рамках одного хода → та же идея, но для навыков (`skill_manage` записывает).

**Оба продолжают работать в среде codex.** Путь `codex` проецирует каждый завершённый элемент `commandExecution` / `fileChange` / `mcpToolCall` / `dynamicToolCall` в синтетическое сообщение `assistant tool_call` + результат `tool`, так что к моменту запуска обзора он видит ту же структуру, что и в обычной среде Hermes.

Как сохраняется эквивалентность соединения:

| | Обычная среда | Среда Codex |
|---|---|---|
| `_turns_since_memory` увеличивается | при каждом запросе пользователя, в `run_conversation` до цикла | тот же путь кода, перед ранним возвратом |
| `_iters_since_skill` увеличивается | при каждой итерации инструмента в цикле `chat-completions` | через `turn.tool_iterations` после возврата хода `codex` |
| Триггер памяти (`_turns_since_memory >= _memory_nudge_interval`) | вычисляется в предцикле, срабатывает после ответа | вычисляется в предцикле, передаётся вспомогательному коду `codex` |
| Триггер навыка (`_iters_since_skill >= _skill_nudge_interval`) | вычисляется после цикла | вычисляется после хода `codex` |
| `_spawn_background_review(messages_snapshot=..., review_memory=..., review_skills=...)` | вызывается, когда срабатывает любой из триггеров | вызывается идентично, когда срабатывает любой из триггеров |

Одна деталь: форк обзора сам должен вызывать инструменты цикла агента Hermes (`memory`, `skill_manage`), которые требуют собственного диспетчера Hermes. Поэтому, когда родительский агент работает на `codex_app_server`, форк обзора **понижается до `codex_responses`** — те же OAuth‑учётные данные, тот же провайдер `openai-codex`, но обращается напрямую к API Responses от OpenAI, так что Hermes сохраняет контроль над циклом и инструменты цикла агента работают. Это скрыто от пользователя.

Итоговый эффект: включи среду `codex`, и твои подсказки памяти + навыков продолжают срабатывать точно так же, как и раньше.
## Как работают одобрения

Codex запрашивает одобрение перед выполнением команд или применением патчей. Эти запросы переводятся в стандартный запрос Hermes «Опасная команда»:

```
╭───────────────────────────────────────╮
│ Dangerous Command                     │
│                                       │
│ /bin/bash -lc 'echo hello > foo.txt'  │
│                                       │
│ ❯ 1. Allow once                       │
│   2. Allow for this session           │
│   3. Deny                             │
│                                       │
│ Codex requests exec in /your/cwd      │
╰───────────────────────────────────────╯
```

- **Allow once** → разрешить эту одну команду.
- **Allow for this session** → разрешить на эту сессию (Codex больше не будет запрашивать подтверждение для похожих команд).
- **Deny** → отказ; команда отклоняется, Codex продолжает работу в режиме только для чтения.

Для одобрений `apply_patch` (изменение файлов) Hermes показывает сводку того, что изменилось (`1 add, 1 update: /tmp/new.py, /tmp/old.py`), когда Codex предоставляет данные через соответствующий элемент `fileChange`.
## Профили разрешений

Codex имеет три встроенных профиля разрешений:
- `:read-only` — без записи; каждая команда оболочки требует одобрения
- `:workspace` — запись в текущем рабочем пространстве разрешена без запросов (значение по умолчанию Hermes при включении runtime)
- `:danger-no-sandbox` — полностью без песочницы (не используй, если не понимаешь, как это работает)

Ты можешь переопределить значение по умолчанию в `~/.codex/config.toml` вне управляемого блока Hermes:

```toml
default_permissions = ":read-only"
```

(Hermes сохранит твоё переопределение при повторной миграции, пока оно находится за пределами маркеров `# managed by hermes-agent`.)
## Вспомогательные задачи и стоимость токенов подписки ChatGPT

Когда эта среда включена с провайдером `openai-codex`, **вспомогательные задачи (генерация заголовка, сжатие контекста, автоопределение визуального контента, форк фонового самообучающегося обзора) также проходят через твою подписку ChatGPT по умолчанию**, потому что вспомогательный клиент Hermes использует основной провайдер/модель, если не задано переопределение для конкретной задачи.

Это относится не только к `codex_app_server` — это верно и для существующего пути `codex_responses` — но здесь это более заметно, поскольку ты явно согласился на биллинг по подписке.

Чтобы направить конкретные вспомогательные задачи к более дешёвой / другой модели, задай явные переопределения в `~/.hermes/config.yaml`:

```yaml
auxiliary:
  title_generation:
    provider: openrouter
    model: google/gemini-3-flash-preview
  context_compression:
    provider: openrouter
    model: google/gemini-3-flash-preview
  vision_detect:
    provider: openrouter
    model: google/gemini-3-flash-preview
  goal_judge:
    provider: openrouter
    model: google/gemini-3-flash-preview
```

Форк обзора самообучающегося агента наследует основную среду выполнения через `_current_main_runtime()` и Hermes автоматически понижает его с `codex_app_server` до `codex_responses` (чтобы форк мог действительно вызывать `memory` и `skill_manage` — инструменты собственного цикла агента Hermes). Этот форк всё равно использует твою аутентификацию подписки, если только ты не перенаправил вспомогательные задачи в другое место.
## Безопасное редактирование `~/.codex/config.toml`

Hermes помещает всё, чем управляет, между двумя маркерными комментариями:

```toml
# managed by hermes-agent — `hermes codex-runtime migrate` regenerates this section
default_permissions = ":workspace"
[mcp_servers.filesystem]
...
[plugins."github@openai-curated"]
...
# end hermes-agent managed section
```

Всё **вне** этого блока — твоё. Повторный запуск миграции (через `/codex-runtime codex_app_server` или каждый раз, когда ты включаешь runtime) заменяет управляемый блок на месте, но сохраняет пользовательский контент выше и ниже его дословно. Это значит, что ты можешь:

- Добавить свои MCP‑серверы, о которых Hermes не знает
- Переопределить `default_permissions` на `:read-only`, если ты предпочитаешь получать запросы
- Настроить параметры, относящиеся только к codex (model, providers, otel и т.д.)
- Добавить пользовательские профили разрешений в таблицах `[permissions.<name>]`

Любое изменение **внутри** управляемого блока будет перезаписано при следующей миграции. Если тебе нужен параметр, требующий правки управляемого блока, создай issue, и мы добавим соответствующий переключатель.
## Мульти‑профильные / мульти‑арендные настройки

По умолчанию Hermes указывает подпроцессу codex каталог `~/.codex/` независимо от того, какой профиль Hermes активен. Это значит, что `hermes -p work` и `hermes -p personal` используют одну и ту же аутентификацию Codex, плагины и конфигурацию. Для большинства пользователей такое поведение правильно — оно соответствует тому, что происходит при прямом запуске CLI `codex`.

Если требуется изоляция Codex — отдельная аутентификация, отдельные установленные плагины и отдельная конфигурация — задай переменную `CODEX_HOME` явно для каждого профиля. Самый чистый способ — указать каталог внутри твоего `HERMES_HOME`:

```bash
# Inside the work profile, you might wrap hermes:
CODEX_HOME=~/.hermes/profiles/work/codex hermes chat
```

Тебе понадобится один раз выполнить `codex login` с установленным `CODEX_HOME`, чтобы токены OAuth оказались в расположении, привязанном к профилю. После этого `hermes -p work` будет работать с изолированным состоянием Codex.

Мы не делаем это автоматически, потому что перемещение существующего каталога `~/.codex/` пользователя без предупреждения сделало бы их аутентификацию в CLI Codex недействительной — каждому, кто уже выполнил `codex login`, пришлось бы повторно аутентифицироваться. Опция включения по желанию выглядит безопаснее, чем неожиданно ломать работу пользователей.
## Переменная окружения HOME передаётся без изменений

Hermes не переписывает `HOME` при запуске подпроцесса сервера приложений codex (мы используем `os.environ.copy()` и накладываем только `CODEX_HOME` и `RUST_LOG`). Это означает:

- Команды, которые codex запускает через свой инструмент `shell`, видят реальный `HOME` пользователя и корректно находят `~/.gitconfig`, `~/.gh/`, `~/.aws/`, `~/.npmrc` и т.д.
- Внутреннее состояние Codex остаётся изолированным через `CODEX_HOME` (который по умолчанию указывает на `~/.codex/`).

Это соответствует границе, к которой пришёл OpenClaw после ранних экспериментов: изолировать состояние Codex, не изменять домашний каталог пользователя. (См. openclaw/openclaw#81562.)
## Миграция сервера MCP

Конфигурация Hermes `mcp_servers` автоматически переводится в формат TOML, который ожидает Codex. Миграция запускается каждый раз при включении runtime и является идемпотентной — повторные выполнения заменяют управляемый раздел, но сохраняют любые пользовательские изменения конфигурации Codex.

Что переводится:

| Hermes (`config.yaml`) | Codex (`config.toml`) |
|---|---|
| `command` + `args` + `env` | stdio transport |
| `url` + `headers` | streamable_http transport |
| `timeout` | `tool_timeout_sec` |
| `connect_timeout` | `startup_timeout_sec` |
| `enabled: false` | `enabled = false` |

Что не мигрируется:
- Ключи, специфичные для Hermes, такие как `sampling` (у клиента MCP в Codex нет эквивалента — они отбрасываются с предупреждением для каждого сервера).
## Миграция плагинов Native Codex

Плагины, установленные через `codex plugin` (Linear, GitHub, Gmail, Calendar, Canva и др.), обнаруживаются через RPC `plugin/list` Codex. Для каждого плагина, у которого `installed: true`, Hermes записывает блок `[plugins."<name>@openai-curated"]`, включающий его в твою сессию Hermes.

Это означает: когда твой друг говорит «У меня настроены Calendar и GitHub в Codex CLI» и включает runtime Codex в Hermes, Hermes активирует их автоматически. Перенастройка не требуется.

Что НЕ мигрируется:
- Плагины, которые ты ещё не установил — установи их в Codex сначала.
- Плагины, у которых Codex возвращает `availability != AVAILABLE` (повреждённая установка, истёкший OAuth, удалённые из маркета и т.п.). Они пропускаются, чтобы не записывать конфигурацию, которая бы не прошла активацию.
- Записи маркета приложений ChatGPT (результаты `app/list` для каждой учётной записи — они уже включены в Codex благодаря авторизации твоей учётной записи).
- OAuth плагина — ты авторизуешь каждый плагин один раз непосредственно в Codex; Hermes не трогает учётные данные.
## Hermes tool callback (новый сервер MCP)

Встроенный набор инструментов Codex охватывает операции с оболочкой, файлами и патчами, но не включает веб‑поиск, автоматизацию браузера, зрение, генерацию изображений и т.п. Чтобы эти возможности оставались доступными в ходе выполнения Codex, Hermes регистрирует себя как сервер MCP в `~/.codex/config.toml`:

```toml
[mcp_servers.hermes-tools]
command = "/path/to/python"
args = ["-m", "agent.transports.hermes_tools_mcp_server"]
env = { HERMES_HOME = "/your/.hermes", PYTHONPATH = "...", HERMES_QUIET = "1" }
startup_timeout_sec = 30.0
tool_timeout_sec = 600.0
```

Когда модель вызывает `web_search` (или другой доступный инструмент Hermes), Codex запускает подпроцесс `hermes_tools_mcp_server` через stdio, запрос передаётся через `model_tools.handle_function_call()`, а результат возвращается в Codex так же, как любой другой ответ MCP.

**Инструменты, доступные через callback:** `web_search`, `web_extract`, `browser_navigate`, `browser_click`, `browser_type`, `browser_press`, `browser_snapshot`, `browser_scroll`, `browser_back`, `browser_get_images`, `browser_console`, `browser_vision`, `vision_analyze`, `image_generate`, `skill_view`, `skills_list`, `text_to_speech`.

**Инструменты, НЕ доступные:** `delegate_task`, `memory`, `session_search`, `todo`. Для их использования требуется контекст работающего AIAgent (состояние в середине цикла), а stateless MCP callback не может их управлять. Используй стандартное окружение Hermes (`/codex-runtime auto`), когда нужны эти инструменты.
## Отключение

Вернуться в любой момент:

```
/codex-runtime auto
```

Изменения вступят в силу со следующей сессии. Блок, управляемый Codex, остаётся в `~/.codex/config.toml`, поэтому ты можешь включить его позже без потери конфигурации — либо удалить его вручную, если предпочитаешь.
## Ограничения

Это окружение **beta по желанию**. Работает в Hermes Agent 2026.5 + Codex CLI 0.130.0:

- Многошаговые диалоги
- Одобрения `commandExecution` и `fileChange` (apply_patch) через интерфейс Hermes UI
- Вызовы инструмента MCP (проверено против `@modelcontextprotocol/server-filesystem` и нового обратного вызова `hermes-tools`)
- Миграция нативных плагинов Codex (проверено против инвентаря Linear / GitHub / Calendar)
- Запрет/отмена путей
- Переключение включения/выключения цикла
- Счётчики подсказок памяти и навыков (проверено в реальном времени интеграционными тестами)
- Веб‑поиск Hermes через codex (проверено в реальном времени: «OpenAI Codex CLI – Getting Started» вернул end-to-end)

Известные ограничения:

- **Hermes auth и codex auth — отдельные сессии.** Требуется выполнить одновременно `codex login` И `hermes auth login codex` для наилучшего UX (runtime использует сессию codex для вызова LLM). Это осознанный выбор дизайна в Hermes `_import_codex_cli_tokens` — Hermes не будет делиться состоянием OAuth с codex CLI, чтобы избежать конфликтов при обновлении токенов.
- **`delegate_task`, `memory`, `session_search`, `todo` недоступны в этом runtime.** Они требуют контекста работающего AIAgent, который stateless‑callback MCP не может предоставить. Используй `/codex-runtime auto`, когда нужны эти функции.
- **Отсутствие предварительного просмотра патча в запросах одобрения, если codex не отслеживает набор изменений.** Параметры одобрения `fileChange` в Codex не всегда содержат changeset. Hermes кэширует данные из соответствующего уведомления `item/started`, когда это возможно, но если одобрение приходит до того, как элемент был передан, запрос падает к тому `reason`, который предоставляет codex.
- **Гарантировать отмену менее чем за секунду нельзя.** Прерывания в середине потока (Ctrl+C, пока codex отвечает) отправляются через `turn/interrupt`, но если codex уже отправил финальное сообщение, ты всё равно получишь ответ.

Если нашёл баг, [открой issue](https://github.com/NousResearch/hermes-agent/issues) с выводом `hermes logs --since 5m`. Укажи `codex-runtime` в заголовке, чтобы его было легче обработать.
