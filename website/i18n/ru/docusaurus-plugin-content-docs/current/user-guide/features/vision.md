# Vision & Image Paste

Hermes Agent поддерживает **мультимодальное зрение** — ты можешь вставлять изображения из буфера обмена напрямую в CLI и просить агента их анализировать, описывать или работать с ними. Изображения отправляются модели как блоки контента, закодированные в base64, поэтому любую модель с поддержкой зрения может их обработать.

:::tip
Подписчики Portal получают модели с поддержкой зрения (Claude, GPT‑5, Gemini) в том же каталоге — дополнительные учётные данные не требуются. См. [Nous Portal](/integrations/nous-portal).
:::
## Как это работает

1. Скопируй изображение в буфер обмена (скриншот, изображение из браузера и т.п.).
2. Прикрепи его одним из способов ниже.
3. Введи свой вопрос и нажми **Enter**.
4. Изображение появляется в виде бейджа `[📎 Image #1]` над полем ввода.
5. При отправке изображение передаётся модели как блок визуального контента.

Можно прикрепить несколько изображений перед отправкой — каждое получит свой бейдж. Нажми `Ctrl+C`, чтобы очистить все прикреплённые изображения.

Изображения сохраняются в `~/.hermes/images/` как PNG‑файлы с именами, содержащими метку времени.
## Методы вставки

То, как ты прикрепляешь изображение, зависит от среды терминала. Не все методы работают везде — ниже полное описание:

### Команда `/paste`

**Самый надёжный явный запасной (вариант) прикрепления изображения.**

```
/paste
```

Введи `/paste` и нажми Enter. Hermes проверяет буфер обмена на наличие изображения и прикрепляет его. Это самый безопасный вариант, когда твой терминал переопределяет `Cmd+V`/`Ctrl+V`, или когда ты скопировал только изображение и в буфере нет текстовой нагрузки с квадратными скобками для анализа.

### Ctrl+V / Cmd+V

Hermes теперь обрабатывает вставку как многоуровневый процесс:
- сначала обычная вставка текста;
- затем запасной (вариант) текста из нативного буфера обмена / OSC52, если терминал не передал текст корректно;
- прикрепление изображения, когда буфер обмена или вставленная нагрузка разрешается в изображение или путь к изображению.

Это означает, что временные пути скриншотов macOS и URI изображений вида `file://…` могут сразу прикрепляться вместо того, чтобы оставаться в редакторе как сырой текст.

:::warning
Если в буфере обмена **только изображение** (без текста), терминалы всё равно не могут отправлять бинарные байты изображения напрямую. Используй `/paste` как явный запасной (вариант) прикрепления изображения.
:::

### `/terminal-setup` для VS Code / Cursor / Windsurf

Если ты запускаешь TUI внутри локального интегрированного терминала семейства VS Code на macOS, Hermes может установить рекомендуемые `workbench.action.terminal.sendSequence`‑биндинги для лучшей поддержки многострочного ввода и отмены / повтора:

```text
/terminal-setup
```

Это особенно полезно, когда `Cmd+Enter`, `Cmd+Z` или `Shift+Cmd+Z` перехватываются IDE. Запускай это только на локальной машине — не внутри SSH‑сессии.
## Совместимость платформ

| Окружение | `/paste` | Cmd/Ctrl+V | `/terminal-setup` | Примечания |
|---|:---:|:---:|:---:|---|
| **macOS Terminal / iTerm2** | ✅ | ✅ | n/a | Оптимальный опыт — нативный буфер обмена + восстановление пути к скриншоту |
| **Apple Terminal** | ✅ | ✅ | n/a | Если Cmd+←/→/⌫ переписывается, используй запасные варианты Ctrl+A / Ctrl+E / Ctrl+U |
| **Linux X11 desktop** | ✅ | ✅ | n/a | Требуется `xclip` (`apt install xclip`) |
| **Linux Wayland desktop** | ✅ | ✅ | n/a | Требуется `wl-paste` (`apt install wl-clipboard`) |
| **WSL2 (Windows Terminal)** | ✅ | ✅ | n/a | Использует `powershell.exe` — дополнительная установка не требуется |
| **VS Code / Cursor / Windsurf (local)** | ✅ | ✅ | ✅ | Рекомендуется для лучшей согласованности Cmd+Enter / отмена / повтор |
| **VS Code / Cursor / Windsurf (SSH)** | ❌² | ❌² | ❌³ | Выполни `/terminal-setup` на локальном компьютере |
| **SSH terminal (any)** | ❌² | ❌² | n/a | Удалённый буфер обмена недоступен |

## Настройка под конкретную платформу

### macOS

**Настройка не требуется.** Hermes использует `osascript` (встроенный в macOS) для чтения буфера обмена. Для повышения производительности, при желании установи `pngpaste`:

```bash
brew install pngpaste
```

### Linux (X11)

Установи `xclip`:

```bash
# Ubuntu/Debian
sudo apt install xclip

# Fedora
sudo dnf install xclip

# Arch
sudo pacman -S xclip
```

### Linux (Wayland)

Современные Linux‑десктопы (Ubuntu 22.04+, Fedora 34+) часто используют Wayland по умолчанию. Установи `wl-clipboard`:

```bash
# Ubuntu/Debian
sudo apt install wl-clipboard

# Fedora
sudo dnf install wl-clipboard

# Arch
sudo pacman -S wl-clipboard
```

:::tip Как проверить, используешь ли ты Wayland
```bash
echo $XDG_SESSION_TYPE
# "wayland" = Wayland, "x11" = X11, "tty" = no display server
```
:::

### WSL2

**Дополнительная настройка не требуется.** Hermes автоматически определяет WSL2 (через `/proc/version`) и использует `powershell.exe` для доступа к буферу обмена Windows через .NET‑класс `System.Windows.Forms.Clipboard`. Это встроено в межоперационную поддержку WSL2 — `powershell.exe` доступен по умолчанию.

Данные буфера обмена передаются как PNG, закодированный в base64, через stdout, поэтому преобразование путей файлов или временные файлы не нужны.

:::info Примечание о WSLg
Если ты используешь WSLg (WSL2 с поддержкой GUI), Hermes сначала пытается использовать PowerShell, затем переходит к `wl-paste`. Мост буфера обмена WSLg поддерживает только формат BMP для изображений — Hermes автоматически конвертирует BMP в PNG с помощью Pillow (если установлен) или команды `convert` из ImageMagick.
:::

#### Проверка доступа к буферу обмена в WSL2

```bash
# 1. Check WSL detection
grep -i microsoft /proc/version

# 2. Check PowerShell is accessible
which powershell.exe

# 3. Copy an image, then check
powershell.exe -NoProfile -Command "Add-Type -AssemblyName System.Windows.Forms; [System.Windows.Forms.Clipboard]::ContainsImage()"
# Should print "True"
```
## SSH & Remote Sessions

**Вставка изображения из буфера обмена не полностью работает через SSH.** Когда ты подключаешься к удалённой машине по SSH, Hermes CLI запускается на удалённом хосте. Инструменты буфера обмена (`xclip`, `wl-paste`, `powershell.exe`, `osascript`) читают буфер обмена машины, на которой они запущены — то есть удалённого сервера, а не твоего локального компьютера. Поэтому локальное изображение из буфера недоступно со стороны удалённого сервера.

Текст иногда всё ещё может передаваться через вставку в терминале или OSC52, но доступ к изображению из буфера и временные пути локального скриншота остаются привязанными к машине, где запущен Hermes.

### Обходные пути для SSH

1. **Загрузить файл изображения** — сохрани изображение локально, загрузив его на удалённый сервер через `scp`, проводник файлов VSCode (перетаскивание) или любой другой метод передачи файлов. Затем укажи путь к нему. *(Команда `/attach <filepath>` запланирована для будущего релиза.)*

2. **Использовать URL** — если изображение доступно в интернете, просто вставь URL в своё сообщение. Агент может использовать `vision_analyze` для просмотра любого URL изображения напрямую.

3. **Перенаправление X11** — подключись с `ssh -X` для перенаправления X11. Это позволяет `xclip` на удалённой машине получать доступ к твоему локальному буферу X11. Требуется запущенный локально X‑сервер (XQuartz на macOS, встроенный в Linux‑десктопы X11). Медленно для больших изображений.

4. **Использовать платформу обмена сообщениями** — отправляй изображения Hermes через Telegram, Discord, Slack или WhatsApp. Эти платформы нативно поддерживают загрузку изображений и не зависят от ограничений буфера обмена/терминала.
## Почему терминалы не могут вставлять изображения

Это частый источник путаницы, поэтому вот техническое объяснение:

Терминалы — **текстовые** интерфейсы. Когда ты нажимаешь Ctrl+V (или Cmd+V), эмулятор терминала:

1. Считывает буфер обмена на **текстовое содержимое**
2. Оборачивает его в escape‑последовательности [bracketed paste](https://en.wikipedia.org/wiki/Bracketed-paste)
3. Отправляет приложение через текстовый поток терминала

Если в буфере обмена находится только изображение (без текста), у терминала нечего отправлять. Стандартной escape‑последовательности для бинарных данных изображения не существует. Терминал просто ничего не делает.

Именно поэтому Hermes использует отдельную проверку буфера обмена — вместо получения данных изображения через событие вставки в терминале, он напрямую вызывает инструменты уровня ОС (`osascript`, `powershell.exe`, `xclip`, `wl-paste`) через `subprocess`, чтобы прочитать буфер обмена независимо.
## Поддерживаемые модели

Вставка изображения работает с любой моделью, поддерживающей зрение. Изображение отправляется в виде URL‑данных, закодированных в base64, в формате контента OpenAI vision:

```json
{
  "type": "image_url",
  "image_url": {
    "url": "data:image/png;base64,..."
  }
}
```

Большинство современных моделей поддерживают этот формат, включая GPT‑4 Vision, Claude (с поддержкой зрения), Gemini и модели с открытым исходным кодом, обслуживаемые через OpenRouter.
## Маршрутизация изображений (модели с поддержкой зрения vs только текстовые модели)

Когда пользователь прикрепляет изображение — из буфера обмена CLI, через gateway (Telegram/Discord фото) или любой другой точкой входа — Hermes маршрутизирует его в зависимости от того, поддерживает ли текущая модель зрение:

| Ваша модель | Что происходит с изображением |
|---|---|
| **Поддерживающая зрение** (GPT-4V, Claude with vision, Gemini, Qwen-VL, MiMo-VL и др.) | Отправляется как **реальные пиксели** с использованием нативного формата изображения провайдера, описанного выше. Без слоя текстового резюме. |
| **Только текстовая** (DeepSeek V3, более мелкие open‑source модели, старые чат‑только эндпоинты) | Маршрутизируется через вспомогательный инструмент `vision_analyze` — вспомогательная модель зрения описывает изображение, и текстовое описание внедряется в разговор. |

Ты не настраиваешь это — Hermes смотрит возможность текущей модели в метаданных провайдера и автоматически выбирает правильный путь. Практический эффект: можно переключаться между моделями с поддержкой зрения и без неё в середине сессии, и обработка изображений «просто работает» без изменения рабочего процесса. Текстовые модели получают согласованный контекст об изображении, а не повреждённый мультимодальный пакет, который они вынуждены отклонять.

Какая вспомогательная модель обрабатывает путь текстового описания, настраивается в `auxiliary.vision` — см. [Auxiliary Models](/user-guide/configuration#auxiliary-models).

### `vision_analyze` имеет такое же двойное поведение

Сам инструмент `vision_analyze` следует тем же правилам маршрутизации. Когда активная основная модель поддерживает зрение **и** её провайдер поддерживает изображение в результатах инструмента (в текущий момент стеки Anthropic, OpenAI, Azure‑OpenAI и Gemini 3.x), `vision_analyze` обходит вспомогательный описыватель и возвращает сырые пиксели изображения в виде мультимодального конверта результата инструмента. Основная модель видит изображение нативно на следующем ходу — без вызова aux, без потери информации о текстовом резюме, без дополнительной задержки.

Для только текстовых основных моделей (или провайдеров, чей канал результата инструмента не передаёт изображения) `vision_analyze` переходит к наследуемому пути: он запрашивает настроенную вспомогательную модель зрения, чтобы описать изображение, и возвращает описание как обычный текст. В любом случае сигнатура вызывающего инструмента остаётся той же — инструмент решает, какой путь выбрать во время выполнения, исходя из активной модели.
