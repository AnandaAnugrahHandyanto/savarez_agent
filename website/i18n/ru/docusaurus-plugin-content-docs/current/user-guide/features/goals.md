# Постоянные цели (`/goal`)

`/goal` задаёт Hermes постоянную цель, которая сохраняется между ходами. После каждого хода лёгкая модель‑судья проверяет, выполнена ли цель последним ответом ассистента. Если нет, Hermes автоматически подаёт запрос‑продолжение в ту же сессию и продолжает работу — пока цель не будет достигнута, ты не приостановишь её или не очистишь, либо не исчерпается бюджет ходов.

Это наш вариант **Ralph loop**, напрямую вдохновлённый [Codex CLI 0.128.0's `/goal`](https://github.com/openai/codex) Эрика Траута (OpenAI). Основная идея — поддерживать цель живой между ходами и не останавливаться, пока она не будет достигнута — принадлежит им. Реализация здесь независима и адаптирована к архитектуре Hermes.
## Когда использовать

Используй `/goal` для задач, где ты хочешь, чтобы Hermes сам итеративно работал без необходимости каждый раз подсказывать ему:

- «Исправить все ошибки линтинга в `src/` и убедиться, что `ruff check` проходит»
- «Перенести функцию X из репозитория Y, включая тесты, и получить зелёный CI»
- «Исследовать, почему идентификаторы сессий иногда отклоняются при сжатии в середине выполнения, и написать отчёт»
- «Создать небольшой CLI для переименования файлов по их датам EXIF, затем протестировать его в папке `photos/`»

Задачи, в которых агент делает один ход и останавливается, не требуют `/goal`. Задачи, где *тебе пришлось бы сказать «продолжай» три раза*, именно здесь проявляют себя лучше всего.
## Быстрый старт

```
/goal Fix every failing test in tests/hermes_cli/ and make sure scripts/run_tests.sh passes for that directory
```

Что ты увидишь:

1. **Goal accepted** — `⊙ Goal set (20-turn budget): <your goal>`
2. **Turn 1 runs** — Hermes начинает работать, как будто ты отправил цель обычным сообщением.
3. **Judge runs** — после хода модель‑судья решает `done` или `continue`.
4. **Loop fires if needed** — если `continue`, ты увидишь `↻ Continuing toward goal (1/20): <judge's reason>` и Hermes автоматически выполнит следующий шаг.
5. **Terminates** — в конце ты увидишь либо `✓ Goal achieved: <reason>`, либо `⏸ Goal paused — N/20 turns used`.
## Команды

| Команда | Что делает |
|---|---|
| `/goal <text>` | Устанавливает (или заменяет) текущую цель. Немедленно запускает первый ход, поэтому отдельное сообщение отправлять не нужно. |
| `/goal` или `/goal status` | Показывает текущую цель, её статус и количество использованных ходов. |
| `/goal pause` | Останавливает цикл автоматического продолжения без очистки цели. |
| `/goal resume` | Возобновляет цикл (сбрасывает счётчик ходов до нуля). |
| `/goal clear` | Полностью удаляет цель. |

Работает одинаково в CLI и на всех платформах шлюза (Telegram, Discord, Slack, Matrix, Signal, WhatsApp, SMS, iMessage, Webhook, API server и веб‑дашборд).
## Добавление критериев в середине цели: `/subgoal`

Пока цель активна, ты можешь добавить дополнительные критерии приёма с помощью `/subgoal <текст>` без сброса цикла. Каждый вызов добавляет один пронумерованный пункт в список **subgoal**‑ов цели; **prompt продолжения**, который агент видит на следующем ходу, включает исходную цель плюс блок «Additional criteria the user added mid-loop», а **prompt судьи** переписывается так, чтобы вердикт учитывал каждую подцель — цель не считается выполненной, пока не выполнены исходная задача **и** все подцели.

| Команда | Что делает |
|---|---|
| `/subgoal <text>` | Добавляет новый критерий к активной цели. Требуется активная `/goal`. |
| `/subgoal` (без аргументов) | Показывает текущий пронумерованный список подцелей. |
| `/subgoal remove <N>` | Удаляет N‑й subgoal (нумерация с 1). |
| `/subgoal clear` | Удаляет все subgoal‑ы, но сохраняет оригинальную цель. |

Subgoal‑ы сохраняются вместе с целью в `SessionDB.state_meta`, поэтому они сохраняются после `/resume`. Установка новой `/goal <text>` заменяет цель и очищает список subgoal‑ов; `/goal clear` делает то же самое.

Используй это, когда начинаешь цикл («исправить падающие тесты») и замечаешь, что в процессе также нужно «добавить регрессионный тест для только что исправленной ошибки» — `/subgoal add a regression test` уточняет критерии успеха без прерывания запущенного цикла.
## Подробности поведения

### Судья

После каждого хода Hermes вызывает вспомогательную модель с:

- Текстом текущей цели
- Последним окончательным ответом агента (около ~4 KB текста)
- Системным запросом, заставляющим судью отвечать строгим JSON: `{"done": <bool>, "reason": "<one-sentence rationale>"}`

Судья преднамеренно консервативен: он помечает цель как `done` только когда ответ **явно** подтверждает завершение цели, когда конечный результат явно произведён, или когда цель недостижима/заблокирована (рассматривается как DONE с причиной блокировки, чтобы не тратить бюджет на невозможные задачи).

### Семантика fail-open

Если судья выдаёт ошибку (сбой сети, некорректный ответ, недоступный вспомогательный клиент), Hermes трактует вердикт как `continue` — сломанный судья никогда не останавливает прогресс. **Бюджет ходов** служит реальным ограничителем.

### Бюджет ходов

По умолчанию — 20 ходов продолжения (`goals.max_turns` в `config.yaml`). Когда бюджет исчерпан, Hermes автоматически ставит паузу и сообщает, как действовать дальше:

```
⏸ Goal paused — 20/20 turns used. Use /goal resume to keep going, or /goal clear to stop.
```

`/goal resume` сбрасывает счётчик в ноль, позволяя продолжать работу небольшими порциями.

### Сообщения пользователя всегда имеют приоритет

Любое реальное сообщение, которое ты отправляешь, пока цель активна, имеет приоритет над циклом продолжения. В CLI твоё сообщение попадает в `_pending_input` перед запланированным продолжением; в шлюзе оно проходит через FIFO‑адаптер тем же способом. Судья запускается снова после твоего хода — если твоё сообщение случайно завершит цель, судья это зафиксирует и остановит процесс.

### Безопасность во время выполнения (шлюз)

Пока агент уже запущен, команды `/goal status`, `/goal pause` и `/goal clear` безопасны — они затрагивают только состояние управляющей плоскости и не прерывают текущий ход. Установка **новой** цели во время выполнения (`/goal <new text>`) отклоняется с сообщением, предлагающим сначала выполнить `/stop`, чтобы старая цепочка не конкурировала с новой.

### Устойчивость

Состояние цели хранится в `SessionDB.state_meta` под ключом `goal:<session_id>`. Это значит, что `/resume` продолжает работу ровно там, где ты остановился — задаёшь цель, закрываешь ноутбук, возвращаешься завтра, вводишь `/resume`, и цель остаётся в том же состоянии (активна, на паузе или завершена).

### Кеш подсказок

Подсказка продолжения представляет собой обычное сообщение роли `user`, добавленное к истории. Она **не** изменяет системный запрос, не меняет набор инструментов и не затрагивает разговор таким образом, который бы инвалидировал кеш подсказок Hermes. Выполнение цели в 20 ходов стоит столько же в плане кеша, сколько 20 обычных ходов диалога.
## Конфигурация

Добавь в `~/.hermes/config.yaml`:

```yaml
goals:
  # Max continuation turns before Hermes auto-pauses and asks you to
  # /goal resume. Default 20. Lower this if you want tighter loops;
  # raise it for long-running refactors.
  max_turns: 20
```

### Выбор модели‑судьи

Судья использует вспомогательную задачу `goal_judge`. По умолчанию используется твоя основная модель (см. [Auxiliary Models](/user-guide/configuration#auxiliary-models)). Если ты хочешь направить судью на дешёвую быструю модель, чтобы сократить расходы, добавь переопределение:

```yaml
auxiliary:
  goal_judge:
    provider: openrouter
    model: google/gemini-3-flash-preview
```

Вызов судьи небольшой (~200 токенов вывода) и выполняется один раз за ход, поэтому обычно имеет смысл использовать дешёвую быструю модель.
## Пример пошагового прохождения

```
You: /goal Create four files /tmp/note_{1..4}.txt, one per turn, each containing its number as text

  ⊙ Goal set (20-turn budget): Create four files /tmp/note_{1..4}.txt, one per turn, each containing its number as text

Hermes: Creating /tmp/note_1.txt now.
  💻 echo "1" > /tmp/note_1.txt   (0.1s)
  I've created /tmp/note_1.txt with the content "1". I'll continue with the remaining files on the next turn as you specified.

  ↻ Continuing toward goal (1/20): Only 1 of 4 files has been created; 3 files remain.

Hermes: [Continuing toward your standing goal]
  💻 echo "2" > /tmp/note_2.txt   (0.1s)
  Created /tmp/note_2.txt. Two more to go.

  ↻ Continuing toward goal (2/20): 2 of 4 files created; 2 remain.

Hermes: [Continuing toward your standing goal]
  💻 echo "3" > /tmp/note_3.txt   (0.1s)
  Created /tmp/note_3.txt.

  ↻ Continuing toward goal (3/20): 3 of 4 files created; 1 remains.

Hermes: [Continuing toward your standing goal]
  💻 echo "4" > /tmp/note_4.txt   (0.1s)
  All four files have been created: /tmp/note_1.txt through /tmp/note_4.txt, each containing its number.

  ✓ Goal achieved: All four files were created with the specified content, completing the goal.

You: _
```

Четыре хода, один вызов `/goal`, ноль запросов «keep going» от тебя.
## Когда судья ошибается

Ни один судья не идеален. Есть два режима сбоя, за которыми стоит следить:

**Ложный отрицательный результат — судья говорит «продолжай», хотя цель уже достигнута.** Ограничение количества ходов ловит это. Ты увидишь `⏸ Goal paused` и сможешь выполнить `/goal clear` или просто отправить новое сообщение.

**Ложный положительный результат — судья говорит «завершено», хотя работа ещё продолжается.** Ты увидишь `✓ Goal achieved`, но знаешь лучше. Отправь последующее сообщение, чтобы продолжить, или уточни цель более точно: `/goal <более конкретный текст>`. Системный запрос судьи намеренно консервативен, чтобы ложные положительные результаты были реже, чем ложные отрицательные.

Если verdict судьи кажется тебе неубедительным, текст причины в строке `↻ Continuing toward goal` или `✓ Goal achieved` подскажет, что именно увидел судья. Обычно этого достаточно, чтобы диагностировать, был ли текст цели неоднозначным или ответ модели был.
## Атрибуция

`/goal` — это реализация Hermes шаблона **Ralph loop**. Дизайн, ориентированный на пользователя — поддерживать цель живой на протяжении ходов, не останавливаясь, пока она не будет достигнута, с элементами управления `create`/`pause`/`resume`/`clear` — был популяризирован и выпущен в [Codex CLI 0.128.0](https://github.com/openai/codex) Эриком Траутом из команды OpenAI Codex. Наша реализация независима (центральный реестр `CommandDef`, постоянное хранение `SessionDB.state_meta`, вспомогательный клиент‑судья, адаптер‑FIFO продолжения на стороне шлюза), но идея принадлежит им. Отдаём должное.
