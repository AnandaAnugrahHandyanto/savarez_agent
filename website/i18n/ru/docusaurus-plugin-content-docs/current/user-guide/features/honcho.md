# Honcli Memory

[Honcho](https://github.com/plastic-labs/honcho) — это AI‑нативный backend памяти, который добавляет диалектическое рассуждение и глубокое моделирование пользователя поверх встроенной системы памяти Hermes. Вместо простого хранилища «ключ‑значение» Honcho поддерживает актуальную модель пользователя — его предпочтения, стиль общения, цели и шаблоны — рассуждая о разговорах после их завершения.

:::info Honcho is a Memory Provider Plugin
Honcho интегрирован в систему [Memory Providers](./memory-providers.md). Все функции ниже доступны через единый интерфейс провайдера памяти.
:::
## Что добавляет Honcho

| Возможность | Встроенная память | Honcho |
|-----------|----------------|--------|
| Сохранение между сессиями | ✔ Файловая MEMORY.md/USER.md | ✔ На сервере через API |
| Профиль пользователя | ✔ Ручное курирование агента | ✔ Автоматическое диалектическое рассуждение |
| Сводка сессии | — | ✔ Внедрение контекста, ограниченного сессией |
| Изоляция мульти‑агентов | — | ✔ Разделение профилей по каждому пиру |
| Режимы наблюдения | — | ✔ Унифицированное или направленное наблюдение |
| Выводы (полученные инсайты) | — | ✔ Серверное рассуждение о паттернах |
| Поиск по истории | ✔ Поиск FTS5 по сессии | ✔ Семантический поиск по выводам |

**Диалектическое рассуждение**: После каждого хода разговора (ограниченного `dialecticCadence`) Honcho анализирует обмен и выводит инсайты о предпочтениях, привычках и целях пользователя. Они накапливаются со временем, давая агенту всё более глубокое понимание, выходящее за рамки явно заявленного пользователем. Диалектическое рассуждение поддерживает многопроходную глубину (1–3 прохода) с автоматическим выбором холодного/тёплого промпта — запросы холодного старта фокусируются на общих фактах о пользователе, а тёплые запросы отдают приоритет контексту, ограниченному текущей сессией.

**Контекст, ограниченный сессией**: Базовый контекст теперь включает сводку сессии вместе с представлением пользователя и картой пира. Это даёт агенту осведомлённость о том, что уже обсуждалось в текущей сессии, снижая повторения и обеспечивая непрерывность.

**Профили мульти‑агентов**: Когда несколько экземпляров Hermes общаются с одним пользователем (например, помощник‑программист и персональный помощник), Honcho поддерживает отдельные профили «пиров». Каждый пир видит только свои наблюдения и выводы, предотвращая перекрёстное загрязнение контекста.
## Настройка

```bash
hermes memory setup    # select "honcho" from the provider list
```

Или настрой вручную:

```yaml
# ~/.hermes/config.yaml
memory:
  provider: honcho
```

```bash
echo 'HONCHO_API_KEY=***' >> ~/.hermes/.env
```

Получить API‑ключ на [honcho.dev](https://honcho.dev).
## Архитектура

### Двухуровневое внедрение контекста

Каждый ход (в режиме `hybrid` или `context`), Honcho собирает два уровня контекста, внедряемые в системный запрос:

1. **Базовый контекст** — сводка сессии, представление пользователя, карточка пользователя‑пира, само‑представление ИИ и карточка идентификации ИИ. Обновляется согласно `contextCadence`. Это слой «кто этот пользователь».
2. **Диалектическое дополнение** — синтезированное LLM‑рассуждение о текущем состоянии и потребностях пользователя. Обновляется согласно `dialecticCadence`. Это слой «что важно сейчас».

Оба уровня конкатенируются и обрезаются до бюджета `contextTokens` (если установлен).

### Выбор холодного/тёплого запроса

Диалектика автоматически выбирает одну из двух стратегий запросов:

- **Холодный старт** (ещё нет базового контекста): общий запрос — «Кто этот человек? Каковы его предпочтения, цели и стиль работы?»
- **Тёплая сессия** (базовый контекст уже существует): запрос в рамках сессии — «Исходя из того, что уже обсуждалось в этой сессии, какой контекст о пользователе наиболее релевантен?»

Это происходит автоматически в зависимости от того, заполнен ли базовый контекст.

### Три независимых регулятора конфигурации

Стоимость и глубина контролируются тремя независимыми регуляторами:

| Регулятор | Что контролирует | По умолчанию |
|------|-------------------|--------------|
| `contextCadence` | Количество ходов между вызовами API `context()` (обновление базового уровня) | `1` |
| `dialecticCadence` | Количество ходов между вызовами LLM `peer.chat()` (обновление диалектического уровня) | `2` (рекомендовано 1–5) |
| `dialecticDepth` | Число проходов `.chat()` за один вызов диалектики (1–3) | `1` |

Эти регуляторы независимы — можно часто обновлять контекст при редком диалектическом обновлении, либо выполнять глубокий многопроходный диалектик при низкой частоте. Пример: `contextCadence: 1, dialecticCadence: 5, dialecticDepth: 2` обновляет базовый контекст каждый ход, запускает диалектику каждые 5 ходов, а каждый запуск диалектики делает 2 прохода.

### Глубина диалектики (многопроходная)

Когда `dialecticDepth` > 1, каждый вызов диалектики выполняет несколько проходов `.chat()`:

- **Проход 0**: холодный или тёплый запрос (см. выше)
- **Проход 1**: самоаудит — выявляет пробелы в начальной оценке и синтезирует доказательства из недавних сессий
- **Проход 2**: согласование — проверяет противоречия между предыдущими проходами и формирует окончательный синтез

Каждый проход использует пропорциональный уровень рассуждения (более лёгкие ранние проходы, базовый уровень для основного прохода). Уровни можно переопределять per‑pass с помощью `dialecticDepthLevels` — например, `["minimal", "medium", "high"]` для глубины 3.

Проходы могут завершиться преждевременно, если предыдущий проход вернул сильный сигнал (длинный, структурированный вывод), поэтому глубина 3 не всегда означает 3 вызова LLM.

### Предразогрев при старте сессии

При инициализации сессии Honcho запускает диалектический вызов в фоне с полной сконфигурированной `dialecticDepth` и передаёт результат напрямую в сборку контекста первого хода. Однопроходный предразогрев на холодном пиру часто возвращает скудный вывод — многопроходная глубина запускает цикл аудит/согласование до того, как пользователь заговорит. Если предразогрев не успел завершиться к первому ходу, первый ход переходит к синхронному вызову с ограниченным тайм‑аутом.

### Адаптивный уровень рассуждения по запросу

Автоматически внедряемая диалектика масштабирует `dialecticReasoningLevel` в зависимости от длины запроса: +1 уровень при ≥120 симв., +2 при ≥400, ограничивается `reasoningLevelCap` (по умолчанию `"high"`). Отключить можно с помощью `reasoningHeuristic: false`, чтобы фиксировать каждый автоматический вызов на `dialecticReasoningLevel`. Доступные уровни: `minimal`, `low`, `medium`, `high`, `max`.
## Параметры конфигурации

Honcho настраивается в `~/.honcho/config.json` (глобально) или `$HERMES_HOME/honcho.json` (локально для профиля). Мастер настройки делает это за тебя.

### Полный справочник конфигурации

| Ключ | По умолчанию | Описание |
|-----|--------------|----------|
| `contextTokens` | `null` (без ограничения) | Бюджет токенов для автоматически внедряемого контекста за ход. Установи целое число (например, 1200), чтобы ограничить. Обрезается на границах слов |
| `contextCadence` | `1` | Минимальное количество ходов между вызовами API `context()` (обновление базового слоя) |
| `dialecticCadence` | `2` | Минимальное количество ходов между вызовами LLM `peer.chat()` (слой диалектики). Рекомендуется 1–5. В режиме `tools` не имеет значения — вызовы модели задаются явно |
| `dialecticDepth` | `1` | Число проходов `.chat()` за один вызов диалектики. Ограничено 1–3 |
| `dialecticDepthLevels` | `null` | Необязательный массив уровней рассуждений за проход, например `["minimal", "low", "medium"]`. Переопределяет пропорциональные значения по умолчанию |
| `dialecticReasoningLevel` | `'low'` | Базовый уровень рассуждений: `minimal`, `low`, `medium`, `high`, `max` |
| `dialecticDynamic` | `true` | При `true` модель может переопределять уровень рассуждений для каждого вызова через параметр инструмента |
| `dialecticMaxChars` | `600` | Максимальное количество символов результата диалектики, внедряемого в системный промпт |
| `recallMode` | `'hybrid'` | `hybrid` (авто‑внедрение + инструменты), `context` (только внедрение), `tools` (только инструменты) |
| `writeFrequency` | `'async'` | Когда сбрасывать сообщения: `async` (фоновый поток), `turn` (синхронно), `session` (пакетно в конце) или целое число N |
| `saveMessages` | `true` | Сохранять ли сообщения через API Honcho |
| `observationMode` | `'directional'` | `directional` (все включены) или `unified` (общий пул). Переопределяется объектом `observation` для более тонкого контроля |
| `messageMaxChars` | `25000` | Максимальное количество символов в одном сообщении, отправляемом через `add_messages()`. При превышении разбивается на части |
| `dialecticMaxInputChars` | `10000` | Максимальное количество символов входных данных запроса диалектики к `peer.chat()` |
| `sessionStrategy` | `'per-directory'` | `per-directory`, `per-repo`, `per-session` или `global` |

**Стратегия сессии** определяет, как сессии Honcho сопоставляются с твоей работой:
- `per-session` — каждый запуск `hermes` получает новую сессию. Чистый старт, память через инструменты. Рекомендуется новичкам.
- `per-directory` — одна сессия Honcho на рабочий каталог. Контекст накапливается между запусками.
- `per-repo` — одна сессия на git‑репозиторий.
- `global` — одна сессия для всех каталогов.

**Режим воспоминаний** определяет, как память попадает в диалоги:
- `hybrid` — контекст автоматически внедряется в системный промпт **и** инструменты доступны (модель решает, когда их использовать).
- `context` — только авто‑внедрение, инструменты скрыты.
- `tools` — только инструменты, авто‑внедрение отключено. Агент должен явно вызывать `honcho_reasoning`, `honcho_search` и т.п.

**Настройки по режиму воспоминаний:**

| Настройка | `hybrid` | `context` | `tools` |
|----------|----------|-----------|---------|
| `writeFrequency` | сбрасывает сообщения | сбрасывает сообщения | сбрасывает сообщения |
| `contextCadence` | управляет обновлением базового контекста | управляет обновлением базового контекста | не имеет значения — нет внедрения |
| `dialecticCadence` | управляет автоматическими вызовами LLM | управляет автоматическими вызовами LLM | не имеет значения — вызовы модели задаются явно |
| `dialecticDepth` | многопроходный вызов за инвокацию | многопроходный вызов за инвокацию | не имеет значения — вызовы модели задаются явно |
| `contextTokens` | ограничивает внедрение | ограничивает внедрение | не имеет значения — нет внедрения |
| `dialecticDynamic` | регулирует возможность модели переопределять уровень рассуждений | N/A (нет инструментов) | регулирует возможность модели переопределять уровень рассуждений |

В режиме `tools` модель полностью контролирует процесс — она вызывает `honcho_reasoning`, когда хочет, с любым выбранным `reasoning_level`. Параметры частоты и бюджета применяются только к режимам с авто‑внедрением (`hybrid` и `context`).
## Наблюдение (Directional vs. Unified)

Honcho моделирует разговор как обмен сообщениями между peers. У каждого peer есть два переключателя наблюдения, которые 1:1 соответствуют `SessionPeerConfig` Honcho:

| Переключатель | Эффект |
|--------|--------|
| `observeMe` | Honcho строит представление этого peer из его собственных сообщений |
| `observeOthers` | Этот peer наблюдает сообщения другого peer (подаёт данные для кросс‑peer рассуждения) |

Два peer × два переключателя = четыре флага. `observationMode` — это сокращённый набор предустановок:

| Предустановка | Флаги пользователя | Флаги ИИ | Семантика |
|--------|-----------|----------|-----------|
| `"directional"` (по умолчанию) | me: on, others: on | me: on, others: on | Полное взаимное наблюдение. Включает кросс‑peer диалектику — «что ИИ знает о пользователе, исходя из того, что сказал пользователь и как ответил ИИ». |
| `"unified"` | me: on, others: off | me: off, others: on | Семантика общего пула — ИИ наблюдает только сообщения пользователя, пользователь‑peer только самомоделирует себя. Один наблюдатель пула. |

Переопределить предустановку можно явным блоком `observation` для управления каждым peer:

```json
"observation": {
  "user": { "observeMe": true,  "observeOthers": true },
  "ai":   { "observeMe": true,  "observeOthers": false }
}
```

Типичные шаблоны:

| Намерение | Конфигурация |
|--------|--------|
| Полное наблюдение (большинство пользователей) | `"observationMode": "directional"` |
| ИИ не должен пере‑моделировать пользователя по своим ответам | `"ai": {"observeMe": true, "observeOthers": false}` |
| Сильная персона: ИИ‑peer не должен обновлять себя по самонаблюдению | `"ai": {"observeMe": false, "observeOthers": true}` |

Переключатели на стороне сервера, задаваемые через [Honcho dashboard](https://app.honcho.dev), имеют приоритет над локальными значениями по умолчанию — Hermes синхронно подтягивает их при инициализации сессии.
## Инструменты

Когда Honcho активен в качестве поставщика памяти, доступны пять инструментов:

| Инструмент | Назначение |
|------|---------|
| `honcho_profile` | Прочитать или обновить карточку пира — передать `card` (список фактов) для обновления, опустить для чтения |
| `honcho_search` | Семантический поиск по контексту — необработанные выдержки, без синтеза LLM |
| `honcho_context` | Полный контекст сессии — сводка, представление, карточка, последние сообщения |
| `honcho_reasoning` | Синтезированный ответ от LLM Honcho — передать `reasoning_level` (minimal/low/medium/high/max) для управления глубиной |
| `honcho_conclude` | Создать или удалить выводы — передать `conclusion` для создания, `delete_id` для удаления (только персональные данные) |
## Команды CLI

Подкоманда `hermes honcho` **регистрируется только когда Honcho является активным провайдером памяти** (`memory.provider: honcho` в `config.yaml`). Запусти `hermes memory setup` и выбери Honcho первым; подкоманда появится при следующем запуске.

```bash
hermes honcho status          # Connection status, config, and key settings
hermes honcho setup           # Redirects to `hermes memory setup`
hermes honcho strategy        # Show or set session strategy (per-session/per-directory/per-repo/global)
hermes honcho peer            # Show or update peer names + dialectic reasoning level
hermes honcho mode            # Show or set recall mode (hybrid/context/tools)
hermes honcho tokens          # Show or set token budget for context and dialectic
hermes honcho identity        # Seed or show the AI peer's Honcho identity
hermes honcho sync            # Sync Honcho config to all existing profiles
hermes honcho peers           # Show peer identities across all profiles
hermes honcho sessions        # List known Honcho session mappings
hermes honcho map             # Map current directory to a Honcho session name
hermes honcho enable          # Enable Honcho for the active profile
hermes honcho disable         # Disable Honcho for the active profile
hermes honcho migrate         # Step-by-step migration guide from openclaw-honcho
```
## Переход с `hermes honcat`

Если ты ранее использовал автономный `hermes honcat setup`:

1. Твоя существующая конфигурация (`honcat.json` или `~/.honcat/config.json`) сохраняется
2. Твои серверные данные (память, заключения, пользовательские профили) остаются нетронутыми
3. Установи `memory.provider: honcat` в `config.yaml`, чтобы повторно активировать

Повторный вход в систему или повторная настройка не требуются. Запусти `hermes memory setup` и выбери «honcat» — мастер обнаружит твою существующую конфигурацию.
