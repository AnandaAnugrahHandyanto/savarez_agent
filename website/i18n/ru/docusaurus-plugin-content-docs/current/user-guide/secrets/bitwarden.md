# Bitwarden Secrets Manager

Получайте API‑ключи из [Bitwarden Secrets Manager](https://bitwarden.com/products/secrets-manager/) при запуске процесса вместо того, чтобы хранить их в открытом виде в `~/.hermes/.env`. Один bootstrap‑секрет (токен доступа машинного аккаунта) заменяет N ключей провайдеров, а ротация учётных данных становится единственным изменением в веб‑приложении Bitwarden.

## Как это работает

1. Создаёшь **machine account** в Bitwarden Secrets Manager, даёшь ему права чтения проекта и генерируешь **access token**.
2. Hermes сохраняет этот единственный токен в `~/.hermes/.env` как `BWS_ACCESS_TOKEN`.
3. Каждый раз, когда запускается `hermes` (или gateway, или cron‑задача), после загрузки `~/.hermes/.env` Hermes вызывает `bws secret list <project_id>` и помещает полученные ключи в `os.environ`.
4. По умолчанию Hermes **перезаписывает** значения, уже находящиеся в окружении, поэтому Bitwarden становится единственным источником правды — ротация ключа в веб‑приложении сразу отражается в каждом процессе Hermes при следующем запуске. Отключи перезапись, установив `override_existing: false` в конфигурации, если хочешь, чтобы приоритет имел `.env`.

Бинарный файл `bws` автоматически скачивается в `~/.hermes/bin/` при первом использовании — без `apt`, без `brew`, без `sudo`.

## Почему машинные аккаунты (и почему без запроса 2FA)

Bitwarden Secrets Manager предназначен для неинтерактивных нагрузок: машинные аккаунты не могут быть защищены 2FA, потому что в процессе нет человека. Токен доступа — это учётные данные. Любой, кто обладает токеном, может читать все секреты, к которым имеет доступ машинный аккаунт, поэтому относись к нему как к ценному bearer‑токену — храни его в `.env` (а не в `config.yaml`) и при утечке отзывай и генерируй новый токен в веб‑приложении Bitwarden.

Машинный аккаунт настраивается *в веб‑приложении*, где уже действует обычный 2FA. После этого токен работает автономно.

## Настройка

### 1. Создай машинный аккаунт и токен доступа

В [Bitwarden web app](https://vault.bitwarden.com) (или [vault.bitwarden.eu](https://vault.bitwarden.eu) для EU‑аккаунтов):

1. Переключись на **Secrets Manager** в переключателе продуктов.
2. Создай или выбери **Project** (например, «Hermes keys»).
3. Добавь ключи провайдеров как секреты. **Name** секрета становится именем переменной окружения — используй `OPENROUTER_API_KEY`, `ANTHROPIC_API_KEY` и т.д.
4. **Machine accounts → New machine account → My Hermes machine → Projects** — выдай права **Read** для своего проекта.
5. На вкладке **Access tokens** нажми **Create access token** → установи **Never** expires (или выбери дату) → скопируй токен (он начинается с `0.`). Bitwarden больше не сможет его показать — сохрани копию.

Secrets Manager включён в бесплатный тариф Bitwarden с ограничениями; платный план не требуется для пробования.

### 2. Запусти мастер‑настройку

```bash
hermes secrets bitwarden setup
```

Он выполнит:

1. Скачает и проверит `bws v2.0.0` в `~/.hermes/bin/bws`.
2. Попросит ввести токен доступа (ввод скрыт). Сохранит его в `~/.hermes/.env` как `BWS_ACCESS_TOKEN`.
3. Спросит, к какому региону Bitwarden относится твой машинный аккаунт — **US Cloud**, **EU Cloud** или **self-hosted / custom URL**. Сохранит в `config.yaml` как `secrets.bitwarden.server_url` и передаст `bws` как `BWS_SERVER_URL`.
4. Выведет список проектов, доступных машинному аккаунту; выбери один. Сохранит в `config.yaml` как `secrets.bitwarden.project_id`.
5. Протестирует получение секретов проекта и покажет, какие переменные окружения будут заданы.
6. Установит `secrets.bitwarden.enabled: true`.

Поддерживается также неинтерактивная настройка через флаги:

```bash
hermes secrets bitwarden setup \
  --access-token "$BWS_ACCESS_TOKEN" \
  --server-url https://vault.bitwarden.eu \
  --project-id <project-uuid>
```

### 3. Подтверди

```bash
hermes secrets bitwarden status
```

С этого момента каждый вызов `hermes` будет вытягивать свежие секреты при старте. При первом применении секретов в процессе ты увидишь однострочное резюме в `stderr`.

## CLI

| Command | Что делает |
|---|---|
| `hermes secrets bitwarden setup` | Интерактивный мастер (установка бинарника, запрос токена, выбор проекта, тестовый fetch) |
| `hermes secrets bitwarden status` | Показать конфиг + версию бинарника + наличие токена |
| `hermes secrets bitwarden sync` | Dry‑run: сейчас вытянуть секреты и показать, что бы было применено |
| `hermes secrets bitwarden sync --apply` | Вытянуть и экспортировать в текущую оболочку |
| `hermes secrets bitwarden install` | Просто скачать зафиксированный бинарник `bws` (без аутентификации) |
| `hermes secrets bitwarden disable` | Установить `enabled: false`; токен и ID проекта остаются на месте |

## Конфигурация

Значения по умолчанию в `~/.hermes/config.yaml`:

```yaml
secrets:
  bitwarden:
    enabled: false
    access_token_env: BWS_ACCESS_TOKEN
    project_id: ""
    server_url: ""
    cache_ttl_seconds: 300
    override_existing: true
    auto_install: true
```

| Key | Default | Что делает |
|---|---|---|
| `enabled` | `false` | Главный переключатель. При `false` Bitwarden никогда не вызывается. |
| `access_token_env` | `BWS_ACCESS_TOKEN` | Имя переменной окружения, в которой хранится bootstrap‑токен. Измени, если уже используешь `BWS_ACCESS_TOKEN` для чего‑то другого. |
| `project_id` | `""` | UUID проекта, из которого синхронизировать. |
| `server_url` | `""` | Регион Bitwarden или собственный endpoint. Пусто = значение по умолчанию `bws` (US Cloud, `https://vault.bitwarden.com`). Установи `https://vault.bitwarden.eu` для EU Cloud или свой URL для self‑hosted. Передаётся в подпроцесс `bws` как `BWS_SERVER_URL`. |
| `cache_ttl_seconds` | `300` | Как долго результат fetch‑а переиспользуется в процессе. Установи `0`, чтобы отключить кэш. Кэш привязан к процессу; новые вызовы `hermes` стартуют с чистого состояния. |
| `override_existing` | `true` | При `true` значения из Bitwarden перезаписывают всё, что уже есть в окружении (поэтому ротация в веб‑приложении действительно срабатывает). Поставь `false`, если хочешь, чтобы приоритет имел `.env` / экспорты оболочки. |
| `auto_install` | `true` | При `true` `bws` автоматически скачивается в `~/.hermes/bin/` при первом использовании. |

## Сценарии отказов

Bitwarden никогда не блокирует запуск Hermes. Если что‑то идёт не так, ты увидишь однострочное предупреждение в `stderr`, а Hermes продолжит работу с теми учётными данными, которые уже есть в `.env`:

| Symptom | Cause | Fix |
|---|---|---|
| `BWS_ACCESS_TOKEN is not set` | В конфиге включено, но токен удалён из `.env` | Перезапусти `hermes secrets bitwarden setup` |
| `bws exited 1: invalid access token` | Токен отозван или неверный | Сгенерируй новый токен, повторно запусти настройку |
| `[400 Bad Request] {"error":"invalid_client"}` | Токен относится к региону Bitwarden, отличному от того, к которому обращается `bws` (например, EU‑токен к US‑endpoint) | Перезапусти настройку и выбери правильный регион, либо установи `secrets.bitwarden.server_url` в `https://vault.bitwarden.eu` (или свой URL) |
| `bws timed out` | Сеть заблокирована или API Bitwarden медленно отвечает | Проверь соединение с `api.bitwarden.com` (или своим `server_url`) |
| `bws binary not available` | `auto_install: false` и `bws` не в PATH | Установи вручную из [github.com/bitwarden/sdk-sm/releases](https://github.com/bitwarden/sdk-sm/releases) или включи `auto_install` |
| `Checksum mismatch` | Скачанный файл повреждён или подменён | Повтори попытку; при постоянных проблемах открой issue |

## Замечания по безопасности

- Bootstrap‑токен (`BWS_ACCESS_TOKEN`) сам по себе чувствителен — любой, кто его получит, сможет читать все секреты машинного аккаунта. Обращайся с ним как с любым другим API‑ключом.
- Hermes не позволит Bitwarden перезаписать сам bootstrap‑токен, даже если `override_existing: true`. Если ты сохранил `BWS_ACCESS_TOKEN` как секрет внутри проекта, он будет тихо пропущен при применении.
- Скачивание бинарника `bws` проверяется по опубликованной SHA‑256 контрольной сумме из того же релиза GitHub. Несоответствие прерывает установку.
- Зафиксированная версия (`bws v2.0.0` на момент написания) обновляется через PR в этом репозитории — Hermes не обновляет `bws` автоматически до «latest», потому что upstream‑релизы могут менять совместимость.

## Когда НЕ стоит использовать

- **Одномашинные персональные установки**, где достаточно `~/.hermes/.env`. Ты просто меняешь одну учётную запись на другую и добавляешь сетевую зависимость при старте.
- **Изолированные (air‑gapped) среды**, которые не могут достучаться до `api.bitwarden.com`.
- **CI/CD**, где уже настроен механизм внедрения секретов (GitHub Actions secrets, Vault и т.п.) — выбирай один путь, а не два.

Хороший сценарий применения — флот из нескольких машин, общие dev‑коробки, VPS‑шлюзы или любая инфраструктура, где нужен централизованный процесс ротации и отзыва учётных данных для множества установок Hermes.
