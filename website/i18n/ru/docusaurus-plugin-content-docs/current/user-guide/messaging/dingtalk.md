# Настройка DingTalk

Hermes Agent интегрируется с DingTalk (钉钉) в виде чат‑бота, позволяя общаться с твоим AI‑ассистентом через личные сообщения или групповые чаты. Бот подключается через **Stream Mode** DingTalk — долгоживое WebSocket‑соединение, не требующее публичного URL или сервера веб‑хуков — и отвечает, отправляя сообщения в формате markdown через **session webhook API** DingTalk.

Перед настройкой вот та часть, которая интересует большинство людей: как Hermes ведёт себя, когда находится в твоём рабочем пространстве DingTalk.
## Как себя ведёт Hermes

| Context | Behavior |
|---------|----------|
| **DMs (1:1 chat)** | Hermes отвечает на каждое сообщение. `@mention` не требуется. Каждый личный чат имеет свою сессию. |
| **Group chats** | Hermes отвечает, когда ты `@mention` его. Без упоминания Hermes игнорирует сообщение. |
| **Shared groups with multiple users** | По умолчанию Hermes изолирует историю сессий для каждого пользователя внутри группы. Два человека, общающиеся в одной группе, не используют одну общую транскрипцию, если ты явно не отключишь это. |

### Модель сессий в DingTalk

По умолчанию:

- каждый личный чат получает свою сессию
- каждый пользователь в общем групповом чате получает свою сессию внутри этой группы

Это контролируется файлом `config.yaml`:

```yaml
group_sessions_per_user: true
```

Установи значение `false` только если ты явно хочешь одну общую беседу для всей группы:

```yaml
group_sessions_per_user: false
```

Это руководство проведёт тебя через весь процесс настройки — от создания бота DingTalk до отправки первого сообщения.
## Требования

Установи необходимые пакеты Python:

```bash
pip install "hermes-agent[dingtalk]"
```

Или по отдельности:

```bash
pip install dingtalk-stream httpx alibabacloud-dingtalk
```

- `dingtalk-stream` — официальный SDK DingTalk для режима Stream (на основе WebSocket)
- `httpx` — асинхронный HTTP‑клиент, используемый для отправки ответов через веб‑хуки сессии
- `alibabacloud-dingtalk` — SDK DingTalk OpenAPI для AI‑карт, реакций‑emoji и загрузки медиа
## Шаг 1: Создать приложение DingTalk

1. Перейди в [DingTalk Developer Console](https://open-dev.dingtalk.com/).
2. Войди в систему своей учётной записью администратора DingTalk.
3. Нажми **Application Development** → **Custom Apps** → **Create App via H5 Micro-App** (или **Robot**, в зависимости от версии консоли).
4. Заполни:
   - **App Name**: например `Hermes Agent`
   - **Description**: по желанию
5. После создания перейди в **Credentials & Basic Info**, чтобы найти свой **Client ID** (AppKey) и **Client Secret** (AppSecret). Скопируй оба значения.

:::warning[Credentials shown only once]
**Client Secret** отображается только один раз при создании приложения. Если ты его потеряешь, потребуется сгенерировать новый. Никогда не публикуй эти учётные данные и не добавляй их в репозиторий Git.
:::
## Шаг 2: Включить возможность **Robot**

1. На странице настроек твоего приложения перейди к **Add Capability** → **Robot**.
2. Включи возможность **Robot**.
3. В разделе **Message Reception Mode** выбери **Stream Mode** (рекомендовано — публичный URL не требуется).

:::tip
**Stream Mode** — рекомендованная конфигурация. Она использует длительное соединение WebSocket, инициируемое с твоего компьютера, поэтому тебе не нужен публичный IP, доменное имя или endpoint веб‑хука. Это работает за NAT, брандмауэрами и на локальных машинах.
:::
## Шаг 3: Найти свой DingTalk User ID

Hermes Agent использует твой DingTalk User ID для контроля того, кто может взаимодействовать с ботом. DingTalk User ID — это буквенно‑цифровые строки, задаваемые администратором твоей организации.

Чтобы найти свой ID:

1. Спроси у администратора DingTalk в твоей организации — User ID настраиваются в консоли администрирования DingTalk в разделе **Contacts** → **Members**.
2. Либо бот записывает `sender_id` для каждого входящего сообщения. Запусти gateway, отправь боту сообщение и проверь логи, чтобы увидеть свой ID.
## Шаг 4: Настройка Hermes Agent

### Вариант A: Интерактивная установка (рекомендовано)

Запусти команду настройки с подсказками:

```bash
hermes gateway setup
```

Выбери **DingTalk** при запросе. Мастер установки может авторизоваться одним из двух способов:

- **QR‑code device flow (рекомендовано).** Отсканируй QR‑код, выводимый в терминале, приложением DingTalk mobile — твой Client ID и Client Secret будут возвращены автоматически и записаны в `~/.hermes/.env`. По консоли разработчика ходить не нужно.
- **Ручная вставка.** Если у тебя уже есть учётные данные (или сканирование QR неудобно), вставь свой Client ID, Client Secret и разрешённые ID пользователей, когда будет запрос.

:::note openClaw branding disclosure
Поскольку `verification_uri_complete` DingTalk жёстко привязан к идентификации **openClaw** на уровне API, QR‑код сейчас авторизует под строкой источника `openClaw`, пока Alibaba / DingTalk‑Real‑AI не зарегистрирует серверный шаблон, специфичный для Hermes. Это просто способ, которым DingTalk отображает экран согласия — бот, который ты создаёшь, полностью твой и приватный для твоего **тенанта**.
:::

### Вариант B: Ручная конфигурация

Добавь следующее в файл `~/.hermes/.env`:

```bash
# Required
DINGTALK_CLIENT_ID=your-app-key
DINGTALK_CLIENT_SECRET=your-app-secret

# Security: restrict who can interact with the bot
DINGTALK_ALLOWED_USERS=user-id-1

# Multiple allowed users (comma-separated)
# DINGTALK_ALLOWED_USERS=user-id-1,user-id-2

# Optional: group-chat gating (mirrors Slack/Telegram/Discord/WhatsApp)
# DINGTALK_REQUIRE_MENTION=true
# DINGTALK_FREE_RESPONSE_CHATS=cidABC==,cidDEF==
# DINGTALK_MENTION_PATTERNS=^小马
# DINGTALK_HOME_CHANNEL=cidXXXX==
# DINGTALK_ALLOW_ALL_USERS=true
```

Опциональные настройки поведения в `~/.hermes/config.yaml`:

```yaml
group_sessions_per_user: true

gateway:
  platforms:
    dingtalk:
      extra:
        # Require @mention in groups before the bot replies (parity with Slack/Telegram/Discord).
        # DMs ignore this — the bot always replies in 1:1 chats.
        require_mention: true

        # Per-platform allowlist. When set, only these DingTalk user IDs can interact with the bot
        # (same semantics as DINGTALK_ALLOWED_USERS, but scoped here instead of in .env).
        allowed_users:
          - user-id-1
          - user-id-2
```

- `group_sessions_per_user: true` сохраняет изолированный контекст каждого участника внутри общих групповых чатов
- `require_mention: true` предотвращает ответ бота на каждое сообщение в группе — он отвечает только при упоминании `@`
- `allowed_users` в `dingtalk.extra` является альтернативой `DINGTALK_ALLOWED_USERS`; если заданы оба, они объединяются

### Запуск шлюза

После настройки запусти шлюз DingTalk:

```bash
hermes gateway
```

Бот должен подключиться к Stream Mode DingTalk в течение нескольких секунд. Отправь ему сообщение — в личных сообщениях или в группе, куда его добавили — чтобы проверить.

:::tip
Ты можешь запустить `hermes gateway` в фоне или как службу systemd для постоянной работы. См. документацию по развертыванию для деталей.
:::
## Возможности

### AI‑карты

Hermes может отвечать, используя AI‑карты DingTalk вместо обычных markdown‑сообщений. Карты предоставляют более богатое, структурированное отображение и поддерживают потоковые обновления по мере генерации ответа агентом.

Чтобы включить AI‑карты, укажи идентификатор шаблона карты в `config.yaml`:

```yaml
platforms:
  dingtalk:
    enabled: true
    extra:
      card_template_id: "your-card-template-id"
```

Идентификатор шаблона карты можно найти в консоли разработчика DingTalk в разделе настроек AI‑карт твоего приложения. Когда AI‑карты включены, все ответы отправляются как карты с потоковыми обновлениями текста.

### Реакции‑эмодзи

Hermes автоматически добавляет эмодзи‑реакции к твоим сообщениям, чтобы показать статус обработки:

- 🤔Thinking — добавляется, когда бот начинает обрабатывать твоё сообщение
- 🥳Done — добавляется, когда ответ завершён (заменяет реакцию Thinking)

Эти реакции работают как в личных сообщениях, так и в групповых чатах.

### Настройки отображения

Ты можешь настроить поведение отображения DingTalk независимо от других платформ:

```yaml
display:
  platforms:
    dingtalk:
      show_reasoning: false   # Show model reasoning/thinking in replies
      streaming: true         # Enable streaming responses (works with AI Cards)
      tool_progress: all      # Show tool execution progress (all/new/off)
      interim_assistant_messages: true  # Show intermediate commentary messages
```

Чтобы отключить прогресс инструмента и промежуточные сообщения для более чистого опыта:

```yaml
display:
  platforms:
    dingtalk:
      tool_progress: off
      interim_assistant_messages: false
```
## Устранение неполадок

### Бот не отвечает на сообщения

**Причина**: Возможность бота не включена, либо `DINGTALK_ALLOWED_USERS` не содержит твой идентификатор пользователя.

**Решение**: Убедись, что возможность бота включена в настройках приложения и выбран режим **Stream Mode**. Проверь, что твой идентификатор пользователя присутствует в `DINGTALK_ALLOWED_USERS`. Перезапусти шлюз.

### Ошибка «dingtalk-stream not installed»

**Причина**: Пакет Python `dingtalk-stream` не установлен.

**Решение**: Установи его:

```bash
pip install dingtalk-stream httpx
```

### Требуются «DINGTALK_CLIENT_ID и DINGTALK_CLIENT_SECRET»

**Причина**: Учётные данные не заданы в окружении или в файле `.env`.

**Решение**: Убедись, что `DINGTALK_CLIENT_ID` и `DINGTALK_CLIENT_SECRET` правильно заданы в `~/.hermes/.env`. Идентификатор клиента — это твой **AppKey**, а секрет клиента — твой **AppSecret** из консоли разработчика DingTalk.

### Отключения потока / циклы переподключения

**Причина**: Нестабильность сети, обслуживание платформы DingTalk или проблемы с учётными данными.

**Решение**: Адаптер автоматически переподключается с экспоненциальным увеличением задержки (2 с → 5 с → 10 с → 30 с → 60 с). Проверь, что твои учётные данные действительны и приложение не деактивировано. Убедись, что сеть позволяет исходящие соединения WebSocket.

### Бот офлайн

**Причина**: Шлюз Hermes не запущен или не смог подключиться.

**Решение**: Проверь, что запущен `hermes gateway`. Посмотри вывод терминала на предмет сообщений об ошибках. Частые причины: неверные учётные данные, деактивированное приложение, отсутствие установленных `dingtalk-stream` или `httpx`.

### «No session_webhook available»

**Причина**: Бот попытался ответить, но у него нет URL веб‑хука сессии. Обычно это происходит, когда веб‑хук истёк или бот был перезапущен между получением сообщения и отправкой ответа.

**Решение**: Отправь боту новое сообщение — каждое входящее сообщение предоставляет свежий веб‑хук сессии для ответов. Это нормальное ограничение DingTalk; бот может отвечать только на сообщения, полученные недавно.
## Безопасность

:::warning
Всегда устанавливай `DINGTALK_ALLOWED_USERS`, чтобы ограничить, кто может взаимодействовать с ботом. Если переменная не задана, gateway по умолчанию отклоняет всех пользователей как меру предосторожности. Добавляй только идентификаторы пользователей, которым ты доверяешь — авторизованные пользователи получают полный доступ к возможностям агента, включая использование инструментов и доступ к системе.
:::

Для получения дополнительной информации о защите развертывания Hermes Agent смотри [Руководство по безопасности](../security.md).
## Примечания

- **Stream Mode**: Не требуется публичный URL, доменное имя или сервер webhook. Соединение инициируется с твоего компьютера через WebSocket, поэтому работает за NAT и брандмауэрами.
- **AI Cards**: При желании отвечай с помощью богатых AI Cards вместо обычного markdown. Настраивается через `card_template_id`.
- **Emoji Reactions**: Автоматические реакции 🤔Thinking/🥳Done для статуса обработки.
- **Markdown responses**: Ответы формируются в markdown‑формате DingTalk для отображения форматированного текста.
- **Media support**: Изображения и файлы во входящих сообщениях автоматически разрешаются и могут обрабатываться инструментами компьютерного зрения.
- **Message deduplication**: Адаптер удаляет дублирующие сообщения в течение 5‑минутного окна, чтобы предотвратить двойную обработку.
- **Auto-reconnection**: Если соединение потока прерывается, адаптер автоматически переподключается с экспоненциальным увеличением интервала.
- **Message length limit**: Ответы ограничены 20 000 символов на сообщение. Более длинные ответы обрезаются.
