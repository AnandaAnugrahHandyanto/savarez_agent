# Настройка Feishu / Lark

Hermes Agent интегрируется с Feishu и Lark как полнофункциональный бот. После подключения ты можешь общаться с агентом в личных сообщениях или групповых чатах, получать результаты cron‑задач в домашний чат и отправлять текст, изображения, аудио и вложения файлов через обычный поток шлюза.

Интеграция поддерживает оба режима подключения:

- `websocket` — рекомендуется; Hermes открывает исходящее соединение, и тебе не нужен публичный webhook‑endpoint
- `webhook` — полезно, когда ты хочешь, чтобы Feishu/Lark отправлял события в твой шлюз по HTTP
## Как себя ведёт Hermes

| Context | Behavior |
|---------|----------|
| Direct messages | Hermes отвечает на каждое сообщение. |
| Group chats | Hermes отвечает только когда бот упомянут (@mentioned) в чате. |
| Shared group chats | По умолчанию история сессии изолирована для каждого пользователя в общем чате. |

Это поведение в общем чате управляется файлом `config.yaml`:

```yaml
group_sessions_per_user: true
```

Установи значение `false` только если действительно хочешь одну общую беседу на каждый чат.
## Шаг 1: Создать приложение Feoursquare / Lark

### Рекомендуется: Scan-to-Create (одна команда)

```bash
hermes gateway setup
```

Выбери **Feishu / Lark** и отсканируй QR‑код в мобильном приложении Feishu или Lark. Hermes автоматически создаст бот‑приложение с нужными правами и сохранит учётные данные.

### Альтернатива: Ручная настройка

Если Scan-to-Create недоступен, мастер переходит к ручному вводу:

1. Открой консоль разработчика Feishu или Lark:
   - Feishu: [https://open.feishu.cn/](https://open.feishu.cn/)
   - Lark: [https://open.larksuite.com/](https://open.larksuite.com/)
2. Создай новое приложение.
3. В **Credentials & Basic Info** скопируй **App ID** и **App Secret**.
4. Включи возможность **Bot** для приложения.
5. Выполни `hermes gateway setup`, выбери **Feishu / Lark** и введи учётные данные, когда будет предложено.

:::warning
Храни App Secret в секрете. Любой, у кого он есть, может выдавать себя за твоё приложение.
:::

### Настройка прав доступа

В консоли разработчика Feishu перейди в **Permission Management** и добавь следующие области. Их можно импортировать пакетно на странице прав.

**Обязательные права:**

| Scope | Purpose |
|-------|---------|
| `im:message` | Приём и чтение сообщений |
| `im:message:send_as_bot` | Отправка сообщений от имени бота |
| `im:resource` | Доступ к изображениям, файлам и аудио, отправленным пользователями |
| `im:chat` | Доступ к метаданным чатов/групп |
| `im:chat:readonly` | Чтение списка чатов и их участников |

**Рекомендуемые права (для полной функциональности):**

| Scope | Purpose |
|-------|---------|
| `im:message.reactions:readonly` | Приём событий реакций‑эмодзи |
| `admin:app.info:readonly` | Автоопределение идентичности бота для ограничения @упоминаний |
| `contact:user.id:readonly` | Разрешение идентификаторов пользователей для сопоставления с allowlist |

### Настройка событий

В **Events and Callbacks**:

1. Установи режим соединения **Long Connection (WebSocket)** (рекомендовано) или укажи URL веб‑хука.
2. В разделе **Event Configuration** подпишись на:
   - `im.message.receive_v1` — необходимо для получения сообщений

### Публикация приложения

После настройки прав и событий перейди в **Version Management** и опубликуй новую версию приложения. Права не вступят в силу, пока версия не будет опубликована и одобрена (для корпоративных приложений может потребоваться одобрение администратора).
## Шаг 2: Выбери режим подключения

### Рекомендуется: режим WebSocket

Используй режим WebSocket, когда Hermes работает на твоём ноутбуке, рабочей станции или частном сервере. Публичный URL не требуется. Официальный Lark SDK открывает и поддерживает постоянное исходящее WebSocket‑соединение с автоматическим переподключением.

```bash
FEISHU_CONNECTION_MODE=websocket
```

**Требования:** Пакет Python `websockets` должен быть установлен. SDK управляет жизненным циклом соединения, heartbeat‑ами и автоматическим переподключением изнутри.

**Как это работает:** Адаптер запускает WebSocket‑клиент Lark SDK в фоновом потоке‑исполнителе. Входящие события (сообщения, реакции, действия карточек) передаются в основной цикл asyncio. При разрыве соединения SDK попытается автоматически переподключиться.

### Опционально: режим Webhook

Используй режим webhook только тогда, когда Hermes уже запущен за доступной HTTP‑точкой.

```bash
FEISHU_CONNECTION_MODE=webhook
```

В режиме webhook Hermes запускает HTTP‑сервер (через `aiohttp`) и обслуживает endpoint Feishu по адресу:

```text
/feishu/webhook
```

**Требования:** Пакет Python `aiohttp` должен быть установлен.

Ты можешь настроить адрес привязки и путь сервера webhook:

```bash
FEISHU_WEBHOOK_HOST=127.0.0.1   # default: 127.0.0.1
FEISHU_WEBHOOK_PORT=8765         # default: 8765
FEISHU_WEBHOOK_PATH=/feishu/webhook  # default: /feishu/webhook
```

Когда Feishu отправляет запрос проверки URL (`type: url_verification`), webhook отвечает автоматически, позволяя завершить настройку подписки в консоли разработчика Feishu. Ответ на проверку проверяется по `FEISHU_VERIFICATION_TOKEN`, если он установлен — запросы проверки без токена или с неверным токеном отклоняются, чтобы неавторизованный клиент не мог подтвердить контроль над конечной точкой, отсылая атакующему управляемые данные проверки.
## Шаг 3: Настройка Hermes

### Вариант A: Интерактивная настройка

```bash
hermes gateway setup
```

Выбери **Feishu / Lark** и введи данные в появившиеся запросы.

### Вариант B: Ручная конфигурация

Добавь следующее в `~/.hermes/.env`:

```bash
FEISHU_APP_ID=cli_xxx
FEISHU_APP_SECRET=secret_xxx
FEISHU_DOMAIN=feishu
FEISHU_CONNECTION_MODE=websocket

# Optional but strongly recommended
FEISHU_ALLOWED_USERS=ou_xxx,ou_yyy
FEISHU_HOME_CHANNEL=oc_xxx
```

`FEISHU_DOMAIN` принимает:

- `feishu` — для Feishu (Китай)
- `lark` — для Lark (международный)
## Шаг 4: Запуск шлюза

```bash
hermes gateway
```

Затем отправь сообщение боту из Feishu/Lark, чтобы убедиться, что соединение работает.
## Домашний чат

Используй `/set-home` в чате Feishu/Lark, чтобы пометить его как домашний канал для результатов cron‑задач и кроссплатформенных уведомлений.

Ты также можешь предварительно настроить его:

```bash
FEISHU_HOME_CHANNEL=oc_xxx
```
## Безопасность

### Allowlist пользователей

Для продакшн‑использования задайте allowlist Feishu Open ID:

```bash
FEISHU_ALLOWED_USERS=ou_xxx,ou_yyy
```

Если оставить allowlist пустым, любой, кто может достучаться до бота, сможет его использовать. В групповых чатах allowlist проверяется по `open_id` отправителя перед обработкой сообщения.

### Ключ шифрования веб‑хука

При работе в режиме веб‑хука задайте ключ шифрования, чтобы включить проверку подписи входящих payload'ов веб‑хука:

```bash
FEISHU_ENCRYPT_KEY=your-encrypt-key
```

Этот ключ находится в разделе **Event Subscriptions** конфигурации вашего Feishu‑приложения. При задании адаптер проверяет каждый запрос веб‑хука, используя алгоритм подписи:

```
SHA256(timestamp + nonce + encrypt_key + body)
```

Вычисленный хеш сравнивается с заголовком `x-lark-signature` с помощью сравнения, устойчивого к тайминговым атакам. Запросы с недействительными или отсутствующими подписями отклоняются с HTTP 401.

:::tip
В режиме WebSocket проверку подписи выполняет сам SDK, поэтому `FEISHU_ENCRYPT_KEY` является опциональным. В режиме веб‑хука настоятельно рекомендуется использовать его в продакшн‑среде.
:::

### Токен проверки

Дополнительный уровень аутентификации, который проверяет поле `token` внутри payload'ов веб‑хука:

```bash
FEISHU_VERIFICATION_TOKEN=your-verification-token
```

Этот токен также находится в разделе **Event Subscriptions** вашего Feishu‑приложения. При задании каждый входящий payload веб‑хука должен содержать соответствующий `token` в объекте `header`. Несоответствующие токены отклоняются с HTTP 401.

`FEISHU_ENCRYPT_KEY` и `FEISHU_VERIFICATION_TOKEN` могут использоваться совместно для защиты в глубину.
## Политика групповых сообщений

Переменная окружения `FEISHU_GROUP_POLICY` управляет тем, отвечает ли Hermes в групповых чатах и как:

```bash
FEISHU_GROUP_POLICY=allowlist   # default
```

| Value | Behavior |
|-------|----------|
| `open` | Hermes отвечает на @упоминания от любого пользователя в любой группе. |
| `allowlist` | Hermes отвечает только на @упоминания от пользователей, перечисленных в `FEISHU_ALLOWED_USERS`. |
| `disabled` | Hermes полностью игнорирует все групповые сообщения. |

Во всех режимах бот должен быть явно @упомянут (или @all) в группе, чтобы сообщение было обработано. Прямые сообщения всегда обходят эту проверку.

Установи `FEISHU_REQUIRE_MENTION=false`, чтобы Hermes читал весь групповой трафик без необходимости @упоминания:

```bash
FEISHU_REQUIRE_MENTION=false
```

Для управления на уровне отдельного чата установи `require_mention` в записи `group_rules` — см. [Per-Group Access Control](#per-group-access-control) ниже.

### Идентификация бота

Hermes автоматически определяет `open_id` бота и отображаемое имя при запуске. Их нужно задавать вручную только тогда, когда автоматическое определение не может достичь API Feishu, или когда приложение использует tenant‑scoped идентификаторы пользователей:

```bash
FEISHU_BOT_OPEN_ID=ou_xxx     # only when auto-detection fails
FEISHU_BOT_USER_ID=xxx        # required if your app uses sender_id_type=user_id
FEISHU_BOT_NAME=MyBot         # only when auto-detection fails
```
## Обмен сообщениями между ботами

По умолчанию Hermes игнорирует сообщения, отправленные другими ботами. Включи обмен сообщениями между ботами, когда нужно, чтобы Hermes участвовал в оркестрации A2A или получал уведомления от других ботов в той же группе.

```bash
FEISHU_ALLOW_BOTS=mentions   # default: none
```

| Value | Behavior |
|-------|----------|
| `none` | Игнорировать все сообщения от других ботов (по умолчанию). |
| `mentions` | Принимать только когда бот‑партнёр упоминает Hermes через `@`. |
| `all` | Принимать каждое сообщение от бота‑партнёра. |

Также можно настроить через `feishu.allow_bots` в `config.yaml` (переменные окружения имеют приоритет, если оба заданы).

Боты‑партнёры не нужно добавлять в `FEISHU_ALLOWED_USERS` — этот список разрешённых отправителей применяется только к людям.

Предоставь область `application:bot.basic_info:read`, чтобы отображать имена ботов‑партнёров; без неё боты‑партнёры всё равно будут маршрутизироваться корректно, но будут отображаться их `open_id`.
## Интерактивные действия карточек

Когда пользователи нажимают кнопки или взаимодействуют с интерактивными карточками, отправленными ботом, адаптер маршрутизирует их как синтетические события команды `/card`:

- Нажатия кнопок превращаются в: `/card button {"key": "value", ...}`
- Полезная нагрузка `value` действия из определения карточки включается в виде JSON.
- Действия карточек дедуплицируются в течение 15‑минутного окна, чтобы предотвратить двойную обработку.

Обновления, инициируемые **gateway**, используют нативную Feishu‑карточку `Yes` / `No` вместо возврата к простым текстовым ответам. Когда `hermes update --gateway` требует подтверждения, адаптер записывает выбранный ответ в файл Hermes `.update_response` и заменяет карточку в строке на состояние «разрешено».

События действий карточек отправляются с `MessageType.COMMAND`, поэтому они проходят через обычный конвейер обработки команд.

Так же работает **одобрение команд** — когда агенту нужно выполнить опасную команду, он отправляет интерактивную карточку с кнопками **Allow Once** / **Session** / **Always** / **Deny**. Пользователь нажимает кнопку, и обратный вызов действия карточки передаёт решение об одобрении обратно агенту.

### Требуемая конфигурация приложения Feishu

Для интерактивных карточек необходимо выполнить **три** шага настройки в Feishu Developer Console. Отсутствие любого из них приводит к ошибке **200340** при нажатии пользователями кнопок карточки.

1. **Подписаться на событие действия карточки:**
   В **Event Subscriptions** добавьте `card.action.trigger` в список подписанных событий.

2. **Включить возможность Interactive Card:**
   В **App Features > Bot** убедитесь, что переключатель **Interactive Card** включён. Это сообщает Feishu, что ваше приложение может получать обратные вызовы действий карточек.

3. **Настроить Card Request URL (только режим webhook):**
   В **App Features > Bot > Message Card Request URL** укажите URL, совпадающий с конечной точкой вашего webhook‑а событий (например, `https://your-server:8765/feishu/webhook`). В режиме WebSocket это обрабатывается автоматически SDK.

:::warning
Без выполнения всех трёх шагов Feishu успешно *отправит* интерактивные карточки (отправка требует только разрешения `im:message:send`), но нажатие любой кнопки вернёт ошибку 200340. Карточка кажется работающей — ошибка проявляется только при взаимодействии пользователя с ней.
:::
## Интеллектуальный ответ на комментарий к документу

Помимо чата, адаптер может также отвечать на `@`‑упоминания в **документах Feishu/Lark**. Когда пользователь комментирует документ (выделяя локальный фрагмент текста или оставляя комментарий ко всему документу) и @‑упоминает бота, Hermes читает документ вместе с окружающей веткой комментариев и публикует ответ LLM непосредственно в этой ветке.

Работает на событии `drive.notice.comment_add_v1`; обработчик:

- Параллельно получает содержимое документа и историю комментариев (20 сообщений для веток ко всему документу, 12 — для веток локального выделения).
- Запускает агент с набором инструментов `feishu_doc` + `feishu_drive`, ограниченным этой единственной сессией комментария.
- Делит ответы на фрагменты по 4000 символов и публикует их как ответные сообщения в ветке.
- Кеширует сессии по документу в течение 1 часа с ограничением в 50 сообщений, чтобы последующие комментарии в том же документе сохраняли контекст.

### 3‑уровневый контроль доступа

Ответы на комментарии к документу доступны **только по явному разрешению** — режима «разрешить всем» нет. Права определяются в следующем порядке (первое совпадение выигрывает, по полям):

1. **Точный документ** — правило, привязанное к конкретному токену документа.
2. **Шаблон** — правило, совпадающее с шаблоном документов.
3. **Верхний уровень** — правило по умолчанию для рабочего пространства.

Для каждого правила доступны два типа политик:

- **`allowlist`** — статический список пользователей / арендаторов.
- **`pairing`** — статический список ∪ одобренный в режиме runtime хранилище. Полезно для развертываний, когда модераторы могут предоставлять доступ «на лету».

Правила хранятся в `~/.hermes/feishu_comment_rules.json` (pairing‑гранты в `~/.hermes/feishu_comment_pairing.json`) с кешированием по времени изменения файла — изменения вступают в силу при следующем событии комментария без перезапуска gateway.

CLI:

```bash
# Inspect current rules and pairing state
python -m gateway.platforms.feishu_comment_rules status

# Simulate an access check for a specific doc + user
python -m gateway.platforms.feishu_comment_rules check <fileType:fileToken> <user_open_id>

# Manage pairing grants at runtime
python -m gateway.platforms.feishu_comment_rules pairing list
python -m gateway.platforms.feishu_comment_rules pairing add <user_open_id>
python -m gateway.platforms.feishu_comment_rules pairing remove <user_open_id>
```

### Требуемая конфигурация приложения Feishu

Помимо уже предоставленных прав чата/карты, добавьте событие комментария к **диску**:

- Подпишитесь на `drive.notice.comment_add_v1` в **Event Subscriptions**.
- Предоставьте области `docs:doc:readonly` и `drive:drive:readonly`, чтобы обработчик мог читать содержимое документа.
## Поддержка медиа

### Входящие (получение)

Адаптер получает и кэширует следующие типы медиа от пользователей:

| Тип | Расширения | Как обрабатывается |
|------|-----------|-------------------|
| **Изображения** | .jpg, .jpeg, .png, .gif, .webp, .bmp | Загружаются через Feishu API и кэшируются локально |
| **Аудио** | .ogg, .mp3, .wav, .m4a, .aac, .flac, .opus, .webm | Загружаются и кэшируются; небольшие текстовые файлы автоматически извлекаются |
| **Видео** | .mp4, .mov, .avi, .mkv, .webm, .m4v, .3gp | Загружаются и кэшируются как документы |
| **Файлы** | .pdf, .doc, .docx, .xls, .xlsx, .ppt, .pptx и другие | Загружаются и кэшируются как документы |

Медиа из сообщений в формате **rich‑text** (post), включая встроенные изображения и вложения файлов, также извлекаются и кэшируются.

Для небольших текстовых документов (.txt, .md) содержимое файла автоматически вставляется в текст сообщения, чтобы агент мог прочитать его напрямую без использования инструментов.

### Исходящие (отправка)

| Метод | Что отправляется |
|--------|-----------------|
| `send` | Текстовые или **rich‑post** сообщения (автоматически определяется по содержимому markdown) |
| `send_image` / `send_image_file` | Загружает изображение в Feishu, затем отправляет как нативное изображение (с необязательной подписью) |
| `send_document` | Загружает файл в Feishu API, затем отправляет как вложение |
| `send_voice` | Загружает аудиофайл как вложение в Feishu |
| `send_video` | Загружает видео и отправляет как нативное медиа‑сообщение |
| `send_animation` | GIF‑файлы преобразуются во вложения (в Feishu нет нативного GIF‑сообщения) |

Маршрутизация загрузки файлов происходит автоматически в зависимости от расширения:

- `.ogg`, `.opus` → загружается как аудио `opus`
- `.mp4`, `.mov`, `.avi`, `.m4v` → загружается как медиа `mp4`
- `.pdf`, `.doc(x)`, `.xls(x)`, `.ppt(x)` → загружается с соответствующим типом документа
- Всё остальное → загружается как общий потоковый файл
## Отображение Markdown и запасной (fallback) вариант поста

Когда исходный текст содержит форматирование markdown (заголовки, жирный шрифт, списки, блоки кода, ссылки и т.д.), адаптер автоматически отправляет его как **post**‑сообщение Feishu со вложенным тегом `md`, а не как обычный текст. Это обеспечивает богатое отображение в клиенте Feishu.

Если API Feishu отклонит payload поста (например, из‑за неподдерживаемых конструкций markdown), адаптер автоматически переходит к отправке в виде простого текста без markdown. Этот двухэтапный запасной (fallback) вариант гарантирует доставку сообщений.

Сообщения простого текста (markdown не обнаружен) отправляются как тип сообщения `text`.
## Реакции статуса обработки

Пока агент работает, бот отображает реакцию `Typing` на твоём сообщении. Она удаляется, когда приходит ответ, или заменяется на `CrossMark`, если обработка завершилась ошибкой.

Установи `FEISHU_REACTIONS=false`, чтобы отключить её.
## Защита от всплесков и пакетирование

Адаптер включает подавление дребезга для быстрых всплесков сообщений, чтобы не перегружать агента:

### Пакетирование текста

Когда пользователь отправляет несколько текстовых сообщений подряд в быстром темпе, они объединяются в одно событие перед отправкой:

| Параметр | Переменная окружения | По умолчанию |
|---------|-----------------------|--------------|
| Период тишины | `HERMES_FEISHU_TEXT_BATCH_DELAY_SECONDS` | 0.6 s |
| Максимальное количество сообщений в пакете | `HERMES_FEISHU_TEXT_BATCH_MAX_MESSAGES` | 8 |
| Максимальное количество символов в пакете | `HERMES_FEISHU_TEXT_BATCH_MAX_CHARS` | 4000 |

### Пакетирование медиа

Несколько медиа‑вложений, отправленных подряд (например, перетаскивание нескольких изображений), объединяются в одно событие:

| Параметр | Переменная окружения | По умолчанию |
|---------|-----------------------|--------------|
| Период тишины | `HERMES_FEISHU_MEDIA_BATCH_DELAY_SECONDS` | 0.8 s |

### Сериализация по чатам

Сообщения в одном и том же чате обрабатываются последовательно (по одному), чтобы сохранять согласованность разговора. Каждый чат имеет собственную блокировку, поэтому сообщения в разных чатах обрабатываются одновременно.
## Ограничение скорости (режим Webhook)

В режиме webhook адаптер применяет ограничение скорости по IP, чтобы защититься от злоупотреблений:

- **Окно:** скользящее окно в 60 секунд
- **Лимит:** 120 запросов за окно для каждой тройки `(app_id, path, IP)`
- **Лимит отслеживаемых ключей:** до 4096 уникальных ключей (предотвращает неограниченный рост памяти)

Запросы, превышающие лимит, получают ответ `HTTP 429` (Too Many Requests).

### Отслеживание аномалий Webhook

Адаптер отслеживает последовательные ответы с ошибкой по каждому IP‑адресу. После 25 подряд ошибок от одного и того же IP в течение 6‑часового окна в журнал записывается предупреждение. Это помогает обнаружить неправильно настроенных клиентов или попытки сканирования.

Дополнительные защиты webhook:
- **Ограничение размера тела:** максимум 1 МБ
- **Таймаут чтения тела:** 30 секунд
- **Принудительная проверка Content-Type:** принимается только `application/json`
## Настройка WebSocket

При использовании режима `websocket` можно настроить поведение переподключения и ping‑ов:

```yaml
platforms:
  feishu:
    extra:
      ws_reconnect_interval: 120   # Seconds between reconnect attempts (default: 120)
      ws_ping_interval: 30         # Seconds between WebSocket pings (optional; SDK default if unset)
```

| Параметр | Ключ конфигурации | По умолчанию | Описание |
|---------|-------------------|--------------|----------|
| Интервал переподключения | `ws_reconnect_interval` | 120s | Как долго ждать между попытками повторного подключения |
| Интервал ping | `ws_ping_interval` | _(SDK default)_ | Частота keepalive‑ping‑ов WebSocket (значение по умолчанию SDK) |
## Управление доступом по группам

Помимо глобального параметра `FEISHU_GROUP_POLICY` можно задавать тонко настроенные правила для каждой групповой беседы, используя `group_rules` в файле `config.yaml`:

```yaml
platforms:
  feishu:
    extra:
      default_group_policy: "open"     # Default for groups not in group_rules
      admins:                          # Users who can manage bot settings
        - "ou_admin_open_id"
      group_rules:
        "oc_group_chat_id_1":
          policy: "allowlist"          # open | allowlist | blacklist | admin_only | disabled
          allowlist:
            - "ou_user_open_id_1"
            - "ou_user_open_id_2"
        "oc_group_chat_id_2":
          policy: "admin_only"
        "oc_group_chat_id_3":
          policy: "blacklist"
          blacklist:
            - "ou_blocked_user"
        "oc_free_chat":
          policy: "open"
          require_mention: false       # overrides FEISHU_REQUIRE_MENTION for this chat
```

| Политика | Описание |
|----------|----------|
| `open` | Любой участник группы может использовать бота |
| `allowlist` | Только пользователи, указанные в `allowlist` группы, могут использовать бота |
| `blacklist` | Все, кроме пользователей, указанных в `blacklist` группы, могут использовать бота |
| `admin_only` | Только пользователи из глобального списка `admins` могут использовать бота в этой группе |
| `disabled` | Бот игнорирует все сообщения в этой группе |

Установи `require_mention: false` в записи `group_rules`, чтобы отключить требование упоминания @ для конкретного чата. Если параметр опущен, чат наследует значение глобального параметра `FEISHU_REQUIRE_MENTION`.

Группы, не указанные в `group_rules`, используют политику `default_group_policy` (по умолчанию берётся значение `FEISHU_GROUP_POLICY`).

## Устранение неполадок

| Проблема | Решение |
|---------|-----|
| `lark-oapi not installed` | Установи SDK: `pip install lark-oapi` |
| `websockets not installed; websocket mode unavailable` | Установи websockets: `pip install websockets` |
| `aiohttp not installed; webhook mode unavailable` | Установи aiohttp: `pip install aiohttp` |
| `FEISHU_APP_ID or FEISHU_APP_SECRET not set` | Установи обе переменные окружения или настрой их через `hermes gateway setup` |
| `Another local Hermes gateway is already using this Feishu app_id` | Один экземпляр Hermes может использовать один app_id одновременно. Останови другой шлюз сначала. |
| Bot doesn't respond in groups | Убедись, что бот упомянут @, проверь `FEISHU_GROUP_POLICY` и убедись, что отправитель находится в `FEISHU_ALLOWED_USERS`, если политика — `allowlist`. |
| `Webhook rejected: invalid verification token` | Убедись, что `FEISHU_VERIFICATION_TOKEN` совпадает с токеном в конфигурации **Event Subscriptions** твоего Feishu‑приложения. |
| `Webhook rejected: invalid signature` | Убедись, что `FEISHU_ENCRYPT_KEY` совпадает с ключом шифрования в конфигурации Feishu‑приложения. |
| Post messages show as plain text | Feishu API отклонил полезную нагрузку сообщения; это нормальное поведение запасного варианта. Проверь логи для деталей. |
| Images/files not received by bot | Предоставь приложению Feishu права `im:message` и `im:resource`. |
| Bot identity not auto-detected | Обычно это временная проблема сети при обращении к эндпоинту информации о боте Feishu. Задай `FEISHU_BOT_OPEN_ID` и `FEISHU_BOT_NAME` вручную как обходной путь. |
| Peer bot messages still ignored after enabling `FEISHU_ALLOW_BOTS` | Hermes ещё не может определить себя — задай `FEISHU_BOT_OPEN_ID` (и `FEISHU_BOT_USER_ID`, если приложение использует `sender_id_type=user_id`). |
| Peer bots show as `ou_xxxxxx` instead of by name | Предоставь право `application:bot.basic_info:read`. |
| Error 200340 when clicking approval buttons | Включи возможность **Interactive Card** и настрой **Card Request URL** в консоли разработчика Feishu. См. [Required Feishu App Configuration](#required-feishu-app-configuration) выше. |
| `Webhook rate limit exceeded` | Более 120 запросов в минуту с одного IP. Обычно это ошибка конфигурации или цикл. |
