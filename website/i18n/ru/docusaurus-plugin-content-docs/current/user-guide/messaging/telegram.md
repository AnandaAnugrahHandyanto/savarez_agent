# Настройка Telegram

Hermes Agent интегрируется с Telegram как полнофункциональный разговорный бот. После подключения ты можешь общаться со своим агентом с любого устройства, отправлять голосовые заметки, которые автоматически транскрибируются, получать результаты запланированных задач и использовать агента в групповых чатах. Интеграция построена на основе [python-telegram-bot](https://python-telegram-bot.org/) и поддерживает текст, голос, изображения и вложения файлов.

## Шаг 1: Создай бота через BotFather

Каждому боту Telegram требуется API‑токен, выданный [@BotFather](https://t.me/BotFather), официальным инструментом управления ботами.

1. Открой Telegram и найди **@BotFather**, либо перейди по ссылке [t.me/BotFather](https://t.me/BotFather)
2. Отправь `/newbot`
3. Выбери **отображаемое имя** (например, «Hermes Agent») — может быть любым
4. Выбери **имя пользователя** — должно быть уникальным и заканчиваться на `bot` (например, `my_hermes_bot`)
5. BotFather пришлёт твой **API‑токен**. Он выглядит так:

```
123456789:ABCdefGHIjklMNOpqrSTUvwxYZ
```

:::warning
Храни токен бота в секрете. Любой, у кого есть токен, может управлять твоим ботом. Если он утечёт, немедленно отзови его через `/revoke` в BotFather.
:::

## Шаг 2: Настрой бота (по желанию)

Эти команды BotFather улучшают пользовательский опыт. Напиши @BotFather и используй:

| Команда | Назначение |
|---------|------------|
| `/setdescription` | Текст «Что может делать этот бот?», отображаемый перед началом чата |
| `/setabouttext` | Краткий текст на странице профиля бота |
| `/setuserpic` | Загрузить аватарку для бота |
| `/setcommands` | Определить меню команд (кнопка `/` в чате) |
| `/setprivacy` | Управлять тем, видит ли бот все сообщения группы (см. Шаг 3) |

:::tip
Для `/setcommands` полезный стартовый набор:

```
help - Show help information
new - Start a new conversation
sethome - Set this chat as the home channel
```
:::

## Шаг 3: Режим конфиденциальности (важно для групп)

У ботов Telegram есть **режим конфиденциальности**, который **включён по умолчанию**. Это самая частая причина путаницы при использовании ботов в группах.

**При включённом режиме конфиденциальности** бот видит только:
- Сообщения, начинающиеся с команды `/`
- Ответы непосредственно на сообщения бота
- Сервисные сообщения (вход/выход участников, закреплённые сообщения и т.п.)
- Сообщения в каналах, где бот администратор

**При отключённом режиме конфиденциальности** бот получает каждое сообщение в группе.

### Как отключить режим конфиденциальности

1. Напиши **@BotFather**
2. Отправь `/mybots`
3. Выбери своего бота
4. Перейди в **Bot Settings → Group Privacy → Turn off**

:::warning
**После изменения настройки конфиденциальности необходимо удалить бота из группы и добавить его снова.** Telegram кэширует состояние конфиденциальности при присоединении бота к группе, и оно обновится только после повторного добавления.
:::

:::tip
Альтернатива отключению конфиденциальности: назначить бота **администратором группы**. Администраторы всегда получают все сообщения независимо от режима конфиденциальности, что избавляет от необходимости переключать глобальный режим.
:::

### Наблюдать чат группы без автоматических ответов

Для поведения группы в стиле OpenClaw/Yuanbao настрой Telegram так, чтобы бот **видел** обычные сообщения группы, но **отвечал** только при прямом упоминании:

```yaml
telegram:
  allowed_chats:
    - "-1001234567890"
  group_allowed_chats:
    - "-1001234567890"
  require_mention: true
  observe_unmentioned_group_messages: true
```

В этом режиме сообщения из явно разрешённых чатов/тем добавляются к транскрипту общей сессии чата/темы как наблюдаемый контекст, но не вызывают агент. `allowed_chats` определяет, где бот отвечает; `group_allowed_chats` разрешает общую групповую сессию, используемую для наблюдаемого контекста, поэтому используй те же ID чатов. Позднее упоминание `@botname`, ответ боту или настроенный шаблон упоминания в том же разрешённом чате/теме могут использовать этот наблюдаемый контекст. Триггерное сообщение также помечается `[nickname|user_id]` и получает per‑turn safety‑prompt, чтобы модель рассматривала предыдущие наблюдаемые строки как контекст, а не как инструкции боту.

Эквивалентная переменная окружения:

```bash
TELEGRAM_ALLOWED_CHATS=-1001234567890
TELEGRAM_GROUP_ALLOWED_CHATS=-1001234567890
TELEGRAM_OBSERVE_UNMENTIONED_GROUP_MESSAGES=true
```

Это требует, чтобы Telegram доставлял обычные групповые сообщения шлюзу, поэтому отключи конфиденциальность BotFather или назначь бота администратором группы, как описано выше.

## Шаг 4: Найди свой пользовательский ID

Hermes Agent использует числовые Telegram‑ID пользователей для контроля доступа. Твой пользовательский ID **не** является именем пользователя — это число вроде `123456789`.

**Метод 1 (рекомендованный):** Напиши [@userinfobot](https://t.me/userinfobot) — он сразу ответит твоим пользовательским ID.
**Метод 2:** Напиши [@get_id_bot](https://t.me/get_id_bot) — тоже надёжный вариант.

Сохрани этот номер; он понадобится на следующем шаге.

## Шаг 5: Настройка Hermes

### Вариант A: Интерактивная настройка (рекомендовано)

```bash
hermes gateway setup
```

Выбери **Telegram**, когда будет предложено. Мастер запросит токен бота и разрешённые пользовательские ID, затем запишет конфигурацию за тебя.

### Вариант B: Ручная конфигурация

Добавь следующее в `~/.hermes/.env`:

```bash
TELEGRAM_BOT_TOKEN=123456789:ABCdefGHIjklMNOpqrSTUvwxYZ
TELEGRAM_ALLOWED_USERS=123456789    # Comma-separated for multiple users
```

### Запуск шлюза

```bash
hermes gateway
```

Бот должен появиться онлайн в течение нескольких секунд. Отправь ему сообщение в Telegram, чтобы проверить.

## Отправка сгенерированных файлов из Docker‑бэкендов терминала

Если твой терминальный бэкенд — `docker`, помни, что вложения Telegram отправляются **процессом шлюза**, а не изнутри контейнера. Поэтому конечный путь `MEDIA:/...` должен быть доступен на хосте, где работает шлюз.

Распространённая ошибка:

- агент пишет файл внутри Docker в `/workspace/report.txt`
- модель выдаёт `MEDIA:/workspace/report.txt`
- доставка в Telegram падает, потому что `/workspace/report.txt` существует только внутри контейнера

Рекомендованный шаблон:

```yaml
terminal:
  backend: docker
  docker_volumes:
    - "/home/user/.hermes/cache/documents:/output"
```

Затем:

- записывай файлы внутри Docker в `/output/...`
- указывай **видимый хостом** путь в `MEDIA:`, например: `MEDIA:/home/user/.hermes/cache/documents/report.txt`

Если у тебя уже есть секция `docker_volumes:`, добавь новый монт в тот же список. Дублирующиеся ключи YAML тихо переопределяют предыдущие.

### Поддерживаемые расширения файлов `MEDIA:`

Шлюз извлекает теги `MEDIA:/path/to/file` из ответов агента и отправляет указанный файл как нативное вложение. Поддерживаемые расширения на всех платформах шлюза:

| Категория | Расширения |
|---|---|
| Images | `png`, `jpg`, `jpeg`, `gif`, `webp`, `bmp`, `tiff`, `svg` |
| Audio | `mp3`, `wav`, `ogg`, `m4a`, `opus`, `flac`, `aac` |
| Video | `mp4`, `mov`, `webm`, `mkv`, `avi` |
| **Documents** | `pdf`, `txt`, `md`, `csv`, `json`, `xml`, `html`, `yaml`, `yml`, `log` |
| **Office** | `docx`, `xlsx`, `pptx`, `odt`, `ods`, `odp` |
| **Archives** | `zip`, `rar`, `7z`, `tar`, `gz`, `bz2` |
| **Books / packages** | `epub`, `apk`, `ipa` |

Все из этого списка доставляются как нативные вложения на платформах, их поддерживающих (Telegram, Discord, Signal, Slack, WhatsApp, Feishu, Matrix и др.). На платформах без нативной поддержки будет ссылка или простой текст. **Жирные** категории добавлены в последних релизах — если ты полагался на модель, говорящую `here is the file: /path/to/report.docx`, переключись на `MEDIA:/path/to/report.docx` для нативной доставки.

## Режим веб‑хука

По умолчанию Hermes подключается к Telegram через **длинный опрос** — шлюз делает исходящие запросы к серверам Telegram, чтобы получать новые обновления. Это удобно для локальных и постоянно работающих развертываний.

Для **облачных развертываний** (Fly.io, Railway, Render и др.) **режим веб‑хука** экономичнее. Такие платформы могут автоматически «просыпаться» при входящем HTTP‑трафике, но не при исходящих соединениях. Поскольку опрос — исходящий, бот в режиме опроса никогда не спит. Веб‑хук меняет направление — Telegram посылает обновления на HTTPS‑URL твоего бота, позволяя спать при бездействии.

| | Опрос (по умолчанию) | Веб‑хук |
|---|---|---|
| Направление | Шлюз → Telegram (исходящий) | Telegram → Шлюз (входящий) |
| Лучшее для | Локальные, постоянно включённые серверы | Облачные платформы с авто‑просыпанием |
| Настройка | Нет дополнительных параметров | Установи `TELEGRAM_WEBHOOK_URL` |
| Стоимость простоя | Машина должна работать постоянно | Машина может спать между сообщениями |

### Конфигурация

Добавь следующее в `~/.hermes/.env`:

```bash
TELEGRAM_WEBHOOK_URL=https://my-app.fly.dev/telegram
TELEGRAM_WEBHOOK_SECRET="$(openssl rand -hex 32)"  # required
# TELEGRAM_WEBHOOK_PORT=8443        # optional, default 8443
```

| Переменная | Обязательно | Описание |
|------------|-------------|----------|
| `TELEGRAM_WEBHOOK_URL` | Да | Публичный HTTPS‑URL, куда Telegram будет отправлять обновления. Путь автоматически извлекается (например, `/telegram` из примера). |
| `TELEGRAM_WEBHOOK_SECRET` | **Да** (если задан `TELEGRAM_WEBHOOK_URL`) | Секретный токен, который Telegram возвращает в каждом запросе веб‑хука для проверки. Шлюз не запустится без него — см. [GHSA-3vpc-7q5r-276h](https://github.com/NousResearch/hermes-agent/security/advisories/GHSA-3vpc-7q5r-276h). Сгенерировать: `openssl rand -hex 32`. |
| `TELEGRAM_WEBHOOK_PORT` | Нет | Локальный порт, на котором слушает сервер веб‑хука (по умолчанию `8443`). |

Если `TELEGRAM_WEBHOOK_URL` задан, шлюз запускает HTTP‑сервер веб‑хука вместо опроса. Если не задан, используется режим опроса — поведение прежнее.

### Пример облачного развертывания (Fly.io)

1. Добавь переменные окружения в секреты приложения Fly.io:

```bash
fly secrets set TELEGRAM_WEBHOOK_URL=https://my-app.fly.dev/telegram
fly secrets set TELEGRAM_WEBHOOK_SECRET=$(openssl rand -hex 32)
```

2. Открой порт веб‑хука в `fly.toml`:

```toml
[[services]]
  internal_port = 8443
  protocol = "tcp"

  [[services.ports]]
    handlers = ["tls", "http"]
    port = 443
```

3. Разверни:

```bash
fly deploy
```

В логах шлюза должно появиться: `[telegram] Connected to Telegram (webhook mode)`.

## Поддержка прокси

Если API Telegram заблокирован или нужен прокси, укажи Telegram‑специфичный URL прокси. Он имеет приоритет над общими `HTTPS_PROXY` / `HTTP_PROXY`.

**Вариант 1: config.yaml (рекомендовано)**

```yaml
telegram:
  proxy_url: "socks5://127.0.0.1:1080"
```

**Вариант 2: переменная окружения**

```bash
TELEGRAM_PROXY=socks5://127.0.0.1:1080
```

Поддерживаемые схемы: `http://`, `https://`, `socks5://`.

Прокси применяется как к основному соединению Telegram, так и к резервному IP‑транспорту. Если Telegram‑специфичный прокси не задан, шлюз использует `HTTPS_PROXY` / `HTTP_PROXY` / `ALL_PROXY` (или автоматическое обнаружение системного прокси macOS).

## Домашний канал

В любом чате Telegram (личном или групповом) используй команду `/sethome`, чтобы назначить его **домашним каналом**. Результаты запланированных задач (cron‑jobs) будут доставляться в этот канал.

Можно также задать вручную в `~/.hermes/.env`:

```bash
TELEGRAM_HOME_CHANNEL=-1001234567890
TELEGRAM_HOME_CHANNEL_NAME="My Notes"
```

:::tip
ID групповых чатов — отрицательные числа (например, `-1001234567890`). Твой личный DM‑чат имеет тот же ID, что и пользовательский ID.
:::

### Доставки cron‑задач в режиме тем

Если у тебя включён режим тем в DM‑бота, сообщения cron, доставляемые в корневой чат, попадают в системный лобби — ответы там не открывают сессию, и ты видишь уведомление «main chat is reserved for system commands». Создай отдельную тему форума (например, `Cron`) и укажи:

```bash
TELEGRAM_CRON_THREAD_ID=<topic_thread_id>
```

`TELEGRAM_CRON_THREAD_ID` переопределяет `TELEGRAM_HOME_CHANNEL_THREAD_ID` только для cron‑доставок. Ответы в этой теме продолжают существующую сессию темы.

## Голосовые сообщения

### Входящие голосовые (Speech‑to‑Text)

Голосовые сообщения, отправленные в Telegram, автоматически транскрибируются выбранным STT‑провайдером Hermes и вставляются в разговор как текст.

- `local` использует `faster-whisper` на машине, где работает Hermes — без API‑ключа
- `groq` использует Groq Whisper и требует `GROQ_API_KEY`
- `openai` использует OpenAI Whisper и требует `VOICE_TOOLS_OPENAI_KEY`

#### Пропуск STT: передать агенту необработанный аудиофайл

Если хочешь, чтобы **сам агент** обрабатывал аудио (диаризация, кастомный транскрибатор или просто архивирование), отключи STT, установив `stt.enabled: false` в `~/.hermes/config.yaml`:

```yaml
stt:
  enabled: false
```

При отключённом STT шлюз всё равно скачивает вложение голосового/аудио в кэш Hermes, но **не транскрибирует его**. Агент получает сообщение с маркером вроде:

```
[The user sent a voice message: /home/<user>/.hermes/cache/audio/<hash>.ogg]
```

Твои инструменты или навыки могут затем прочитать этот путь напрямую (например, передать в локальный пайплайн диаризации, более мощный транскрибатор или загрузить в долговременное хранилище). Расширение файла отражает оригинальный формат, который Telegram доставил (`.ogg` для голосовых заметок, `.mp3`/`.m4a`/… для аудио‑вложений).

Это естественно сочетается с разделом [локального Bot API сервера](#large-files-20mb-via-local-bot-api-server) ниже, который поднимает лимит `getFile` с 20 МБ до 2 ГБ — полезно, когда записи длиннее нескольких минут.

### Исходящие голосовые (Text‑to‑Speech)

Когда агент генерирует аудио через TTS, оно доставляется как нативные **голосовые пузырьки** Telegram — круглые, воспроизводимые в‑строке.

- **OpenAI и ElevenLabs** генерируют Opus напрямую — дополнительных настроек не требуется
- **Edge TTS** (провайдер по умолчанию) выдаёт MP3 и требует **ffmpeg** для конвертации в Opus:

```bash
# Ubuntu/Debian
sudo apt install ffmpeg

# macOS
brew install ffmpeg
```

Без ffmpeg Edge TTS будет отправлен как обычный аудиофайл (всё равно воспроизводимый, но через прямоугольный плеер вместо голосового пузырька).

Укажи провайдера TTS в `config.yaml` под ключом `tts.provider`.

## Большие файлы (>20 МБ) через локальный Bot API сервер

Публичный Bot API Telegram ограничивает загрузку `getFile` до **20 МБ**, поэтому любые голосовые заметки, аудио, видео или документы больше этого размера тихо отклоняются Hermes с ответом «слишком большой». Обходом является запуск **локального** [telegram-bot-api](https://github.com/tdlib/telegram-bot-api) демона — того же серверного ПО, что использует Telegram, но работающего в твоей сети. Локальный сервер поднимает лимит до **2 ГБ**, а Hermes автоматически снимает собственный внутренний лимит, когда видит настроенный `base_url`.

Это открывает такие сценарии:

- Отправка длинных голосовых заметок (45‑минутные встречи, подкасты) боту
- Загрузка больших видео для обработки vision‑инструментами
- Архивирование сырого аудио для офлайн‑пайплайнов (диаризация, выравнивание, обучение)

### Шаг 1: Получи Telegram‑API‑учётные данные

Локальный сервер общается напрямую с MTProto‑слой Telegram (а не с публичным Bot API), поэтому нужны **MTProto‑учётные данные**:

1. Перейди на [my.telegram.org/apps](https://my.telegram.org/apps) и войди в свой аккаунт Telegram.
2. Создай новое приложение (название и короткое описание — любые).
3. Скопируй `api_id` и `api_hash` — оба обязательны.

### Шаг 2: Запусти сервер telegram-bot-api

Самый простой путь — Docker‑образ [`aiogram/telegram-bot-api`](https://hub.docker.com/r/aiogram/telegram-bot-api). Минимальный `docker-compose.yaml` (используй режим `--local` для повышения лимитов):

```yaml
services:
  tg-bot-api:
    image: aiogram/telegram-bot-api:latest
    container_name: tg-bot-api
    restart: unless-stopped
    ports:
      - "127.0.0.1:8081:8081"   # bind to loopback only; see security note
    environment:
      TELEGRAM_API_ID: "12345"           # your api_id from Step 1
      TELEGRAM_API_HASH: "abcdef..."     # your api_hash from Step 1
      TELEGRAM_LOCAL: "1"                # enable --local mode (raises 20MB → 2GB)
    volumes:
      - ./tg-bot-api-data:/var/lib/telegram-bot-api
```

Подними контейнер:

```bash
docker compose up -d tg-bot-api
docker logs --tail 20 tg-bot-api
```

:::warning
Локальный Bot API сервер принимает токен бота в пути URL (например, `/bot<TOKEN>/getMe`) **без дополнительной аутентификации**. Любой, кто может достучаться до порта, полностью контролирует бота — читает все сообщения, отправляет от его имени и т.д. Привязывай контейнер к `127.0.0.1` и/или ставь перед ним обратный прокси в частной сети. **Никогда не открывай порт 8081 в публичный интернет.**
:::

### Шаг 3: Выход из публичного API (один раз)

Бот может работать только на **одном** Bot API сервере одновременно. Если твой бот уже подключён к `api.telegram.org` (что почти наверняка), нужно явно выйти из него, иначе локальный сервер его не примет:

```bash
curl "https://api.telegram.org/bot<YOUR_BOT_TOKEN>/logOut"
# expected response: {"ok":true,"result":true}
```

Это одноразовый шаг миграции — повторять его при каждом перезапуске не нужно. После `logOut` все новые сообщения будут приходить через новый сервер.

Проверь, что локальный сервер может общаться с Telegram от имени бота:

```bash
curl "http://127.0.0.1:8081/bot<YOUR_BOT_TOKEN>/getMe"
# expected response: {"ok":true,"result":{"id":...,"is_bot":true,...}}
```

### Шаг 4: Укажи Hermes локальный сервер

Добавь URL‑ы в `platforms.telegram.extra` в `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      base_url: "http://127.0.0.1:8081/bot"
      base_file_url: "http://127.0.0.1:8081/file/bot"
      local_mode: true        # see Step 5 below — only set this if the bot's data
                              # directory is readable by the Hermes process
```

:::caution
В текущей версии только форма `platforms.<name>.extra` глубоко сливается в конфиг платформы. Ключи, помещённые напрямую в `telegram.extra`, просто игнорируются.
:::

Когда задан `base_url`, Hermes:

- Строит клиент python‑telegram‑bot против локального сервера
- Автоматически поднимает внутренний лимит документов/аудио с 20 МБ → 2 ГБ
- Сообщает активный лимит в ошибке «слишком большой» (`Maximum: 2048 MB.`), чтобы было ясно, в каком режиме ты находишься

Перезапусти шлюз и ищи подтверждающую строку в логах:

```bash
hermes gateway restart
grep -E "Using custom Telegram base_url|Using Telegram local_mode" ~/.hermes/logs/gateway.log | tail
```

### Шаг 5: `local_mode` — доступ к файлам на диске

Локальный сервер может доставлять файлы двумя способами:

1. **Без `--local`** (по умолчанию): файлы отдаются по HTTP‑URL `/file/bot<TOKEN>/<path>`, как в публичном API. Лимит 20 МБ остаётся. Полезно лишь как сетевой фикс (например, когда `api.telegram.org` недоступен).
2. **С `--local`** (устанавливается через `TELEGRAM_LOCAL=1` выше): файлы записываются в файловую систему сервера, а ответ `getFile` возвращает **абсолютный путь** вместо HTTP‑URL. Лимит 20 МБ снимается. Hermes затем читает байты **с диска**, а не по HTTP.

Чтобы путь к файлу на диске работал, установи `local_mode: true` в конфиге выше **и** убедись, что процесс Hermes может читать возвращённый путь. Два сценария:

- **Тот же хост** — telegram‑bot‑api и Hermes работают на одной машине. Примонтируй том данных в директорию, доступную Hermes (например, `/var/lib/telegram-bot-api`), и проверь совпадение владельцев. Контейнер по умолчанию работает от пользователя `telegram-bot-api` (uid различается); проще добавить `user: "<UID>:<GID>"` в сервис compose, чтобы файлы принадлежали тому же uid, что и Hermes.
- **Разные машины** — сервер бота на одном хосте (NAS, отдельная VM), а Hermes на другом. Директория данных сервера должна быть доступна Hermes по **тому же абсолютному пути**, который сервер возвращает (обычно `/var/lib/telegram-bot-api`). Хорошо работает NFS; CIFS/SMB с `uid=` монтированием удобнее, если не хочется разбираться с uid‑несовпадениями.

Если `local_mode: true`, но Hermes не может `stat` возвращённый путь (проблемы с правами или монтированием), python‑telegram‑bot тихо переключается на HTTP‑`getFile` к локальному серверу, который в режиме `--local` отвечает `404 Not Found`. В логах это выглядит так:

```
[Telegram] Failed to cache voice: Not Found
telegram.error.InvalidToken: Not Found
```

Если видишь это, лимит поднят, но файловый шаринг не работает. Проверь `ls -la /var/lib/telegram-bot-api/<TOKEN>/voice/` с правами пользователя шлюза и убедись, что файл можно `cat` без ошибок.

### Шаг 6: Протестировать

Отправь боту голосовую заметку или аудиофайл размером более 20 МБ. Проверь лог шлюза:

```bash
tail -f ~/.hermes/logs/gateway.log | grep -iE "telegram|cache"
```

Ты должен увидеть строку `[Telegram] Cached user voice at /home/<user>/.hermes/cache/audio/...` и **не** получить отказ «слишком большой». В сочетании с `stt.enabled: false` путь к оригинальному аудио попадёт в входящее сообщение агента для дальнейшей обработки.

## Использование в групповых чатах

Hermes Agent работает в группах Telegram с некоторыми нюансами:

- **Режим конфиденциальности** определяет, какие сообщения бот видит (см. Шаг 3)
- `TELEGRAM_ALLOWED_USERS` всё равно применяется — только авторизованные пользователи могут вызывать бота, даже в группах
- Можно запретить боту отвечать на обычный групповой чат, установив `telegram.require_mention: true`
- При `telegram.require_mention: true` групповые сообщения принимаются, если они:
  - ответы на одно из сообщений бота
  - упоминания `@botusername`
  - команды вида `/command@botusername` (формат меню бота)
  - совпадают с одним из настроенных regex‑wake‑words в `telegram.mention_patterns`
- В группах с несколькими Hermes‑ботами `telegram.exclusive_bot_mentions` делает маршрутизацию детерминированной. Когда сообщение явно упоминает одного или нескольких Telegram‑ботов, только упомянутые боты обрабатывают его; остальные игнорируют до выполнения fallback‑триггеров. По умолчанию включено.
- `telegram.ignored_threads` позволяет заставить Hermes молчать в конкретных темах форума, даже если группа в целом разрешает свободные ответы или упоминания.
- Если `telegram.require_mention` не задан или `false`, Hermes сохраняет прежнее поведение «открытой группы» и отвечает на все видимые сообщения.

### Несколько Hermes‑ботов в одной группе

Если в одной группе запущено несколько профилей Hermes, создай отдельный токен бота для каждого профиля и запусти отдельный шлюз для каждого. Не используй один и тот же токен в нескольких работающих шлюзах — Telegram отклонит одновременный опрос.

Рекомендуемая конфигурация группы:

```yaml
telegram:
  require_mention: true
  exclusive_bot_mentions: true
  mention_patterns: []
```

С такой настройкой сообщение вроде `@research_bot @ops_bot summarize this` будет обработано только `research_bot` и `ops_bot`. Остальные Hermes‑боты в группе останутся тихими, даже если сообщение является ответом на их предыдущие сообщения или совпадает с общим wake‑word.

Отключи `exclusive_bot_mentions: false` только для устаревших групп, где явные упоминания не должны переопределять ответы и wake‑word триггеры.

Чтобы запустить несколько профилей, выполните команду шлюза один раз для каждого профиля, например:

```bash
# default profile
hermes gateway start
hermes gateway status
hermes gateway stop

# named profiles
hermes -p research gateway start
hermes -p research gateway status
hermes -p research gateway stop
```

Для небольшого фиксированного набора можно использовать цикл оболочки или скрипт, вызывающий `hermes gateway <action>` для профиля по умолчанию и `hermes -p <profile> gateway <action>` для каждого именованного профиля. Это надёжнее, чем полагаться на один процесс, управляющий всеми профилями.

### Устранение неполадок: работает в DM, но не в группах

Если бот отвечает в личных сообщениях, но молчит в группе, проверь следующие пункты по порядку:

1. **Доставка Telegram:** отключи конфиденциальность BotFather, назначь бота администратором или упомяни бота явно. Hermes не может отвечать на сообщения, которые Telegram не доставил боту.
2. **Повторное присоединение после изменения конфиденциальности:** удали бота из группы и добавь его снова после изменения настроек BotFather. Telegram может сохранять старое поведение для уже существующих участников.
3. **Авторизация Hermes:** убедись, что отправитель включён в `TELEGRAM_ALLOWED_USERS` или `TELEGRAM_GROUP_ALLOWED_USERS`, либо разреши чат через `TELEGRAM_GROUP_ALLOWED_CHATS`.
4. **Фильтры упоминаний:** если установлен `telegram.require_mention: true`, обычный групповой чат игнорируется, если сообщение не является slash‑командой, ответом боту, упоминанием `@botusername` или не совпадает с `mention_patterns`.
5. **Маршрутизация нескольких ботов:** если в группе несколько ботов, убедись, что каждый профиль Hermes использует уникальный токен бота и что `exclusive_bot_mentions` включён, если только не нужен старый совместный режим.

Отрицательные ID чатов — нормальная ситуация для групп и супергрупп Telegram. Если ты используешь авторизацию по чатам, помещай эти ID в `TELEGRAM_GROUP_ALLOWED_CHATS`, а не в список разрешённых отправителей.

### Пример конфигурации триггеров группы

Добавь следующее в `~/.hermes/config.yaml`:

```yaml
telegram:
  require_mention: true
  exclusive_bot_mentions: true
  mention_patterns:
    - "^\\s*chompy\\b"
  ignored_threads:
    - 31
    - "42"
```

Этот пример позволяет всем обычным прямым триггерам плюс сообщения, начинающиеся с `chompy`, даже без `@упоминания`. Сообщения в темах Telegram `31` и `42` всегда игнорируются до проверки упоминаний и свободных ответов.

### Примечания к `mention_patterns`

- Шаблоны используют регулярные выражения Python
- Сравнение нечувствительно к регистру
- Шаблоны проверяются как в текстовых сообщениях, так и в подписях к медиа
- Неправильные regex‑шаблоны игнорируются с предупреждением в логах шлюза, а не приводят к падению бота
- Если нужно, чтобы шаблон срабатывал только в начале сообщения, зафиксируй его с помощью `^`

## Приватные темы чатов (Bot API 9.4)

Bot API Telegram 9.4 (февраль 2026) ввёл **приватные темы чатов** — боты могут создавать форумы‑темы прямо в 1‑на‑1 DM‑чатах, без необходимости супергруппы. Это позволяет иметь несколько изолированных рабочих пространств внутри одного DM‑чата с Hermes.

### Сценарий использования

Если ты работаешь над несколькими длительными проектами, темы позволяют держать их контекст раздельным:

- **Тема «Website»** — работа над продакшн‑веб‑сервисом
- **Тема «Research»** — обзор литературы и исследование статей
- **Тема «General»** — прочие задачи и быстрые вопросы

Каждая тема имеет собственную сессию, историю и контекст — полностью изолированы друг от друга.

### Конфигурация

:::caution
Перед добавлением тем в конфиг пользователь должен **включить режим Тем** в DM‑чате с ботом:

1. Открой личный чат с Hermes‑ботом в Telegram
2. Нажми на имя бота вверху, чтобы открыть информацию о чате
3. Включи **Topics** (переключатель, превращающий чат в форум)

Если это не сделано, Hermes запишет в лог `The chat is not a forum` при старте и пропустит создание тем. Это клиентская настройка Telegram — бот не может включить её программно.
:::

Добавь темы в `platforms.telegram.extra.dm_topics` в `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      dm_topics:
      - chat_id: 123456789        # Your Telegram user ID
        topics:
        - name: General
          icon_color: 7322096
        - name: Website
          icon_color: 9367192
        - name: Research
          icon_color: 16766590
          skill: arxiv              # Auto-load a skill in this topic
```

**Поля:**

| Поле | Обязательно | Описание |
|------|--------------|----------|
| `name` | Да | Отображаемое название темы |
| `icon_color` | Нет | Код цвета иконки Telegram (целое число) |
| `icon_custom_emoji_id` | Нет | ID пользовательского эмодзи для иконки темы |
| `skill` | Нет | Навык, который автоматически загружается в новых сессиях этой темы |
| `thread_id` | Нет | Заполняется автоматически после создания темы — не задавай вручную |

### Как это работает

1. При старте шлюза Hermes вызывает `createForumTopic` для каждой темы без `thread_id`.
2. `thread_id` сохраняется обратно в `config.yaml` автоматически — при последующих запусках вызов API пропускается.
3. Каждая тема сопоставляется с изолированным ключом сессии: `agent:main:telegram:dm:{chat_id}:{thread_id}`.
4. Сообщения в каждой теме имеют собственную историю, очистку памяти и окно контекста.

### Обработка корневого DM

По умолчанию сообщения, отправленные в корневой DM (вне любой темы), обрабатываются как обычно. Установи `ignore_root_dm: true`, чтобы превратить корневой DM в лобби — обычные сообщения будут игнорироваться для пользователей, у которых настроены темы, в то время как системные команды (`/start`, `/help`, `/status` и др.) продолжат работать.

```yaml
platforms:
  telegram:
    extra:
      ignore_root_dm: true
      dm_topics:
        - chat_id: 123456789
          topics:
            - name: General
```

Проверка происходит **по чату**: только пользователи, имеющие хотя бы одну запись в `dm_topics`, получат такое поведение. Пользователи без тем не затронуты.

### Привязка навыка

Тема с полем `skill` автоматически загружает указанный навык при запуске новой сессии в этой теме. Это эквивалентно вводу `/skill-name` в начале разговора — содержимое навыка вставляется в первое сообщение, а последующие сообщения видят его в истории.

:::tip
Темы, созданные вне конфига (например, вручную через API Telegram), автоматически обнаруживаются, когда приходит сервисное сообщение `forum_topic_created`. Ты также можешь добавить темы в конфиг во время работы шлюза — они будут подхвачены при следующем кэше.
:::

## Многосессионный режим DM (`/topic`)

ChatGPT‑подобный многосессионный DM — один бот, несколько параллельных разговоров. В отличие от оператор‑заданного `extra.dm_topics`, этот режим **управляется пользователем**: без конфига, без предопределённых названий тем. Пользователь включает его командой `/topic`, затем нажимает кнопку **+** в Telegram, чтобы создать столько тем, сколько захочет, каждая со своей независимой сессией Hermes.

### Подкоманды `/topic`

| Форма | Контекст | Эффект |
|------|----------|--------|
| `/topic` | Корневой DM, ещё не включён | Проверить возможности BotFather, включить многосессионный режим, создать закреплённую системную тему |
| `/topic` | Корневой DM, уже включён | Показать статус: доступные несвязанные сессии для восстановления |
| `/topic` | Внутри темы | Показать привязку текущей темы к сессии |
| `/topic help` | Любой | Инлайн‑помощь |
| `/topic off` | Корневой DM | Отключить многосессионный режим и очистить все привязки тем для этого чата |
| `/topic <session-id>` | Внутри темы | Восстановить предыдущую Telegram‑сессию в текущей теме |

Только авторизованные пользователи (через `TELEGRAM_ALLOWED_USERS` / конфиг авторизации платформы) могут выполнять `/topic`. Неавторизованный отправитель получит отказ вместо активации.

### DM‑темы vs многосессионный режим

| | `extra.dm_topics` (конфиг‑управляемый) | `/topic` (управляемый пользователем) |
|---|---|---|
| Кто активирует | Оператор, в `config.yaml` | Конечный пользователь, отправив `/topic` |
| Список тем | Фиксированный набор в конфиге | Пользователь свободно создаёт/удаляет темы |
| Имена тем | Задаёт оператор | Задаёт пользователь; автоматически переименовывается в заголовок сессии Hermes |
| Поведение корневого DM | Обычный чат (лобби, если `ignore_root_dm: true`) | Становится системным лобби (не командные сообщения отклоняются) |
| Основное назначение | Постоянные рабочие пространства с возможной привязкой навыков | Ад‑hoc параллельные сессии |
| Сохранение | `extra.dm_topics` в конфиге | Таблицы SQLite `telegram_dm_topic_mode` + `telegram_dm_topic_bindings` |

Обе функции могут сосуществовать: ты можешь использовать `/topic` в пользовательском DM, а `extra.dm_topics` продолжит управлять операторскими темами для других чатов.

### Предварительные требования

В **@BotFather** открой настройки бота → **Bot Settings → Threads Settings**:

1. Включи **Threaded Mode** (включает `has_topics_enabled`)
2. Не отключай возможность пользователям создавать темы (`allows_users_to_create_topics`)

Когда пользователь впервые выполнит `/topic`, Hermes вызовет `getMe` для проверки обоих флагов. Если какой‑то выключен, Hermes пришлёт скриншот страницы настроек BotFather с объяснением, что нужно переключить — активация не произойдёт, пока требования не выполнены.

### Поток активации

Из корневого DM отправь:

```
/topic
```

Hermes:

1. Проверит `has_topics_enabled` и `allows_users_to_create_topics`.
2. Если оба `true`, включит многосессионный режим для этого DM.
3. Попробует создать и закрепить **System**‑тему для статуса/команд (по возможности).
4. Ответит списком предыдущих несвязанных Telegram‑сессий, которые пользователь может восстановить.

После активации корневой DM становится лобби: обычные запросы отклоняются с подсказкой перейти в **All Messages**. Системные команды (`/status`, `/sessions`, `/usage`, `/help` и др.) работают как обычно.

### Создание новой темы (пользовательский поток)

1. Открой DM‑бота в Telegram
2. Нажми **All Messages** вверху интерфейса бота, затем отправь любое сообщение
3. Telegram создаст новую тему для этого сообщения
4. Hermes ответит внутри этой темы — теперь тема представляет отдельную сессию

Каждая тема имеет собственную историю, состояние модели, выполнение инструментов и идентификатор сессии. Ключ изоляции: `agent:main:telegram:dm:{chat_id}:{thread_id}` — тот же, что и у конфиг‑управляемых DM‑тем.

### Автоматическое переименование тем

Когда Hermes генерирует заголовок сессии для темы (через пайплайн авто‑заголовка после первого обмена), тема в Telegram переименовывается, чтобы соответствовать — например, «New Topic» становится «Database migration plan». Переименование работает в лучшем случае: ошибки логируются, но не ломают сессию.

Чтобы отключить переименование и оставить вручную выбранные названия тем, задай:

```yaml
gateway:
  platforms:
    telegram:
      extra:
        disable_topic_auto_rename: true
```

При включённом флаге Hermes всё равно генерирует внутренний заголовок сессии (используется в `hermes sessions`, TUI и т.п.), но не меняет название темы в Telegram. Полезно, если ты вручную организуешь темы через BotFather Threaded Mode и не хочешь, чтобы каждый первый ответ их переименовывал.

### `/new` внутри темы

Сбрасывает текущую сессию темы (новый ID сессии, чистая история) без изменения других тем. Hermes отвечает напоминанием, что для параллельной работы обычно лучше создать новую тему (через **All Messages**).

### Восстановление предыдущей сессии

Внутри темы отправь:

```
/topic <session-id>
```

Это привязывает текущую тему к уже существующей сессии Hermes вместо создания новой. Полезно, если нужно продолжить разговор, начатый до включения режима тем. Ограничения:

- Целевая сессия должна принадлежать тому же пользователю Telegram
- Целевая сессия не должна уже быть привязана к другой теме

Hermes подтверждает заголовком сессии и воспроизводит последнее сообщение ассистента для контекста.

Чтобы увидеть доступные ID сессий, отправь `/topic` (без аргументов) в корневом DM — Hermes перечислит несвязанные Telegram‑сессии пользователя.

### `/topic` внутри темы (без аргументов)

Показывает привязку текущей темы: заголовок сессии, ID сессии и подсказки для `/new` vs создания новой темы.

### Как это реализовано

- При старте шлюза Hermes вызывает `createForumTopic` для каждой темы без `thread_id`.
- `thread_id` сохраняется обратно в `config.yaml`.
- Каждый входящий DM‑сообщение ищет привязку `(chat_id, thread_id)` в таблице `telegram_dm_topic_bindings`. Если найдена, сообщение маршрутизируется к соответствующей сессии через `SessionStore.switch_session()`.
- `/new` внутри темы обновляет строку привязки, указывая новый `session_id`.
- Темы из `extra.dm_topics` **никогда** не переименовываются автоматически — сохраняется имя, заданное оператором, даже если включён многосессионный режим.
- Установи `extra.disable_topic_auto_rename: true`, чтобы отключить автоматическое переименование **всех** тем в чате (включая ад‑hoc темы, созданные через Threaded Mode).
- Общая (закреплённая) тема в форуме‑DM считается корневым лобби, независимо от того, доставляет ли Telegram её с `message_thread_id=1` или без `thread_id`.
- Напоминания в корневом лобби ограничены одним сообщением каждые 30 секунд на чат — пользователь, забывший, что режим тем включён, не получит десяток ответов.
- Скриншоты настроек BotFather ограничены одним отправлением каждые 5 минут на чат — повторные попытки `/topic`, пока Threads отключены, не будут повторно загружать изображение.
- Команда `/background <prompt>` внутри темы отправляет результат в ту же тему; фоновые сессии не вызывают автоматическое переименование темы.
- `/topic` сам проверяется на авторизацию бота — неавторизованные DMs получают отказ вместо активации.

### Отключение многосессионного режима

Отправь `/topic off` в корневом DM. Hermes выключит режим, очистит все привязки `(thread_id → session_id)` для этого чата, а корневой DM вернётся к обычному поведению Hermes. Существующие темы в Telegram не удаляются — они просто перестают быть привязанными к отдельным сессиям. Позже можно снова включить режим, выполнив `/topic`.

Если нужно очистить вручную (например, массовый сброс по многим чатам), удали строки напрямую:

```bash
sqlite3 ~/.hermes/state.db \
  "UPDATE telegram_dm_topic_mode SET enabled = 0 WHERE chat_id = '<your_chat_id>'; \
   DELETE FROM telegram_dm_topic_bindings WHERE chat_id = '<your_chat_id>';"
```

### Понижение версии Hermes

Если ты откатываешься к версии Hermes, предшествующей `/topic`, функция просто перестаёт работать — таблицы `telegram_dm_topic_mode` и `telegram_dm_topic_bindings` остаются в `state.db`, но игнорируются старым кодом. DM‑чаты возвращаются к нативной изоляции по `message_thread_id`, так что существующие темы Telegram продолжают работать как параллельные сессии. Корневой DM больше не является лобби — сообщения туда попадают в агент как обычно. При повторном обновлении режим многосессий активируется точно там, где был.

## Привязка навыков к темам группы

Супергруппы с включённым **режимом Тем** (forum topics) уже имеют изоляцию сессий по теме — каждый `thread_id` сопоставлен своей беседе. Но ты можешь **автоматически загружать навык**, когда сообщения приходят в конкретную тему группы, аналогично привязке навыков к DM‑темам.

### Сценарий использования

Супергруппа команды с темами для разных потоков работы:

- Тема **Engineering** → автоматически загружает навык `software-development`
- Тема **Research** → автоматически загружает навык `arxiv`
- Тема **General** → без навыка, обычный помощник

### Конфигурация

Добавь привязки тем в `platforms.telegram.extra.group_topics` в `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      group_topics:
      - chat_id: -1001234567890       # Supergroup ID
        topics:
        - name: Engineering
          thread_id: 5
          skill: software-development
        - name: Research
          thread_id: 12
          skill: arxiv
        - name: General
          thread_id: 1
          # No skill — general purpose
```

**Поля:**

| Поле | Обязательно | Описание |
|------|--------------|----------|
| `chat_id` | Да | Числовой ID супергруппы (отрицательное число, начинающееся с `-100`) |
| `name` | Нет | Человекочитаемая метка темы (только для информации) |
| `thread_id` | Да | ID темы форума Telegram — виден в ссылках `t.me/c/<group_id>/<thread_id>` |
| `skill` | Нет | Навык, автоматически загружаемый в новых сессиях этой темы |

### Как это работает

1. Когда сообщение приходит в сопоставленную групповую тему, Hermes ищет `chat_id` и `thread_id` в конфиге `group_topics`.
2. Если найдено поле `skill`, этот навык автоматически загружается для сессии — точно так же, как в DM‑темах.
3. Темы без `skill` получают только изоляцию сессии (поведение по умолчанию, без изменений).
4. Несопоставленные `thread_id` или `chat_id` просто игнорируются — без ошибок и без навыков.

### Отличия от DM‑тем

| | DM‑темы | Темы группы |
|---|---|---|
| Ключ конфигурации | `extra.dm_topics` | `extra.group_topics` |
| Создание темы | Hermes создаёт темы через API, если `thread_id` отсутствует | Администратор создаёт темы в UI Telegram |
| `thread_id` | Автоматически заполняется после создания | Должен быть указан вручную |
| `icon_color` / `icon_custom_emoji_id` | Поддерживаются | Не применимо (внешний вид контролирует админ) |
| Привязка навыка | ✓ | ✓ |
| Изоляция сессии | ✓ | ✓ (уже встроена для форум‑тем) |

:::tip
Чтобы узнать `thread_id` темы, открой её в Telegram Web или Desktop и посмотри URL: `https://t.me/c/1234567890/5` — последняя цифра (`5`) и есть `thread_id`. `chat_id` для супергрупп — это ID группы с префиксом `-100` (например, группа `1234567890` превращается в `-1001234567890`).
:::

## Недавние возможности Bot API

- **Bot API 9.4 (февраль 2026):** Приватные чат‑темы — боты могут создавать форумы‑темы в 1‑на‑1 DM‑чатах через `createForumTopic`. Hermes использует это для двух функций: оператор‑заданных [Приватных тем чата](#private-chat-topics-bot-api-94) (конфиг‑управляемый фиксированный список) и пользователь‑управляемого [Многосессионного DM‑режима](#multi-session-dm-mode-topic) (активируется `/topic`, неограниченное количество тем).
- **Политика конфиденциальности:** Telegram теперь требует, чтобы у ботов была политика конфиденциальности. Установи её через BotFather командой `/setprivacy_policy`, иначе Telegram может сгенерировать заглушку. Это особенно важно для публичных ботов.
- **Bot API 9.5 (март 2026):** Нативный стриминг через `sendMessageDraft`. Hermes поддерживает нативный стриминг‑черновиков в приватных чатах как опцию. По умолчанию используется старый путь `editMessageText`, потому что черновики могут визуально схлопываться в некоторых клиентах.

### Транспорт стриминга (`gateway.streaming.transport`)

Когда стриминг включён (`gateway.streaming.enabled: true`), Hermes выбирает один из четырёх транспортов:

| Значение | Поведение |
|---|---|
| `auto` | Нативный черновик‑стриминг в поддерживаемых чатах (сейчас только DM Telegram); иначе старый путь `edit`. При сбое черновика автоматически переключается на `edit`. |
| `draft` | Принудительно использовать нативные черновики. Если чат не поддерживает черновики (группы/темы), логируется откат к `edit`. |
| `edit` (по умолчанию) | Прогрессивный `editMessageText` для всех типов чатов. |
| `off` | Отключить стриминг полностью (только финальный ответ). |

В `~/.hermes/config.yaml`:

```yaml
gateway:
  streaming:
    enabled: true
    transport: edit    # edit | auto | draft | off
```

**Что ты увидишь в DM при `edit` (по умолчанию)** — шлюз отправит обычное предварительное сообщение и будет постепенно обновлять его через `editMessageText`, избегая эффекта схлопывания черновика.

**Что ты увидишь в DM при `auto` или `draft`** — Telegram покажет анимированный черновик‑предпросмотр, обновляющийся токен‑за‑токеном. Когда ответ завершится, он будет доставлен как обычное сообщение, а черновик исчезнет. Черновики не имеют `message_id`, поэтому финальный ответ остаётся в истории чата.

**А как насчёт групп, супергрупп, форум‑тем?** `sendMessageDraft` ограничен только приватными чатами (DM). Шлюз автоматически переходит к пути `edit` для остальных — прежний UX.

**Что если черновик не удался?** Любая ошибка (временная сеть, отказ сервера, устаревший python‑telegram‑bot) переключает дальнейшую часть потока на путь `edit`. Следующий ответ получит новую попытку.

## Рендеринг: таблицы и превью ссылок

MarkdownV2 Telegram не поддерживает таблицы — pipe‑таблицы превращаются в бессмысленный шум, если передать их «как есть». Hermes автоматически нормализует таблицы:

- **Небольшие таблицы** преобразуются в **список пунктов** — каждая строка становится читаемым пунктом под заголовками колонок. Хорошо подходит для 2‑4 колонок и коротких ячеек.
- **Большие или широкие таблицы** выводятся как **блок кода** с выровненными колонками, чтобы ничего не «сломалось». Добавляется однострочная подсказка, чтобы агент предпочитал prose‑ответы вместо дальнейших таблиц в Telegram.

Настраивать это не требуется — адаптер выбирает подходящий fallback автоматически. Если нужен старый «всегда блок кода», отключи нормализацию, задав `telegram.pretty_tables: false` в `config.yaml` (по умолчанию `true`).

### Предпросмотр ссылок

Telegram автоматически генерирует превью для URL в сообщениях бота. Если ты хочешь их подавить (длинный вывод `/tools`, ответы с множеством ссылок и т.п.):

```yaml
gateway:
  platforms:
    telegram:
      extra:
        disable_link_previews: true
```

При включении Hermes добавляет к каждому исходящему сообщению `LinkPreviewOptions(is_disabled=True)`, а на старых версиях python‑telegram‑bot использует параметр `disable_web_page_preview`.

## Групповое разрешение (allowlisting)

В группах и форумах Telegram есть два независимых механизма, которые можно настроить:

- **Разрешённые пользовательские ID** (`group_allow_from` / `TELEGRAM_GROUP_ALLOWED_USERS`) — список отправителей, применимый только к сообщениям в группах/форумах. Используй, если хочешь, чтобы конкретные пользователи могли вызывать бота в группах, но не имели доступа к DM (что требует их добавления в `TELEGRAM_ALLOWED_USERS`).
- **Разрешённые чат‑ID** (`group_allowed_chats` / `TELEGRAM_GROUP_ALLOWED_CHATS`) — список чатов, где любой участник может взаимодействовать с ботом. Удобно для командных/поддерживающих ботов, где членство в группе само по себе является сигналом доступа.

```yaml
gateway:
  platforms:
    telegram:
      extra:
        # Global access (DMs + groups). Users here can always invoke the bot.
        allow_from:
          - "123456789"
        # Sender IDs allowed in groups/forums only. Does NOT grant DM access.
        group_allow_from:
          - "987654321"
        # Entire groups/forums — any member is authorized.
        group_allowed_chats:
          - "-1001234567890"
```

Эквивалентные переменные окружения:

```bash
TELEGRAM_ALLOWED_USERS="123456789"
TELEGRAM_GROUP_ALLOWED_USERS="987654321"
TELEGRAM_GROUP_ALLOWED_CHATS="-1001234567890"
```

Поведение:

- `TELEGRAM_ALLOWED_USERS` охватывает все типы чатов (DM, группы, форумы).
- `TELEGRAM_GROUP_ALLOWED_USERS` авторизует только указанных отправителей в группах/форумах. Они всё равно не могут писать боту в DM, если не указаны в `TELEGRAM_ALLOWED_USERS`.
- Чат, указанный в `TELEGRAM_GROUP_ALLOWED_CHATS`, авторизует всех его участников, независимо от их ID.
- Можно использовать `*` в любой из этих переменных, чтобы разрешить всех отправителей/чаты.
- Эти правила накладываются поверх уже существующих упоминаний/шаблонов и `group_topics` + `ignored_threads`.

### Миграция из до PR #17686

Раньше единственным параметром был `TELEGRAM_GROUP_ALLOWED_USERS`, и пользователи помещали туда **ID чатов**. Для обратной совместимости значения, начинающиеся с `-`, всё ещё трактуются как чат‑ID и выводят предупреждение о депрецировании. Миграция:

```bash
# Old (still works, but deprecated)
TELEGRAM_GROUP_ALLOWED_USERS="-1001234567890"

# New
TELEGRAM_GROUP_ALLOWED_CHATS="-1001234567890"
```

### Guest‑mode (`guest_mode`)

В типичной настройке `group_allowed_chats` — жёсткий фильтр: сообщения из групп, не входящих в список, просто отбрасываются, даже если в них явно упомянут бот. Это правильный вариант для поддерж‑ботов.

Для более «дружеских» групп, где бот должен быть в основном тихим, но иногда доступным по упоминанию, включи `guest_mode`:

```yaml
gateway:
  platforms:
    telegram:
      extra:
        group_allowed_chats:
          - "-1001234567890"   # your main allowlisted group
        guest_mode: true       # non-allowlisted groups: allow on @mention only
```

Эквивалент в переменных окружения:

```bash
TELEGRAM_GUEST_MODE=true
```

По умолчанию `false`.

При `guest_mode: true` сообщение из неразрешённой группы обрабатывается **только**, если в нём явно упомянут бот. Упоминание требуется на каждом шаге — нет «прилипшего» сеанса для гостей, поэтому бот никогда не будет автоматически участвовать в беседе в чужой группе без явного пинга.

DM и разрешённые группы работают как обычно.

## Управление доступом к slash‑командам

По умолчанию каждый разрешённый пользователь может выполнять любую slash‑команду. Чтобы разделить пользователей на **администраторов** (полный доступ к slash‑командам) и **обычных** (только к явно разрешённым), добавь `allow_admin_from` и `user_allowed_commands` в блок `extra` платформы:

```yaml
gateway:
  platforms:
    telegram:
      extra:
        # Existing allowlists (unchanged)
        allow_from:
          - "123456789"     # admin
          - "555555555"     # regular user
          - "777777777"     # regular user

        # NEW — admins get all slash commands (built-in + plugin)
        allow_admin_from:
          - "123456789"

        # NEW — non-admin allowed users can only run these slash commands.
        # /help and /whoami are always allowed so users can see their access.
        user_allowed_commands:
          - status
          - model
          - history

        # Optional: separate admin/command lists for groups
        group_allow_admin_from:
          - "123456789"
        group_user_allowed_commands:
          - status
```

**Поведение:**

- Пользователь, указанный в `allow_admin_from` для области (DM или группа), может выполнять **все** зарегистрированные slash‑команды — как встроенные, так и добавленные плагинами — через живой реестр.
- Пользователь, присутствующий в `allow_from`, но **не** в `allow_admin_from`, может запускать только команды, перечисленные в `user_allowed_commands`, плюс всегда разрешённые базовые: `/help` и `/whoami`.
- Обычные сообщения в чате не затронуты. Пользователи без админ‑прав всё равно могут разговаривать с агентом, просто без возможности вызывать произвольные slash‑команды.
- **Обратная совместимость:** если `allow_admin_from` не задана для области, фильтрация slash‑команд отключена для этой области. Существующие установки продолжают работать без изменений.
- Статус админа в DM не подразумевает админ‑статус в группе; каждая область имеет свой список.
- Если задан только `group_allow_admin_from`, область DM остаётся в режиме без ограничений (как в старой версии).

Команда `/whoami` покажет текущую область, твой уровень (admin / user / unrestricted) и какие slash‑команды доступны.

## Интерактивный выбор модели

Отправив `/model` без аргументов в чате Telegram, Hermes покажет интерактивную inline‑клавиатуру для переключения моделей:

1. **Выбор провайдера** — кнопки с названиями провайдеров и количеством моделей (например, «OpenAI (15)», «✓ Anthropic (12)» для текущего провайдера).
2. **Выбор модели** — постраничный список моделей с кнопками **Prev**/**Next**, кнопкой **Back** для возврата к провайдерам и **Cancel**.

Текущий провайдер и модель отображаются вверху. Всё навигационное происходит путём редактирования того же сообщения (без захламления чата).

:::tip
Если знаешь точное название модели, просто введи `/model <name>` — это пропустит выборщик. Можно добавить `--global`, чтобы сохранить выбор для всех сессий.
:::

## DNS‑over‑HTTPS резервные IP

В ограниченных сетях `api.telegram.org` может резолвиться в недоступный IP. Адаптер Telegram включает механизм **резервных IP**, который автоматически пытается альтернативные IP, сохраняя правильный TLS‑хостнейм и SNI.

### Как это работает

1. Если задан `TELEGRAM_FALLBACK_IPS`, эти IP используются напрямую.
2. Иначе адаптер автоматически запрашивает **Google DNS** и **Cloudflare DNS** через DNS‑over‑HTTPS (DoH), чтобы найти альтернативные IP для `api.telegram.org`.
3. IP, полученные от DoH и отличные от системного DNS‑результата, используются как резервные.
4. Если DoH тоже заблокирован, используется жёстко закодированный IP‑seed (`149.154.167.220`) как последняя попытка.
5. Как только резервный IP успешно отвечает, он становится «липким» — последующие запросы идут напрямую к нему без повторных попыток основного пути.

### Конфигурация

```bash
# Explicit fallback IPs (comma-separated)
TELEGRAM_FALLBACK_IPS=149.154.167.220,149.154.167.221
```

Или в `~/.hermes/config.yaml`:

```yaml
platforms:
  telegram:
    extra:
      fallback_ips:
        - "149.154.167.220"
```

:::tip
Обычно вручную настраивать не требуется. Авто‑обнаружение через DoH покрывает большинство ограниченных сетей. Переменная `TELEGRAM_FALLBACK_IPS` нужна только если DoH тоже заблокирован.
:::

## Поддержка прокси (повтор)

Если в сети нужен HTTP‑прокси для выхода в интернет (часто в корпоративных средах), адаптер Telegram автоматически читает стандартные переменные прокси и направляет весь трафик через прокси.

### Поддерживаемые переменные

Проверяются в порядке:

1. `HTTPS_PROXY`
2. `HTTP_PROXY`
3. `ALL_PROXY`
4. `https_proxy` / `http_proxy` / `all_proxy` (нижний регистр)

### Конфигурация

Установи прокси в окружении перед запуском шлюза:

```bash
export HTTPS_PROXY=http://proxy.example.com:8080
hermes gateway
```

Или добавь в `~/.hermes/.env`:

```bash
HTTPS_PROXY=http://proxy.example.com:8080
```

Прокси применяется как к основному транспорту, так и к резервным IP‑транспорту. Дополнительных настроек Hermes не требуется — если переменная окружения установлена, она используется автоматически.

:::note
Это относится к кастомному слою транспортов, который Hermes использует для соединений Telegram. Стандартный клиент `httpx`, используемый в остальных частях, уже нативно поддерживает переменные прокси.
:::

## Реакции на сообщения

Бот может добавлять эмодзи‑реакции к сообщениям как визуальную обратную связь:

- 👀 — когда бот начинает обрабатывать твоё сообщение
- ✅ — когда ответ успешно доставлен
- ❌ — если произошла ошибка

Реакции **по умолчанию отключены**. Включи их в `config.yaml`:

```yaml
telegram:
  reactions: true
```

Или через переменную окружения:

```bash
TELEGRAM_REACTIONS=true
```

:::note
В отличие от Discord, где реакции добавляются к уже существующим, Bot API Telegram заменяет все реакции бота одним вызовом. Переход от 👀 к ✅/❌ происходит атомарно — ты не увидишь их одновременно.
:::

:::tip
Если у бота нет прав добавлять реакции в группе, вызовы реакций тихо падают, а обработка сообщения продолжается нормально.
:::

## Промпты per‑channel

Назначай эфемерные системные промпты отдельным Telegram‑группам или темам форума. Промпт вставляется в каждый ход, но не сохраняется в истории транскрипта, поэтому изменения вступают в силу мгновенно.

```yaml
telegram:
  channel_prompts:
    "-1001234567890": |
      You are a research assistant. Focus on academic sources,
      citations, and concise synthesis.
    "42":  |
      This topic is for creative writing feedback. Be warm and
      constructive.
```

Ключи — это ID чатов (группы/супергруппы) или ID тем форума. Для групп‑форумов промпт темы переопределяет групповой промпт:

- Сообщение в теме `42` внутри группы `-1001234567890` → использует промпт темы `42`
- Сообщение в теме `99` (без явного ввода) → падает к групповому промпту `-1001234567890`
- Сообщение в группе без записи → промпт не применяется

Числовые YAML‑ключи автоматически нормализуются в строки.

## Устранение неполадок

| Проблема | Решение |
|----------|---------|
| Бот не отвечает вообще | Проверь правильность `TELEGRAM_BOT_TOKEN`. Посмотри логи `hermes gateway` на ошибки. |
| Бот отвечает «unauthorized» | Твой пользовательский ID не включён в `TELEGRAM_ALLOWED_USERS`. Проверь через @userinfobot. |
| Бот игнорирует сообщения группы | Скорее всего включён режим конфиденциальности. Отключи его (Шаг 3) или сделай бота администратором. **Не забудь удалить и добавить бота после изменения конфиденциальности.** |
| Голосовые сообщения не транскрибируются | Убедись, что STT доступен: установи `faster-whisper` для локальной транскрипции, либо задай `GROQ_API_KEY` / `VOICE_TOOLS_OPENAI_KEY` в `~/.hermes/.env`. |
| Голосовые ответы приходят как файлы, а не пузырьки | Установи `ffmpeg` (нужен для конвертации Edge TTS в Opus). |
| Токен бота отозван/некорректен | Сгенерируй новый токен через `/revoke`, затем `/newbot` или `/token` в BotFather. Обнови `.env`. |
| Веб‑хук не получает обновления | Убедись, что `TELEGRAM_WEBHOOK_URL` публично доступен (проверь `curl`). Проверь, что платформа/обратный прокси направляет входящий HTTPS‑трафик с порта URL к локальному порту, указанному в `TELEGRAM_WEBHOOK_PORT` (они могут различаться). Убедись, что включён SSL/TLS — Telegram отправляет только на HTTPS. Проверь правила firewall. |

## Подтверждение выполнения команд

Когда агент пытается выполнить потенциально опасную команду, он запрашивает подтверждение в чате:

> ⚠️ Эта команда потенциально опасна (рекурсивное удаление). Ответь «yes», чтобы подтвердить.

Ответь «yes»/«y», чтобы разрешить, или «no»/«n», чтобы отклонить.

## Интерактивные запросы (clarify)

Когда агент вызывает инструмент `clarify` — чтобы спросить, какой подход предпочесть, собрать обратную связь после задачи или уточнить перед важным решением — Telegram отображает вопрос с **inline‑кнопками**:

> ❓ Какой фреймворк использовать для дашборда?
>
> [1. Next.js] [2. Remix] [3. Astro]
> [✏️ Other (type answer)]

Нажми кнопку, чтобы ответить, или выбери **Other**, чтобы ввести свободный ответ (следующее твоё сообщение будет воспринято как ответ). Если у `clarify` нет предустановленных вариантов, кнопки не показываются, и просто фиксируется твой следующий ввод.

Настрой таймаут ответа через `agent.clarify_timeout` в `~/.hermes/config.yaml` (по умолчанию `600` сек). Если ты не ответишь вовремя, агент продолжит работу с «запасным» сообщением и не будет ждать бесконечно.

## Объём push‑уведомлений

Telegram посылает push‑уведомление на каждое сообщение, которое бот отправляет. При длинных ответах агента, включающих прогресс инструментов, стриминг‑обновления и статус‑колбэки, это быстро становится шумным. Адаптер Telegram имеет два режима уведомлений:

| Режим | Поведение |
|------|-----------|
| `important` (по умолчанию) | Уведомления только для **финальных ответов**, **запросов подтверждения** и **подтверждений slash‑команд**. Прогресс инструментов, стриминг‑чанки и статус‑сообщения отправляются с `disable_notification=true`. |
| `all` | Уведомление отправляется на каждое исходящее сообщение. Наследственное поведение; включай, если действительно хочешь слышать каждый вызов инструмента. |

Настрой в `~/.hermes/config.yaml`:

```yaml
display:
  platforms:
    telegram:
      notifications: important   # or "all"
```

Переопределить через переменную окружения (удобно для быстрого A/B‑тестинга):

```bash
HERMES_TELEGRAM_NOTIFICATIONS=all
```

Неизвестные значения логируются как предупреждение и переключаются на `important`.

## Статус‑сообщения, редактируемые на месте

Адаптер Telegram маршрутизирует повторяющиеся статус‑колбэки агента (например, «Compressing context…», «Calling tool…») через `send_or_update_status()`, который хранит кэш `{(chat_id, status_key) → message_id}` и **редактирует** существующее сообщение при последующих эмиссиях вместо создания новых. Разные `status_key` получают отдельные сообщения; разные чаты не конфликтуют. Если редактирование не удаётся (сообщение удалено пользователем или слишком старое для редактирования), кэш‑запись удаляется, и следующий эмит создаёт новое сообщение и заново кэширует его ID. Конфигурация не требуется — это стандартное поведение Telegram‑адаптера. Другие адаптеры, не реализующие `send_or_update_status`, просто используют обычный `send()`.

## Закрепление входящего сообщения пользователя на время хода агента

Когда пользователь отправляет сообщение, вызывающее ход агента, адаптер Telegram закрепляет это входящее сообщение на время обработки и снимает закрепление после завершения ответа — лёгкий визуальный индикатор того, что бот активно работает над запросом, а не игнорирует его. При закреплении используется `disable_notification=true`, чтобы не создавать лишних пингов. Конфигурация не требуется.

## Безопасность

:::warning
Всегда задавай `TELEGRAM_ALLOWED_USERS`, чтобы ограничить, кто может взаимодействовать с ботом. По умолчанию шлюз отклоняет всех пользователей как меру предосторожности.
:::

Никогда не публикуй токен бота. Если он скомпрометирован, сразу отзови его через команду `/revoke` в BotFather.

Для более подробной информации смотри [документацию по безопасности](/user-guide/security). Ты также можешь воспользоваться [DM‑парой](/user-guide/messaging#dm-pairing-alternative-to-allowlists) для более динамичного подхода к авторизации пользователей.
