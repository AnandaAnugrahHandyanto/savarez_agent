# Настройка Matrix

Hermes Agent интегрируется с Matrix, открытым федеративным протоколом обмена сообщениями. Matrix позволяет запускать собственный homeserver или использовать публичный, например matrix.org — в любом случае ты сохраняешь контроль над своей коммуникацией. Бот подключается через Python‑SDK `mautrix`, обрабатывает сообщения в конвейере Hermes Agent (включая использование инструментов, память и рассуждения) и отвечает в реальном времени. Поддерживаются текст, вложения файлов, изображения, аудио, видео и опциональное сквозное шифрование (E2EE).

Hermes работает с любым homeserver Matrix — Synapse, Conduit, Dendrite или matrix.org.

Перед настройкой — вот то, что интересует большинство людей: как себя ведёт Hermes после подключения.
## Как себя ведёт Hermes

| Context | Behavior |
|---------|----------|
| **DMs** | Hermes отвечает на каждое сообщение. `@mention` не требуется. Каждый DM имеет свою собственную сессию. Установи `MATRIX_DM_MENTION_THREADS=true`, чтобы начинать ветку, когда бот `@mentioned` в DM. |
| **Rooms** | По умолчанию Hermes требует `@mention` для ответа. Установи `MATRIX_REQUIRE_MENTION=false` или добавь ID комнат в `MATRIX_FREE_RESPONSE_ROOMS` для комнат со свободным ответом. Приглашения в комнаты принимаются автоматически. |
| **Threads** | Hermes поддерживает ветки Matrix (MSC3440). Если ты отвечаешь в ветке, Hermes сохраняет контекст ветки изолированным от основной ленты комнаты. Ветки, в которых бот уже участвовал, не требуют упоминания. |
| **Auto-threading** | По умолчанию Hermes автоматически создаёт ветку для каждого сообщения, на которое он отвечает в комнате. Это изолирует разговоры. Установи `MATRIX_AUTO_THREAD=false`, чтобы отключить. |
| **Shared rooms with multiple users** | По умолчанию Hermes изолирует историю сессий для каждого пользователя внутри комнаты. Два человека, общающиеся в одной комнате, не делят один транскрипт, если ты явно не отключишь это. |

:::tip
Бот автоматически присоединяется к комнатам при приглашении. Просто пригласи Matrix‑пользователя бота в любую комнату — он присоединится и начнёт отвечать.
:::

### Модель сессий в Matrix

По умолчанию:

- каждый DM получает свою собственную сессию
- каждая ветка получает своё собственное пространство имён сессии
- каждый пользователь в общей комнате получает свою собственную сессию внутри этой комнаты

Это контролируется в `config.yaml`:

```yaml
group_sessions_per_user: true
```

Установи значение `false` только если ты явно хочешь одну общую беседу для всей комнаты:

```yaml
group_sessions_per_user: false
```

Общие сессии могут быть полезны для совместной комнаты, но они также означают:

- пользователи делят рост контекста и затраты токенов
- длительная задача одного человека с большим количеством инструментов может «раздуть» контекст всех остальных
- запущенный процесс одного человека может прервать последующее действие другого человека в той же комнате

### Настройки упоминаний и ветвления

Ты можешь настроить поведение упоминаний и автоматического ветвления через переменные окружения или `config.yaml`:

```yaml
matrix:
  require_mention: true           # Require @mention in rooms (default: true)
  free_response_rooms:            # Rooms exempt from mention requirement
    - "!abc123:matrix.org"
  auto_thread: true               # Auto-create threads for responses (default: true)
  dm_mention_threads: false       # Create thread when @mentioned in DM (default: false)
```

Или через переменные окружения:

```bash
MATRIX_REQUIRE_MENTION=true
MATRIX_FREE_RESPONSE_ROOMS=!abc123:matrix.org,!def456:matrix.org
MATRIX_AUTO_THREAD=true
MATRIX_DM_MENTION_THREADS=false
MATRIX_REACTIONS=true          # default: true — emoji reactions during processing
```

:::tip Отключение реакций
`MATRIX_REACTIONS=false` отключает эмодзи‑реакции жизненного цикла обработки (👀/✅/❌), которые бот ставит на входящие сообщения. Полезно для комнат, где события реакций шумные или не поддерживаются всеми клиентами.
:::

:::note
Если ты обновляешься с версии, в которой не было `MATRIX_REQUIRE_MENTION`, бот ранее отвечал на все сообщения в комнатах. Чтобы сохранить это поведение, установи `MATRIX_REQUIRE_MENTION=false`.
:::

Это руководство проведёт тебя через весь процесс настройки — от создания учётной записи бота до отправки первого сообщения.
## Шаг 1: Создай учётную запись бота

Тебе нужна учётная запись Matrix для бота. Существует несколько способов сделать это:

### Вариант A: Зарегистрировать на своём homeserver (рекомендовано)

Если ты управляешь собственным homeserver (Synapse, Conduit, Dendrite):

1. Используй admin API или инструмент регистрации, чтобы создать нового пользователя:

```bash
# Synapse example
register_new_matrix_user -c /etc/synapse/homeserver.yaml http://localhost:8008
```

2. Выбери имя пользователя, например `hermes` — полный ID пользователя будет `@hermes:your-server.org`.

### Вариант B: Использовать matrix.org или другой публичный homeserver

1. Перейди на [Element Web](https://app.element.io) и создай новую учётную запись.
2. Выбери имя пользователя для бота (например, `hermes-bot`).

### Вариант C: Использовать свою собственную учётную запись

Ты также можешь запускать Hermes от своего пользователя. В этом случае бот будет публиковать сообщения от твоего имени — удобно для персональных помощников.
## Шаг 2: Получить токен доступа

Hermes нужен токен доступа для аутентификации на homeserver. У тебя есть два варианта:

### Вариант A: Токен доступа (рекомендовано)

Самый надёжный способ получить токен:

**Через Element:**
1. Войди в [Element](https://app.element.io) под учётной записью бота.
2. Перейди в **Settings** → **Help & About**.
3. Прокрути вниз и разверни **Advanced** — токен доступа будет отображён там.
4. **Скопируй его сразу.**

**Через API:**

```bash
curl -X POST https://your-server/_matrix/client/v3/login \
  -H "Content-Type: application/json" \
  -d '{
    "type": "m.login.password",
    "user": "@hermes:your-server.org",
    "password": "your-password"
  }'
```

В ответе будет поле `access_token` — скопируй его.

:::warning[Храни токен доступа в безопасности]
Токен доступа предоставляет полный доступ к учётной записи бота в Matrix. Никогда не публикуй его и не добавляй в репозиторий Git. Если он будет скомпрометирован, отозови его, завершив все сессии этого пользователя.
:::

### Вариант B: Вход по паролю

Вместо токена доступа можно передать Hermes идентификатор пользователя бота и пароль. Hermes выполнит вход автоматически при запуске. Это проще, но пароль будет храниться в файле `.env`.

```bash
MATRIX_USER_ID=@hermes:your-server.org
MATRIX_PASSWORD=your-password
```
## Шаг 3: Найди свой Matrix User ID

Hermes Agent использует твой Matrix User ID, чтобы контролировать, кто может взаимодействовать с ботом. Matrix User ID имеет формат `@username:server`.

Чтобы найти его:

1. Открой [Element](https://app.element.io) (или любой другой предпочитаемый клиент Matrix).
2. Нажми на свой аватар → **Settings**.
3. Твой Matrix User ID отображается в верхней части профиля (например, `@alice:matrix.org`).

:::tip
Matrix User ID всегда начинается с `@` и содержит `:` с именем сервера. Например: `@alice:matrix.org`, `@bob:your-server.com`.
:::
## Шаг 4: Настройка Hermes Agent

### Вариант A: Интерактивная настройка (рекомендовано)

Запусти команду интерактивной настройки:

```bash
hermes gateway setup
```

Выбери **Matrix**, когда будет предложено, затем укажи URL твоего homeserver, токен доступа (или ID пользователя + пароль) и разрешённые ID пользователей, когда будет запрошено.

### Вариант B: Ручная конфигурация

Добавь следующее в файл `~/.hermes/.env`:

**Использование токена доступа:**

```bash
# Required
MATRIX_HOMESERVER=https://matrix.example.org
MATRIX_ACCESS_TOKEN=***

# Optional: user ID (auto-detected from token if omitted)
# MATRIX_USER_ID=@hermes:matrix.example.org

# Security: restrict who can interact with the bot
MATRIX_ALLOWED_USERS=@alice:matrix.example.org

# Multiple allowed users (comma-separated)
# MATRIX_ALLOWED_USERS=@alice:matrix.example.org,@bob:matrix.example.org
```

**Вход по паролю:**

```bash
# Required
MATRIX_HOMESERVER=https://matrix.example.org
MATRIX_USER_ID=@hermes:matrix.example.org
MATRIX_PASSWORD=***

# Security
MATRIX_ALLOWED_USERS=@alice:matrix.example.org
```

Опциональные настройки поведения в `~/.hermes/config.yaml`:

```yaml
group_sessions_per_user: true
```

- `group_sessions_per_user: true` изолирует контекст каждого участника внутри общих комнат

### Запуск шлюза

После настройки запусти шлюз Matrix:

```bash
hermes gateway
```

Бот подключится к твоему homeserver и начнёт синхронизацию в течение нескольких секунд. Отправь ему сообщение — либо в личных сообщениях, либо в комнате, в которой он находится — чтобы проверить.

:::tip
Ты можешь запустить `hermes gateway` в фоне или как службу systemd для постоянной работы. Смотри документацию по развертыванию для деталей.
:::
## Сквозное шифрование (E2EE)

Hermes поддерживает сквозное шифрование Matrix, поэтому ты можешь общаться с ботом в зашифрованных комнатах.

### Требования

Для E2EE требуется библиотека `mautrix` с extras для шифрования и библиотека C `libolm`:

```bash
# Install mautrix with E2EE support
pip install 'mautrix[encryption]'

# Or install with hermes extras
pip install 'hermes-agent[matrix]'
```

Тебе также нужно установить `libolm` в системе:

```bash
# Debian/Ubuntu
sudo apt install libolm-dev

# macOS
brew install libolm

# Fedora
sudo dnf install libolm-devel
```

### Включение E2EE

Добавь в свой `~/.hermes/.env`:

```bash
MATRIX_ENCRYPTION=true
```

Когда E2EE включено, Hermes:

- Хранит ключи шифрования в `~/.hermes/platforms/matrix/store/` (установки legacy: `~/.hermes/matrix/store/`)
- Загружает ключи устройства при первом подключении
- Автоматически расшифровывает входящие сообщения и шифрует исходящие
- Автоматически присоединяется к зашифрованным комнатам по приглашению

### Проверка кросс‑подписи (рекомендовано)

Если в твоей учётной записи Matrix включена кросс‑подпись (по умолчанию в Element), задай ключ восстановления, чтобы бот мог самостоятельно подписать своё устройство при запуске. Без этого другие клиенты Matrix могут отказать в обмене сеансами шифрования с ботом после ротации ключа устройства.

```bash
MATRIX_RECOVERY_KEY=EsT... your recovery key here
```

**Где его найти:** В Element открой **Settings** → **Security & Privacy** → **Encryption** → твой ключ восстановления (также называется «Security Key»). Это ключ, который тебя просили сохранить при первой настройке кросс‑подписи.

При каждом запуске, если установлен `MATRIX_RECOVERY_KEY`, Hermes импортирует ключи кросс‑подписи из защищённого хранилища сервера и подписывает текущее устройство. Это идемпотентно и безопасно оставлять включённым постоянно.

:::warning[Удаление крипто‑хранилища]
Если удалить `~/.hermes/platforms/matrix/store/crypto.db`, бот потеряет свою идентичность шифрования. Простая перезагрузка с тем же ID устройства **не** восстановит её полностью — сервер всё ещё хранит одноразовые ключи, подписанные старым ключом идентичности, и пиры не смогут установить новые сеансы Olm.

Hermes обнаруживает эту ситуацию при запуске и отказывается включать E2EE, выводя в лог: `device XXXX has stale one-time keys on the server signed with a previous identity key`.

**Самый простой способ восстановления:** сгенерировать новый токен доступа (что даст новое ID устройства без истории устаревших ключей). См. раздел «Обновление с предыдущей версии с E2EE» ниже. Это самый надёжный путь и он не требует вмешательства в базу данных сервера.

**Ручное восстановление** (для продвинутых — сохраняет тот же ID устройства):

1. Останови Synapse и удали старое устройство из его базы данных:
      ```bash
   sudo systemctl stop matrix-synapse
   sudo sqlite3 /var/lib/matrix-synapse/homeserver.db "
     DELETE FROM e2e_device_keys_json WHERE device_id = 'DEVICE_ID' AND user_id = '@hermes:your-server';
     DELETE FROM e2e_one_time_keys_json WHERE device_id = 'DEVICE_ID' AND user_id = '@hermes:your-server';
     DELETE FROM e2e_fallback_keys_json WHERE device_id = 'DEVICE_ID' AND user_id = '@hermes:your-server';
     DELETE FROM devices WHERE device_id = 'DEVICE_ID' AND user_id = '@hermes:your-server';
   "
   sudo systemctl start matrix-synapse
   ```
   Или через API администрирования Synapse (обрати внимание на URL‑закодированный ID пользователя):
      ```bash
   curl -X DELETE -H "Authorization: Bearer ADMIN_TOKEN" \
     'https://your-server/_synapse/admin/v2/users/%40hermes%3Ayour-server/devices/DEVICE_ID'
   ```
   Примечание: удаление устройства через API администрирования может также аннулировать связанный токен доступа. После этого, возможно, придётся сгенерировать новый токен.

2. Удали локальное крипто‑хранилище и перезапусти Hermes:
      ```bash
   rm -f ~/.hermes/platforms/matrix/store/crypto.db*
   # restart hermes
   ```

Другие клиенты Matrix (Element, matrix-commander) могут кэшировать старые ключи устройства. После восстановления введи `/discardsession` в Element, чтобы принудительно создать новый сеанс шифрования с ботом.
:::

:::info
Если `mautrix[encryption]` не установлен или `libolm` отсутствует, бот автоматически переходит в обычный (нешифрованный) режим. В логах будет предупреждение.
:::
## Домашняя комната

Ты можешь назначить «домашнюю комнату», в которую бот будет отправлять проактивные сообщения (например, вывод cron‑задач, напоминания и уведомления). Есть два способа её установить:

### Использование слеш‑команды

Введи `/sethome` в любой комнате Matrix, где присутствует бот. Эта комната станет домашней.

### Ручная настройка

Добавь следующее в свой `~/.hermes/.env`:

```bash
MATRIX_HOME_ROOM=!abc123def456:matrix.example.org
```
## Белый список комнат (`allowed_rooms`)

Ограничивает бота фиксированным набором комнат Matrix. Когда параметр установлен, бот **отвечает только** в комнатах, чьи ID присутствуют в списке — сообщения из любых других комнат игнорируются без уведомления, даже если бот упомянут.

**Прямые чаты (direct chat rooms) исключаются** из этого фильтра, поэтому уполномоченные пользователи всегда могут связаться с ботом один‑на‑один.

```yaml
matrix:
  allowed_rooms:
    - "!abc123def456:matrix.example.org"
    - "!opsroom789:matrix.example.org"
```

Или через переменную окружения (через запятую):

```bash
MATRIX_ALLOWED_ROOMS="!abc123def456:matrix.example.org,!opsroom789:matrix.example.org"
```

**Поведение**

- Пустой / не установлен → без ограничений (по умолчанию).
- Не пустой → ID комнаты должен быть в списке. Проверка выполняется **до** любой другой фильтрации (требование упоминания, белый список отправителей и т.д.).
- Используй **внутренний ID** комнаты (`!abc...:server`), а не её алиас (`#room:server`). Внутренний ID комнаты можно найти в Element через **Комната → Настройки → Расширенные**.

См. также: [разделение slash‑команд админа/пользователя](../../reference/slash-commands.md#permissions-and-adminuser-split).


:::tip
Чтобы найти ID комнаты: в Element открой комнату → **Settings** → **Advanced** → там отображается **Internal room ID** (начинается с `!`).
:::
## Устранение неполадок

### Бот не отвечает на сообщения

**Причина**: Бот не присоединился к комнате, или `MATRIX_ALLOWED_USERS` не содержит ваш идентификатор пользователя.

**Решение**: Пригласи бота в комнату — он автоматически присоединяется по приглашению. Убедись, что твой идентификатор пользователя присутствует в `MATRIX_ALLOWED_USERS` (используй полный формат `@user:server`). Перезапусти **gateway**.

### Бот присоединяется к комнатам, но молча отбрасывает каждое сообщение (смещение часов)

**Причина**: Системные часы хоста опережают реальное время. Адаптер Matrix применяет фильтр «grace period» в 5 секунд при запуске (`event_ts < startup_ts - 5`), чтобы игнорировать события, воспроизведённые из начальной синхронизации. Когда часы опережают реальное время, каждое входящее событие выглядит «старее запуска» и отбрасывается до попадания в обработчик сообщений — бот выглядит подключённым, но никогда не отвечает. См. [#12614](https://github.com/NousResearch/hermes-agent/issues/12614).

**Симптом**: В журнале **gateway** появляется `Matrix: dropped N live events as 'too old' more than 30s after startup`.

**Решение**: Синхронизируй часы хоста с NTP и перезапусти бота:

```bash
# Debian/Ubuntu
sudo timedatectl set-ntp true
timedatectl status   # confirm "System clock synchronized: yes"

# macOS
sudo sntp -sS time.apple.com
```

### «Failed to authenticate» / «whoami failed» при запуске

**Причина**: Неправильный токен доступа или URL homeserver.

**Решение**: Убедись, что `MATRIX_HOMESERVER` указывает на твой homeserver (включи `https://`, без завершающего слеша). Проверь, что `MATRIX_ACCESS_TOKEN` действителен — попробуй его с помощью curl:

```bash
curl -H "Authorization: Bearer YOUR_TOKEN" \
  https://your-server/_matrix/client/v3/account/whoami
```

Если запрос возвращает информацию о твоём пользователе, токен действителен. Если возвращается ошибка, сгенерируй новый токен.

### Ошибка «mautrix not installed»

**Причина**: Пакет Python `mautrix` не установлен.

**Решение**: Установи его:

```bash
pip install 'mautrix[encryption]'
```

Или с extras Hermes:

```bash
pip install 'hermes-agent[matrix]'
```

### Ошибки шифрования / «could not decrypt event»

**Причина**: Отсутствуют ключи шифрования, не установлен `libolm` или устройство бота не доверено.

**Решение**:
1. Убедись, что `libolm` установлен в системе (см. раздел E2EE выше).
2. Убедись, что в твоём `.env` установлено `MATRIX_ENCRYPTION=true`.
3. В клиенте Matrix (Element) открой профиль бота → **Sessions** → проверь/доверяй устройству бота.
4. Если бот только что присоединился к зашифрованной комнате, он может расшифровывать только сообщения, отправленные *после* его присоединения. Более ранние сообщения недоступны.

### Обновление с предыдущей версии с E2EE

:::tip
Если ты также вручную удалил `crypto.db`, см. предупреждение «Deleting the crypto store» в разделе E2EE выше — есть дополнительные шаги для очистки устаревших одноразовых ключей с homeserver.
:::

Если ты ранее использовал Hermes с `MATRIX_ENCRYPTION=true` и обновляешься до версии, использующей новое хранилище криптографии на базе SQLite, идентичность шифрования бота изменилась. Твой клиент Matrix (Element) может кэшировать старые ключи устройства и отказываться делиться с ботом сеансами шифрования.

**Симптомы**: Бот подключается и в журналах показывает «E2EE enabled», но все сообщения отображаются как «could not decrypt event», и бот не отвечает.

**Что происходит**: Состояние шифрования из предыдущего бэкенда `matrix-nio` или основанного на сериализации `mautrix` несовместимо с новым SQLite‑хранилищем. Бот создаёт новую идентичность шифрования, но твой клиент Matrix всё ещё хранит старые ключи и не будет делиться сеансом шифрования с устройством, у которого ключи изменились. Это функция безопасности Matrix — клиенты рассматривают изменённые ключи устройства как подозрительные.

**Решение** (однократная миграция):

1. **Создай новый токен доступа**, чтобы получить новое ID устройства. Самый простой способ:

      ```bash
   curl -X POST https://your-server/_matrix/client/v3/login \
     -H "Content-Type: application/json" \
     -d '{
       "type": "m.login.password",
       "identifier": {"type": "m.id.user", "user": "@hermes:your-server.org"},
       "password": "***",
       "initial_device_display_name": "Hermes Agent"
     }'
   ```

   Скопируй новый `access_token` и обнови `MATRIX_ACCESS_TOKEN` в `~/.hermes/.env`.

2. **Удали старое состояние шифрования**:

      ```bash
   rm -f ~/.hermes/platforms/matrix/store/crypto.db
   rm -f ~/.hermes/platforms/matrix/store/crypto_store.*
   ```

3. **Установи свой ключ восстановления** (если используешь кросс‑подпись — это делают большинство пользователей Element). Добавь в `~/.hermes/.env`:

      ```bash
   MATRIX_RECOVERY_KEY=EsT... your recovery key here
   ```

   Это позволит боту самоподписаться кросс‑подписными ключами при запуске, и Element сразу доверит новому устройству. Без этого Element может увидеть новое устройство как непроверенное и отказаться делиться сеансами шифрования. Найди свой ключ восстановления в Element в **Settings** → **Security & Privacy** → **Encryption**.

4. **Принуди клиент Matrix обновить сеанс шифрования**. В Element открой личный чат с ботом и введи `/discardsession`. Это заставит Element создать новый сеанс шифрования и поделиться им с новым устройством бота.

5. **Перезапусти gateway**:

      ```bash
   hermes gateway run
   ```

   Если установлен `MATRIX_RECOVERY_KEY`, в журналах должно появиться `Matrix: cross-signing verified via recovery key`.

6. **Отправь новое сообщение**. Бот должен расшифровать его и ответить нормально.

:::note
После миграции сообщения, отправленные *до* обновления, расшифровать нельзя — старые ключи шифрования удалены. Это затрагивает только переходный период; новые сообщения работают как обычно.
:::

:::tip
**Новые установки не затронуты.** Эта миграция нужна только если у тебя уже была рабочая настройка E2EE в предыдущей версии Hermes и ты обновляешься.

**Зачем нужен новый токен доступа?** Каждый токен доступа Matrix привязан к конкретному ID устройства. Повторное использование того же ID устройства с новыми ключами шифрования приводит к тому, что другие клиенты Matrix не доверяют устройству (они видят изменённые ключи как потенциальную угрозу). Новый токен доступа создаёт новый ID устройства без истории устаревших ключей, поэтому другие клиенты сразу ему доверяют.
:::
## Режим прокси (E2EE на macOS)

Matrix E2EE требует `libolm`, который не компилируется на macOS ARM64 (Apple Silicon). Дополнение `hermes-agent[matrix]` доступно только для Linux. Если ты используешь macOS, режим прокси позволяет запускать E2EE в контейнере Docker на Linux‑VM, тогда как сам агент работает нативно на macOS с полным доступом к локальным файлам, памяти и навыкам.

### Как это работает

```
macOS (Host):
  └─ hermes gateway
       ├─ api_server adapter ← listens on 0.0.0.0:8642
       ├─ AIAgent ← single source of truth
       ├─ Sessions, memory, skills
       └─ Local file access (Obsidian, projects, etc.)

Linux VM (Docker):
  └─ hermes gateway (proxy mode)
       ├─ Matrix adapter ← E2EE decryption/encryption
       └─ HTTP forward → macOS:8642/v1/chat/completions
           (no LLM API keys, no agent, no inference)
```

Контейнер Docker обрабатывает только протокол Matrix + E2EE. Когда приходит сообщение, контейнер расшифровывает его и пересылает текст хосту через обычный HTTP‑запрос. Хост запускает агент, вызывает инструменты, генерирует ответ и передаёт его обратно. Контейнер шифрует и отправляет ответ в Matrix. Все сессии объединены — CLI, Matrix, Telegram и любые другие платформы используют одну и ту же память и историю диалога.

### Шаг 1: Настройка хоста (macOS)

Включи API‑сервер, чтобы хост принимал входящие запросы от контейнера Docker.

Добавь в `~/.hermes/.env`:

```bash
API_SERVER_ENABLED=true
API_SERVER_KEY=your-secret-key-here
API_SERVER_HOST=0.0.0.0
```

- `API_SERVER_HOST=0.0.0.0` привязывает сервер ко всем интерфейсам, чтобы контейнер Docker мог к нему обратиться.
- `API_SERVER_KEY` требуется для привязки, отличной от loopback. Выбери надёжную случайную строку.
- API‑сервер работает на порту 8642 по умолчанию (можно изменить с помощью `API_SERVER_PORT`, если нужно).

Запусти шлюз:

```bash
hermes gateway
```

Ты должен увидеть, как API‑сервер стартует вместе с другими настроенными платформами. Проверь, доступен ли он из VM:

```bash
# From the Linux VM
curl http://<mac-ip>:8642/health
```

### Шаг 2: Настройка контейнера Docker (Linux VM)

Контейнеру нужны учётные данные Matrix и URL прокси. Ему НЕ нужны ключи API LLM.

**`docker-compose.yml`:**

```yaml
services:
  hermes-matrix:
    build: .
    environment:
      # Matrix credentials
      MATRIX_HOMESERVER: "https://matrix.example.org"
      MATRIX_ACCESS_TOKEN: "syt_..."
      MATRIX_ALLOWED_USERS: "@you:matrix.example.org"
      MATRIX_ENCRYPTION: "true"
      MATRIX_DEVICE_ID: "HERMES_BOT"

      # Proxy mode — forward to host agent
      GATEWAY_PROXY_URL: "http://192.168.1.100:8642"
      GATEWAY_PROXY_KEY: "your-secret-key-here"
    volumes:
      - ./matrix-store:/root/.hermes/platforms/matrix/store
```

**`Dockerfile`:**

```dockerfile
FROM python:3.11-slim

RUN apt-get update && apt-get install -y libolm-dev && rm -rf /var/lib/apt/lists/*
RUN pip install 'hermes-agent[matrix]'

CMD ["hermes", "gateway"]
```

Это полностью весь контейнер. Нет API‑ключей для OpenRouter, Anthropic или любого другого провайдера инференса.

### Шаг 3: Запуск обеих сторон

1. Сначала запусти шлюз на хосте:
      ```bash
   hermes gateway
   ```

2. Затем запусти контейнер Docker:
      ```bash
   docker compose up -d
   ```

3. Отправь сообщение в зашифрованной комнате Matrix. Контейнер расшифрует его, пересылает хосту и стримит ответ обратно.

### Справочник конфигурации

Режим прокси настраивается **со стороны контейнера** (тонкий шлюз):

| Setting | Description |
|---------|-------------|
| `GATEWAY_PROXY_URL` | URL удалённого Hermes API‑сервера (например, `http://192.168.1.100:8642`) |
| `GATEWAY_PROXY_KEY` | Токен‑Bearer для аутентификации (должен совпадать с `API_SERVER_KEY` на хосте) |
| `gateway.proxy_url` | То же, что `GATEWAY_PROXY_URL`, но в `config.yaml` |

На стороне хоста нужны:

| Setting | Description |
|---------|-------------|
| `API_SERVER_ENABLED` | Установить в `true` |
| `API_SERVER_KEY` | Токен‑Bearer (общий с контейнером) |
| `API_SERVER_HOST` | Установить в `0.0.0.0` для сетевого доступа |
| `API_SERVER_PORT` | Номер порта (по умолчанию: `8642`) |

### Работает на любой платформе

Режим прокси не ограничен Matrix. Любой адаптер платформы может его использовать — укажи `GATEWAY_PROXY_URL` в любой инстанции шлюза, и он будет пересылать запросы к удалённому агенту вместо локального. Это полезно в любой ситуации, когда адаптер платформы должен работать в отдельной среде от агента (сетевой изоляции, требования E2EE, ограничения ресурсов).

:::tip
Непрерывность сессии поддерживается через заголовок `X-Hermes-Session-Id`. API‑сервер хоста отслеживает сессии по этому ID, поэтому диалоги сохраняются между сообщениями так же, как при работе с локальным агентом.
:::

:::note
**Ограничения (v1):** Сообщения о прогрессе инструмента от удалённого агента не передаются обратно — пользователь видит только финальный стриминговый ответ, без отдельных вызовов инструментов. Запросы подтверждения опасных команд обрабатываются на хосте и не пересылаются пользователю Matrix. Эти ограничения могут быть устранены в будущих обновлениях.
:::

### Проблемы синхронизации / бот отстаёт

**Причина**: Длительные выполнения инструментов могут задерживать цикл синхронизации, либо homeserver работает медленно.

**Решение**: Цикл синхронизации автоматически повторяет попытку каждые 5 секунд при ошибке. Проверь логи Hermes на предмет предупреждений, связанных с синхронизацией. Если бот постоянно отстаёт, убедись, что у homeserver достаточно ресурсов.

### Бот офлайн

**Причина**: Шлюз Hermes не запущен или не смог подключиться.

**Решение**: Убедись, что запущен `hermes gateway`. Посмотри вывод терминала на наличие сообщений об ошибках. Частые причины: неверный URL homeserver, просроченный токен доступа, недоступный homeserver.

### «User not allowed» / бот игнорирует тебя

**Причина**: Твой User ID отсутствует в `MATRIX_ALLOWED_USERS`.

**Решение**: Добавь свой User ID в `MATRIX_ALLOWED_USERS` в `~/.hermes/.env` и перезапусти шлюз. Используй полный формат `@user:server`.
## Безопасность

:::warning
Всегда задавай `MATRIX_ALLOWED_USERS`, чтобы ограничить, кто может взаимодействовать с ботом. Без этого **gateway** по умолчанию отклоняет всех пользователей как меру предосторожности. Добавляй только идентификаторы пользователей, которым ты доверяешь — авторизованные пользователи получают полный доступ к возможностям агента, включая использование инструментов и доступ к системе.
:::

Для получения дополнительной информации о защите развертывания Hermes Agent смотри [Руководство по безопасности](../security.md).
## Примечания

- **Any homeserver**: Работает с Synapse, Conduit, Dendrite, matrix.org или любым homeserver, соответствующим спецификации Matrix. Не требуется конкретное программное обеспечение homeserver.
- **Federation**: Если ты используешь федеративный homeserver, бот может общаться с пользователями с других серверов — просто добавь их полные `@user:server` ID в `MATRIX_ALLOWED_USERS`.
- **Auto-join**: Бот автоматически принимает приглашения в комнаты и присоединяется. Он начинает отвечать сразу после присоединения.
- **Media support**: Hermes может отправлять и получать изображения, аудио, видео и вложения файлов. Медиа загружается на твой homeserver с помощью API репозитория контента Matrix.
- **Native voice messages (MSC3245)**: Адаптер Matrix автоматически помечает исходящие голосовые сообщения флагом `org.matrix.msc3245.voice`. Это означает, что ответы TTS и голосовое аудио отображаются как **нативные голосовые сообщения** в Element и других клиентах, поддерживающих MSC3245, а не как обычные вложения аудиофайлов. Входящие голосовые сообщения с флагом MSC3245 также правильно распознаются и направляются на транскрипцию речи в текст. Настройка не требуется — всё работает автоматически.
