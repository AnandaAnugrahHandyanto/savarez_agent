# Настройка Google Chat

Подключи Hermes Agent к Google Chat в качестве бота. Интеграция использует подписки Cloud Pub/Sub типа **pull** для входящих событий и Chat REST API для исходящих сообщений. Аналогичное удобство, как в Slack Socket Mode или Telegram long‑polling: твоему процессу Hermes не нужен публичный URL, туннель или TLS‑сертификат. Он подключается, аутентифицируется и слушает подписку — так же, как бот Telegram слушает токен.

> Запусти `hermes gateway setup` и выбери **Google Chat** для пошагового руководства.

:::note Workspace edition
Google Chat является частью Google Workspace. Ты можешь использовать эту интеграцию с личным Workspace (`@yourdomain.com`, зарегистрированным через Google) или рабочим Workspace, где у тебя есть права администратора для публикации приложения. Учётные записи только Gmail не могут размещать приложения Chat.
:::
## Обзор

| Компонент | Значение |
|-----------|----------|
| **Libraries** | `google-cloud-pubsub`, `google-api-python-client`, `google-auth` |
| **Входящий транспорт** | Cloud Pub/Sub pull subscription (no public endpoint) |
| **Исходящий транспорт** | Chat REST API (`chat.googleapis.com`) |
| **Аутентификация** | Service Account JSON with `roles/pubsub.subscriber` on the subscription |
| **Идентификация пользователя** | Chat resource names (`users/{id}`) + email |

---
## Шаг 1: Создай или выбери проект GCP

Тебе нужен проект Google Cloud, чтобы создать тему Pub/Sub. Если у тебя его нет, создай его в [console.cloud.google.com](https://console.cloud.google.com) — личные аккаунты получают бесплатный тариф, который без труда покрывает трафик бота.

Обрати внимание на идентификатор проекта (например, `my-chat-bot-123`). Ты будешь использовать его на каждом последующем шаге.

---
## Шаг 2: Включить два API

В консоли перейди в **APIs & Services → Library** и включи:

- **Google Chat API**
- **Cloud Pub/Sub API**

Оба бесплатны в пределах объёмов, генерируемых персональным ботом.
## Шаг 3: Создать сервисный аккаунт

**IAM & Admin → Service Accounts → Create Service Account.**

- Name: `hermes-chat-bot`
- Пропусти шаг «Grant this service account access to project». IAM на конкретной подписке — всё, что тебе нужно; **НЕ** предоставляй роли Pub/Sub уровня проекта.

После создания открой сервисный аккаунт, перейди в **Keys → Add Key → Create new key → JSON** и скачай файл. Сохрани его в месте, доступном только Hermes (например, `~/.hermes/google-chat-sa.json`, `chmod 600`).

:::caution Нет роли «Chat Bot Caller»
Распространённая ошибка — искать IAM‑роль, специфичную для Chat, и назначать её на уровне проекта. Такая роль не существует. Права чат‑бота получаются за счёт установки в пространство, а не через IAM. Всё, что нужен твоему сервисному аккаунту, — это роль подписчика Pub/Sub на подписку, которую ты создашь на следующем шаге.
:::
## Шаг 4: Создай тему Pub/Sub и подписку

**Pub/Sub → Topics → Create topic.**

- Topic ID: `hermes-chat-events`
- Оставь значения по умолчанию для остальных параметров.

После создания на странице детали темы есть вкладка **Subscriptions**. Создай подписку:

- Subscription ID: `hermes-chat-events-sub`
- Delivery type: **Pull**
- Message retention: **7 дней** (чтобы backlog сохранялся при перезапуске hermes)
- Оставь остальные параметры по умолчанию.

---
## Шаг 5: Привязка IAM к топику (критически важно)

На **топике** (а не на подписке) добавьте принципал IAM:

- Principal: `chat-api-push@system.gserviceaccount.com`
- Role: `Pub/Sub Publisher`

Без этого Google Chat не сможет публиковать события в ваш топик, и ваш бот никогда ничего не получит.

---
## Шаг 6: Привязка IAM к подписке

В **подписке** добавь свой собственный сервисный аккаунт в качестве принципала:

- Принципал: `hermes-chat-bot@<your-project>.iam.gserviceaccount.com`
- Роль: `Pub/Sub Subscriber`

Также предоставь роль `Pub/Sub Viewer` на той же подписке — Hermes вызывает
`subscription.get()` при запуске в качестве проверки доступности.
## Шаг 7: Настройка чат‑приложения

Перейди в **APIs & Services → Google Chat API → Configuration**.

- **App name**: укажи любое название, которое будут видеть пользователи (подойдёт «Hermes»).
- **Avatar URL**: любой публичный PNG (у Google есть несколько вариантов по умолчанию).
- **Description**: короткое предложение, отображаемое в каталоге приложений.
- **Functionality**: включи **Receive 1:1 messages** и **Join spaces and group conversations**.
- **Connection settings**: выбери **Cloud Pub/Sub**, укажи имя топика `projects/<your-project>/topics/hermes-chat-events`.
- **Visibility**: ограничь доступ только твоим рабочим пространством (или конкретными пользователями) — не публикуй приложение для всех, пока тестируешь.

Save.
## Шаг 8: Установить бота в тестовое пространство

Открой Google Chat в браузере. Начни личный чат с приложением, найдя его название в меню **+ New Chat**. При первом отправлении сообщения Google генерирует событие `ADDED_TO_SPACE`, которое Hermes использует для кэширования собственного `users/{id}` бота — это необходимо для фильтрации собственных сообщений.
## Шаг 9: Настройка Hermes

Добавь раздел Google Chat в `~/.hermes/.env`:

```bash
# Required
GOOGLE_CHAT_PROJECT_ID=my-chat-bot-123
GOOGLE_CHAT_SUBSCRIPTION_NAME=projects/my-chat-bot-123/subscriptions/hermes-chat-events-sub
GOOGLE_CHAT_SERVICE_ACCOUNT_JSON=/home/you/.hermes/google-chat-sa.json

# Authorization — paste the emails of people allowed to talk to the bot
GOOGLE_CHAT_ALLOWED_USERS=you@yourdomain.com,coworker@yourdomain.com

# Optional
GOOGLE_CHAT_HOME_CHANNEL=spaces/AAAA...         # default delivery destination for cron jobs
GOOGLE_CHAT_MAX_MESSAGES=1                      # Pub/Sub FlowControl; 1 serializes commands per session
GOOGLE_CHAT_MAX_BYTES=16777216                  # 16 MiB — cap on in-flight message bytes
```

Идентификатор проекта также может быть получен из `GOOGLE_CLOUD_PROJECT`, а путь к сервисному аккаунту — из `GOOGLE_APPLICATION_CREDENTIALS`. Используй любой из этих вариантов.

Установи зависимости, необходимые адаптеру Google Chat (в данный момент отдельный пакет Hermes не опубликован — установи их напрямую):

```bash
pip install google-cloud-pubsub google-api-python-client google-auth google-auth-oauthlib
```

Запусти шлюз:

```bash
hermes gateway
```

Ты должен увидеть строку журнала вроде:

```
[GoogleChat] Connected; project=my-chat-bot-123, subscription=<redacted>,
             bot_user_id=users/XXXX, flow_control(msgs=1, bytes=16777216)
```

Отправь «hola» в тестовом личном сообщении. Бот разместит маркер «Hermes is thinking…», а затем отредактирует то же сообщение реальным ответом — без «message deleted»‑тумбстонов.
## Форматирование и возможности

Google Chat отображает ограниченный набор markdown:

| Поддерживается | Не поддерживается |
|----------------|-------------------|
| `*bold*`, `_italic_`, `~strike~`, `` `code` `` | Заголовки, списки |
| Встроенные изображения через URL | Кнопки Interactive Card v2 (v1 этого шлюза) |
| Нативные вложения файлов (после `/setup-files` — см. Шаг 10) | Нативные голосовые заметки / круглые видеозаметки |

Системный запрос агента включает специфическую подсказку для Google Chat, чтобы он знал об этих ограничениях и избегал форматирования, которое не будет отображено.

Ограничение размера сообщения: 4000 символов на сообщение. Более длинные ответы агента автоматически разбиваются на несколько сообщений.

Поддержка веток: когда пользователь отвечает внутри ветки, Hermes обнаруживает `thread.name` и публикует свой ответ в той же ветке, поэтому каждая ветка получает отдельную сессию Hermes.

## Шаг 10: Доставка нативных вложений (опционально)

Из коробки бот может публиковать текст, встроенные изображения по URL и загружать карточки для аудио/видео/документов. Чтобы доставлять **нативные** вложения Chat — тот же файловый виджет, который появляется, когда человек перетаскивает файл — каждый пользователь авторизует бота один раз через OAuth‑поток для конкретного пользователя.

### Почему нужен отдельный поток

Конечная точка Google Chat `media.upload` жёстко отклоняет аутентификацию сервис‑аккаунтом:

> This method doesn't support app authentication with a service account.
> Authenticate with a user account.

Нет роли IAM или области, которые бы это исправляли. Конечная точка принимает только учётные данные пользователя. Поэтому бот должен действовать *от имени пользователя* каждый раз, когда загружает файл — конкретно от имени того пользователя, который запросил файл.

### Одноразовая настройка хоста

1. Перейди в **APIs & Services → Credentials** того же проекта GCP.
2. **Create credentials → OAuth client ID → Desktop app**.
3. Скачай JSON. Перемести его на хост, где работает Hermes.
4. На хосте зарегистрируй клиент в Hermes:

```bash
python -m plugins.platforms.google_chat.oauth \
    --client-secret /path/to/client_secret.json
```

Это запишет `~/.hermes/google_chat_user_client_secret.json`. Это общая инфраструктура — она идентифицирует OAuth‑*приложение*, а не отдельного пользователя. Достаточно одного файла на хост, независимо от количества пользователей, которые позже авторизуются.

### Авторизация per‑user (в чате)

Каждый пользователь запускает поток один раз в своём личном чате с ботом:

1. Он отправляет `/setup-files` боту. Бот отвечает статусом и следующими шагами.
2. Он отправляет `/setup-files start`. Бот отвечает OAuth‑URL.
3. Он открывает URL, нажимает **Allow** и видит, как браузер не может загрузить `http://localhost:1/?...&code=...`. Эта ошибка ожидаема — код авторизации находится в адресной строке.
4. Он копирует неудавшийся URL (или только значение `code=...`) и вставляет его обратно в чат как `/setup-files <PASTED_URL>`. Бот обменяет его на refresh‑токен.

Токен сохраняется в `~/.hermes/google_chat_user_tokens/<sanitized_email>.json`. Последующие запросы файлов в личном чате этого пользователя используют *его* токен, поэтому бот загружает файлы от его имени, а сообщение появляется в его пространстве.

Чтобы отозвать позже: `/setup-files revoke` удалит только токен этого пользователя. Токены остальных пользователей останутся нетронутыми.

### Область

Поток запрашивает ровно одну область: `chat.messages.create`. Это покрывает как `media.upload`, так и `messages.create`, который ссылается на загруженный `attachmentDataRef`. Нет доступа к Drive и более широким областям Chat — реализовано с минимальными привилегиями.

### Поведение в мультипользовательской среде

Если у задающего вопроса ещё нет per‑user токена, бот переходит к резервному одно‑пользовательскому токену `~/.hermes/google_chat_user_token.json` (если он есть от предшествующей установки без мультипользовательской поддержки). Если ни один токен недоступен, бот публикует чёткое текстовое сообщение с инструкцией запустить `/setup-files`.

Отзыв токена пользователем очищает только его собственный слот. Ошибка 401/403 от токена одного пользователя удаляет только кэш этого пользователя. Пользователи не мешают друг другу.
## Устранение неполадок

**Бот молчит после отправки «hola».**

1. Проверь, есть ли у подписки Pub/Sub недоставленные сообщения в консоли.
   Если есть, Hermes не аутентифицирован — проверь `GOOGLE_CHAT_SERVICE_ACCOUNT_JSON` и то, что сервисный аккаунт указан как `Pub/Sub Subscriber` у подписки.
2. Если у подписки ноль сообщений, Google Chat не публикует.
   Дважды проверь привязку IAM у **топика**: `chat-api-push@system.gserviceaccount.com` должен иметь `Pub/Sub Publisher`.
3. Проверь логи `hermes gateway` на наличие `[GoogleChat] Connected`. Если видишь
   `[GoogleChat] Config validation failed`, сообщение об ошибке подскажет, какую переменную окружения нужно исправить.

**Бот отвечает, но вместо ответа агента появляется сообщение об ошибке.**

Проверь логи на наличие `[GoogleChat] Pub/Sub stream died` — если такие сообщения повторяются, учётные данные сервисного аккаунта могли быть обновлены или подписка удалена. После 10 попыток адаптер помечает себя как фатальный.

**«403 Forbidden» при каждой исходящей отправке сообщения.**

Бот был удалён из пространства, либо ты отозвал его в консоли Chat API. Переустанови его в пространстве (следующее событие `ADDED_TO_SPACE` автоматически включит обмен сообщениями).

**Слишком много предупреждений «Rate limit hit».**

Стандартные квоты Chat API позволяют 60 сообщений на пространство в минуту. Если твой агент генерирует длительные потоковые ответы, превышающие эту норму, адаптер будет повторять попытки с экспоненциальным откатом — но пользователь всё равно будет видеть задержку. Рассмотри более лаконичные ответы или увеличение квоты в консоли GCP.

**Бот продолжает публиковать уведомление «/setup-files» вместо файлов.**

У задающего нет OAuth‑токена пользователя и нет резервного варианта. Выполни
`/setup-files` в их личных сообщениях и следуй Шагу 10. После завершения обмена следующий запрос файла будет загружен напрямую без перезапуска шлюза.

**`/setup-files start` сообщает «No client credentials stored on the host».**

Одноразовая настройка хоста не была выполнена. На терминале хоста, где запущен Hermes:

```bash
python -m plugins.platforms.google_chat.oauth \
    --client-secret /path/to/client_secret.json
```

Затем снова отправь `/setup-files start`.

**`/setup-files <PASTED_URL>` сообщает «Token exchange failed».**

Код аутентификации одноразовый и имеет короткий срок жизни (обычно несколько минут). Выполни
`/setup-files start`, чтобы получить свежий URL, и попробуй снова.
## Примечания по безопасности

- **Область доступа Service Account**: адаптер запрашивает области `chat.bot` и `pubsub`.
  IAM должен быть реальным механизмом принудительного контроля — предоставь своему SA минимум
  (`roles/pubsub.subscriber` + `roles/pubsub.viewer` на подписке), а не роли Pub/Sub уровня проекта или организации.
- **Защита загрузки вложений**: Hermes будет прикреплять токен SA к URL, чей хост находится в коротком списке разрешённых доменов, принадлежащих Google (`googleapis.com`, `drive.google.com`, `lh[3-6].googleusercontent.com` и несколько других). Любой другой хост отклоняется до выполнения HTTP‑запроса, чтобы защитить от сценариев SSRF, когда специально сформированное событие может перенаправить токен к сервису метаданных GCE.
- **Редактирование**: электронные адреса Service Account, пути подписок и пути топиков удаляются из вывода журналов скриптом `agent/redact.py`. Дамп отладочного конверта (`GOOGLE_CHAT_DEBUG_RAW=1`) проходит через тот же фильтр редактирования и записывается на уровне DEBUG.
- **Соответствие требованиям**: если планируешь подключать этого бота к регулируемому рабочему пространству (любая среда с политикой резидентности данных или управления ИИ), получи одобрение до первой установки.
- **Область OAuth пользователя**: поток вложений для отдельного пользователя запрашивает *только* `chat.messages.create` — минимум, покрывающий `media.upload` и последующий `messages.create`. Токены сохраняются в виде обычного JSON по пути `~/.hermes/google_chat_user_tokens/<sanitized_email>.json` (защита обеспечивается правами доступа к файловой системе — та же модель, что и у файла ключа SA). Каждый токен принадлежит ровно одному пользователю; отзыв ограничен этим пользователем.
