# WeCom Callback (Self-Built App)

Подключи Hermes к WeCom (Enterprise WeChat) как самостоятельное корпоративное приложение, используя модель callback/webhook.

:::info WeCom Bot vs WeCom Callback
Hermes поддерживает два режима интеграции с WeCom:
- **[WeCom Bot](wecom.md)** — бот‑стиль, подключается через WebSocket. Проще в настройке, работает в групповых чатах.
- **WeCom Callback** (эта страница) — самостоятельное приложение, получает зашифрованные XML‑callback’и. Отображается как полноценное приложение в боковой панели пользователей WeCom. Поддерживает маршрутизацию между несколькими корпорациями.
:::

См. также: [WeCom Bot](./wecom.md) для интеграции в стиле бота.

> Выполни `hermes gateway setup` и выбери **WeCom Callback** для пошагового руководства.

## Как это работает

1. Ты регистрируешь самостоятельное приложение в консоли администрирования WeCom.
2. WeCom отправляет зашифрованный XML на твой HTTP‑endpoint callback’а.
3. Hermes расшифровывает сообщение, ставит его в очередь агенту.
4. Немедленно отправляет подтверждение (тихо — пользователю ничего не показывается).
5. Агент обрабатывает запрос (обычно 3–30 минут).
6. Ответ доставляется проактивно через API WeCom `message/send`.

## Требования

- Корпоративный аккаунт WeCom с правами администратора.
- Python‑пакеты `aiohttp` и `httpx` (включены в стандартную установку).
- Публично доступный сервер для URL callback’а (или туннель типа ngrok).

## Настройка

### 1. Создай самостоятельное приложение в WeCom

1. Перейди в [WeCom Admin Console](https://work.weixin.qq.com/) → **Applications** → **Create App**.
2. Запиши свой **Corp ID** (отображается в верхней части консоли).
3. В настройках приложения создай **Corp Secret**.
4. Запиши **Agent ID** со страницы обзора приложения.
5. В разделе **Receive Messages** укажи URL callback’а:
   - URL: `http://YOUR_PUBLIC_IP:8645/wecom/callback`
   - Token: сгенерируй случайный токен (WeCom предоставляет).
   - EncodingAESKey: сгенерируй ключ (WeCom предоставляет).

### 2. Настрой переменные окружения

Добавь в файл `.env`:

```bash
WECOM_CALLBACK_CORP_ID=your-corp-id
WECOM_CALLBACK_CORP_SECRET=your-corp-secret
WECOM_CALLBACK_AGENT_ID=1000002
WECOM_CALLBACK_TOKEN=your-callback-token
WECOM_CALLBACK_ENCODING_AES_KEY=your-43-char-aes-key

# Optional
WECOM_CALLBACK_HOST=0.0.0.0
WECOM_CALLBACK_PORT=8645
WECOM_CALLBACK_ALLOWED_USERS=user1,user2
```

### 3. Запусти шлюз

```bash
hermes gateway
```

(Используй `hermes gateway start` только после того, как `hermes gateway install` зарегистрировал сервис systemd/launchd.)

Адаптер callback’а запускает HTTP‑сервер на указанном порту. WeCom проверит URL callback’а запросом GET, после чего начнёт отправлять сообщения запросами POST.

## Справочник конфигурации

Укажи эти параметры в `config.yaml` под `platforms.wecom_callback.extra` или через переменные окружения:

| Setting | Default | Description |
|---------|---------|-------------|
| `corp_id` | — | Corp ID предприятия WeCom (обязательно). |
| `corp_secret` | — | Corp secret для самостоятельного приложения (обязательно). |
| `agent_id` | — | Agent ID самостоятельного приложения (обязательно). |
| `token` | — | Токен проверки callback’а (обязательно). |
| `encoding_aes_key` | — | 43‑символьный AES‑ключ для шифрования callback’ов (обязательно). |
| `host` | `0.0.0.0` | Адрес привязки HTTP‑сервера callback’а. |
| `port` | `8645` | Порт HTTP‑сервера callback’а. |
| `path` | `/wecom/callback` | Путь URL для endpoint’а callback’а. |

## Маршрутизация между несколькими приложениями

Для предприятий, использующих несколько самостоятельных приложений (например, в разных отделах или дочерних компаниях), настрой список `apps` в `config.yaml`:

```yaml
platforms:
  wecom_callback:
    enabled: true
    extra:
      host: "0.0.0.0"
      port: 8645
      apps:
        - name: "dept-a"
          corp_id: "ww_corp_a"
          corp_secret: "secret-a"
          agent_id: "1000002"
          token: "token-a"
          encoding_aes_key: "key-a-43-chars..."
        - name: "dept-b"
          corp_id: "ww_corp_b"
          corp_secret: "secret-b"
          agent_id: "1000003"
          token: "token-b"
          encoding_aes_key: "key-b-43-chars..."
```

Пользователи идентифицируются как `corp_id:user_id`, чтобы избежать конфликтов между корпорациями. Когда пользователь отправляет сообщение, адаптер фиксирует, к какому приложению (корпорации) он относится, и направляет ответы через соответствующий токен доступа приложения.

## Управление доступом

Ограничь, какие пользователи могут взаимодействовать с приложением:

```bash
# Allowlist specific users
WECOM_CALLBACK_ALLOWED_USERS=zhangsan,lisi,wangwu

# Or allow all users
WECOM_CALLBACK_ALLOW_ALL_USERS=true
```

## Endpoint’ы

Адаптер предоставляет:

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/wecom/callback` | Хэндшейк проверки URL (WeCom отправляет его при настройке). |
| POST | `/wecom/callback` | Защищённый callback сообщения (WeCom отправляет сюда пользовательские сообщения). |
| GET | `/health` | Проверка работоспособности — возвращает `{"status": "ok"}`. |

## Шифрование

Все payload’ы callback’ов зашифрованы AES‑CBC с использованием EncodingAESKey. Адаптер обрабатывает:

- **Inbound**: расшифровка XML‑payload, проверка подписи SHA1.
- **Outbound**: ответы отправляются проактивно через API (ответный callback не шифруется).

Криптографическая реализация совместима с официальным SDK Tencent WXBizMsgCrypt.

## Ограничения

- **Нет потоковой передачи** — ответы приходят целыми сообщениями после завершения работы агента.
- **Нет индикаторов набора** — модель callback не поддерживает статус «печатает».
- **Только текст** — пока поддерживаются только текстовые сообщения на вход; ввод изображений, файлов и голоса ещё не реализован. Агент знает о возможностях исходящих медиа через подсказки платформы WeCom (изображения, документы, видео, голос).
- **Задержка ответа** — сессии агента занимают 3–30 минут; пользователи видят ответ только после завершения обработки.

## Устранение неполадок

**Проверка подписи не проходит.**
WeCom подписывает каждый запрос токеном **Token**, который ты указал в консоли администрирования. Наиболее частая причина — несовпадение токена, настроенного в Hermes, и токена, ожидаемого консолью. Скопируй заново **Token** и **EncodingAESKey** из консоли — их легко обрезать. Пробелы в значениях `~/.hermes/.env` вокруг `=` также ломают проверку подписи. После исправления перезапусти `hermes gateway run`.

**URL callback’а недоступен / шаг проверки не проходит.**
WeCom обращается к публичному URL, который ты указал. Убедись:
1. Твой обратный прокси / туннель перенаправляет `/wecom/callback` на порт шлюза.
2. URL в консоли администрирования использует HTTPS (WeCom отклоняет простой HTTP).
3. С внешней сети команда `curl -i https://<your-domain>/wecom/callback` возвращает что‑то, кроме таймаута (ответ 4xx без параметров запроса тоже подходит — значит слушатель доступен).

**Порт недоступен / слушатель не привязан.**
Проверь логи `hermes gateway run` на предмет привязанного хоста/порта. Если адаптер привязан к `127.0.0.1`, тебе нужен обратный прокси или туннель — серверы WeCom не могут достать localhost. Установи `extra.host: 0.0.0.0` в `config.yaml` (и `allowed_source_cidrs`, если открываешь напрямую) или оставь привязку к localhost и используй туннель, например Cloudflare Tunnel / nginx.
