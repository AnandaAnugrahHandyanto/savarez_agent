# ntfy

[ntfy](https://ntfy.sh/) — это простой HTTP‑based сервис публикаций‑подписок уведомлений. Он работает с бесплатным публичным сервером `ntfy.sh` или любой самохост‑инстанс и поддерживает любой клиент, способный делать HTTP‑запросы — телефоны, браузеры, скрипты, часы.

ntfy — отличный лёгкий канал push‑уведомлений для Hermes: подпишись на топик через [мобильное приложение ntfy](https://ntfy.sh/docs/subscribe/phone/), отправляй сообщения в топик, чтобы поговорить с агентом, получай ответ обратно на телефон.

> Запусти `hermes gateway setup` и выбери **ntfy** для пошагового руководства.

## Требования

- Имя топика (любая уникальная строка — `hermes-myname-2026` подойдёт)
- Установленное [мобильное приложение ntfy](https://ntfy.sh/docs/subscribe/phone/) и подписка на этот топик
- Необязательно: самохост‑инстанс ntfy или токен аккаунта `ntfy.sh` для приватных/зарезервированных топиков

Вот и всё. Никакого SDK, демонов, Node.js. Адаптер использует `httpx`, который уже является зависимостью Hermes.

## Настройка Hermes

### Через мастер‑настройку

```bash
hermes setup gateway
```

Выбери **ntfy** и следуй подсказкам.

### Через переменные окружения

Добавь следующее в `~/.hermes/.env`:

```
NTFY_TOPIC=hermes-myname-2026
NTFY_ALLOWED_USERS=hermes-myname-2026
NTFY_HOME_CHANNEL=hermes-myname-2026
```

| Variable | Required | Description |
|---|---|---|
| `NTFY_TOPIC` | Yes | Топик для подписки (входящие сообщения) |
| `NTFY_SERVER_URL` | Optional | URL сервера (по умолчанию: `https://ntfy.sh`) — укажи самохост‑ntfy для повышения приватности |
| `NTFY_TOKEN` | Optional | Bearer‑токен (например `tk_xyz`) или `user:pass` для Basic‑auth |
| `NTFY_PUBLISH_TOPIC` | Optional | Другой топик для исходящих ответов (по умолчанию `NTFY_TOPIC`) |
| `NTFY_MARKDOWN` | Optional | Установи `true`, чтобы отправлять ответы с заголовком `X-Markdown: true` |
| `NTFY_ALLOWED_USERS` | Recommended | Список имён топиков, разрешённых к использованию (через запятую; рассматриваются как ID пользователей; см. ниже) |
| `NTFY_ALLOW_ALL_USERS` | Optional | Установи `true`, чтобы разрешить всем публиковать — безопасно только для приватных топиков с токенами чтения |
| `NTFY_HOME_CHANNEL` | Optional | Топик по умолчанию для cron‑уведомлений |
| `NTFY_HOME_CHANNEL_NAME` | Optional | Человекочитаемая метка для домашнего канала |

## Модель идентификации — читай перед развертыванием

ntfy не имеет встроенной аутентификации пользователей. Поле `title` в опубликованном сообщении полностью контролируется издателем и может быть любым. Адаптер Hermes НЕ использует `title` для авторизации — иначе любой, кто знает топик, мог бы подделать разрешённого пользователя.

Вместо этого **само имя топика является идентификатором**. Каждое сообщение, опубликованное в топик, считается пришедшим от одного логического пользователя (топика). Поэтому `NTFY_ALLOWED_USERS` обычно содержит лишь имя самого топика — одноэлементный allowlist, который защищает весь канал.

Это значит, **любой, кто знает топик, может общаться с агентом**. Чтобы сделать это реальной границей доверия:

- **Самохост ntfy** и зафиксируй топик с помощью [Access Control](https://docs.ntfy.sh/config/#access-control). Только авторизованные клиенты с токеном чтения/записи смогут публиковать.
- Или **используй приватный топик на ntfy.sh** (для [зарезервированных топиков](https://docs.ntfy.sh/publish/#reserved-topics) нужен аккаунт) и защити его `NTFY_TOKEN`.
- Или **выбери длинное, трудно угадываемое имя топика** (`hermes-7d4f9c8b-2026`) и рассматривай его как общий секрет. Это самая простая настройка, но имя топика может утечь через логи или скриншоты.

Во всех случаях не передавай через ntfy конфиденциальные данные, если топик не защищён контролем доступа.

## Быстрый старт — общайся с агентом со смартфона

1. Выбери имя топика: `hermes-myname-2026`
2. На телефоне: установи [приложение ntfy](https://ntfy.sh/docs/subscribe/phone/), нажми **+**, введи `hermes-myname-2026`
3. На хосте:
   ```bash
   echo 'NTFY_TOPIC=hermes-myname-2026' >> ~/.hermes/.env
   echo 'NTFY_ALLOWED_USERS=hermes-myname-2026' >> ~/.hermes/.env
   hermes gateway restart
   ```
4. В приложении ntfy отправь сообщение в топик. Ответ агента придёт в виде push‑уведомления.

## Использование ntfy в cron‑задачах

После установки `NTFY_HOME_CHANNEL` cron‑задачи могут доставлять сообщения в ntfy:

```python
cronjob(
    action="create",
    schedule="every 1h",
    deliver="ntfy",          # uses NTFY_HOME_CHANNEL
    prompt="Check for alerts and summarise."
)
```

Или явно указать конкретный топик:

```python
send_message(target="ntfy:alerts-channel", message="Done!")
```

Это работает даже когда cron запускается вне процесса шлюза — плагин регистрирует `standalone_sender_fn`, открывающий собственное HTTP‑соединение.

## Самохостинг ntfy

Если нужен полный контроль:

```bash
# Docker
docker run -p 80:80 -it binwiederhier/ntfy serve

# Native
go install heckel.io/ntfy/v2@latest
ntfy serve
```

Затем укажи Hermes путь к нему:

```
NTFY_SERVER_URL=https://ntfy.mydomain.com
NTFY_TOPIC=hermes
NTFY_TOKEN=tk_abc123  # if you've set up access control
```

Самохостинг даёт контроль доступа к топикам, политики сохранения сообщений, вложения и эмодзи‑теги. См. [документацию сервера ntfy](https://docs.ntfy.sh/install/).

## Форматирование Markdown

Клиенты ntfy рендерят markdown, когда издатель задаёт заголовок `X-Markdown: true`. Чтобы включить это для исходящих ответов Hermes:

```
NTFY_MARKDOWN=true
```

Или в `config.yaml`:

```yaml
platforms:
  ntfy:
    extra:
      markdown: true
```

Мобильное приложение поддерживает подмножество CommonMark — жирный, курсив, списки, ссылки, блоки кода. Подробнее в [документации markdown ntfy](https://docs.ntfy.sh/publish/#markdown-formatting).

## Настройка только исходящих уведомлений (без входящих)

Если нужно, чтобы Hermes лишь *отправлял* уведомления в ntfy (сводки cron, алерты) и не принимал сообщения обратно, укажи одинаковое значение для `NTFY_TOPIC` и `NTFY_PUBLISH_TOPIC` и полностью опусти `NTFY_ALLOWED_USERS`. Без allowlist агент не будет отвечать на входящие сообщения — телефон получает только push‑уведомления, а диалог односторонний.

## Ограничения

- **Размер сообщения**: ntfy ограничивает тело сообщения 4096 символами. Hermes обрезает его с предупреждением, если превышено.
- **Нет индикаторов набора текста**: протокол не предоставляет такой возможности; `send_typing` — пустая операция.
- **Нет веток и вложений**: ntfy — просто push‑уведомления. Длинные ответы остаются в теле сообщения, нет разветвления по темам.
- **Отсутствие нативной идентификации пользователя**: см. раздел «модель идентификации» выше.

## Устранение неполадок

**Ошибка аутентификации / 401** — `NTFY_TOKEN` неверен, либо токен не имеет прав публикации/подписки на данный топик. Адаптер останавливает цикл переподключения при 401, и статус шлюза покажет `fatal: ntfy_unauthorized`. Исправь токен и перезапусти шлюз.

**Топик не найден / 404** — `NTFY_TOPIC` отсутствует на указанном сервере. На ntfy.sh топики создаются автоматически при первой публикации, поэтому 404 обычно означает, что ты указываешь самохост‑сервер без предварительно созданного топика. Адаптер останавливает цикл переподключения с `fatal: ntfy_topic_not_found`.

**Подключено, но нет сообщений** — Проверь, что `NTFY_ALLOWED_USERS` включает само имя топика. В модели идентификации ntfy топик — это пользователь; пустой allowlist отклонит всё.

**Переподключения каждые 60 с** — keepalive потока по умолчанию 55 с; у ntfy могут быть периодические сетевые сбои. Адаптер применяет экспоненциальный backoff (2 → 5 → 10 → 30 → 60 с) и сбрасывает его до 0, как только поток живёт ≥ 60 с.
