# Настройка Discord

Hermes Agent интегрируется с Discord в виде бота, позволяя тебе общаться с твоим AI‑ассистентом через личные сообщения или каналы сервера. Бот получает твои сообщения, обрабатывает их через pipeline Hermes Agent (включая использование инструментов, память и рассуждения) и отвечает в реальном времени. Он поддерживает текст, голосовые сообщения, вложения файлов и slash‑команды.

Перед настройкой — вот та часть, которая интересует большинство людей: как Hermes ведёт себя, когда уже находится на твоём сервере.
## Как себя ведёт Hermes

| Context | Behavior |
|---------|----------|
| **DMs** | Hermes отвечает на каждое сообщение. `@mention` не требуется. Каждый DM имеет свою сессию. |
| **Server channels** | По умолчанию Hermes отвечает только когда ты `@mention` его. Если отправляешь сообщение в канал без упоминания, Hermes игнорирует его. |
| **Free-response channels** | Ты можешь сделать конкретные каналы без упоминаний с помощью `DISCORD_FREE_RESPONSE_CHANNELS`, либо отключить упоминания глобально с `DISCORD_REQUIRE_MENTION=false`. Сообщения в этих каналах отвечаются inline — авто‑создание веток пропускается, и канал остаётся лёгким чатом. |
| **Threads** | Hermes отвечает в той же ветке. Правила упоминаний по‑прежнему применяются, если только эта ветка или её родительский канал не настроены как free‑response. Ветки изолированы от родительского канала для истории сессии. |
| **Shared channels with multiple users** | По умолчанию Hermes изолирует историю сессии для каждого пользователя внутри канала ради безопасности и ясности. Два человека, говорящие в одном канале, не делят один транскрипт, если ты явно не отключил эту изоляцию. |
| **Messages mentioning other users** | Когда `DISCORD_IGNORE_NO_MENTION` установлен в `true` (по умолчанию), Hermes молчит, если сообщение @упоминает других пользователей, но **не** упоминает бота. Это предотвращает вмешательство бота в разговоры, адресованные другим людям. Установи `false`, если хочешь, чтобы бот отвечал на все сообщения независимо от того, кого упомянули. Это действует только в серверных каналах, не в DM. |

:::tip
Если ты хочешь обычный канал помощи бота, где люди могут говорить с Hermes без тегания его каждый раз, добавь этот канал в `DISCORD_FREE_RESPONSE_CHANNELS`.
:::

### Модель шлюза Discord

Hermes в Discord — это не веб‑хук, отвечающий без состояния. Он проходит через полноценный шлюз сообщений, что значит, каждое входящее сообщение проходит через:

1. авторизацию (`DISCORD_ALLOWED_USERS`);
2. проверки упоминаний / свободного ответа;
3. поиск сессии;
4. загрузку транскрипта сессии;
5. обычное выполнение агента Hermes, включая инструменты, память и slash‑команды;
6. доставку ответа обратно в Discord.

Это важно, потому что поведение в загруженном сервере зависит как от маршрутизации Discord, так и от политики сессий Hermes.

### Модель сессий в Discord

По умолчанию:

- каждый DM получает свою сессию;
- каждая серверная ветка получает своё пространство имён сессии;
- каждый пользователь в общем канале получает свою сессию внутри этого канала.

То есть если Alice и Bob оба разговаривают с Hermes в `#research`, Hermes рассматривает их как отдельные беседы по умолчанию, хотя они используют один и тот же видимый канал Discord.

Это контролируется в `config.yaml`:

```yaml
group_sessions_per_user: true
```

Установи `false` только если ты явно хочешь одну общую беседу для всей комнаты:

```yaml
group_sessions_per_user: false
```

Общие сессии могут быть полезны для совместной работы, но они также означают:

- пользователи делят рост контекста и затраты токенов;
- задача одного человека, тяжёлая по инструментам, может «раздуть» контекст всех остальных;
- запущенный процесс одного человека может прервать последующий запрос другого в той же комнате.

### Прерывания и параллелизм

Hermes отслеживает работающих агентов по ключу сессии.

С настройкой по умолчанию `group_sessions_per_user: true`:

- прерывание Alice своей текущей задачи влияет только на её сессию в этом канале;
- Bob может продолжать говорить в том же канале, не наследуя историю Alice и не прерывая её задачу.

С `group_sessions_per_user: false`:

- вся комната делит один слот работающего агента для данного канала/ветки;
- последующие сообщения от разных людей могут прерывать друг друга или ставиться в очередь.

Это руководство проведёт тебя через весь процесс настройки — от создания бота в Developer Portal Discord до отправки первого сообщения.
## Шаг 1: Создай приложение Discord

1. Перейди в [Discord Developer Portal](https://discord.com/developers/applications) и войди в свой аккаунт Discord.
2. Нажми **New Application** в правом верхнем углу.
3. Введи название для своего приложения (например, «Hermes Agent») и согласись с Условиями обслуживания разработчиков.
4. Нажми **Create**.

Ты окажешься на странице **General Information**. Обрати внимание на **Application ID** — он понадобится позже для создания URL‑приглашения.
## Шаг 2: Создай бота

1. В левом сайдбаре нажми **Bot**.
2. Discord автоматически создаёт учётную запись бота для твоего приложения. Ты увидишь имя бота, которое можешь изменить.
3. В разделе **Authorization Flow**:
   - Установи **Public Bot** в **ON** — требуется для использования ссылки‑приглашения, предоставленной Discord (рекомендовано). Это позволяет вкладке **Installation** генерировать URL авторизации по умолчанию.
   - Оставь **Require OAuth2 Code Grant** в положении **OFF**.

:::tip
Ты можешь задать собственный аватар и баннер для бота на этой странице. Именно это увидят пользователи в Discord.
:::

:::info[Private Bot Alternative]
Если ты предпочитаешь держать бота приватным (Public Bot = OFF), **обязательно** используй метод **Manual URL** в Шаге 5 вместо вкладки **Installation**. Ссылка, предоставленная Discord, требует включённый Public Bot.
:::
## Шаг 3: Включение привилегированных интентов шлюза

Это самый критичный шаг во всей настройке. Без включённых правильных интентов ваш бот подключится к Discord, но **не сможет читать содержимое сообщений**.

На странице **Bot** прокрутите вниз до **Privileged Gateway Intents**. Вы увидите три переключателя:

| Интент | Назначение | Требуется? |
|--------|------------|------------|
| **Presence Intent** | Видеть статус пользователя онлайн/офлайн | Необязательно |
| **Server Members Intent** | Доступ к списку участников, разрешение имён пользователей | **Требуется** |
| **Message Content Intent** | Чтение текстового содержимого сообщений | **Требуется** |

**Включите оба интента Server Members Intent и Message Content Intent**, переключив их **ON**.

- Без **Message Content Intent** ваш бот получает события сообщений, но текст сообщения пуст — бот буквально не видит, что вы написали.
- Без **Server Members Intent** бот не может разрешать имена пользователей в списке разрешённых и может не определить, кто ему пишет.

:::warning[Это #1 причина, почему боты Discord не работают]
Если ваш бот онлайн, но никогда не отвечает на сообщения, **Message Content Intent** почти наверняка отключён. Вернитесь в [Developer Portal](https://discord.com/developers/applications), выберите ваше приложение → Bot → Privileged Gateway Intents и убедитесь, что **Message Content Intent** включён. Нажмите **Save Changes**.
:::

**Относительно количества серверов:**
- Если ваш бот находится **на менее чем 100 серверах**, вы можете свободно включать и отключать интенты.
- Если ваш бот находится **на 100 и более серверах**, Discord требует отправить заявку на верификацию для использования привилегированных интентов. Для личного использования это не актуально.

Нажмите **Save Changes** внизу страницы.
## Шаг 4: Получить токен бота

Токен бота — учётные данные, которые Hermes Agent использует для входа от имени твоего бота. На странице **Bot**:

1. В разделе **Token** нажми **Reset Token**.
2. Если у тебя включена двухфакторная аутентификация в аккаунте Discord, введи код 2FA.
3. Discord отобразит новый токен. **Сразу скопируй его.**

:::warning[Токен отображается только один раз]
Токен показывается лишь один раз. Если ты его потеряешь, придётся сбросить его и сгенерировать новый. Никогда не публикуй токен и не коммитить его в Git — любой, кто получит токен, получит полный контроль над твоим ботом.
:::

Сохрани токен в надёжном месте (например, в менеджере паролей). Он понадобится в Шаге 8.
## Шаг 5: Генерация URL‑приглашения

Тебе нужен OAuth2‑URL, чтобы пригласить бота на свой сервер. Есть два способа сделать это:

### Вариант A: Через вкладку **Installation** (рекомендовано)

:::note[Requires Public Bot]
Этот метод требует, чтобы **Public Bot** был включён (**ON**) на Шаге 2. Если ты отключил **Public Bot** (**OFF**), используй метод с ручным URL ниже.
:::

1. В левом боковом меню нажми **Installation**.
2. В разделе **Installation Contexts** включи **Guild Install**.
3. Для **Install Link** выбери **Discord Provided Link**.
4. В **Default Install Settings** для **Guild Install**:
   - **Scopes**: выбери `bot` и `applications.commands`
   - **Permissions**: выбери перечисленные ниже разрешения.

### Вариант B: Ручной URL

Ты можешь собрать URL‑приглашения вручную, используя следующий формат:

```
https://discord.com/oauth2/authorize?client_id=YOUR_APP_ID&scope=bot+applications.commands&permissions=274878286912
```

Замени `YOUR_APP_ID` на идентификатор приложения из Шага 1.

### Требуемые разрешения

Это минимальный набор разрешений, необходимых боту:

- **View Channels** — видеть каналы, к которым у него есть доступ
- **Send Messages** — отвечать на твои сообщения
- **Embed Links** — форматировать богатые ответы
- **Attach Files** — отправлять изображения, аудио и файлы‑результаты
- **Read Message History** — поддерживать контекст беседы

### Рекомендуемые дополнительные разрешения

- **Send Messages in Threads** — отвечать в ветках обсуждений
- **Add Reactions** — ставить реакции на сообщения для подтверждения

### Целочисленные значения разрешений

| Уровень | Permissions Integer | Что включено |
|--------|---------------------|--------------|
| Minimal | `117760` | View Channels, Send Messages, Read Message History, Attach Files |
| Recommended | `274878286912` | Всё выше перечисленное плюс Embed Links, Send Messages in Threads, Add Reactions |
## Шаг 6: Приглашение на твой сервер

1. Открой URL‑ссылку приглашения в браузере (из вкладки **Installation** или вручную сконструированный URL).
2. В выпадающем списке **Add to Server** выбери свой сервер.
3. Нажми **Continue**, затем **Authorize**.
4. При необходимости пройди CAPTCHA.

:::info
Тебе требуется право **Manage Server** на Discord‑сервере, чтобы пригласить бота. Если ты не видишь свой сервер в выпадающем списке, попроси администратора сервера использовать ссылку приглашения.
:::

После авторизации бот появится в списке участников твоего сервера (он будет отображаться как офлайн, пока ты не запустишь шлюз Hermes).
## Шаг 7: Найти свой Discord User ID

Hermes Agent использует твой Discord User ID, чтобы контролировать, кто может взаимодействовать с ботом. Чтобы найти его:

1. Открой Discord (десктопное приложение или веб‑версию).
2. Перейди в **Settings** → **Advanced** → включи **Developer Mode** (переключатель **ON**).
3. Закрой настройки.
4. Щёлкни правой кнопкой мыши по своему имени пользователя (в сообщении, в списке участников или в профиле) → **Copy User ID**.

Твой User ID — это длинное число, например `284102345871466496`.

:::tip
Режим разработчика также позволяет копировать **Channel IDs** и **Server IDs** тем же способом — щёлкни правой кнопкой мыши по названию канала или сервера и выбери **Copy ID**. Тебе понадобится идентификатор канала, если ты хочешь вручную задать домашний канал.
:::
## Шаг 8: Настройка Hermes Agent

### Вариант A: Интерактивная настройка (рекомендовано)

Запусти команду интерактивного мастера настройки:

```bash
hermes gateway setup
```

Выбери **Discord**, когда будет предложено, затем вставь токен бота и ID пользователя, когда будет запрошено.

### Вариант B: Ручная конфигурация

Добавь следующее в файл `~/.hermes/.env`:

```bash
# Required
DISCORD_BOT_TOKEN=your-bot-token
DISCORD_ALLOWED_USERS=284102345871466496

# Multiple allowed users (comma-separated)
# DISCORD_ALLOWED_USERS=284102345871466496,198765432109876543
```

Затем запусти `gateway`:

```bash
hermes gateway
```

Бот должен появиться в Discord в течение нескольких секунд. Отправь ему сообщение — в личных сообщениях или в канале, где он виден — чтобы проверить.

:::tip
Ты можешь запускать `hermes gateway` в фоне или как службу `systemd` для постоянной работы. См. документацию по развертыванию для деталей.
:::
## Справочник конфигурации

Поведение Discord контролируется двумя файлами: **`~/.hermes/.env`** для учётных данных и переключателей уровня окружения, и **`~/.hermes/config.yaml`** для структурированных настроек. Переменные окружения всегда имеют приоритет над значениями из `config.yaml`, если обе заданы.

### Переменные окружения (`.env`)

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `DISCORD_BOT_TOKEN` | **Yes** | — | Токен бота из [Discord Developer Portal](https://discord.com/developers/applications). |
| `DISCORD_ALLOWED_USERS` | **Yes** | — | Список ID пользователей Discord через запятую, которым разрешено взаимодействовать с ботом. Без этого **или** `DISCORD_ALLOWED_ROLES` шлюз отклонит всех пользователей. |
| `DISCORD_ALLOWED_ROLES` | No | — | Список ID ролей Discord через запятую. Любой участник с одной из этих ролей авторизован — семантика ИЛИ с `DISCORD_ALLOWED_USERS`. Автоматически включает **Server Members Intent** при подключении. Полезно, когда команды модерации часто меняются: новые модераторы получают доступ сразу после назначения роли, без необходимости обновлять конфиг. |
| `DISCORD_HOME_CHANNEL` | No | — | ID канала, в который бот отправляет проактивные сообщения (вывод cron, напоминания, уведомления). |
| `DISCORD_HOME_CHANNEL_NAME` | No | `"Home"` | Отображаемое имя домашнего канала в логах и статусе. |
| `DISCORD_COMMAND_SYNC_POLICY` | No | `"safe"` | Управляет синхронизацией нативных slash‑команд при старте. `"safe"` сравнивает существующие глобальные команды и обновляет только изменённые, пересоздавая команды, когда изменения метаданных Discord нельзя применить патчем. `"bulk"` сохраняет старое поведение `tree.sync()`. `"off"` полностью пропускает синхронизацию при старте. |
| `DISCORD_REQUIRE_MENTION` | No | `true` | При `true` бот отвечает в серверных каналах только когда его **@упомянут**. Установи `false`, чтобы отвечать на все сообщения во всех каналах. |
| `DISCORD_THREAD_REQUIRE_MENTION` | No | `false` | При `true` отключается упрощённый способ упоминания в ветке — ветки защищаются так же, как каналы, требуя **@упоминание** даже после того, как бот уже принял участие. Используй, когда в одной ветке работают несколько ботов и каждый должен реагировать только на явное **@упоминание**. |
| `DISCORD_FREE_RESPONSE_CHANNELS` | No | — | Список ID каналов через запятую, где бот отвечает без требования **@упоминания**, даже если `DISCORD_REQUIRE_MENTION` включён. |
| `DISCORD_IGNORE_NO_MENTION` | No | `true` | При `true` бот молчит, если сообщение **@упоминает** других пользователей, но **не** упоминает бота. Предотвращает вмешательство бота в разговоры, адресованные другим людям. Применяется только в серверных каналах, не в личных сообщениях. |
| `DISCORD_AUTO_THREAD` | No | `true` | При `true` автоматически создаёт новую ветку для каждого **@упоминания** в текстовом канале, так каждый разговор изолирован (подобно Slack). Сообщения, уже находящиеся в ветках или личных сообщениях, не затрагиваются. |
| `DISCORD_ALLOW_BOTS` | No | `"none"` | Управляет тем, как бот обрабатывает сообщения от других ботов Discord. `"none"` — игнорировать всех остальных ботов. `"mentions"` — принимать только те сообщения ботов, которые **@упоминают** Hermes. `"all"` — принимать сообщения от всех ботов. |
| `DISCORD_REACTIONS` | No | `true` | При `true` бот добавляет эмодзи‑реакции к сообщениям во время обработки (👀 при старте, ✅ при успехе, ❌ при ошибке). Установи `false`, чтобы полностью отключить реакции. |
| `DISCORD_IGNORED_CHANNELS` | No | — | Список ID каналов через запятую, где бот **никогда** не отвечает, даже при **@упоминании**. Имеет приоритет над всеми другими настройками каналов. |
| `DISCORD_ALLOWED_CHANNELS` | No | — | Список ID каналов через запятую. При задании бот **отвечает только** в этих каналах (плюс личные сообщения, если разрешены). Переопределяет `config.yaml` `discord.allowed_channels`. Комбинируй с `DISCORD_IGNORED_CHANNELS` для правил разрешения/запрета. |
| `DISCORD_NO_THREAD_CHANNELS` | No | — | Список ID каналов через запятую, где бот отвечает напрямую в канале, а не создаёт ветку. Имеет смысл только когда `DISCORD_AUTO_THREAD` включён. |
| `DISCORD_HISTORY_BACKFILL` | No | `true` | При `true` добавляет к пользовательскому сообщению недавнюю прокрутку канала (с момента последнего ответа бота), когда бот **@упомянут**. Восстанавливает контекст, который иначе был бы упущен при `require_mention`. Пропускается в личных сообщениях и свободных каналах. Установи `false`, чтобы отключить. |
| `DISCORD_HISTORY_BACKFILL_LIMIT` | No | `50` | Максимальное количество сообщений, которое сканируется назад при формировании блока backfill. На практике сканирование обычно останавливается раньше — на последнем собственном сообщении бота в канале. |
| `DISCORD_REPLY_TO_MODE` | No | `"first"` | Управляет поведением ссылки‑ответа: `"off"` — никогда не отвечать на оригинальное сообщение, `"first"` — ссылка‑ответ только на первый фрагмент сообщения (по умолчанию), `"all"` — ссылка‑ответ на каждый фрагмент. |
| `DISCORD_ALLOW_MENTION_EVERYONE` | No | `false` | При `false` (по умолчанию) бот не может упоминать `@everyone` или `@here`, даже если его ответ содержит эти токены. Установи `true`, чтобы вернуть возможность. См. [Контроль упоминаний](#mention-control) ниже. |
| `DISCORD_ALLOW_MENTION_ROLES` | No | `false` | При `false` (по умолчанию) бот не может упоминать роли `@role`. Установи `true`, чтобы разрешить. |
| `DISCORD_ALLOW_MENTION_USERS` | No | `true` | При `true` (по умолчанию) бот может упоминать отдельных пользователей по ID. |
| `DISCORD_ALLOW_MENTION_REPLIED_USER` | No | `true` | При `true` (по умолчанию) ответ на сообщение упоминает оригинального автора. |
| `DISCORD_PROXY` | No | — | URL прокси для соединений Discord (HTTP, WebSocket, REST). Переопределяет `HTTPS_PROXY`/`ALL_PROXY`. Поддерживает схемы `http://`, `https://` и `socks5://`. |
| `DISCORD_ALLOW_ANY_ATTACHMENT` | No | `false` | При `true` бот принимает вложения любого типа файлов (не только из встроенного списка PDF/text/zip/office). Неизвестные типы кэшируются на диск и передаются агенту как локальный путь с MIME `application/octet-stream`, чтобы их можно было проанализировать через `terminal` / `read_file` / `ffprobe` и т.д. |
| `DISCORD_MAX_ATTACHMENT_BYTES` | No | `33554432` | Максимальный размер в байтах для одного вложения, которое шлюз загрузит и кэширует. По умолчанию — 32 MiB. Установи `0` для снятия ограничения (вложения хранятся в памяти во время записи, поэтому отсутствие лимита приводит к реальному потреблению памяти). |
| `HERMES_DISCORD_TEXT_BATCH_DELAY_SECONDS` | No | `0.6` | Окно задержки, которое адаптер ждёт перед сбросом накопленного текстового фрагмента. Полезно для сглаживания потокового вывода. |
| `HERMES_DISCORD_TEXT_BATCH_SPLIT_DELAY_SECONDS` | No | `2.0` | Задержка между разделёнными фрагментами, когда одно сообщение превышает лимит длины Discord. |

### Файл конфигурации (`config.yaml`)

Раздел `discord` в `~/.hermes/config.yaml` отражает переменные окружения выше. Настройки из `config.yaml` применяются как значения по умолчанию — если соответствующая переменная окружения уже задана, переменная окружения имеет приоритет.

```yaml
# Discord-specific settings
discord:
  require_mention: true           # Require @mention in server channels
  thread_require_mention: false   # If true, require @mention in threads too (multi-bot threads)
  free_response_channels: ""      # Comma-separated channel IDs (or YAML list)
  auto_thread: true               # Auto-create threads on @mention
  reactions: true                 # Add emoji reactions during processing
  ignored_channels: []            # Channel IDs where bot never responds
  no_thread_channels: []          # Channel IDs where bot responds without threading
  history_backfill: true          # Prepend recent channel scrollback on mention (default: true)
  history_backfill_limit: 50      # Max messages to scan backwards (default: 50)
  channel_prompts: {}             # Per-channel ephemeral system prompts
  allow_mentions:                 # What the bot is allowed to ping (safe defaults)
    everyone: false               # @everyone / @here pings (default: false)
    roles: false                  # @role pings (default: false)
    users: true                   # @user pings (default: true)
    replied_user: true            # reply-reference pings the author (default: true)

# Session isolation (applies to all gateway platforms, not just Discord)
group_sessions_per_user: true     # Isolate sessions per user in shared channels
```

#### `discord.require_mention`

**Type:** boolean — **Default:** `true`

Когда включено, бот отвечает в серверных каналах только при прямом **@упоминании**. Личные сообщения всегда получают ответ независимо от этой настройки.

#### `discord.thread_require_mention`

**Type:** boolean — **Default:** `false`

По умолчанию, после того как бот принял участие в ветке (автоматически созданной при **@упоминании** или после ответа), он продолжает отвечать на каждое последующее сообщение в этой ветке без необходимости повторного **@упоминания**. Это правильный вариант по умолчанию для одно‑на‑одно разговоров.

В **ветках с несколькими ботами**, где пользователи обращаются к одному боту за раз, такое поведение становится проблемой — каждый другой бот в ветке тоже реагирует на каждое сообщение, тратя кредиты и спамя канал. Установи `thread_require_mention: true`, чтобы отключить упрощённый способ упоминания в ветке и защищать ветки так же, как каналы. Явные **@упоминания** продолжают работать как прежде.

```yaml
discord:
  require_mention: true
  thread_require_mention: true    # multi-bot setup
```

#### `discord.free_response_channels`

**Type:** string or list — **Default:** `""`

ID каналов, где бот отвечает на все сообщения без требования **@упоминания**. Принимает либо строку с запятыми, либо список YAML:

```yaml
# String format
discord:
  free_response_channels: "1234567890,9876543210"

# List format
discord:
  free_response_channels:
    - 1234567890
    - 9876543210
```

Если родительский канал ветки находится в этом списке, ветка также становится свободной от упоминаний.

Свободные каналы также **пропускают автоматическое создание веток** — бот отвечает в строке, а не создаёт новую ветку для каждого сообщения. Это сохраняет канал пригодным для лёгкого чата. Если нужен режим ветвления, не включай канал в список свободных (используй обычный поток **@упоминания**).

#### `discord.auto_thread`

**Type:** boolean — **Default:** `true`

Когда включено, каждое **@упоминание** в обычном текстовом канале автоматически создаёт новую ветку для разговора. Это сохраняет основной канал чистым и даёт каждому разговору собственную изолированную историю сессии. После создания ветки последующие сообщения в ней не требуют **@упоминания** — бот уже участвует. Установи [`thread_require_mention`](#discordthread_require_mention) в `true`, чтобы отключить этот упрощённый путь в ветках для сценариев с несколькими ботами.

Сообщения, отправленные в уже существующих ветках или личных сообщениях, не затрагиваются этой настройкой. Каналы, указанные в `discord.free_response_channels` или `discord.no_thread_channels`, также обходят авто‑создание веток и получают ответы в строке.

#### `discord.reactions`

**Type:** boolean — **Default:** `true`

Управляет тем, добавляет ли бот эмодзи‑реакции к сообщениям в качестве визуального отклика:
- 👀 добавляется, когда бот начинает обработку сообщения;
- ✅ добавляется при успешной доставке ответа;
- ❌ добавляется при ошибке обработки.

Отключи, если реакции отвлекают или у роли бота нет права **Add Reactions**.

#### `discord.ignored_channels`

**Type:** string or list — **Default:** `[]`

ID каналов, где бот **никогда** не отвечает, даже при прямом **@упоминании**. Это имеет высший приоритет — если канал находится в этом списке, бот молча игнорирует все сообщения там, независимо от `require_mention`, `free_response_channels` или любой другой настройки.

```yaml
# String format
discord:
  ignored_channels: "1234567890,9876543210"

# List format
discord:
  ignored_channels:
    - 1234567890
    - 9876543210
```

Если родительский канал ветки находится в этом списке, сообщения в этой ветке также игнорируются.

#### `discord.no_thread_channels`

**Type:** string or list — **Default:** `[]`

ID каналов, где бот отвечает напрямую в канале вместо автоматического создания ветки. Имеет смысл только когда `auto_thread` включён (по умолчанию). В этих каналах бот отвечает в строке, как обычное сообщение, а не создаёт новую ветку.

```yaml
discord:
  no_thread_channels:
    - 1234567890  # Bot responds inline here
```

Полезно для каналов, предназначенных исключительно для взаимодействия с ботом, где ветки создавали бы лишний шум.

#### `discord.channel_prompts`

**Type:** mapping — **Default:** `{}`

Эфемерные системные подсказки per‑channel, которые внедряются на каждый ход в соответствующем канале или ветке Discord без сохранения в историю транскрипта.

```yaml
discord:
  channel_prompts:
    "1234567890": |
      This channel is for research tasks. Prefer deep comparisons,
      citations, and concise synthesis.
    "9876543210": |
      This forum is for therapy-style support. Be warm, grounded,
      and non-judgmental.
```

Поведение:
- Приоритет имеют точные совпадения ID ветки/канала.
- Если сообщение приходит в ветку или форум и для этой ветки нет явного entry, Hermes переходит к ID родительского канала/форума.
- Подсказки применяются эфемерно во время выполнения, поэтому их изменение сразу влияет на будущие ходы без переписывания прошлой истории сессии.

#### `discord.history_backfill`

**Type:** boolean — **Default:** `true`

Когда включено, бот восстанавливает пропущенные сообщения канала при каждом **@упоминании**. При `require_mention: true` бот обрабатывает только сообщения, которые явно его упоминают — всё остальное в канале невидимо для транскрипта сессии. Backfill сканирует назад по недавней истории канала, собирая сообщения между последним ответом бота и текущим упоминанием, и включает их как контекст.

Поведение по типу поверхности:

- **Серверные каналы** (с `require_mention: true`): backfill сканирует канал с момента последнего ответа бота. Полезно, когда другие участники писали, пока бот не был упомянут.
- **Ветки**: backfill сканирует только ветку — `channel.history()` для ветки возвращает только её сообщения, а не родительского канала. Это правильный диапазон, так как ветки обычно являются самостоятельными разговорами.
- **Личные сообщения**: пропускаются. Каждое личное сообщение вызывает бота, поэтому транскрипт уже полон — нет пропусков упоминаний.
- **Свободные каналы** и **автоматически созданные ботом ветки**: также пропускаются по той же причине — отсутствие ограничения упоминанием значит отсутствие разрыва.

Сессии per‑user (`group_sessions_per_user: true`, значение по умолчанию) тоже выигрывают: у пользователя может не быть контекста, опубликованного другими участниками канала, и его собственных сообщений до того, как он упомянул бота. Backfill заполняет оба пробела.

```yaml
discord:
  history_backfill: true   # default
```

Чтобы отключить:

```yaml
discord:
  history_backfill: false
```

> **Note:** Сообщения, которые приходят *во время* обработки бота (между триггером и ответом), не захватываются. Это принятое упрощение — пользователь может переслать или упомянуть снова.

#### `discord.history_backfill_limit`

**Type:** integer — **Default:** `50`

Максимальное количество сообщений, которое сканируется назад при восстановлении контекста канала. На практике сканирование обычно останавливается гораздо раньше — на последнем собственном сообщении бота в канале, которое служит естественной границей между ходами. Этот лимит служит защитой от холодных запусков и больших разрывов, когда в недавней истории нет сообщения бота.

```yaml
discord:
  history_backfill: true
  history_backfill_limit: 50
```

#### `group_sessions_per_user`

**Type:** boolean — **Default:** `true`

Это глобальная настройка шлюза (не специфична для Discord), управляющая тем, получают ли пользователи в одном канале изолированные истории сессий.

Когда `true`: у Алисы и Боба в `#research` каждый свой отдельный разговор с Hermes. Когда `false`: весь канал делит одну общую историю и один слот работающего агента.

```yaml
group_sessions_per_user: true
```

См. раздел [Модель сессий](#session-model-in-discord) выше для полного понимания последствий каждого режима.

#### `display.tool_progress`

**Type:** string — **Default:** `"all"` — **Values:** `off`, `new`, `all`, `verbose`

Управляет тем, отправляет ли бот сообщения о прогрессе в чат во время обработки (например, «Чтение файла…», «Выполнение команды терминала…»). Это глобальная настройка шлюза, применимая ко всем платформам.

```yaml
display:
  tool_progress: "all"    # off | new | all | verbose
```

- `off` — без сообщений о прогрессе
- `new` — показывать только первый вызов инструмента за ход
- `all` — показывать все вызовы инструментов (обрезанные до 40 символов в сообщениях шлюза)
- `verbose` — показывать полные детали вызова инструмента (может генерировать длинные сообщения)

#### `display.tool_progress_command`

**Type:** boolean — **Default:** `false`

Когда включено, делает доступной slash‑команду `/verbose` в шлюзе, позволяя переключать режимы прогресса инструмента (`off → new → all → verbose → off`) без редактирования `config.yaml`.

```yaml
display:
  tool_progress_command: true
```
## Управление доступом к слеш‑командам

По умолчанию каждый разрешённый пользователь может выполнять любую слеш‑команду. Чтобы разделить ваш список разрешённых на **администраторов** (полный доступ к слеш‑командам) и **обычных пользователей** (только те команды, которые ты явно разрешишь), добавь `allow_admin_from` и `user_allowed_commands` в блок `extra` платформы Discord:

```yaml
gateway:
  platforms:
    discord:
      extra:
        # Existing user allowlist (unchanged)
        allow_from:
          - "123456789012345678"  # admin user ID
          - "999888777666555444"  # regular user ID

        # NEW — admins get all slash commands (built-in + plugin)
        allow_admin_from:
          - "123456789012345678"

        # NEW — non-admin allowed users can only run these slash commands.
        # /help and /whoami are always allowed so users can see their access.
        user_allowed_commands:
          - status
          - model
          - history

        # Optional: separate admin / command lists for server channels
        group_allow_admin_from:
          - "123456789012345678"
        group_user_allowed_commands:
          - status
```

**Поведение:**

- Пользователь, указанный в `allow_admin_from` для области (прямое сообщение или канал сервера), может выполнять **любую** зарегистрированную слеш‑команду — как встроенную, так и добавленную плагинами — через живой реестр команд.
- Пользователь, не входящий в `allow_admin_from`, может выполнять только команды, перечисленные в `user_allowed_commands`, плюс всегда разрешённые базовые: `/help` и `/whoami`.
- Обычный чат (сообщения без слеша) не затронут. Пользователи без прав администратора по‑прежнему могут общаться с агентом как обычно; они просто не могут вызывать произвольные команды.
- **Обратная совместимость:** если `allow_admin_from` не задана для области, фильтрация слеш‑команд отключена для этой области. Существующие установки продолжают работать без изменений.
- Статус администратора в ДМ не подразумевает статус администратора в канале сервера. У каждой области свой список администраторов.

Используй `/whoami`, чтобы увидеть текущую область, свой уровень (admin / user / unrestricted) и какие слеш‑команды ты можешь выполнять.
## Интерактивный выбор модели

Отправь `/model` без аргументов в канале Discord, чтобы открыть выпадающий список выбора модели:

1. **Выбор провайдера** — выпадающий список, показывающий доступные провайдеры (до 25).
2. **Выбор модели** — второй выпадающий список с моделями выбранного провайдера (до 25).

Выборщик завершает работу через 120 секунд. Только авторизованные пользователи (те, кто находятся в `DISCORD_ALLOWED_USERS`) могут взаимодействовать с ним. Если ты знаешь название модели, введи `/model <name>` напрямую.
## Native Slash Commands for Skills

Hermes автоматически регистрирует установленные навыки как **native Discord Application Commands**. Это значит, что навыки появляются в автодополнении меню `/` в Discord наряду со встроенными командами.

- Каждый навык становится slash‑командой Discord (например, `/code-review`, `/ascii-art`)
- Навыки принимают необязательный строковый параметр `args`
- В Discord есть ограничение в 100 команд приложения на бота — если у тебя больше навыков, чем доступных слотов, лишние навыки пропускаются с предупреждением в логах
- Навыки регистрируются при запуске бота вместе со встроенными командами, такими как `/model`, `/reset` и `/background`

Дополнительная конфигурация не требуется — любой навык, установленный через `hermes skills install`, автоматически регистрируется как slash‑команда Discord при следующем перезапуске шлюза.

### Отключение регистрации slash‑команд

Если ты запускаешь несколько шлюзов Hermes к одному и тому же приложению Discord (например, staging + production), только один из них должен владеть глобальной регистрацией slash‑команд — иначе последняя загрузка «перезапишет» регистрации и они будут «перепрыгивать». Отключи регистрацию slash‑команд на шлюзе‑«подписчике»:

```yaml
gateway:
  platforms:
    discord:
      extra:
        slash_commands: false   # default: true
```

Оставив это значение `true` на «основном» шлюзе, ты сохраняешь обычное поведение — глобальные команды меню `/` для встроенных функций и установленных навыков.
## Отправка медиа (`send_message` + `MEDIA:` теги)

Адаптер Discord поддерживает нативную загрузку файлов для всех распространённых типов медиа через инструмент `send_message` и встроенные теги `MEDIA:/path/to/file`, генерируемые агентом:

| Тип | Как доставляется |
|---|---|
| Изображения (PNG/JPG/WebP) | Нативное вложение изображения Discord с встроенным превью |
| Анимированные GIF | `send_animation` загружает как `animation.gif`, поэтому Discord воспроизводит его встроенно (не как статическую миниатюру) |
| Видео (MP4/MOV) | `send_video` — нативный видеоплеер |
| Аудио / Голос | `send_voice` — нативное голосовое сообщение, когда это возможно, иначе вложение файла |
| Документы (PDF/ZIP/docx/и др.) | `send_document` — нативное вложение с кнопкой загрузки |

Ограничение размера загрузки в Discord зависит от уровня буста сервера (25 МБ бесплатно, до 500 МБ). Если Hermes получает HTTP 413, адаптер переходит к ссылке, указывающей путь к локальному кэшу, вместо того чтобы молча завершиться с ошибкой.
## Приём произвольных типов файлов

По умолчанию бот кэширует загрузки, соответствующие встроенному allowlist — изображения, аудио, видео, PDF, текст/markdown/csv/log, JSON/XML/YAML/TOML, zip, docx/xlsx/pptx. Всё остальное (например, `.wav`, `.bin`, дамп с пользовательским расширением) фиксируется как `Unsupported document type` и отбрасывается до того, как агент его увидит.

Чтобы принимать произвольные типы файлов, включи `discord.allow_any_attachment`:

```yaml
discord:
  allow_any_attachment: true
  # Optional — raise/disable the per-file size cap. Default is 32 MiB.
  # The whole file is held in memory while being cached, so unlimited
  # uploads carry a real memory cost.
  max_attachment_bytes: 33554432   # bytes; 0 = unlimited
```

Когда флаг включён, любой загруженный файл скачивается, кэшируется в `~/.hermes/cache/documents/` и передаётся агенту как событие сообщения типа `DOCUMENT` с MIME `application/octet-stream`. Агент получает примечание контекста, указывающее на локальный путь (автоматически переводится для терминалов Docker/Modal в изолированных средах через `to_agent_visible_cache_path`) и может исследовать файл с помощью `terminal` (`ffprobe`, `unzip`, `file`, `strings` и т.п.) или `read_file`. Содержимое файла **не** встраивается в prompt — только путь — поэтому бинарные загрузки не раздувают окно контекста.

Известные текстовые форматы, уже присутствующие в allowlist (`.txt`, `.md`, `.log`), продолжают автоматически внедряться до 100 КиБ; это поведение не меняется при включённом флаге.

Эквивалентные переменные окружения: `DISCORD_ALLOW_ANY_ATTACHMENT=true` и `DISCORD_MAX_ATTACHMENT_BYTES=33554432` (или `0` для снятия ограничения).

:::warning Стоимость memory при отсутствии ограничений
Отключение ограничения размера (`max_attachment_bytes: 0`) означает, что пользователь может бросить на бота файл размером в несколько ГиБ, и gateway будет буферизовать его в memory, пока кэширует на диск. Делай это только в доверенных установках с одним пользователем. Для общих ботов оставляй значение по умолчанию 32 MiB или повышай его консервативно.
:::
## Интерактивные запросы (clarify)

Когда агент вызывает инструмент `clarify` — чтобы спросить, какой подход ты предпочитаешь, получить обратную связь после задачи или уточнить перед нетривиальным решением — Discord отображает вопрос с **одной кнопкой на каждый вариант**:

> Какой фреймворк использовать для панели управления?
>
> [1. Next.js] [2. Remix] [3. Astro] [Other (type answer)]

Нажми кнопку с номером, чтобы ответить, или нажми **Other**, чтобы ввести свободный ответ (следующее сообщение, которое ты отправишь в этом канале, станет ответом). Открытые вызовы `clarify` (без предустановленных вариантов) пропускают кнопки и просто фиксируют твоё следующее сообщение.

Кнопки отключаются после выбора, чтобы повторные нажатия не приводили к двойному разрешению запроса. Настрой тайм‑аут ответа через `agent.clarify_timeout` в `~/.hermes/config.yaml` (по умолчанию `600` секунд). Если ты не ответишь в течение тайм‑аута, агент разблокируется с помощью контрольного сообщения и адаптируется, вместо того чтобы зависнуть.

## Каналы форума

Каналы форума Discord (тип 15) не принимают прямые сообщения — каждый пост в форуме должен быть тредом. Hermes автоматически обнаруживает каналы форума и создаёт новое сообщение‑тред, когда ему нужно отправить туда сообщение, поэтому `send_message`, TTS, изображения, голосовые сообщения и вложения файлов работают без специальной обработки со стороны агента.

- **Имя треда** формируется из первой строки сообщения (префикс заголовка markdown удаляется, ограничено 100 символами). Когда сообщение состоит только из вложения, в качестве запасного имени треда используется имя файла.
- **Вложения** идут вместе с начальным сообщением нового треда — нет отдельного шага загрузки, нет частичных отправок.
- **Один вызов — один тред**: каждая отправка в форум создаёт новый тред. Последующие отправки в тот же форум, соответственно, будут создавать отдельные треды.
- **Обнаружение трёхуровневое**: сначала проверяется кэш каталога каналов, затем кэш проб локального процесса, и в последнюю очередь делается живой запрос `GET /channels/{id}` (результат которого затем кэшируется на время жизни процесса).

Обновление каталога (`/channels refresh` на платформах, где это доступно, или перезапуск шлюза) заполняет кэш всеми каналами форума, созданными после запуска бота.
## Устранение неполадок

### Бот онлайн, но не отвечает на сообщения

**Причина**: Intent «Message Content» отключён.

**Решение**: Перейди в [Developer Portal](https://discord.com/developers/applications) → твоё приложение → Bot → Privileged Gateway Intents → включи **Message Content Intent** → Save Changes. Перезапусти шлюз.

### Ошибка «Disallowed Intents» при запуске

**Причина**: Твой код запрашивает intents, которые не включены в Developer Portal.

**Решение**: В настройках бота включи все три Privileged Gateway Intents (Presence, Server Members, Message Content), затем перезапусти.

### Бот не видит сообщения в конкретном канале

**Причина**: У роли бота нет разрешения просматривать этот канал.

**Решение**: В Discord открой настройки канала → Permissions → добавь роль бота и включи **View Channel** и **Read Message History**.

### Ошибки 403 Forbidden

**Причина**: У бота отсутствуют необходимые разрешения.

**Решение**: Пригласи бота заново с правильными разрешениями, используя URL из Шага 5, или вручную отрегулируй разрешения роли бота в Server Settings → Roles.

### Бот офлайн

**Причина**: Шлюз Hermes не запущен, либо токен неверен.

**Решение**: Убедись, что `hermes gateway` работает. Проверь `DISCORD_BOT_TOKEN` в файле `.env`. Если ты недавно сбросил токен, обнови его.

### «User not allowed» / Бот игнорирует тебя

**Причина**: Твой User ID отсутствует в `DISCORD_ALLOWED_USERS`.

**Решение**: Добавь свой User ID в `DISCORD_ALLOWED_USERS` в `~/.hermes/.env` и перезапусти шлюз.

### Люди в одном канале неожиданно делятся контекстом

**Причина**: `group_sessions_per_user` отключён, либо платформа не может предоставить User ID для сообщений в этом контексте.

**Решение**: Установи значение в `~/.hermes/config.yaml` и перезапусти шлюз:

```yaml
group_sessions_per_user: true
```

Если ты намеренно хочешь совместный разговор в комнате, оставь параметр выключенным — просто ожидай совместную историю транскриптов и совместное поведение прерываний.
## Безопасность

:::warning
Всегда задавай `DISCORD_ALLOWED_USERS` (или `DISCORD_ALLOWED_ROLES`), чтобы ограничить, кто может взаимодействовать с ботом. Если ни один из параметров не указан, шлюз по умолчанию отклоняет всех пользователей как меру предосторожности. Разрешай только людям, которым доверяешь — у авторизованных пользователей будет полный доступ к возможностям агента, включая использование инструментов и доступ к системе.
:::

### Управление доступом на основе ролей

Для серверов, где доступ управляется ролями, а не списками отдельных пользователей (команды модераторов, служба поддержки, внутренние инструменты), используй `DISCORD_ALLOWED_ROLES` — список идентификаторов ролей, разделённых запятыми. Любой участник, имеющий одну из этих ролей, считается авторизованным.

```bash
# ~/.hermes/.env — works alongside or instead of DISCORD_ALLOWED_USERS
DISCORD_ALLOWED_ROLES=987654321098765432,876543210987654321
```

Семантика:

- **ИЛИ со списком разрешённых пользователей.** Пользователь считается авторизованным, если его ID присутствует в `DISCORD_ALLOWED_USERS` **или** у него есть любая роль из `DISCORD_ALLOWED_ROLES`.
- **Server Members Intent автоматически включён.** Когда задаётся `DISCORD_ALLOWED_ROLES`, бот включает Intent Members при подключении — это требуется, чтобы Discord отправлял информацию о ролях вместе с записями участников.
- **Идентификаторы ролей, а не имена.** Получи их в Discord: **User Settings → Advanced → Developer Mode ON**, затем щёлкни правой кнопкой мыши по любой роли → **Copy Role ID**.
- **Запасной (вариант) в личных сообщениях.** В личных сообщениях проверка ролей сканирует общие серверы; пользователь с разрешённой ролью в любом совместном сервере также считается авторизованным в личных сообщениях.

Это предпочтительный подход, когда состав команды модерации часто меняется — новые модераторы получают доступ сразу после назначения роли, без правки `.env` и перезапуска шлюза.

### Управление упоминаниями

По умолчанию Hermes блокирует бота от упоминаний `@everyone`, `@here` и ролей, даже если их токены присутствуют в ответе. Это предотвращает спам всего сервера из‑за плохо сформулированного запроса или отзеркалённого пользовательского контента. Упоминания отдельных `@user` и упоминания в ответах (маленькая «replying to…»‑метка) остаются включёнными, чтобы обычный диалог работал.

Ты можешь ослабить эти ограничения через переменные окружения или `config.yaml`:

```yaml
# ~/.hermes/config.yaml
discord:
  allow_mentions:
    everyone: false      # allow the bot to ping @everyone / @here
    roles: false         # allow the bot to ping @role mentions
    users: true          # allow the bot to ping individual @users
    replied_user: true   # ping the author when replying to their message
```

```bash
# ~/.hermes/.env — env vars win over config.yaml
DISCORD_ALLOW_MENTION_EVERYONE=false
DISCORD_ALLOW_MENTION_ROLES=false
DISCORD_ALLOW_MENTION_USERS=true
DISCORD_ALLOW_MENTION_REPLIED_USER=true
```

:::tip
Оставляй `everyone` и `roles` со значением `false`, если только точно не знаешь, зачем они нужны. Очень легко, что LLM сгенерирует строку `@everyone` в обычном ответе; без этой защиты это уведомит каждого участника твоего сервера.
:::
