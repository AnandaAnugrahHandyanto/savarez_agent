# WeCom (Корпоративный WeChat)

Подключи Hermes к [WeCom](https://work.weixin.qq.com/) (企业微信) — корпоративной платформе обмена сообщениями от Tencent. Адаптер использует WebSocket‑шлюз AI‑бота WeCom для двунаправлённой связи в реальном времени, поэтому публичный эндпоинт или webhook не требуются.

См. также: [WeCom Callback](./wecom-callback.md) для настройки входящего webhook.
## Предварительные требования

- Учётная запись организации WeCom
- AI‑бот, созданный в консоли администрирования WeCom
- ID бота и Secret со страницы учётных данных бота
- Пакеты Python: `aiohttp` и `httpx`
## Настройка

### Шаг 1: Создать AI‑бота

#### Рекомендовано: Scan-to-Create (одна команда)

```bash
hermes gateway setup
```

Выбери **WeCom** и отсканируй QR‑код в мобильном приложении WeCom. Hermes автоматически создаст приложение‑бота с нужными правами и сохранит учётные данные.

Мастер настройки выполнит:
1. Отобразит QR‑код в терминале
2. Ожидет, пока ты отсканируешь его в мобильном приложении WeCom
3. Автоматически получит Bot ID и Secret
4. Проведёт тебя через настройку контроля доступа

#### Альтернатива: Ручная настройка

Если Scan-to-Create недоступен, мастер переходит к ручному вводу:

1. Войди в [WeCom Admin Console](https://work.weixin.qq.com/wework_admin/frame)
2. Перейди в **Applications** → **Create Application** → **AI Bot**
3. Укажи название и описание бота
4. Скопируй **Bot ID** и **Secret** со страницы учётных данных
5. Выполни `hermes gateway setup`, выбери **WeCom** и введи учётные данные, когда будет запрос

:::warning
Храни Secret бота в тайне. Любой, кто его получит, сможет выдавать себя за твоего бота.
:::

### Шаг 2: Настроить Hermes

#### Вариант A: Интерактивная настройка (рекомендовано)

```bash
hermes gateway setup
```

Выбери **WeCom** и следуй подсказкам. Мастер проведёт тебя через:
- Учётные данные бота (через сканирование QR или ручной ввод)
- Параметры контроля доступа (белый список, режим паринга или открытый доступ)
- Основной канал для уведомлений

#### Вариант B: Ручная конфигурация

Добавь следующее в `~/.hermes/.env`:

```bash
WECOM_BOT_ID=your-bot-id
WECOM_SECRET=your-secret

# Optional: restrict access
WECOM_ALLOWED_USERS=user_id_1,user_id_2

# Optional: home channel for cron/notifications
WECOM_HOME_CHANNEL=chat_id
```

### Шаг 3: Запустить шлюз

```bash
hermes gateway
```
## Возможности

- **WebSocket transport** — постоянное соединение, публичный эндпоинт не требуется
- **DM and group messaging** — настраиваемые политики доступа
- **Per-group sender allowlists** — списки разрешённых отправителей для каждой группы
- **Media support** — загрузка и скачивание изображений, файлов, голосовых сообщений, видео
- **AES-encrypted media** — автоматическое расшифрование входящих вложений
- **Quote context** — сохраняет контекст цитаты
- **Markdown rendering** — ответы в формате богатого текста
- **Reply-mode streaming** — стриминг в режиме ответа, сопоставляющий ответы с контекстом входящего сообщения
- **Auto-reconnect** — экспоненциальный backoff при разрывах соединения
## Параметры конфигурации

Устанавливай эти параметры в `config.yaml` под `platforms.wecom.extra`:

| Key | Default | Description |
|-----|---------|-------------|
| `bot_id` | — | ID AI‑бота WeCom (обязательно) |
| `secret` | — | Секрет AI‑бота WeCom (обязательно) |
| `websocket_url` | `wss://openws.work.weixin.qq.com` | URL шлюза WebSocket |
| `dm_policy` | `open` | Доступ к прямым сообщениям: `open`, `allowlist`, `disabled`, `pairing` |
| `group_policy` | `open` | Доступ к группам: `open`, `allowlist`, `disabled` |
| `allow_from` | `[]` | ID пользователей, которым разрешены прямые сообщения (когда `dm_policy=allowlist`) |
| `group_allow_from` | `[]` | ID групп, которым разрешён доступ (когда `group_policy=allowlist`) |
| `groups` | `{}` | Конфигурация для каждой группы (см. ниже) |
## Политики доступа

### DM Policy

Определяет, кто может отправлять прямые сообщения боту:

| Value | Behavior |
|-------|----------|
| `open` | Любой может написать боту (по умолчанию) |
| `allowlist` | Только пользовательские ID, указанные в `allow_from`, могут писать |
| `disabled` | Все прямые сообщения игнорируются |
| `pairing` | Режим сопряжения (для первоначальной настройки) |

```bash
WECOM_DM_POLICY=allowlist
```

### Group Policy

Определяет, в каких группах бот отвечает:

| Value | Behavior |
|-------|----------|
| `open` | Бот отвечает во всех группах (по умолчанию) |
| `allowlist` | Бот отвечает только в группах, ID которых перечислены в `group_allow_from` |
| `disabled` | Все сообщения в группах игнорируются |

```bash
WECOM_GROUP_POLICY=allowlist
```

### Per-Group Sender Allowlists

Для более тонкой настройки можно ограничить, какие пользователи могут взаимодействовать с ботом в конкретных группах. Это настраивается в `config.yaml`:

```yaml
platforms:
  wecom:
    enabled: true
    extra:
      bot_id: "your-bot-id"
      secret: "your-secret"
      group_policy: "allowlist"
      group_allow_from:
        - "group_id_1"
        - "group_id_2"
      groups:
        group_id_1:
          allow_from:
            - "user_alice"
            - "user_bob"
        group_id_2:
          allow_from:
            - "user_charlie"
        "*":
          allow_from:
            - "user_admin"
```

**Как это работает:**

1. Политики `group_policy` и `group_allow_from` определяют, разрешена ли группа вообще.
2. Если группа проходит проверку верхнего уровня, список `groups.<group_id>.allow_from` (если он присутствует) дополнительно ограничивает, какие отправители внутри этой группы могут взаимодействовать с ботом.
3. Запись `"*"` в списке групп служит значением по умолчанию для групп, не указанных явно.
4. Записи в `allowlist` поддерживают подстановочный символ `*`, позволяющий всем пользователям, а сравнение производится без учёта регистра.
5. Записи могут опционально использовать префикс `wecom:user:` или `wecom:group:` — префикс автоматически удаляется.

Если для группы не настроен `allow_from`, всем пользователям в этой группе разрешён доступ (при условии, что сама группа прошла проверку верхнего уровня).
## Поддержка медиа

### Входящие (получение)

Адаптер получает медиа‑вложения от пользователей и кэширует их локально для обработки агентом:

| Тип | Как обрабатывается |
|------|--------------------|
| **Изображения** | Загружаются и кэшируются локально. Поддерживаются как URL‑изображения, так и изображения в формате base64. |
| **Файлы** | Загружаются и кэшируются. Имя файла сохраняется из оригинального сообщения. |
| **Голос** | При наличии извлекается транскрипция текста голосового сообщения. |
| **Смешанные сообщения** | Смешанные сообщения WeCom (текст + изображения) разбираются, и все компоненты извлекаются. |

**Цитируемые сообщения:** Медиа из цитируемых (ответных) сообщений также извлекаются, чтобы агент имел контекст того, на что пользователь отвечает.

### Дешифрование AES‑зашифрованного медиа

WeCom шифрует некоторые входящие медиа‑вложения с помощью AES‑256‑CBC. Адаптер обрабатывает это автоматически:

- Когда входящий медиа‑элемент содержит поле `aeskey`, адаптер загружает зашифрованные байты и расшифровывает их с помощью AES‑256‑CBC с паддингом PKCS#7.
- AES‑ключ — это значение поля `aeskey`, декодированное из base64 (должно быть ровно 32 байта).
- IV берётся из первых 16 байт ключа.
- Для этого требуется пакет Python `cryptography` (`pip install cryptography`).

Никакой настройки не требуется — дешифрование происходит прозрачно при получении зашифрованного медиа.

### Исходящие (отправка)

| Метод | Что отправляется | Ограничение размера |
|--------|-------------------|----------------------|
| `send` | Текстовые сообщения в Markdown | 4000 символов |
| `send_image` / `send_image_file` | Нативные сообщения‑изображения | 10 МБ |
| `send_document` | Вложенные файлы | 20 МБ |
| `send_voice` | Голосовые сообщения (только формат AMR для нативного голоса) | 2 МБ |
| `send_video` | Видео‑сообщения | 10 МБ |

**Загрузка кусками:** Файлы загружаются кусками по 512 KB через трёхшаговый протокол (init → chunks → finish). Адаптер обрабатывает это автоматически.

**Автоматическое понижение:** Когда медиа превышает ограничение размера своего нативного типа, но находится ниже абсолютного предела в 20 МБ, оно автоматически отправляется как обычное файловое вложение:

- Изображения > 10 МБ → отправляются как файл
- Видео > 10 МБ → отправляются как файл
- Голос > 2 МБ → отправляются как файл
- Аудио не в формате AMR → отправляются как файл (WeCom поддерживает AMR только для нативного голоса)

Файлы, превышающие абсолютный предел в 20 МБ, отклоняются с информационным сообщением, отправляемым в чат.
## Ответы в режиме Reply-Mode с потоковой передачей

Когда бот получает сообщение через обратный вызов WeCom, адаптер запоминает идентификатор входящего запроса. Если ответ отправляется, пока контекст запроса ещё активен, адаптер использует reply‑mode WeCom (`aibot_respond_msg`) с потоковой передачей, чтобы напрямую сопоставить ответ с входящим сообщением. Это обеспечивает более естественное взаимодействие в клиенте WeCom.

Если контекст входящего запроса истёк или недоступен, адаптер переходит к запасному варианту — проактивной отправке сообщения через `aibot_send_msg`.

Reply‑mode также работает с медиа: загруженные медиа‑файлы могут быть отправлены в качестве ответа на исходное сообщение.
## Подключение и повторное подключение

Адаптер поддерживает постоянное WebSocket‑соединение с шлюзом WeCom по адресу `wss://openws.work.weixin.qq.com`.

### Жизненный цикл подключения

1. **Connect:** Открывает WebSocket‑соединение и отправляет аутентификационный кадр `aibot_subscribe` с `bot_id` и `secret`.
2. **Heartbeat:** Отправляет приложенческие ping‑кадры каждые 30 секунд, чтобы поддерживать соединение живым.
3. **Listen:** Непрерывно читает входящие кадры и распределяет обратные вызовы сообщений.

### Поведение при повторном подключении

При потере соединения адаптер использует экспоненциальную задержку для повторных попыток:

| Attempt | Delay |
|---------|-------|
| 1‑я попытка | 2 секунды |
| 2‑я попытка | 5 секунд |
| 3‑я попытка | 10 секунд |
| 4‑я попытка | 30 секунд |
| 5‑я и последующие | 60 секунд |

После каждой успешной повторной попытки счётчик задержки сбрасывается в ноль. Все ожидающие запросы‑futures завершаются ошибкой при отключении, чтобы вызывающие функции не зависали бесконечно.

### Дедупликация

Входящие сообщения дедуплицируются по идентификаторам сообщений с окном в 5 минут и максимальным кэшем в 1000 записей. Это предотвращает двойную обработку сообщений во время повторного подключения или сетевых сбоев.

## Устранение неполадок

| Проблема | Решение |
|----------|---------|
| `WECOM_BOT_ID and WECOM_SECRET are required` | Установи обе переменные окружения или настрой их в мастере настройки |
| `WeCom startup failed: aiohttp not installed` | Установи aiohttp: `pip install aiohttp` |
| `WeCom startup failed: httpx not installed` | Установи httpx: `pip install httpx` |
| `invalid secret (errcode=40013)` | Проверь, что секрет совпадает с учётными данными твоего бота |
| `Timed out waiting for subscribe acknowledgement` | Проверь сетевое соединение с `openws.work.weixin.qq.com` |
| Bot doesn't respond in groups | Проверь настройку `group_policy` и убедись, что ID группы присутствует в `group_allow_from` |
| Bot ignores certain users in a group | Проверь списки `allow_from` для каждой группы в секции конфигурации `groups` |
| Media decryption fails | Установи `cryptography`: `pip install cryptography` |
| `cryptography is required for WeCom media decryption` | Входящие медиа зашифрованы AES. Установи: `pip install cryptography` |
| Voice messages sent as files | WeCom поддерживает только формат AMR для нативных голосовых сообщений. Другие форматы автоматически преобразуются в файл. |
| `File too large` error | У WeCom абсолютный лимит 20 МБ на все загрузки файлов. Сожми или раздели файл. |
| Images sent as files | Изображения более 10 МБ превышают нативный лимит и автоматически преобразуются в вложения‑файлы. |
| `Timeout sending message to WeCom` | Возможно, WebSocket был разорван. Проверь логи на наличие сообщений о переподключении. |
| `WeCom websocket closed during authentication` | Проблема с сетью или неверные учётные данные. Проверь `bot_id` и `secret`. |
