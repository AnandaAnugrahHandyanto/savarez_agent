# Запуск локальных LLM на Mac

Это руководство проведёт тебя через запуск локального LLM‑сервера на macOS с совместимым с OpenAI API. Ты получаешь полную приватность, нулевые затраты на API и удивительно хорошую производительность на Apple Silicon.

Мы рассматриваем два бэкенда:

| Бэкенд | Установка | Лучшее в | Формат |
|---------|---------|---------|--------|
| **llama.cpp** | `brew install llama.cpp` | Самое быстрое время до первого токена, квантованный KV‑кеш для небольшого объёма памяти | GGUF |
| **omlx** | [omlx.ai](https://omlx.ai) | Самая быстрая генерация токенов, нативная оптимизация под Metal | MLX (safetensors) |

Оба предоставляют совместимый с OpenAI эндпоинт `/v1/chat/completions`. Hermes работает с любым из них — просто укажи `http://localhost:8080` или `http://localhost:8000`.

:::info Apple Silicon only
Это руководство ориентировано на Mac с Apple Silicon (M1 и новее). На Intel‑Mac можно запустить llama.cpp, но без ускорения GPU — ожидай заметно более медленную работу.
:::

## Выбор модели

Для начала мы рекомендуем **Qwen3.5-9B** — сильную модель рассуждений, которая комфортно помещается в 8 ГБ+ объединённой памяти при квантовании.

| Вариант | Размер на диске | Память (контекст 128K) | Бэкенд |
|---------|----------------|------------------------|--------|
| Qwen3.5-9B-Q4_K_M (GGUF) | 5.3 GB | ~10 GB с квантованным KV‑кешем | llama.cpp |
| Qwen3.5-9B-mlx-lm-mxfp4 (MLX) | ~5 GB | ~12 GB | omlx |

**Правило памяти:** размер модели + KV‑кеш. Модель 9B Q4 занимает ~5 GB. KV‑кеш при контексте 128K с квантованием Q4 добавляет ~4‑5 GB. При использовании KV‑кеша по умолчанию (f16) объём растёт до ~16 GB. Флаги квантования KV‑кеша в llama.cpp — ключевая оптимизация для систем с ограниченной памятью.

Для более крупных моделей (27B, 35B) понадобится 32 GB+ объединённой памяти. 9B — оптимальный вариант для машин с 8‑16 GB.

## Вариант A: llama.cpp

llama.cpp — самый переносимый локальный рантайм LLM. На macOS он использует Metal для ускорения GPU «из коробки».

### Установка

```bash
brew install llama.cpp
```

Это установит глобальную команду `llama-server`.

### Скачивание модели

Требуется модель в формате GGUF. Самый простой источник — Hugging Face через `huggingface-cli`:

```bash
brew install huggingface-cli
```

Затем скачай:

```bash
huggingface-cli download unsloth/Qwen3.5-9B-GGUF Qwen3.5-9B-Q4_K_M.gguf --local-dir ~/models
```

:::tip Gated models
Некоторые модели на Hugging Face требуют аутентификации. Сначала выполни `huggingface-cli login`, если получаешь ошибку 401 или 404.
:::

### Запуск сервера

```bash
llama-server -m ~/models/Qwen3.5-9B-Q4_K_M.gguf \
  -ngl 99 \
  -c 131072 \
  -np 1 \
  -fa on \
  --cache-type-k q4_0 \
  --cache-type-v q4_0 \
  --host 0.0.0.0
```

Что делает каждый флаг:

| Флаг | Описание |
|------|----------|
| `-ngl 99` | Переносит все слои на GPU (Metal). Укажи большое число, чтобы ничего не оставалось на CPU. |
| `-c 131072` | Размер окна контекста (128 K токенов). Уменьши, если не хватает памяти. |
| `-np 1` | Количество параллельных слотов. Оставь 1 для одиночного пользователя — больше слотов делят твой бюджет памяти. |
| `-fa on` | Flash attention. Снижает потребление памяти и ускоряет вывод при длинном контексте. |
| `--cache-type-k q4_0` | Квантование кеша ключей до 4‑бит. **Это главный экономитель памяти.** |
| `--cache-type-v q4_0` | Квантование кеша значений до 4‑бит. Вместе с предыдущим уменьшает память KV‑кеша примерно на 75 % по сравнению с f16. |
| `--host 0.0.0.0` | Слушать на всех интерфейсах. Используй `127.0.0.1`, если не нужен сетевой доступ. |

Сервер готов, когда ты видишь:

```
main: server is listening on http://0.0.0.0:8080
srv  update_slots: all slots are idle
```

### Оптимизация памяти для ограниченных систем

Флаги `--cache-type-k q4_0 --cache-type-v q4_0` — самая важная оптимизация для систем с ограниченной памятью. Их влияние при контексте 128K:

| Тип KV‑кеша | Память KV‑кеша (128K ctx, 9B модель) |
|------------|--------------------------------------|
| f16 (по умолчанию) | ~16 GB |
| q8_0 | ~8 GB |
| **q4_0** | **~4 GB** |

На Mac с 8 GB используй KV‑кеш `q4_0` и выбери более маленькую модель, которая всё ещё укладывается в минимум контекста Hermes — 64K. На 16 GB ты спокойно можешь работать с 128K контекстом. На 32 GB+ можно запускать более крупные модели или несколько параллельных слотов.

Если всё ещё не хватает памяти, уменьшай только контекст, оставаясь хотя бы на минимуме Hermes — 64K; иначе переключись на более маленькую модель или другую квантовку (Q3_K_M вместо Q4_K_M).

### Проверка

```bash
curl -s http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen3.5-9B-Q4_K_M.gguf",
    "messages": [{"role": "user", "content": "Hello!"}],
    "max_tokens": 50
  }' | jq .choices[0].message.content
```

### Получение названия модели

Если забыл название модели, запроси эндпоинт моделей:

```bash
curl -s http://localhost:8080/v1/models | jq '.data[].id'
```

## Вариант B: MLX через omlx

[omlx](https://omlx.ai) — нативное macOS‑приложение, которое управляет и обслуживает модели MLX. MLX — собственный фреймворк Apple для машинного обучения, оптимизированный под архитектуру объединённой памяти Apple Silicon.

### Установка

Скачай и установи с [omlx.ai](https://omlx.ai). Приложение предоставляет GUI для управления моделями и встроенный сервер.

### Скачивание модели

Используй приложение omlx для поиска и скачивания моделей. Найди `Qwen3.5-9B-mlx-lm-mxfp4` и скачай её. Модели хранятся локально (обычно в `~/.omlx/models/`).

### Запуск сервера

omlx обслуживает модели по умолчанию на `http://127.0.0.1:8000`. Запусти обслуживание через UI приложения или через CLI, если он доступен.

### Проверка

```bash
curl -s http://127.0.0.1:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen3.5-9B-mlx-lm-mxfp4",
    "messages": [{"role": "user", "content": "Hello!"}],
    "max_tokens": 50
  }' | jq .choices[0].message.content
```

### Список доступных моделей

omlx может обслуживать несколько моделей одновременно:

```bash
curl -s http://127.0.0.1:8000/v1/models | jq '.data[].id'
```

## Бенчмарки: llama.cpp vs MLX

Оба бэкенда протестированы на одной машине (Apple M5 Max, 128 GB объединённой памяти) с одной и той же моделью (Qwen3.5-9B) при сопоставимых уровнях квантования (Q4_K_M для GGUF, mxfp4 для MLX). Пять разнообразных запросов, три прогонки каждый, бэкенды тестировались последовательно, чтобы избежать конкуренции за ресурсы.

### Результаты

| Метрика | llama.cpp (Q4_K_M) | MLX (mxfp4) | Победитель |
|--------|-------------------|-------------|------------|
| **TTFT (avg)** | **67 ms** | 289 ms | llama.cpp (4.3× быстрее) |
| **TTFT (p50)** | **66 ms** | 286 ms | llama.cpp (4.3× быстрее) |
| **Generation (avg)** | 70 tok/s | **96 tok/s** | MLX (37 % быстрее) |
| **Generation (p50)** | 70 tok/s | **96 tok/s** | MLX (37 % быстрее) |
| **Total time (512 tokens)** | 7.3 s | **5.5 s** | MLX (25 % быстрее) |

### Что это значит

- **llama.cpp** превосходит в обработке подсказок — его flash attention + квантованный KV‑кеш дают первый токен за ~66 ms. Если ты создаёшь интерактивные приложения, где важна воспринимаемая отзывчивость (чат‑боты, автодополнение), это значительное преимущество.

- **MLX** генерирует токены примерно на 37 % быстрее после начала работы. Для пакетных задач, генерации длинных текстов или любых сценариев, где важнее общее время завершения, MLX выигрывает.

- Оба бэкенда **чрезвычайно стабильны** — разброс между запусками пренебрежимо мал. На эти цифры можно рассчитывать.

### Какой выбрать?

| Сценарий | Рекомендация |
|----------|---------------|
| Интерактивный чат, инструменты с низкой задержкой | llama.cpp |
| Длинная генерация, массовая обработка | MLX (omlx) |
| Ограниченная память (8‑16 GB) | llama.cpp (квантованный KV‑кеш непревзойдён) |
| Обслуживание нескольких моделей одновременно | omlx (встроенная поддержка мульти‑моделей) |
| Максимальная совместимость (Linux тоже) | llama.cpp |

## Подключение к Hermes

Когда локальный сервер запущен:

```bash
hermes model
```

Выбери **Custom endpoint** и следуй подсказкам. Тебе понадобится базовый URL и название модели — используй значения из выбранного бэкенда.

## Таймауты

Hermes автоматически обнаруживает локальные эндпоинты (localhost, LAN‑IP) и ослабляет свои таймауты потоков. Для большинства конфигураций ничего настраивать не нужно.

Если всё же появляются ошибки таймаута (например, очень большие контексты на медленном железе), можешь переопределить таймаут чтения потока:

```bash
# In your .env — raise from the 120s default to 30 minutes
HERMES_STREAM_READ_TIMEOUT=1800
```

| Таймаут | По умолчанию | Авто‑коррекция для локального | Переопределение переменной |
|--------|--------------|--------------------------------|----------------------------|
| Чтение потока (уровень сокета) | 120 s | Повышен до 1800 s | `HERMES_STREAM_READ_TIMEOUT` |
| Обнаружение «застрявшего» потока | 180 s | Отключён полностью | `HERMES_STREAM_STALE_TIMEOUT` |
| API‑вызов (не потоковый) | 1800 s | Не меняется | `HERMES_API_TIMEOUT` |

Таймаут чтения потока — это тот, который чаще всего вызывает проблемы: он задаёт предельный срок получения следующего куска данных на уровне сокета. При предзаполнении больших контекстов локальные модели могут не выдавать вывод в течение минут, пока обрабатывают запрос. Авто‑детекция обрабатывает это прозрачно.
