# Советы и лучшие практики

Быстрая подборка практических советов, которые сразу делают тебя более эффективным с Hermes Agent. Каждый раздел охватывает отдельный аспект — сканируй заголовки и переходи к нужному.

:::tip Не можешь решить, какую модель выбрать?
Запусти `hermes setup --portal` — получишь более 300 моделей, включая Claude, GPT‑5 и Gemini, в рамках одной подписки. Смотри [Nous Portal](/integrations/nous-portal).
:::

## Получение наилучших результатов

### Будь конкретен в том, чего ты хочешь

Неясные подсказки дают неясные результаты. Вместо «исправить код» скажи «исправить TypeError в `api/handlers.py` на строке 47 — функция `process_request()` получает `None` от `parse_body()`». Чем больше контекста ты предоставляешь, тем меньше итераций понадобится.

### Предоставь контекст сразу

Начни запрос с релевантных деталей: пути к файлам, сообщения об ошибках, ожидаемое поведение. Одно хорошо сформулированное сообщение лучше трёх раундов уточнений. Вставляй трассировки ошибок напрямую — агент сможет их разобрать.

### Используй файлы контекста для повторяющихся инструкций

Если ты постоянно повторяешь одни и те же указания («использовать табуляцию, а не пробелы», «мы используем pytest», «API находится по пути `/api/v2`»), помести их в файл `AGENTS.md`. Агент будет автоматически читать его в каждой сессии — ноль усилий после настройки.

### Позволь агенту использовать свои инструменты

Не пытайся контролировать каждый шаг. Скажи «найти и исправить падающий тест» вместо «открыть `tests/test_foo.py`, посмотреть строку 42, затем…». У агента есть поиск файлов, доступ к терминалу и выполнение кода — дай ему возможность исследовать и итеративно работать.

### Используй skill для сложных рабочих процессов

Прежде чем писать длинную подсказку о том, как что‑то сделать, проверь, есть ли уже skill для этого. Введи `/skills`, чтобы просмотреть доступные skill, или сразу вызови нужный, например `/axolotl` или `/github-pr-workflow`.
## Советы для продвинутых пользователей CLI

### Многострочный ввод

Нажми **Alt+Enter**, **Ctrl+J** или **Shift+Enter**, чтобы вставить перевод строки без отправки. `Shift+Enter` работает только тогда, когда терминал передаёт его как отдельный нажатие клавиши (по умолчанию в Kitty / foot / WezTerm / Ghostty; в iTerm2 / Alacritty / VS Code terminal после включения протокола клавиатуры Kitty). Остальные два работают во всех терминалах.

### Обнаружение вставки

CLI автоматически определяет многострочные вставки. Просто вставь блок кода или трассировку ошибки напрямую — он не будет отправлять каждую строку как отдельное сообщение. Вставка буферизуется и отправляется одним сообщением.

### Прерывание и перенаправление

Нажми **Ctrl+C** один раз, чтобы прервать агента посреди ответа. Затем можешь ввести новое сообщение, чтобы перенаправить его. Двойное нажатие **Ctrl+C** в течение 2 секунд принудительно завершит работу. Это бесценно, когда агент начинает идти по неверному пути.

### Возобновление сессий с `-c`

Забыл что‑то из последней сессии? Выполни `hermes -c`, чтобы возобновить точно там, где остановился, с полной историей разговора. Можно также возобновить по названию: `hermes -r "my research project"`.

### Вставка изображения из буфера обмена

Нажми **Ctrl+V**, чтобы вставить изображение из буфера обмена напрямую в чат. Агент использует зрение для анализа скриншотов, диаграмм, всплывающих окон ошибок или макетов UI — сохранять файл сначала не требуется.

### Автодополнение слеш‑команд

Введи `/` и нажми **Tab**, чтобы увидеть все доступные команды. Это включает встроенные команды (`/compress`, `/model`, `/title`) и все установленные **skill**. Запоминать ничего не нужно — автодополнение покрывает всё.

:::tip
Используй `/verbose`, чтобы переключать режимы отображения вывода инструмента: **off → new → all → verbose**. Режим «all» отлично подходит для наблюдения за действиями агента; «off» — самый чистый для простых вопросов‑ответов.
:::
## Файлы контекста

### AGENTS.md: Мозг твоего проекта

Создай `AGENTS.md` в корне проекта с решениями по архитектуре, соглашениями о кодировании и инструкциями, специфичными для проекта. Этот файл автоматически внедряется в каждую сессию, поэтому агент всегда знает правила твоего проекта.

```markdown
# Project Context
- This is a FastAPI backend with SQLAlchemy ORM
- Always use async/await for database operations
- Tests go in tests/ and use pytest-asyncio
- Never commit .env files
```

### SOUL.md: Настройка личности

Хочешь, чтобы у Hermes был стабильный голос по умолчанию? Отредактируй `~/.hermes/SOUL.md` (или `$HERMES_HOME/SOUL.md`, если используешь пользовательскую домашнюю папку Hermes). Hermes теперь автоматически создаёт стартовый SOUL и использует этот глобальный файл как источник личности для всего экземпляра.

Для полного пошагового руководства смотри [Use SOUL.md with Hermes](/guides/use-soul-with-hermes).

```markdown
# Soul
You are a senior backend engineer. Be terse and direct.
Skip explanations unless asked. Prefer one-liners over verbose solutions.
Always consider error handling and edge cases.
```

Используй `SOUL.md` для постоянной личности. Используй `AGENTS.md` для инструкций, специфичных для проекта.

### Совместимость с .cursorrules

Уже есть файл `.cursorrules` или файлы `.cursor/rules/*.mdc`? Hermes читает их тоже. Нет необходимости дублировать свои соглашения о кодировании — они загружаются автоматически из рабочей директории.

### Обнаружение

Hermes загружает `AGENTS.md` верхнего уровня из текущей рабочей директории при запуске сессии. Файлы `AGENTS.md` в подкаталогах обнаруживаются лениво во время вызовов инструментов (через `subdirectory_hints.py`) и внедряются в результаты инструмента — они не загружаются заранее в системный запрос.

:::tip
Держи файлы контекста сфокусированными и лаконичными. Каждый символ учитывается в твоём токен‑бюджете, так как они внедряются в каждое отдельное сообщение.
:::
## Память и навыки

### Память vs. Навыки: что где использовать

**Память** — для фактов: твоя среда, предпочтения, расположения проектов и то, что агент узнал о тебе. **Навыки** — для процедур: многошаговые рабочие процессы, инструкции, специфичные для инструмента, и переиспользуемые рецепты. Используй память для «что», навыки — для «как».

### Когда создавать навыки

Если ты находишь задачу, требующую более 5 шагов и планируешь выполнять её снова, попроси агента создать навык для неё. Скажи «сохрани то, что ты только что сделал, как навык под названием `deploy-staging`». В следующий раз просто введи `/deploy-staging`, и агент загрузит весь процесс.

### Управление ёмкостью памяти

Память намеренно ограничена (~2 200 символов для `MEMORY.md`, ~1 375 символов для `USER.md`). Когда она заполняется, агент консолидирует записи. Ты можешь помочь, сказав «очисти свою память» или «замени старую заметку о Python 3.9 — теперь у нас 3.12».

### Позволь агенту запомнить

После продуктивной сессии скажи «запомни это на будущее», и агент сохранит ключевые выводы. Можно быть более конкретным: «сохрани в памяти, что наш CI использует GitHub Actions с рабочим процессом `deploy.yml`».

:::warning
Память — это замороженный снимок: изменения, внесённые во время сессии, не появляются в системном промпте до начала следующей сессии. Агент записывает данные на диск сразу, но кеш промпта не инвалидируется в середине сессии.
:::
## Производительность и стоимость

### Не нарушай кэш подсказок

Большинство провайдеров LLM кэшируют префикс системной подсказки. Если ты оставляешь системную подсказку стабильной (одни и те же файлы контекста, одна и та же память), последующие сообщения в сессии получают **кэш‑хиты**, которые значительно дешевле. Избегай изменения модели или системной подсказки в середине сессии.

### Используй /compress перед достижением лимитов

Длинные сессии накапливают токены. Когда ты замечаешь, что ответы замедляются или обрезаются, запусти `/compress`. Эта команда суммирует историю разговора, сохраняет ключевой контекст и резко уменьшает количество токенов. Используй `/usage`, чтобы проверить своё текущее состояние.

### Делегируй для параллельной работы

Нужно исследовать три темы одновременно? Попроси агента использовать `delegate_task` с параллельными подзадачами. Каждый субагент работает независимо со своим контекстом, а возвращаются только финальные резюме — это сильно снижает расход токенов основной беседы.

### Используй execute_code для пакетных операций

Вместо выполнения терминальных команд по одной попроси агента написать скрипт, который сделает всё сразу. «Write a Python script to rename all `.jpeg` files to `.jpg` and run it» дешевле и быстрее, чем переименовывать файлы по отдельности.

### Выбирай правильную модель

Используй `/model`, чтобы переключать модели в середине сессии. Выбирай передовую модель (Claude Sonnet/Opus, GPT‑4o) для сложных рассуждений и архитектурных решений. Переключайся на более быструю модель для простых задач, таких как форматирование, переименование или генерация шаблонного кода.

:::tip
Запускай `/usage` периодически, чтобы видеть потребление токенов. Запускай `/insights` для более широкого обзора шаблонов использования за последние 30 дней.
:::
## Советы по обмену сообщениями

### Установить домашний канал

Используй `/sethome` в предпочитаемом чате Telegram или Discord, чтобы назначить его домашним каналом. Результаты cron‑задач и выводы запланированных задач будут доставляться сюда. Без этого у агента нет куда отправлять проактивные сообщения.

### Используй /title для организации сессий

Назначай названия своим сессиям с помощью `/title auth‑refactor` или `/title research‑llm‑quantization`. Именованные сессии легко находить с помощью `hermes sessions list` и возобновлять командой `hermes -r "auth‑refactor"`. Без названий сессии накапливаются и их становится невозможно различить.

### DM‑pairing для доступа команды

Вместо ручного сбора идентификаторов пользователей для allowlists включи DM‑pairing. Когда коллега отправит боту личное сообщение, он получит одноразовый код pairing. Ты одобряешь его командой `hermes pairing approve telegram XKGH5N7P` — просто и безопасно.

### Режимы отображения прогресса инструмента

Используй `/verbose`, чтобы контролировать, сколько активности инструмента ты видишь. В платформах обмена сообщениями обычно меньше — лучше. Оставь режим «new», чтобы видеть только новые вызовы инструмента. В CLI режим «all» предоставляет полноценный живой обзор всего, что делает агент.

:::tip
В платформах обмена сообщениями сессии автоматически сбрасываются после простоя (по умолчанию: 24 часа) или ежедневно в 4 утра. При необходимости более длительных сессий настройте параметр per‑platform в `~/.hermes/config.yaml`.
:::
## Безопасность

### Используй Docker для недоверенного кода

При работе с недоверёнными репозиториями или запуске незнакомого кода используй Docker или Daytona в качестве терминального backend. Установи `TERMINAL_BACKEND=docker` в своём файле `.env`. Разрушительные команды внутри контейнера не могут навредить хост‑системе.

```bash
# In your .env:
TERMINAL_BACKEND=docker
TERMINAL_DOCKER_IMAGE=hermes-sandbox:latest
```

### Избегай проблем с кодировкой в Windows

В Windows некоторые кодировки по умолчанию (например, `cp125x`) не могут представлять все символы Unicode, что может вызвать `UnicodeEncodeError` при записи файлов в тестах или скриптах.

- Предпочитай открывать файлы с явным указанием кодировки UTF‑8:

```python
with open("results.txt", "w", encoding="utf-8") as f:
    f.write("✓ All good\n")
```

- В PowerShell также можно переключить текущую сессию на UTF‑8 для вывода консоли и нативных команд:

```powershell
$OutputEncoding = [Console]::OutputEncoding = [Text.UTF8Encoding]::new($false)
```

Это переводит PowerShell и дочерние процессы на UTF‑8 и помогает избежать ошибок, характерных только для Windows.

### Проверяй перед выбором «always»

Когда агент запрашивает подтверждение опасной команды (`rm -rf`, `DROP TABLE` и т.п.), у тебя есть четыре варианта: **once**, **session**, **always**, **deny**. Подумай дважды, прежде чем выбрать «always» — это навсегда добавит шаблон в список разрешённых. Начинай с «session», пока не будешь уверен.

### Подтверждение команд — твоя страховка

Hermes проверяет каждую команду по тщательно отобранному списку опасных шаблонов перед выполнением. Это включает рекурсивные удаления, SQL‑удаления, передачу вывода `curl` в оболочку и многое другое. Не отключай эту проверку в продакшене — она существует по важным причинам.

:::warning
При работе в контейнерном backend (Docker, Singularity, Modal, Daytona) проверки опасных команд **пропускаются**, потому что контейнер является границей безопасности. Убедись, что образы контейнеров правильно защищены.
:::

### Используй allowlist‑ы для ботов обмена сообщениями

Никогда не устанавливай `GATEWAY_ALLOW_ALL_USERS=true` для бота с доступом к терминалу. Всегда используй платформенно‑специфические allowlist‑ы (`TELEGRAM_ALLOWED_USERS`, `DISCORD_ALLOWED_USERS`) или парную DM‑аутентификацию, чтобы контролировать, кто может взаимодействовать с твоим агентом.

```bash
# Recommended: explicit allowlists per platform
TELEGRAM_ALLOWED_USERS=123456789,987654321
DISCORD_ALLOWED_USERS=123456789012345678

# Or use cross-platform allowlist
GATEWAY_ALLOWED_USERS=123456789,987654321
```
