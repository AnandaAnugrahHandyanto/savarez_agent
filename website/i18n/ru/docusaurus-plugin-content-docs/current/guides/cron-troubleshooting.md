# Устранение проблем с cron

Когда задача cron ведёт себя не так, как ожидается, последовательно проверь следующие пункты. Большинство проблем попадают в одну из четырёх категорий: тайминг, доставка, разрешения или загрузка **skill**.
## Задачи не запускаются

### Проверка 1: Убедись, что задача существует и активна

```bash
hermes cron list
```

Найди задачу и проверь, что её состояние `[active]` (а не `[paused]` или `[completed]`). Если отображается `[completed]`, счётчик повторов может быть исчерпан — отредактируй задачу, чтобы сбросить его.

### Проверка 2: Убедись, что расписание корректно

Неправильно сформатированное расписание молча переходит в однократный запуск или полностью отклоняется. Проверь своё выражение:

| Твоё выражение | Должно оцениваться как |
|----------------|------------------------|
| `0 9 * * *` | 9:00 утра каждый день |
| `0 9 * * 1` | 9:00 утра каждый понедельник |
| `every 2h` | Каждые 2 часа, начиная с сейчас |
| `30m` | Через 30 минут |
| `2025-06-01T09:00:00` | 1 июня 2025 г., 9:00 утра UTC |

Если задача запускается один раз и затем исчезает из списка, это однократное расписание (`30m`, `1d` или ISO‑таймстамп) — ожидаемое поведение.

### Проверка 3: Запущен ли gateway?

Cron‑задачи запускаются фоновым тикером gateway, который срабатывает каждые 60 секунд. Обычная CLI‑сессия чата **не** автоматически запускает cron‑задачи.

Если ты ожидаешь автоматический запуск задач, нужен запущенный gateway (`hermes gateway` для foreground или `hermes gateway start` для установленного сервиса). Для одноразовой отладки можно вручную вызвать тикер командой `hermes cron tick`.

### Проверка 4: Проверь системные часы и часовой пояс

Задачи используют локальный часовой пояс. Если часы машины неверны или находятся в другом часовом поясе, задачи будут запускаться в неправильное время. Убедись:

```bash
date
hermes cron list   # Compare next_run times with local time
```

## Ошибки доставки

### Проверка 1: Убедись, что цель доставки указана правильно

Цели доставки чувствительны к регистру и требуют правильной настройки платформы. Неправильно настроенная цель тихо отбрасывает ответ.

| Цель | Требует |
|--------|----------|
| `telegram` | `TELEGRAM_BOT_TOKEN` в `~/.hermes/.env` |
| `discord` | `DISCORD_BOT_TOKEN` в `~/.hermes/.env` |
| `slack` | `SLACK_BOT_TOKEN` в `~/.hermes/.env` |
| `whatsapp` | Настроенный шлюз WhatsApp |
| `signal` | Настроенный шлюз Signal |
| `matrix` | Настроенный homeserver Matrix |
| `email` | SMTP, настроенный в `config.yaml` |
| `sms` | Настроенный провайдер SMS |
| `local` | Права записи в `~/.hermes/cron/output/` |
| `origin` | Доставляет в чат, где была создана задача |

Другие поддерживаемые платформы включают `mattermost`, `homeassistant`, `dingtalk`, `feishu`, `wecom`, `weixin`, `bluebubbles`, `qqbot` и `webhook`. Можно также указать конкретный чат с помощью синтаксиса `platform:chat_id` (например, `telegram:-1001234567890`).

Если доставка не удалась, задача всё равно выполняется — просто ничего не будет отправлено. Проверь `hermes cron list` на наличие обновлённого поля `last_error` (если доступно).

### Проверка 2: Использование `[SILENT]`

Если твой cron‑задание не выводит ничего или агент отвечает `[SILENT]`, доставка подавляется. Это сделано намеренно для мониторинговых задач — но убедись, что твой запрос случайно не подавляет всё.

Запрос вроде «respond with [SILENT] if nothing changed» будет тихо поглощать и непустые ответы. Проверь свою условную логику.

### Проверка 3: Разрешения токенов платформ

Каждому боту мессенджера нужны специфические разрешения для получения сообщений. Если доставка тихо не проходит:

- **Telegram**: Бот должен быть администратором в целевой группе/канале
- **Discord**: Бот должен иметь разрешение отправлять сообщения в целевом канале
- **Slack**: Бот должен быть добавлен в рабочее пространство и иметь область `chat:write`

### Проверка 4: Обёртка ответа

По умолчанию ответы cron‑задач оборачиваются заголовком и подвалом (`cron.wrap_response: true` в `config.yaml`). Некоторые платформы или интеграции могут плохо с этим справляться. Чтобы отключить:

```yaml
cron:
  wrap_response: false
```

## Ошибки загрузки навыков

### Проверка 1: Убедись, что навыки установлены

```bash
hermes skills list
```

Навыки должны быть установлены, прежде чем их можно будет привязать к заданиям cron. Если какой‑то навык отсутствует, установи его сначала с помощью `hermes skills install <skill-name>` или через `/skills` в CLI.

### Проверка 2: Сравни имя навыка и имя папки навыка

Имена навыков чувствительны к регистру и должны совпадать с именем папки установленного навыка. Если твоя задача указывает `ai-funding-daily-report`, а папка навыка называется иначе, проверь точное имя с помощью `hermes skills list`.

### Проверка 3: Навыки, требующие интерактивных инструментов

Задания cron запускаются с отключёнными наборами инструментов `cronjob`, `messaging` и `clarify`. Это предотвращает рекурсивное создание cron‑заданий, отправку прямых сообщений (доставка обрабатывается планировщиком) и интерактивные подсказки. Если навык полагается на эти наборы инструментов, он не будет работать в контексте cron.

Проверь документацию навыка, чтобы убедиться, что он работает в неинтерактивном (headless) режиме.

### Проверка 4: Порядок загрузки нескольких навыков

При использовании нескольких навыков они загружаются последовательно. Если Навык A зависит от контекста, предоставляемого Навыком B, убедись, что B загружается первым:

```bash
/cron add "0 9 * * *" "..." --skill context-skill --skill target-skill
```

В этом примере `context-skill` загружается перед `target-skill`.
## Ошибки и сбои заданий

### Проверка 1: Просмотр недавнего вывода задания

Если задание запустилось и завершилось с ошибкой, контекст ошибки можно увидеть в:

1. Чате, где задание доставляет результат (если доставка прошла успешно)
2. `~/.hermes/logs/agent.log` — сообщения планировщика (или `errors.log` — предупреждения)
3. Метаданные `last_run` задания через `hermes cron list`

### Проверка 2: Распространённые шаблоны ошибок

**«No such file or directory» для скриптов**
Путь к `script` должен быть абсолютным (или относительным к каталогу конфигурации Hermes). Проверь:

```bash
ls ~/.hermes/scripts/your-script.py   # Must exist
hermes cron edit <job_id> --script ~/.hermes/scripts/your-script.py
```

**«Skill not found» при выполнении задания**
Skill должен быть установлен на машине, где работает планировщик. При переходе между машинами skills не синхронизируются автоматически — переустанови их командой `hermes skills install <skill-name>`.

**Задание запускается, но ничего не доставляет**
Вероятно, проблема с целью доставки (см. раздел «Сбои доставки» выше) или тихо подавленный ответ (`[SILENT]`).

**Задание зависает или истекает время ожидания**
Планировщик использует таймаут, основанный на простое́нии (по умолчанию 600 с, настраивается переменной окружения `HERMES_CRON_TIMEOUT`, `0` — без ограничения). Агент может работать столько, сколько активно вызываются инструменты — таймер срабатывает только после продолжительного простоя. Длительные задания следует реализовывать скриптами, которые собирают данные и доставляют только итоговый результат.

### Проверка 3: Конкуренция за блокировки

Планировщик использует файловые блокировки, чтобы предотвратить наложение тиков. Если запущено два экземпляра gateway (или сессия CLI конфликтует с gateway), задания могут быть отложены или пропущены.

Убей дублирующие процессы gateway:

```bash
ps aux | grep hermes
# Kill duplicate processes, keep only one
```

### Проверка 4: Права доступа к `jobs.json`

Задания хранятся в `~/.hermes/cron/jobs.json`. Если этот файл недоступен для чтения/записи текущим пользователем, планировщик завершится тихо:

```bash
ls -la ~/.hermes/cron/jobs.json
chmod 600 ~/.hermes/cron/jobs.json   # Your user should own it
```
## Проблемы производительности

### Медленный запуск задания

Каждое задание `cron` создаёт новую сессию AIAgent, что может включать аутентификацию провайдера и загрузку модели. Для расписаний, чувствительных ко времени, добавь запас времени (например, `0 8 * * *` вместо `0 9 * * *`).

### Слишком много перекрывающихся заданий

Планировщик выполняет задания последовательно на каждом тике. Если несколько заданий запланированы на одно и то же время, они выполняются одно за другим. Рассмотри возможность распределения расписаний (например, `0 9 * * *` и `5 9 * * *` вместо обоих в `0 9 * * *`), чтобы избежать задержек.

### Большой объём вывода скрипта

Скрипты, выводящие мегабайты данных, замедляют агента и могут превысить лимиты токенов. Фильтруй/суммируй вывод на уровне скрипта — выводи только то, что агенту нужно для рассуждений.
## Команды диагностики

```bash
hermes cron list                    # Show all jobs, states, next_run times
hermes cron run <job_id>            # Schedule for next tick (for testing)
hermes cron edit <job_id>           # Fix configuration issues
hermes logs                         # View recent Hermes logs
hermes skills list                  # Verify installed skills
```

## Получение дополнительной помощи

Если ты прошёл этот гид, а проблема остаётся:

1. Запусти задачу командой `hermes cron run <job_id>` (сработает на следующем tick gateway) и следи за ошибками в выводе чата
2. Проверь `~/.hermes/logs/agent.log` на предмет сообщений планировщика и `~/.hermes/logs/errors.log` на предупреждения
3. Открой issue на [github.com/NousResearch/hermes-agent](https://github.com/NousResearch/hermes-agent) с:
   - ID задачи и расписанием
   - Целью доставки
   - Что ты ожидал и что произошло
   - Соответствующими сообщениями об ошибках из логов
