# Справочник слеш‑команд

Hermes имеет два интерфейса слеш‑команд, оба управляются центральным `COMMAND_REGISTRY` в `hermes_cli/commands.py`:

- **Интерактивные слеш‑команды CLI** — обрабатываются `cli.py`, с автодополнением из реестра
- **Слеш‑команды обменом сообщениями** — обрабатываются `gateway/run.py`, справочный текст и меню платформ генерируются из реестра

Установленные навыки также доступны как динамические слеш‑команды на обоих интерфейсах. Это включает встроенные навыки, такие как `/plan`, который открывает режим планирования и сохраняет markdown‑планы в `.hermes/plans/` относительно активного рабочего каталога/рабочего каталога бэкенда.
## Права доступа и разделение администраторов/пользователей

Каждая платформа обмена сообщениями, поддерживающая список разрешённых пользователей (Telegram, Discord, Slack, Matrix, Mattermost, Signal, …), также поддерживает двухуровневое разделение слеш‑команд: **администраторы** получают все зарегистрированные команды, **обычные пользователи** — только те, которые указаны в `user_allowed_commands` (плюс всегда разрешённые базовые `/help` и `/whoami`). Настрой `allow_admin_from` и `user_allowed_commands` (а также эквиваленты для групп `group_allow_admin_from` / `group_user_allowed_commands`) внутри блока `extra:` в файле `~/.hermes/gateway-config.yaml`.

См. документацию по каждой платформе для примеров — структура одинаковая для всех платформ:

- [Telegram](../user-guide/messaging/telegram.md#slash-command-access-control)
- [Discord](../user-guide/messaging/discord.md)
- [Slack](../user-guide/messaging/slack.md)
- [Matrix](../user-guide/messaging/matrix.md)
- [Mattermost](../user-guide/messaging/mattermost.md)
- [Signal](../user-guide/messaging/signal.md)

Если `allow_admin_from` не задан для области, эта область остаётся в режиме неограниченной обратной совместимости — каждый разрешённый пользователь может выполнять любую команду.
## Интерактивные CLI‑команды со слешем

Нажми `/` в CLI, чтобы открыть меню автодополнения. Встроенные команды нечувствительны к регистру.

### Сессия

| Команда | Описание |
|---------|----------|
| `/new [name]` (alias: `/reset`) | Начать новую сессию (новый ID сессии + история). Необязательный `[name]` задаёт начальный заголовок сессии — например, `/new my-experiment` открывает свежую сессию уже с заголовком `my-experiment`, чтобы её было легко найти позже с помощью `/resume` или `/sessions`. Добавь `now`, `--yes` или `-y`, чтобы пропустить модальное окно подтверждения — например, `/reset now`, `/new --yes my-experiment`. |
| `/clear` | Очистить экран и начать новую сессию |
| `/history` | Показать историю диалога |
| `/save` | Сохранить текущий диалог |
| `/retry` | Повторить последнее сообщение (отправить агенту заново) |
| `/undo` | Удалить последний обмен пользователь / ассистент |
| `/title` | Установить заголовок для текущей сессии (использование: `/title My Session Name`) |
| `/compress [focus topic]` | Вручную сжать контекст разговора (сбросить память + суммировать). Необязательная тема фокуса ограничивает, что сохраняется в сводке. |
| `/rollback` | Показать или восстановить контрольные точки файловой системы (использование: `/rollback [number]`) |
| `/snapshot [create\|restore <id>\|prune]` (alias: `/snap`) | Создать или восстановить снимок состояния конфигурации/состояния Hermes. `create [label]` сохраняет снимок, `restore <id>` откатывает к нему, `prune [N]` удаляет старые снимки, без аргументов выводит список. |
| `/stop` | Убить все запущенные фоновые процессы |
| `/queue <prompt>` (alias: `/q`) | Поставить запрос в очередь на следующий ход (не прерывает текущий ответ агента). |
| `/steer <prompt>` | Вставить заметку в середине выполнения, которая придёт агенту **после следующего вызова инструмента** — без прерывания, без нового хода пользователя. Текст добавляется к содержимому последнего результата инструмента после его завершения, предоставляя агенту новый контекст без разрыва текущего цикла вызовов инструментов. Используй для корректировки направления в середине задачи (например, «сфокусироваться на модуле auth», пока агент запускает тесты). |
| `/goal <text>` | Установить постоянную цель, к которой Hermes будет стремиться на протяжении ходов — наша реализация цикла Ralph. После каждого хода вспомогательная модель‑судья решает, выполнена ли цель; если нет, Hermes продолжает автоматически. Подкоманды: `/goal status`, `/goal pause`, `/goal resume`, `/goal clear`. По умолчанию бюджет — 20 ходов (`goals.max_turns`); любое реальное сообщение пользователя прерывает цикл продолжения, а состояние сохраняется при `/resume`. См. [Persistent Goals](/user-guide/features/goals) для полного руководства. |
| `/subgoal <text>` | Добавить пользовательский критерий к активной цели в середине цикла. Промпт продолжения передаёт все подцели агенту дословно, и судья учитывает их при вынесении вердикта DONE/CONTINUE — цель считается выполненной только когда выполнена оригинальная цель **и** все подцели. Подкоманды: `/subgoal` (список), `/subgoal remove <N>`, `/subgoal clear`. Требуется активный `/goal`. |
| `/resume [name]` | Возобновить ранее названную сессию |
| `/sessions` (TUI alias: `/switch`) | Классический CLI: просмотреть и возобновить предыдущие сессии в интерактивном выборе. TUI: открыть переключатель живых сессий для текущих открытых TUI‑сессий. В TUI можно выполнить `/sessions new`, чтобы сразу запустить ещё одну живую сессию. |
| `/redraw` | Принудительно полностью перерисовать UI (восстанавливает после дрейфа терминала после изменения размера tmux, артефактов выделения мышью и т.п.) |
| `/status` | Показать информацию о сессии — модель, провайдер, профиль, ID сессии, рабочий каталог, заголовок, timestamps создания/обновления, суммарные токены, состояние запущенного агента — затем локальный блок **Session recap** (количество недавних ходов пользователя/ассистента, количество результатов инструментов, топ‑используемые инструменты, последние изменённые файлы, последний пользовательский запрос и последний ответ ассистента). Резюме вычисляется локально из памяти разговора; без вызова LLM, без влияния кеша промптов. |
| `/agents` (alias: `/tasks`) | Показать активных агентов и запущенные задачи в текущей сессии. |
| `/background <prompt>` (alias: `/bg`, `/btw`) | Запустить запрос в отдельной фоновой сессии. Агент обрабатывает запрос независимо — текущая сессия остаётся свободной для другой работы. Результаты появляются в панели после завершения задачи. См. [CLI Background Sessions](/user-guide/cli#background-sessions). |
| `/branch [name]` (alias: `/fork`) | Создать ветку текущей сессии (исследовать иной путь) |
| `/handoff <platform>` | **Только CLI.** Передать текущую сессию в мессенджер (Telegram, Discord, Slack, WhatsApp, Signal, Matrix). Шлюз сразу подхватывает её, создаёт новую ветку в платформах, поддерживающих ветки (темы Telegram, ветки текстовых каналов Discord, ветки сообщений Slack), пере‑привязывает назначение к `session_id` CLI, воспроизводит полный ролевой транскрипт и генерирует синтетический пользовательский ход, подтверждая работу в новом месте. При успешном завершении CLI выходит чисто с подсказкой `/resume`; локально можно возобновить в любое время с `/resume <title>`. Отклонено в середине хода. Требуется запущенный шлюз и настроенный домашний канал для целевой платформы (`/sethome` из чата назначения). См. [Cross-Platform Handoff](/user-guide/sessions#cross-platform-handoff). |

### Быстрые команды

Пользовательские быстрые команды сопоставляют короткую слеш‑команду либо shell‑команде, либо другой слеш‑команде. Настраиваются в `~/.hermes/config.yaml`:

```yaml
quick_commands:
  status:
    type: exec
    command: systemctl status hermes-agent
  deploy:
    type: exec
    command: scripts/deploy.sh
  inbox:
    type: alias
    target: /gmail unread
```

Затем вводи `/status`, `/deploy` или `/inbox` в CLI или в мессенджере. Быстрые команды разрешаются во время диспетчеризации и могут не отображаться во всех встроенных таблицах автодополнения/справки.

Команды‑сокращения, состоящие только из строки‑промпта, не поддерживаются как быстрые команды. Помести более длинные переиспользуемые промпты в навык или используй `type: alias`, чтобы сослаться на существующую слеш‑команду.

### Пользовательские алиасы моделей

Определи свои короткие имена для часто используемых моделей, а затем вызывай их через `/model <alias>` в CLI или в любом мессенджере. Алиасы работают одинаково в обоих случаях, как в режиме только сессии (по умолчанию), так и с переключателем `--global`.

Поддерживаются два формата конфигурации:

**Полная форма** — зафиксировать точную модель, провайдера и, при необходимости, базовый URL. Помести в `~/.hermes/config.yaml`:

```yaml
model_aliases:
  fav:
    model: claude-sonnet-4.6
    provider: anthropic
  grok:
    model: grok-4
    provider: x-ai
  ollama-qwen:
    model: qwen3-coder:30b
    provider: custom
    base_url: http://localhost:11434/v1
```

**Краткая форма** — `provider/model` в одной строке. Устанавливается из оболочки без редактирования YAML:

```bash
hermes config set model.aliases.fav anthropic/claude-opus-4.6
hermes config set model.aliases.grok x-ai/grok-4
```

Затем в чате:

```
/model fav            # session-only
/model grok --global  # also persists current-model change to config.yaml
```

Пользовательские алиасы имеют приоритет над встроенными короткими именами, поэтому объявление алиаса `sonnet`, `kimi`, `opus` и т.п. перекроет встроенный. Имена алиасов нечувствительны к регистру.

### Разрешение алиасов

Команды поддерживают поиск по префиксу: ввод `/h` разрешается как `/help`, `/mod` — как `/model`. Когда префикс неоднозначен (соответствует нескольким командам), победителем становится первая команда в порядке реестра. Полные имена команд и зарегистрированные алиасы всегда имеют приоритет над префиксными совпадениями.
## Команды слеш‑мессенджинга

Шлюз обмена сообщениями поддерживает следующие встроенные команды в чатах Telegram, Discord, Slack, WhatsApp, Signal, Email, Home Assistant и Teams:

| Команда | Описание |
|---------|----------|
| `/start` | Команда протокола платформы. Многие чат‑платформы (Telegram, Discord, …) отправляют `/start` автоматически при первом открытии пользователем диалога с ботом. Hermes молча подтверждает пинг — без ответа агента, без расхода сессии — так что рукопожатия при первом контакте не тратят ход. Также её можно отправить явно, чтобы убедиться, что шлюз доступен. |
| `/new` | Начать новую сессию. |
| `/reset` | Сбросить историю сессии. |
| `/status` | Показать информацию о сессии, затем блок **Session recap** с локальными данными (количество недавних ходов, топ‑инструменты, затронутые файлы, последний запрос + ответ). |
| `/stop` | Прервать все запущенные фоновые процессы и остановить работающего агента. |
| `/model [provider:model]` | Показать или изменить модель. Поддерживает переключение провайдеров (`/model zai:glm-5`), пользовательские эндпоинты (`/model custom:model`), именованные пользовательские провайдеры (`/model custom:local:qwen`), автоопределение (`/model custom`) и пользовательские алиасы (`/model fav`, `/model grok` — см. [Custom model aliases](#custom-model-aliases)). Используй `--global`, чтобы сохранить изменение в `config.yaml`. **Примечание:** `/model` может переключать только уже сконфигурированные провайдеры. Чтобы добавить новый провайдер или задать API‑ключи, используй `hermes model` в терминале (вне чат‑сессии). |
| `/codex-runtime [auto\|codex_app_server\|on\|off]` | Переключить необязательный [Codex app‑server runtime](../user-guide/features/codex-app-server-runtime). Сохраняется в `model.openai_runtime` в `config.yaml` и сбрасывает кэш агента, чтобы следующее сообщение использовало новый рантайм. Действует со следующей сессии. |
| `/personality [name]` | Установить наложение личности для сессии. |
| `/fast [normal\|fast\|status]` | Переключить быстрый режим — OpenAI Priority Processing / Anthropic Fast Mode. |
| `/retry` | Повторить последнее сообщение. |
| `/undo` | Удалить последний обмен. |
| `/sethome` (alias: `/set-home`) | Пометить текущий чат как домашний канал платформы для доставки. |
| `/compress [focus topic]` | Вручную сжать контекст сессии. Необязательная тема фокусировки уточняет, что именно сохраняется в сводке. |
| `/topic [off\|help\|session-id]` | **Только в Telegram DM.** Управление режимом многосессионных тем, задаваемым пользователем. `/topic` включает его или показывает статус; `/topic off` отключает и очищает привязки; `/topic help` выводит справку; `/topic <session-id>` внутри темы восстанавливает предыдущую сессию. См. [Multi-session DM mode](/user-guide/messaging/telegram#multi-session-dm-mode-topic). |
| `/title [name]` | Установить или показать заголовок сессии. |
| `/resume [name]` | Возобновить ранее названную сессию. |
| `/usage` | Показать использование токенов, оценку стоимости (ввод/вывод), состояние окна контекста, длительность сессии и — при наличии у активного провайдера — раздел **Account limits** с оставшейся квотой/кредитами, полученными в реальном времени из API провайдера. |
| `/insights [days]` | Показать аналитические данные об использовании. |
| `/reasoning [level\|show\|hide]` | Изменить уровень рассуждений или переключить их отображение. |
| `/voice [on\|off\|tts\|join\|channel\|leave\|status]` | Управление голосовыми ответами в чате. `join`/`channel`/`leave` управляют режимом голосового канала Discord. |
| `/rollback [number]` | Вывести список или восстановить контрольные точки файловой системы. |
| `/background <prompt>` | Запустить запрос в отдельной фоновой сессии. Результаты возвращаются в тот же чат после завершения задачи. См. [Messaging Background Sessions](/user-guide/messaging/#background-sessions). |
| `/queue <prompt>` (alias: `/q`) | Поставить запрос в очередь на следующий ход, не прерывая текущий. |
| `/steer <prompt>` | Вставить сообщение после следующего вызова инструмента без прерывания — модель учтёт его в следующей итерации, а не как новый ход. |
| `/goal <text>` | Установить постоянную цель, к которой Hermes будет стремиться на протяжении ходов — наш вариант цикла Ralph. Модель‑судья проверяет после каждого хода; если цель не достигнута, Hermes автоматически продолжает, пока цель не выполнена, пользователь её не приостановит/не очистит, или не будет исчерпан бюджет ходов (по умолчанию 20). Подкоманды: `/goal status`, `/goal pause`, `/goal resume`, `/goal clear`. Безопасно запускать в середине работы агента для статуса/паузы/очистки; установка новой цели требует предварительного `/stop`. См. [Persistent Goals](/user-guide/features/goals). |
| `/footer [on\|off\|status]` | Переключить отображение футера с метаданными рантайма в финальных ответах (модель, количество инструментов, время). |
| `/curator [status\|run\|pin\|archive]` | Управление обслуживанием фоновых навыков. |
| `/kanban <action>` | Управлять доской совместной работы (мульти‑профиль, мульти‑проект) из чата — те же аргументы, что и в CLI. Обходит защиту работающего агента, поэтому `/kanban unblock t_abc`, `/kanban comment t_abc "…"`, `/kanban list --mine`, `/kanban boards switch <slug>` и т.п. работают в середине хода. `/kanban create …` автоматически подписывает исходный чат на события терминала новой задачи. См. [Kanban slash command](/user-guide/features/kanban#kanban-slash-command). |
| `/reload-mcp` (alias: `/reload_mcp`) | Перезагрузить серверы MCP из конфигурации. |
| `/yolo` | Переключить режим YOLO — пропустить все запросы подтверждения опасных команд. |
| `/commands [page]` | Просмотр всех команд и навыков (постранично). |
| `/approve [session\|always]` | Утвердить и выполнить ожидающую опасную команду. `session` одобряет только для текущей сессии; `always` добавляет в постоянный список разрешённых. |
| `/deny` | Отклонить ожидающую опасную команду. |
| `/update` | Обновить Hermes Agent до последней версии. |
| `/restart` | Аккуратно перезапустить шлюз после завершения активных запусков. Когда шлюз снова станет онлайн, он отправит подтверждение в чат/тред запросившего. |
| `/debug` | Загрузить отчёт отладки (системная информация + логи) и получить ссылки для совместного доступа. |
| `/help` | Показать справку по мессенджингу. |
| `/<skill-name>` | Вызвать любой установленный навык по имени. |
## Примечания

- `/skin`, `/snapshot`, `/gquota`, `/reload`, `/tools`, `/toolsets`, `/browser`, `/config`, `/cron`, `/skills`, `/platforms`, `/paste`, `/image`, `/statusbar`, `/plugins`, `/busy`, `/indicator`, `/redraw`, `/clear`, `/history`, `/save`, `/copy`, `/handoff` и `/quit` — команды **только CLI**.
- `/verbose` **по умолчанию только CLI**, но может быть включён для платформ обмена сообщениями, задав `display.tool_progress_command: true` в `config.yaml`. При включении он переключает режим `display.tool_progress` и сохраняет его в конфиг.
- `/sethome`, `/update`, `/restart`, `/approve`, `/deny`, `/topic` и `/commands` — команды **только для обмена сообщениями**.
- `/status`, `/background`, `/queue`, `/steer`, `/voice`, `/reload-mcp`, `/reload-skills`, `/rollback`, `/debug`, `/fast`, `/footer`, `/curator`, `/kanban`, `/sessions` и `/yolo` работают **как в CLI, так и в шлюзе обмена сообщениями**.
- `/voice join`, `/voice channel` и `/voice leave` имеют смысл только в Discord.
- В TUI команда `/sessions` показывает активные сессии в текущем процессе TUI. Используй `/resume [name]` или `hermes --tui --resume <id-or-title>` для сохранённых или закрытых транскриптов.
## Confirmation prompts for destructive commands

CLI запрашивает подтверждение перед выполнением slash‑команд, которые удаляют несохранённое состояние сессии. Текущий набор деструктивных команд:

| Команда | Что она удаляет |
|---------|-----------------|
| `/clear` | Очищает экран и начинает новую сессию — текущий ID сессии и история в памяти исчезают. |
| `/new` / `/reset` | Запускает новую сессию (новый ID сессии + пустая история). |
| `/undo` | Удаляет последний обмен пользователь/ассистент из истории. |
| `/exit --delete` / `/quit --delete` | Выходит **и** навсегда удаляет SQLite‑историю текущей сессии и файлы транскриптов на диске. |

Для каждой из этих команд CLI открывает модальное окно с тремя вариантами: **Approve Once** (подтвердить только сейчас), **Always Approve** (подтвердить и сохранить `approvals.destructive_slash_confirm: false`, чтобы в будущем деструктивные команды выполнялись без запроса) или **Cancel**.

**Inline skip:** добавь `now`, `--yes` или `-y`, чтобы обойти модальное окно для одного вызова — например, `/reset now`, `/new --yes my-session`, `/clear -y`, `/undo -y`. Это полезно, когда модальное окно не отображается корректно в твоём терминале (см. [issue #30768](https://github.com/NousResearch/hermes-agent/issues/30768) для нативного Windows PowerShell) или при скриптинге CLI.

Установи `approvals.destructive_slash_confirm: false` в `~/.hermes/config.yaml`, чтобы отключить запросы глобально; верни значение в `true`, чтобы включить их обратно. См. [Security — Destructive slash command confirmation](../user-guide/security.md#dangerous-command-approval) для контекста.
