# Каталог моделей

Hermes получает отобранные списки моделей для **OpenRouter** и **Nous Portal** из JSON‑манифеста, размещённого рядом с сайтом документации. Это позволяет поддерживающим обновлять списки выбора без выпуска новой версии `hermes-agent`.

Когда манифест недоступен (офлайн, блокировка сети, сбой хостинга), Hermes тихо переходит к снимку в репозитории, который поставляется вместе с CLI. Манифест никогда не ломает выборщик — в худшем случае ты увидишь список, который был включён в установленную версию.

## URL живого манифеста

```
https://hermes-agent.nousresearch.com/docs/api/model-catalog.json
```

Публикуется при каждом слиянии в `main` через существующий pipeline `deploy-site.yml` GitHub Pages. Истина находится в репозитории по пути `website/static/api/model-catalog.json`.

## Схема

```json
{
  "version": 1,
  "updated_at": "2026-04-25T22:00:00Z",
  "metadata": {},
  "providers": {
    "openrouter": {
      "metadata": {},
      "models": [
        {"id": "moonshotai/kimi-k2.6", "description": "recommended", "metadata": {}},
        {"id": "openai/gpt-5.4",       "description": ""}
      ]
    },
    "nous": {
      "metadata": {},
      "models": [
        {"id": "anthropic/claude-opus-4.7"},
        {"id": "moonshotai/kimi-k2.6"}
      ]
    }
  }
}
```

Примечания к полям:

- **`version`** — целочисленная версия схемы. В будущих версиях значение будет увеличиваться; Hermes отклоняет манифесты с версиями, которые он не понимает, и переходит к жёстко заданному снимку.
- **`metadata`** — произвольный словарь на уровне манифеста, провайдера и модели. Любые ключи. Hermes игнорирует неизвестные поля, поэтому ты можешь аннотировать записи (`"tier": "paid"`, `"tags": [...]` и т.д.) без необходимости менять схему.
- **`description`** — только для OpenRouter. Управляет текстом бейджа в выборщике (`"recommended"`, `"free"` или пусто). Nous Portal не использует это — определение бесплатного уровня берётся в реальном времени из эндпоинта ценообразования Портала.
- **Цены и длина контекста** НЕ находятся в манифесте. Они получаются из живых API провайдеров (`/v1/models` эндпоинты, models.dev) во время запроса.

## Поведение при получении

| Когда | Что происходит |
|---|---|
| `/model` или `hermes model` | Запрашивает, если кэш на диске устарел, иначе использует кэш |
| Кэш на диске свежий (< TTL) | Нет сетевого запроса |
| Сетевая ошибка при наличии кэша | Тихий переход к кэшу, одна строка в логе |
| Сетевая ошибка, кэша нет | Тихий переход к снимку в репозитории |
| Манифест не проходит проверку схемы | Считается недоступным |

Расположение кэша: `~/.hermes/cache/model_catalog.json`.

## Конфигурация

```yaml
model_catalog:
  enabled: true
  url: https://hermes-agent.nousresearch.com/docs/api/model-catalog.json
  ttl_hours: 24
  providers: {}
```

Установи `enabled: false`, чтобы полностью отключить удалённый запрос и всегда использовать снимок в репозитории.

### Переопределение URL‑ов per‑provider

Третьи стороны могут самостоятельно хостить свой список курируемых моделей, используя ту же схему. Укажи провайдеру пользовательский URL:

```yaml
model_catalog:
  providers:
    openrouter:
      url: https://example.com/my-openrouter-curation.json
```

Переопределяющий манифест должен заполнить только те блоки провайдера, которые ему нужны. Остальные провайдеры продолжают получать данные из основного URL.

## Обновление манифеста

Поддерживающие:

```bash
# Re-generate from the in-repo hardcoded lists (keeps manifest in sync after
# editing OPENROUTER_MODELS or _PROVIDER_MODELS["nous"] in hermes_cli/models.py).
python scripts/build_model_catalog.py
```

Затем создай PR с изменением `website/static/api/model-catalog.json` в `main`. Сайт документации автоматически разворачивается после слияния, и новый манифест становится доступным в течение нескольких минут.

Ты также можешь вручную отредактировать JSON для точечных изменений метаданных, которые не должны попадать в снимок репозитория — скрипт‑генератор служит лишь удобством, а не единственным источником правды.
